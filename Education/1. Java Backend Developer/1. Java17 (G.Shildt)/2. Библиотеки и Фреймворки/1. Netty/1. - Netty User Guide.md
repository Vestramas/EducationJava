## 1.1 Написание Discard-сервера
**Самый простой протокол в мире** — это не 'Hello, World!', а `DISCARD`. 
Это протокол, который отбрасывает любые полученные данные без какого-либо ответа.

Чтобы реализовать протокол `DISCARD`, все, что вам нужно сделать, — это игнорировать все полученные данные. 

Давайте начнем сразу с реализации обработчика, который обрабатывает события ввода-вывода, генерируемые Netty.
```java
package io.netty.example.discard;

import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;

/**
 * Обрабатывает канал на стороне сервера.
 */
public class DiscardServerHandler extends ChannelInboundHandlerAdapter { // (1)

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) { // (2)
        // Молча игнорируем полученные данные.
        ((ByteBuf) msg).release(); // (3)
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) { // (4)
        // Закрываем соединение при возникновении исключения.
        cause.printStackTrace();
        ctx.close();
    }
}
```

1. `DiscardServerHandler` расширяет [`ChannelInboundHandlerAdapter`](https://netty.io/4.1/api/io/netty/channel/ChannelInboundHandlerAdapter.html), который является **реализацией** [`ChannelInboundHandler`](https://netty.io/4.1/api/io/netty/channel/ChannelInboundHandler.html). 
   
   [`ChannelInboundHandler`](https://netty.io/4.1/api/io/netty/channel/ChannelInboundHandler.html) **предоставляет различные** ==методы-обработчики событий==, которые вы можете переопределить. 
   Пока что достаточно расширить `ChannelInboundHandlerAdapter`, а не реализовывать интерфейс обработчика самостоятельно.
    
2. Здесь мы переопределяем метод-обработчик событий `channelRead()`. 
   Этот **метод вызывается с полученным сообщением всякий раз, когда от клиента поступают новые данные**. В этом примере тип полученного сообщения — `ByteBuf`.
    
3. *Чтобы реализовать протокол* `DISCARD`, *обработчик должен игнорировать полученное сообщение.* 
   `ByteBuf` — это **объект с подсчетом ссылок,** ***который должен быть явно освобожден с помощью метода*** `release()`. 
   Помните, что ==обработчик несет ответственность за освобождение любого переданного ему объекта с подсчетом ссылок.== Обычно метод-обработчик `channelRead()` реализуется следующим образом:
```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    try {
        // Что-то делаем с msg
    } finally {
        ReferenceCountUtil.release(msg);
    }
}
```
	
4. Метод-обработчик событий `exceptionCaught()` **вызывается с объектом** `Throwable`, **когда исключение было вызвано Netty из-за ошибки ввода-вывода** ИЛИ **реализацией обработчика из-за исключения, возникшего во время обработки событий.** 
   В большинстве случаев *пойманное исключение должно быть зарегистрировано, а связанный с ним канал должен быть закрыт,* хотя реализация этого метода может отличаться в зависимости от того, что вы хотите сделать для обработки исключительной ситуации. Например, вы можете отправить ответное сообщение с кодом ошибки перед закрытием соединения.

Пока все хорошо. Мы реализовали первую половину `DISCARD`-сервера. Теперь осталось написать метод `main()`, который запускает сервер с `DiscardServerHandler`.
```java
package io.netty.example.discard;

import io.netty.bootstrap.ServerBootstrap;
import io.netty.channel.ChannelFuture;
import io.netty.channel.ChannelInitializer;
import io.netty.channel.ChannelOption;
import io.netty.channel.EventLoopGroup;
import io.netty.channel.nio.NioEventLoopGroup;
import io.netty.channel.socket.SocketChannel;
import io.netty.channel.socket.nio.NioServerSocketChannel;

/**
 * Отбрасывает любые входящие данные.
 */
public class DiscardServer {

    private int port;

    public DiscardServer(int port) {
        this.port = port;
    }

    public void run() throws Exception {
        EventLoopGroup bossGroup = new NioEventLoopGroup(); // (1)
        EventLoopGroup workerGroup = new NioEventLoopGroup();
        try {
            ServerBootstrap b = new ServerBootstrap(); // (2)
            b.group(bossGroup, workerGroup)
             .channel(NioServerSocketChannel.class) // (3)
             .childHandler(new ChannelInitializer<SocketChannel>() { // (4)
                 @Override
                 public void initChannel(SocketChannel ch) throws Exception {
                     ch.pipeline().addLast(new DiscardServerHandler());
                 }
             })
             .option(ChannelOption.SO_BACKLOG, 128)          // (5)
             .childOption(ChannelOption.SO_KEEPALIVE, true); // (6)

            // Привязываемся к порту и начинаем принимать входящие соединения.
            ChannelFuture f = b.bind(port).sync(); // (7)

            // Ждем, пока серверный сокет не будет закрыт.
            // В этом примере этого не происходит, но вы можете сделать это для корректного
            // завершения работы вашего сервера.
            f.channel().closeFuture().sync();
        } finally {
            workerGroup.shutdownGracefully();
            bossGroup.shutdownGracefully();
        }
    }

    public static void main(String[] args) throws Exception {
        int port = 8080;
        if (args.length > 0) {
            port = Integer.parseInt(args[0]);
        }

        new DiscardServer(port).run();
    }
}
```

1. `NioEventLoopGroup` — это **многопоточный цикл обработки событий,** (типо `ExecutorService`) который управляет операциями ввода-вывода. 
   **Netty предоставляет различные реализации** `EventLoopGroup` **для разных видов транспорта**. 
   В этом примере мы реализуем серверное приложение, поэтому будут использоваться две группы `NioEventLoopGroup`. 
	- Первая, часто называемая `boss`, **принимает входящие соединения.** 
	- Вторая, часто называемая `worker`, **обрабатывает трафик принятого соединения** после того, как босс принял соединение и зарегистрировал его в рабочей группе.
	Количество используемых потоков и то, как они сопоставляются с созданными каналами, зависит от реализации `EventLoopGroup` и может быть даже настраиваемым через конструктор.
    
2. `ServerBootstrap` — ==это вспомогательный класс, который настраивает сервер.== 
   ***Вы можете настроить сервер, используя напрямую*** `Channel`. Однако обратите внимание, что **это утомительный процесс, и в большинстве случаев вам не нужно этого делать.**
    
3. Здесь мы указываем использовать класс `NioServerSocketChannel`, который **используется для создания нового канала, принимающего входящие соединения.**
    
4. **Обработчик**, ***указанный здесь***, **будет всегда использоваться для каждого вновь принятого** `Channel`. 
   
   `ChannelInitializer` — это специальный **обработчик**, предназначенный д**ля помощи пользователю настроить новый** `Channel`. 
   
   Скорее всего, ***вы захотите настроить*** `ChannelPipeline` нового `Channel`, **добавив несколько обработчиков, таких как** `DiscardServerHandler`, для реализации вашего сетевого приложения. 
   
   По мере усложнения приложения вы, вероятно, добавите больше обработчиков в конвейер и в конечном итоге выделите этот анонимный класс в класс верхнего уровня.
    
5. Вы также можете **установить параметры, специфичные для реализации** `Channel`. 
   Мы пишем TCP/IP сервер, поэтому мы можем **устанавливать параметры сокетов, такие как** `tcpNoDelay` и `keepAlive`. Пожалуйста, обратитесь к документации по API `ChannelOption` и конкретным реализациям `ChannelConfig`, чтобы получить общее представление о поддерживаемых `ChannelOption`.
    
6. Заметили разницу между `option()` и `childOption()`? 
   
   - `option()` применяется к `NioServerSocketChannel`, который **принимает входящие соединения.** 
   - `childOption()` *применяется к каналам*, **принятым родительским** `ServerChannel`, в данном случае это `NioSocketChannel`.
    
7. Теперь мы готовы к работе. Осталось привязаться к порту и запустить сервер. Здесь мы привязываемся к порту `8080` на всех сетевых интерфейсах машины. 
   Теперь вы можете вызывать метод `bind()` столько раз, сколько захотите (с разными адресами).
    

Поздравляем! Вы только что завершили свой первый сервер на основе Netty.

### Изучение полученных данных
Теперь, когда мы написали наш первый сервер, нужно проверить, действительно ли он работает. Самый простой способ протестировать его — использовать команду telnet. Например, вы можете ввести `telnet localhost 8080` в командной строке и что-нибудь напечатать.

Однако можем ли мы сказать, что сервер работает правильно? Мы не можем этого знать наверняка, потому что это сервер отбрасывания данных. Вы вообще не получите никакого ответа. Чтобы доказать, что он действительно работает, давайте изменим сервер так, чтобы он выводил то, что получил.

Мы уже знаем, что метод `channelRead()` вызывается каждый раз при получении данных. Давайте добавим код в метод `channelRead()` класса `DiscardServerHandler`:
```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    ByteBuf in = (ByteBuf) msg;
    try {
        while (in.isReadable()) { // (1)
            System.out.print((char) in.readByte());
            System.out.flush();
        }
    } finally {
        ReferenceCountUtil.release(msg); // (2)
    }
}
```
1. Этот неэффективный цикл можно упростить до: `System.out.println(in.toString(io.netty.util.CharsetUtil.US_ASCII))`
    
2. В качестве альтернативы вы можете использовать `in.release()`.
    

Если вы снова запустите telnet, то увидите, что сервер печатает полученные данные.

В моем случае telnet в cmd не оказалось, поэтому использую `curl`
```bash
curl http://localhost:8080
```

## 1.2 Написание Echo-сервера
До сих пор мы только потребляли данные, никак не отвечая. Однако сервер обычно должен отвечать на запрос. Давайте научимся отправлять ответ клиенту, реализовав протокол `ECHO`, при котором любые полученные данные отправляются обратно.

Единственное отличие от discard-сервера, который мы реализовали в предыдущих разделах, заключается в том, что он отправляет полученные данные обратно, а не выводит их на консоль. Поэтому снова достаточно изменить метод `channelRead()`:
```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    ctx.write(msg); // (1)
    ctx.flush(); // (2)
}
```
1. Объект `ChannelHandlerContext` **предоставляет различные операции, позволяющие инициировать различные события** ==и операции ввода-вывода. ==Здесь мы вызываем `write(Object)`, чтобы записать полученное сообщение дословно. Обратите внимание, что мы не освободили полученное сообщение, как в примере с `DISCARD`. Это потому, что Netty освобождает его за вас, когда оно записывается в канал.
    
2. `ctx.write(Object)` не приводит к немедленной записи сообщения в канал. Оно внутренне буферизуется, а затем сбрасывается в канал с помощью `ctx.flush()`. Как вариант, для краткости вы можете вызвать `ctx.writeAndFlush(msg)`.
    

Если вы снова запустите telnet, то увидите, что сервер отправляет обратно все, что вы ему отправили.

Полный исходный код echo-сервера находится в пакете `io.netty.example.echo` дистрибутива.

## 1.3 Написание Time-сервера
Протокол, который мы реализуем в этом разделе, — это протокол `TIME`. Он отличается от предыдущих примеров тем, что отправляет сообщение, содержащее 32-битное целое число, без получения каких-либо запросов и закрывает соединение после отправки сообщения. В этом примере вы узнаете, как сконструировать и отправить сообщение, а также закрыть соединение по завершении.

Поскольку мы собираемся игнорировать любые полученные данные, но отправлять сообщение сразу после установления соединения, на этот раз мы не можем использовать метод `channelRead()`. Вместо этого мы должны переопределить метод `channelActive()`. Ниже приведена реализация:
```java
package io.netty.example.time;

public class TimeServerHandler extends ChannelInboundHandlerAdapter {

    @Override
    public void channelActive(final ChannelHandlerContext ctx) { // (1)
        final ByteBuf time = ctx.alloc().buffer(4); // (2)
        time.writeInt((int) (System.currentTimeMillis() / 1000L + 2208988800L));

        final ChannelFuture f = ctx.writeAndFlush(time); // (3)
        f.addListener(new ChannelFutureListener() {
            @Override
            public void operationComplete(ChannelFuture future) {
                assert f == future;
                ctx.close();
            }
        }); // (4)
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}
```

1. Как объяснялось, метод `channelActive()` будет вызван, когда соединение установлено и готово к обмену данными. Давайте запишем 32-битное целое число, представляющее текущее время, в этом методе.
    
2. Чтобы отправить новое сообщение, нам нужно выделить новый буфер, который будет содержать сообщение. Мы собираемся записать 32-битное целое число, поэтому нам нужен `ByteBuf` емкостью не менее 4 байт. Получите текущий `ByteBufAllocator` через `ChannelHandlerContext.alloc()` и выделите новый буфер.
    
3. Как обычно, мы записываем сконструированное сообщение.
    
4. Но подождите, где же `flip`? Разве в NIO мы не вызывали `java.nio.ByteBuffer.flip()` перед отправкой сообщения? У `ByteBuf` нет такого метода, потому что у него есть два указателя: один для операций чтения, другой для операций записи. Индекс записи увеличивается, когда вы что-то записываете в `ByteBuf`, в то время как индекс чтения не меняется. Индекс чтения и индекс записи представляют собой соответственно начало и конец сообщения.  
    В отличие от этого, NIO-буфер не предоставляет простого способа определить, где начинается и заканчивается содержимое сообщения, без вызова метода `flip`. Вы попадете в беду, если забудете перевернуть буфер, потому что будет отправлено ничего или неверные данные. Такая ошибка не происходит в Netty, потому что у нас есть разные указатели для разных типов операций. Вы обнаружите, что это делает вашу жизнь намного проще, как только вы привыкнете к этому — жизнь без переворачивания!
    
    Еще один момент, который следует отметить: `ChannelHandlerContext.write()` (и `writeAndFlush()`) возвращает `ChannelFuture`. `ChannelFuture` представляет операцию ввода-вывода, которая еще не произошла. Это означает, что любая запрошенная операция могла быть еще не выполнена, поскольку все операции в Netty асинхронны. Например, следующий код может закрыть соединение еще до отправки сообщения:
```java
Channel ch = ...;
ch.writeAndFlush(message);
ch.close();
```

Поэтому вам нужно вызвать метод `close()` после завершения `ChannelFuture`, который был возвращен методом `write()`, и он уведомит своих слушателей о завершении операции записи. Имейте в виду, что `close()` также может не закрыть соединение немедленно и возвращает `ChannelFuture`.

Как же узнать, когда запрос на запись завершен? Это просто: добавить `ChannelFutureListener` к возвращенному `ChannelFuture`. Здесь мы создали новый анонимный `ChannelFutureListener`, который закрывает `Channel` после завершения операции.  
В качестве альтернативы вы можете упростить код, используя предопределенный слушатель:
```java
f.addListener(ChannelFutureListener.CLOSE);
```
Чтобы проверить, правильно ли работает наш Time-сервер, вы можете использовать UNIX-команду `rdate`:
```java
$ rdate -o <port> -p <host>
```

де `<port>` — это номер порта, который вы указали в методе `main()`, а `<host>` — обычно `localhost`.

### Написание Time-клиента
В отличие от серверов `DISCARD` и `ECHO`, для протокола `TIME` нам нужен клиент, потому что человек не может преобразовать 32-битные двоичные данные в дату в календаре. В этом разделе мы обсудим, как убедиться, что сервер работает правильно, и узнаем, как написать клиента с помощью Netty.

Самое большое и единственное различие между сервером и клиентом в Netty заключается в том, что используются разные реализации `Bootstrap` и `Channel`. Взгляните на следующий код:
```java
package io.netty.example.time;

public class TimeClient {
    public static void main(String[] args) throws Exception {
        String host = args[0];
        int port = Integer.parseInt(args[1]);
        EventLoopGroup workerGroup = new NioEventLoopGroup();

        try {
            Bootstrap b = new Bootstrap(); // (1)
            b.group(workerGroup); // (2)
            b.channel(NioSocketChannel.class); // (3)
            b.option(ChannelOption.SO_KEEPALIVE, true); // (4)
            b.handler(new ChannelInitializer<SocketChannel>() {
                @Override
                public void initChannel(SocketChannel ch) throws Exception {
                    ch.pipeline().addLast(new TimeClientHandler());
                }
            });

            // Запускаем клиента.
            ChannelFuture f = b.connect(host, port).sync(); // (5)

            // Ждем, пока соединение не будет закрыто.
            f.channel().closeFuture().sync();
        } finally {
            workerGroup.shutdownGracefully();
        }
    }
}
```

1. `Bootstrap` похож на `ServerBootstrap`, за исключением того, что он предназначен для не-серверных каналов, таких как клиентские или бессоединенные каналы.
    
2. Если вы укажете только одну `EventLoopGroup`, она будет использоваться и как босс-группа, и как рабочая группа. Однако босс-группа не используется на стороне клиента.
    
3. Вместо `NioServerSocketChannel` для создания клиентского канала используется `NioSocketChannel`.
    
4. Обратите внимание, что здесь мы не используем `childOption()`, в отличие от `ServerBootstrap`, потому что у клиентского `SocketChannel` нет родителя.
    
5. Мы должны вызвать метод `connect()` вместо `bind()`.
    

Как вы видите, это не сильно отличается от кода на стороне сервера. А что насчет реализации `ChannelHandler`? Он должен получить 32-битное целое число от сервера, преобразовать его в удобочитаемый формат, вывести преобразованное время и закрыть соединение:
```java
package io.netty.example.time;

import java.util.Date;
import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;

public class TimeClientHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        ByteBuf m = (ByteBuf) msg; // (1)
        try {
            long currentTimeMillis = (m.readUnsignedInt() - 2208988800L) * 1000L;
            System.out.println(new Date(currentTimeMillis));
            ctx.close();
        } finally {
            m.release();
        }
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}
```

1. При использовании TCP/IP Netty считывает данные, отправленные удаленным узлом, в `ByteBuf`.
    

Выглядит очень просто и не отличается от примера на стороне сервера. Однако иногда этот обработчик отказывается работать, вызывая `IndexOutOfBoundsException`. Мы обсудим, почему это происходит, в следующем разделе.

## 1.4 Работа с транспортом, основанным на потоках

### Один небольшой недостаток сокетного буфера
В транспорте, основанном на потоках, таком как TCP/IP, полученные данные сохраняются в буфере приема сокета. К сожалению, буфер транспорта, основанного на потоках, представляет собой не очередь пакетов, а очередь байтов. Это означает, что даже если вы отправили два сообщения как два независимых пакета, операционная система не будет рассматривать их как два сообщения, а просто как набор байтов. Следовательно, нет гарантии, что то, что вы читаете, является именно тем, что написал ваш удаленный peer. Например, предположим, что стек TCP/IP операционной системы получил три пакета:

`+-----+-----+-----+`  
`| ABC | DEF | GHI |`  
`+-----+-----+-----+`

Из-за этого общего свойства протоколов, основанных на потоках, велика вероятность прочитать их в вашем приложении в следующем фрагментированном виде:

`+----+-------+---+`  
`| AB | CDEFG | HI |`  
`+----+-------+---+`

Поэтому принимающая сторона, будь то сервер или клиент, должна дефрагментировать полученные данные в одно или несколько осмысленных сообщений (кадров), которые могут быть легко поняты логикой приложения. В случае приведенного выше примера полученные данные должны быть оформлены в кадры следующим образом:

`+-----+-----+-----+`  
`| ABC | DEF | GHI |`  
`+-----+-----+-----+`

### Первое решение
Теперь вернемся к примеру с `TIME` клиентом. У нас здесь та же проблема. 32-битное целое число — это очень маленький объем данных, и маловероятно, что оно часто будет фрагментироваться. Однако проблема в том, что оно может быть фрагментировано, и вероятность фрагментации будет увеличиваться по мере роста трафика.

Простое решение — создать внутренний накопительный буфер и ждать, пока все 4 байта не будут получены во внутренний буфер. Ниже приведена измененная реализация `TimeClientHandler`, которая решает проблему:
```java
package io.netty.example.time;

public class TimeClientHandler extends ChannelInboundHandlerAdapter {
    private ByteBuf buf;

    @Override
    public void handlerAdded(ChannelHandlerContext ctx) {
        buf = ctx.alloc().buffer(4); // (1)
    }

    @Override
    public void handlerRemoved(ChannelHandlerContext ctx) {
        buf.release(); // (1)
        buf = null;
    }

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        ByteBuf m = (ByteBuf) msg;
        buf.writeBytes(m); // (2)
        m.release();

        if (buf.readableBytes() >= 4) { // (3)
            long currentTimeMillis = (buf.readUnsignedInt() - 2208988800L) * 1000L;
            System.out.println(new Date(currentTimeMillis));
            ctx.close();
        }
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}
```

1. `ChannelHandler` имеет два метода-слушателя жизненного цикла: `handlerAdded()` и `handlerRemoved()`. Вы можете выполнять любые задачи (де)инициализации, если они не блокируются надолго.
    
2. Во-первых, все полученные данные должны быть накоплены в `buf`.
    
3. Затем обработчик должен проверить, достаточно ли данных в `buf` (в данном примере 4 байта), и перейти к реальной бизнес-логике. В противном случае Netty снова вызовет метод `channelRead()`, когда поступят дополнительные данные, и в итоге все 4 байта будут накоплены.
    

### Второе решение
Хотя первое решение решило проблему с `TIME` клиентом, измененный обработчик выглядит не очень чистым. Представьте себе более сложный протокол, состоящий из нескольких полей, например, поле переменной длины. Ваша реализация `ChannelInboundHandler` очень быстро станет неподдерживаемой.

Как вы могли заметить, вы можете добавить более одного `ChannelHandler` в `ChannelPipeline`, и, следовательно, вы можете разделить один монолитный `ChannelHandler` на несколько модульных, чтобы уменьшить сложность вашего приложения. Например, вы могли бы разделить `TimeClientHandler` на два обработчика:

- `TimeDecoder`, который занимается проблемой фрагментации, и
    
- первоначальную простую версию `TimeClientHandler`.
    

К счастью, Netty предоставляет расширяемый класс, который помогает написать первый "из коробки":
```java
package io.netty.example.time;

public class TimeDecoder extends ByteToMessageDecoder { // (1)
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) { // (2)
        if (in.readableBytes() < 4) {
            return; // (3)
        }

        out.add(in.readBytes(4)); // (4)
    }
}
```

1. `ByteToMessageDecoder` — это реализация `ChannelInboundHandler`, которая упрощает работу с проблемой фрагментации.
    
2. `ByteToMessageDecoder` вызывает метод `decode()` с внутренним накопительным буфером всякий раз, когда получены новые данные.
    
3. `decode()` может решить ничего не добавлять в `out`, если в накопительном буфере недостаточно данных. `ByteToMessageDecoder` снова вызовет `decode()`, когда будет получено больше данных.
    
4. Если `decode()` добавляет объект в `out`, это означает, что декодер успешно декодировал сообщение. `ByteToMessageDecoder` отбросит прочитанную часть накопительного буфера. Помните, что вам не нужно декодировать несколько сообщений за один вызов. `ByteToMessageDecoder` будет продолжать вызывать метод `decode()` до тех пор, пока он ничего не добавит в `out`.
    

Теперь, когда у нас есть еще один обработчик для вставки в `ChannelPipeline`, мы должны изменить реализацию `ChannelInitializer` в `TimeClient`:
```java
b.handler(new ChannelInitializer<SocketChannel>() {
    @Override
    public void initChannel(SocketChannel ch) throws Exception {
        ch.pipeline().addLast(new TimeDecoder(), new TimeClientHandler());
    }
});
```

Если вы любите приключения, вы можете попробовать `ReplayingDecoder`, который упрощает декодер еще больше. Однако вам нужно будет обратиться к справочнику по API для получения дополнительной информации.
```java
public class TimeDecoder extends ReplayingDecoder<Void> {
    @Override
    protected void decode(
            ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        out.add(in.readBytes(4));
    }
}
```

Кроме того, Netty предоставляет готовые декодеры, которые позволяют очень легко реализовать большинство протоколов и помогают избежать создания монолитной, плохо поддерживаемой реализации обработчика. Пожалуйста, обратитесь к следующим пакетам для более подробных примеров:

- `io.netty.example.factorial` для бинарного протокола и
    
- `io.netty.example.telnet` для текстового протокола на основе строк.

### Использование POJO вместо `ByteBuf`
Во всех примерах, которые мы рассмотрели до сих пор, в качестве основной структуры данных сообщения протокола использовался `ByteBuf`. В этом разделе мы улучшим пример клиента и сервера протокола `TIME`, используя POJO вместо `ByteBuf`.

Преимущество использования POJO в ваших `ChannelHandler` очевидно: ваш обработчик становится более поддерживаемым и пригодным для повторного использования за счет отделения кода, извлекающего информацию из `ByteBuf`, от обработчика. В примерах клиента и сервера `TIME` мы читаем только одно 32-битное целое число, и использование напрямую `ByteBuf` не является большой проблемой. Однако вы поймете, что необходимо сделать такое разделение по мере реализации реального протокола.

Во-первых, давайте определим новый тип под названием `UnixTime`.
```java
package io.netty.example.time;

import java.util.Date;

public class UnixTime {

    private final long value;

    public UnixTime() {
        this(System.currentTimeMillis() / 1000L + 2208988800L);
    }

    public UnixTime(long value) {
        this.value = value;
    }

    public long value() {
        return value;
    }

    @Override
    public String toString() {
        return new Date((value() - 2208988800L) * 1000L).toString();
    }
}
```
Теперь мы можем изменить `TimeDecoder` так, чтобы он создавал `UnixTime` вместо `ByteBuf`.
```java
@Override
protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
    if (in.readableBytes() < 4) {
        return;
    }

    out.add(new UnixTime(in.readUnsignedInt()));
}
```

С обновленным декодером `TimeClientHandler` больше не использует `ByteBuf`:
```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    UnixTime m = (UnixTime) msg;
    System.out.println(m);
    ctx.close();
}
```

Гораздо проще и элегантнее, правда? Тот же метод можно применить и на стороне сервера. 

Давайте сначала обновим `TimeServerHandler` на этот раз:
```java 
@Override
public void channelActive(ChannelHandlerContext ctx) {
    ChannelFuture f = ctx.writeAndFlush(new UnixTime());
    f.addListener(ChannelFutureListener.CLOSE);
}
```

Теперь единственным недостающим элементом является кодировщик — реализация `ChannelOutboundHandler`, которая преобразует `UnixTime` обратно в `ByteBuf`. Это намного проще, чем писать декодер, потому что при кодировании сообщения не нужно иметь дело с фрагментацией и сборкой пакетов.
```java
package io.netty.example.time;

public class TimeEncoder extends ChannelOutboundHandlerAdapter {
    @Override
    public void write(ChannelHandlerContext ctx, Object msg, ChannelPromise promise) {
        UnixTime m = (UnixTime) msg;
        ByteBuf encoded = ctx.alloc().buffer(4);
        encoded.writeInt((int)m.value());
        ctx.write(encoded, promise); // (1)
    }
}
```

1. В этой одной строке есть несколько важных моментов.  
    Во-первых, мы передаем исходный `ChannelPromise` как есть, чтобы Netty отметил его как успех или неудачу, когда закодированные данные действительно будут записаны в канал.  
    Во-вторых, мы не вызывали `ctx.flush()`. Существует отдельный метод обработчика `void flush(ChannelHandlerContext ctx)`, предназначенный для переопределения операции `flush()`.
    

Чтобы еще больше упростить задачу, вы можете использовать `MessageToByteEncoder`:
```java
public class TimeEncoder extends MessageToByteEncoder<UnixTime> {
    @Override
    protected void encode(ChannelHandlerContext ctx, UnixTime msg, ByteBuf out) {
        out.writeInt((int)msg.value());
    }
}
```

Последняя задача — вставить `TimeEncoder` в `ChannelPipeline` на стороне сервера перед `TimeServerHandler`, и это оставляется в качестве тривиального упражнения.
