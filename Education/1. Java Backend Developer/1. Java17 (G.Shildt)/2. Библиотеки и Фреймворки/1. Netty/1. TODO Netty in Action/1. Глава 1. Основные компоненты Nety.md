## 1.3 Основные компоненты Netty (Netty's core components)
В этом разделе мы обсудим основные строительные блоки Netty:
- **Каналы (Channels)**
    
- **Обратные вызовы (Callbacks)**
    
- **Фьючерсы (Futures)**
    
- **События и обработчики (Events and handlers)**

Эти строительные блоки представляют различные типы конструкций: ресурсы, логику и уведомления. Ваши приложения будут использовать их для доступа к сети и данным, которые через неё проходят.

Для каждого компонента мы дадим базовое определение и, где уместно, простой пример кода, иллюстрирующий его использование.

---
### 1.3.1 Каналы (Channels)
> [!NOTE] Канал - открытое соединение
> **Канал (Channel)** — это базовая конструкция Java NIO. Он представляет **открытое соединение** с сущностью, такой как *аппаратное устройство, файл, сетевой сокет или компонент программы*, способный выполнять одну или несколько различных операций ввода-вывода, например чтение или запись.

А пока думайте о `Channel` как о транспортном средстве для входящих (inbound) и исходящих (outbound) данных. Как таковой, он может быть открыт или закрыт, подключен или отключен.

---
### 1.3.2 Обратные вызовы (Callbacks)

> [!NOTE] Обратный вызов - ссылка на метод, переданная другому для вызова в нужный момент
> **Обратный вызов (callback)** — это ***просто метод, ссылка на который была передана другому методу***. 
> 
> Это *позволяет последнему вызвать первый в подходящий момент времени.* 

Обратные вызовы используются в широком круге задач программирования и представляют собой *один из самых распространённых способов уведомить заинтересованную сторону о том, что операция завершилась.*

**Netty использует обратные вызовы** внутри себя ==при обработке событий==; 

> [!IMPORTANT] Обрабатывают события - РЕАЛИЗАЦИИ интерфейса `ChannelHanlder`
> Когда обратный вызов срабатывает, событие может быть обработано реализацией интерфейса `ChannelHandler`.

В следующем листинге показан пример: когда установлено новое соединение, вызывается callback-метод `channelActive()` обработчика `ChannelHandler`, который выводит сообщение.

**Листинг 1.2. ChannelHandler, срабатывающий от обратного вызова**
```java
public class ConnectHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelActive(ChannelHandlerContext ctx) throws Exception {    // (1)
        System.out.println(
	        "Клиент " + ctx.channel().remoteAddress() + " подключен");
    }
}
```

`(1)` Метод `channelActive()` вызывается, когда установлено новое соединение.

### 1.3.3 Фьючерсы (Futures)

> [!NOTE] Собственная реализация `Future`, для хранения результата асинхронной операции
> **Фьючерс (Future)** предоставляет другой способ уведомить приложение о завершении операции. 
> 
> Этот объект действует как _заполнитель (placeholder)_ для результата асинхронной операции; он будет завершён в какой-то момент в будущем и предоставит доступ к результату.

*JDK поставляется с интерфейсом* `java.util.concurrent.Future`, *но его реализации позволяют вам **только вручную проверять***, завершилась ли операция, или блокироваться до её завершения. 

**Это довольно громоздко**, ==поэтому Netty предоставляет свою собственную реализацию== — `ChannelFuture`, для использования при выполнении асинхронных операций.


`ChannelFuture` предоставляет дополнительные методы, которые **позволяют нам зарегистрировать один или несколько экземпляров** `ChannelFutureListener`. 
**Callback-метод слушателя** `operationComplete()` **вызывается, когда операция завершается**. Затем слушатель может определить, завершилась ли операция успешно или с ошибкой. В последнем случае мы можем получить возникшее исключение (`Throwable`). 

> Короче говоря, ==механизм уведомлений, предоставляемый== `ChannelFutureListener`, **устраняет необходимость вручную проверять завершение операции.**


> [!WARNING] Outbound операции в Netty возвращает `ChannelFuture`
> Каждая исходящая (outbound) операция ввода-вывода Netty возвращает `ChannelFuture`; то есть ни одна из них не блокируется. Как мы уже говорили ранее, Netty является асинхронной и событийно-ориентированной "с самого низа".

**Листинг 1.3** показывает, что `ChannelFuture` возвращается как часть операции ввода-вывода. Здесь `connect()` вернётся напрямую, без блокировки, а вызов завершится в фоновом режиме. Когда это произойдёт, может зависеть от нескольких факторов, но эта проблема абстрагирована от кода. 
Поскольку поток не блокируется в ожидании завершения операции, он может делать другую работу в это время, используя ресурсы более эффективно.

**Листинг 1.3. Асинхронное подключение (connect)**
```java
Channel channel = ...;
// Не блокируется
ChannelFuture future = channel.connect(                          // (1)
    new InetSocketAddress("192.168.0.1", 25));
```

`(1)` Асинхронное подключение к удалённому узлу.

---
**Листинг 1.4** показывает, как использовать `ChannelFutureListener`. Сначала вы подключаетесь к удалённому узлу. 

Затем вы регистрируете новый `ChannelFutureListener` с `ChannelFuture`, возвращённым вызовом `connect()`. 

Когда слушатель будет уведомлён об установлении соединения, вы проверяете статус **(b)**. Если операция успешна, вы записываете данные в `Channel`. В противном случае вы получаете `Throwable` из `ChannelFuture`.

```java
Channel channel = ...;
// Не блокируется
ChannelFuture future = channel.connect(                          // (1)
    new InetSocketAddress("192.168.0.1", 25));
    
future.addListener(new ChannelFutureListener() {                 // (2)
    @Override
    public void operationComplete(ChannelFuture future) {
    
        if (future.isSuccess()) {                                // (3)
            ByteBuf buffer = Unpooled.copiedBuffer(              // (4)
                "Hello", Charset.defaultCharset());
                
            ChannelFuture wf = future.channel()
                .writeAndFlush(buffer);                          // (5)
            // ... остальная логика
        } else {
            Throwable cause = future.cause();                    // (6)
            cause.printStackTrace();
        }
    }
});
```

**1)** Асинхронное подключение к удалённому узлу.  
**(2)** Регистрация `ChannelFutureListener` для уведомления о завершении операции.  
**(3)** Проверка статуса операции.  
**(4)** Если операция успешна, создаётся `ByteBuf` для хранения данных.  
**(5)** Асинхронная отправка данных удалённому узлу. Возвращает `ChannelFuture`.  
**(6)** Если произошла ошибка, получаем доступ к `Throwable`, описывающему причину.

Обратите внимание, что обработка ошибок полностью зависит от вас, с учётом любых ограничений, накладываемых конкретной ошибкой. Например, в случае сбоя подключения вы можете попытаться переподключиться или установить соединение с другим удалённым узлом.

Если вы думаете, что `ChannelFutureListener` — это более сложная версия обратного вызова (callback), вы правы. Фактически, обратные вызовы и фьючерсы являются взаимодополняющими механизмами; в сочетании они составляют один из ключевых строительных блоков самого Netty.

### 1.3.4 События и обработчики (Events and handlers)

*Netty использует различные* **события (events)** **для уведомления нас об изменении состояния или статуса операций.** 

Это позволяет нам запускать соответствующее действие на основе произошедшего события. 

Такие действия могут включать:
- Логирование
    
- Преобразование данных
    
- Управление потоком (flow-control)
    
- Бизнес-логику приложения
    

Netty — это сетевой фреймворк, поэтому события классифицируются по их отношению к **входящему (inbound)** или **исходящему (outbound)** потоку данных.

**События, которые могут быть вызваны входящими данными или связанным с ними изменением состояния, включают:**
- Активные или неактивные соединения
    
- Чтение данных
    
- Пользовательские события
    
- Ошибки
    

**Исходящее событие** является результатом операции, которая вызовет действие в будущем, например:
- Открытие или закрытие соединения с удалённым узлом
    
- Запись или сброс (flushing) данных в сокет

Каждое событие может быть направлено в реализованный пользователем метод класса-обработчика. Это хороший пример того, как событийно-ориентированная парадигма напрямую транслируется в строительные блоки приложения.

Мы ещё многое скажем о `ChannelHandler` в своё время, но сейчас вы можете думать о каждом экземпляре обработчика как о своего рода обратном вызове, который выполняется в ответ на конкретное событие.

Netty предоставляет обширный набор предопределённых обработчиков, которые можно использовать "из коробки", включая обработчики для таких протоколов, как HTTP и SSL/TLS. Внутри себя `ChannelHandler` сами используют события и фьючерсы, являясь потребителями тех же абстракций, которые будут использовать и ваши приложения.

### 1.3.5 Собираем всё вместе (Putting it all together)

В этой главе вы познакомились с подходом Netty к высокопроизводительным сетям и с некоторыми основными компонентами его реализации. Давайте соберём общую картину того, что мы обсудили.

**ФЬЮЧЕРСЫ, ОБРАТНЫЕ ВЫЗОВЫ И ОБРАБОТЧИКИ (FUTURES, CALLBACKS, AND HANDLERS)**

Асинхронная модель программирования Netty построена на концепциях фьючерсов и обратных вызовов, причём диспетчеризация событий методам-обработчикам происходит на более глубоком уровне. 

В совокупности эти элементы обеспечивают среду обработки, которая позволяет логике вашего приложения развиваться независимо от каких-либо проблем, связанных с сетевыми операциями. Это ключевая цель архитектурного подхода Netty.

Перехват операций и преобразование входящих или исходящих данных "на лету" требует только предоставить обратные вызовы или использовать фьючерсы, возвращаемые операциями. *Это упрощает и делает эффективным объединение операций в цепочки*, а также способствует написанию пригодного для повторного использования, обобщённого кода.

---
**СЕЛЕКТОРЫ, СОБЫТИЯ И ЦИКЛЫ СОБЫТИЙ (SELECTORS, EVENTS, AND EVENT LOOPS)**

Netty абстрагирует `Selector` от приложения, генерируя события, устраняя весь код ручной диспетчеризации, который в противном случае потребовался бы. 

==Под капотом КАЖДОМУ КАНАЛУ (Channel) назначается== `EventLoop` для обработки всех событий, включая:
- Регистрацию интересующих событий
    
- Диспетчеризацию событий в `ChannelHandler`
    
- Планирование дальнейших действий


Сам `EventLoop` управляется только одним потоком, который обрабатывает все события ввода-вывода для одного канала и не меняется в течение времени жизни `EventLoop`. 

Эта простая и мощная конструкция **УСТРАНЯЕТ любые ПРОБЛЕМЫ с СИНХРОНИЗАЦИЕЙ в ваших** `ChannelHandler`, поэтому вы можете сосредоточиться на предоставлении правильной логики для выполнения, когда появятся интересующие данные для обработки. Как мы увидим, когда будем детально исследовать модель потоков Netty, API является простым и компактным.
