Сеть видит данные просто как последовательность сырых байтов. Однако наши приложения структурируют эти байты так, чтобы они имели смысл как информация. Преобразование данных в сетевой байтовый поток и из него — одна из самых распространённых задач программирования. Вам может потребоваться, например, работать со стандартным форматом или протоколом, таким как FTP или Telnet, реализовать проприетарный бинарный протокол, определённый третьей стороной, или расширить устаревший формат сообщений, созданный вашей собственной организацией.

Компоненты, которые обрабатывают преобразование данных приложения в сетевой формат и обратно, называются соответственно **кодировщиками (encoders)** и **декодировщиками (decoders)**, ==а единый компонент, обладающий обеими способностями, называется ==**кодеком (codec)**. 

**Netty предоставляет ряд инструментов для создания всех этих компонентов:** 
- от *готовых классов, специализированных для хорошо известных протоколов, таких как HTTP и base64,* 
- до универсальных кодеков преобразования сообщений, которые вы можете настроить под свои конкретные потребности.

Глава 10 представляет введение в кодировщики и декодировщики. Вы узнаете о базовых классах кодеков Netty, изучая некоторые типичные случаи использования. По мере того как вы будете узнавать, как эти классы вписываются в общий фреймворк, вы обнаружите, что они построены на тех же API, которые вы уже изучили, поэтому вы сможете использовать их сразу.

В главе 11 вы изучите некоторые кодировщики и декодировщики, которые Netty предоставляет для обработки более специализированных сценариев. 

Раздел о WebSocket представляет особый интерес и подготовит вас к детальному обсуждению продвинутых сетевых протоколов в части 3.

# 10. Фреймворк кодеков (The codec framework)

**В этой главе рассматриваются:**
- Обзор декодировщиков, кодировщиков и кодеков
    
- Классы кодеков Netty
    

Подобно тому, как многие стандартные архитектурные паттерны поддерживаются специализированными фреймворками, общие паттерны обработки данных часто являются хорошими кандидатами для целевых реализаций, что может сэкономить разработчикам значительное время и усилия.

Это, безусловно, относится к теме этой главы: к***одированию и декодированию, или преобразованию данных из одного протокол-специфичного формата в другой***. 

Эти задачи обрабатываются компонентами, обычно называемыми **кодеками**. **Netty предоставляет компоненты, которые упрощают создание ПОЛЬЗОВАТЕЛЬСКИХ кодеков** для широкого круга протоколов. Например, если вы создаёте почтовый сервер на основе Netty, вы обнаружите, что поддержка кодеков Netty неоценима для реализации протоколов POP3, IMAP и SMTP.

---
## 10.1 Что такое кодек? (What is a codec?)

Каждое сетевое приложение должно определять, **как необработанные байты**, *передаваемые между узлами*, **должны быть распаршены и преобразованы в формат данных целевой программы и обратно.** 


> [!NOTE] Кодеки добавляются в Pipeline
> Кодеки — это обычные `ChannelHandler`. 
> 
> Их добавляют в `ChannelPipeline` так же, как и любые другие обработчики.
> ```java
> // Сервер
> .childHandler(new ChannelInitializer<SocketChannel>() {
> 	@Override
> 	protected void initChannel(SocketChannel ch) {
> 		ChannelPipeline p = ch.pipeline();
> 		
> 		// Декодировщик: байты → объект
> 		p.addLast(new ByteToMessageDecoder() { ... });
> 		
> 		// Кодировщик: объект → байты
> 		p.addLast(new MessageToByteEncoder<MyMessage>() { ... });
> 		
> 		// Ваш бизнес-обработчик
> 		p.addLast(new MyBusinessHandler());
> 	}}
> );
>```


Эта логика преобразования обрабатывается **кодеком**, который состоит из **кодировщика (encoder)** и **декодировщика (decoder)**, каждый из которых преобразует поток байтов из одного формата в другой. Что их отличает?

Думайте о **сообщении** как о структурированной последовательности байтов, имеющей значение для конкретного приложения — его данные. 
- **Кодировщик** преобразует это с**ообщение в формат, подходящий для передачи** (скорее всего, в байтовый поток); 
- соответствующий **декодировщик** преобразует **сетевой поток обратно в формат сообщения программы.** 
Таким образом, кодировщик работает с исходящими данными, а декодировщик обрабатывает входящие данные.

Имея эту информацию, давайте рассмотрим классы, которые Netty предоставляет для реализации обоих типов компонентов.

## 10.2 Декодировщики (Decoders)

В этом разделе мы рассмотрим классы декодировщиков Netty и представим конкретные примеры того, когда и как вы можете их использовать. 

Эти классы охватывают ==два различных варианта использования==:
- **Декодирование БАЙТОВ В СООБЩЕНИЯ** 
	  — `ByteToMessageDecoder` и `ReplayingDecoder`
    
- **Декодирование ОДНОГО ТИПА СООБЩЕНИЯ В ДРУГОЙ** 
	  — `MessageToMessageDecoder`
    

Поскольку декодировщики отвечают за преобразование входящих данных из одного формата в другой, вас не удивит, что декодировщики Netty реализуют `ChannelInboundHandler`.

**Когда бы вы использовали декодировщик?** Просто: всякий раз, когда вам нужно преобразовать входящие данные для следующего `ChannelInboundHandler` в `ChannelPipeline`. Более того, благодаря архитектуре `ChannelPipeline`, **вы можете объединять несколько декодировщиков в цепочку для реализации** сколь угодно сложной логики преобразования — это отличный пример того, как Netty поддерживает модульность и повторное использование кода.

---
### 10.2.1 Абстрактный класс ByteToMessageDecoder

Декодирование ==из байтов в сообщения== (или в другую последовательность байтов) является настолько распространённой задачей, что **Netty предоставляет для неё АБСТРАКТНЫЙ базовый класс:** **`ByteToMessageDecoder`**. 

Поскольку вы не можете знать, отправит ли удалённый узел полное сообщение сразу, ==этот класс БУФЕРИЗИРУЕТ входящие данные, пока они не будут готовы к обработке==. Таблица 10.1 объясняет два его наиболее важных метода.

**Таблица 10.1. API ByteToMessageDecoder**

| Метод                                                                 | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out)`     | Это ==единственный абстрактный метод, который вы должны реализовать==. <br><br>`decode()` вызывается с `ByteBuf`, **содержащим входящие данные, и `List`, в который добавляются декодированные сообщения.** <br><br>Этот вызов повторяется до тех пор, пока не будет определено, что в `List` не добавлено новых элементов или в `ByteBuf` не осталось читаемых байтов. Затем, если `List` не пуст, его содержимое передаётся следующему обработчику в конвейере. |
| `decodeLast(ChannelHandlerContext ctx, ByteBuf in, List<Object> out)` | **Реализация по умолчанию**, предоставляемая Netty, просто вызывает `decode()`. <br><br>Этот метод вызывается один раз, когда канал становится неактивным (`inactive`). Переопределите метод, чтобы обеспечить специальную обработку.                                                                                                                                                                                                                             |

Для примера использования этого класса предположим, что вы получаете байтовый поток, содержащий простые целые числа (`int`), каждое из которых должно быть обработано отдельно. В этом случае вы будете читать каждое целое число из входящего `ByteBuf` и передавать его следующему `ChannelInboundHandler` в конвейере.

Для декодирования байтового потока вы расширите `ByteToMessageDecoder`. (Обратите внимание, что примитивный `int` будет автоматически упакован (autoboxed) в `Integer` при добавлении в `List`.)

Дизайн проиллюстрирован на рисунке 10.1.

![[Pasted image 20260612092946.png]]
**Рисунок 10.1. ToIntegerDecoder**


Каждый раз из входящего `ByteBuf` читаются четыре байта, декодируются в `int` и добавляются в `List`. Когда в `List` больше нельзя добавить элементов, его содержимое будет отправлено следующему `ChannelInboundHandler`.

Этот листинг показывает код для `ToIntegerDecoder`.

**Листинг 10.1. Класс ToIntegerDecoder расширяет ByteToMessageDecoder**
```java
// (1) Расширяет ByteToMessageDecoder для декодирования байтов в определённый формат
public class ToIntegerDecoder extends ByteToMessageDecoder {          
    @Override
    public void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
	    // (2) Проверяет, есть ли хотя бы 4 читаемых байта (длина int)
        if (in.readableBytes() >= 4) {  
        
			// (3) Читает int из входящего ByteBuf и добавляет его в список декодированных сообщений
            out.add(in.readInt());                                     
        }
    }
}
```

Хотя `ByteToMessageDecoder` упрощает реализацию этого паттерна, вам **может показаться немного УТОМИТЕЛЬНЫМ ПРОВЕРЯТЬ, что во входящем `ByteBuf` ДОСТАТОЧНО ДАННЫХ ДЛЯ ВЫЗОВА** `readInt()`. 
> В следующем разделе **мы обсудим `ReplayingDecoder` — специальный декодер, который устраняет этот шаг** ценой небольшого объёма накладных расходов.

> **Подсчёт ссылок в кодеках**
> 
> Как мы упоминали в главах 5 и 6, подсчёт ссылок требует особого внимания. В случае кодировщиков и декодировщиков процедура довольно проста: ***как только сообщение закодировано или декодировано***, **оно будет автоматически освобождено вызовом** `ReferenceCountUtil.release(message)`. 
> 
> Если вам ==нужно сохранить ссылку== для последующего использования, вы ==можете вызвать ==`ReferenceCountUtil.retain(message)`. 
> Это увеличивает счётчик ссылок, предотвращая освобождение сообщения.

### 10.2.2 Абстрактный класс ReplayingDecoder

`ReplayingDecoder` расширяет `ByteToMessageDecoder` и и**збавляет нас от необходимости вызывать** `readableBytes()` (как в листинге 10.1). 

Он достигает этого, оборачивая входящий `ByteBuf` пользовательской реализацией `ByteBuf` — `ReplayingDecoderBuffer`, которая выполняет проверку внутренне.

Полное объявление этого класса:
```java
public abstract class ReplayingDecoder<S> extends ByteToMessageDecoder
```

Параметр `S` **указывает тип, используемый для управления состоянием,** ***где `Void` означает, что управление состоянием не выполняется.*** 
Следующий листинг показывает повторную реализацию `ToIntegerDecoder` на основе `ReplayingDecoder`.

**Листинг 10.2. Класс ToIntegerDecoder2 расширяет ReplayingDecoder**
```java
// (1) Расширяет ReplayingDecoder<Void> для декодирования байтов в сообщения
public class ToIntegerDecoder2 extends ReplayingDecoder<Void> {        
    @Override
    public void decode(
	    ChannelHandlerContext ctx, 
	    ByteBuf in, 
	    List<Object> out
	) throws Exception {
	    // (3) Читает int из входящего ByteBuf и добавляет его в список декодированных сообщений
        out.add(in.readInt());                                          
    }                                                                   // (2) Входящий ByteBuf является ReplayingDecoderBuffer
}
```

Как и прежде, целые числа, извлечённые из `ByteBuf`, добавляются в `List`. Если доступно недостаточно байтов, эта реализация `readInt()` выбрасывает `Error`, который будет перехвачен и обработан в базовом классе. Метод `decode()` будет вызван снова, когда для чтения будет готово больше данных (см. описание `decode()` в таблице 10.1).

**Пожалуйста, обратите внимание на следующие аспекты `ReplayingDecoderBuffer`:**

- Поддерживаются не все операции `ByteBuf`. Если вызывается неподдерживаемый метод, будет выброшено `UnsupportedOperationException`.
    
- `ReplayingDecoder` немного медленнее, чем `ByteToMessageDecoder`.
    

Если сравнить листинги 10.1 и 10.2, очевидно, что последний проще. Сам пример очень прост, поэтому имейте в виду, что в реальной, более сложной ситуации разница между использованием одного или другого базового класса может быть значительной. 
> **Вот простое правило:** 
> используйте `ByteToMessageDecoder`, если он не вносит излишней сложности; в противном случае используйте `ReplayingDecoder`.

### 10.2.3 Абстрактный класс MessageToMessageDecoder

В этом разделе мы объясним, как преобразовывать между форматами сообщений (например, из одного типа POJO в другой) с помощью абстрактного базового класса:
```java
public abstract class MessageToMessageDecoder<I> extends ChannelInboundHandlerAdapter
```

Параметр `I` указывает тип аргумента `msg` для `decode()`, который является единственным методом, который вы должны реализовать. Таблица 10.2 показывает детали этого метода.

**Таблица 10.2. API MessageToMessageDecoder**

| Метод                                                        | Описание                                                                                                                                                                        |

| `decode(ChannelHandlerContext ctx, I msg, List<Object> out)` | Вызывается для каждого входящего сообщения для декодирования в другой формат. <br><br>Декодированные сообщения затем передаются следующему `ChannelInboundHandler` в конвейере. |

В этом примере **мы напишем декодер** `IntegerToStringDecoder`, который расширяет `MessageToMessageDecoder<Integer>`. Его метод `decode()` будет преобразовывать аргумент `Integer` в его строковое представление (`String`) и будет иметь следующую сигнатуру:
```java
public void decode(ChannelHandlerContext ctx, Integer msg, List<Object> out) throws Exception
```

Как и прежде, декодированная `String` будет добавлена в исходящий `List` и передана следующему `ChannelInboundHandler`.

### 10.2.4 Класс TooLongFrameException

Поскольку Netty является асинхронным фреймворком, **вам нужно буферизировать байты в памяти, пока вы не сможете их декодировать.** 

Следовательно, ***вы не должны позволять вашему декодеру буферизировать достаточно данных, чтобы исчерпать доступную память.*** 

Для решения этой общей проблемы Netty предоставляет `TooLongFrameException`, ==который предназначен для выбрасывания декодерами, если кадр превышает указанный лимит размера.==

Чтобы избежать этого, **вы можете установить порог максимального количества байтов,** превышение которого вызовет выбрасывание `TooLongFrameException` (и будет перехвачено `ChannelHandler.exceptionCaught()`). 

Затем пользователь декодера должен решить, как обработать исключение. Некоторые протоколы, такие как HTTP, могут позволить вам вернуть специальный ответ. В других случаях единственным вариантом может быть закрытие соединения.

В листинге 10.4 показано, как `ByteToMessageDecoder` может использовать `TooLongFrameException` для уведомления других `ChannelHandler` в `ChannelPipeline` о возникновении превышения размера кадра. Обратите внимание, что такая защита особенно важна, если вы работаете с протоколом, имеющим переменный размер кадра.

**Листинг 10.4. TooLongFrameException**
```java
public class SafeByteToMessageDecoder extends ByteToMessageDecoder {      // (1) Расширяет ByteToMessageDecoder для декодирования байтов в сообщения
    private static final int MAX_FRAME_SIZE = 1024;

    @Override
    public void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
        int readable = in.readableBytes();
        
        // (2) Проверяет, не превышает ли буфер MAX_FRAME_SIZE байтов
        if (readable > MAX_FRAME_SIZE) { 
	        // (3) Пропускает все читаемые байты
            in.skipBytes(readable);    
            
            // выбрасывает TooLongFrameException и уведомляет ChannelHandler                                   
            throw new TooLongFrameException("Frame too big!");            
        }
        // do something
        // ...
    }
}
```

## 10.3 Кодировщики (Encoders)

Возвращаясь к нашему предыдущему определению, **кодировщик** реализует `ChannelOutboundHandler` и **преобразует исходящие данные из одного формата в другой** — это обратная функция по отношению к декодировщикам, которые мы только что изучили. 

Netty предоставляет набор классов, которые помогут вам писать кодировщики со следующими возможностями:

- Кодирование **из сообщений в байты**
    
- Кодирование **из сообщений в сообщения**
    

Мы начнём наше рассмотрение этих классов с абстрактного базового класса `MessageToByteEncoder`.

### 10.3.1 Абстрактный класс MessageToByteEncoder

Ранее мы рассмотрели, как преобразовывать байты в сообщения с помощью `ByteToMessageDecoder`. Теперь мы сделаем обратное с помощью `MessageToByteEncoder`. Таблица 10.3 показывает API.

**Таблица 10.3. API MessageToByteEncoder**

| Метод                                                   | Описание                                                                                                                                                                                                                                                                |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `encode(ChannelHandlerContext ctx, I msg, ByteBuf out)` | Метод `encode` — единственный абстрактный метод, который вам нужно реализовать. <br><br>Он вызывается с исходящим сообщением (типа `I`), которое этот класс закодирует в `ByteBuf`. <br><br>Затем `ByteBuf` передаётся следующему `ChannelOutboundHandler` в конвейере. |

> Вы могли заметить, что у этого класса есть только один метод, в то время как у декодировщиков их два. 
> 
> Причина в том, что декодировщикам часто нужно создать последнее сообщение после закрытия канала (отсюда метод `decodeLast()`). 
> 
> Это, очевидно, не относится к кодировщику — нет смысла создавать сообщение после того, как соединение закрыто.

Рисунок 10.3 показывает `ShortToByteEncoder`, который получает экземпляр `Short` как сообщение, кодирует его в примитивный `short` и записывает в `ByteBuf`, который затем передаётся следующему `ChannelOutboundHandler` в конвейере. Каждый исходящий `Short` будет занимать два байта в `ByteBuf`.

![[Pasted image 20260612093728.png]]
**Рисунок 10.3. ShortToByteEncoder**

Реализация `ShortToByteEncoder` показана в следующем листинге.

**Листинг 10.5. Класс ShortToByteEncoder**
```java
public class ShortToByteEncoder extends MessageToByteEncoder<Short> {   // (1) Расширяет MessageToByteEncoder
    @Override
    public void encode(ChannelHandlerContext ctx, Short msg, ByteBuf out) throws Exception {
        out.writeShort(msg);                                            // (2) Записывает Short в ByteBuf
    }
}
```

### 10.3.2 Абстрактный класс MessageToMessageEncoder

Вы уже видели, как декодировать входящие данные из одного формата сообщения в другой. Чтобы завершить картину, мы покажем, как кодировать из одного сообщения в другое для исходящих данных. Метод `encode()` из `MessageToMessageEncoder` предоставляет эту возможность, как описано в таблице 10.4.

**Таблица 10.4. API MessageToMessageEncoder**

|Название|Описание|
|---|---|
|`encode(ChannelHandlerContext ctx, I msg, List<Object> out)`|Это единственный метод, который вам нужно реализовать. Каждое сообщение, записанное с помощью `write()`, передаётся в `encode()` для кодирования в одно или несколько исходящих сообщений. Затем они передаются следующему `ChannelOutboundHandler` в конвейере.|

Для демонстрации листинг 10.6 расширяет `MessageToMessageEncoder` с помощью `IntegerToStringEncoder`. Дизайн показан на рисунке 10.4.

**Рисунок 10.4. IntegerToStringEncoder**

Как показано в следующем листинге, кодировщик добавляет строковое представление каждого исходящего `Integer` в `List`.

## 10.4 Абстрактные классы кодеков (Abstract codec classes) - объединение декодеров и кодеров

Хотя мы обсуждали декодировщики и кодировщики как отдельные сущности, иногда вам **будет полезно УПРАВЛЯТЬ ПРЕОБРАЗОВАНИЯМИ как входящих, так и исходящих данных и сообщений В ОДНОМ КЛАССЕ.** Абстрактные классы кодеков Netty полезны для этой цели, поскольку каждый из них объединяет пару декодировщик/кодировщик для обработки обоих типов операций, которые мы изучали. Как вы могли предположить, эти классы реализуют как `ChannelInboundHandler`, так и `ChannelOutboundHandler`.

**Почему бы нам не использовать эти составные классы всегда вместо отдельных декодировщиков и кодировщиков?** Потому что разделение двух функций везде, где это возможно, максимизирует возможность повторного использования кода и расширяемость — это основной принцип архитектуры Netty.

Когда мы будем рассматривать абстрактные классы кодеков, мы будем сравнивать их с соответствующими отдельными декодировщиками и кодировщиками.

### 10.4.1 Абстрактный класс ByteToMessageCodec

Давайте рассмотрим случай, когда нам нужно декодировать байты в какой-либо тип сообщения (возможно, POJO), а затем снова закодировать его. `ByteToMessageCodec` справится с этим за нас, поскольку он объединяет `ByteToMessageDecoder` и обратный ему `MessageToByteEncoder`. Важные методы перечислены в таблице 10.5.

**Таблица 10.5. API ByteToMessageCodec**

|Название метода|Описание|
|---|---|
|`decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out)`|Этот метод вызывается до тех пор, пока есть доступные для чтения байты. Он преобразует входящий `ByteBuf` в указанный формат сообщения и передаёт его следующему `ChannelInboundHandler` в конвейере.|
|`decodeLast(ChannelHandlerContext ctx, ByteBuf in, List<Object> out)`|Реализация этого метода по умолчанию делегирует выполнение `decode()`. Он вызывается только один раз, когда канал становится неактивным (`inactive`). Для специальной обработки может быть переопределён.|
|`encode(ChannelHandlerContext ctx, I msg, ByteBuf out)`|Этот метод вызывается для каждого сообщения (типа `I`), которое нужно закодировать и записать в исходящий `ByteBuf`.|

Любой протокол "запрос-ответ" может быть хорошим кандидатом для использования `ByteToMessageCodec`. Например, в реализации SMTP кодек будет читать входящие байты и декодировать их в пользовательский тип сообщения, скажем `SmtpRequest`. На принимающей стороне, когда создаётся ответ, будет создан `SmtpResponse`, который будет закодирован обратно в байты для передачи.

### 10.4.2 Абстрактный класс MessageToMessageCodec

В разделе 9.2.2 вы видели пример `MessageToMessageEncoder`, расширенный для преобразования одного формата сообщения в другой. С помощью `MessageToMessageCodec` мы можем сделать полный цикл с одним классом. `MessageToMessageCodec` — это параметризованный класс, определённый следующим образом:
```java
public abstract class MessageToMessageCodec<INBOUND_IN, OUTBOUND_IN>
```

Важные методы перечислены в таблице 10.6.

**Таблица 10.6. Методы MessageToMessageCodec**

|Название метода|Описание|
|---|---|
|`protected abstract decode(ChannelHandlerContext ctx, INBOUND_IN msg, List<Object> out)`|Этот метод вызывается с сообщениями типа `INBOUND_IN`. Он декодирует их в сообщения типа `OUTBOUND_IN`, которые затем передаются следующему `ChannelInboundHandler` в `ChannelPipeline`.|
|`protected abstract encode(ChannelHandlerContext ctx, OUTBOUND_IN msg, List<Object> out)`|Этот метод вызывается для каждого сообщения типа `OUTBOUND_IN`. Они кодируются в сообщения типа `INBOUND_IN` и передаются следующему `ChannelOutboundHandler` в конвейере.|

Метод `decode()` преобразует сообщение `INBOUND_IN` в тип `OUTBOUND_IN`, а `encode()` делает обратное. Возможно, будет полезно думать о сообщениях `INBOUND_IN` как о типе, отправляемом по сети, а о сообщениях `OUTBOUND_IN` — как о типе, обрабатываемом приложением.

Хотя этот кодек может показаться несколько эзотерическим, вариант его использования довольно распространён: преобразование данных туда и обратно между двумя различными API обмена сообщениями. Мы часто сталкиваемся с этим паттерном, когда нам нужно взаимодействовать с API, использующим устаревший или проприетарный формат сообщений.

> **Протокол WebSocket**
> 
> Следующий пример `MessageToMessageCodec` ссылается на WebSocket — современный протокол, обеспечивающий полную двунаправленную связь между веб-браузерами и серверами. Мы подробно обсудим поддержку WebSocket в Netty в главе 11.

Листинг 10.7 показывает, как может происходить такой обмен. Наш `WebSocketConvertHandler` параметризует `MessageToMessageCodec` с типом `INBOUND_IN` `WebSocketFrame` и типом `OUTBOUND_IN` `MyWebSocketFrame`, причём последний является статическим вложенным классом самого `WebSocketConvertHandler`.

**Листинг 10.7. Использование MessageToMessageCodec**
```java
// (1) Объявляет тип OUTBOUND_IN, используемый WebSocketConvertHandler
public class WebSocketConvertHandler 
extends 
MessageToMessageCodec<
	WebSocketFrame, 
	WebSocketConvertHandler.MyWebSocketFrame> 
{ 

    @Override
    // (2) Кодирует MyWebSocketFrame в указанный подтип WebSocketFrame
    protected void encode(
	    ChannelHandlerContext ctx, 
	    WebSocketConvertHandler.MyWebSocketFrame msg, 
	    List<Object> out
	) throws Exception 
	{ 
        ByteBuf payload = msg.getData().duplicate().retain();
        switch (msg.getType()) {
            case BINARY:
                out.add(new BinaryWebSocketFrame(payload));
                break;
            case TEXT:
                out.add(new TextWebSocketFrame(payload));
                break;
            case CLOSE:
                out.add(new CloseWebSocketFrame(true, 0, payload));
                break;
            case CONTINUATION:
                out.add(new ContinuationWebSocketFrame(payload));
                break;
            case PONG:
                out.add(new PongWebSocketFrame(payload));
                break;
            case PING:
                out.add(new PingWebSocketFrame(payload));
                break;
            default:
                throw new IllegalStateException("Unsupported websocket msg " + msg);
        }
    }

    @Override
    protected void decode(ChannelHandlerContext ctx, WebSocketFrame msg, List<Object> out) throws Exception { // (3) Декодирует WebSocketFrame в MyWebSocketFrame и устанавливает FrameType
        ByteBuf payload = msg.getData().duplicate().retain();
        if (msg instanceof BinaryWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.BINARY, payload));
        } else if (msg instanceof CloseWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.CLOSE, payload));
        } else if (msg instanceof PingWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.PING, payload));
        } else if (msg instanceof PongWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.PONG, payload));
        } else if (msg instanceof TextWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.TEXT, payload));
        } else if (msg instanceof ContinuationWebSocketFrame) {
            out.add(new MyWebSocketFrame(MyWebSocketFrame.FrameType.CONTINUATION, payload));
        } else {
            throw new IllegalStateException("Unsupported websocket msg " + msg);
        }
    }

    public static final class MyWebSocketFrame {                         // (4) Определяет тип WebSocketFrame, который владеет обёрнутой полезной нагрузкой
        public enum FrameType {                                          // (5) Определяет тип WebSocketFrame
            BINARY,
            CLOSE,
            PING,
            PONG,
            TEXT,
            CONTINUATION
        }
        private final FrameType type;
        private final ByteBuf data;

        public MyWebSocketFrame(FrameType type, ByteBuf data) {
            this.type = type;
            this.data = data;
        }
        public FrameType getType() {
            return type;
        }
        public ByteBuf getData() {
            return data;
        }
    }
}
```

### 10.4.3 Класс CombinedChannelDuplexHandler

Как мы упоминали ранее, объединение декодировщика и кодировщика может повлиять на возможность повторного использования. Однако есть способ избежать этого недостатка, не жертвуя удобством развёртывания декодировщика и кодировщика как единого блока. Решение предоставляется классом `CombinedChannelDuplexHandler`, объявленным как:
```java
public class CombinedChannelDuplexHandler<I extends ChannelInboundHandler, O extends ChannelOutboundHandler>
```

Этот класс действует как контейнер для `ChannelInboundHandler` и `ChannelOutboundHandler` (параметры класса `I` и `O`). Предоставляя типы, которые расширяют, соответственно, класс декодировщика и класс кодировщика, мы можем реализовать кодек без необходимости напрямую расширять абстрактные классы кодеков. Мы проиллюстрируем это в следующем примере.

Сначала рассмотрим `ByteToCharDecoder` в этом листинге. Обратите внимание, что реализация расширяет `ByteToMessageDecoder`, потому что он читает символы (`char`) из `ByteBuf`.

**Листинг 10.8. Класс ByteToCharDecoder**
```java
public class ByteToCharDecoder extends ByteToMessageDecoder {          // (1) Расширяет ByteToMessageDecoder
    @Override
    public void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
        while (in.readableBytes() >= 2) {                               // (2) Каждый char занимает 2 байта
            out.add(in.readChar());                                     // (3) Добавляет один или несколько объектов Character в исходящий List
        }
    }
}
```

Здесь `decode()` извлекает по 2 байта за раз из `ByteBuf` и записывает их в `List` как символы (`char`), которые будут автоматически упакованы в объекты `Character`.

Этот листинг показывает `CharToByteEncoder`, который преобразует `Character` обратно в байты. Этот класс расширяет `MessageToByteEncoder`, потому что ему нужно закодировать сообщения `char` в `ByteBuf`. Это делается путём прямой записи в `ByteBuf`.

**Листинг 10.9. Класс CharToByteEncoder**
```java
public class CharToByteEncoder extends MessageToByteEncoder<Character> { // (1) Расширяет MessageToByteEncoder
    @Override
    public void encode(ChannelHandlerContext ctx, Character msg, ByteBuf out) throws Exception {
        out.writeChar(msg);                                               // (2) Декодирует Character в char и записывает в исходящий ByteBuf
    }
}
```

Теперь, когда у нас есть декодировщик и кодировщик, мы объединим их, чтобы построить кодек. Этот листинг показывает, как это делается.

**Листинг 10.10. CombinedChannelDuplexHandler<I, O>**
```java
public class CombinedByteCharCodec extends CombinedChannelDuplexHandler<ByteToCharDecoder, CharToByteEncoder> { // (1) Параметризует CombinedByteCharCodec реализациями декодировщика и кодировщика
    public CombinedByteCharCodec() {
        super(new ByteToCharDecoder(), new CharToByteEncoder());          // (2) Передаёт экземпляры делегатов родительскому классу
    }
}
```

Как вы видите, в некоторых случаях может быть проще и гибче комбинировать реализации таким образом, чем использовать один из классов кодеков. Это также может быть вопросом личных предпочтений.

## 10.5 Резюме (Summary)

В этой главе мы изучили использование API кодеков Netty для написания декодировщиков и кодировщиков. Вы узнали, почему использование этого API предпочтительнее, чем использование напрямую API `ChannelHandler`.

Вы видели, как абстрактные классы кодеков обеспечивают поддержку обработки декодирования и кодирования в одной реализации.
