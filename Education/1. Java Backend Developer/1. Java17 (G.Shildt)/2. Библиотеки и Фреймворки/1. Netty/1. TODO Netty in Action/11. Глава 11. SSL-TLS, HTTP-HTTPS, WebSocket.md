## 11.1 Защита приложений Netty с помощью SSL/TLS

Конфиденциальность данных сегодня является предметом большой озабоченности, и как разработчики мы должны быть готовы её обеспечить. Как минимум мы должны быть знакомы с протоколами шифрования, такими как SSL и TLS, которые накладываются поверх других протоколов для обеспечения безопасности данных. 

Все мы сталкивались с этими протоколами при доступе к защищённым веб-сайтам, но они также используются в приложениях, не основанных на HTTP, таких как защищённые почтовые сервисы Secure SMTP (SMTPS) и даже системы реляционных баз данных.

**Для поддержки SSL/TLS Java предоставляет пакет** `javax.net.ssl`, *чьи классы `SSLContext` и `SSLEngine` делают реализацию дешифрования и шифрования довольно простой.* 

> [!IMPORTANT] `SslHandler` 
> **Netty использует этот API через реализацию `ChannelHandler` под названием `SslHandler`**, *который внутренне использует* `SSLEngine` для выполнения реальной работы.


> **Реализация OpenSSL/SSLEngine в Netty**
> 
> *Netty также предоставляет реализацию* `SSLEngine`, которая использует инструментарий OpenSSL ([www.openssl.org](https://www.openssl.org/)). 
> **Этот класс,** `OpenSslEngine`, обеспечивает лучшую производительность, чем реализация `SSLEngine`, поставляемая с JDK.
> 
> Приложения Netty (клиенты и серверы) могут быть настроены на использование `OpenSslEngine` по умолчанию, если доступны библиотеки OpenSSL. Если нет, Netty возвращается к реализации JDK. Для получения подробных инструкций по настройке поддержки OpenSSL, пожалуйста, обратитесь к документации Netty.
> 
> Обратите внимание, что API SSL и поток данных идентичны независимо от того, используете ли вы JDK `SSLEngine` или Netty `OpenSslEngine`.

Листинг 11.1 показывает, как `SslHandler` добавляется в `ChannelPipeline` с помощью `ChannelInitializer`. Напомним, что `ChannelInitializer` используется для настройки `ChannelPipeline` после регистрации канала.

**Листинг 11.1. Добавление поддержки SSL/TLS**
```java
public class SslChannelInitializer extends ChannelInitializer<Channel> {
    private final SslContext context;
    private final boolean startTls;
	
	// (1) Передаёт SslContext для использования
    public SslChannelInitializer(SslContext context, boolean startTls) { 
        this.context = context;
        this.startTls = startTls;
    }

    @Override
    // (2) Для каждого экземпляра SslHandler получает новый SSLEngine из SslContext, используя ByteBufAllocator канала
    protected void initChannel(Channel ch) throws Exception {
        SSLEngine engine = context.newEngine(ch.alloc());  
        
        // (3) Добавляет SslHandler в конвейер первым обработчиком
        ch.pipeline().addFirst("ssl", new SslHandler(engine, startTls)); 
    }
}
```

> В большинстве случаев `SslHandler` будет первым `ChannelHandler` в `ChannelPipeline`. Это гарантирует, что шифрование будет происходить только после того, как все остальные `ChannelHandler` применят свою логику к данным.

`SslHandler` имеет **несколько полезных методов**, как показано в таблице 11.1. Например, во время фазы handshake два узла проверяют друг друга и согласовывают метод шифрования. Вы можете настроить `SslHandler` для изменения его поведения или получения уведомления после завершения handshake SSL/TLS, после чего все данные будут зашифрованы. Handshake SSL/TLS будет выполнен автоматически.

**Таблица 11.1. Методы SslHandler**

| Название                                                                                                                  | Описание                                                                                                                                                                                               |
| ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `setHandshakeTimeout(long, TimeUnit)`<br><br>`setHandshakeTimeoutMillis(long)`  <br><br>`getHandshakeTimeoutMillis()`     | Устанавливает и получает таймаут, по истечении которого `ChannelFuture` handshake будет уведомлён об ошибке.                                                                                           |
| `setCloseNotifyTimeout(long, TimeUnit)`  <br><br>`setCloseNotifyTimeoutMillis(long)`  <br>`getCloseNotifyTimeoutMillis()` | Устанавливает и получает таймаут, по истечении которого будет отправлено уведомление о закрытии (close_notify) и соединение будет закрыто. Это также приводит к уведомлению `ChannelFuture` об ошибке. |
| `handshakeFuture()`                                                                                                       | Возвращает `ChannelFuture`, который будет уведомлён о завершении handshake. Если handshake уже был выполнен, возвращает `ChannelFuture`, содержащий результат предыдущего handshake.                   |
| `close()`  <br><br>`close(ChannelPromise)`  <br><br>`close(ChannelHandlerContext, ChannelPromise)`                        | Отправляет `close_notify` для запроса закрытия и уничтожает нижележащий `SSLEngine`.                                                                                                                   |

## 11.2 Создание HTTP/HTTPS приложений на Netty

HTTP/HTTPS — один из самых распространённых наборов протоколов, и с успехом смартфонов он используется всё шире с каждым днём, поскольку для любой компании стало практически обязательным иметь веб-сайт, доступный с мобильных устройств. Эти протоколы используются и другими способами. API веб-сервисов, предоставляемые многими организациями для связи с их бизнес-партнёрами, обычно основаны на HTTP(S).

Далее мы рассмотрим `ChannelHandler`, которые Netty предоставляет для использования HTTP и HTTPS без необходимости писать пользовательские кодеки.

**Рисунки 11.2 и 11.3 показывают методы для создания и потребления HTTP-запросов и ответов** соответственно.

Как показано на рисунках 11.2 и 11.3, ***HTTP-запрос/ответ может состоять из более чем одной части данных*** **и всегда заканчивается частью** `LastHttpContent`. 

**Сообщения `FullHttpRequest` и `FullHttpResponse` являются специальными подтипами, которые представляют полный запрос и ответ** соответственно. 
Все типы HTTP-сообщений (`FullHttpRequest`, `LastHttpContent` и показанные в листинге 11.2) **реализуют интерфейс** `HttpObject`.

Таблица 11.2 даёт обзор HTTP-декодировщиков и кодировщиков, которые обрабатывают и создают эти сообщения.

**Таблица 11.2. HTTP-декодировщики и кодировщики**

| Название              | Описание                                                                           |
| --------------------- | ---------------------------------------------------------------------------------- |
| `HttpRequestEncoder`  | **Кодирует сообщения** `HttpRequest`, `HttpContent` и `LastHttpContent` в байты.   |
| `HttpResponseEncoder` | Кодирует сообщения `HttpResponse`, `HttpContent` и `LastHttpContent` в байты.      |
| `HttpRequestDecoder`  | **Декодирует байты в сообщения `HttpRequest`, `HttpContent` и `LastHtt**pContent`. |
| `HttpResponseDecoder` | Декодирует байты в сообщения `HttpResponse`, `HttpContent` и `LastHttpContent`.    |

Класс `HttpPipelineInitializer` в следующем листинге показывает, насколько просто добавить поддержку HTTP в ваше приложение — просто добавьте правильные `ChannelHandler` в `ChannelPipeline`.

**Листинг 11.2. Добавление поддержки HTTP**
```java
public class HttpPipelineInitializer extends ChannelInitializer<Channel> {
    private final boolean client;

    public HttpPipelineInitializer(boolean client) {
        this.client = client;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        if (client) {
	        // (1) Если клиент, добавляет HttpResponseDecoder для обработки ответов от сервера
            pipeline.addLast("decoder", new HttpResponseDecoder());   
            
            // (2) Если клиент, добавляет HttpRequestEncoder для отправки запросов серверу
            pipeline.addLast("encoder", new HttpRequestEncoder());
                 
        } else {
	        // (3) Если сервер, добавляет HttpRequestDecoder для получения запросов от клиента
            pipeline.addLast("decoder", new HttpRequestDecoder());  
            
			// (4) Если сервер, добавляет HttpResponseEncoder для отправки ответов клиенту   
            pipeline.addLast("encoder", new HttpResponseEncoder());    
        }
    }
}
```

### 11.2.2 Агрегация HTTP-сообщений

После того как инициализатор установил обработчики в `ChannelPipeline`, вы можете работать с различными объектами `HttpObject`. 

*Но поскольку HTTP-запросы и ответы могут состоять из многих частей, вам нужно будет агрегировать их для формирования полных сообщений*. 

*Чтобы устранить эту утомительную задачу,* **Netty предоставляет агрегатор, который объединяет части сообщений в сообщения** `FullHttpRequest` и `FullHttpResponse`. Таким образом, вы всегда видите полное содержимое сообщения.

У этой операции есть небольшая стоимость, поскольку **сегменты сообщений должны буферизироваться до тех пор, пока полные сообщения не могут быть переданы следующему** `ChannelInboundHandler`. Компромисс заключается в том, что вам не нужно беспокоиться о фрагментации сообщений.

Внедрение этой автоматической агрегации сводится к добавлению ещё одного `ChannelHandler` в конвейер. В этом листинге показано, как это делается.

**Листинг 11.3. Автоматическая агрегация фрагментов HTTP-сообщений**
```java
public class HttpAggregatorInitializer extends ChannelInitializer<Channel> {
    private final boolean isClient;

    public HttpAggregatorInitializer(boolean isClient) {
        this.isClient = isClient;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        if (isClient) {
	        // (1) Если клиент, добавляет HttpClientCodec
            pipeline.addLast("codec", new HttpClientCodec());        
        } else {
	        // (2) Если сервер, добавляет HttpServerCodec
            pipeline.addLast("codec", new HttpServerCodec());        
        }
        
	    // (3) Добавляет HttpObjectAggregator с максимальным размером сообщения 512 КБ в конвейер
        pipeline.addLast(
	        "aggregator", new HttpObjectAggregator(512 * 1024)
	    ); 
    }
}
```

### 11.2.3 Сжатие HTTP

При использовании HTTP целесообразно применять сжатие для максимального уменьшения размера передаваемых данных. Хотя сжатие имеет некоторую стоимость в циклах процессора, в целом это хорошая идея, особенно для текстовых данных.

==Netty предоставляет реализации== `ChannelHandler` ==для сжатия и декомпрессии==, которые поддерживают кодировки gzip и deflate.

---
#### **Заголовок HTTP-запроса**

Клиент может указать поддерживаемые режимы шифрования, предоставив следующий заголовок:
```http
GET /encrypted-area HTTP/1.1
Host: www.example.com
Accept-Encoding: gzip, deflate
```

Однако обратите внимание, *что сервер не обязан сжимать отправляемые данные.*

---
Пример показан в следующем листинге.

**Листинг 11.4. Автоматическое сжатие HTTP-сообщений**
```java
public class HttpCompressionInitializer extends ChannelInitializer<Channel> {
    private final boolean isClient;

    public HttpCompressionInitializer(boolean isClient) {
        this.isClient = isClient;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        if (isClient) {
	        // (1) Если клиент, добавляет HttpClientCodec
            pipeline.addLast("codec", new HttpClientCodec());   
            
			// (2) Если клиент, добавляет HttpContentDecompressor для обработки сжатого контента от сервера
            pipeline.addLast(
	            "decompressor", new HttpContentDecompressor()); 
	        
        } else {
	        // (3) Если сервер, добавляет HttpServerCodec
            pipeline.addLast("codec", new HttpServerCodec()); 
            
			// (4) Если сервер, добавляет HttpContentCompressor для сжатия данных (если клиент поддерживает)
            pipeline.addLast("compressor", new HttpContentCompressor()); 
        }
    }
}
```

> **Сжатие и зависимости** 
> 
> Если вы используете JDK 6 или более раннюю версию, вам нужно добавить JZlib ([www.jcraft.com/jzlib/](https://www.jcraft.com/jzlib/)) в CLASSPATH для поддержки сжатия.

Для Maven добавьте следующую зависимость:
```xml
<dependency>
    <groupId>com.jcraft</groupId>
    <artifactId>jzlib</artifactId>
    <version>1.1.3</version>
</dependency>
```

### 11.2.4 Использование HTTPS

Следующий листинг показывает, **что включение HTTPS — это всего лишь вопрос добавления** `SslHandler` в конвейер.

**Листинг 11.5. Использование HTTPS**
```java
public class HttpsCodecInitializer extends ChannelInitializer<Channel> {
    private final SslContext context;
    private final boolean isClient;

    public HttpsCodecInitializer(SslContext context, boolean isClient) {
        this.context = context;
        this.isClient = isClient;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        SSLEngine engine = context.newEngine(ch.alloc());
        
        // (1) Добавляет SslHandler в конвейер для использования HTTPS
        pipeline.addFirst("ssl", new SslHandler(engine));            
        if (isClient) {
	        // (2) Если клиент, добавляет HttpClientCodec
            pipeline.addLast("codec", new HttpClientCodec()); 
                   
        } else {
	        // (3) Если сервер, добавляет HttpServerCodec
            pipeline.addLast("codec", new HttpServerCodec());        
        }
    }
}
```

Предыдущий код является хорошим примером того, как архитектурный подход Netty превращает повторное использование в рычаг. Просто добавив `ChannelHandler` в `ChannelPipeline`, вы можете предоставить новую возможность, даже такую значительную, как шифрование.

### 11.2.5 WebSocket

Обширный инструментарий Netty для приложений на основе HTTP включает поддержку некоторых из его самых передовых функций. В этом разделе мы рассмотрим WebSocket — протокол, стандартизированный IETF в 2011 году.

WebSocket решает давнюю проблему: как публиковать информацию в реальном времени, учитывая, что нижележащий протокол HTTP представляет собой последовательность взаимодействий "запрос-ответ". AJAX обеспечивает некоторое улучшение, но поток данных по-прежнему управляется запросами со стороны клиента. Были и другие более или менее изобретательные подходы, но в конечном итоге они остались обходными путями с ограниченной масштабируемостью.

Спецификация WebSocket и её реализации представляют собой попытку более эффективного решения. **Проще говоря, WebSocket обеспечивает "одно TCP-соединение для трафика в обоих направлениях... В сочетании с WebSocket API... он предоставляет альтернативу HTTP-опросу для двусторонней связи с веб-страницы с удалённым сервером"**.

То есть ==WebSocket обеспечивает истинный двунаправленный обмен данными между клиентом и сервером.== Мы не будем углубляться в детали внутреннего устройства, *но следует упомянуть, что хотя самые ранние реализации были ограничены текстовыми данными, **сейчас это уже не так;** **WebSocket теперь можно использовать для любых данных, подобно обычному сокету.***

Рисунок 11.4 даёт общее представление о протоколе WebSocket. В этом сценарии связь начинается как обычный HTTP и обновляется до двунаправленного WebSocket.

**Рисунок 11.4. Протокол WebSocket**

Этот класс будет обрабатывать ==СПЕЦИАЛЬНЫЕ ТИПЫ СООБЩЕНИЙ, определённые WebSocket, известные как== **фреймы (frames)**. Как показано в таблице 11.3, `WebSocketFrame` можно разделить на фреймы данных или управляющие фреймы.

| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------- |

Листинг 11.6 представляет ==простой пример использования **==`WebSocketServerProtocolHandler`.** 
*Этот класс обрабатывает handshake обновления протокола, а также три управляющих фрейма — Close, Ping и Pong.* Фреймы данных Text и Binary будут переданы следующим обработчикам (реализованным вами) для обработки.

public class WebSocketServerInitializer extends ChannelInitializer<Channel> {
    @Override

        }
    }

Для более обширного примера, пожалуйста, обратитесь к главе 12, где подробно рассматривается проектирование приложения WebSocket в реальном времени.
