## 11.3 Неактивные соединения и таймауты

До сих пор наше обсуждение было сосредоточено на поддержке Netty вариантов HTTP, HTTPS и WebSocket через специализированные кодеки и обработчики. Эти технологии могут сделать ваши веб-приложения более эффективными, удобными и безопасными при условии, что вы эффективно управляете сетевыми ресурсами. 

Итак, давайте поговорим о главной проблеме — управлении соединениями.

Обнаружение неактивных соединений и таймаутов необходимо для своевременного освобождения ресурсов. **Это настолько распространённая задача, что Netty предоставляет несколько реализаций `ChannelHandler` именно для этой цели.** В таблице 11.4 дан обзор этих обработчиков.

**Таблица 11.4. ChannelHandler для неактивных соединений и таймаутов**

| Название              | Описание                                                                                                                                                                                                                                           |
| --------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `IdleStateHandler`    | Генерирует `IdleStateEvent`, **если соединение простаивает слишком долго.** <br><br>Затем вы можете обработать `IdleStateEvent`, переопределив `userEventTriggered()` в вашем `ChannelInboundHandler`.                                             |
| `ReadTimeoutHandler`  | Выбрасывает `ReadTimeoutException` и закрывает канал, **если в течение указанного интервала не получено входящих данных.** <br><br>`ReadTimeoutException` может быть обнаружена переопределением `exceptionCaught()` в вашем `ChannelHandler`.     |
| `WriteTimeoutHandler` | Выбрасывает `WriteTimeoutException` и закрывает канал, **если в течение указанного интервала данные не могут быть записаны.** <br><br>`WriteTimeoutException` может быть обнаружена переопределением `exceptionCaught()` в вашем `ChannelHandler`. |

Давайте подробнее рассмотрим `IdleStateHandler`, который чаще всего используется на практике. 
*Листинг 11.7 показывает, как получить уведомление, если в течение 60 секунд не было получено или отправлено данных*, **используя распространённый метод отправки heartbeat-сообщения удалённому узлу**; если ответа нет, соединение закрывается.

**Листинг 11.7. Отправка heartbeat-сообщений**
```java
public class IdleStateHandlerInitializer extends ChannelInitializer<Channel> {
    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        
        pipeline.addLast(
	        // (1) IdleStateHandler генерирует IdleStateEvent при срабатывании
	        new IdleStateHandler(0, 0, 60, TimeUnit.SECONDS)
	    ); 
	    
	    // (2) Добавляет HeartbeatHandler в конвейер
        pipeline.addLast(new HeartbeatHandler());                            
    }

    public static final class HeartbeatHandler 
    extends ChannelInboundHandlerAdapter {
	    // (3) Heartbeat для отправки удалённому узлу
        private static final ByteBuf HEARTBEAT_SEQUENCE = 
	        Unpooled.unreleasableBuffer( 
                Unpooled.copiedBuffer(
	                "HEARTBEAT", 
	                CharsetUtil.ISO_8859_1)
	            );

        @Override
        public void userEventTriggered(
	        ChannelHandlerContext ctx, 
	        Object evt
	    ) throws Exception {
		    // (4) Реализует userEventTriggered() для отправки heartbeat
            if (evt instanceof IdleStateEvent) {                          
	            ctx
		            // (5) Отправляет heartbeat
		            .writeAndFlush(HEARTBEAT_SEQUENCE.duplicate())
		            // и закрывает соединение, если отправка не удалась
		            .addListener(ChannelFutureListener.CLOSE_ON_FAILURE);      
            } else {
	            // (6) Не IdleStateEvent, передаёт следующему обработчику
                super.userEventTriggered(ctx, evt);
            }
        }
    }
}
```

Этот пример иллюстрирует, как использовать `IdleStateHandler`, чтобы проверить, жив ли удалённый узел, и освободить ресурсы, закрыв соединение, если это не так. 

`IdleStateHandler` **(1)** вызовет `userEventTriggered()` с `IdleStateEvent`, если соединение не получало и не отправляло данные в течение 60 секунд. `HeartbeatHandler` реализует `userEventTriggered()`. 

Если этот метод обнаруживает `IdleStateEvent`, он отправляет heartbeat-сообщение и добавляет `ChannelFutureListener`, который закрывает соединение в случае сбоя операции отправки **(5)**.

## 11.4 Декодирование протоколов с разделителями и основанных на длине

*При работе с Netty вы столкнётесь с протоколами с разделителями и основанными на длине, которые требуют декодеров.* В следующих разделах объясняются реализации, которые Netty предоставляет для обработки этих случаев.

### 11.4.1 Протоколы с разделителями (Delimited protocols)

==Протоколы сообщений с разделителями используют ОПРЕДЕЛЕННЫЕ СИМВОЛЫ ДЛЯ ОБОЗНАЧЕНИЯ НАЧАЛА И КОНЦА СООБЩЕНИЯ== или сегмента сообщения, часто называемого **кадром (frame)**. Это верно для многих протоколов, формально определённых документами RFC, таких как *SMTP, POP3, IMAP и Telnet.* 

И, конечно, частные организации часто имеют свои собственные проприетарные форматы. С каким бы протоколом вы ни работали, декодеры, перечисленные в таблице 11.5, помогут вам определить пользовательские декодеры, которые могут извлекать кадры, разделённые любой последовательностью токенов.

**Таблица 11.5. Декодеры для обработки протоколов с разделителями и основанных на длине**

| Название                     | Описание                                                                                                                                                  |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `DelimiterBasedFrameDecoder` | Универсальный декодер, который извлекает кадры, **используя любой предоставленный пользователем разделитель.**                                            |
| `LineBasedFrameDecoder`      | Декодер, **который извлекает кадры, разделённые символами конца строки `\n` ил**и `\r\n`. <br><br>Этот декодер быстрее, чем `DelimiterBasedFrameDecoder`. |

Рисунок 11.5 показывает, как обрабатываются кадры, когда они разделяются последовательностью конца строки `\r\n` (возврат каретки + перевод строки).

![[Pasted image 20260612101616.png]]
**Рисунок 11.5. Кадры, разделённые концами строк**

Следующий листинг показывает, как вы можете использовать `LineBasedFrameDecoder` для обработки случая, показанного на рисунке 11.5.

**Листинг 11.8. Обработка кадров, разделённых концами строк**
```java
public class LineBasedHandlerInitializer extends ChannelInitializer<Channel> {
    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
	    
	    // (1) LineBasedFrameDecoder передаёт извлечённые кадры следующему обработчику
        pipeline.addLast(new LineBasedFrameDecoder(64 * 1024)); 
	    
	    // (2) Добавляет FrameHandler для получения кадров
        pipeline.addLast(new FrameHandler());                            
    }

    public static final class FrameHandler 
    extends SimpleChannelInboundHandler<ByteBuf> {
        @Override
        // (3) Передаёт содержимое одного кадра
        
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        ByteBuf msg
	    ) throws Exception { 
		    // Делаем что-то с данными, извлечёнными из кадра
        }
    }
}
```

Если вы работаете с кадрами, разделёнными чем-то отличным от концов строк, вы можете использовать `DelimiterBasedFrameDecoder` аналогичным образом, указав конкретную последовательность разделителя в конструкторе.

Эти декодеры являются инструментами для реализации ваших собственных протоколов с разделителями. 

**В качестве примера** мы будем использовать следующую спецификацию протокола:
- Входящий поток данных представляет собой серию кадров, каждый из которых разделён переводом строки (`\n`).
    
- Каждый кадр состоит из серии элементов, каждый из которых разделён одним символом пробела.
    
- Содержимое кадра представляет команду, определённую как имя, за которым следует переменное количество аргументов.
    

Наш пользовательский декодер для этого протокола будет определять следующие классы:

- `Cmd` — хранит содержимое кадра (команду) в одном `ByteBuf` для имени и другом для аргументов.
    
- `CmdDecoder` — извлекает строку из переопределённого метода `decode()` и создаёт экземпляр `Cmd` из её содержимого.
    
- `CmdHandler` — получает декодированный объект `Cmd` от `CmdDecoder` и выполняет некоторую обработку.
    
- `CmdHandlerInitializer` — для простоты мы определим предыдущие классы как вложенные классы специализированного `ChannelInitializer`, который установит обработчики в конвейере.
    

Как вы можете видеть в следующем листинге, ключ к этому декодеру — расширение `LineBasedFrameDecoder`.

**Листинг 11.9. Использование ChannelInitializer в качестве установщика декодера**
```java
public class CmdHandlerInitializer extends ChannelInitializer<Channel> {
    final byte SPACE = (byte) ' ';

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
	    // (1) Добавляет CmdDecoder для извлечения 
	    // объекта Cmd и передачи его следующему обработчику
        pipeline.addLast(new CmdDecoder(64 * 1024));  
        
		// (2) Добавляет CmdHandler 
		// для получения и обработки объектов Cmd   
        pipeline.addLast(new CmdHandler());              
    }

    public static final class Cmd {  // (3) POJO Cmd
        private final ByteBuf name;
        private final ByteBuf args;

        public Cmd(ByteBuf name, ByteBuf args) {
            this.name = name;
            this.args = args;
        }

        public ByteBuf name() {
            return name;
        }

        public ByteBuf args() {
            return args;
        }
    }

    public static final class CmdDecoder extends LineBasedFrameDecoder {
        public CmdDecoder(int maxLength) {
            super(maxLength);
        }

        @Override
        protected Object decode(ChannelHandlerContext ctx, ByteBuf buffer) throws Exception {
	        // (4) Извлекает кадр, разделённый 
	        // последовательностью конца строки, из ByteBuf
            ByteBuf frame = (ByteBuf) super.decode(ctx, buffer); 
            
            // (5) Null возвращается, если во входных данных нет кадра  
            if (frame == null) {                                    
                return null;
            }
            
            // (6) Находит индекс первого пробела. Имя команды предшествует ему, аргументы следуют за ним.
            int index = frame.indexOf(
	            frame.readerIndex(), 
	            frame.writerIndex(), 
	            SPACE
	        ); 
	        
	        // (7) Создаёт новый объект Cmd с срезами, 
	        // содержащими имя команды и аргументы
            return new Cmd(
	            frame.slice(frame.readerIndex(), index),  
                frame.slice(index + 1, frame.writerIndex())
            );
        }
    }

    public static final class CmdHandler extends 
    SimpleChannelInboundHandler<Cmd> {
        @Override
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        Cmd msg
	    ) throws Exception {
            // Обрабатываем объект Cmd, переданный через конвейер
        }
    }
}
```

### 11.4.2 Протоколы, основанные на длине (Length-based protocols)

Протокол, основанный на длине, определяет кадр, кодируя его длину в сегменте заголовка кадра, а не отмечая его конец специальным разделителем. В таблице 11.6 перечислены два декодера, которые Netty предоставляет для обработки этого типа протоколов.

**Таблица 11.6. Декодеры для протоколов, основанных на длине**

| Название                       | Описание                                                                                                                                  |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `FixedLengthFrameDecoder`      | **Извлекает кадры фиксированного размера**, указанного при вызове конструктора.                                                           |
| `LengthFieldBasedFrameDecoder` | **Извлекает кадры на основе значения длины, закодированного в поле в заголовке кадра;** смещение и длина поля указываются в конструкторе. |

Рисунок 11.6 показывает работу `FixedLengthFrameDecoder`, который был создан с длиной кадра 8 байт.

![[Pasted image 20260612101730.png]]
**Рисунок 11.6. Декодирование с длиной кадра 8 байт**

Вы часто будете встречать протоколы, где размер кадра, закодированный в заголовке сообщения, не является фиксированным значением. Для обработки таких кадров переменной длины вы будете использовать `LengthFieldBasedFrameDecoder`, который определяет длину кадра из поля заголовка и извлекает указанное количество байт из потока данных.

Рисунок 11.7 показывает пример, где поле длины в заголовке находится по смещению 0 и имеет длину 2 байта.

`LengthFieldBasedFrameDecoder` предоставляет несколько конструкторов для охвата различных вариантов конфигурации заголовка. Листинг 11.10 показывает использование конструктора, три аргумента которого: `maxFrameLength`, `lengthFieldOffset` и `lengthFieldLength`. В этом случае длина кадра закодирована в первых 8 байтах кадра.

**Листинг 11.10. Декодер для команды и обработчик**
```java
public class LengthBasedInitializer extends ChannelInitializer<Channel> {
    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        // (1) LengthFieldBasedFrameDecoder для сообщений, 
        // кодирующих длину кадра в первых 8 байтах
        pipeline.addLast(
	        new LengthFieldBasedFrameDecoder(64 * 1024, 0, 8)
	    ); 
	    
	    // (2) Добавляет FrameHandler для обработки каждого кадра
        pipeline.addLast(new FrameHandler());                                 
    }

    public static final class FrameHandler extends SimpleChannelInboundHandler<ByteBuf> {
        @Override
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        ByteBuf msg
	    ) throws Exception {
            // Обрабатываем данные кадра
        }
    }
}
```

Вы теперь видели кодеки, которые Netty предоставляет для поддержки протоколов, определяющих структуру байтовых потоков путём указания либо разделителей, либо длины (фиксированной или переменной) кадра протокола. Вы найдёте множество применений для этих кодеков, поскольку очень многие распространённые протоколы попадают в одну или другую категорию.
