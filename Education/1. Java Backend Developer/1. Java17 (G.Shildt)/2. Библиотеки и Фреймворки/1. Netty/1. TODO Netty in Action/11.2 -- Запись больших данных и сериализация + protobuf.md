## 11.5 Запись больших объёмов данных (Writing big data)

Эффективная запись больших фрагментов данных является особой проблемой в асинхронных фреймворках из-за возможности насыщения сети. 

**Поскольку операции записи являются неблокирующими, они возвращают управление и уведомляют `ChannelFuture`, ДАЖЕ ЕСЛИ НЕ ВСЕ ДАННЫЕ БЫЛИ ЗАПИСАНЫ.** 

Когда это происходит, *если вы не остановите запись, вы рискуете исчерпать память* (`OutOfMemoryError`). Поэтому при записи больших объёмов данных вы должны быть готовы обрабатывать случаи, когда медленное соединение с удалённым узлом может вызвать задержки в освобождении памяти. Рассмотрим случай записи содержимого файла в сеть.

В нашем обсуждении транспортов (см. раздел 4.2) мы упомянули функцию **zero-copy** в NIO, которая устраняет шаги копирования при перемещении содержимого файла из файловой системы в сетевой стек. 

Всё это происходит внутри Netty, **поэтому всё, что требуется от приложения, — это использовать реализацию интерфейса** `FileRegion`, определённого в документации Netty API как "область файла, которая отправляется через канал, поддерживающий zero-copy передачу файлов".

Следующий листинг показывает, как вы можете передать содержимое файла, используя zero-copy, создав `DefaultFileRegion` из `FileInputStream` и записав его в канал.

**Листинг 11.11. Передача содержимого файла с помощью FileRegion**
```java
 // (1) Создаёт FileInputStream
FileInputStream in = new FileInputStream(file);  

// (2) Создаёт новый DefaultFileRegion для полной длины файла             
FileRegion region = new DefaultFileRegion(
	in.getChannel(), 
	0, 
	file.length()
); 

channel.writeAndFlush(region).addListener(new ChannelFutureListener() { // (3) Отправляет DefaultFileRegion и регистрирует ChannelFutureListener
    @Override
    public void operationComplete(ChannelFuture future) throws Exception {
        if (!future.isSuccess()) {
	        // (4) Обрабатывает ошибку
            Throwable cause = future.cause();                         
            // Do something
        }
    }
});
```

Этот пример применим **только к прямой передаче содержимого файла без какой-либо обработки данных приложением.** 

В случаях, **когда вам нужно скопировать данные из файловой системы в память пользовательского пространства**, вы можете ==использовать== **`ChunkedWriteHandler`**, который обеспечивает поддержку асинхронной записи большого потока данных без высокого потребления памяти.

Ключевым является интерфейс `ChunkedInput<B>`, где параметр `B` — это тип, возвращаемый методом `readChunk()`. Предоставляются четыре реализации этого интерфейса, перечисленные в таблице 11.7. Каждая из них представляет поток данных неопределённой длины, который будет потребляться `ChunkedWriteHandler`.

**Таблица 11.7. Реализации ChunkedInput**

| Название           | Описание                                                                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ChunkedFile`      | **Извлекает данные из файла фрагмент за фрагментом** (chunk by chunk) для использования, *когда ваша платформа не поддерживает zero-copy или когда вам нужно преобразовать данные.* |
| `ChunkedNioFile`   | Похож на `ChunkedFile`, за исключением того, что использует `FileChannel`.                                                                                                          |
| `ChunkedStream`    | Передаёт содержимое фрагмент за фрагментом из `InputStream`.                                                                                                                        |
| `ChunkedNioStream` | Передаёт содержимое фрагмент за фрагментом из `ReadableByteChannel`.                                                                                                                |

Листинг 11.12 иллюстрирует использование `ChunkedStream` — реализации, которая чаще всего используется на практике. Показанный класс создаётся с экземплярами `File` и `SslContext`. Когда вызывается `initChannel()`, он инициализирует канал показанной цепочкой обработчиков.

Когда канал становится активным, `WriteStreamHandler` будет записывать данные из файла фрагмент за фрагментом как `ChunkedStream`. Данные будут зашифрованы `SslHandler` перед передачей.

**Листинг 11.12. Передача содержимого файла с помощью ChunkedStream**
```java
public class ChunkedWriteHandlerInitializer extends ChannelInitializer<Channel> {
    private final File file;
    private final SslContext sslCtx;

    public ChunkedWriteHandlerInitializer(File file, SslContext sslCtx) {
        this.file = file;
        this.sslCtx = sslCtx;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        // (1) Добавляет SslHandler в ChannelPipeline
        pipeline.addLast(new SslHandler(sslCtx.newEngine(ch.alloc())));
        
        // (2) Добавляет ChunkedWriteHandler для обработки данных, передаваемых как ChunkedInput 
        pipeline.addLast(new ChunkedWriteHandler());
        
		// (3) WriteStreamHandler начинает запись данных файла после установления соединения
        pipeline.addLast(new WriteStreamHandler());                     
    }

    public final class WriteStreamHandler extends ChannelInboundHandlerAdapter {
        @Override
        // (4) channelActive() записывает данные файла, используя ChunkedInput, когда соединение установлено
        public void channelActive(ChannelHandlerContext ctx) throws Exception { 
            super.channelActive(ctx);
            ctx.writeAndFlush(
	            new ChunkedStream(new FileInputStream(file))
	        );
        }
    }
}
```

> **CHUNKED INPUT**: Чтобы использовать свою собственную реализацию `ChunkedInput`, установите `ChunkedWriteHandler` в конвейере.

В этом разделе мы обсудили, как эффективно передавать файлы, используя функцию zero-copy, и как записывать большие данные без риска `OutOfMemoryError`, используя `ChunkedWriteHandler`. В следующем разделе мы рассмотрим несколько подходов к сериализации POJO.

## 11.6 Сериализация данных (Serializing data)

JDK предоставляет `ObjectOutputStream` и `ObjectInputStream` для сериализации и десериализации примитивных типов данных и графов POJO по сети. 

API не сложен и может применяться к любому объекту, реализующему `java.io.Serializable`. Но он также не очень эффективен. В этом разделе мы посмотрим, что предлагает Netty.

### 11.6.1 Сериализация JDK

Если вашему приложению приходится взаимодействовать с узлами, которые используют `ObjectOutputStream` и `ObjectInputStream`, и совместимость является вашей главной заботой, то сериализация JDK — правильный выбор. 

В таблице 11.8 перечислены классы сериализации, которые Netty предоставляет для взаимодействия с JDK.

**Таблица 11.8. Кодеки сериализации JDK**

|Название|Описание|
|---|---|
|`CompatibleObjectDecoder`|Декодер для взаимодействия с не-Netty узлами, использующими сериализацию JDK.|
|`CompatibleObjectEncoder`|Кодировщик для взаимодействия с не-Netty узлами, использующими сериализацию JDK.|
|`ObjectDecoder`|Декодер, использующий пользовательскую сериализацию поверх сериализации JDK; обеспечивает повышение скорости, когда внешние зависимости исключены. В противном случае другие реализации сериализации предпочтительнее.|
|`ObjectEncoder`|Кодировщик, использующий пользовательскую сериализацию поверх сериализации JDK; обеспечивает повышение скорости, когда внешние зависимости исключены. В противном случае другие реализации сериализации предпочтительнее.|

### 11.6.2 Сериализация с JBoss Marshalling

Если вы можете свободно использовать внешние зависимости, **JBoss Marshalling** идеален: он до трёх раз быстрее, чем сериализация JDK, и более компактен. В обзоре на домашней странице JBoss Marshalling это определяется так:

> JBoss Marshalling — это альтернативный API сериализации, который исправляет многие проблемы, обнаруженные в API сериализации JDK, оставаясь полностью совместимым с `java.io.Serializable` и его родственниками, и добавляет несколько новых настраиваемых параметров и дополнительных функций, все из которых подключаются через конфигурацию фабрики.

Netty поддерживает JBoss Marshalling с двумя парами декодер/кодировщик, показанными в таблице 11.9. Первый набор совместим с узлами, использующими только сериализацию JDK. Второй, обеспечивающий максимальную производительность, предназначен для использования с узлами, использующими JBoss Marshalling.

**Таблица 11.9. Кодеки JBoss Marshalling**

|Название|Описание|
|---|---|
|`CompatibleMarshallingDecoder`  <br>`CompatibleMarshallingEncoder`|Для совместимости с узлами, использующими сериализацию JDK.|
|`MarshallingDecoder`  <br>`MarshallingEncoder`|Для использования с узлами, использующими JBoss Marshalling. Эти классы должны использоваться вместе.|

Следующий листинг показывает, как использовать `MarshallingDecoder` и `MarshallingEncoder`. Опять же, это в основном вопрос правильной настройки `ChannelPipeline`.

**Листинг 11.13. Использование JBoss Marshalling**
```java
public class MarshallingInitializer extends ChannelInitializer<Channel> {
    private final MarshallerProvider marshallerProvider;
    private final UnmarshallerProvider unmarshallerProvider;

    public MarshallingInitializer(UnmarshallerProvider unmarshallerProvider,
                                  MarshallerProvider marshallerProvider) {
        this.marshallerProvider = marshallerProvider;
        this.unmarshallerProvider = unmarshallerProvider;
    }

    @Override
    protected void initChannel(Channel channel) throws Exception {
        ChannelPipeline pipeline = channel.pipeline();
        
        // (1) Добавляет MarshallingDecoder 
        // для преобразования ByteBuf в POJO
        pipeline.addLast(new MarshallingDecoder(unmarshallerProvider)); 
        
        // (2) Добавляет MarshallingEncoder 
        // для преобразования POJO в ByteBuf
        pipeline.addLast(new MarshallingEncoder(marshallerProvider));
        // (3) Добавляет ObjectHandler для обычных POJO, 
        // реализующих Serializable   
        pipeline.addLast(new ObjectHandler());                          
    }

    public static final class ObjectHandler extends SimpleChannelInboundHandler<Serializable> {
        @Override
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        Serializable serializable
	    ) throws Exception 
	    {
            // Do something
        }
    }
}
```

### 11.6.3 Сериализация через Protocol Buffers

Последним из решений Netty для сериализации является кодек, использующий **Protocol Buffers** — формат обмена данными, разработанный Google и теперь доступный с открытым исходным кодом. Код можно найти на [https://github.com/google/protobuf](https://github.com/google/protobuf).

Protocol Buffers кодирует и декодирует структурированные данные компактным и эффективным способом. Он имеет привязки для многих языков программирования, что делает его хорошим выбором для кроссплатформенных проектов. В таблице 11.10 показаны реализации `ChannelHandler`, которые Netty предоставляет для поддержки protobuf.

**Таблица 11.10. Кодек Protobuf**

|Название|Описание|
|---|---|
|`ProtobufDecoder`|Декодирует сообщение с помощью protobuf.|
|`ProtobufEncoder`|Кодирует сообщение с помощью protobuf.|
|`ProtobufVarint32FrameDecoder`|Разделяет полученные `ByteBuf` динамически по значению целочисленного поля длины "Base 128 Varints" протокола Google.|

И снова использование protobuf — это вопрос добавления правильного `ChannelHandler` в `ChannelPipeline`, как показано в листинге 11.14.

**Листинг 11.14. Использование protobuf**
```java
public class ProtoBufInitializer extends ChannelInitializer<Channel> {
    private final MessageLite lite;

    public ProtoBufInitializer(MessageLite lite) {
        this.lite = lite;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        ChannelPipeline pipeline = ch.pipeline();
        
        // (1) Добавляет ProtobufVarint32FrameDecoder для разбивки кадров
        pipeline.addLast(new ProtobufVarint32FrameDecoder());
        
		// (2) Добавляет ProtobufEncoder для кодирования сообщений        
        pipeline.addLast(new ProtobufEncoder()); 
        
		// (3) Добавляет ProtobufDecoder для декодирования сообщений
        pipeline.addLast(new ProtobufDecoder(lite));    
        
		// (4) Добавляет ObjectHandler для обработки 
		// декодированных сообщений
        pipeline.addLast(new ObjectHandler());                         
    }

    public static final class ObjectHandler extends SimpleChannelInboundHandler<Object> {
        @Override
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        Object msg
	    ) throws Exception {
            // Do something with the object
        }
    }
}
```

В этом разделе мы исследовали различные варианты сериализации, поддерживаемые специализированными декодерами и кодировщиками Netty: стандартную сериализацию JDK, JBoss Marshalling и Google Protocol Buffers.
