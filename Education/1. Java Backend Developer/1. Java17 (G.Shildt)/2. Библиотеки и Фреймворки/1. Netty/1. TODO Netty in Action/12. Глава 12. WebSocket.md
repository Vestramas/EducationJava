**В этой главе рассматриваются:**
- Концепция "реального времени" в вебе
    
- Протокол WebSocket
    
- Создание сервера чат-комнаты на основе WebSocket с помощью Netty
    

Если вы следите за последними событиями в веб-технологиях, вы, вероятно, встречали термин "веб реального времени" (real-time web), и если у вас был опыт работы с приложениями реального времени в инженерных областях, вы можете быть немного скептически настроены относительно того, что подразумевает этот термин.

Итак, давайте сразу проясним: речь идёт не о так называемом "жёстком реальном времени" (hard real-time) и качестве обслуживания (QoS), где гарантируется доставка результатов вычислений в течение заданного интервала времени. Уже сама архитектура "запрос-ответ" HTTP делает это крайне проблематичным, о чём свидетельствует тот факт, что ни один из подходов, разработанных в прошлом, не дал удовлетворительного решения.

И хотя велись академические дискуссии о формальном определении семантики временных веб-сервисов, общепринятые определения, похоже, не появятся в ближайшее время. Поэтому пока мы примем следующее неавторитетное описание из Википедии как достаточное:

> Веб реального времени — это сетевая сеть, использующая технологии и практики, которые позволяют пользователям получать информацию сразу после её публикации авторами, а не требовать, чтобы они или их программное обеспечение периодически проверяли источник на наличие обновлений.

Короче говоря, полноценный веб реального времени может быть не за горами, но идея, лежащая в его основе, подпитывает растущие ожидания почти мгновенного доступа к информации. Протокол WebSocket, который мы обсудим в этой главе, является хорошо поддерживаемым шагом в этом направлении.

## 12.1 Знакомство с WebSocket

**Протокол WebSocket** был разработан с нуля для обеспечения практического **решения проблемы двунаправленной передачи данных в вебе**, позволяя клиенту и серверу передавать сообщения в любое время и, следовательно, требуя от них асинхронной обработки получения сообщений. 
> (Большинство современных браузеров поддерживают WebSocket как клиентский API HTML5.)

Поддержка WebSocket в Netty включает все основные используемые реализации, поэтому внедрение его в ваше следующее приложение не составит труда. Как обычно в Netty, вы можете полностью использовать протокол, не беспокоясь о деталях его внутренней реализации.

## 12.3 Добавление поддержки WebSocket

> [!WARNING] Переключение с HTTP на WebSocket
> Механизм, известный как **upgrade handshake** (рукопожатие обновления), ==используется для переключения со стандартного протокола HTTP или HTTPS на WebSocket.== 

Таким образом, *приложение, использующее WebSocket*, **всегда начинается с HTTP/S, а затем выполняет обновление**. Когда именно это происходит, зависит от конкретного приложения; это может быть при запуске или когда запрошен определённый URL.

***Наше приложение принимает следующее соглашение:*** *если запрошенный URL заканчивается на `/ws`, мы обновляем протокол до WebSocket.* В противном случае сервер использует базовый HTTP/S. После того как соединение обновлено, все данные передаются с использованием WebSocket.

### 12.3.1 Обработка HTTP-запросов

Сначала **мы реализуем компонент, который обрабатывает HTTP-запросы.** Этот *компонент будет обслуживать страницу, предоставляющую доступ к чат-комнате, и отображать сообщения*, отправленные подключёнными клиентами. 

В листинге 12.1 приведён код этого `HttpRequestHandler`, который расширяет `SimpleChannelInboundHandler` для сообщений `FullHttpRequest`. Обратите внимание, как реализация `channelRead0()` перенаправляет любые запросы для URI `/ws`.

**Листинг 12.1. HTTPRequestHandler**
```java
// (1) Расширяет SimpleChannelInboundHandler для обработки сообщений FullHttpRequest
public class HttpRequestHandler 
extends SimpleChannelInboundHandler<FullHttpRequest> 
{ 
    private final String wsUri;
    private static final File INDEX;

    static {
        URL location = HttpRequestHandler.class
	        .getProtectionDomain()
	        .getCodeSource()
	        .getLocation();
	    
        try {
            String path = location.toURI() + "index.html";
            path = !path.contains("file:") ? path : path.substring(5);
            INDEX = new File(path);
        } catch (URISyntaxException e) {
            throw new IllegalStateException(
	            "Unable to locate index.html", e);
        }
    }

    public HttpRequestHandler(String wsUri) {
        this.wsUri = wsUri;
    }

    @Override
    public void channelRead0(
		ChannelHandlerContext ctx, 
		FullHttpRequest request
	) throws Exception 
	{
		// (2) Если запрошено обновление WebSocket, 
		// увеличивает счётчик ссылок (retain) 
		// и передаёт следующему ChannelInboundHandler
        if (wsUri.equalsIgnoreCase(request.getUri())) {                   
            ctx.fireChannelRead(request.retain());
            
        } else {
	        // (3) Обрабатывает запросы 100 Continue 
	        // в соответствии с HTTP 1.1
            if (HttpHeaders.is100ContinueExpected(request)) {             
                send100Continue(ctx);
            }
            
            // (4) Читает index.html
            RandomAccessFile file = new RandomAccessFile(INDEX, "r");       
            HttpResponse response = 
	            new DefaultHttpResponse(
		            request.getProtocolVersion(), 
		            HttpResponseStatus.OK
		        );
            response.headers()
	            .set(
		            HttpHeaders.Names.CONTENT_TYPE, 
		            "text/plain; charset=UTF-8"
		        );
		        
            boolean keepAlive = HttpHeaders.isKeepAlive(request);
            
            // (5) Если запрошено keep-alive, добавляет необходимые 
            // заголовки
            if (keepAlive) {                                              
                response.headers()
	                .set(
		                HttpHeaders.Names.CONTENT_LENGTH, 
		                file.length()
		            );
                response.headers()
	                .set(
		                HttpHeaders.Names.CONNECTION,
		                HttpHeaders.Values.KEEP_ALIVE
		            );
            }
            // (6) Записывает HttpResponse клиенту
            ctx.write(response);                                            
            // (7) Если нет шифрования, использует zero-copy через 
            // DefaultFileRegion
            if (ctx.pipeline().get(SslHandler.class) == null) {           
                ctx.write(
	                new DefaultFileRegion(
		                file.getChannel(), 
		                0, 
		                file.length()
		            )
		        );
            } else {
	            // (8) Иначе использует ChunkedNioFile
                ctx.write(new ChunkedNioFile(file.getChannel()));           
            }
            // (9) Записывает и сбрасывает LastHttpContent клиенту
            ChannelFuture future = 
	            ctx.writeAndFlush(LastHttpContent.EMPTY_LAST_CONTENT);
	            
			// (10) Если keep-alive не запрошен, закрывает канал после 
			// завершения записи 
            if (!keepAlive) {                                            
                future.addListener(ChannelFutureListener.CLOSE);
            }
        }
    }

    private static void send100Continue(ChannelHandlerContext ctx) {
        FullHttpResponse response = 
	        new DefaultFullHttpResponse(
		        HttpVersion.HTTP_1_1, 
		        HttpResponseStatus.CONTINUE
		    );
        ctx.writeAndFlush(response);
    }

    @Override
    public void exceptionCaught(
	    ChannelHandlerContext ctx, 
	    Throwable cause
	) throws Exception 
	{
        cause.printStackTrace();
        ctx.close();
    }
}
```

Если HTTP-запрос ссылается на URI `/ws`, `HttpRequestHandler` вызывает `retain()` на `FullHttpRequest` и передаёт его следующему `ChannelInboundHandler` **(2)** с помощью `fireChannelRead(msg)`. 

Вызов `retain()` **необходим, потому что после завершения `channelRead()` будет вызван** `release()` на `FullHttpRequest` для освобождения его ресурсов 
> (пожалуйста, обратитесь к нашему обсуждению `SimpleChannelInboundHandler` в главе 6).

Если клиент отправляет заголовок HTTP 1.1 `Expect: 100-continue`, `HttpRequestHandler` отправляет ответ `100 Continue` **(3)**. `HttpRequestHandler` записывает `HttpResponse` **(6)** обратно клиенту после установки заголовков. Это не `FullHttpResponse`, так как это только первая часть ответа. Также здесь не вызывается `writeAndFlush()`. Это делается в конце.

Если не требуется ни шифрование, ни сжатие, наибольшая эффективность может быть достигнута путём хранения содержимого `index.html` **(4)** в `DefaultFileRegion`. Это будет использовать zero-copy для выполнения передачи. Поэтому вы проверяете, есть ли `SslHandler` в `ChannelPipeline` **(7)**. В противном случае вы используете `ChunkedNioFile` **(8)**.

`HttpRequestHandler` записывает `LastHttpContent` **(9)**, чтобы отметить конец ответа. Если keep-alive не запрошен **(10)**, `HttpRequestHandler` добавляет `ChannelFutureListener` к `ChannelFuture` последней записи и закрывает соединение. Здесь вызывается `writeAndFlush()`, чтобы сбросить все ранее записанные сообщения.

Это первая часть сервера чата, которая управляет чистыми HTTP-запросами и ответами. Далее мы будем обрабатывать фреймы WebSocket, которые передают фактические сообщения чата.

> **ФРЕЙМЫ WEBSOCKET**: WebSocket передают данные в фреймах, каждый из которых представляет часть сообщения. Полное сообщение может состоять из многих фреймов.

### 12.3.2 Обработка фреймов WebSocket

==RFC WebSocket, опубликованный IETF, определяет шесть фреймов==; 

**Netty предоставляет реализацию POJO для каждого из них**. В таблице 12.1 перечислены типы фреймов и описано их использование.

**Таблица 12.1. Типы WebSocketFrame**

|Тип фрейма|Описание|
|---|---|
|`BinaryWebSocketFrame`|Содержит бинарные данные.|
|`TextWebSocketFrame`|Содержит текстовые данные.|
|`ContinuationWebSocketFrame`|Содержит текстовые или бинарные данные, принадлежащие предыдущему `BinaryWebSocketFrame` или `TextWebSocketFrame`.|
|`CloseWebSocketFrame`|Представляет запрос CLOSE и содержит код статуса закрытия и фразу.|
|`PingWebSocketFrame`|Запрашивает передачу `PongWebSocketFrame`.|
|`PongWebSocketFrame`|Отправляется в ответ на `PingWebSocketFrame`.|

Наше чат-приложение будет использовать следующие типы фреймов:
- `CloseWebSocketFrame`
    
- `PingWebSocketFrame`
    
- `PongWebSocketFrame`
    
- `TextWebSocketFrame`

`TextWebSocketFrame` — единственный, который нам действительно нужно обрабатывать. В соответствии с RFC WebSocket, Netty предоставляет `WebSocketServerProtocolHandler` для управления остальными.

Следующий листинг показывает наш `ChannelInboundHandler` для `TextWebSocketFrame`, который также будет отслеживать все активные WebSocket-соединения в своей `ChannelGroup`.

**Листинг 12.2. Обработка текстовых фреймов**
```java
// (1) Расширяет SimpleChannelInboundHandler и обрабатывает сообщения TextWebSocketFrame
public class TextWebSocketFrameHandler
extends SimpleChannelInboundHandler<TextWebSocketFrame> { 
    private final ChannelGroup group;

    public TextWebSocketFrameHandler(ChannelGroup group) {
        this.group = group;
    }

    @Override
    // (2) Переопределяет userEventTriggered() 
    // для обработки пользовательских событий
    public void userEventTriggered(
	    ChannelHandlerContext ctx, 
	    Object evt
	) throws Exception 
	{ 
        if (evt == WebSocketServerProtocolHandler
	        .ServerHandshakeStateEvent.HANDSHAKE_COMPLETE
	    ) {
		    // (3) Если событие указывает на успешное завершение handshake, удаляет HttpRequestHandler из ChannelPipeline, так как дальнейшие HTTP-сообщения не будут получены.
            ctx.pipeline().remove(HttpRequestHandler.class); 
            
			// (4) Уведомляет всех подключённых WebSocket-клиентов о том, что новый клиент подключился   
            group.writeAndFlush(
	            new TextWebSocketFrame(
		            "Client " + ctx.channel() + " joined")
		        ); 
		        
		    // (5) Добавляет новый WebSocket канал в ChannelGroup
            group.add(ctx.channel());                                   
        } else {
            super.userEventTriggered(ctx, evt);
        }
    }

    @Override
    public void channelRead0(
	    ChannelHandlerContext ctx, 
	    TextWebSocketFrame msg
	) throws Exception {
	    // (6) Увеличивает счётчик ссылок сообщения и записывает его всем подключённым клиентам в ChannelGroup
        group.writeAndFlush(msg.retain());                               
    }
}
```

`TextWebSocketFrameHandler` имеет очень небольшой набор обязанностей. Когда рукопожатие WebSocket с новым клиентом успешно завершено **(3)**, он уведомляет всех подключённых клиентов, записывая сообщение во все каналы `ChannelGroup`, а затем добавляет новый канал в `ChannelGroup` **(5)**.

Если получен `TextWebSocketFrame` **(6)**, он вызывает `retain()` на нём и использует `writeAndFlush()` для передачи его в `ChannelGroup`, чтобы все подключённые WebSocket-каналы получили его.

Как и прежде, вызов `retain()` необходим, потому что счётчик ссылок `TextWebSocketFrame` будет уменьшен, когда `channelRead0()` вернёт управление. Поскольку все операции асинхронны, `writeAndFlush()` может завершиться позже, и он не должен обращаться к ссылке, которая стала недействительной.

Поскольку Netty обрабатывает большую часть оставшейся функциональности внутренне, единственное, что осталось сделать сейчас, — это инициализировать `ChannelPipeline` для каждого нового созданного канала. Для этого нам понадобится `ChannelInitializer`.

### 12.3.3 Инициализация ChannelPipeline

Как вы уже узнали, для установки `ChannelHandler` в `ChannelPipeline` вы расширяете `ChannelInitializer` и реализуете `initChannel()`. Следующий листинг показывает код для результирующего `ChatServerInitializer`.

**Листинг 12.3. Инициализация ChannelPipeline**
```java
public class ChatServerInitializer extends ChannelInitializer<Channel> { // (1) Расширяет ChannelInitializer
    private final ChannelGroup group;

    public ChatServerInitializer(ChannelGroup group) {
        this.group = group;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {           // (2) Добавляет все необходимые ChannelHandler в ChannelPipeline
        ChannelPipeline pipeline = ch.pipeline();
        pipeline.addLast(new HttpServerCodec());
        pipeline.addLast(new ChunkedWriteHandler());
        pipeline.addLast(new HttpObjectAggregator(64 * 1024));
        pipeline.addLast(new HttpRequestHandler("/ws"));
        pipeline.addLast(new WebSocketServerProtocolHandler("/ws"));
        pipeline.addLast(new TextWebSocketFrameHandler(group));
    }
}
```

Вызов `initChannel()` настраивает `ChannelPipeline` вновь зарегистрированного канала, устанавливая все необходимые `ChannelHandler`. Они обобщены в таблице 12.2 вместе с их индивидуальными обязанностями.

**Таблица 12.2. ChannelHandler для WebSocket-сервера чата**

|ChannelHandler|Обязанность|
|---|---|
|`HttpServerCodec`|Декодирует байты в `HttpRequest`, `HttpContent` и `LastHttpContent`. Кодирует `HttpRequest`, `HttpContent` и `LastHttpContent` в байты.|
|`ChunkedWriteHandler`|Записывает содержимое файла.|
|`HttpObjectAggregator`|Агрегирует `HttpMessage` и следующие за ним `HttpContent` в один `FullHttpRequest` или `FullHttpResponse` (в зависимости от того, используется ли он для обработки запросов или ответов). С установленным этим обработчиком следующий `ChannelHandler` в конвейере будет получать только полные HTTP-запросы.|
|`HttpRequestHandler`|Обрабатывает `FullHttpRequest` (те, которые не отправлены на URI `/ws`).|
|`WebSocketServerProtocolHandler`|В соответствии со спецификацией WebSocket обрабатывает рукопожатие обновления WebSocket, `PingWebSocketFrame`, `PongWebSocketFrame` и `CloseWebSocketFrame`.|
|`TextWebSocketFrameHandler`|Обрабатывает `TextWebSocketFrame` и события завершения handshake.|

`WebSocketServerProtocolHandler` в Netty обрабатывает все обязательные типы фреймов WebSocket и само рукопожатие обновления. Если рукопожатие успешно, необходимые `ChannelHandler` добавляются в конвейер, а те, которые больше не нужны, удаляются.

### 12.3.4 Бутстраппинг

Последней частью картины является код, который выполняет бутстраппинг сервера и устанавливает `ChatServerInitializer`. Это будет обрабатываться классом `ChatServer`, как показано здесь.

**Листинг 12.4. Бутстраппинг сервера**
```java
public class ChatServer {
	// (1) Создаёт DefaultChannelGroup, которая будет содержать все подключённые WebSocket-каналы
    private final ChannelGroup channelGroup = 
	    new DefaultChannelGroup(ImmediateEventExecutor.INSTANCE); 
	    
    private final EventLoopGroup group = new NioEventLoopGroup();
    private Channel channel;

    public ChannelFuture start(InetSocketAddress address) {
	    // (2) Выполняет бутстраппинг сервера
        ServerBootstrap bootstrap = new ServerBootstrap();  
                      
        bootstrap.group(group)
                .channel(NioServerSocketChannel.class)
                .childHandler(createInitializer(channelGroup));
                
        ChannelFuture future = bootstrap.bind(address);
        future.syncUninterruptibly();
        channel = future.channel();
        return future;
    }
    
	// (3) Создаёт ChatServerInitializer
    protected ChannelInitializer<Channel> createInitializer(
	    ChannelGroup group
	) { 
        return new ChatServerInitializer(group);
    }

    public void destroy() {                                                 // (4) Обрабатывает завершение работы сервера и освобождает все ресурсы
        if (channel != null) {
            channel.close();
        }
        channelGroup.close();
        group.shutdownGracefully();
    }

    public static void main(String[] args) throws Exception {
        if (args.length != 1) {
            System.err.println("Please give port as argument");
            System.exit(1);
        }
        int port = Integer.parseInt(args[0]);
        
        final ChatServer endpoint = new ChatServer();
        
        ChannelFuture future = endpoint.start(
	        new InetSocketAddress(port)
	    );
        Runtime.getRuntime().addShutdownHook(new Thread() {
            @Override
            public void run() {
                endpoint.destroy();
            }
        });
        future.channel().closeFuture().syncUninterruptibly();
    }
}
```

### 12.4.1 А как насчёт шифрования?

В реальном сценарии вас скоро попросят добавить шифрование на этот сервер. С Netty это просто вопрос добавления `SslHandler` в `ChannelPipeline` и его настройки. Следующий листинг показывает, как это можно сделать, расширив наш `ChatServerInitializer` для создания `SecureChatServerInitializer`.

**Листинг 12.6. Добавление шифрования в ChannelPipeline**
```java
public class SecureChatServerInitializer extends ChatServerInitializer { // (1) Расширяет ChatServerInitializer для добавления шифрования
    private final SslContext context;

    public SecureChatServerInitializer(ChannelGroup group, SslContext context) {
        super(group);
        this.context = context;
    }

    @Override
    protected void initChannel(Channel ch) throws Exception {
        super.initChannel(ch);                                          // (2) Вызывает initChannel() родительского класса
        SSLEngine engine = context.newEngine(ch.alloc());
        ch.pipeline().addFirst(new SslHandler(engine));                 // (3) Добавляет SslHandler в ChannelPipeline
    }
}
```

Последний шаг — адаптировать `ChatServer` для использования `SecureChatServerInitializer`, чтобы установить `SslHandler` в конвейер. Это даёт нам `SecureChatServer`, показанный здесь.

**Листинг 12.7. Добавление шифрования в ChatServer**
```java
public class SecureChatServer extends ChatServer {                       // (1) SecureChatServer расширяет ChatServer для поддержки шифрования
    private final SslContext context;

    public SecureChatServer(SslContext context) {
        this.context = context;
    }

    @Override
    protected ChannelInitializer<Channel> createInitializer(ChannelGroup group) {
        return new SecureChatServerInitializer(group, context);         // (2) Возвращает ранее созданный SecureChatServerInitializer для включения шифрования
    }

    public static void main(String[] args) throws Exception {
        if (args.length != 1) {
            System.err.println("Please give port as argument");
            System.exit(1);
        }
        int port = Integer.parseInt(args[0]);
        SelfSignedCertificate cert = new SelfSignedCertificate();
        SslContext context = SslContext.newServerContext(cert.certificate(), cert.privateKey());
        final SecureChatServer endpoint = new SecureChatServer(context);
        ChannelFuture future = endpoint.start(new InetSocketAddress(port));
        Runtime.getRuntime().addShutdownHook(new Thread() {
            @Override
            public void run() {
                endpoint.destroy();
            }
        });
        future.channel().closeFuture().syncUninterruptibly();
    }
}
```

Это всё, что нужно для включения шифрования SSL/TLS всех коммуникаций. Как и прежде, вы можете использовать Apache Maven для запуска приложения. Он также загрузит все необходимые зависимости.
