# 1. Введение

## 1.1 Проблема HTTP
Обычный HTTP построен по принципу **Request → Response**.

Клиент отправляет запрос:
```http
GET /users HTTP/1.1
Host: localhost
```

Сервер отвечает:
```http
HTTP/1.1 200 OK
Content-Type: application/json

[...]
```

После отправки ответа сервер считается выполнившим свою задачу.

Схема:
```
Client                             Server
   | -------- Request -----------> |
   |                               |
   | <-------- Response ---------- |
   |                               |
```

Проблема заключается в том, что сервер **не может самостоятельно отправить данные клиенту**.

Если сервер хочет уведомить клиента о событии:
- новое сообщение в чате
- изменение курса валют
- обновление котировок
- игровой ивент

то ему приходится ждать очередного HTTP-запроса.

---
### 1.1.1 Как решали проблему до появления WebSocket

#### Polling
Клиент постоянно спрашивает сервер:
```
Есть новости?
Есть новости?
Есть новости?
Есть новости?
```

Схема:
```
Client                             Server
   | ---- GET /events -----------> |
   | <--------- none ------------ |
   |
   | ---- GET /events -----------> |
   | <--------- none ------------ |
   |
   | ---- GET /events -----------> |
   | <------- new event --------- |
```

Недостатки:
- огромное количество запросов
- лишняя нагрузка
- высокая задержка

---
#### Long Polling
Сервер удерживает соединение открытым.

```
Client                             Server
   | ---- GET /events -----------> |
   |                               |
   |         waiting...            |
   |                               |
   | <-------- event ------------ |
```

После получения ответа клиент открывает новый запрос.

Минусы:
- постоянное создание HTTP соединений
- лишние заголовки
- неудобная реализация

---
## 1.2 Решение — WebSocket

WebSocket создаёт **постоянное двустороннее соединение**.

После установки соединения:
- клиент может отправлять данные серверу
- сервер может отправлять данные клиенту
- любое направление может инициировать передачу

Схема:
```
		WebSocket Connection

Client <=======================> Server
          Full Duplex
```

Преимущества:
- минимальная задержка
- отсутствие постоянных HTTP-запросов
- двусторонняя связь
- меньший объём служебных данных

---
### 1.2.1 Переход с HTTP на WebSocket

WebSocket **начинается как обычный HTTP.**

Клиент отправляет специальный запрос:
```http
GET /chat HTTP/1.1
Host: localhost:8080

Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Version: 13
Sec-WebSocket-Key: abc123...
```

---
Сервер отвечает:
```http
HTTP/1.1 101 Switching Protocols

Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: ...
```

Код:
```http
101 Switching Protocols
```
говорит:
> С этого момента мы больше не используем HTTP.

---
### 1.2.2 Схема Upgrade

```
Client                                       Server

   | -------- HTTP GET --------------------> |
   |      Upgrade: websocket                 |
   |                                          |
   | <---- 101 Switching Protocols ---------- |
   |                                          |
   |==========================================|
   |                                          |
   |        WebSocket Frames                  |
   |                                          |                                    
```

---
### 1.2.3 Обмен данными после Upgrade

После успешного рукопожатия HTTP исчезает.

Больше нет:
- GET
- POST
- PUT
- DELETE
- HTTP заголовков

Начинается передача WebSocket кадров (Frame).

Схема:
```
Client                                 Server

   | ---- Text Frame ----------------> |
   |                                   |
   | <--- Text Frame ----------------- |
   |                                   |
   | ---- Binary Frame --------------> |
   |                                   |
   | <--- Ping Frame ----------------- |
   |                                   |
   | ---- Pong Frame ----------------> |
```

---
### 1.2.4 Кадр WebSocket

Упрощённо:
```
+----------+----------+-----------+
| Header   | Length   | Payload   |
+----------+----------+-----------+
```

Payload:
- текст
- бинарные данные
- heartbeat
- команды управления

---
# 2. Основные классы для работы WebSocket в Netty
Логически все классы можно разделить на 6 групп:
1. HTTP Upgrade
2. Handshake
3. Protocol Handler
4. Frames
5. Codec Layer
6. Конфигурация WebSocket-протокола

---
## Группа 1. HTTP Upgrade
Эти классы работают только до момента перехода на WebSocket.

Эти классы являются комбинацией декодера и кодера для HTTP-протокола. Они преобразуют входящие байты в объекты HTTP-запросов/ответов и наоборот.

**Устанавливаются первыми** (*или вторыми после SSL-обработчика*), так как он должен обрабатывать сырые байты.

### 1. HttpServerCodec
Комбинация: `HttpRequestDecoder` + `HttpResponseEncoder`

Используется на сервере.
Преобразует: 
- `ByteBuf` в `HttpRequest` (байты в запрос)
- и обратно: `HttpResponse` в `ByteBuf` (ответ в байты)

---
#### Конфигурация
Минимальная:
```java
pipeline.addLast(new HttpServerCodec());
```

Расширенная:
```java
new HttpServerCodec(4096, 8192, 8192);
```

| Параметр             | Назначение                                               |
| -------------------- | -------------------------------------------------------- |
| maxInitialLineLength | лимит длины начальной строки <br>(GET /path HTTP/1.1)    |
| maxHeaderSize        | максимальный суммарный размер всех заголовков            |
| maxChunkSize         | максимальный размер чанка при фрагментированной передаче |

---
#### Место в пайплайне

```
HttpServerCodec      
↓
HttpObjectAggregator      
↓
WebSocket...
```

---
### 2. HttpClientCodec

Комбинация: `HttpRequestEncoder`+`HttpResponseDecoder`

Используется на клиенте.

Преобразует:
- `HttpRequest` → `ByteBuf` (запрос в байты)
- `ByteBuf` → `HttpResponse` (байты в ответ сервера)

---
#### Конфигурация
Минимальная:
```java
pipeline.addLast(new HttpClientCodec());
```

Расширенная:
```java
new HttpClientCodec(4096, 8192,8192);
```

| Параметр             | Назначение                          |
| -------------------- | ----------------------------------- |
| maxInitialLineLength | максимальная длина стартовой строки |
| maxHeaderSize        | максимальный размер заголовков      |
| maxChunkSize         | максимальный размер чанка           |

---
#### Место в пайплайне
```
HttpClientCodec       
↓
HttpObjectAggregator       
↓
WebSocket...
```

---
### 3. HttpObjectAggregator
Этот класс **агрегирует фрагментированные HTTP-сообщения** в полноценные объекты `FullHttpRequest` или `FullHttpResponse`.

Без агрегатора:
```
HttpRequest
HttpContent
HttpContent
LastHttpContent
```

С агрегатором:
```
FullHttpRequest
```

или
```
FullHttpResponse
```

---
#### Конфигурация
```java
// Для сервера
pipeline.addLast("httpAggregator", new HttpObjectAggregator(
    65536     // maxContentLength - максимальный размер содержимого
));

// Расширенная конфигурация
pipeline.addLast("httpAggregator", new HttpObjectAggregator(
    65536,               // maxContentLength
    true                 // closeOnExpectationFailed - закрыть соединение при ошибке ожидания
));
```

| Параметр                 | Назначение                                                         |
| ------------------------ | ------------------------------------------------------------------ |
| maxContentLength         | максимальный размер HTTP-сообщения                                 |
| closeOnExpectationFailed | закрывать ли соединение при ошибке ожидания (Expect: 100-continue) |

---
#### Место в пайплайне
Устанавливается **сразу после кодера/декодера**, так как он обрабатывает уже декодированные HTTP-объекты.

```
HttpServerCodec       
↓
HttpObjectAggregator       
↓
WebSocket...
```

или
```
HttpClientCodec       
↓
HttpObjectAggregator       
↓
WebSocket...
```

---
## Группа 2. Handshake
Это низкоуровневые **классы рукопожатия.**
> Обычно напрямую не используются.

Протокол WebSocket имеет несколько версий:
- Версия 0 (Hixie-76, устаревшая)
- Версия 7 (HyBi-07)
- Версия 8 (HyBi-08)
- Версия 13 (HyBi-13, RFC 6455 — современная)

Каждая версия требует разных алгоритмов вычисления ключа и форматирования заголовков. 

**Использование специализированных классов с постфиксами** (`WebSocketServerHandshaker00`, `WebSocketServerHandshaker08`, `WebSocketServerHandshaker13`) **усложняет код** и требует ручного определения версии.

**Фабрика решает эту проблему

---
### 1. WebSocketServerHandshakerFactory
Фабрика **для создания серверного Handshaker (WebSocketServerHandshaker).**

Самостоятельно определяет версию WebSocket клиента и создаёт соответствующую реализацию.

Используется вместо:
```java
WebSocketServerHandshaker13
WebSocketServerHandshaker08
WebSocketServerHandshaker07
```

---
#### Почему рекомендуется использовать фабрику
Версия протокола приходит в заголовке:
```http
Sec-WebSocket-Version: 13
```

Сервер заранее не знает её значение.

Поэтому фабрика автоматически выбирает нужную реализацию.

---
#### Конфигурация
```java
WebSocketServerHandshakerFactory factory =
    new WebSocketServerHandshakerFactory(
        "ws://localhost:8080/chat",
        null,
        true
    );
```

Создание Handshaker:
```java
WebSocketServerHandshaker handshaker =
    factory.newHandshaker(request);
```

| Параметр              | Назначение                                                 |
| --------------------- | ---------------------------------------------------------- |
| websocketURL          | URL, по которому доступен WebSocket                        |
| subprotocols          | список поддерживаемых подпротоколов (разделенных запятыми) |
| allowExtensions       | разрешать ли WebSocket-расширения                          |
| maxFramePayloadLength | максимальный размер полезной нагрузки фрейма               |

---
#### Место в пайплайне
Обычно **не добавляется в pipeline**.

**Используется внутри пользовательского handler.**
```
HttpServerCodec
       ↓
HttpObjectAggregator
       ↓
CustomHandler
       ↓
HandshakerFactory
```

---
### 2. WebSocketClientHandshakerFactory
Фабрика для **создания клиентского Handshaker (WebSocketClientHandshaker).**

Создаёт ***объект, выполняющий клиентское рукопожатие.***

---
#### Конфигурация
```java
WebSocketClientHandshaker handshaker =
    WebSocketClientHandshakerFactory
        .newHandshaker(
            uri,
            WebSocketVersion.V13,
            null,
            true,
            new DefaultHttpHeaders()
        );
```

|Параметр|Назначение|
|---|---|
|URI|адрес подключения|
|Version|версия WebSocket|
|Subprotocol|субпротокол|
|AllowExtensions|разрешить расширения|
|Headers|дополнительные HTTP-заголовки|

---
#### Место в пайплайне
```
HttpClientCodec
       ↓
HttpObjectAggregator
       ↓
CustomHandler
       ↓
ClientHandshaker
```

---
## Группа 3. Protocol Handler
Эти обработчики предоставляют **высокоуровневый интерфейс для работы с WebSocket** и автоматизируют многие задачи.

На практике именно эти классы используются чаще всего.

Они автоматически выполняют:
- handshake
- close handshake
- ping/pong
- обновление pipeline после Upgrade
- проверку ошибок протокола

---
### 1. WebSocketServerProtocolHandler
Полностью **автоматизирует работу WebSocket на сервере**.
#### Конфигурация
Минимальная:
```java
pipeline.addLast(
    new WebSocketServerProtocolHandler("/chat")
);
```

Расширенная:
```java
pipeline.addLast(
    new WebSocketServerProtocolHandler(
        "/chat",
        null,
        true,
        65536
    )
);
```

|Параметр|Назначение|
|---|---|
|websocketPath|путь подключения|
|subprotocols|список субпротоколов|
|allowExtensions|разрешить расширения|
|maxFramePayloadLength|максимальный размер кадра|

#### Место в пайплайне
```
HttpServerCodec
       ↓
HttpObjectAggregator
       ↓
WebSocketServerProtocolHandler
       ↓
BusinessHandler
```

---
### 2. WebSocketClientProtocolHandler
Полностью **автоматизирует работу WebSocket на клиенте.**

#### Конфигурация
```java
pipeline.addLast(
    new WebSocketClientProtocolHandler(
        config
    )
);
```

Параметры задаются через:
```java
WebSocketClientProtocolConfig
```

---
#### Место в пайплайне
```
HttpClientCodec
       ↓
HttpObjectAggregator
       ↓
WebSocketClientProtocolHandler
       ↓
BusinessHandler
```

---
## Группа 4. Frames
После успешного Upgrade HTTP больше не используется.

**Обмен данными происходит через объекты типа:**
```java
WebSocketFrame
```
> Все сообщения WebSocket являются наследниками этого класса.

```
WebSocketFrame (абстрактный)
├── TextWebSocketFrame - текстовые данные (UTF-8)
├── BinaryWebSocketFrame - бинарные данные
├── PingWebSocketFrame - ping-запрос
├── PongWebSocketFrame - pong-ответ
├── CloseWebSocketFrame - закрытие соединения
├── ContinuationWebSocketFrame - продолжение фрагментированного сообщения
```

---
### 1. TextWebSocketFrame
Используется **для передачи текстовых данных.**

Наиболее распространённый тип сообщений.

#### Конфигурация
Отправка:
```java
ctx.writeAndFlush(
    new TextWebSocketFrame(
        "Hello World"
    )
);
```

Получение:
```java
if (msg instanceof TextWebSocketFrame frame) {
    String text = frame.text();
}
```

---
#### Основные методы

|Метод|Назначение|
|---|---|
|text()|получить строку|
|content()|получить ByteBuf|
|copy()|создать копию кадра|
|retain()|увеличить reference count|

---
#### Место в пайплайне
```
WebSocketServerProtocolHandler
              ↓
         BusinessHandler
              ↓
      TextWebSocketFrame
```

---
#### Когда использовать

Подходит для:
- чатов
- JSON API
- уведомлений
- команд управления

Пример JSON:
```
{
  "type": "message",
  "text": "Hello"
}
```

---
### 2. BinaryWebSocketFrame
Используется **для передачи бинарных данных**.

---
#### Конфигурация

Отправка:
```java
ByteBuf payload = 
ctx.alloc()
	.buffer();

payload.writeBytes(data);

ctx.writeAndFlush(
    new BinaryWebSocketFrame(payload)
);
```

Получение:
```java
if (msg instanceof BinaryWebSocketFrame frame) {

    ByteBuf payload =
        frame.content();
}
```

---
#### Основные методы

|Метод|Назначение|
|---|---|
|content()|получить ByteBuf|
|copy()|копировать кадр|
|retain()|увеличить reference count|

---
#### Место в пайплайне
```
WebSocketServerProtocolHandler
              ↓
         BusinessHandler
              ↓
    BinaryWebSocketFrame
```

---
#### Когда использовать

Подходит для:
- файлов
- изображений
- protobuf
- сериализованных объектов
- собственного бинарного протокола

---
### 3. PingWebSocketFrame
Используется **для проверки доступности второй стороны.**

Является служебным кадром.

---
#### Конфигурация

Отправка:
```java
ctx.writeAndFlush(
    new PingWebSocketFrame()
);
```

С полезной нагрузкой:
```java
ctx.writeAndFlush(
    new PingWebSocketFrame(
        Unpooled.copiedBuffer(
            "ping",
            CharsetUtil.UTF_8
        )
    )
);
```

---
#### Место в пайплайне
```
Client
   |
   | Ping
   |
   V
Server
```

---
#### Когда использовать

Подходит для:
- heartbeat
- проверки соединения
- определения разрыва связи

---
### 4. PongWebSocketFrame
**Ответ на Ping.**

#### Конфигурация
Обычно вручную не создаётся.

Если используется:
```java
ctx.writeAndFlush(
    new PongWebSocketFrame()
);
```

---
#### Особенности
При использовании:
```java
WebSocketServerProtocolHandler
```
или
```java
WebSocketClientProtocolHandler
```

**ответ на Ping формируется автоматически.**

---
### 5. CloseWebSocketFrame
Используется **для корректного закрытия WebSocket-соединения.**

---
#### Конфигурация
Закрытие:
```java
ctx.writeAndFlush(
    new CloseWebSocketFrame()
);
```

С кодом ошибки:
```java
ctx.writeAndFlush(
    new CloseWebSocketFrame(
        1000,
        "Normal closure"
    )
);
```

| Параметр   | Назначение       |
| ---------- | ---------------- |
| statusCode | код закрытия     |
| reasonText | причина закрытия |

---
### 6. ContinuationWebSocketFrame

Используется **при фрагментации больших сообщений.**

---
#### Зачем нужен

Если сообщение слишком большое:
```
100 МБ файл
```
его можно разбить на несколько кадров.

---
#### Схема

```
Text Frame (fragment 1)
       ↓
Continuation Frame
       ↓
Continuation Frame
       ↓
Continuation Frame
```

---
#### Конфигурация

```java
new ContinuationWebSocketFrame(
    true,
    0,
    payload
);
```

---
#### Когда используется
Обычно автоматически.

Большинство приложений никогда не создают его вручную.

---
## Группа 5. Codec Layer
Низкоуровневые кодеры и декодеры WebSocket.
> Большинство разработчиков их никогда не используют напрямую.

Эти классы преобразуют данные между:`ByteBuf` и `WebSocketFrame`

Обычно используются внутренне и редко добавляются вручную.

---
### 1. WebSocketFrameDecoder
Преобразует входящие байты в объекты WebSocket.

```java
new WebSocket13FrameDecoder(
    true,
    true,
    65536
);
```

|Параметр|Назначение|
|---|---|
|expectMaskedFrames|ожидать маскированные кадры|
|allowExtensions|разрешить расширения|
|maxFramePayloadLength|максимальный размер кадра|

---
### 2. WebSocketFrameEncoder
Преобразует объекты WebSocket в набор байтов.

```java
new WebSocket13FrameEncoder(
    false
);
```

|Параметр|Назначение|
|---|---|
|maskPayload|использовать маскирование|
#### Особенности
**На клиенте кадры должны быть замаскированы** согласно RFC6455.

Поэтому обычно используется:
```java
WebSocket13FrameEncoder(true)
```

На сервере:
```java
WebSocket13FrameEncoder(false)
```

---
## Группа 6. Конфигурация WebSocket-протокола
Эти классы позволяют **настраивать поведение высокоуровневых обработчиков.**

*Начиная с Netty 4.1.44, появились конфигурационные классы*, которые предоставляют типобезопасный и читаемый способ конфигурации.

**Преимущества:**
- Удобная билдер-конфигурация
    
- Все параметры в одном месте
    
- Лучшая читаемость кода
    
- Легче поддерживать и изменять

---
### 1. WebSocketServerProtocolConfig
**Конфигурация сервера.**

Используется совместно с:
```java
WebSocketServerProtocolHandler
```

---
#### Конфигурация
```java
WebSocketServerProtocolConfig config =
    WebSocketServerProtocolConfig
        .newBuilder()
        .websocketPath("/chat")
        .allowExtensions(true)
        .checkStartsWith(true)
        .maxFramePayloadLength(65536)
        .build();
```

|Параметр|Описание|Значение по умолчанию|
|---|---|---|
|`websocketPath`|Путь для обработки WebSocket|Обязательный|
|`subprotocols`|Список поддерживаемых подпротоколов|null|
|`allowExtensions`|Разрешить WebSocket-расширения|false|
|`maxFramePayloadLength`|Максимальный размер фрейма|65536|
|`allowMaskMismatch`|Разрешить несоответствие маски|false|
|`checkStartsWith`|Проверять начало пути|true|
|`dropPongFrames`|Отбрасывать Pong-фреймы|false|
|`decodeCloseReason`|Декодировать причину закрытия|true|
|`handshakeTimeoutMillis`|Таймаут рукопожатия|10000|
|`forceCloseTimeoutMillis`|Таймаут принудительного закрытия|3000|
|`allowUtf8Fragments`|Разрешить фрагментированный UTF-8|true|
|`useDefaultExtensions`|Использовать расширения по умолчанию|true|

---
#### Использование
```java
pipeline.addLast(
    new WebSocketServerProtocolHandler(
        config
    )
);
```

---
#### Место в пайплайне
```
HttpServerCodec
       ↓
HttpObjectAggregator
       ↓
WebSocketServerProtocolHandler(config)
       ↓
BusinessHandler
```

---
### 2. WebSocketClientProtocolConfig
**Конфигурация клиента.**

Используется совместно с:
```java
WebSocketClientProtocolHandler
```

---
#### Конфигурация
```java
WebSocketClientProtocolConfig config =
    WebSocketClientProtocolConfig
        .newBuilder()
        .webSocketUri(uri)
        .version(WebSocketVersion.V13)
        .allowExtensions(true)
        .maxFramePayloadLength(65536)
        .build();
```

|Параметр|Описание|Значение по умолчанию|
|---|---|---|
|`webSocketUri`|URL WebSocket-сервера|Обязательный|
|`version`|Версия протокола|Обязательный|
|`subprotocol`|Запрашиваемый подпротокол|null|
|`customHeaders`|Дополнительные заголовки|null|
|`handleCloseFrames`|Автоматически обрабатывать Close|true|
|`absoluteWebSocketUri`|Использовать абсолютный URI|true|

---
#### Использование
```java
pipeline.addLast(
    new WebSocketClientProtocolHandler(
        config
    )
);
```

---
#### Место в пайплайне
```
HttpClientCodec
       ↓
HttpObjectAggregator
       ↓
WebSocketClientProtocolHandler(config)
       ↓
BusinessHandler
```

# Итоговая карта классов в пайплайне

## Сервер

```
       ↓

       ↓
WebSocketServerProtocolHandler
       ↓
────────────────────────────
       ↓
WebSocketFrameDecoder
       ↓
BusinessHandler
       ↓
WebSocketFrameEncoder
```

---
## Клиент
```
HttpClientCodec
       ↓
HttpObjectAggregator
       ↓
WebSocketClientProtocolHandler
       ↓
────────────────────────────
       ↓
WebSocketFrameDecoder
       ↓
BusinessHandler
       ↓
WebSocketFrameEncoder
```

Важно понимать, что после установки `WebSocketServerProtocolHandler` и `WebSocketClientProtocolHandler` **разработчик обычно работает только с объектами ФРЕЙМОВ:**

```java
TextWebSocketFrame
BinaryWebSocketFrame
PingWebSocketFrame
PongWebSocketFrame
CloseWebSocketFrame
```

*а кодеры, декодеры, handshake и обработка служебных кадров* **выполняются Netty автоматически.** 

Это и есть типичная архитектура большинства WebSocket-приложений на Netty.
