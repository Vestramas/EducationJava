## Зачем это нужно: Архитектура реального времени

Традиционный протокол **HTTP** построен по схеме «запрос-ответ», где инициатором всегда выступает клиент. 
Это создает **две проблемы для реального времени:**
1. **Push-уведомления:** Сервер не может сам отправить данные клиенту (новое сообщение в чате, уведомление о цене акций).
    
2. **Overhead:** Постоянные HTTP-опросы (polling) создают огромную нагрузку на сеть и сервер[](https://juejin.cn/post/7258670816433799205)[](https://cloud.tencent.cn/developer/article/2587653?from=15425&frompage=seopage).


**Решение — WebSocket.** Он устанавливает постоянное двунаправленное (Full-Duplex) соединение через фиксированный TCP-канал. После рукопожатия (Upgrade Handshake) сервер и клиент могут посылать данные независимо друг от друга[](https://developer.aliyun.com/article/1287875).

> [!NOTE] Реализация технологии WebSocket в Netty
> В Netty за реализацию этого механизма отвечает `WebSocketServerProtocolHandler`.

## 1. Пошаговая конфигурация сервера

**Любое WebSocket-соединение в Netty начинается как HTTP.** ***Пайплайн настраивается последовательно.***

### Шаг 1: Настройка бутстрапа (EventLoopGroup)
Правильная настройка пулов потоков критична для производительности при большом количестве подключений[](https://www.php.cn/faq/2561081.html):

- **Boss Group:** Нужен всего **1 поток**. Он только принимает новые соединения. Больше потоков здесь — лишние накладные расходы.
    
- **Worker Group:** Отвечает за обработку I/O операций (чтение/запись).
    - _Формула:_ `Math.min(8, CPU cores * 2)`
        
    - Слишком много потоков вызовут частые переключения контекста (Context Switching), слишком мало — не смогут обработать нагрузку.

```java
// Рекомендованная конфигурация для высокой нагрузки
EventLoopGroup bossGroup = new NioEventLoopGroup(1);   // Только на Accept
EventLoopGroup workerGroup = new NioEventLoopGroup(
	Math.min(8, Runtime.getRuntime().availableProcessors() * 2)
);

ServerBootstrap b = new ServerBootstrap();
b.group(bossGroup, workerGroup)
  .channel(NioServerSocketChannel.class)
  .option(ChannelOption.SO_BACKLOG, 128)       // Размер очереди соединений
  .childOption(ChannelOption.SO_KEEPALIVE, true); // Важно для NAT и балансировщиков[citation:10]
```

### Шаг 2: Конфигурация ChannelPipeline (Порядок имеет значение!)

Ниже приведен эталонный пайплайн с комментариями о назначении каждого звена.
```java
.childHandler(new ChannelInitializer<SocketChannel>() {
    @Override
    protected void initChannel(SocketChannel ch) {
        ChannelPipeline p = ch.pipeline();

        // 1. HTTP кодировщик/декодировщик (обязателен для рукопожатия)
        p.addLast(new HttpServerCodec());
        
        // 2. Поддержка чанков (для передачи больших файлов/данных частями)
        p.addLast(new ChunkedWriteHandler());
        
        // 3. Агрегатор HTTP сообщений (собирает фрагментированные запросы в FullHttpRequest)
        //    Размер буфера (64KB) обычно достаточен. Не ставьте слишком большим.
        p.addLast(new HttpObjectAggregator(64 * 1024));
        
        // 4. ЯДРО: Обработчик WebSocket
        // Параметры детально разобраны в таблице ниже.
        p.addLast(
	        new WebSocketServerProtocolHandler(
		        "/ws", 
		        null, 
		        true, 
		        65536, 
		        false)
		    );
        
        // 5. Ваш бизнес-обработчик (работает с TextWebSocketFrame)
        p.addLast(new WebSocketFrameHandler());
    }
});
```

#### Шаг 3: Детальная настройка WebSocketServerProtocolHandler

Этот хендлер выполняет "тяжелую работу" (апгрейд, обработка контрольных фреймов). Его конструктор имеет 5 ключевых параметров[](https://juejin.cn/post/7258670816433799205)[](https://javadoc.io/static/io.netty/netty-all/4.0.36.Final/io/netty/handler/codec/http/websocketx/WebSocketServerProtocolHandler.html)[](https://www.php.cn/faq/2561081.html):

| Параметр                | Обязательность | Назначение                                                     | Рекомендация                                                                                                                               |
| ----------------------- | -------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **`websocketPath`**     | **Да**         | URL-путь для соединения (например, `/chat`).                   | Строгое совпадение по умолчанию. Если нужно принимать `/ws?id=123`, смотрите настройку `checkStartsWith` ниже.                             |
| **`subprotocols`**      | Нет            | Список поддерживаемых подпротоколов (например, `chat-v1`).     | Если клиент запрашивает определенный протокол, он должен быть в этом списке[](https://www.php.cn/faq/2561081.html).                        |
| **`allowExtensions`**   | Нет            | Разрешает расширения WebSocket.                                | Обычно `false` (безопасность/стабильность). Включайте только если точно знаете зачем.                                                      |
| **`maxFrameSize`**      | Нет            | Максимальный размер одного фрейма (байт).                      | **Критично для безопасности!** Защита от OOM через слишком большие сообщения. Рекомендуется `65536` (64KB).                                |
| **`checkStartsWith`**   | Нет            | По умолчанию `false` (строгое совпадение пути).                | `true` = путь должен **начинаться** с указанной строки. Решает проблему с параметрами в URL[](https://juejin.cn/post/7258670816433799205). |
| **`allowMaskMismatch`** | Нет            | Соответствие RFC: данные от клиента ДОЛЖНЫ быть замаскированы. | Обычно `false` (сторогое соблюдение RFC).                                                                                                  |

## 2. Реализация бизнес-логики

**После установки соединения все входящие данные превращаются во фреймы** (`WebSocketFrame`). Чаще всего мы работаем с текстом.

### Типы фреймов (POJO в Netty)
**Вам нужно обрабатывать только** `TextWebSocketFrame` и `BinaryWebSocketFrame`. 

Остальными управляет `WebSocketServerProtocolHandler`[](https://cloud.tencent.cn/developer/article/2587653?from=15425&frompage=seopage)[](https://developer.aliyun.com/article/1287875):
- `BinaryWebSocketFrame` — бинарные данные.
    
- `TextWebSocketFrame` — текст (JSON, XML, plain).
    
- `CloseWebSocketFrame` — закрытие соединения.
    
- `Ping` / `PongWebSocketFrame` — проверка связи (Heartbeat).
    

#### Пример: Обработчик чата (Трансляция сообщений)
```java
public class WebSocketFrameHandler 
extends SimpleChannelInboundHandler<TextWebSocketFrame> {

    // Потокобезопасная коллекция для хранения всех активных каналов
    private static final ChannelGroup channels = 
	    new DefaultChannelGroup(GlobalEventExecutor.INSTANCE);

    @Override
    public void handlerAdded(ChannelHandlerContext ctx) {
        channels.add(ctx.channel());
        
        // Оповещаем всех о новом пользователе
        channels.writeAndFlush(
	        new TextWebSocketFrame(
		        "User " + ctx.channel().id() + " joined!")
		);
    }

    @Override
    protected void channelRead0(
	    ChannelHandlerContext ctx, 
	    TextWebSocketFrame msg
	) {
        // msg.text() содержит полученную строку.
        // Ретрансляция ВСЕМ подключенным клиентам
        channels.writeAndFlush(
	        new TextWebSocketFrame(
		        "[" + ctx.channel().id() + "]: " + msg.text()
		    )
		);
    }

    @Override
    public void handlerRemoved(ChannelHandlerContext ctx) {
        // Канал удаляется из группы автоматически, но можно добавить логику
        channels.writeAndFlush(
	        new TextWebSocketFrame(
		        "User " + ctx.channel().id() + " left. Active: " + 
		        channels.size()));
    }
    
    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}
```

> _Важно: `SimpleChannelInboundHandler` автоматически освобождает ресурсы фрейма после использования `channelRead0`. Если вы используете обычный адаптер, не забывайте про `msg.release()`[](https://developer.aliyun.com/article/1287875)._

## 3. Интеграция с клиентом (JavaScript)

Для стыковки бекенда с фронтендом достаточно нативного WebSocket API.
```java
// URL должен совпадать с websocketPath и хостом сервера
const socket = new WebSocket("ws://localhost:8080/ws");

socket.onopen = function(event) {
    console.log("Connected");
    socket.send("Hello Server!");
};

socket.onmessage = function(event) {
    console.log("Message from server: ", event.data);
};
```

## 4. Дополнительные возможности конфигурации

1. **Обнаружение рукопожатия и проверка авторизации**  
    Вам может понадобиться проверить токен пользователя ДО того, как соединение будет установлено. `WebSocketServerProtocolHandler` генерирует кастомное событие после успешного апгрейда[](https://www.tabnine.com/code/java/methods/com.netflix.zuul.netty.server.push.PushConnectionRegistry$PushConnection/%3Cinit%3E)[](https://developer.aliyun.com/article/871508):
```java
@Override
public void userEventTriggered(ChannelHandlerContext ctx, Object evt) {
    if (evt == WebSocketServerProtocolHandler.
	    ServerHandshakeStateEvent.HANDSHAKE_COMPLETE
	) {
        // Соединение апгрейднуто, но мы можем его закрыть, если проверка не пройдена
        // Достаем параметры из HTTP запроса (который уже в прошлом)
        System.out.println(
	        "Handshake finished. Можно удалить Http handlers");
        
        // Удаляем хендлеры HTTP из пайплайна, так как они больше не нужны
        ctx.pipeline().remove(HttpServerCodec.class);
        
    } else if (
	    evt instanceof WebSocketServerProtocolHandler.HandshakeComplete
	) {
        // Объект HandshakeComplete содержит запрошенный URI и заголовки
        WebSocketServerProtocolHandler.HandshakeComplete handshake = 
	        (WebSocketServerProtocolHandler.HandshakeComplete) evt;
        // Проверка handshake.requestHeaders().get("Authorization")...
    }
    super.userEventTriggered(ctx, evt);
}
```

2. **Отказоустойчивость и Heartbeat**  
RFC предписывает серверу отвечать на Ping, но **не требует** отправлять их самому. Если вам нужно детектировать "мертвые" соединения (например, при обрыве wifi), используйте `IdleStateHandler` в пайплайне[](https://www.php.cn/faq/2561081.html):
```java
// Вместе с WebSocketServerProtocolHandler
p.addLast(new IdleStateHandler(30, 0, 0)); // Читаем таймаут 30 сек
p.addLast(new WebSocketServerProtocolHandler("/ws"));
// ...
// В хендлере:
if (evt instanceof IdleStateEvent) {
    ctx.writeAndFlush(new PingWebSocketFrame()); // Запросить подтверждение
}
```

### Частые ошибки и как их избежать

1. **Забыли про `HttpObjectAggregator`**:
    
    - _Ошибка:_ `IllegalStateException: Message is not a FullHttpRequest`
        
    - _Решение:_ Он собирает куски HTTP запроса, иначе `WebSocketServerProtocolHandler` не увидит заголовки.
        
2. **Неправильный порядок хендлеров**:
    
    - `WebSocketServerProtocolHandler` должен быть **ДО** `HttpObjectAggregator`.
        
3. **Утечка памяти (ByteBuf)**:
    
    - Если вы реализуете `ChannelInboundHandler`, а не `Simple...`, и перехватываете сообщение (не передавая дальше), вы обязаны вызвать `ReferenceCountUtil.release(msg)`, иначе память будет течь[](https://developer.aliyun.com/article/1287875).
        
4. **Игнорирование флага `SO_KEEPALIVE`**[](https://www.php.cn/faq/2561081.html):
    
    - Без него при обрыве кабеля или сне NAT (мобильный интернет) соединение может висеть "мертвым" часами. Всегда включайте `.childOption(ChannelOption.SO_KEEPALIVE, true)`.
