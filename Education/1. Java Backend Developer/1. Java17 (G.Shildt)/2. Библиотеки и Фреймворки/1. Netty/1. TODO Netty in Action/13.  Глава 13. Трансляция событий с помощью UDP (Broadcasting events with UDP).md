## 13.1 Основы UDP

Протоколы, ориентированные на соединение (такие как TCP), управляют установлением соединения между двумя сетевыми конечными точками, упорядочиванием и надёжной доставкой сообщений, отправленных в течение времени жизни соединения, и, наконец, упорядоченным завершением соединения. 

В противоположность этому, в протоколе без установления соединения, таком как UDP, нет понятия долговременного соединения, и каждое сообщение (UDP-дейтаграмма) является независимой передачей.

Более того, в UDP нет механизма исправления ошибок, как в TCP, где каждый узел подтверждает полученные пакеты, а неподтверждённые пакеты повторно отправляются отправителем.

По аналогии, TCP-соединение подобно телефонному разговору, где серия упорядоченных сообщений течёт в обоих направлениях. UDP, наоборот, напоминает опускание пачки открыток в почтовый ящик. Вы не можете знать порядок, в котором они прибудут к месту назначения, или даже все ли они вообще прибудут.

Эти аспекты UDP могут показаться вам серьёзными ограничениями, но они также объясняют, почему он намного быстрее TCP: все накладные расходы на рукопожатие и управление сообщениями были устранены. Очевидно, что UDP хорошо подходит для приложений, которые могут обрабатывать или допускать потерю сообщений, в отличие от тех, которые обрабатывают финансовые транзакции.

## 13.2 Широковещательная рассылка UDP

Все наши предыдущие примеры использовали режим передачи, называемый **unicast** (одноадресная рассылка), определяемый как отправка сообщений одному сетевому получателю, идентифицируемому уникальным адресом. Этот режим поддерживается как протоколами с установкой соединения, так и без неё.

==UDP предоставляет дополнительные режимы передачи== для отправки сообщения нескольким получателям:
- **Multicast (многоадресная рассылка)** — передача определённой группе хостов.
    
- **Broadcast (широковещательная рассылка)** — передача всем хостам в сети (или подсети).

Пример приложения в этой главе продемонстрирует использование широковещательной рассылки UDP путём отправки сообщений, которые могут быть получены всеми хостами в той же сети. 

Для этой цели мы будем использовать специальный "ограниченный широковещательный" или нулевой сетевой адрес **255.255.255.255**. Сообщения, отправленные на этот адрес, предназначены для всех хостов в локальной сети (0.0.0.0) и никогда не пересылаются маршрутизаторами в другие сети.

## 13.3 Пример приложения для UDP

**Наше примерное приложение будет открывать файл и транслировать каждую строку как сообщение на указанный порт через UDP.** Если вы знакомы с UNIX-подобными ОС, вы можете узнать в этом очень упрощённую версию стандартной утилиты syslog. UDP идеально подходит для такого приложения, потому что периодическая потеря строки лог-файла допустима, учитывая, что сам файл хранится в файловой системе. Более того, приложение предоставляет очень ценную возможность эффективно обрабатывать большие объёмы данных.

**А что насчёт получателя?** С помощью UDP broadcast вы можете создать монитор событий для получения лог-сообщений, просто запустив программу-слушатель на указанном порту. Обратите внимание, что эта лёгкость доступа вызывает потенциальную проблему безопасности, что является одной из причин, почему UDP broadcast, как правило, не используется в небезопасных средах. По той же причине маршрутизаторы часто блокируют широковещательные сообщения, ограничивая их сетью, в которой они были созданы.

Передатчик отслеживает появление нового содержимого и, когда оно появляется, передаёт его как широковещательное сообщение через UDP.

## 13.4 POJO сообщения: LogEvent

**В приложениях обмена сообщениями данные часто представлены POJO,** который может содержать конфигурацию или информацию обработки в дополнение к фактическому содержимому сообщения. 

В этом приложении мы будем обрабатывать сообщение как событие, и поскольку данные поступают из лог-файла, мы назовём его `LogEvent`. Листинг 13.1 показывает детали этого простого POJO.

**Листинг 13.1. Сообщение LogEvent**
```java
public final class LogEvent {
    public static final byte SEPARATOR = (byte) ':';  // (1) Разделитель
    private final InetSocketAddress source;
    private final String logfile;
    private final String msg;
    private final long received;

    public LogEvent(String logfile, String msg) {// (2) Конструктор для исходящего сообщения
        this(null, -1, logfile, msg);
    }

    public LogEvent(InetSocketAddress source, long received,// (3) Конструктор для входящего сообщения
                    String logfile, String msg) {
        this.source = source;
        this.logfile = logfile;
        this.msg = msg;
        this.received = received;
    }

    public InetSocketAddress getSource() { // (4) Возвращает InetSocketAddress источника, отправившего LogEvent
        return source;
    }

    public String getLogfile() { // (5) Возвращает имя лог-файла, для которого был отправлен LogEvent
        return logfile;
    }

    public String getMsg() {// (6) Возвращает содержимое сообщения
        return msg;
    }

    public long getReceivedTimestamp() {  // (7) Возвращает время получения LogEvent
        return received;
    }
}
```

## 13.5 Написание широковещательного передатчика (Writing the broadcaster)

==Netty предоставляет ряд классов для поддержки написания UDP-приложений==. 

Основные из них, которые мы будем использовать, — это контейнеры сообщений и типы каналов, перечисленные в таблице 13.1.

**Таблица 13.1. UDP классы Netty, используемые в передатчике**

| Название                                                                                                     | Описание                                                                                                                                   |
| ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `interface AddressedEnvelope<M, A extends SocketAddress> extends ReferenceCounted`                           | **Определяет сообщение, которое оборачивает другое сообщение с адресами отправителя и получателя.** `M` — тип сообщения; `A` — тип адреса. |
| `class DefaultAddressedEnvelope<M, A extends SocketAddress> implements AddressedEnvelope<M, A>`              | Предоставляет реализацию интерфейса `AddressedEnvelope` по умолчанию.                                                                      |
| `class DatagramPacket extends DefaultAddressedEnvelope<ByteBuf, InetSocketAddress> implements ByteBufHolder` | Расширяет `DefaultAddressedEnvelope`, чтобы использовать `ByteBuf` **в качестве контейнера данных сообщения**.                             |
| `interface DatagramChannel extends Channel`                                                                  | *Расширяет абстракцию* `Channel` от Netty **для поддержки управления UDP-мультикаст-группами.**                                            |
| `class NioDatagramChannel extends AbstractNioMessageChannel implements DatagramChannel`                      | Определяет тип канала, который может отправлять и получать сообщения `AddressedEnvelope`.                                                  |

`DatagramPacket` от Netty — это простой контейнер сообщений, используемый реализациями `DatagramChannel` для связи с удалёнными узлами. Как и открытки, на которые мы ссылались в нашей предыдущей аналогии, он содержит адрес получателя (и, опционально, отправителя), а также саму полезную нагрузку сообщения.

Как вы видели, все данные для передачи инкапсулируются в сообщения `LogEvent`. `LogEventBroadcaster` записывает их в канал, отправляя через `ChannelPipeline`, где они преобразуются (кодируются) в сообщения `DatagramPacket`. Наконец, они транслируются через UDP и принимаются удалёнными узлами (мониторами).

Следующий листинг показывает нашу настраиваемую версию `MessageToMessageEncoder`, которая выполняет только что описанное преобразование.

**Листинг 13.2. LogEventEncoder**
```java
public class LogEventEncoder extends MessageToMessageEncoder<LogEvent> {
    private final InetSocketAddress remoteAddress;

	// (1) LogEventEncoder создаёт сообщения DatagramPacket для отправки на указанный InetSocketAddress
    public LogEventEncoder(InetSocketAddress remoteAddress) {          
        this.remoteAddress = remoteAddress;
    }

    @Override
    protected void encode(ChannelHandlerContext ctx, LogEvent logEvent, List<Object> out) throws Exception {
        byte[] file = logEvent.getLogfile().getBytes(CharsetUtil.UTF_8);
        byte[] msg = logEvent.getMsg().getBytes(CharsetUtil.UTF_8);
        
        ByteBuf buf = ctx.alloc().buffer(file.length + msg.length + 1);
        
        buf.writeBytes(file);// (2) Записывает имя файла в ByteBuf
        buf.writeByte(LogEvent.SEPARATOR);  // (3) Добавляет разделитель
        buf.writeBytes(msg); // (4) Записывает сообщение лога в ByteBuf
        out.add(new DatagramPacket(buf, remoteAddress));// (5) Добавляет новый DatagramPacket с данными и адресом назначения в список исходящих сообщений
    }
}
```

С реализованным `LogEventEncoder` мы готовы к бутстраппингу сервера, который включает установку различных `ChannelOptions` и установку необходимых `ChannelHandler` в конвейер. Это будет сделано главным классом `LogEventBroadcaster`, показанным далее.

**Листинг 13.3. LogEventBroadcaster**
```java
public class LogEventBroadcaster {
    private final EventLoopGroup group;
    private final Bootstrap bootstrap;
    private final File file;

    public LogEventBroadcaster(InetSocketAddress address, File file) {
        group = new NioEventLoopGroup();
        bootstrap = new Bootstrap();
        bootstrap.group(group)
                 .channel(NioDatagramChannel.class)                    // (1) Выполняет бутстраппинг NioDatagramChannel (без соединения)
                 .option(ChannelOption.SO_BROADCAST, true)             // (2) Устанавливает опцию сокета SO_BROADCAST
                 .handler(new LogEventEncoder(address));               // (3) Устанавливает LogEventEncoder
        this.file = file;
    }

    public void run() throws Exception {
        Channel ch = bootstrap.bind(0).sync().channel();               // (4) Привязывает канал
        long pointer = 0;
        for (;;) {                                                      // (5) Запускает основной цикл обработки
            long len = file.length();
            if (len < pointer) {
                // файл был сброшен
                pointer = len;                                          // (6) При необходимости устанавливает указатель файла на последний байт файла
            } else if (len > pointer) {
                // Контент был добавлен
                RandomAccessFile raf = new RandomAccessFile(file, "r");
                raf.seek(pointer);                                      // (7) Устанавливает текущий указатель файла, чтобы ничего старого не отправлялось
                String line;
                while ((line = raf.readLine()) != null) {
                    ch.writeAndFlush(new LogEvent(null, -1,             // (8) Для каждой записи лога записывает LogEvent в канал
                            file.getAbsolutePath(), line));
                }
                pointer = raf.getFilePointer();                         // (9) Сохраняет текущую позицию в файле
                raf.close();
            }
            try {
                Thread.sleep(1000);                                     // (10) Спит 1 секунду. Если прерван, выходит из цикла; иначе перезапускает его.
            } catch (InterruptedException e) {
                Thread.interrupted();
                break;
            }
        }
    }

    public void stop() {
        group.shutdownGracefully();
    }

    public static void main(String[] args) throws Exception {
        if (args.length != 2) {
            throw new IllegalArgumentException();
        }
        LogEventBroadcaster broadcaster = new LogEventBroadcaster(     // (11) Создаёт и запускает новый экземпляр LogEventBroadcaster
                new InetSocketAddress("255.255.255.255", Integer.parseInt(args[0])),
                new File(args[1]));
        try {
            broadcaster.run();
        } finally {
            broadcaster.stop();
        }
    }
}
```

## 13.6 Написание монитора (Writing the monitor)

Наша цель — заменить netcat более полноценным потребителем событий, который мы назовём `LogEventMonitor`. Эта программа будет:

1. Получать UDP `DatagramPacket`, транслируемые `LogEventBroadcaster`.
    
2. Декодировать их в сообщения `LogEvent`.
    
3. Записывать сообщения `LogEvent` в `System.out`.
    

Как и прежде, логика будет реализована пользовательскими `ChannelHandler` — для нашего декодера мы расширим `MessageToMessageDecoder`. Рисунок 13.4 изображает `ChannelPipeline` `LogEventMonitor` и показывает, как `LogEvent` будут проходить через него.

![[Pasted image 20260613183913.png]]
**Рисунок 13.4. LogEventMonitor**

Первый декодер в конвейере, `LogEventDecoder`, отвечает за декодирование входящих `DatagramPacket` в сообщения `LogEvent` (типичная настройка для любого приложения Netty, преобразующего входящие данные). Следующий листинг показывает реализацию.

**Листинг 13.6. LogEventDecoder**
```java
public class LogEventDecoder extends MessageToMessageDecoder<DatagramPacket> {
    @Override
    protected void decode(
	    ChannelHandlerContext ctx, 
	    DatagramPacket datagramPacket, 
	    List<Object> out
	) throws Exception 
	{
        ByteBuf data = datagramPacket.data();                            // (1) Получает ссылку на данные в DatagramPacket (ByteBuf)
        
        int idx = data.indexOf(
	        0, 
	        data.readableBytes(), 
	        LogEvent.SEPARATOR
	    ); // (2) Получает индекс разделителя
	    
	    // (3) Извлекает имя файла
        String filename = data.slice(0, idx)                             
                .toString(CharsetUtil.UTF_8);
                
		// (4) Извлекает сообщение лога
        String logMsg = data.slice(idx + 1, data.readableBytes())        
                .toString(CharsetUtil.UTF_8);
        LogEvent event = new LogEvent(
	        datagramPacket.remoteAddress(),    // (5) Создаёт новый объект LogEvent и добавляет его в список
                System.currentTimeMillis(), filename, logMsg);
        out.add(event);
    }
}
```

Задача второго `ChannelHandler` — выполнить некоторую обработку сообщений `LogEvent`, созданных первым. В этом случае он просто записывает их в `System.out`. В реальном приложении вы могли бы агрегировать их с событиями из другого лог-файла или отправлять в базу данных. Этот листинг, показывающий `LogEventHandler`, иллюстрирует основные шаги.

**Листинг 13.7. LogEventHandler**
```java
public class LogEventHandler extends SimpleChannelInboundHandler<LogEvent> { // (1) Расширяет SimpleChannelInboundHandler для обработки сообщений LogEvent
    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) throws Exception {
        cause.printStackTrace();                                         // (2) При исключении печатает трассировку стека и закрывает канал
        ctx.close();
    }

    @Override
    public void channelRead0(ChannelHandlerContext ctx, LogEvent event) throws Exception {
        StringBuilder builder = new StringBuilder();                     // (3) Создаёт StringBuilder и формирует вывод
        builder.append(event.getReceivedTimestamp());
        builder.append(" [");
        builder.append(event.getSource().toString());
        builder.append("] [");
        builder.append(event.getLogfile());
        builder.append("] : ");
        builder.append(event.getMsg());
        System.out.println(builder.toString());                          // (4) Печатает данные LogEvent
    }
}
```

`LogEventHandler` печатает `LogEvent` в удобочитаемом формате, который состоит из:

- Временной метки получения в миллисекундах.
    
- `InetSocketAddress` отправителя (IP-адрес и порт).
    
- Абсолютного имени файла, из которого был сгенерирован `LogEvent`.
    
- Фактического сообщения лога (одна строка в лог-файле).
    

Теперь нам нужно установить наши обработчики в `ChannelPipeline`, как показано на рисунке 13.4. Этот листинг показывает, как это делается главным классом `LogEventMonitor`.

**Листинг 13.8. LogEventMonitor**
```java
public class LogEventMonitor {
    private final EventLoopGroup group;
    private final Bootstrap bootstrap;

    public LogEventMonitor(InetSocketAddress address) {
        group = new NioEventLoopGroup();
        bootstrap = new Bootstrap();
        bootstrap.group(group)                                           // (1) Выполняет бутстраппинг NioDatagramChannel
                .channel(NioDatagramChannel.class)
                .option(ChannelOption.SO_BROADCAST, true)                // (2) Устанавливает опцию сокета SO_BROADCAST
                .handler(new ChannelInitializer<Channel>() {             // (3) Добавляет ChannelHandler в ChannelPipeline
                    @Override
                    protected void initChannel(Channel channel) throws Exception {
                        ChannelPipeline pipeline = channel.pipeline();
                        pipeline.addLast(new LogEventDecoder());
                        pipeline.addLast(new LogEventHandler());
                    }
                })
                .localAddress(address);
    }

    public Channel bind() {
        return bootstrap.bind().sync().channel();                         // (4) Привязывает канал (DatagramChannel не требует соединения)
    }

    public void stop() {
        group.shutdownGracefully();
    }

    public static void main(String[] main) throws Exception {
        if (args.length != 1) {
            throw new IllegalArgumentException("Usage: LogEventMonitor <port>");
        }
        LogEventMonitor monitor = new LogEventMonitor(                   // (5) Создаёт новый LogEventMonitor
                new InetSocketAddress(Integer.parseInt(args[0])));
        try {
            Channel channel = monitor.bind();
            System.out.println("LogEventMonitor running");
            channel.closeFuture().sync();
        } finally {
            monitor.stop();
        }
    }
}
```
