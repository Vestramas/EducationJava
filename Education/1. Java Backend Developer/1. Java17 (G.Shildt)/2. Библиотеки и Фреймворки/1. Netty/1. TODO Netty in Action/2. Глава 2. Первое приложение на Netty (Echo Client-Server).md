## 2.3 Написание Echo-сервера (Writing the Echo server)

==ВСЕ СЕРВЕРЫ Netty ТРЕБУЮТ следующего:==
- **Как минимум один `ChannelHandler`** — этот компонент ***реализует обработку данных, полученных от клиента***, то есть его бизнес-логику.

### 2.3.1 `ChannelHandler` и бизнес-логика

В главе 1 мы представили фьючерсы (Futures) и обратные вызовы (callbacks) и проиллюстрировали их использование в событийно-ориентированном дизайне. Мы также **обсудили** `ChannelHandler` — родителя семейства интерфейсов, **реализации которых получают и реагируют на уведомления о событиях**. 


> [!NOTE] Вся логика работы, описывается в `ChannelHanlder`
> В приложениях Netty вся логика обработки данных содержится в реализациях этих основных абстракций.

Поскольку ваш Echo-сервер должен отвечать на входящие сообщения, ему потребуется реализовать интерфейс `ChannelInboundHandler`, ==который определяет методы ДЛЯ ОБРАБОТКИ ВХОДЯЩИХ событий.== 

Этому простому приложению потребуется лишь несколько из этих методов, поэтому будет достаточно унаследоваться от класса `ChannelInboundHandlerAdapter`, который предоставляет реализацию `ChannelInboundHandler` по умолчанию.

**Нас интересуют следующие методы:**
- **`channelRead()`** — Вызывается для каждого входящего сообщения.
    
- **`channelReadComplete()`** — Уведомляет обработчик о том, что последний вызов `channelRead()` был последним сообщением в текущей партии (batch).
    
- **`exceptionCaught()`** — Вызывается, если во время операции чтения возникло исключение.

Реализация `ChannelHandler` для Echo-сервера — это `EchoServerHandler`, показанный в следующем листинге.

**Листинг 2.1. EchoServerHandler**
```java
@Sharable                                                          // (1)
public class EchoServerHandler extends ChannelInboundHandlerAdapter {

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        ByteBuf in = (ByteBuf) msg;
        System.out.println(
	        "Server received: " + in.toString(CharsetUtil.UTF_8)); // (2)
        ctx.write(in);                                             // (3)
    }

    @Override
    public void channelReadComplete(ChannelHandlerContext ctx) {
        ctx.writeAndFlush(Unpooled.EMPTY_BUFFER)                   // (4)
            .addListener(ChannelFutureListener.CLOSE);
    }

    @Override
    public void exceptionCaught(
	    ChannelHandlerContext ctx, 
	    Throwable cause
	) {
        cause.printStackTrace();                                   // (5)
        ctx.close();                                               // (6)
    }
}
```

**(1)** Указывает, что `ChannelHandler` может быть безопасно разделён (shared) между несколькими каналами.  
**(2)** Логирование полученного сообщения в консоль.  
**(3)** **Запись полученного сообщения отправителю** без сброса (flushing) исходящих сообщений.  
**(4)** Сброс ожидающих сообщений удалённому злу и закрытие канала.  
**(5)** Печать стека исключения.  
**(6)** Закрытие канала.

`ChannelInboundHandlerAdapter` имеет простой API, и каждый его метод может быть переопределён для подключения к жизненному циклу событий в подходящей точке. Поскольку вам нужно обрабатывать все полученные данные, вы переопределяете `channelRead()`. В этом сервере вы просто отправляете данные обратно удалённому узлу (эхо).

Переопределение `exceptionCaught()` позволяет вам реагировать на любые подтипы `Throwable` — здесь вы логируете исключение и закрываете соединение. Более сложное приложение может попытаться восстановиться после исключения, но в этом случае простое закрытие соединения сигнализирует удалённому узлу, что произошла ошибка.

> **Что произойдёт, если исключение не будет перехвачено?**
> 
> Каждый `Channel` имеет связанный с ним `ChannelPipeline`, **который содержит цепочку экземпляров** `ChannelHandler`. 
> 
> По умолчанию ==обработчик передаёт вызов метода следующему обработчику в цепочке. ==
> Следовательно, если `exceptionCaught()` не реализован где-то в цепочке, полученные исключения дойдут до конца `ChannelPipeline` и будут залогированы. 
> 
> По этой причине ваше приложение должно предоставить по крайней мере один `ChannelHandler`, который реализует `exceptionCaught()`.


Помимо `ChannelInboundHandlerAdapter`, существует множество других подтипов и реализаций `ChannelHandler`, которые необходимо изучить, и мы подробно рассмотрим их в главах 6 и 7. 

А пока запомните следующие ключевые моменты:
- `ChannelHandler` вызываются для различных типов событий.
    
- Приложения реализуют или расширяют `ChannelHandler`, чтобы подключиться к жизненному циклу событий и обеспечить пользовательскую бизнес-логику.
    
- Архитектурно `ChannelHandler` помогают отделить вашу бизнес-логику от сетевого кода. Это упрощает разработку по мере развития кода в ответ на меняющиеся требования.

### 2.3.2 Бутстраппинг сервера (Bootstrapping the server)

Обсудив основную бизнес-логику, реализованную в `EchoServerHandler`, теперь мы можем рассмотреть бутстраппинг самого сервера, который включает следующее:

- Привязку к порту, на котором сервер будет прослушивать и принимать входящие запросы на подключение.
    
- Настройку каналов (`Channels`) для уведомления экземпляра `EchoServerHandler` о входящих сообщениях.


> **Транспорты (Transports)**
> 
> В этом разделе вы встретите термин "транспорт". В стандартном многоуровневом представлении сетевых протоколов транспортный уровень — это тот, который предоставляет услуги для сквозной (end-to-end) или межхостовой (host-to-host) связи.
> 
> Интернет-коммуникации основаны на транспорте **TCP**. 
> `NIO transport` **относится к транспорту, который в основном ИДЕНТИЧЕН TCP,** за исключением улучшений производительности на стороне сервера, предоставляемых реализацией Java NIO.
> 
> Транспорты будут подробно рассмотрены в главе 4.

В следующем листинге показан полный код класса `EchoServer`.

**Листинг 2.2. Класс EchoServer**
```java
public class EchoServer {
    private final int port;

    public EchoServer(int port) {
        this.port = port;
    }

    public static void main(String[] args) throws Exception {
        if (args.length != 1) {
            System.err.println(
	            "Usage: " + EchoServer.class.getSimpleName() + " <port>");
            return;
        }
        int port = Integer.parseInt(args[0]);          // (1)
        new EchoServer(port).start();                  // (2)
    }

    public void start() throws Exception {
        final EchoServerHandler serverHandler = new EchoServerHandler();
        
        EventLoopGroup group = new NioEventLoopGroup();        // (3)
        try {
            ServerBootstrap b = new ServerBootstrap();         // (4)
            
            b.group(group)                                    
                .channel(NioServerSocketChannel.class)         // (5)
                .localAddress(new InetSocketAddress(port))     // (6)
                .childHandler(
	                new ChannelInitializer<SocketChannel>() { // (7)
	                    @Override
	                    public void initChannel(SocketChannel ch) 
	                    throws Exception 
	                    {
	                        ch.pipeline().addLast(serverHandler); // (8)
	                    }
	                }
	            );

            ChannelFuture f = b.bind().sync();                 // (9)
            f.channel().closeFuture().sync();                  // (10)
        } finally {
            group.shutdownGracefully().sync();                 // (11)
        }
    }
}
```

**(1)** Установка значения порта (генерирует `NumberFormatException`, если аргумент порта некорректен).  
**(2)** Вызов метода `start()` сервера. 

**(3)** Создание `EventLoopGroup`.  
**(4)** Создание `ServerBootstrap`.  

**(5)** Указание на ==использование NIO транспортного канала==.  
**(6)** Установка адреса сокета с использованием указанного порта.  

**(7)** Добавление `EchoServerHandler` в `ChannelPipeline` канала (объяснено ниже).  
**(8)** `EchoServerHandler` **помечен как** `@Sharable`, **поэтому мы всегда можем использовать один и тот же экземпляр.**  

**(9)** Асинхронная привязка сервера; вызов `sync()` ожидает завершения привязки.  
**(10)** Получение `CloseFuture` канала и блокировка текущего потока до его завершения.  
**(11)** Завершение работы `EventLoopGroup` с освобождением всех ресурсов.

---
ПОСЯНЕНИЕ в книге
В **(4)** вы создаёте экземпляр `ServerBootstrap`. 

Поскольку ==вы используете NIO транспорт, вы указываете== 
- `NioEventLoopGroup` **(3)** для приёма и обработки новых соединений 
- и `NioServerSocketChannel` **(5)** в качестве типа канала. 

После этого вы устанавливает локальный адрес как `InetSocketAddress` с выбранным портом **(6)**. Сервер будет привязан к этому адресу для прослушивания новых запросов на подключение.

В **(7)** и **(8)** вы используете специальный класс `ChannelInitializer`. Это ключевой момент. Когда новое соединение принято, будет создан новый дочерний канал (`child Channel`), и `ChannelInitializer` добавит экземпляр вашего `EchoServerHandler` в `ChannelPipeline` этого канала. Как мы объясняли ранее, этот обработчик будет получать уведомления о входящих сообщениях.


> Хотя NIO масштабируем, его правильная настройка, особенно в части многопоточности, нетривиальна. 
> 
> Архитектура Netty инкапсулирует большую часть сложности, *и мы обсудим соответствующие абстракции* (`EventLoopGroup`, `SocketChannel` и `ChannelInitializer`) более подробно в главе 3.

Затем вы привязываете сервер **(9)** и ждёте завершения привязки (вызов `sync()` блокирует текущий поток до этого момента). В **(10)** приложение будет ждать, пока канал сервера не закроется (поскольку вы вызываете `sync()` на `CloseFuture` канала). Затем вы можете завершить работу `EventLoopGroup` и освободить все ресурсы, включая все созданные потоки **(11)**.

В этом примере **используется NIO**, потому что в настоящее время ==это наиболее широко используемый транспорт благодаря своей масштабируемости и полной асинхронности==. 
> Но можно было бы использовать и другую реализацию транспорта. 
> 
> Если бы вы захотели использовать OIO транспорт в вашем сервере, вы бы указали `OioServerSocketChannel` и `OioEventLoopGroup`. Мы подробнее рассмотрим транспорты в главе 4.

**А пока давайте ПОВТОРИМ важные шаги в реализации сервера, которые вы только что завершили. Это основные компоненты кода сервера:**
- `EchoServerHandler` реализует бизнес-логику.
    
- Метод `main()` выполняет бутстраппинг сервера.


**Следующие шаги требуются при бутстраппинге:**
1. Создайте экземпляр `ServerBootstrap` для бутстраппинга и привязки сервера.
    
2. Создайте и назначьте экземпляр `NioEventLoopGroup` для обработки событий, таких как принятие новых соединений и чтение/запись данных.
    
3. Укажите локальный `InetSocketAddress`, к которому привязывается сервер.
    
4. Инициализируйте каждый новый канал с помощью экземпляра `EchoServerHandler`.
    
5. Вызовите `ServerBootstrap.bind()` для привязки сервера.

## 2.4 Написание Echo-клиента (Writing the Echo client)

**Echo-клиент будет:**
1. Подключаться к серверу.
    
2. Отправлять одно или несколько сообщений.
    
3. Для каждого сообщения ждать и получать то же самое сообщение обратно от сервера.
    
4. Закрывать соединение.
    

Написание клиента включает те же две основные области кода, которые вы видели в сервере: бизнес-логику и бутстраппинг.

---
### 2.4.1 Реализация логики клиента с помощью ChannelHandler

Как и сервер, клиент будет иметь `ChannelInboundHandler` для обработки данных. В этом случае вы расширите класс `SimpleChannelInboundHandler`, чтобы обработать все необходимые задачи, как показано в листинге 2.3. Это

требует переопределения следующих методов:
- **`channelActive()`** — Вызывается после установления соединения с сервером.
    
- **`channelRead0()`** — Вызывается при получении сообщения от сервера.
    
- **`exceptionCaught()`** — Вызывается, если во время обработки возникло исключение.
    

**Листинг 2.3. ChannelHandler для клиента**
```java
@Sharable                                                          // (1)
public class EchoClientHandler extends SimpleChannelInboundHandler<ByteBuf> {

    @Override
    public void channelActive(ChannelHandlerContext ctx) {
        ctx.writeAndFlush(Unpooled.copiedBuffer("Netty rocks!",   // (2)
            CharsetUtil.UTF_8));
    }

    @Override
    public void channelRead0(ChannelHandlerContext ctx, ByteBuf in) {
        System.out.println(
	        "Client received: " + in.toString(CharsetUtil.UTF_8)); // (3)
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();                                   // (4)
        ctx.close();                                               // (5)
    }
}
```

**(1)** Указывает, что этот класс может быть безопасно разделён между каналами.  
**(2)** ==Когда уведомлён о том, что канал активен, отправляет сообщение.==  
**(3)** Логирование полученного сообщения.  
**(4)** При возникновении исключения логирует ошибку.  
**(5)** Закрывает канал.

Сначала вы переопределяете `channelActive()`, **который вызывается после установления соединения**. Это гарантирует, что данные будут записаны на сервер как можно скорее; в данном случае это байтовый буфер, содержащий строку "Netty rocks!".

Затем вы переопределяете метод `channelRead0()`. **Этот метод вызывается всякий раз, когда получены данные**. 

***Обратите внимание, что сообщение, отправленное сервером, может быть получено частями (chunks)***. То есть, если сервер отправляет 5 байт, нет гарантии, что все 5 байт будут получены за один раз. Даже для такого небольшого объёма данных метод `channelRead0()` может быть вызван дважды: первый раз с `ByteBuf` (байтовый контейнер Netty), содержащим 3 байта, а второй — с `ByteBuf`, содержащим 2 байта. 
Как протокол, ориентированный на поток данных (`stream-oriented`), TCP гарантирует, что байты будут получены в том порядке, в котором они были отправлены сервером.

Третий переопределяемый метод — `exceptionCaught()`. Так же как и в `EchoServerHandler` (листинг 2.2), исключение логируется, и канал закрывается, в данном случае завершая соединение с сервером.

> **`SimpleChannelInboundHandler` против `ChannelInboundHandler`**
> 
> Возможно, вы задаётесь вопросом, почему мы использовали `SimpleChannelInboundHandler` в клиенте вместо `ChannelInboundHandlerAdapter`, который использовался в `EchoServerHandler`.
> 
> Это связано с взаимодействием двух факторов: как бизнес-логика обрабатывает сообщения и как Netty управляет ресурсами.
> 
> В клиенте, когда `channelRead0()` завершается, вы получили входящее сообщение и больше оно вам не нужно. Когда метод возвращает управление, `SimpleChannelInboundHandler` **заботится об освобождении ссылки на память, занимаемую `ByteBuf`, который содержит сообщение.**
> 
> В `EchoServerHandler` вам всё ещё нужно отправить входящее сообщение обратно отправителю (эхо), **и операция `write()`, которая является асинхронной, МОЖЕТ НЕ ЗАВЕРШИТСЯ ДО ТОГО, как `channelRead()` вернёт управление** (как показано в листинге 2.1). 
> 
> То есть, когда вызовем метод `write()`, который асинхронно отправит ПРИШЕДШЕЕ СООБЩЕНИЕ клиенту, то есть вероятность, что он не успеет выполнится до завершения метода `channelRead()`, в котором он выполняется. Поэтому нам категорично нельзя освобождать буффер от сообщения.
> 
> По этой причине `EchoServerHandler` расширяет `ChannelInboundHandlerAdapter`, который **НЕ освобождает сообщение в этот момент.**
> 
> Сообщение освобождается в `channelReadComplete()` в `EchoServerHandler`, когда вызывается `writeAndFlush()` (листинг 2.1).
> 
> Управление ресурсами сообщений будет подробно рассмотрено в главах 5 и 6.

### 2.4.2 Бутстраппинг клиента (Bootstrapping the client)

Как вы увидите в следующем листинге, бутстраппинг клиента похож на бутстраппинг сервера, с той разницей, что вместо привязки к порту прослушивания клиент использует параметры хоста и порта для подключения к удалённому адресу (в данном случае адресу Echo-сервера).

**Листинг 2.4. Главный класс клиента**
```java
public class EchoClient {
    private final String host;
    private final int port;

    public EchoClient(String host, int port) {
        this.host = host;
        this.port = port;
    }

    public void start() throws Exception {
        EventLoopGroup group = new NioEventLoopGroup();            // (1)
        try {
            Bootstrap b = new Bootstrap();                         // (2)
            b.group(group)                                         // (3)
                .channel(NioSocketChannel.class)                   // (4)
                .remoteAddress(new InetSocketAddress(host, port))  // (5)
                .handler(new ChannelInitializer<SocketChannel>() { // (6)
                    @Override
                    public void initChannel(SocketChannel ch) throws Exception {
                        ch.pipeline().addLast(
	                        new EchoClientHandler());              // (7)
                    }
                });

            ChannelFuture f = b.connect().sync();                  // (8)
            f.channel().closeFuture().sync();                      // (9)
        } finally {
            group.shutdownGracefully().sync();                     // (10)
        }
    }

    public static void main(String[] args) throws Exception {
        if (args.length != 2) {
            System.err.println(
	            "Usage: " + EchoClient.class.getSimpleName() +
                " <host> <port>");
            return;
        }
        String host = args[0];
        int port = Integer.parseInt(args[1]);
        new EchoClient(host, port).start();
    }
}
```

**(1)** Создание `EventLoopGroup`.  
**(2)** Создание `Bootstrap`.  
**(3)** Указание `EventLoopGroup` для обработки событий клиента; необходима реализация NIO.  
**(4)** Тип канала — для NIO транспорта.  
**(5)** Установка адреса сервера.  
**(6)** Добавление `EchoClientHandler` в конвейер при создании канала.  
**(7)** `EchoClientHandler` добавляется в конвейер.  
**(8)** Подключение к удалённому узлу; ожидание завершения подключения.  
**(9)** Блокировка до тех пор, пока канал не закроется.  
**(10)** Завершение работы пулов потоков и освобождение всех ресурсов.

Как и прежде, используется NIO транспорт. Обратите внимание, что ==вы МОЖЕТЕ ИСПОЛЬЗОВАТЬ РАЗНЫЕ транспорты на клиенте и сервере==; например, NIO транспорт на стороне сервера и OIO транспорт на стороне клиента. В главе 4 мы рассмотрим факторы и сценарии, которые приведут вас к выбору конкретного транспорта для конкретного случая использования.

---
**Давайте ПОВТОРИМ важные моменты, представленные в этом разделе:**
- Экземпляр `Bootstrap` создаётся для инициализации клиента.
    
- Экземпляр `NioEventLoopGroup` ***назначается для обработки событий,*** **что включает создание новых соединений и обработку входящих и исходящих данных.**
    
- `InetSocketAddress` создаётся для подключения к серверу.
    
- `EchoClientHandler` будет установлен в конвейере (`pipeline`), когда соединение будет установлено.
    
- После того как всё настроено, вызывается `Bootstrap.connect()` для подключения к удалённому узлу.

Завершив написание клиента, вы можете приступить к сборке системы и её тестированию.
