**В этой главе рассматриваются:**
- **OIO** — блокирующий транспорт (старый I/O)
    
- **NIO** — асинхронный транспорт
    
- **Local transport** — асинхронные коммуникации внутри одной JVM
    
- **Embedded transport** — тестирование ваших ChannelHandler


Данные, которые текут через сеть, всегда имеют один и тот же тип: байты. То, как эти байты перемещаются, зависит в основном от того, что мы называем **сетевым транспортом** — *концепцией, которая помогает нам абстрагироваться от низкоуровневой механики передачи данных.* Пользователям не важны детали; они просто хотят быть уверенными, что их байты будут надёжно отправлены и получены.

Если у вас есть опыт сетевого программирования на Java, вы, возможно, в какой-то момент обнаружили, что вам нужно поддерживать гораздо больше одновременных соединений, чем ожидалось. **Если затем вы попытаетесь переключиться с блокирующего на неблокирующий транспорт, вы можете столкнуться с проблемами**, потому что два сетевых API совершенно разные.

Однако **Netty предоставляет ОБЩИЙ API поверх всех своих реализаций транспорта**, *делая такое преобразование гораздо более простым, чем при использовании напрямую JDK.* Получившийся код не будет загрязнён деталями реализации, и вам не нужно будет выполнять масштабный рефакторинг всей вашей кодовой базы. Короче говоря, вы можете тратить своё время на продуктивную работу.

В этой главе *мы изучим этот общий API, противопоставляя его JDK, чтобы продемонстрировать его гораздо более высокую простоту использования*. Мы объясним реализации транспортов, которые поставляются в комплекте с Netty, и варианты использования, подходящие для каждого. 
> Имея эту информацию под рукой, вы сможете легко выбрать лучший вариант для вашего приложения.

Единственным prerequisite для этой главы является знание языка программирования Java. Опыт работы с сетевыми фреймворками или сетевым программированием является плюсом, но не обязателен.

Мы начнём с того, как транспорты работают в реальной ситуации.

## 4.1 Практический пример: миграция транспорта (Case study: transport migration)

Мы начнём изучение транспортов с приложения, которое просто принимает соединение, записывает "Hi!" клиенту и закрывает соединение.

### 4.1.1 Использование OIO и NIO БЕЗ Netty

Мы представим блокирующую (OIO) и асинхронную (NIO) версии приложения, которые используют только JDK API. 

Следующий листинг показывает блокирующую реализацию. Если вы когда-либо испытывали радость сетевого программирования с JDK, этот код вызовет приятные воспоминания.

**Листинг 4.1. Блокирующий сетевой код без Netty**
```java
public class PlainOioServer {
    public void serve(int port) throws IOException {
	    // (1) Привязка сервера к указанному порту
        final ServerSocket socket = new ServerSocket(port); 
        
        try {
            for (;;) {
	            // (2) Приём соединения
                final Socket clientSocket = socket.accept();    
                
                System.out.println(
	                "Accepted connection from " + clientSocket);
	                
                // (3) Создание нового потока для обработки соединения
                new Thread(new Runnable() {                    
                    @Override
                    public void run() {
                        OutputStream out;
                        try {
                            out = clientSocket.getOutputStream();
                            
                            // (4) Запись сообщения клиенту
                            out.write(
	                            "Hi!\r\n".getBytes(
		                            Charset.forName("UTF-8")
		                        )
		                    ); 
                            out.flush();
                            
                            // (5) Закрытие соединения
                            clientSocket.close();  
                            
                        } catch (IOException e) {
                            e.printStackTrace();
                        } finally {
                            try {
                                clientSocket.close();
                            } catch (IOException ex) {
                                // ignore on close
                            }
                        }
                    }
                }).start(); // (6) Запуск потока
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

Этот код адекватно обрабатывает умеренное количество одновременных клиентов. Но по мере того как приложение становится популярным, вы замечаете, что *оно не очень хорошо масштабируется до десятков тысяч одновременных входящих соединений.* 

*Вы решаете перейти на асинхронный сетевой ввод-вывод*, ***но вскоре обнаруживаете, что асинхронный API полностью отличается*,** **и теперь вам нужно переписывать ваше приложение.**

Неблокирующая версия показана в следующем листинге.

**Листинг 4.2. Асинхронный сетевой код без Netty**
```java
public class PlainNioServer {
    public void serve(int port) throws IOException {
        ServerSocketChannel serverChannel = ServerSocketChannel.open();
        serverChannel.configureBlocking(false);
        
        ServerSocket ssocket = serverChannel.socket();
        
        InetSocketAddress address = new InetSocketAddress(port);
        
        // (1) Привязка сервера к выбранному порту
        ssocket.bind(address); 
        
		// (2) Открытие Selector'а для обработки каналов 
        Selector selector = Selector.open();                  
        
		// (3) Регистрация ServerSocket с Selector'ом для приёма соединений    
        serverChannel.register(selector, SelectionKey.OP_ACCEPT); 
             
        final ByteBuffer msg = ByteBuffer.wrap("Hi!\r\n".getBytes());
        for (;;) {
            try {
	            // (4) Ожидание новых событий для обработки; блокируется до следующего события
                selector.select();                                     
            } catch (IOException ex) {
                ex.printStackTrace();
                break;
            }
            
            // (5) Получение всех SelectionKey, получивших события
            Set<SelectionKey> readyKeys = selector.selectedKeys(); 
                 
            Iterator<SelectionKey> iterator = readyKeys.iterator();
            while (iterator.hasNext()) {
                SelectionKey key = iterator.next();
                iterator.remove();
                try {
	                // (6) Проверка, является ли событие новым готовым 
	                // к приёму соединением
                    if (key.isAcceptable()) {                           
                        ServerSocketChannel server = 
	                        (ServerSocketChannel) key.channel();
	                    
						// (7) Приём клиента   
                        SocketChannel client = server.accept();         
                        client.configureBlocking(false);
                        
                        client.register(
		                    selector, 
		                    SelectionKey.OP_WRITE | SelectionKey.OP_READ, 
		                    msg.duplicate()
		                ); // (8) Регистрация клиента с Selector'ом
		                
                        System.out.println(
	                        "Accepted connection from " + client);
                    }
                    // (9) Проверка, готов ли сокет к записи данных
                    if (key.isWritable()) {                             
                    
                        SocketChannel client = 
	                        (SocketChannel) key.channel();
	                        
                        ByteBuffer buffer = (ByteBuffer) key.attachment();
                        
                        while (buffer.hasRemaining()) {
	                        // (10) Запись данных клиенту
                            if (client.write(buffer) == 0) {            
                                break;
                            }
                        }
                        // (11) Закрытие соединения
                        client.close();                                 
                    }
                } catch (IOException ex) {
                    key.cancel();
                    try {
                        key.channel().close();
                    } catch (IOException cex) {
                        // ignore on close
                    }
                }
            }
        }
    }
}
```

Как вы видите, хотя этот код делает то же самое, что и предыдущая версия, он совершенно другой. Если для переписывания этого простого приложения под неблокирующий I/O требуется полный пересмотр кода, представьте себе уровень усилий, который потребовался бы для переноса чего-то действительно сложного.

Имея это в виду, давайте **посмотрим, как выглядит приложение, реализованное с помощью Netty.**

### 4.1.2 Использование OIO и NIO с Netty

Мы начнём с написания ещё одной блокирующей версии приложения, на этот раз с использованием фреймворка Netty, как показано в следующем листинге.

**Листинг 4.3. Блокирующий сетевой код с Netty**
```java
public class NettyOioServer {
    public void server(int port) throws Exception {
    
        final ByteBuf buf = Unpooled.unreleasableBuffer(
                Unpooled.copiedBuffer(
	                "Hi!\r\n", 
	                Charset.forName("UTF-8")
	            )
	        );
	    
	    // (1) Использование OioEventLoopGroup 
	    // для блокирующего режима (old I/O)
        EventLoopGroup group = new OioEventLoopGroup();                    
        try {
	        // (2) Создание ServerBootstrap
            ServerBootstrap b = new ServerBootstrap();  
             
            b.group(group)
	             // (3) Использование OioServerSocketChannel
                .channel(OioServerSocketChannel.class)                    
                .localAddress(new InetSocketAddress(port))
                // (4) ChannelInitializer будет вызван для каждого 
                // принятого соединения
                .childHandler(new ChannelInitializer<SocketChannel>() {   
                    @Override
                    public void initChannel(SocketChannel ch) 
                    throws Exception 
	                {
                        ch.pipeline().addLast(
	                        // (5) Добавление ChannelInboundHandlerAdapter
	                        new ChannelInboundHandlerAdapter() 
	                        { 
                            @Override
                            public void channelActive(
	                            ChannelHandlerContext ctx
	                        ) throws Exception 
	                        {
		                        // (6) Запись сообщения клиенту
                                ctx.writeAndFlush(buf.duplicate()) 
                                .addListener(
	                                ChannelFutureListener.CLOSE
	                            ); // и закрытие соединения после записи
                            }
                        });
                    }
                });
            // (7) Привязка сервера к порту
            ChannelFuture f = b.bind().sync();                            
            f.channel().closeFuture().sync();
            
        } finally {
	        // (8) Освобождение всех ресурсов
            group.shutdownGracefully().sync();                            
        }
    }
}
```

Далее мы реализуем ту же логику с неблокирующим I/O, используя Netty.

## 4.2 Транспортный API (Transport API)

В сердце транспортного API находится интерфейс `Channel`, который используется для всех операций ввода-вывода. Иерархия классов `Channel` показана на рисунке 4.1.

```

      │
      └── AbstractChannel
              ▲
              │
              ├── AbstractNioChannel
              │         ▲
              │         │
              │         ├── NioSocketChannel
              │         └── NioServerSocketChannel

              │         │

              └── EmbeddedChannel
```

**Рисунок 4.1. Иерархия интерфейса Channel**

На рисунке показано, что с `Channel` связаны `ChannelPipeline` и `ChannelConfig`. 

`ChannelConfig` **содержит все настройки конфигурации для канала** и *поддерживает "горячие" изменения (hot changes)*. Поскольку конкретный транспорт может иметь уникальные настройки, он может реализовывать подтип `ChannelConfig`. 
> Пожалуйста, обратитесь к Javadoc для реализаций `ChannelConfig`

Поскольку каналы уникальны, объявление `Channel` как подинтерфейса `java.lang.Comparable` предназначено для гарантии упорядоченности. Таким образом, реализация `compareTo()` в `AbstractChannel` выбрасывает `Error`, если два различных экземпляра `Channel` возвращают одинаковый хэш-код.

`ChannelPipeline` *содержит все экземпляры `ChannelHandler`, которые будут применяться к входящим и исходящим данным и событиям.* Эти `ChannelHandler` реализуют логику приложения для обработки изменений состояния и обработки данных.


**Типичные варианты использования `ChannelHandler` включают:**
- Преобразование данных из одного формата в другой
    
- Уведомление об исключениях
    
- Уведомление об активации или деактивации канала
    
- Уведомление о регистрации канала в `EventLoop` или о дерегистрации
    
- Уведомление о пользовательских событиях
    

> **ПЕРЕХВАТЫВАЮЩИЙ ФИЛЬТР (INTERCEPTING FILTER)** : 
> 
> `ChannelPipeline` **реализует** распространённый шаблон проектирования — **"Перехватывающий фильтр"**. 
> 
> UNIX-конвейеры (pipes) — ещё один знакомый пример: команды объединяются в цепочку, где вывод одной команды соединяется с вводом следующей.

Вы также можете изменять `ChannelPipeline` на лету, добавляя или удаляя экземпляры `ChannelHandler` по мере необходимости. Эта возможность Netty может быть использована для создания очень гибких приложений. 

> Например, вы могли бы поддерживать протокол *STARTTLS* по требованию, просто добавляя соответствующий `ChannelHandler` (`SslHandler`) в `ChannelPipeline` всякий раз, когда протокол запрашивается.

В дополнение к доступу к назначенным `ChannelPipeline` и `ChannelConfig`, вы можете использовать методы `Channel`, наиболее важные из которых перечислены в таблице 4.1.

**Таблица 4.1. Методы Channel**

| Название метода   | Описание                                                                                                                                                                                                                                        |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `eventLoop()`     | Возвращает `EventLoop`, назначенный каналу.                                                                                                                                                                                                     |
| `pipeline()`      | Возвращает `ChannelPipeline`, назначенный каналу.                                                                                                                                                                                               |
| `isActive()`      | Возвращает `true`, если канал активен. Значение "активен" может зависеть от нижележащего транспорта. Например, сокетный транспорт активен после подключения к удалённому узлу, тогда как дейтаграммный транспорт был бы активен после открытия. |
| `localAddress()`  | Возвращает локальный `SocketAddress`.                                                                                                                                                                                                           |
| `remoteAddress()` | Возвращает удалённый `SocketAddress`.                                                                                                                                                                                                           |
| `write()`         | Записывает данные удалённому узлу. Данные передаются в `ChannelPipeline` и ставятся в очередь до тех пор, пока не будут сброшены (flushed).                                                                                                     |
| `flush()`         | **Сбрасывает** ранее записанные данн**ые в нижележащий транспорт** (например, в сокет).                                                                                                                                                         |
| `writeAndFlush()` | Удобный метод для вызова `write()`, за которым следует `flush()`.                                                                                                                                                                               |

Позже мы подробно обсудим использование всех этих функций. А пока имейте в виду, что широкий спектр функциональности, предлагаемый Netty, опирается на небольшое количество интерфейсов. Это означает, что вы можете вносить существенные изменения в логику приложения без массового рефакторинга вашей кодовой базы.

Рассмотрим типичную задачу записи данных и их сброса удалённому узлу. Следующий листинг иллюстрирует использование `Channel.writeAndFlush()` для этой цели.

**Листинг 4.5. Запись в Channel**
```java
Channel channel = ...

// (1) Создание ByteBuf, содержащего данные для записи
ByteBuf buf = Unpooled.copiedBuffer("your data", CharsetUtil.UTF_8);
   
// (2) Запись данных и их сброс
ChannelFuture cf = channel.writeAndFlush(buf); 

// (3) Добавление ChannelFutureListener для уведомления о завершении записи
cf.addListener(new ChannelFutureListener() {                           
    @Override
    public void operationComplete(ChannelFuture future) {
	    // (4) Операция записи завершена без ошибок
        if (future.isSuccess()) {                                      
            System.out.println("Write successful");
        
        // (5) Логирование ошибки
        } else { 
            System.err.println("Write error");                         
            future.cause().printStackTrace();
        }
    }
});
```

Реализации `Channel` в Netty потокобезопасны (thread-safe), поэтому вы можете хранить ссылку на `Channel` и использовать её всякий раз, когда вам нужно что-то записать удалённому узлу, даже когда задействовано много потоков. 

Следующий листинг показывает простой пример записи из нескольких потоков. 
> Обратите внимание, что сообщения гарантированно отправляются в порядке очереди.

**Листинг 4.6. Использование Channel из нескольких потоков**
```java
final Channel channel = ...

// (1) Создание ByteBuf, содержащего данные для записи
final ByteBuf buf = Unpooled.copiedBuffer(
	"your data",
    CharsetUtil.UTF_8
).retain();        

// (2) Создание Runnable, который записывает данные в канал 
Runnable writer = new Runnable() {                             
    @Override
    public void run() {
        channel.write(buf.duplicate());
    }
};

// (3) Получение ссылки на пул потоков Executor
Executor executor = Executors.newCachedThreadPool();            

// запись в одном потоке
// (4) Передача задачи записи исполнителю (executor) для выполнения в одном потоке
executor.execute(writer);  

// запись в другом потоке
// (5) Передача другой задачи записи для выполнения в другом потоке
executor.execute(writer);                                       
// ...
```

## 4.3 Встроенные транспорты (Included transports)

**Netty поставляется с несколькими готовыми к использованию транспортами.** 

Поскольку не все из них поддерживают каждый протокол, вы должны выбрать транспорт, совместимый с протоколами, используемыми вашим приложением. В этом разделе мы обсудим эти взаимосвязи.

**Таблица 4.2. Транспорты, предоставляемые Netty**

| Имя          | Пакет                         | Описание                                                                                                                                                                                                     |
| ------------ | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **NIO**      | `io.netty.channel.socket.nio` | Использует пакет `java.nio.channels` как основу — подход на основе селекторов.                                                                                                                               |
| **Epoll**    | `io.netty.channel.epoll`      | Использует JNI для `epoll()` и неблокирующего I/O. <br><br>Этот транспорт поддерживает функции, доступные только на Linux (например, `SO_REUSEPORT`), быстрее, чем NIO транспорт, и полностью неблокирующий. |
| **OIO**      | `io.netty.channel.socket.oio` | Использует пакет `java.net` как основу — использует блокирующие потоки.                                                                                                                                      |
| **Local**    | `io.netty.channel.local`      | Локальный транспорт, *который может использоваться для связи внутри JVM через pipes.*                                                                                                                        |
| **Embedded** | `io.netty.channel.embedded`   | Встроенный транспорт, который позволяет использовать `ChannelHandler` **без реального сетевого транспорта.** <br><br>Это может быть очень полезно для тестирования ваших реализаций `ChannelHandler`.        |

Мы обсудим эти транспорты более подробно в следующих разделах.

### 4.3.1 NIO — неблокирующий I/O
==NIO обеспечивает полностью асинхронную реализацию всех операций ввода-вывода.== 
Он ***использует API на основе селекторов***, доступный с момента появления подсистемы NIO в JDK 1.4.

**Основная концепция, лежащая в основе селектора, заключается в том, чтобы служить реестром, где вы запрашиваете уведомление об изменении состояния канала. **

Возможные изменения состояния включают:
- Новый канал был принят и готов к работе.
    
- Соединение канала было завершено.
    
- У канала есть данные, готовые для чтения.
    
- Канал доступен для записи данных.

**После того как ПРИЛОЖЕНИЕ РЕАГИРУЕТ на изменение состояния, СЕЛЕКТОР СБРАСЫВАЕТСЯ**, и процесс повторяется, работая в потоке, который проверяет изменения и соответствующим образом реагирует на них.

Константы, показанные в таблице 4.3, представляют битовые маски, определённые классом `java.nio.channels.SelectionKey`. 
Эти маски комбинируются для указания набора изменений состояния, о которых приложение запрашивает уведомление.

**Таблица 4.3. Битовые маски операций выбора (Selection operation bit-set)**

| Название     | Описание                                                                                                                                                                                                                             |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `OP_ACCEPT`  | Запрашивает уведомление, когда новое соединение принято и создан канал.                                                                                                                                                              |
| `OP_CONNECT` | Запрашивает уведомление, когда соединение установлено.                                                                                                                                                                               |
| `OP_READ`    | Запрашивает уведомление, когда данные готовы для чтения из канала.                                                                                                                                                                   |
| `OP_WRITE`   | Запрашивает уведомление, когда можно записать больше данных в канал. Это обрабатывает случаи, когда буфер сокета полностью заполнен, что обычно происходит, когда данные передаются быстрее, чем удалённый узел может их обработать. |

### 4.3.2 Epoll — нативный неблокирующий транспорт для Linux

Как мы объясняли ранее, NIO транспорт Netty основан на общей абстракции для асинхронного/неблокирующего сетевого взаимодействия, предоставляемой Java. 
Хотя это гарантирует, что неблокирующий API Netty будет usable на любой платформе, это также влечёт за собой ограничения, поскольку JDK вынужден идти на компромиссы, чтобы предоставить одинаковые возможности на всех системах.

Растущая важность Linux как платформы для высокопроизводительных сетей привела к разработке ряда расширенных функций, включая **epoll** — высокомасштабируемую функцию уведомлений о событиях I/O. 
Этот API, доступный с версии 2.5.44 (2002) ядра Linux, обеспечивает лучшую производительность, чем старые системные вызовы POSIX `select` и `poll`, и в настоящее время является де-факто стандартом для неблокирующих сетей на Linux. JDK NIO API для Linux использует эти вызовы epoll.

Netty предоставляет NIO API для Linux, который использует epoll способом, более соответствующим собственному дизайну Netty и менее затратным с точки зрения использования прерываний. Рассмотрите возможность использования этой версии, если ваши приложения предназначены для Linux; вы обнаружите, что производительность под высокой нагрузкой превосходит производительность реализации NIO от JDK.

Семантика этого транспорта идентична показанной на рисунке 4.2, и его использование просто. Для примера обратитесь к листингу 4.4. Чтобы заменить в этом листинге NIO на epoll, замените `NioEventLoopGroup` на `EpollEventLoopGroup`, а `NioServerSocketChannel.class` — на `EpollServerSocketChannel.class`.

### 4.3.3 OIO — старый блокирующий I/O (old blocking I/O)

Реализация транспорта OIO в Netty представляет собой компромисс: он доступен через общий транспортный API, но, поскольку он построен на блокирующей реализации `java.net`, он **не является асинхронным**. Тем не менее, он очень хорошо подходит для определённых случаев использования.

Например, вам может понадобиться портировать устаревший код, который использует библиотеки, совершающие блокирующие вызовы (например, JDBC[^4]), и может быть непрактично преобразовывать логику в неблокирующую. Вместо этого вы могли бы использовать OIO транспорт Netty в краткосрочной перспективе, а позже портировать ваш код на один из чисто асинхронных транспортов. Давайте посмотрим, как это работает.


**В API `java.net` у вас обычно есть один поток, который принимает новые соединения**, *поступающие на слушающий* `ServerSocket`. 

*Новый сокет создаётся для взаимодействия с узлом*, и для обработки трафика выделяется новый поток. 
*Это необходимо, потому что любая операция I/O на конкретном сокете может заблокироваться в любой момент*. Обработка нескольких сокетов одним потоком может легко привести к тому, что блокирующая операция на одном сокете свяжет все остальные.

![[Pasted image 20260611185453.png]]
**Рисунок 4.3. Логика обработки OIO**

_(Схема: Поток, выделенный для Socket. Выполняется read(). Операция чтения (может заблокироваться). Если чтение завершено успешно — обрабатываются читаемые байты. Если читать нечего — поток пытается прочитать снова. Также обрабатываются другие задачи, поданные для этого сокета)_

*Учитывая это, вы можете задаться вопросом, как Netty может поддерживать NIO с тем же API, что и для асинхронных транспортов.* 

Ответ заключается в том, что **Netty использует флаг сокета `SO_TIMEOUT`, который указывает максимальное количество миллисекунд ожидания завершения операции I/O**. 

Если операция не завершается в течение указанного интервала, выбрасывается `SocketTimeoutException`. Netty перехватывает это исключение и продолжает цикл обработки. 

При следующем запуске `EventLoop` он попытается снова. Это единственный способ, которым асинхронный фреймворк, такой как Netty, может поддерживать OIO. Рисунок 4.3 иллюстрирует эту логику.

### 4.3.4 Локальный транспорт (Local transport) для связи внутри JVM

Netty **предоставляет локальный транспорт** *для асинхронной связи* **между клиентами и серверами, работающими в одной JVM**. Опять же, этот транспорт поддерживает API, общий для всех реализаций транспорта Netty.

В этом транспорте `SocketAddress`, связанный с серверным каналом, привязан не к физическому сетевому адресу; скорее, он хранится в реестре до тех пор, пока работает сервер, и дерегистрируется, когда канал закрывается. 

Поскольку этот транспорт не принимает реальный сетевой трафик, он не может взаимодействовать с другими реализациями транспорта. 
Следовательно, клиент, желающий подключиться к серверу (в той же JVM), использующему этот транспорт, также должен использовать его. Помимо этого ограничения, его использование идентично использованию других транспортов.

---
### 4.3.5 Встроенный транспорт (Embedded transport)

Netty предоставляет дополнительный транспорт, который позволяет встраивать `ChannelHandler` в качестве вспомогательных классов внутрь других `ChannelHandler`. Таким образом, вы можете расширить функциональность `ChannelHandler`, не изменяя его внутренний код.

Ключом к этому встроенному транспорту является конкретная реализация `Channel` под названием, как несложно догадаться, `EmbeddedChannel`. В главе 9 мы подробно обсудим, как использовать этот класс для создания модульных тестов для реализаций `ChannelHandler`.

Как упоминалось ранее, ==не все транспорты поддерживают все основные протоколы==, что может ограничивать ваш выбор. Таблица 4.4 показывает матрицу поддерживаемых транспортов и протоколов на момент публикации.

**Таблица 4.4. Поддержка транспортами сетевых протоколов**
![[Pasted image 20260611190210.png]]

* _См. объяснение протокола SCTP (Stream Control Transmission Protocol) в RFC 2960._


**Включение SCTP на Linux**
SCTP требует поддержки ядра, а также установки пользовательских библиотек.

Например, для Ubuntu вы бы использовали следующую команду:
```bash
sudo apt-get install libsctp1
```

Хотя только SCTP имеет эти специфические требования, другие транспорты могут иметь свои собственные параметры конфигурации, которые следует учитывать. Кроме того, серверная платформа, вероятно, должна быть настроена иначе, чем клиентская, если только для поддержки большего количества одновременных соединений.

**Вот варианты использования, с которыми вы, скорее всего, столкнётесь:**

- **Неблокирующая кодовая база (Non-blocking code base)** — 
  *Если* в вашей кодовой базе *нет блокирующих вызовов (или вы можете их ограничить)*, **всегда хорошей идеей является использование NIO** или epoll (если вы на Linux). Хотя NIO/epoll предназначен для обработки многих одновременных соединений, он также хорошо работает с небольшим их количеством, особенно учитывая способ разделения потоков между соединениями.
    
- **Блокирующая кодовая база (Blocking code base)** — Как мы уже отмечали, если ваша кодовая база сильно зависит от блокирующего I/O и ваши приложения имеют соответствующую архитектуру, вы, вероятно, столкнётесь с проблемами блокирующих операций, если попытаетесь напрямую перейти на NIO транспорт Netty. Вместо того чтобы переписывать ваш код для этого, рассмотрите поэтапную миграцию: начните с OIO и переходите на NIO (или epoll, если вы на Linux), когда вы пересмотрите свой код.
    
- **Связь внутри одной JVM** — Связь внутри одной JVM без необходимости предоставлять сервис через сеть представляет собой идеальный вариант использования для локального транспорта (local transport). Это устранит все накладные расходы реальных сетевых операций, всё ещё используя вашу кодовую базу Netty. Если возникнет необходимость предоставить сервис через сеть, вы просто замените транспорт на NIO или OIO.
    
- **Тестирование ваших реализаций ChannelHandler** — Если вы хотите писать модульные тесты для ваших реализаций `ChannelHandler`, рассмотрите возможность использования встроенного транспорта (embedded transport). Это позволит легко тестировать ваш код без необходимости создавать множество mock-объектов. Ваши классы всё равно будут соответствовать общему API и потоку событий, гарантируя, что `ChannelHandler` будет правильно работать с реальными транспортами. Вы найдёте больше информации о тестировании `ChannelHandler` в главе 9.


**Таблица 4.5. Оптимальный транспорт для приложения**

|Потребности приложения|Рекомендуемый транспорт|
|---|---|
|Неблокирующая кодовая база или общая начальная точка|NIO (или epoll на Linux)|
|Блокирующая кодовая база|OIO|
|Связь внутри одной JVM|Local|
|Тестирование реализаций ChannelHandler|Embedded|
