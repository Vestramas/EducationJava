**В этой главе рассматриваются:**
- **ByteBuf** — контейнер данных Netty
    
- Детали API
    
- Варианты использования
    
- Выделение памяти
    

Как мы отмечали ранее, **фундаментальной единицей сетевых данных всегда является байт**. 
**Java NIO предоставляет `ByteBuffer` в качестве своего байтового контейнера**, но этот класс делает использование излишне сложным и несколько громоздким.

> [!NOTE] `ByteBuf` - аналог ByteBuffer из NIO
> Альтернатива Netty для `ByteBuffer` — это `ByteBuf`, мощная реализация, которая устраняет ограничения JDK API и предоставляет лучший API для разработчиков сетевых приложений.

В этой главе мы проиллюстрируем превосходную функциональность и гибкость `ByteBuf` по сравнению с JDK `ByteBuffer`. Это также даст вам лучшее понимание подхода Netty к обработке данных в целом и подготовит вас к нашему обсуждению `ChannelPipeline` и `ChannelHandler` в главе 6.

## 5.1 API ByteBuf

**API Netty для обработки данных** предоставляется **через два компонента:** 
- абстрактный класс `ByteBuf` 
- и интерфейс `ByteBufHolder`.


**Вот некоторые из преимуществ API ByteBuf:**
- Он расширяем для пользовательских типов буферов.
    
- *Прозрачное нулевое копирование (zero-copy) достигается с помощью встроенного составного типа буфера.*
    
- *Ёмкость расширяется по требованию* (как в JDK `StringBuilder`).
    
- **Переключение между режимами чтения и записи не требует вызова метода** `flip()` класса `ByteBuffer`.
    
- *Чтение и запись используют разные индексы.*
    
- Поддерживается "цепочка вызовов" (method chaining).
    
- Поддерживается подсчёт ссылок (reference counting).
    
- Поддерживается пулинг (pooling).
    

Доступны другие классы для управления выделением экземпляров `ByteBuf` и для выполнения различных операций над контейнером и хранящимися в нём данными. Мы рассмотрим эти функции по мере детального изучения `ByteBuf` и `ByteBufHolder`.

## 5.2 Класс ByteBuf — контейнер данных Netty

Давайте начнём с рассмотрения того, как `ByteBuf` *использует индексы для упрощения доступа к содержащимся в нём данным*.

### 5.2.1 Как это работает

`ByteBuf` поддерживает ==два различных индекса==: 
- один для чтения 
- и один для записи. 

**Когда вы читаете из `ByteBuf`, его `readerIndex` увеличивается** на количество прочитанных байтов. 
***Аналогично***, **когда вы пишете в `ByteBuf`, его `writerIndex` увеличивается.** 

Рисунок 5.1 показывает структуру и состояние пустого `ByteBuf`.

![[Pasted image 20260611191628.png]]
**Рисунок 5.1. ByteBuf ёмкостью 16 байт с индексами, установленными в 0**

Когда **`readIndex` дойдет до `writeIndex`, при чтении, то при попытке прочитать дальше получим исключение** `IndexOutOfBoundsException`, точно так же, как при попытке доступа к данным за пределами массива.

> [!NOTE] Какие методы увеличивают индекс
> Методы `ByteBuf`, **чьи имена начинаются с `read` или `write`, увеличивают соответствующий индекс**
> 
> Операции, начинающиеся с `set` и `get`, **не делают этого**. Последние методы работают с относительным индексом, который передаётся в качестве аргумента методу.


Максимальная ёмкость (`maxCapacity`) `ByteBuf` может быть указана, и попытка переместить индекс записи за это значение вызовет исключение (лимит по умолчанию — `Integer.MAX_VALUE`).

При работе с Netty вы столкнётесь с несколькими распространёнными шаблонами использования, построенными вокруг `ByteBuf`. Когда мы будем их рассматривать, полезно держать в уме рисунок 5.1 — массив байтов с отдельными индексами для управления доступом на чтение и запись.

(подробнее про то, как их получить, смотри раздел про Пуллинг [[5.1 -- ButeBufHolder, Пуллинг, Утилитарные классы, и подсчет ссылок]])

#### HEAP BUFFERS (КУЧНЫЕ БУФЕРЫ)

Наиболее часто используемый шаблон `ByteBuf` ==хранит данные в куче (heap space) JVM. ==

Называемый _резервным массивом (backing array)_, этот шаблон **обеспечивает быстрое выделение и освобождение в ситуациях, когда пулинг не используется.** 

Этот подход, показанный в листинге 5.1, хорошо подходит для случаев, когда вам приходится работать с устаревшими данными.

**Листинг 5.1. Резервный массив (Backing array)**
```java
ByteBuf heapBuf = ...;

// (1) Проверяет, есть ли у ByteBuf РЕЗЕРВНЫЙ МАССИВ
if (heapBuf.hasArray()) {  
	// (2) Если да, получает ссылку на массив
    byte[] array = heapBuf.array();         
    
	// (3) Вычисляет смещение первого байта     
    int offset = heapBuf.arrayOffset() + heapBuf.readerIndex(); 
    
    // (4) Получает количество читаемых байтов
    int length = heapBuf.readableBytes();    
    
	// (5) Вызывает ваш метод с массивом, смещением и длиной          
    handleArray(array, offset, length);                
}
```

> **ПРИМЕЧАНИЕ:** Попытка доступа к резервному массиву, когда `hasArray()` возвращает `false`, вызовет `UnsupportedOperationException`. 
> 
> Этот шаблон похож на использование JDK `ByteBuffer`.

---
#### DIRECT BUFFERS (ПРЯМЫЕ БУФЕРЫ)

Прямой буфер — это другой шаблон `ByteBuf`. Можно ожидать, что память, выделяемая для создания объектов, всегда будет из кучи, но это не обязательно — класс `ByteBuffer`, представленный в JDK 1.4 с NIO, позволяет реализации JVM *выделять память через нативные вызовы.* 

Это делается для того, чтобы избежать копирования содержимого буфера в промежуточный буфер (или из него) до (или после) каждого вызова нативной операции ввода-вывода.

В Javadoc для `ByteBuffer` прямо сказано: 
"**Содержимое прямых буферов будет находиться вне обычной кучи, управляемой сборщиком мусора**". 
> Это объясняет, почему ==прямые буферы идеально подходят для передачи сетевых данных.== 
> 
> *Если бы ваши данные содержались в буфере, выделенном в куче, JVM фактически скопировала бы ваш буфер во внутренний прямой буфер перед отправкой через сокет.*

Основным **недостатком прямых буферов является то, что их выделение и освобождение несколько дороже**, чем у кучных буферов. Вы также можете столкнуться с другим недостатком, если работаете с устаревшим кодом: поскольку данные находятся не в куче, вам, возможно, придётся сделать копию, как показано далее.

**Листинг 5.2. Доступ к данным прямого буфера**
```java
ByteBuf directBuf = ...;

// (1) Проверяет, не имеет ли ByteBuf резервного массива (значит, это прямой буфер)
if (!directBuf.hasArray()) {       
	// (2) Получает количество читаемых байтов                  
    int length = directBuf.readableBytes();   
    
	// (3) Выделяет новый массив для хранения байтов  
    byte[] array = new byte[length];         
    
    // (4) Копирует байты в массив
    directBuf.getBytes(directBuf.readerIndex(), array);
    
    // (5) Вызывает метод с массивом, смещением и длиной
    handleArray(array, 0, length);                     
}
```

Очевидно, что это требует немного больше работы, чем использование резервного массива, поэтому, если вы заранее знаете, что к данным в контейнере будут обращаться как к массиву, вы можете предпочесть использование кучной памяти.

---
#### COMPOSITE BUFFERS (СОСТАВНЫЕ БУФЕРЫ)

Третий и последний шаблон использует _составной буфер (composite buffer)_, который **представляет агрегированное представление нескольких** `ByteBuf`. Здесь вы можете добавлять и удалять экземпляры `ByteBuf` по мере необходимости — функциональность, полностью отсутствующая в реализации `ByteBuffer` от JDK.

**Netty реализует этот шаблон с помощью подкласса** `ByteBuf` — `CompositeByteBuf`, ***который обеспечивает виртуальное представление нескольких буферов как одного объединённого буфера.***

> **ПРЕДУПРЕЖДЕНИЕ:** Экземпляры `ByteBuf` в `CompositeByteBuf` могут включать как прямые, так и не-прямые выделения. 
> 
> Если есть только один экземпляр, вызов `hasArray()` на `CompositeByteBuf` вернёт значение `hasArray()` этого компонента; в противном случае он вернёт `false`.

##### ПРИМЕР - Сообщение из заголовка и тела

Чтобы проиллюстрировать, давайте рассмотрим сообщение, состоящее из двух частей — заголовка (header) и тела (body), которое должно быть передано через HTTP. 

**Две части создаются разными модулями приложения и собираются вместе при отправке сообщения.** У приложения есть возможность повторно использовать одно и то же тело сообщения для нескольких сообщений. Когда это происходит, для каждого сообщения создаётся новый заголовок.

Поскольку мы не хотим перевыделять оба буфера для каждого сообщения, `CompositeByteBuf` идеально подходит; он устраняет ненужное копирование, предоставляя общий API `ByteBuf`. Рисунок 5.2 показывает структуру результирующего сообщения.

![[Pasted image 20260611194231.png]]
**Рисунок 5.2. CompositeByteBuf, содержащий заголовок и тело**


Следующий листинг показывает, как эта задача была бы реализована с использованием JDK `ByteBuffer`. 
Создаётся массив из двух `ByteBuffer` для хранения компонентов сообщения, и третий создаётся для хранения копии всех данных.

**Листинг 5.3. Шаблон составного буфера с использованием ByteBuffer**
```java
// Использование массива для хранения частей сообщения
ByteBuffer[] message = new ByteBuffer[] { header, body };

// Создание нового ByteBuffer и объединение заголовка и тела с помощью копирования
ByteBuffer message2 = ByteBuffer.allocate(
	header.remaining() + body.remaining()
);
message2.put(header);
message2.put(body);
message2.flip();
```

***Операции выделения и копирования, а также необходимость управления массивом делают*** **эту версию неэффективной и неудобной.** 

Следующий листинг **показывает версию с использованием** `CompositeByteBuf`.

**Листинг 5.4. Шаблон составного буфера с использованием CompositeByteBuf**
```java
CompositeByteBuf messageBuf = Unpooled.compositeBuffer();

ByteBuf headerBuf = ...; // может быть кучным или прямым
ByteBuf bodyBuf = ...;   // может быть кучным или прямым

// (1) Добавляет экземпляры ByteBuf в CompositeByteBuf
messageBuf.addComponents(headerBuf, bodyBuf);              
// .....

// (2) Удаляет ByteBuf по индексу 0 (первый компонент)
messageBuf.removeComponent(0); // удаление заголовка  
     
// (3) Перебирает все экземпляры ByteBuf
for (ByteBuf buf : messageBuf) {                           
    System.out.println(buf.toString());
}
```

`CompositeByteBuf` может не разрешать доступ к резервному массиву, поэтому доступ к данным в `CompositeByteBuf` напоминает шаблон прямого буфера, как показано далее.

**Листинг 5.5. Доступ к данным в CompositeByteBuf**
```java
CompositeByteBuf compBuf = Unpooled.compositeBuffer();

// (1) Получает количество читаемых байтов
int length = compBuf.readableBytes();       

// (2) Выделяет новый массив для хранения байтов                
byte[] array = new byte[length];           

// (3) Читает байты в массив
compBuf.getBytes(compBuf.readerIndex(), array);  

// (4) Использует массив с параметрами смещения и длины
handleArray(array, 0, array.length);                       
```

Обратите внимание, что **Netty оптимизирует операции сокетного ввода-вывода, использующие** `CompositeByteBuf`, **устраняя, когда это возможно, штрафы по производительности и использованию памяти**, которые возникают при реализации буфера в JDK. 

Эта оптимизация происходит во внутреннем коде Netty и поэтому не раскрывается, но вы должны знать о её влиянии.

> **API COMPOSITEBYTEBUF:** Помимо методов, унаследованных от `ByteBuf`, `CompositeByteBuf` предлагает множество дополнительных функций. Полный список API смотрите в Javadoc Netty.

## 5.3 Побайтовые операции (Byte-level operations)

`ByteBuf` предоставляет **множество методов помимо базовых операций чтения и записи для изменения своих данных.** 

В следующих разделах мы обсудим наиболее важные из них.

---
### 5.3.1 Произвольный доступ по индексу (Random access indexing)

Как и в обычном байтовом массиве Java, индексация `ByteBuf` начинается с нуля: 
- индекс первого байта — 0, 
- а последнего байта — `capacity() - 1`. 

Следующий листинг показывает, что инкапсуляция механизмов хранения делает очень простым перебор содержимого `ByteBuf`.

**Листинг 5.6. Доступ к данным**
```java
ByteBuf buffer = ...;
for (int i = 0; i < buffer.capacity(); i++) {
    byte b = buffer.getByte(i);
    
    System.out.println((char) b);
}
```

> [!WARNING] `get/set` методы, НЕ ДВИГАЮТ ИНДЕКСЫ
> Обратите внимание, что доступ к данным с использованием одного из методов, принимающих индекс в качестве аргумента, не изменяет значение ни `readerIndex`, ни `writerIndex`. 

Любой из них может быть перемещён вручную при необходимости вызовом `readerIndex(index)` или `writerIndex(index)`.

---
### 5.3.2 Последовательный доступ по индексу (Sequential access indexing)

В то время как **`ByteBuf` имеет индексы читателя и писателя,** *JDK `ByteBuffer` имеет только один, поэтому вам приходится вызывать `flip()` для переключения между режимами чтения и записи.*

Рисунок 5.3 показывает, как `ByteBuf` разделяется своими двумя индексами на три области.

Они могут быть отброшены, а пространство восстановлено вызовом `discardReadBytes()`. Начальный размер этого сегмента, хранящийся в `readerIndex`, равен 0, увеличиваясь по мере выполнения операций чтения

Хотя у вас ***может возникнуть соблазн часто вызывать `discardReadBytes()` для максимизации сегмента записи,*** имейте в виду, что *это, скорее всего, вызовет копирование памяти, поскольку читаемые байты (помеченные как CONTENT на рисунках) должны быть перемещены в начало буфера.* Мы советуем делать это только тогда, когда это действительно необходимо; например, когда память в дефиците.

---
### 5.3.3 Читаемые байты (Readable bytes)

**Сегмент читаемых байтов `ByteBuf` хранит актуальные данные.** 

Значение по умолчанию `readerIndex` для нового выделенного, "обёрнутого" или скопированного буфера равно 0. Любая операция, имя которой начинается с `read` или `skip`, будет извлекать или пропускать данные на текущем `readerIndex` и увеличивать его на количество прочитанных байтов.

Если вызываемый метод принимает `ByteBuf` в качестве цели записи и не имеет аргумента индекса назначения, `writerIndex` буфера назначения также будет увеличен; например:
```java
readBytes(ByteBuf dest);
```

> Если предпринята попытка чтения из буфера, когда читаемые байты исчерпаны, выбрасывается `IndexOutOfBoundsException`.

В этом листинге ==показано, как прочитать все читаемые байты.==

**Листинг 5.7. Чтение всех данных**
```java
ByteBuf buffer = ...;
while (buffer.isReadable()) {
    System.out.println(buffer.readByte());
}
```

---
### 5.3.4 Записываемые байты (Writable bytes)

**Сегмент записываемых байтов — это область памяти** с неопределённым содержимым, **готовая для записи**. 

Значение по умолчанию `writerIndex` для нового выделенного буфера равно 0. **Любая операция, имя которой начинается с `write`, начнёт запись данных на текущем `writerIndex`, увеличивая его на количество записанных байтов.** 

**Если целью операции записи также является `ByteBuf` и исходный индекс не указан**, ***`readerIndex` исходного буфера будет увеличен на ту же величину.*** Этот вызов выглядел бы следующим образом:
```java
writeBytes(ByteBuf dest);
```

> Если предпринята попытка записи за пределы целевой ёмкости, будет выброшено `IndexOutOfBoundsException`.

Следующий листинг является примером, который заполняет буфер случайными целыми числами, пока не закончится место. 

Метод `writableBytes()` используется здесь для определения, достаточно ли места в буфере.

**Листинг 5.8. Запись данных**
```java
// Заполняет записываемые байты буфера случайными целыми числами.
ByteBuf buffer = ...;
while (buffer.writableBytes() >= 4) {
    buffer.writeInt(random.nextInt());
}
```

### 5.3.5 Управление индексами (Index management)

JDK `InputStream` определяет методы `mark(int readlimit)` и `reset()`. Они используются, соответственно, для отметки текущей позиции в потоке указанным значением и для сброса потока на эту позицию.

Аналогично, **вы можете установить и переместить** `readerIndex` и `writerIndex` у `ByteBuf`, вызывая `markReaderIndex()`, `markWriterIndex()`, `resetReaderIndex()` и `resetWriterIndex()`. 
Они похожи на вызовы `InputStream`, за исключением того, что нет параметра `readlimit`, указывающего, когда метка становится недействительной.

Вы также можете **переместить индексы в указанные позиции, вызвав** `readerIndex(int)` или `writerIndex(int)`. Попытка установить любой индекс в недопустимую позицию вызовет `IndexOutOfBoundsException`.

Вы можете **установить и `readerIndex`, и `writerIndex` в 0, вызвав** `clear()`. 
Обратите внимание, что это не очищает содержимое памяти. Рисунок 5.5 (повторяющий рисунок 5.3) показывает, как это работает.

### 5.3.6 Поисковые операции (Search operations)

Существует *несколько способов определить индекс указанного значения в* `ByteBuf`. 

Самый простой из них **использует методы** `indexOf()`.

Более **сложные поиски могут выполняться с помощью методов,** **принимающих аргумент** `ByteBufProcessor`. Этот интерфейс определяет единственный метод:
```java
boolean process(byte value)
```
который сообщает, является ли входное значение искомым.

`ByteBufProcessor` определяет множество удобных методов, нацеленных на общие значения. Предположим, вашему приложению нужно интегрироваться с так называемыми _Flash sockets_[^3], которые имеют содержимое, оканчивающееся нулём (NULL). Вызов
```java
forEachByte(ByteBufProcessor.FIND_NUL)
```

потребляет данные Flash просто и эффективно, поскольку при обработке выполняется меньше проверок границ.

В этом листинге показан пример поиска символа возврата каретки (`\r`).

**Листинг 5.9. Использование ByteBufProcessor для поиска \r**
```java
ByteBuf buffer = ...;
int index = buffer.forEachByte(ByteBufProcessor.FIND_CR);
```

### 5.3.7 Производные буферы (Derived buffers)

Производный буфер предоставляет представление `ByteBuf`, которое отображает его содержимое специализированным способом. 

Такие представления создаются следующими методами:
- `duplicate()`
    
- `slice()`
    
- `slice(int, int)`
    
- `Unpooled.unmodifiableBuffer(...)`
    
- `order(ByteOrder)`
    
- `readSlice(int)`

**Каждый из них возвращает новый экземпляр** `ByteBuf` со своими собственными индексами чтения, записи и маркерами. Внутреннее хранилище является общим, как и в JDK `ByteBuffer`. Это делает создание производного буфера недорогим, но также означает, что если вы измените его содержимое, вы измените и исходный экземпляр, так что будьте осторожны.

> **КОПИРОВАНИЕ BYTEBUF:** 
> **Если вам нужна настоящая копия существующего буфера, используйте** `copy()` или `copy(int,int)`. 
> 
> В отличие от производного буфера, `ByteBuf`, возвращённый этим вызовом, имеет независимую копию данных.

Следующий листинг показывает, как работать с сегментом `ByteBuf`, используя `slice(int, int)`.

**Листинг 5.10. Нарезка (Slice) ByteBuf**
```java
Charset utf8 = Charset.forName("UTF-8");

// (1) Создаёт ByteBuf, содержащий байты для данной строки
ByteBuf buf = Unpooled.copiedBuffer("Netty in Action rocks!", utf8); 

// (2) Создаёт новый "срез" ByteBuf, начиная с индекса 0 и заканчивая индексом 14
ByteBuf sliced = buf.slice(0, 14);         

// (3) Печатает "Netty in Action"                          
System.out.println(sliced.toString(utf8));            

// (4) Обновляет байт по индексу 0               
buf.setByte(0, (byte)'J');                          

// (5) Успешно, так как данные общие — изменения, сделанные в одном, видны в другом                 
assert buf.getByte(0) == sliced.getByte(0);                          
```

Теперь посмотрим, чем отличается копия сегмента `ByteBuf` от "среза".

**Листинг 5.11. Копирование ByteBuf**
```java

Charset utf8 = Charset.forName("UTF-8");
// (1) Создаёт ByteBuf для хранения байтов предоставленной строки
ByteBuf buf = Unpooled.copiedBuffer("Netty in Action rocks!", utf8); 

// (2) Создаёт копию сегмента ByteBuf, начиная с индекса 0 и заканчивая индексом 14
ByteBuf copy = buf.copy(0, 14);            
            
// (3) Печатает "Netty in Action"              
System.out.println(copy.toString(utf8));             

// (4) Обновляет байт по индексу 0                
buf.setByte(0, (byte)'J');                          

// (5) Успешно, так как данные не являются общими                 
assert buf.getByte(0) != copy.getByte(0);                            
```

Два случая идентичны, за исключением эффекта модификации "среза" или копии исходного `ByteBuf`. Когда это возможно, используйте `slice()`, чтобы избежать затрат на копирование памяти.

### 5.3.8 Операции чтения/записи (Read/write operations)

Как мы упоминали, существует две категории операций чтения/записи:

- **`get()` и `set()` операции**, которые начинаются с заданного индекса и оставляют его неизменным.
    
- **`read()` и `write()` операции**, которые начинаются с заданного индекса и корректируют его на количество доступных байтов.
    

**Таблица 5.1. Операции get()**

|Название|Описание|
|---|---|
|`getBoolean(int)`|Возвращает логическое значение по заданному индексу|
|`getByte(int)`|Возвращает байт по заданному индексу|
|`getUnsignedByte(int)`|Возвращает беззнаковое байтовое значение по заданному индексу как `short`|
|`getMedium(int)`|Возвращает 24-битное среднее целое значение по заданному индексу|
|`getUnsignedMedium(int)`|Возвращает беззнаковое 24-битное среднее целое значение по заданному индексу|
|`getInt(int)`|Возвращает `int` значение по заданному индексу|
|`getUnsignedInt(int)`|Возвращает беззнаковое `int` значение по заданному индексу как `long`|
|`getLong(int)`|Возвращает `long` значение по заданному индексу|
|`getShort(int)`|Возвращает `short` значение по заданному индексу|
|`getUnsignedShort(int)`|Возвращает беззнаковое `short` значение по заданному индексу как `int`|
|`getBytes(int, ...)`|Передаёт данные этого буфера в указанное место назначения, начиная с заданного индекса|

Большинство этих операций имеют соответствующий метод `set()`. Они перечислены в таблице 5.2.

**Таблица 5.2. Операции set()**

|Название|Описание|
|---|---|
|`setBoolean(int, boolean)`|Устанавливает логическое значение по заданному индексу|
|`setByte(int index, int value)`|Устанавливает байтовое значение по заданному индексу|
|`setMedium(int index, int value)`|Устанавливает 24-битное среднее значение по заданному индексу|
|`setInt(int index, int value)`|Устанавливает `int` значение по заданному индексу|
|`setLong(int index, long value)`|Устанавливает `long` значение по заданному индексу|
|`setShort(int index, int value)`|Устанавливает `short` значение по заданному индексу|

Следующий листинг иллюстрирует использование методов `get()` и `set()`, показывая, что они не изменяют индексы чтения и записи.

**Листинг 5.12. Использование get() и set()**
```java
Charset utf8 = Charset.forName("UTF-8");

// (1) Создаёт новый ByteBuf для хранения байтов заданной строки
ByteBuf buf = Unpooled.copiedBuffer("Netty in Action rocks!", utf8); 

// (2) Печатает первый символ, 'N'
System.out.println((char)buf.getByte(0));           
  
// (3) Сохраняет текущие readerIndex               
int readerIndex = buf.readerIndex();          
  
// и writerIndex                     
int writerIndex = buf.writerIndex();           

// (4) Обновляет байт по индексу 0 символом 'B'                      
buf.setByte(0, (byte)'B');                              

// (5) Печатает первый символ, теперь 'B'             
System.out.println((char)buf.getByte(0));               
     
// (6) Успешно, так как эти операции не изменяют индексы        
assert readerIndex == buf.readerIndex();                             
assert writerIndex == buf.writerIndex();
```

Теперь рассмотрим операции `read()`, которые действуют на текущем `readerIndex` или `writerIndex`. Эти методы используются для чтения из `ByteBuf` так, как если бы он был потоком. В таблице 5.3 показаны наиболее часто используемые методы.

**Таблица 5.3. Операции read()**

| Название                                                                | Описание                                                                                                                                                                                                                                                   |
| ----------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `readBoolean()`                                                         | Возвращает логическое значение на текущем `readerIndex` и увеличивает `readerIndex` на 1.                                                                                                                                                                  |
| `readByte()`                                                            | Возвращает байтовое значение на текущем `readerIndex` и увеличивает `readerIndex` на 1.                                                                                                                                                                    |
| `readUnsignedByte()`                                                    | Возвращает беззнаковое байтовое значение на текущем `readerIndex` как `short` и увеличивает `readerIndex` на 1.                                                                                                                                            |
| `readMedium()`                                                          | Возвращает 24-битное среднее значение на текущем `readerIndex` и увеличивает `readerIndex` на 3.                                                                                                                                                           |
| `readUnsignedMedium()`                                                  | Возвращает беззнаковое 24-битное среднее значение на текущем `readerIndex` и увеличивает `readerIndex` на 3.                                                                                                                                               |
| `readInt()`                                                             | Возвращает `int` значение на текущем `readerIndex` и увеличивает `readerIndex` на 4.                                                                                                                                                                       |
| `readUnsignedInt()`                                                     | Возвращает беззнаковое `int` значение на текущем `readerIndex` как `long` и увеличивает `readerIndex` на 4.                                                                                                                                                |

| `readShort()`                                                           | Возвращает `short` значение на текущем `readerIndex` и увеличивает `readerIndex` на 2.                                                                                                                                                                     |
| `readUnsignedShort()`                                                   | Возвращает беззнаковое `short` значение на текущем `readerIndex` как `int` и увеличивает `readerIndex` на 2.                                                                                                                                               |
| `readBytes(ByteBuf \| byte[] destination, int dstIndex [, int length])` | Передаёт данные из текущего `ByteBuf`, начиная с текущего `readerIndex` (и, если указано, длиной `length` байт), в целевой `ByteBuf` или `byte[]`, начиная с `dstIndex` получателя. Локальный `readerIndex` увеличивается на количество переданных байтов. |

Почти для каждого метода `read()` существует соответствующий метод `write()`, используемый для добавления данных в `ByteBuf`. Обратите внимание, что аргументы этих методов, перечисленные в таблице 5.4, являются значениями для записи, а не индексами.

**Таблица 5.4. Операции write()**

|Название|Описание|
|---|---|
|`writeBoolean(boolean)`|Записывает логическое значение на текущем `writerIndex` и увеличивает `writerIndex` на 1.|
|`writeByte(int)`|Записывает байтовое значение на текущем `writerIndex` и увеличивает `writerIndex` на 1.|
|`writeMedium(int)`|Записывает среднее (medium) значение на текущем `writerIndex` и увеличивает `writerIndex` на 3.|
|`writeInt(int)`|Записывает `int` значение на текущем `writerIndex` и увеличивает `writerIndex` на 4.|
|`writeLong(long)`|Записывает `long` значение на текущем `writerIndex` и увеличивает `writerIndex` на 8.|
|`writeShort(int)`|Записывает `short` значение на текущем `writerIndex` и увеличивает `writerIndex` на 2.|
|`writeBytes(source ByteBuf \| byte[] [, int srcIndex, int length])`|Передаёт данные, начиная с текущего `writerIndex`, из указанного источника (`ByteBuf` или `byte[]`). Если предоставлены `srcIndex` и `length`, чтение начинается с `srcIndex` и продолжается в течение `length` байт. Текущий `writerIndex` увеличивается на количество записанных байтов.|

**Листинг 5.13. Операции read() и write() над ByteBuf**
```java
Charset utf8 = Charset.forName("UTF-8");

// (1) Создаёт ByteBuf для хранения байтов заданной строки
ByteBuf buf = Unpooled.copiedBuffer("Netty in Action rocks!", utf8);

// (2) Печатает первый символ, 'N' 
System.out.println((char)buf.readByte());                 

// (3) Сохраняет текущий readerIndex           
int readerIndex = buf.readerIndex();           

// и writerIndex                      
int writerIndex = buf.writerIndex();               

// (4) Добавляет '?' в буфер                  
buf.writeByte((byte)'?');                            

// (5) Успешно, потому что writeByte() переместил writerIndex             
assert readerIndex == buf.readerIndex();                             
assert writerIndex != buf.writerIndex();
```

### 5.3.9 Дополнительные операции (More operations)

Таблица 5.5 перечисляет дополнительные полезные операции, предоставляемые `ByteBuf`.

**Таблица 5.5. Другие полезные операции**

|Название|Описание|
