## 5.4 Интерфейс ByteBufHolder

Мы часто сталкиваемся с **необходимостью хранить различные значения свойств** *в дополнение к фактической полезной нагрузке* (payload) данных. 

***HTTP-ответ — хороший пример***: наряду с содержимым, представленным в виде байтов, ***существуют код состояния, cookies и так далее.***

Netty предоставляет `ByteBufHolder` для обработки этого распространённого случая использования. `ByteBufHolder` также обеспечивает поддержку расширенных возможностей Netty, таких как пулинг буферов, когда `ByteBuf` может быть заимствован из пула, а также автоматически освобождён при необходимости.

`ByteBufHolder` имеет всего несколько методов для доступа к нижележащим данным и подсчёта ссылок. Таблица 5.6 перечисляет их (опуская те, которые он наследует от `ReferenceCounted`).

**Таблица 5.6. Операции ByteBufHolder**

|Название|Описание|
|---|---|
|`content()`|Возвращает `ByteBuf`, хранящийся в этом `ByteBufHolder`.|
|`copy()`|Возвращает глубокую копию этого `ByteBufHolder`, включая независимую копию данных содержащегося `ByteBuf`.|
|`duplicate()`|Возвращает неглубокую копию этого `ByteBufHolder`, включая разделяемую копию данных содержащегося `ByteBuf`.|

`ByteBufHolder` — хороший выбор, если вы хотите реализовать объект сообщения, который хранит свою полезную нагрузку в `ByteBuf`.

## 5.5 Выделение ByteBuf (ByteBuf allocation)

В этом разделе мы опишем **способы управления экземплярами** `ByteBuf`

### 5.5.1 По требованию: интерфейс ByteBufAllocator

Чтобы ***снизить накладные расходы на выделение и освобождение памяти***, **Netty реализует ПУЛИНГ с помощью интерфейса** `ByteBufAllocator`, который может использоваться для выделения экземпляров любого из описанных нами типов `ByteBuf`.

Использование пулинга — это решение, специфичное для приложения, которое никак не изменяет API `ByteBuf`.

**Таблица 5.7. Методы ByteBufAllocator**

|Название|Описание|
|---|---|
|`buffer()`  <br>`buffer(int initialCapacity)`  <br>`buffer(int initialCapacity, int maxCapacity)`|Возвращает `ByteBuf` с кучным (heap-based) или прямым (direct) хранением данных.|
|`heapBuffer()`  <br>`heapBuffer(int initialCapacity)`  <br>`heapBuffer(int initialCapacity, int maxCapacity)`|Возвращает `ByteBuf` с кучным хранением.|
|`directBuffer()`  <br>`directBuffer(int initialCapacity)`  <br>`directBuffer(int initialCapacity, int maxCapacity)`|Возвращает `ByteBuf` с прямым хранением.|
|`compositeBuffer()`  <br>`compositeBuffer(int maxNumComponents)`  <br>`compositeDirectBuffer()`  <br>`compositeDirectBuffer(int maxNumComponents)`  <br>`compositeHeapBuffer()`  <br>`compositeHeapBuffer(int maxNumComponents)`|Возвращает `CompositeByteBuf`, который может быть расширен путём добавления кучных или прямых буферов до указанного количества компонентов.|
|`ioBuffer()`|Возвращает `ByteBuf`, который будет использоваться для операций ввода-вывода на сокете.|

Вы можете **получить ссылку на `ByteBufAllocator`** 
- либо из `Channel` (каждый из которых может иметь свой экземпляр), 
- либо через `ChannelHandlerContext`, связанный с `ChannelHandler`. Следующий листинг иллюстрирует оба этих метода.

**Листинг 5.14. Получение ссылки на ByteBufAllocator**
```java
Channel channel = ...;

// (1) Получение ByteBufAllocator из Channel
ByteBufAllocator allocator = channel.alloc();           
// ...


ChannelHandlerContext ctx = ...;
// (2) Получение ByteBufAllocator из ChannelHandlerContext
ByteBufAllocator allocator2 = ctx.alloc();              
```

> [!WARNING] РЕАЛИЗАЦИИ `ByteBuffAllocator` (пуллинг и не пуллинг)
> Netty предоставляет две реализации `ByteBufAllocator`:
> - **`PooledByteBufAllocator`** 
> - и **`UnpooledByteBufAllocator`**. 

**Первый использует пулинг экземпляров `ByteBuf`** для повышения производительности и минимизации фрагментации памяти. 
*Эта реализация использует эффективный подход к выделению памяти, известный как* **jemalloc**, который был принят рядом современных операционных систем. 

**Вторая реализация не использует пулинг** экземпляров `ByteBuf` и возвращает новый экземпляр при каждом вызове.

Хотя Netty **по умолчанию использует `PooledByteBufAllocator`, это можно легко изменить через API** `ChannelConfig` или указав другой аллокатор при бутстраппинге вашего приложения. Более подробную информацию можно найти в главе 8.

---
### 5.5.2 Непулированные буферы (Unpooled buffers)

Могут быть ситуации, когда у вас нет ссылки на `ByteBufAllocator`. 

Для этого случая **Netty предоставляет утилитарный класс** `Unpooled`, который содержит статические вспомогательные методы для создания непулированных экземпляров `ByteBuf`. 

В таблице 5.8 перечислены наиболее важные из этих методов.

**Таблица 5.8. Методы Unpooled**

|Название|Описание|
|---|---|
|`buffer()`  <br>`buffer(int initialCapacity)`  <br>`buffer(int initialCapacity, int maxCapacity)`|Возвращает непулированный `ByteBuf` с кучным хранением.|
|`directBuffer()`  <br>`directBuffer(int initialCapacity)`  <br>`directBuffer(int initialCapacity, int maxCapacity)`|Возвращает непулированный `ByteBuf` с прямым хранением.|
|`wrappedBuffer()`|Возвращает `ByteBuf`, который оборачивает (wraps) данные.|
|`copiedBuffer()`|Возвращает `ByteBuf`, который копирует (copies) данные.|

Класс `Unpooled` также делает `ByteBuf` доступным для не-сетевых проектов, которые могут извлечь выгоду из высокопроизводительного расширяемого буферного API и не требуют других компонентов Netty.

---
### 5.5.3 Класс ByteBufUtil

`ByteBufUtil` предоставляет **статические** вспомогательные **методы для управления `ByteBuf`**. Поскольку этот API является общим и не связан с пулингом, эти методы реализованы вне классов выделения памяти.

Наиболее ценным из этих статических методов, вероятно, является `hexdump()`, ==который печатает шестнадцатеричное представление содержимого== `ByteBuf`. 

Это полезно в различных ситуациях, например, для логирования содержимого `ByteBuf` в целях отладки. Шестнадцатеричное представление обычно обеспечивает более удобоваримую запись в логе, чем прямое представление байтовых значений. Кроме того, шестнадцатеричную версию можно легко преобразовать обратно в фактическое байтовое представление.

Другой полезный метод — `boolean equals(ByteBuf, ByteBuf)`, который определяет равенство двух экземпляров `ByteBuf`. Вы можете найти другие методы `ByteBufUtil` полезными, если реализуете свои собственные подклассы `ByteBuf`.

## 5.6 Подсчёт ссылок (Reference counting)

Подсчёт ссылок — это **метод оптимизации использования памяти и производительности путём освобождения ресурсов**, удерживаемых объектом, когда на него больше нет ссылок из других объектов.

Netty представил подсчёт ссылок в версии 4 для `ByteBuf` и `ByteBufHolder`, *оба из которых реализуют интерфейс* `ReferenceCounted`

Идея подсчёта ссылок не особенно сложна; в основном она включает отслеживание количества активных ссылок на указанный объект. Экземпляр реализации `ReferenceCounted` обычно начинает работу с активным счётчиком ссылок, равным 1. **Пока счётчик ссылок больше 0, объект гарантированно не будет освобождён**. **Когда количество активных ссылок уменьшается до 0, экземпляр освобождается.** 
> Обратите внимание, что хотя точное значение "освобождения" может зависеть от реализации, по крайней мере, объект, который был освобождён, больше не должен быть доступен для использования.

**Подсчёт ссылок необходим для реализаций пулинга**, таких как `PooledByteBufAllocator`, который снижает накладные расходы на выделение памяти. Примеры показаны в следующих двух листингах.

**Листинг 5.15. Подсчёт ссылок**
```java
Channel channel = ...;
// (1) Получение ByteBufAllocator из канала
ByteBufAllocator allocator = channel.alloc();      
        
// ...

// (2) Выделение ByteBuf из ByteBufAllocator
ByteBuf buffer = allocator.directBuffer();  

// (3) Проверка ожидаемого счётчика ссылок, равного 1
assert buffer.refCnt() == 1;                               
// ...
```

**Листинг 5.16. Освобождение объекта с подсчётом ссылок**
```java
ByteBuf buffer = ...;

// (1) Уменьшает активные ссылки на объект. При достижении 0 объект освобождается и метод возвращает true.
boolean released = buffer.release();                       
// ...
```

Попытка доступа к объекту с подсчётом ссылок, который был освобождён, приведёт к `IllegalReferenceCountException`.

Обратите внимание, что конкретный класс может определить свой собственный контракт подсчёта ссылок уникальным образом. Например, мы можем представить себе класс, чья реализация `release()` всегда устанавливает счётчик ссылок в ноль, независимо от его текущего значения, таким образом делая недействительными все активные ссылки сразу.

> **КТО ОТВЕЧАЕТ ЗА ОСВОБОЖДЕНИЕ?** В общем случае, последняя сторона, которая обращается к объекту, несёт ответственность за его освобождение. В главе 6 мы объясним актуальность этой концепции для `ChannelHandler` и `ChannelPipeline`.
