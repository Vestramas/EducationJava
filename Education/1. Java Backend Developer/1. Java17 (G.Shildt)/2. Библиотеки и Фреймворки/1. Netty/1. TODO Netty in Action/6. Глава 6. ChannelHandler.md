## 6.1 Семейство ChannelHandler (The ChannelHandler family)

Чтобы подготовиться к нашему детальному изучению `ChannelHandler`, мы потратим время на некоторые основы этой части компонентной модели Netty.

### 6.1.1 Жизненный цикл Channel (The Channel lifecycle)

Интерфейс `Channel` определяет простую, но мощную модель состояний, которая тесно связана с API `ChannelInboundHandler`. Четыре состояния канала перечислены в таблице 6.1.

**Таблица 6.1. Состояния жизненного цикла Channel**

| Состояние             | Описание                                                                                    |
| --------------------- | ------------------------------------------------------------------------------------------- |
| `ChannelUnregistered` | **Канал создан, но НЕ ЗАРЕГИСТРИРОВАН** в `EventLoop`.                                      |
| `ChannelRegistered`   | **Канал зарегистрирован в** `EventLoop`.                                                    |
| `ChannelActive`       | **Канал активен (подключён к удалённому узлу)**. Теперь можно получать и отправлять данные. |
| `ChannelInactive`     | **Канал не подключён к удалённому узлу.**                                                   |

Нормальный жизненный цикл `Channel` показан на рисунке 6.1. По мере того как эти изменения состояния происходят, генерируются соответствующие события. Они передаются `ChannelHandler` в `ChannelPipeline`, которые затем могут реагировать на них.

### 6.1.2 Жизненный цикл ChannelHandler (The ChannelHandler lifecycle)

**Операции жизненного цикла**, определённые интерфейсом `ChannelHandler` и перечисленные в таблице 6.2, **вызываются ПОСЛЕ ТОГО, как `ChannelHandler` был добавлен в** `ChannelPipeline` или удалён из него. 

Каждый метод принимает аргумент `ChannelHandlerContext`.

**Таблица 6.2. Методы жизненного цикла ChannelHandler**

|Тип|Описание|
|---|---|
|`handlerAdded`|Вызывается, когда `ChannelHandler` добавляется в `ChannelPipeline`.|
|`handlerRemoved`|Вызывается, когда `ChannelHandler` удаляется из `ChannelPipeline`.|
|`exceptionCaught`|Вызывается, если во время обработки в `ChannelPipeline` произошла ошибка.|

Netty определяет следующие ==два важных подинтерфейса== `ChannelHandler`:

- **`ChannelInboundHandler`** — **Обрабатывает ВХОДЯЩИЕ данные** и изменения состояния всех видов.
    
- **`ChannelOutboundHandler`** — **Обрабатывает ИСХОДЯЩИЕ данные** и ***позволяет перехватывать все операции.***

В следующих разделах мы подробно обсудим эти интерфейсы.

### 6.1.3 Интерфейс ChannelInboundHandler

В таблице 6.3 перечислены методы жизненного цикла интерфейса `ChannelInboundHandler`. 

Они вызываются, когда данные получены или когда изменилось состояние связанного канала. Как мы упоминали ранее, эти методы тесно связаны с жизненным циклом канала.

**Таблица 6.3. Методы ChannelInboundHandler**

| Тип                         | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `channelRegistered`         | Вызывается, когда канал зарегистрирован в своём `EventLoop` и готов обрабатывать I/O.                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `channelUnregistered`       | Вызывается, когда канал дерегистрирован из своего `EventLoop` и не может обрабатывать I/O.                                                                                                                                                                                                                                                                                                                                                                                                              |
| `channelActive`             | Вызывается, когда канал активен; канал подключён/привязан и готов.                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `channelInactive`           | Вызывается, когда канал выходит из активного состояния и больше не подключён к удалённому узлу.                                                                                                                                                                                                                                                                                                                                                                                                         |
| `channelReadComplete`       | Вызывается, **когда операция чтения на канале завершена.**                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| `channelRead`               | Вызывается, если данные прочитаны из канала.                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `channelWritabilityChanged` | Вызывается, когда изменяется состояние возможности записи канала. Пользователь может гарантировать, что запись не происходит слишком быстро (чтобы избежать `OutOfMemoryError`), или может возобновить запись, когда канал снова становится доступным для записи. Метод канала `isWritable()` может быть вызван для определения возможности записи канала. Порог доступности записи может быть установлен через `Channel.config().setWriteHighWaterMark()` и `Channel.config().setWriteLowWaterMark()`. |
| `userEventTriggered`        | Вызывается, когда `ChannelInboundHandler.fireUserEventTriggered()` был вызван, потому что POJO был передан через `ChannelPipeline`.                                                                                                                                                                                                                                                                                                                                                                     |

Когда реализация `ChannelInboundHandler` переопределяет `channelRead()`, она отвечает за явное освобождение памяти, связанной с пулированными экземплярами `ByteBuf`. Netty предоставляет для этой цели вспомогательный метод `ReferenceCountUtil.release()`, как показано далее.

**Листинг 6.1. Освобождение ресурсов сообщения**
```java

@Sharable
// (1) Расширяет ChannelInboundHandlerAdapter
public class DiscardHandler extends ChannelInboundHandlerAdapter {    
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
	    // (2) Отбрасывает полученное сообщение
        ReferenceCountUtil.release(msg);   
    }
}
```

> **Netty регистрирует НЕОСВОБОЖДЕННЫЕ ресурсы с помощью сообщения уровня WARN**, что позволяет довольно просто найти проблемные экземпляры в коде. 

Но управление ресурсами таким способом может быть громоздким. 

==Более простая альтернатива== — использовать `SimpleChannelInboundHandler`. Следующий листинг является вариацией листинга 6.1, которая это иллюстрирует.

**Листинг 6.2. Использование SimpleChannelInboundHandler**
```java
@Sharable
// (1) Расширяет SimpleChannelInboundHandler
public class SimpleDiscardHandler extends SimpleChannelInboundHandler<Object> { 
    @Override
    public void channelRead0(ChannelHandlerContext ctx, Object msg) {
		// Не нужно ничего специального делать
		// (2) Нет необходимости в явном освобождении ресурсов
    }
}
```

> [!WARNING] `SimpleChannelInboundHandler` **автоматически освобождает ресурсы**
> 

Поскольку `SimpleChannelInboundHandler` ***автоматически освобождает ресурсы, вы не должны хранить ссылки на какие-либо сообщения для последующего использования***, так как они станут недействительными.

В разделе 6.1.6 представлено более подробное обсуждение обработки ссылок.

### 6.1.4 Интерфейс ChannelOutboundHandler

**Исходящие операции и данные обрабатываются** `ChannelOutboundHandler`.
Его методы вызываются 
- `Channel`, 
- `ChannelPipeline` 
- и `ChannelHandlerContext`.

**Мощная возможность `ChannelOutboundHandler` — откладывать операцию или событие по требованию**, что позволяет использовать сложные подходы к обработке запросов. 
> Например, если запись удалённому узлу приостановлена, вы можете отложить операции flush и возобновить их позже.

В таблице 6.4 показаны все методы, определённые локально в `ChannelOutboundHandler` (опуская унаследованные от `ChannelHandler`).

**Таблица 6.4. Методы ChannelOutboundHandler**

|Тип|Описание|
|---|---|
|`bind(ChannelHandlerContext, SocketAddress, ChannelPromise)`|Вызывается по запросу для привязки канала к локальному адресу.|
|`connect(ChannelHandlerContext, SocketAddress, SocketAddress, ChannelPromise)`|Вызывается по запросу для подключения канала к удалённому узлу.|
|`disconnect(ChannelHandlerContext, ChannelPromise)`|Вызывается по запросу для отключения канала от удалённого узла.|
|`close(ChannelHandlerContext, ChannelPromise)`|Вызывается по запросу для закрытия канала.|
|`deregister(ChannelHandlerContext, ChannelPromise)`|Вызывается по запросу для дерегистрации канала из его `EventLoop`.|
|`read(ChannelHandlerContext)`|Вызывается по запросу для чтения дополнительных данных из канала.|
|`flush(ChannelHandlerContext)`|Вызывается по запросу для сброса (flush) данных из очереди удалённому узлу через канал.|
|`write(ChannelHandlerContext, Object, ChannelPromise)`|Вызывается по запросу для записи данных через канал удалённому узлу.|

> **CHANNELPROMISE VS. CHANNELFUTURE:** Большинство методов в `ChannelOutboundHandler` принимают аргумент `ChannelPromise` для уведомления о завершении операции. 
> 
> `ChannelPromise` ==является подинтерфейсом== `ChannelFuture`, который определяет доступные для записи методы, такие как `setSuccess()` или `setFailure()`, делая `ChannelFuture` неизменяемым (immutable).

Далее мы рассмотрим классы, которые упрощают задачу написания `ChannelHandler`.

### 6.1.5 АДАПТЕРЫ ChannelHandler (ChannelHandler adapters)

Вы можете использовать классы `ChannelInboundHandlerAdapter` и `ChannelOutboundHandlerAdapter` в качестве отправных точек для ваших собственных `ChannelHandler`. 

Эти **адаптеры предоставляют базовые реализации** соответственно `ChannelInboundHandler` и `ChannelOutboundHandler`. 

Они ***получают методы своего общего суперинтерфейса*** `ChannelHandler`, *расширяя абстрактный класс* `ChannelHandlerAdapter`. Получившаяся иерархия классов показана на рисунке 6.2.

![[Pasted image 20260611205737.png]]
**Рисунок 6.2. Иерархия классов ChannelHandlerAdapter**

`ChannelHandlerAdapter` также предоставляет вспомогательный метод `isSharable()`. Этот метод возвращает `true`, если реализация аннотирована как `@Sharable`, **что указывает на то, что она может быть добавлена в несколько** `ChannelPipeline` (как обсуждалось в разделе 2.3.1).

Тела методов, предоставленные в `ChannelInboundHandlerAdapter` и `ChannelOutboundHandlerAdapter`, вызывают эквивалентные методы на связанном `ChannelHandlerContext`, тем самым передавая события следующему `ChannelHandler` в конвейере.

Чтобы использовать эти классы-адаптеры в ваших собственных обработчиках, просто расширьте их и переопределите методы, которые вы хотите настроить.

### 6.1.6 Управление ресурсами (Resource management)

Всякий раз, когда вы действуете с данными, вызывая `ChannelInboundHandler.channelRead()` или `ChannelOutboundHandler.write()`, ==вы должны убедиться, что нет утечек ресурсов==. 

Как вы, возможно, помните из предыдущей главы, *Netty использует подсчёт ссылок для обработки пулированных* `ByteBuf`. **Поэтому важно корректировать счётчик ссылок после того, как вы закончили использовать `ByteBuf`.**

**Чтобы помочь вам в диагностике потенциальных проблем, Netty предоставляет класс** `ResourceLeakDetector`, который будет анализировать около 1% выделений буферов в вашем приложении на предмет утечек памяти. Связанные накладные расходы очень малы.

***Если утечка обнаружена, будет выведено сообщение в лог,*** подобное следующему:
```
LEAK: ByteBuf.release() was not called before it's garbage-collected. Enable 
advanced leak reporting to find out where the leak occurred. To enable 
advanced leak reporting, specify the JVM option
'-Dio.netty.leakDetectionLevel=ADVANCED' or call 
ResourceLeakDetector.setLevel().
```

Netty в настоящее время определяет четыре уровня обнаружения утечек, перечисленных в таблице 6.5.

**Таблица 6.5. Уровни обнаружения утечек (Leak-detection levels)**

|Уровень|Описание|
|---|---|
|`DISABLED`|Отключает обнаружение утечек. Используйте это только после обширного тестирования.|
|`SIMPLE`|Сообщает о любых найденных утечках, используя частоту выборки по умолчанию 1%. Это уровень по умолчанию и хорошо подходит для большинства случаев.|
|`ADVANCED`|Сообщает об утечках и о том, где был получен доступ к сообщению. Использует частоту выборки по умолчанию.|
|`PARANOID`|Как и `ADVANCED`, за исключением того, что каждый доступ подвергается выборке. Это оказывает сильное влияние на производительность и должно использоваться только на этапе отладки.|

Уровень обнаружения утечек определяется путём установки следующего системного свойства Java в одно из значений в таблице:
```bash
java -Dio.netty.leakDetectionLevel=ADVANCED
```

Если вы перезапустите своё приложение с этой опцией JVM, вы увидите недавние места в вашем приложении, где был получен доступ к "утекшему" буферу. Ниже приведён типичный отчёт об утечке, сгенерированный модульным тестом:
```

ByteBuf.release() was not called before it's garbage-collected.
```

Как вы используете этот диагностический инструмент для предотвращения утечек при реализации `ChannelInboundHandler.channelRead()` и `ChannelOutboundHandler.write()`? 

Давайте рассмотрим случай, когда ваша операция `channelRead()` потребляет входящее сообщение, то есть не передаёт его следующему `ChannelInboundHandler` вызовом `ChannelHandlerContext.fireChannelRead()`. Этот листинг показывает, как освободить сообщение.

**Листинг 6.3. Потребление и освобождение входящего сообщения**
```java
@Sharable
public class DiscardInboundHandler extends ChannelInboundHandlerAdapter {    // (1) Расширяет ChannelInboundHandlerAdapter
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
	    // (2) Освобождает ресурс с помощью ReferenceCountUtil.release()
        ReferenceCountUtil.release(msg);                                 
        
    }
}
```

> **ПОТРЕБЛЕНИЕ ВХОДЯЩИХ СООБЩЕНИЙ ПРОСТЫМ СПОСОБОМ:** Поскольку потребление входящих данных и их освобождение является очень распространённой задачей, Netty предоставляет специальную реализацию `ChannelInboundHandler` под названием `SimpleChannelInboundHandler`. 
> 
> Эта реализация автоматически освобождает сообщение после того, как оно было потреблено `channelRead0()`. (***после того как метод завершится***)

На исходящей стороне, если вы обрабатываете операцию `write()` и отбрасываете сообщение, вы несёте ответственность за его освобождение. Следующий листинг показывает реализацию, которая отбрасывает все записанные данные.

**Листинг 6.4. Отбрасывание и освобождение исходящих данных**
```java
@Sharable
public class DiscardOutboundHandler extends ChannelOutboundHandlerAdapter {  // (1) Расширяет ChannelOutboundHandlerAdapter
    @Override
    public void write(
	    ChannelHandlerContext ctx, 
	    Object msg, 
	    ChannelPromise promise
	) {
		// (2) Освобождает ресурс с помощью ReferenceCountUtil.release()
        ReferenceCountUtil.release(msg);                            
              
        // (3) Уведомляет ChannelPromise, что данные были обработаны
        promise.setSuccess();                                               
    }
}
```

Важно не только освобождать ресурсы, но и уведомлять `ChannelPromise`. В противном случае может возникнуть ситуация, когда `ChannelFutureListener` не был уведомлён о сообщении, которое было обработано.

Таким образом, **пользователь несёт ответственность за вызов `ReferenceCountUtil.release()`**, если сообщение потребляется или отбрасывается и не передаётся следующему `ChannelOutboundHandler` в `ChannelPipeline`. Если сообщение достигает реального транспортного уровня, оно будет освобождено автоматически, когда оно будет записано или канал будет закрыт.
