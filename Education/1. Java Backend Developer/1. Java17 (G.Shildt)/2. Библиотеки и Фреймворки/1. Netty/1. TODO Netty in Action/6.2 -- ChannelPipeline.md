## 6.2 Интерфейс ChannelPipeline

Если вы думаете о `ChannelPipeline` как о цепочке экземпляров `ChannelHandler`, которые перехватывают входящие и исходящие события, проходящие через канал, легко понять, как взаимодействие этих `ChannelHandler` может составлять ядро логики обработки данных и событий приложения.

**Каждый новый созданный канал получает новый** `ChannelPipeline`. ==Эта связь является постоянной==; канал не может ни присоединить другой `ChannelPipeline`, ни отсоединить текущий. 
> Это фиксированная операция в жизненном цикле компонентов Netty и не требует никаких действий со стороны разработчика.

В зависимости от своего происхождения, событие будет обработано либо `ChannelInboundHandler`, либо `ChannelOutboundHandler`. 
Затем оно будет передано следующему обработчику того же супертипа вызовом реализации `ChannelHandlerContext`.

> **ChannelHandlerContext**
> 
> `ChannelHandlerContext` **позволяет `ChannelHandler` взаимодействовать со своим `ChannelPipeline` и с другими обработчиками.** 
> 
> Обработчик может уведомить следующий `ChannelHandler` в `ChannelPipeline` и даже динамически изменять `ChannelPipeline`, которому он принадлежит.
> 
> `ChannelHandlerContext` имеет богатый API для обработки событий и выполнения операций ввода-вывода. В разделе 6.3 будет представлено больше информации о `ChannelHandlerContext`.

Рисунок 6.3 иллюстрирует типичную структуру `ChannelPipeline` с входящими и исходящими `ChannelHandler` и иллюстрирует наше предыдущее утверждение, что `ChannelPipeline` — это, по сути, серия `ChannelHandler`. `ChannelPipeline` также предоставляет методы для распространения событий через сам `ChannelPipeline`. Если входящее событие инициировано, оно передаётся от начала к концу `ChannelPipeline`. На рисунке 6.3 исходящее событие I/O начнётся с правого конца `ChannelPipeline` и пойдёт налево.

**Рисунок 6.3. ChannelPipeline и ChannelHandler**

> [!WARNING] Цепочка направления IN и OUT - двусвязный список
> Netty идет по этой цепочке в зависимости от направления сообщения:
> - IN - идет слева на право, и пропускает все OUT обработчики 
> - OUT -идет справа на лево, и пропускает все IN обработчики
> 
> Проще представить так: 
> - что **левее, то ближе к сети**
> - *правее - ближе к бизнес логике*
> 
> Поэтому, логичнее, кодеры, и декодеры объявлять первыми. А дальше уже смотреть на логику работы. 
> 
> Ведь есть методы 
> - `ctx.channel.writeAndFlush()` - от крайнего правого положения и на лево
> - `ctx.writeAndFlush()` - от положения текущего обработчика и на лево
> Эти методы, могут не совсем очевидным способом изменить направления обработки сообщения


> **Относительность ChannelPipeline** (НУМЕРАЦИЯ ОБРАБОТЧИКОВ - СЛЕВА НАПРАВО)
> 
> Можно сказать, что с точки зрения события, проходящего через `ChannelPipeline`, начальный конец зависит от того, является ли событие входящим или исходящим. 
> 
> Но Netty всегда идентифицирует входящий вход в `ChannelPipeline` (левая сторона на рисунке 6.3) как начало, а исходящий вход (правая сторона) как конец.
> 
> Когда вы закончите добавлять смесь входящих и исходящих обработчиков в `ChannelPipeline` с помощью методов `ChannelPipeline.add*()`, **порядковый номер каждого `ChannelHandler` будет его позицией от начала до конца**, как мы только что определили. 
> Таким образом, ***если вы пронумеруете обработчики на рисунке 6.3 слева направо***:
> - ==первый== `ChannelHandler`, который **увидит ВХОДЯЩЕЕ событие, будет 1;** 
> - ==первый обработчик==, который увидит **ИСХОДЯЩЕЕ событие, будет 5.**
>   
>   
> По мере того как **конвейер** распространяет событие, он **определяет**, **соответствует ли тип следующего** `ChannelHandler` **в конвейере направлению движения.** 
> 
> **Если нет, `ChannelPipeline` пропускает этот** `ChannelHandler` и переходит к следующему, пока не найдёт соответствующий желаемому направлению (***конечно, обработчик может реализовывать оба интерфейса*** — `ChannelInboundHandler` и `ChannelOutboundHandler`).

**ПРИМЕР**
```java
public class DebugHandler 
extends ChannelDuplexHandler { // реализует и Inbound, и Outbound
    private final String name;
    
    public DebugHandler(String name) { this.name = name; }
    
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        System.out.println("INBOUND: " + name);
        ctx.fireChannelRead(msg);
    }
    
    @Override
    public void write(ChannelHandlerContext ctx, Object msg, ChannelPromise promise) {
        System.out.println("OUTBOUND: " + name);
        ctx.write(msg, promise);
    }
}
```

**Pipeline:**
```java
pipeline.addLast("A", new DebugHandler("A"));
pipeline.addLast("B", new DebugHandler("B"));
pipeline.addLast("C", new DebugHandler("C"));
```

**Вывод при входящем сообщении:**
```
INBOUND: A
INBOUND: B
INBOUND: C

//Первым вызываем самый первый
```

**Вывод при `ctx.writeAndFlush()`:**
```
OUTBOUND: C
OUTBOUND: B
OUTBOUND: A

//Первым вызывается самый последний
```

### 6.2.1 Изменение ChannelPipeline

`ChannelHandler` может изменять структуру `ChannelPipeline` в реальном времени, добавляя, удаляя или заменяя других `ChannelHandler` (он также может удалить себя из `ChannelPipeline`). Это одна из наиболее важных возможностей `ChannelHandler`, поэтому мы подробно рассмотрим, как это делается. Соответствующие методы перечислены в таблице 6.6.

**Таблица 6.6. Методы ChannelHandler для изменения ChannelPipeline**

|Название|Описание|
|---|---|
|`addFirst`, `addBefore`, `addAfter`, `addLast`|Добавляет `ChannelHandler` в `ChannelPipeline`.|
|`remove`|Удаляет `ChannelHandler` из `ChannelPipeline`.|
|`replace`|Заменяет `ChannelHandler` в `ChannelPipeline` другим `ChannelHandler`.|

Этот листинг показывает эти методы в действии.

**Листинг 6.5. Изменение ChannelPipeline**
```java
ChannelPipeline pipeline = ...;

// (1) Создаёт экземпляр FirstHandler
FirstHandler firstHandler = new FirstHandler();  

// (2) Добавляет этот экземпляр в ChannelPipeline как "handler1"       
pipeline.addLast("handler1", firstHandler); 

// (3) Добавляет экземпляр SecondHandler в первое место как "handler2" (перед "handler1")                       
pipeline.addFirst("handler2", new SecondHandler());  

// (4) Добавляет экземпляр ThirdHandler в последнее место как "handler3"  
pipeline.addLast("handler3", new ThirdHandler());       
           
// ...

// (5) Удаляет "handler3" по имени
pipeline.remove("handler3");   

// (6) Удаляет FirstHandler по ссылке (имя не нужно)   
pipeline.remove(firstHandler);                      

// (7) Заменяет SecondHandler ("handler2") на FourthHandler ("handler4")  
pipeline.replace("handler2", "handler4", new FourthHandler());     
```

Позже вы увидите, что эта возможность легко реорганизовывать `ChannelHandler` способствует реализации чрезвычайно гибкой логики.

#### **Выполнение ChannelHandler и блокировка**

Обычно каждый `ChannelHandler` в `ChannelPipeline` обрабатывает события, переданные ему его `EventLoop` (I/O поток). ***Крайне важно*** **не блокировать этот поток**, так как это отрицательно скажется на общей обработке I/O.


*Иногда может потребоваться взаимодействие* с устаревшим кодом, использующим *блокирующие API.* 
Д*ля этого случая в `ChannelPipeline` есть методы `add()`, которые принимают `EventExecutorGroup`*. 

Если событие передаётся в пользовательскую `EventExecutorGroup`, оно будет обработано одним из `EventExecutor`, содержащихся в этой группе, и, таким образом, будет удалено из `EventLoop` самого канала. Для этого случая использования Netty предоставляет реализацию под названием `DefaultEventExecutorGroup`.

Помимо этих операций, существуют другие для доступа к `ChannelHandler` либо по типу, либо по имени. Они перечислены в таблице 6.7.

**Таблица 6.7. Операции ChannelPipeline для доступа к ChannelHandler**

|Название|Описание|
|---|---|
|`get`|Возвращает `ChannelHandler` по типу или имени.|
|`context`|Возвращает `ChannelHandlerContext`, связанный с `ChannelHandler`.|
|`names`|Возвращает имена всех `ChannelHandler` в `ChannelPipeline`.|

### 6.2.2 Генерация событий (Firing events)

API `ChannelPipeline` **предоставляет дополнительные методы для вызова входящих и исходящих операций.** 

В таблице 6.8 **перечислены входящие операции, которые УВЕДОМЛЯЮТ** `ChannelInboundHandler` **О СОБЫТИЯХ**, происходящих в `ChannelPipeline`.

**Таблица 6.8. Входящие операции ChannelPipeline**

|Название метода|Описание|
|---|---|
|`fireChannelRegistered`|Вызывает `channelRegistered(ChannelHandlerContext)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireChannelUnregistered`|Вызывает `channelUnregistered(ChannelHandlerContext)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireChannelActive`|Вызывает `channelActive(ChannelHandlerContext)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireChannelInactive`|Вызывает `channelInactive(ChannelHandlerContext)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireExceptionCaught`|Вызывает `exceptionCaught(ChannelHandlerContext, Throwable)` на следующем `ChannelHandler` в `ChannelPipeline`.|
|`fireUserEventTriggered`|Вызывает `userEventTriggered(ChannelHandlerContext, Object)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireChannelRead`|Вызывает `channelRead(ChannelHandlerContext, Object msg)` на следующем `ChannelInboundHandler` в `ChannelPipeline`.|
|`fireChannelReadComplete`|Вызывает `channelReadComplete(ChannelHandlerContext)` на следующем `ChannelStateHandler` в `ChannelPipeline`.|

На исходящей стороне обработка события вызовет некоторое действие на нижележащем сокете. Таблица 6.9 перечисляет исходящие операции API `ChannelPipeline`.

**Таблица 6.9. Исходящие операции ChannelPipeline**

|Название метода|Описание|
|---|---|
|`bind`|Привязывает канал к локальному адресу. Вызывает `bind(ChannelHandlerContext, SocketAddress, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`connect`|Подключает канал к удалённому адресу. Вызывает `connect(ChannelHandlerContext, SocketAddress, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`disconnect`|Отключает канал. Вызывает `disconnect(ChannelHandlerContext, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`close`|Закрывает канал. Вызывает `close(ChannelHandlerContext, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`deregister`|Дерегистрирует канал из ранее назначенного `EventExecutor` (`EventLoop`). Вызывает `deregister(ChannelHandlerContext, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`flush`|Сбрасывает (flushes) все ожидающие записи канала. Вызывает `flush(ChannelHandlerContext)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|
|`write`|Записывает сообщение в канал. Вызывает `write(ChannelHandlerContext, Object msg, ChannelPromise)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`. **Примечание:** это не записывает сообщение в нижележащий сокет, а только ставит его в очередь. Чтобы записать его в сокет, вызовите `flush()` или `writeAndFlush()`.|
|`writeAndFlush`|Удобный метод для вызова `write()`, за которым следует `flush()`.|
|`read`|Запрашивает чтение дополнительных данных из канала. Вызывает `read(ChannelHandlerContext)` на следующем `ChannelOutboundHandler` в `ChannelPipeline`.|

**Резюмируя:**
- `ChannelPipeline` содержит `ChannelHandler`, связанные с каналом.
    
- `ChannelPipeline` может динамически изменяться путём добавления и удаления `ChannelHandler` по мере необходимости.
    
- `ChannelPipeline` имеет богатый API для вызова действий в ответ на входящие и исходящие события.
