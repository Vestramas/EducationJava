## 6.3 Интерфейс ChannelHandlerContext

`ChannelHandlerContext` **представляет связь между `ChannelHandler` и `ChannelPipeline`** ***и создаётся всякий раз, когда `ChannelHandler` добавляется в `ChannelPipeline`.*** 

Основная функция `ChannelHandlerContext` — **управлять взаимодействием связанного с ним `ChannelHandler` с другими в том же `ChannelPipeline`.**

---

> [!NOTE] Вызов методов 
> Методы ChannelHandlerContext есть в Channel и ChannelPipeline. 
> 
> Разница между ними В РАСПРОСТРАНЕНИИ ДЕЙСТВИЯ 

`ChannelHandlerContext` ***имеет множество методов, некоторые из которых также присутствуют в*** `Channel` и самом `ChannelPipeline`, но есть важное отличие. 

***Если вы вызываете эти методы на экземпляре*** `Channel` или `ChannelPipeline`, **они распространяются через весь конвейер**. 

*Те же методы, вызванные на* `ChannelHandlerContext`, **начнутся с текущего связанного `ChannelHandler` и будут распространяться только до следующего `ChannelHandler` в конвейере**, который способен обработать событие.


---
В таблице 6.10 обобщён API `ChannelHandlerContext`.

**Таблица 6.10. API ChannelHandlerContext**

| Название метода           | Описание                                                                                                                                              |

| `executor`                | Возвращает `EventExecutor`, который диспетчеризирует события.                                                                                         |
| `fireChannelActive`       | Инициирует вызов `channelActive()` (подключен) на следующем `ChannelInboundHandler`.                                                                  |

При использовании API `ChannelHandlerContext`, пожалуйста, имейте в виду следующие моменты:

- `ChannelHandlerContext`, **связанный с `ChannelHandler`, НИКОГДА НЕ МЕНЯЕТСЯ**, поэтому безопасно кешировать ссылку на него.
    
- Методы `ChannelHandlerContext`, как мы объясняли в начале этого раздела, включают более короткий поток событий, чем одноимённые методы, доступные в других классах. Это следует использовать, где это возможно, для обеспечения максимальной производительности.

В следующем листинге вы получаете ссылку на `Channel` из `ChannelHandlerContext`. Вызов `write()` на `Channel` вызывает событие записи, которое проходит через весь конвейер.

**Листинг 6.6. Доступ к Channel из ChannelHandlerContext**

ChannelHandlerContext ctx = ...;

ChannelHandlerContext ctx = ...;

// (1) Получает ссылку на ChannelPipeline, связанный с ChannelHandlerContext
ChannelPipeline pipeline = ctx.pipeline(); 

// (2) Записывает буфер через ChannelPipeline
pipeline.write(

---
**Почему вы хотите распространять событие, начиная с определённой точки в ChannelPipeline?**
- Чтобы уменьшить накладные расходы на передачу события через `ChannelHandler`, которые не заинтересованы в нём.
    
- Чтобы предотвратить обработку события обработчиками, которые были бы заинтересованы в нём.


**Чтобы начать обработку с определённого** `ChannelHandler`, вы должны ссылаться на `ChannelHandlerContext`, связанный с `ChannelHandler` перед этим. **Этот `ChannelHandlerContext` вызовет `ChannelHandler`, который следует за тем, с которым он связан.**

Следующий листинг и рисунок 6.6 иллюстрируют это использование.

**Листинг 6.8. Вызов ChannelHandlerContext.write()**
```java

ChannelHandlerContext ctx = ...;    

// (2) write() отправляет буфер следующему ChannelHandler   
ctx.write(Unpooled.copiedBuffer("Netty in Action", CharsetUtil.UTF_8)); 
```

Как показано на рисунке 6.6, сообщение проходит через `ChannelPipeline`, начиная со следующего `ChannelHandler`, минуя все предыдущие.

### 6.3.2 Продвинутое использование ChannelHandler и ChannelHandlerContext

Как вы видели в листинге 6.6, вы можете получить ссылку на содержащий `ChannelPipeline`, вызвав метод `pipeline()` у `ChannelHandlerContext`. 

Это позволяет во время выполнения манипулировать `ChannelHandler` конвейера, что может быть использовано для реализации сложных архитектур. **Например, вы могли бы добавить `ChannelHandler` в конвейер ДЛЯ ПОДДЕРЖКИ ДИНАМИЧЕСКОГО ИЗМЕНЕНИЯ ПРОТОКОЛА.**

Другие продвинутые варианты использования могут поддерживаться кешированием ссылки на `ChannelHandlerContext` для последующего использования, которое может происходить вне любых методов `ChannelHandler` и даже может исходить из другого потока. Этот листинг показывает этот шаблон, используемый для инициирования события.

**Листинг 6.9. Кеширование ChannelHandlerContext**
```java
public class WriteHandler extends ChannelHandlerAdapter {
    private ChannelHandlerContext ctx;

    @Override
    public void handlerAdded(ChannelHandlerContext ctx) {
        this.ctx = ctx;  // (1) Сохраняет ссылку на ChannelHandlerContext для последующего использования
    }

    public void send(String msg) {
        ctx.writeAndFlush(msg); // (2) Отправляет сообщение, используя ранее сохранённый ChannelHandlerContext
    }
}
```

Поскольку `ChannelHandler` может принадлежать более чем одному `ChannelPipeline`, он может быть связан с несколькими экземплярами `ChannelHandlerContext`. 

> [!NOTE] `@Sharable` - для указания, что этот обработчик может использоваться в нескольких пайплайнах
> `ChannelHandler`, предназначенный для такого использования, должен быть аннотирован `@Sharable`; в противном случае попытка добавить его более чем в один `ChannelPipeline` вызовет исключение. 

Очевидно, что для безопасного использования с несколькими одновременными каналами (то есть соединениями) такой `ChannelHandler` должен быть потокобезопасным (thread-safe).

Этот листинг показывает правильную реализацию этого шаблона.

**Листинг 6.10. Совместно используемый ChannelHandler (Sharable)**
```java
@Sharable                                                              // (1) Аннотирует @Sharable
public class SharableHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        System.out.println("Channel read message: " + msg);
        ctx.fireChannelRead(msg);                                       // (2) Логирует вызов метода и передаёт следующему ChannelHandler
    }
}
```

Предыдущая реализация `ChannelHandler` соответствует всем требованиям для включения в несколько конвейеров, а именно: она аннотирована `@Sharable` и не хранит никакого состояния. И наоборот, код в листинге 6.11 вызовет проблемы.

**Листинг 6.11. Неверное использование @Sharable**
```java
@Sharable                                                              // (1) Аннотирует @Sharable
public class UnsharableHandler extends ChannelInboundHandlerAdapter {
    private int count;  // (2) Состояние — переменная экземпляра

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        count++;  // (3) Увеличивает поле count
        System.out.println(
	        "channelRead(...) called the " + count + " time");
        ctx.fireChannelRead(msg);
    }
}
```

Проблема с этим кодом в том, что он имеет состояние, а именно переменную экземпляра `count`, которая отслеживает количество вызовов метода. Добавление экземпляра этого класса в `ChannelPipeline` с большой вероятностью вызовет ошибки при одновременном доступе из нескольких каналов (конечно, этот простой случай можно было бы исправить, сделав `channelRead()` синхронизированным).

**Резюмируя:** используйте `@Sharable`, ==только если вы уверены, что ваш `ChannelHandler` является потокобезопасным==.

> **ЗАЧЕМ ДЕЛАТЬ CHANNELHANDLER СОВМЕСТНО ИСПОЛЬЗУЕМЫМ?** Распространённой причиной установки одного `ChannelHandler` в несколько `ChannelPipeline` является сбор статистики по нескольким каналам.
