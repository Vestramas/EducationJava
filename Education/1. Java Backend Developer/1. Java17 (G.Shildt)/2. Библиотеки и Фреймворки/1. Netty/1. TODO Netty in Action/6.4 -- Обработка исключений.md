## 6.4 Обработка исключений (Exception handling)

Обработка исключений является важной частью любого серьёзного приложения, и к ней можно подходить разными способами. 

Соответственно, *Netty предоставляет несколько вариантов обработки исключений*, возникающих во время входящей или исходящей обработки. Этот раздел поможет вам понять, как спроектировать подход, который наилучшим образом соответствует вашим потребностям.

---
### 6.4.1 Обработка входящих исключений (Handling inbound exceptions)

Если ***во время обработки входящего события возникает исключение***, **оно НАЧИНАЕТ РАСПРОСТРАНЯТЬСЯ ПО `ChannelPipeline`, начиная с точки в `ChannelInboundHandler`, где оно было инициировано.** 

Чтобы обработать такое входящее исключение, ==вам нужно переопределить следующий метод в вашей реализации== `ChannelInboundHandler`:
```java
public void exceptionCaught(
	ChannelHandlerContext ctx, 
	Throwable cause
) throws Exception
```

Следующий листинг показывает простой пример, который закрывает канал и печатает трассировку стека исключения.

**Листинг 6.12. Базовая обработка входящих исключений**
```java
public class InboundExceptionHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void exceptionCaught(
	    ChannelHandlerContext ctx, 
	    Throwable cause
	) {
        cause.printStackTrace();
        ctx.close();
    }
}
```


> [!NOTE] СОВЕТ
> Размещай обработчик с методом обработки ошибки ПОСЛЕДНИМ в пайплайне (внимательно смотри индекс!!!), так как исключение в любом случае будет прокидываться дальше по цепочке 

Поскольку **исключение будет продолжать распространяться во входящем направлении** (как и все входящие события), `ChannelInboundHandler`, **реализующий вышеуказанную логику, ОБЫЧНО ПОМЕЩАЕТСЯ ПОСЛЕДНИМ в** `ChannelPipeline`. 

Это гарантирует, что все входящие исключения всегда будут обработаны, независимо от того, где в `ChannelPipeline` они могут возникнуть.


То, как вы должны реагировать на исключение, скорее всего, будет специфичным для вашего приложения. Вы можете захотеть закрыть канал (и соединения) или попытаться восстановиться. Если вы не реализуете никакой обработки входящих исключений (или не потребляете исключение), Netty залогирует факт, что исключение не было обработано.

**Резюмируя:**
- Реализация `ChannelHandler.exceptionCaught()` ==по умолчанию передаёт текущее исключение следующему обработчику в конвейере.==
    
- Если исключение достигает конца конвейера, оно логируется как необработанное (unhandled).
    
- Чтобы определить пользовательскую обработку, вы переопределяете `exceptionCaught()`. Затем вы сами решаете, распространять ли исключение за пределы этой точки.

### 6.4.2 Обработка исходящих исключений (Handling outbound exceptions)

Варианты обработки нормального завершения и исключений в исходящих операциях основаны на следующих механизмах уведомления:

- **Каждая исходящая операция возвращает** `ChannelFuture`. `ChannelFutureListener`, зарегистрированные с `ChannelFuture`, уведомляются об успехе или ошибке при завершении операции.
    
- **Почти все методы `ChannelOutboundHandler` получают экземпляр `ChannelPromise`.** Как подкласс `ChannelFuture`, `ChannelPromise` также может иметь назначенные слушатели для асинхронного уведомления. 
  Но `ChannelPromise` также имеет доступные для записи методы, которые обеспечивают немедленное уведомление:
    - `ChannelPromise setSuccess();`
        
    - `ChannelPromise setFailure(Throwable cause);`
    

Добавление `ChannelFutureListener` сводится к вызову `addListener(ChannelFutureListener)` на экземпляре `ChannelFuture`, и есть два способа сделать это. 

**Наиболее часто используемый — ВЫЗВАТЬ `addListener()` на `ChannelFuture`, КОТОРЫЙ ВОЗВРАЩАЕТСЯ ИСХОДЯЩЕЙ ОПЕРАЦИЕЙ** (например, `write()`).

Следующий листинг использует этот подход для добавления `ChannelFutureListener`, который напечатает трассировку стека, а затем закроет канал.

**Листинг 6.13. Добавление ChannelFutureListener к ChannelFuture**
```java
ChannelFuture future = channel.write(someMessage);

future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture f) {
        if (!f.isSuccess()) {
            f.cause().printStackTrace();
            f.channel().close();
        }
    }
});
```

Второй вариант — добавить `ChannelFutureListener` к `ChannelPromise`, который передаётся в качестве аргумента методам `ChannelOutboundHandler`. Код, показанный далее, будет иметь тот же эффект, что и предыдущий листинг.

**Листинг 6.14. Добавление ChannelFutureListener к ChannelPromise**
```java
public class OutboundExceptionHandler extends ChannelOutboundHandlerAdapter {
    @Override
    public void write(
	    ChannelHandlerContext ctx, 
	    Object msg, 
	    ChannelPromise promise
	) {
        promise.addListener(new ChannelFutureListener() {
            @Override
            public void operationComplete(ChannelFuture f) {
                if (!f.isSuccess()) {
                    f.cause().printStackTrace();
                    f.channel().close();
                }
            }
        });
    }
}
```

> **Доступные для записи методы ChannelPromise**
> 
> Вызывая `setSuccess()` и `setFailure()` на `ChannelPromise`, вы можете сделать статус операции известным, как только метод `ChannelHandler` вернёт управление вызывающему коду.

**Почему стоит выбрать один подход вместо другого?** Для детальной обработки исключения вы, вероятно, найдёте более подходящим добавление `ChannelFutureListener` при вызове исходящей операции, как показано в листинге 6.13. Для менее специализированного подхода к обработке исключений вы можете найти пользовательскую реализацию `ChannelOutboundHandler`, показанную в листинге 6.14, более простой.

**Что произойдёт, если ваш `ChannelOutboundHandler` сам выбросит исключение?** В этом случае Netty сам уведомит всех слушателей, которые были зарегистрированы с соответствующим `ChannelPromise`.
