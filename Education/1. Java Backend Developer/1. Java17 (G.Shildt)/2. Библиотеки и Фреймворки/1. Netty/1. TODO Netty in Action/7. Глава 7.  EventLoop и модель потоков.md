## 7.1 Обзор модели потоков (Threading model overview)

В этом разделе мы представим модели потоков в целом, а затем обсудим прошлую и настоящую модели потоков Netty, рассмотрев преимущества и ограничения каждой.

Как мы отмечали в начале главы, модель потоков определяет, как будет выполняться код. Поскольку мы всегда должны остерегаться возможных побочных эффектов конкурентного выполнения, важно понимать последствия применяемой модели (существуют также однопоточные модели). Игнорирование этих вопросов и просто надежда на лучшее равносильны азартной игре — с odds определённо не в вашу пользу.

*Поскольку компьютеры с несколькими ядрами или CPU стали обычным явлением, большинство современных приложений используют сложные методы многопоточности* для эффективного использования системных ресурсов. В отличие от этого, наш подход к многопоточности в ранние дни Java был немногим больше, чем создание и запуск новых потоков по требованию для выполнения конкурентных единиц работы — примитивный подход, который плохо работает под высокой нагрузкой. *Затем Java 5 представила API Executor*, чьи пулы потоков значительно улучшили производительность за счёт кеширования и повторного использования потоков.

**Базовый шаблон пулинга потоков можно описать так:**
- Поток выбирается из списка свободных в пуле и назначается для выполнения переданной задачи (реализации `Runnable`).
    
- Когда задача завершается, поток возвращается в список и становится доступным для повторного использования.

## 7.2 Интерфейс EventLoop - единичный поток, который крутится в цикле ожидания событий

Запуск задач для обработки событий, возникающих в течение жизни соединения, — это базовая функция любого сетевого фреймворка. 

Соответствующая программная конструкция часто называется **event loop** (цикл событий) — термин, который Netty использует с интерфейсом `io.netty.channel.EventLoop`.

==Основная идея event loop проиллюстрирована в следующем листинге==, где каждая задача является экземпляром `Runnable` (как на рисунке 7.1).

**Листинг 7.1. Выполнение задач в event loop**
```java
while (!terminated) {
    List<Runnable> readyEvents = blockUntilEventsReady(); // (1) Блокируется, пока не появятся события, готовые к выполнению
    
    for (Runnable ev: readyEvents) {
        ev.run(); // (2) Перебирает и запускает все события
    }
}
```

`EventLoop` от Netty является частью совместного дизайна, использующего два фундаментальных API: конкурентности и сетевого взаимодействия. 
- Во-первых, пакет `io.netty.util.concurrent` расширяет JDK-пакет `java.util.concurrent` для предоставления исполнителей потоков (thread executors). 
- Во-вторых, классы в пакете `io.netty.channel` расширяют их для взаимодействия с событиями `Channel`. Получившаяся иерархия классов показана на рисунке 7.2.

![[Pasted image 20260611220416.png]]
**Рисунок 7.2. Иерархия классов EventLoop**


В этой модели `EventLoop` ==УПРАВЛЯЕТСЯ РОВНО ОДНИМ ПОТОКОМ==, который никогда не меняется, и задачи (`Runnable` или `Callable`) могут быть переданы непосредственно реализациям `EventLoop` для немедленного или запланированного выполнения. 

В зависимости от конфигурации и доступных ядер может быть создано несколько `EventLoop` для оптимизации использования ресурсов, и один `EventLoop` может быть назначен для обслуживания нескольких каналов.

Обратите внимание, что `EventLoop` от Netty, хотя и расширяет `ScheduledExecutorService`, определяет только один метод — `parent()`. Этот метод, показанный в следующем фрагменте кода, предназначен для возврата ссылки на `EventLoopGroup`, к которой принадлежит текущий экземпляр реализации `EventLoop`.
```java
public interface EventLoop extends EventExecutor, EventLoopGroup {
    @Override
    EventLoopGroup parent();
}
```

> **ПОРЯДОК ВЫПОЛНЕНИЯ СОБЫТИЙ/ЗАДАЧ:** События и задачи выполняются в порядке FIFO (первым пришёл — первым вышел). Это устраняет возможность повреждения данных, гарантируя, что байтовое содержимое обрабатывается в правильном порядке.

### 7.2.1 I/O и обработка событий в Netty 4

Как мы подробно описали в главе 6, события, инициируемые операциями I/O, проходят через `ChannelPipeline`, в котором установлен один или несколько `ChannelHandler`. Затем вызовы методов, распространяющие эти события, могут быть перехвачены `ChannelHandler`, и события обработаны по мере необходимости.

Природа события обычно определяет, как его следует обрабатывать; оно может передавать данные из сетевого стека в ваше приложение, делать обратное или делать что-то совсем другое. Но логика обработки событий должна быть достаточно общей и гибкой, чтобы охватить все возможные случаи использования. Поэтому в Netty 4 **все операции I/O и события обрабатываются потоком, который назначен `EventLoop`.**

Это отличается от модели, которая использовалась в Netty 3. В следующем разделе мы обсудим более раннюю модель и причины, по которым она была заменена.

---
### 7.2.2 Операции I/O в Netty 3

Модель потоков, использовавшаяся в предыдущих релизах, гарантировала только то, что ВХОДЯЩИЕ (ранее называвшиеся _upstream_) события будут выполняться в так называемом I/O потоке (соответствующем `EventLoop` Netty 4). 

Все *ИСХОДЯЩИЕ* (_downstream_) *события обрабатывались вызывающим потоком, который мог быть I/O потоком или любым другим.* Это казалось хорошей идеей поначалу, но оказалось проблематичным из-за необходимости тщательной синхронизации исходящих событий в `ChannelHandler`. 
> Короче говоря, невозможно было гарантировать, что несколько потоков не попытаются одновременно получить доступ к исходящему событию. 
> 
> Это могло произойти, например, если бы вы инициировали одновременные downstream события для одного и того же канала, вызывая `Channel.write()` в разных потоках.

Другой негативный побочный эффект возникал, когда входящее событие инициировалось в результате исходящего события. Когда `Channel.write()` вызывает исключение, вам нужно сгенерировать и запустить событие `exceptionCaught`. Но в модели Netty 3, поскольку это входящее событие, вы в итоге выполняли код в вызывающем потоке, затем передавали событие I/O потоку для выполнения, с consequent дополнительным переключением контекста.

Модель потоков, принятая в Netty 4, решает эти проблемы, обрабатывая всё, что происходит в данном `EventLoop`, в одном потоке. Это обеспечивает более простую архитектуру выполнения и устраняет необходимость синхронизации в `ChannelHandler` (за исключением тех, которые могут быть разделены между несколькими каналами).

Теперь, когда вы понимаете роль `EventLoop`, давайте посмотрим, как задачи планируются для выполнения.

## 7.3 Планирование задач (Task scheduling)

Иногда вам потребуется запланировать задачу для последующего (отложенного) или периодического выполнения. Например, вы можете захотеть зарегистрировать задачу, которая запустится через пять минут после подключения клиента. 
> Распространённый случай использования — *отправка heartbeat-сообщения удалённому узлу, чтобы проверить, живо ли ещё соединение.* Если ответа нет, вы знаете, что можете закрыть канал.

В следующих разделах мы покажем вам, как планировать задачи с помощью как основного Java API, так и `EventLoop` от Netty. Затем мы рассмотрим внутренности реализации Netty и обсудим её преимущества и ограничения.

---
### 7.3.1 JDK API планирования

***До Java 5 планирование задач было построено на*** `java.util.Timer`, который использует фоновый поток и имеет те же ограничения, что и стандартные потоки. 

**Впоследствии JDK предоставил пакет `java.util.concurrent`, который определяет интерфейс** `ScheduledExecutorService`. 

В таблице 7.1 показаны соответствующие фабричные методы `java.util.concurrent.Executors`.

**Таблица 7.1. Фабричные методы java.util.concurrent.Executors**

|Методы|Описание|
|---|---|
|`newScheduledThreadPool(int corePoolSize)`  <br>`newScheduledThreadPool(int corePoolSize, ThreadFactory threadFactory)`|Создаёт `ScheduledThreadExecutorService`, который может планировать команды для выполнения после задержки или для периодического выполнения. Использует аргумент `corePoolSize` для расчёта количества потоков.|
|`newSingleThreadScheduledExecutor()`  <br>`newSingleThreadScheduledExecutor(ThreadFactory threadFactory)`|Создаёт `ScheduledThreadExecutorService`, который может планировать команды для выполнения после задержки или для периодического выполнения. Использует один поток для выполнения запланированных задач.|

Хотя вариантов не так много, предоставленных достаточно для большинства случаев использования. Следующий листинг показывает, как использовать `ScheduledExecutorService` для выполнения задачи после 60-секундной задержки.

**Листинг 7.2. Планирование задачи с помощью ScheduledExecutorService**
```java
ScheduledExecutorService executor = Executors.newScheduledThreadPool(10); // (1) Создаёт ScheduledExecutorService с пулом из 10 потоков

ScheduledFuture<?> future = executor.schedule(                            // (3) Планирует задачу для выполнения через 60 секунд от текущего момента
    new Runnable() {                                                       // (2) Создаёт Runnable для планирования последующего выполнения
        @Override
        public void run() {
            System.out.println("60 seconds later");                        // Сообщение, которое будет напечатано задачей
        }
    }, 60, TimeUnit.SECONDS);

// ...

executor.shutdown();                                                       // (4) Завершает работу ScheduledExecutorService для освобождения ресурсов после завершения задачи
```

Хотя API `ScheduledExecutorService` прост, под высокой нагрузкой он может создавать затраты на производительность. В следующем разделе мы увидим, как Netty предоставляет ту же функциональность с большей эффективностью.

---
### 7.3.2 Планирование задач с использованием EventLoop (и ScheduledFuture)

Реализация `ScheduledExecutorService` ***имеет ограничения***, *такие как создание дополнительных потоков в рамках управления пулом.* 

Это может стать узким местом, если много задач планируется агрессивно. Netty решает эту проблему, реализуя планирование с использованием `EventLoop` канала, как показано в следующем листинге.

**Листинг 7.3. Планирование задачи с помощью EventLoop**
```java
Channel ch = ...

ScheduledFuture<?> future = ch.eventLoop().schedule(
    new Runnable() {                                                      // (1) Создаёт Runnable для планирования последующего выполнения
        @Override
        public void run() {
            System.out.println("60 seconds later");                        // (2) Код для выполнения
        }
    }, 60, TimeUnit.SECONDS);                                             // (3) Планирует выполнение через 60 секунд от текущего момента
```

Через 60 секунд экземпляр `Runnable` будет выполнен `EventLoop`, назначенным каналу. **Чтобы запланировать задачу для выполнения каждые 60 секунд, используйте** `scheduleAtFixedRate()`, как показано далее.

**Листинг 7.4. Планирование повторяющейся задачи с помощью EventLoop**
```java
Channel ch = ...
ScheduledFuture<?> future = ch.eventLoop().scheduleAtFixedRate(           // (2) Планирует выполнение через 60 секунд и каждые 60 секунд после этого
    new Runnable() {                                                      // (1) Создаёт Runnable для планирования последующего выполнения
        @Override
        public void run() {
            System.out.println("Run every 60 seconds");                   // Код для выполнения
        }
    }, 60, 60, TimeUnit.SECONDS);                                         // (3) Это будет выполняться до тех пор, пока ScheduledFuture не будет отменён
```

Как мы отмечали ранее, `EventLoop` **от Netty расширяет `ScheduledExecutorService` (см. рисунок 7.2), поэтому он ПРЕДОСТАВЛЯЕТ ВСЕ МЕТОДЫ, ДОСТУПНЫЕ В РЕАЛИЗАЦИИ JDK**, включая `schedule()` и `scheduleAtFixedRate()`, использованные в предыдущих примерах. 
> Полный список всех операций можно найти в Javadoc для `ScheduledExecutorService`.

Чтобы отменить или проверить состояние выполнения, используйте `ScheduledFuture`, который возвращается для каждой асинхронной операции. Этот листинг показывает простую операцию отмены.

**Листинг 7.5. Отмена задачи с помощью ScheduledFuture**
```java
ScheduledFuture<?> future = ch.eventLoop().scheduleAtFixedRate(...);      // (1) Планирует задачу и получает возвращённый ScheduledFuture
// ...
boolean mayInterruptIfRunning = false;
future.cancel(mayInterruptIfRunning);                                     // (2) Отменяет задачу, что предотвращает её повторный запуск
```

Эти примеры иллюстрируют выигрыш в производительности, который может быть достигнут за счёт использования возможностей планирования Netty. Они зависят, в свою очередь, от нижележащей модели потоков, которую мы рассмотрим далее.

## 7.4 Детали реализации (Implementation details)

В этом разделе более подробно рассматриваются основные элементы модели потоков Netty и реализации планирования. Мы также упомянем ограничения, о которых следует знать, а также области продолжающейся разработки.

### 7.4.1 Управление потоками (Thread management)

Превосходная производительность модели потоков Netty зависит от определения идентичности текущего выполняемого потока, то есть является ли он тем, который назначен текущему каналу и его `EventLoop` 
(*напомним, что* `EventLoop` *отвечает за обработку всех событий для канала в течение его времени жизни*).

**Если вызывающий поток является потоком `EventLoop`,** блок кода выполняется. **В противном случае** `EventLoop` планирует задачу для последующего выполнения и помещает **её во внутреннюю очередь.** Когда `EventLoop` в следующий раз обрабатывает свои события, он выполняет задачи из очереди. Это объясняет, как любой поток может напрямую взаимодействовать с каналом без необходимости синхронизации в `ChannelHandler`.

Обратите внимание, что каждый `EventLoop` и*меет свою собственную очередь задач, независимую от очереди любого другого* `EventLoop`. Рисунок 7.3 показывает логику выполнения, используемую `EventLoop` для планирования задач. Это критический компонент модели потоков Netty.

### 7.4.2 Распределение EventLoop/потоков (EventLoop/thread allocation)

`EventLoop`, **которые обслуживают I/O и события для каналов, содержатся в** `EventLoopGroup`. Способ создания и назначения `EventLoop` варьируется в зависимости от реализации транспорта.

---
#### **АСИНХРОННЫЕ ТРАНСПОРТЫ**

**Асинхронные реализации используют только несколько** `EventLoop` (и связанных с ними потоков), и в текущей модели они могут быть разделены между каналами. Это позволяет обслуживать множество каналов наименьшим возможным количеством потоков, вместо назначения одного потока на канал.

Рисунок 7.4 отображает `EventLoopGroup` **фиксированного размера из трёх** `EventLoop` (каждый управляется одним потоком). 

`EventLoop` (и их потоки) выделяются непосредственно при создании `EventLoopGroup`, чтобы гарантировать их доступность, когда они понадобятся.

![[Pasted image 20260611223329.png]]
**Рисунок 7.4. Распределение EventLoop для неблокирующих транспортов (например, NIO и AIO)**

`EventLoopGroup` отвечает за выделение `EventLoop` каждому вновь созданному каналу. **В текущей реализации используется подход round-robin** для достижения сбалансированного распределения, и один и тот же `EventLoop` может быть назначен нескольким каналам (это может измениться в будущих версиях).

**Как только каналу назначен `EventLoop`, он будет использовать этот `EventLoop` (и связанный с ним поток)** на протяжении всего своего времени жизни. 
> Имейте это в виду, поскольку это освобождает вас от проблем с потокобезопасностью и синхронизацией в ваших реализациях `ChannelHandler`.

Также имейте в виду ***последствия распределения `EventLoop` для использования `ThreadLocal`.*** 
***Поскольку `EventLoop` обычно обслуживает более одного канала, `ThreadLocal` будет одинаковым для всех связанных каналов.*** 

Это делает его плохим выбором для реализации такой функции, как отслеживание состояния. Однако в контексте без сохранения состояния он всё ещё может быть полезен для разделения тяжёлых или дорогих объектов или даже событий между каналами.

![[Pasted image 20260611223535.png]]
**Рисунок 7.5. Распределение EventLoop для блокирующих транспортов (например, OIO)**

Здесь один `EventLoop` (и его поток) назначается каждому каналу. Вы могли столкнуться с этим шаблоном, если разрабатывали приложения, использующие реализацию блокирующего I/O в пакете `java.io`.
