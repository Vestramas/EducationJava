**В этой главе рассматриваются:**
- Бутстраппинг клиентов и серверов
    
- Бутстраппинг клиентов изнутри канала
    
- Добавление ChannelHandler
    
- Использование ChannelOptions и атрибутов
    

После глубокого изучения `ChannelPipeline`, `ChannelHandler` и классов кодеков ваш следующий вопрос, вероятно: ***«Как все эти части складываются в работающее приложение?»***

Ответ? **«Бутстраппинг»**. До сих пор мы использовали этот термин несколько расплывчато, и пришло время определить его более точно. Проще говоря, **бутстраппинг приложения — это процесс его настройки для запуска** — хотя детали процесса могут быть не такими простыми, как его определение, особенно в сетевых приложениях.

В соответствии со своим подходом к архитектуре приложений, ***Netty обрабатывает бутстраппинг таким образом, что изолирует ваше приложение (будь то клиент или сервер) от сетевого уровня.*** Как вы увидите, все компоненты фреймворка соединяются и активируются в фоновом режиме. Бутстраппинг — это недостающий кусочек пазла, который мы собирали; когда вы вставите его на место, ваше приложение Netty будет завершено.

## 8.1 Классы бутстраппинга (Bootstrap classes)

**Иерархия классов бутстраппинга состоит из одного абстрактного родительского класса** *и двух конкретных подклассов бутстрапа,* как показано на рисунке 8.1.

![[Pasted image 20260612081952.png]]
**Рисунок 8.1. Иерархия классов бутстраппинга**

Вместо того чтобы думать о конкретных классах как о серверном и клиентском бутстрапах, полезно помнить о различных функциях приложения, которые они предназначены поддерживать. 

А именно, **сервер выделяет родительский канал для приёма соединений от клиентов и создания дочерних каналов для общения с ними**, *тогда как клиенту, скорее всего, потребуется только один, не-родительский канал для всех сетевых взаимодействий.* 
> (Как мы увидим, это также применимо к бессоединённым транспортам, таким как *UDP, поскольку они не требуют канала для каждого соединения.*)

Некоторые из компонентов Netty, которые мы изучали в предыдущих главах, участвуют в процессе бутстраппинга, и некоторые из них используются как в клиентах, так и в серверах. 

Шаги бутстраппинга, общие для обоих типов приложений, обрабатываются `AbstractBootstrap`, тогда как специфичные для клиентов или серверов обрабатываются соответственно `Bootstrap` или `ServerBootstrap`.

В оставшейся части этой главы мы подробно рассмотрим эти два класса, начиная с менее сложного — `Bootstrap`.

> **Почему классы бутстраппинга реализуют Cloneable?**
> 
> Иногда вам нужно создать несколько каналов с похожими или идентичными настройками. Чтобы поддержать этот шаблон без необходимости создавать и настраивать новый экземпляр бутстрапа для каждого канала, `AbstractBootstrap` помечен как `Cloneable`. Вызов `clone()` на уже настроенном бутстрапе вернёт другой экземпляр бутстрапа, готовый к немедленному использованию.
> 
> Обратите внимание, что *при этом создаётся только поверхностная копия (shallow copy)* `EventLoopGroup` бутстрапа, поэтому последняя будет разделяться между всеми клонированными каналами. Это приемлемо, поскольку клонированные каналы часто являются недолговечными; типичный случай — канал, созданный для выполнения HTTP-запроса.

Полное объявление `AbstractBootstrap` выглядит так:

```java
public abstract class AbstractBootstrap
    <B extends AbstractBootstrap<B,C>, C extends Channel>
```

В этой сигнатуре подкласс `B` является параметром типа для суперкласса, так что ссылка на экземпляр времени выполнения может быть возвращена для поддержки "цепочки вызовов" (так называемый _fluent syntax_).

Подклассы объявлены следующим образом:
```java
public class Bootstrap extends AbstractBootstrap<Bootstrap, Channel>
```
и
```java
public class ServerBootstrap 
	extends AbstractBootstrap<ServerBootstrap, ServerChannel>
```

### 8.2.1 Бутстраппинг клиента

Класс `Bootstrap` отвечает за создание каналов для клиентов и для приложений, использующих бессоединённые протоколы, как проиллюстрировано на рисунке 8.2.

![[Pasted image 20260612082531.png]]
**Рисунок 8.2. Процесс бутстраппинга**

Код в следующем листинге выполняет бутстраппинг клиента, использующего NIO TCP транспорт.

**Листинг 8.1. Бутстраппинг клиента**
```java
EventLoopGroup group = new NioEventLoopGroup();

// (1) Создаёт Bootstrap для создания и подключения новых клиентских каналов
Bootstrap bootstrap = new Bootstrap();  

// (2) Устанавливает EventLoopGroup, которая предоставляет EventLoop для обработки событий канала                              
bootstrap.group(group)  
	
	// (3) Указывает реализацию Channel для использования
    .channel(NioSocketChannel.class)       
    
	// (4) Устанавливает обработчик для событий канала и данных
    .handler(new SimpleChannelInboundHandler<ByteBuf>() {             
        @Override
        protected void channelRead0(
	        ChannelHandlerContext ctx, 
	        ByteBuf byteBuf
	    ) throws Exception 
	    {
            System.out.println("Received data");
        }
    });
    
ChannelFuture future = bootstrap.connect(
	// (5) Подключается к удалённому хосту
	new InetSocketAddress("www.manning.com", 80));
	
 
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture) throws Exception {
        if (channelFuture.isSuccess()) {
            System.out.println("Connection established");
        } else {
            System.err.println("Connection attempt failed");
            channelFuture.cause().printStackTrace();
        }
    }
});
```

Этот пример использует упомянутый ранее _fluent syntax_; методы (кроме `connect()`) объединены в цепочку ссылкой на экземпляр `Bootstrap`, которую каждый из них возвращает.

---
#### **Совместимость Channel и EventLoopGroup**

Следующий листинг каталога взят из пакета `io.netty.channel`. Вы можете видеть по именам пакетов и соответствующим префиксам имён классов, что существуют связанные реализации `EventLoopGroup` и `Channel` как для NIO, так и для OIO транспортов.

**Листинг 8.2. Совместимые EventLoopGroup и Channel**
```
channel
├─── nio
│       NioEventLoopGroup
├─── oio
│       OioEventLoopGroup
└─── socket
	├─── nio
	│       NioDatagramChannel
	│       NioServerSocketChannel
	│       NioSocketChannel
	├─── oio
	│       OioDatagramChannel
	│       OioServerSocketChannel
	│       OioSocketChannel
	
```

Эта совместимость должна поддерживаться; вы не можете смешивать компоненты с разными префиксами, например `NioEventLoopGroup` и `OioSocketChannel`. Следующий листинг показывает попытку сделать именно это.

**Листинг 8.3. НЕСОВМЕСТИМЫЕ Channel и EventLoopGroup**
```java
// (1) Указывает реализацию NIO EventLoopGroup
EventLoopGroup group = new NioEventLoopGroup(); 

// (2) Создаёт новый Bootstrap для создания новых клиентских каналов  
Bootstrap bootstrap = new Bootstrap();    

// (3) Устанавливает EventLoopGroup                      
bootstrap.group(group)      

	// (4) Указывает класс реализации OIO Channel
    .channel(OioSocketChannel.class)     
    
    // (5) Устанавливает обработчик для событий I/O канала и данных
    .handler(new SimpleChannelInboundHandler<ByteBuf>() {             
        @Override
        protected void channelRead0(ChannelHandlerContext ctx, ByteBuf byteBuf) throws Exception {
            System.out.println("Received data");
        }
    });
ChannelFuture future = bootstrap.connect(
	// (6) Пытается подключиться к удалённому узлу
	new InetSocketAddress("www.manning.com", 80)); 
	
future.syncUninterruptibly();
```

Этот код вызовет `IllegalStateException`, поскольку смешивает несовместимые транспорты:
```java
Exception in thread "main" java.lang.IllegalStateException: incompatible event loop type: io.netty.channel.nio.NioEventLoop
```

> **Подробнее об IllegalStateException**
> 
> При бутстраппинге, перед вызовом `bind()` или `connect()`, вы должны вызвать следующие методы для настройки необходимых компонентов:
> 
> - `group()`
>     
> - `channel()` или `channelFactory()`
>     
> - `handler()`
>     
> 
> Несоблюдение этого требования вызовет `IllegalStateException`. Вызов `handler()` особенно важен, поскольку он необходим для настройки `ChannelPipeline`.

### 8.3.2 Бутстраппинг сервера

## 8.4 Бутстраппинг клиентов из канала (Bootstrapping clients from a Channel)

Предположим, ***ваш сервер обрабатывает запрос клиента***, **который требует от него выступить в роли клиента по отношению к третьей системе**. (ПРОКСИРОВАНИЕ)

Это может произойти, когда приложение, например прокси-сервер, должно интегрироваться с существующими системами организации, такими как веб-сервисы или базы данных. 

В таких случаях вам **нужно будет выполнить бутстраппинг клиентского канала из `ServerChannel`.**

> **ПОЧЕМУ НЕ СОЗДАТЬ ОТДЕЛЬНЫЙ Bootstrap как было описано ранее?**
> 
> Вы могли бы создать новый `Bootstrap`, как описано в разделе 8.2.1, но это не самое эффективное решение, **так как потребовало бы определения другого `EventLoop`** для нового клиентского канала. 
> 
> Это **привело бы к созданию дополнительных потоков**, necessitating **переключение контекста при обмене данными** между принятым каналом и клиентским каналом.

**Лучшее решение — разделить `EventLoop` принятого канала**, п*ередав его методу* `group()` `Bootstrap`. 
Поскольку ***все каналы, назначенные `EventLoop`, используют один и тот же поток,*** *это позволяет избежать создания дополнительных потоков* и связанного с этим переключения контекста, упомянутого ранее. 

Это решение проиллюстрировано на рисунке 8.4.

![[Pasted image 20260612083530.png]]
**Рисунок 8.4. Разделяемый EventLoop между каналами**

Реализация разделения `EventLoop` включает установку `EventLoop` путём вызова метода `group()`, как показано в следующем листинге.

**Листинг 8.5. Бутстраппинг сервера с разделением EventLoop**
```java
// (1) Создаёт ServerBootstrap для создания SocketChannel и их привязки
ServerBootstrap bootstrap = new ServerBootstrap();  

// (2) Устанавливает EventLoopGroup, которые предоставляют EventLoop для обработки событий канала
bootstrap.group(new NioEventLoopGroup(), new NioEventLoopGroup())       
    // (3) Указывает реализацию Channel для использования
    .channel(NioServerSocketChannel.class)    
	
	// (4) Устанавливает ChannelInboundHandler для I/O и данных для принятых каналов
    .childHandler(new SimpleChannelInboundHandler<ByteBuf>() {          
        ChannelFuture connectFuture;
        @Override
        public void channelActive(ChannelHandlerContext ctx) throws Exception {
	        // (5) Создаёт Bootstrap для подключения к удалённому хосту
            Bootstrap bootstrap = new Bootstrap();                      
            bootstrap
	            .channel(NioSocketChannel.class)
	            // (6) Устанавливает обработчик для входящего I/O
                .handler(new SimpleChannelInboundHandler<ByteBuf>() {   
                    @Override
                    protected void channelRead0(
	                    ChannelHandlerContext ctx,
	                    ByteBuf in
	                ) throws Exception 
	                {
                        System.out.println("Received data");
                    }
                });
			// (7) Использует тот же EventLoop, что и у принятого канала
            bootstrap.group(ctx.channel().eventLoop());                  
            connectFuture = bootstrap.connect(
	            // (8) Подключается к удалённому узлу
	            new InetSocketAddress("www.manning.com", 80)); 
        }
        @Override
        protected void channelRead0(
	        ChannelHandlerContext ctx, 
	        ByteBuf byteBuf
	    ) throws Exception 
	    {
            if (connectFuture.isDone()) {
                // do something with the data
            }
        }
    });
// (9) Привязывает канал через настроенный Bootstrap
ChannelFuture future = bootstrap.bind(new InetSocketAddress(8080));     
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture) throws Exception {
        if (channelFuture.isSuccess()) {
            System.out.println("Server bound");
        } else {
            System.err.println("Bind attempt failed");
            channelFuture.cause().printStackTrace();
        }
    }
});
```

Тема, которую мы обсуждали в этом разделе, и представленное решение отражают общее правило при написании приложений Netty: **по возможности повторно используйте EventLoop, чтобы снизить затраты на создание потоков**.

## 8.5 Добавление нескольких ChannelHandler во время бутстраппинга (ChannelInitializer)

Во всех показанных нами примерах кода мы вызывали `handler()` или `childHandler()` во время бутстраппинга для добавления одного `ChannelHandler`. Этого может быть достаточно для простых приложений, но этого будет недостаточно для более сложных.

> Например, приложение, которое должно поддерживать несколько протоколов, будет иметь много `ChannelHandler`; альтернативой будет большой и неуклюжий класс.

Как вы неоднократно видели, вы можете развернуть столько `ChannelHandler`, сколько требуется, объединив их в цепочку в `ChannelPipeline`. **Но как это сделать, если во время бутстраппинга можно установить только один** `ChannelHandler`?

Именно для этого случая Netty предоставляет специальный подкласс `ChannelInboundHandlerAdapter`:
```java
public abstract class ChannelInitializer<C extends Channel> extends ChannelInboundHandlerAdapter
```

который определяет следующий метод:
```java
protected abstract void initChannel(C ch) throws Exception;
```

Этот метод предоставляет простой способ добавления нескольких `ChannelHandler` в `ChannelPipeline`. Вы просто предоставляете свою реализацию `ChannelInitializer` бутстрапу, и как только канал регистрируется со своим `EventLoop`, вызывается ваша версия `initChannel()`. После возврата из метода экземпляр `ChannelInitializer` удаляет сам себя из `ChannelPipeline`.

Следующий листинг определяет класс `ChannelInitializerImpl` и регистрирует его с помощью `childHandler()` бутстрапа. Вы можете видеть, что эта, на первый взгляд, сложная операция на самом деле довольно проста.

**Листинг 8.6. Бутстраппинг с использованием ChannelInitializer**
```java
// (1) Создаёт ServerBootstrap для создания и привязки новых каналов
ServerBootstrap bootstrap = new ServerBootstrap();    

// (2) Устанавливает EventLoopGroup, которая предоставляет EventLoop для обработки событий канала
bootstrap.group(new NioEventLoopGroup(), new NioEventLoopGroup())  
	// (3) Указывает реализацию Channel     
    .channel(NioServerSocketChannel.class)                              
    
    // (4) Регистрирует экземпляр ChannelInitializerImpl для настройки ChannelPipeline
    .childHandler(new ChannelInitializerImpl()
);  

// (5) Привязывает канал 
ChannelFuture future = bootstrap.bind(new InetSocketAddress(8080));     
future.sync();

// (6) Пользовательская реализация ChannelInitializerImpl для настройки ChannelPipeline
final class ChannelInitializerImpl extends ChannelInitializer<Channel> { 
    @Override
    // (7) Добавляет требуемые обработчики в ChannelPipeline
    protected void initChannel(Channel ch) throws Exception {           
        ChannelPipeline pipeline = ch.pipeline();
        
        pipeline.addLast(new HttpClientCodec());
        pipeline.addLast(new HttpObjectAggregator(Integer.MAX_VALUE));
    }
}
```

Если ваше приложение использует много `ChannelHandler`, определите свой собственный `ChannelInitializer`, чтобы установить их в конвейере.

## 8.6 Использование ChannelOptions и атрибутов Netty

***Ручная настройка каждого канала при его создании может стать довольно утомительной***. 

К счастью, вам не нужно этого делать. **Вместо этого вы можете использовать `option()` для применения `ChannelOption` к бутстрапу**. 

**Значения, которые вы предоставляете, будут автоматически применены ко всем каналам, созданным в этом бутстрапе.** Доступные `ChannelOption` включают низкоуровневые детали соединения, ==такие как параметры keep-alive или таймаута, а также настройки буферов.==

Приложения Netty часто интегрируются с проприетарным программным обеспечением организации, и такие компоненты, как `Channel`, могут даже использоваться вне обычного жизненного цикла Netty. 
***В случае, если некоторые из обычных свойств и данных недоступны,*** **Netty предлагает абстракцию** `AttributeMap` — *коллекцию*, предоставляемую классами канала и бутстрапа, *и `AttributeKey<T>` — обобщённый класс **для вставки и извлечения значений атрибутов***. С помощью этих инструментов вы можете безопасно связывать любые виды данных как с клиентскими, так и с серверными каналами.


Рассмотрим, **например, серверное приложение, которое отслеживает взаимосвязь между пользователями и каналами.** 
Это может быть достигнуто ==путём хранения идентификатора пользователя в качестве атрибута канала.== Похожая техника может использоваться для маршрутизации сообщений пользователям на основе их ID или для закрытия канала при низкой активности.

Следующий листинг показывает, как вы можете использовать `ChannelOptions` для настройки канала и атрибут для хранения целочисленного значения.

**Листинг 8.7. Использование атрибутов**
```java
// (1) Создаёт AttributeKey для идентификации атрибута
final AttributeKey<Integer> id = new AttributeKey<Integer>("ID");  

// (2) Создаёт Bootstrap для создания клиентских каналов и их подключения
Bootstrap bootstrap = new Bootstrap();

// (3) Устанавливает EventLoopGroup, которая предоставляет EventLoop для обработки событий канала                                  
bootstrap.group(new NioEventLoopGroup()) 
	// (4) Указывает реализацию Channel
    .channel(NioSocketChannel.class)    
      
    // (5) Устанавливает ChannelInboundHandler для обработки I/O и данных канала
    .handler(new SimpleChannelInboundHandler<ByteBuf>() {               
        @Override
        public void channelRegistered(ChannelHandlerContext ctx) throws Exception {
	        // (6) Получает атрибут с помощью AttributeKey и его значение
            // do something with the idValue
            Integer idValue = ctx.channel().attr(id).get();              
        }
        @Override
        protected void channelRead0(
	        ChannelHandlerContext ctx, 
	        ByteBuf byteBuf
	    ) throws Exception 
	    {
            System.out.println("Received data");
        }
    });

// (7) Устанавливает ChannelOptions, которые будут установлены на созданных каналах при connect() или bind()   
bootstrap
	.option(ChannelOption.SO_KEEPALIVE, true)
    .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 5000);   

// (8) Сохраняет атрибут id
bootstrap.attr(id, 123456);

// (9) Подключается к удалённому хосту с настроенным Bootstrap
ChannelFuture future = bootstrap.connect(
	new InetSocketAddress("www.manning.com", 80)); 
	
future.syncUninterruptibly();
```

## 8.7 Бутстраппинг DatagramChannel (UDP)

Предыдущие примеры кода бутстраппинга использовали `SocketChannel`, который основан на TCP, **но `Bootstrap` может использоваться и для бессоединённых протоколов**. 

Netty предоставляет для этой цели различные реализации `DatagramChannel`. 
==Единственное отличие в том, что вы вызываете не== `connect()`, ==а только ==`bind()`, как показано далее.

**Листинг 8.8. Использование Bootstrap с DatagramChannel**
```java
// (1) Создаёт Bootstrap для создания и привязки новых дейтаграммных каналов
Bootstrap bootstrap = new Bootstrap();

// (2) Устанавливает EventLoopGroup, которая предоставляет EventLoop для обработки событий канала                                   
bootstrap.group(new OioEventLoopGroup()) 

	// (3) Указывает реализацию Channel                                
    .channel(OioDatagramChannel.class)     
    
    // (4) Устанавливает ChannelInboundHandler для обработки I/O и данных канала
    .handler(new SimpleChannelInboundHandler<DatagramPacket>() {         
        @Override
        public void channelRead0(
	        ChannelHandlerContext ctx, 
	        DatagramPacket msg
	    ) throws Exception 
	    {
            // Do something with the packet
        }
    });
    
// (5) Вызывает bind(), потому что протокол бессоединённый  
ChannelFuture future = bootstrap.bind(new InetSocketAddress(0));  
       
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture) throws Exception {
        if (channelFuture.isSuccess()) {
            System.out.println("Channel bound");
        } else {
            System.err.println("Bind attempt failed");
            channelFuture.cause().printStackTrace();
        }
    }
});
```

## 8.8 Завершение работы (Shutdown)

Бутстраппинг запускает ваше приложение, но рано или поздно вам нужно будет корректно его остановить. Вы могли бы, конечно, просто позволить JVM обработать всё при выходе, но это не соответствовало бы определению _graceful_ (корректный), которое подразумевает чистое освобождение ресурсов. Для завершения работы приложения Netty не нужно много магии, но есть несколько вещей, которые следует иметь в виду.

Прежде всего, вам **нужно завершить работу** `EventLoopGroup`, которая обработает все ожидающие события и задачи и впоследствии *освободит все активные потоки.* 
**Для этого нужно вызвать** `EventLoopGroup.shutdownGracefully()`. 

***Этот вызов вернёт `Future`, который будет уведомлён о завершении остановки.*** 
Обратите внимание, что `shutdownGracefully()` **также является асинхронной операцией**, поэтому вам нужно 
- **либо заблокироваться до её завершения,** 
- **либо зарегистрировать слушателя на возвращённом** `Future`, чтобы получить уведомление о завершении.

Следующий листинг соответствует определению корректного завершения работы.

**Листинг 8.9. Корректное завершение работы**
```java
// (1) Создаёт EventLoopGroup, которая обрабатывает I/O
EventLoopGroup group = new NioEventLoopGroup();  

// (2) Создаёт Bootstrap и настраивает его
Bootstrap bootstrap = new Bootstrap();   
                               
bootstrap.group(group)
    .channel(NioSocketChannel.class);
// ...

// (3) shutdownGracefully() освобождает ресурсы и закрывает все используемые в данный момент каналы
// block until the group has shutdown
Future<?> future = group.shutdownGracefully();  

// (4) Блокируется до завершения остановки группы                        
future.syncUninterruptibly();    
```

Альтернативно, вы можете явно вызвать `Channel.close()` на всех активных каналах перед вызовом `EventLoopGroup.shutdownGracefully()`. Но в любом случае не забудьте завершить работу самой `EventLoopGroup`.
