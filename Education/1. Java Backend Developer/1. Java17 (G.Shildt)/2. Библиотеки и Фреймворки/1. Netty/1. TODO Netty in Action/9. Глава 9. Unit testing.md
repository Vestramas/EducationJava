**В этой главе рассматриваются:**

- Модульное тестирование (unit testing)
    
- Обзор EmbeddedChannel
    
- Тестирование ChannelHandler с помощью EmbeddedChannel
    

`ChannelHandler` являются критическими элементами приложения Netty, поэтому их тщательное тестирование должно быть стандартной частью вашего процесса разработки. Лучшие практики диктуют, что вы тестируете не только для того, чтобы доказать правильность вашей реализации, но и чтобы легко изолировать проблемы, возникающие при модификации кода. Такой тип тестирования называется **модульным тестированием** (unit testing).

Хотя не существует универсального определения модульного тестирования, большинство практиков сходятся во мнении об основах. **Основная идея заключается в тестировании вашего кода максимально маленькими частями, изолированными, насколько это возможно,** от других модулей кода и от зависимостей времени выполнения, таких как базы данных и сети. Если вы можете проверить с помощью тестов, что каждый модуль работает правильно сам по себе, будет гораздо легче найти виновника, когда что-то пойдёт не так.

В этой главе мы изучим специальную реализацию `Channel` — `EmbeddedChannel`, которую Netty предоставляет специально для облегчения модульного тестирования `ChannelHandler`.

Поскольку тестируемый модуль кода будет выполняться вне его нормальной среды выполнения, вам нужен фреймворк или среда для его запуска. ***В наших примерах мы будем использовать*** **JUnit 4** в качестве нашего тестового фреймворка, поэтому вам потребуется базовое понимание его использования. 

Если вы с ним не знакомы, не бойтесь; хотя он мощный, он прост, и вы найдёте всю необходимую информацию на сайте JUnit ([www.junit.org](https://www.junit.org/)).

Возможно, вам будет полезно просмотреть предыдущие главы о `ChannelHandler` и кодеках, поскольку они предоставят материал для наших примеров.

## 9.1 Обзор EmbeddedChannel

Вы уже знаете, что реализации `ChannelHandler` могут быть объединены в цепочку в `ChannelPipeline` для построения бизнес-логики вашего приложения. Ранее мы объясняли, что эта архитектура поддерживает декомпозицию потенциально сложной обработки на небольшие и пригодные для повторного использования компоненты, каждый из которых обрабатывает чётко определённую задачу или шаг. В этой главе мы покажем вам, как это также упрощает тестирование.

Netty предоставляет так называемый **встроенный транспорт (embedded transport)** для тестирования `ChannelHandler`. Этот транспорт является особенностью специальной реализации `Channel` — `EmbeddedChannel`, которая предоставляет простой способ передачи событий через конвейер.

**Идея проста:** 
Вы записываете входящие или исходящие данные в `EmbeddedChannel`, ***а затем проверяете, достигло ли что-либо конца*** `ChannelPipeline`. Таким образом, вы можете определить, были ли сообщения закодированы или декодированы и были ли инициированы какие-либо действия `ChannelHandler`.

Соответствующие методы `EmbeddedChannel` перечислены в таблице 9.1.

**Таблица 9.1. Специальные методы EmbeddedChannel**

|Название|Описание|
|---|---|
|`writeInbound(Object... msgs)`|Записывает входящее сообщение в `EmbeddedChannel`. Возвращает `true`, если данные могут быть прочитаны из `EmbeddedChannel` с помощью `readInbound()`.|
|`readInbound()`|Читает входящее сообщение из `EmbeddedChannel`. Всё, что возвращено, прошло через весь `ChannelPipeline`. Возвращает `null`, если ничего не готово к чтению.|
|`writeOutbound(Object... msgs)`|Записывает исходящее сообщение в `EmbeddedChannel`. Возвращает `true`, если что-то теперь можно прочитать из `EmbeddedChannel` с помощью `readOutbound()`.|
|`readOutbound()`|Читает исходящее сообщение из `EmbeddedChannel`. Всё, что возвращено, прошло через весь `ChannelPipeline`. Возвращает `null`, если ничего не готово к чтению.|
|`finish()`|Помечает `EmbeddedChannel` как завершённый и возвращает `true`, если могут быть прочитаны входящие или исходящие данные. Также вызовет `close()` на `EmbeddedChannel`.|

![[Pasted image 20260612085929.png]]
**Рисунок 9.1. Поток данных EmbeddedChannel**


Входящие данные обрабатываются `ChannelInboundHandler` и представляют данные, прочитанные от удалённого узла. Исходящие данные обрабатываются `ChannelOutboundHandler` и представляют данные, которые должны быть записаны удалённому узлу. В зависимости от `ChannelHandler`, который вы тестируете, вы будете использовать пары методов `*Inbound()` или `*Outbound()`, а возможно, и оба.

Рисунок 9.1 показывает, как данные проходят через `ChannelPipeline` с использованием методов `EmbeddedChannel`. Вы можете использовать `writeOutbound()` для записи сообщения в канал и его передачи через `ChannelPipeline` в исходящем направлении. Затем вы можете прочитать обработанное сообщение с помощью `readOutbound()`, чтобы определить, соответствует ли результат ожидаемому. Аналогично, для входящих данных вы используете `writeInbound()` и `readInbound()`.

В каждом случае сообщения передаются через `ChannelPipeline` и обрабатываются соответствующими `ChannelInboundHandler` или `ChannelOutboundHandler`. Если сообщение не было потреблено, вы можете использовать `readInbound()` или `readOutbound()` для чтения сообщений из канала после их обработки.

Давайте подробнее рассмотрим оба сценария и посмотрим, как они применяются для тестирования вашей бизнес-логики.

## 9.2 Тестирование ChannelHandler с помощью EmbeddedChannel

В этом разделе мы объясним, как тестировать `ChannelHandler` с помощью `EmbeddedChannel`.

> **Утверждения JUnit (JUnit assertions)**
> 
> Класс `org.junit.Assert` *предоставляет множество статических методов для использования в тестах.* ***Неудачное утверждение (assertion) вызовет исключение*** и завершит текущий выполняющийся тест. 
> 
> Наиболее эффективный способ импортировать эти утверждения — с помощью оператора `import static`:
```java
import static org.junit.Assert.*;
```
> После этого вы можете вызывать методы `Assert` напрямую:
```java
assertEquals(buf.readSlice(3), read);
```

### 9.2.1 Тестирование входящих сообщений

Рисунок 9.2 представляет простую реализацию `ByteToMessageDecoder`. При наличии достаточного количества данных она будет создавать кадры фиксированного размера. Если данных недостаточно для чтения, она будет ждать следующего фрагмента данных и снова проверять, может ли быть создан кадр.

![[Pasted image 20260612090052.png]]
**Рисунок 9.2. Декодирование с помощью FixedLengthFrameDecoder**

Как вы можете видеть из кадров в правой части рисунка, этот конкретный декодер создаёт кадры фиксированного размера 3 байта. Таким образом, может потребоваться более одного события, чтобы предоставить достаточно байтов для создания кадра.

Наконец, каждый кадр будет передан следующему `ChannelHandler` в `ChannelPipeline`.

Реализация декодера показана в следующем листинге.

**Листинг 9.1. FixedLengthFrameDecoder**
```java
// (1) Расширяет ByteToMessageDecoder для обработки входящих байтов и их декодирования в сообщения
public class FixedLengthFrameDecoder extends ByteToMessageDecoder {   
    private final int frameLength;
    
	// (2) Указывает длину создаваемых кадров
    public FixedLengthFrameDecoder(int frameLength) {                 
        if (frameLength <= 0) {
            throw new IllegalArgumentException(
	            "frameLength must be a positive integer: " + frameLength);
        }
        this.frameLength = frameLength;
    }

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
	    // (3) Проверяет, достаточно ли байтов для создания следующего кадра
        while (in.readableBytes() >= frameLength) {    
	        // (4) Читает новый кадр из ByteBuf               
            ByteBuf buf = in.readBytes(frameLength);   
            
            // (5) Добавляет кадр в список декодированных сообщений       
            out.add(buf);                                             
        }
    }
}
```

Теперь давайте создадим модульный тест, чтобы убедиться, что этот код работает как ожидается. Как мы отмечали ранее, даже для простого кода модульные тесты помогают предотвратить проблемы, которые могут возникнуть, если код будет реорганизован в будущем, и диагностировать их, если они возникнут.

В этом листинге показан тест предыдущего кода с использованием `EmbeddedChannel`.

**Листинг 9.2. Тестирование FixedLengthFrameDecoder**
```java
public class FixedLengthFrameDecoderTest {
	// (1) Аннотирован @Test, чтобы JUnit выполнил метод
    @Test       
    
	// (2) Первый тестовый метод: testFramesDecoded()
    public void testFramesDecoded() {  
	    // (3) Создаёт ByteBuf и сохраняет 9 байтов
        ByteBuf buf = Unpooled.buffer();                               
        for (int i = 0; i < 9; i++) {
            buf.writeByte(i);
        }
        ByteBuf input = buf.duplicate();
        // (4) Создаёт EmbeddedChannel и добавляет FixedLengthFrameDecoder для тестирования с длиной кадра 3 байта
        EmbeddedChannel channel = new EmbeddedChannel(                
            new FixedLengthFrameDecoder(3));
        // запись байтов
        // (5) Записывает данные в EmbeddedChannel
        assertTrue(channel.writeInbound(input.retain())); 
        
        // (6) Помечает канал как завершённый             
        assertTrue(channel.finish());                                  

        // чтение сообщений
        // (7) Читает созданные сообщения и проверяет, что есть 3 кадра (среза) по 3 байта каждый
        ByteBuf read = (ByteBuf) channel.readInbound();                
        assertEquals(buf.readSlice(3), read);
        
        read.release();
        read = (ByteBuf) channel.readInbound();
        assertEquals(buf.readSlice(3), read);
        read.release();
        
        read = (ByteBuf) channel.readInbound();
        assertEquals(buf.readSlice(3), read);
        read.release();
        assertNull(channel.readInbound());
        buf.release();
    }

    @Test
    // (8) Второй тестовый метод: testFramesDecoded2()
    public void testFramesDecoded2() {                                 
        ByteBuf buf = Unpooled.buffer();
        for (int i = 0; i < 9; i++) {
            buf.writeByte(i);
        }
        ByteBuf input = buf.duplicate();

        EmbeddedChannel channel = new EmbeddedChannel(
            new FixedLengthFrameDecoder(3));
            
		// (9) Возвращает false, потому что полный кадр ещё не готов к чтению
        assertFalse(channel.writeInbound(input.readBytes(2)));         
        assertTrue(channel.writeInbound(input.readBytes(7)));
        assertTrue(channel.finish());

        ByteBuf read = (ByteBuf) channel.readInbound();
        assertEquals(buf.readSlice(3), read);
        read.release();
        read = (ByteBuf) channel.readInbound();
        assertEquals(buf.readSlice(3), read);
        read.release();
        read = (ByteBuf) channel.readInbound();
        assertEquals(buf.readSlice(3), read);
        read.release();
        assertNull(channel.readInbound());
        buf.release();
    }
}
```

Метод `testFramesDecoded()` проверяет, что `ByteBuf`, содержащий 9 читаемых байтов, декодируется в 3 `ByteBuf`, каждый из которых содержит 3 байта. Обратите внимание, как `ByteBuf` заполняется 9 читаемыми байтами за один вызов `writeInbound()`. После этого выполняется `finish()`, чтобы пометить `EmbeddedChannel` как завершённый. Наконец, `readInbound()` вызывается для чтения ровно трёх кадров и `null` из `EmbeddedChannel`.

Метод `testFramesDecoded2()` похож, с одним отличием: входящие `ByteBuf` записываются в два этапа. Когда вызывается `writeInbound(input.readBytes(2))`, возвращается `false`. Почему? Как указано в таблице 9.1, `writeInbound()` возвращает `true`, если последующий вызов `readInbound()` вернул бы данные. Но `FixedLengthFrameDecoder` будет создавать выходные данные только тогда, когда доступны три или более байта. Остальная часть теста идентична `testFramesDecoded()`.

### 9.2.2 Тестирование исходящих сообщений

Тестирование обработки исходящих сообщений аналогично тому, что вы только что видели. В следующем примере мы покажем, как вы можете использовать `EmbeddedChannel` для тестирования `ChannelOutboundHandler` в форме кодировщика — компонента, который преобразует один формат сообщения в другой. Вы подробно изучите кодировщики и декодировщики в следующей главе, поэтому сейчас мы просто упомянем, что тестируемый нами обработчик, `AbsIntegerEncoder`, является специализацией `MessageToMessageEncoder` от Netty, которая преобразует отрицательные целые числа в абсолютные значения.

**Пример будет работать следующим образом:**

1. `EmbeddedChannel`, содержащий `AbsIntegerEncoder`, будет записывать исходящие данные в виде 4-байтовых отрицательных целых чисел.
    
2. Декодер будет читать каждое отрицательное целое число из входящего `ByteBuf` и вызывать `Math.abs()` для получения абсолютного значения.
    
3. Декодер будет записывать абсолютное значение каждого целого числа в `ChannelPipeline`.


![[Pasted image 20260612090425.png]]
**Рисунок 9.3. Кодирование с помощью AbsIntegerEncoder**

Следующий листинг реализует эту логику, проиллюстрированную на рисунке 9.3. Метод `encode()` записывает полученные значения в `List`.

**Листинг 9.3. AbsIntegerEncoder**
```java
// (1) Расширяет MessageToMessageEncoder для кодирования сообщения в другой формат
public class AbsIntegerEncoder extends MessageToMessageEncoder<ByteBuf> { 
    @Override
    protected void encode(
	    ChannelHandlerContext ctx, 
	    ByteBuf in, 
	    List<Object> out
	) throws Exception 
	{
		// (2) Проверяет, достаточно ли байтов для кодирования
        while (in.readableBytes() >= 4) {   
	        // (3) Читает следующее целое число из входного ByteBuf и вычисляет абсолютное значение
            int value = Math.abs(in.readInt());  
            
            // (4) Записывает целое число в список закодированных сообщений
            out.add(value);                                               
        }
    }
}
```

Следующий листинг тестирует код с использованием `EmbeddedChannel`.

**Листинг 9.4. Тестирование AbsIntegerEncoder**
```java
public class AbsIntegerEncoderTest {
    @Test
    public void testEncoded() {
	    // (1) Создаёт ByteBuf и записывает 9 отрицательных целых чисел
        ByteBuf buf = Unpooled.buffer();                                 
        for (int i = 1; i < 10; i++) {
            buf.writeInt(i * -1);
        }

        assertTrue(channel.finish());                                    

        // read bytes
        for (int i = 1; i < 10; i++) {

        }
        assertNull(channel.readOutbound());
    }

```

**Вот шаги, выполняемые в коде:**
1. **B** Записывает отрицательные 4-байтовые целые числа в новый `ByteBuf`.
    
2. **c** Создаёт `EmbeddedChannel` и назначает ему `AbsIntegerEncoder`.
    
3. **d** Вызывает `writeOutbound()` на `EmbeddedChannel`, чтобы записать `ByteBuf`.
    
4. **e** Помечает канал как завершённый.
    
5. **f** Читает все целые числа из исходящей стороны `EmbeddedChannel` и проверяет, что были созданы только абсолютные значения.

## 9.3 Тестирование обработки исключений

Приложения обычно имеют дополнительные задачи для выполнения помимо преобразования данных. Например, вам может потребоваться обработать некорректный ввод или чрезмерный объём данных. В следующем примере мы выбросим `TooLongFrameException`, если количество прочитанных байтов превысит указанный лимит. Этот подход часто используется для защиты от исчерпания ресурсов.

На рисунке 9.4 максимальный размер кадра установлен в 3 байта. Если размер кадра превышает этот лимит, его байты отбрасываются и выбрасывается `TooLongFrameException`. Другие `ChannelHandler` в конвейере могут либо обработать исключение в `exceptionCaught()`, либо игнорировать его.


![[Pasted image 20260612090443.png]]
**Рисунок 9.4. Декодирование с помощью FrameChunkDecoder**

Реализация показана в следующем листинге.

**Листинг 9.5. FrameChunkDecoder**
```java
//(1) Расширяет ByteToMessageDecoder для декодирования входящих байтов в сообщения
public class FrameChunkDecoder extends ByteToMessageDecoder {           
    private final int maxFrameSize;
	
	// (2) Указывает максимально допустимый размер создаваемых кадров
    public FrameChunkDecoder(int maxFrameSize) {                       
        this.maxFrameSize = maxFrameSize;
    }

    @Override
    protected void decode(
	    ChannelHandlerContext ctx, 
	    ByteBuf in, 
	    List<Object> out
	) throws Exception 
	{
        int readableBytes = in.readableBytes();
        if (readableBytes > maxFrameSize) {
            // отбрасываем байты
            // (3) Отбрасывает кадр, если он слишком велик
            in.clear();  
            
			// и выбрасывает TooLongFrameException
            throw new TooLongFrameException();                         
        }
        // (4) В противном случае читает новый кадр из ByteBuf
        ByteBuf buf = in.readBytes(readableBytes);    
        
        // (5) Добавляет кадр в список декодированных сообщений
        out.add(buf);                                                  
    }
}
```

Опять же, мы протестируем код с использованием `EmbeddedChannel`.

**Листинг 9.6. Тестирование FrameChunkDecoder**
```java
public class FrameChunkDecoderTest {
    @Test
    public void testFramesDecoded() {
	    // (1) Создаёт ByteBuf и записывает 9 байтов
        ByteBuf buf = Unpooled.buffer();                               
        for (int i = 0; i < 9; i++) {
            buf.writeByte(i);
        }
        ByteBuf input = buf.duplicate();
        
		// (2) Создаёт EmbeddedChannel и устанавливает FrameChunkDecoder с размером кадра 3
        EmbeddedChannel channel = new EmbeddedChannel(                
            new FrameChunkDecoder(3));
            
		// (3) Записывает 2 байта и утверждает, что они создали новый кадр
        assertTrue(channel.writeInbound(input.readBytes(2)));          
        try {
	        // (4) Записывает 4-байтовый кадр и перехватывает ожидаемый TooLongFrameException
            channel.writeInbound(input.readBytes(4));  
            
			// (5) Если исключение не выброшено, это утверждение достигается и тест проваливается
            Assert.fail();                                             
        } catch (TooLongFrameException e) {
            // ожидаемое исключение
        }
		// (6) Записывает оставшиеся 2 байта и утверждает допустимый кадр
        assertTrue(channel.writeInbound(input.readBytes(3))); 
        
		// (7) Помечает канал как завершённый         
        assertTrue(channel.finish());                                  

        // чтение кадров
        // (8) Читает созданные сообщения и проверяет значения
        ByteBuf read = (ByteBuf) channel.readInbound();                
        assertEquals(buf.readSlice(2), read);
        read.release();

        read = (ByteBuf) channel.readInbound();
        assertEquals(buf.skipBytes(4).readSlice(3), read);
        read.release();
        buf.release();
    }
}
```

На первый взгляд это очень похоже на тест из листинга 9.2, но есть интересный поворот: а именно, обработка `TooLongFrameException`. Используемый здесь блок `try/catch` является особой функцией `EmbeddedChannel`. Если один из методов `write*` создаёт проверяемое исключение, оно будет выброшено, обёрнутым в `RuntimeException`[^1]. Это позволяет легко проверить, было ли исключение обработано во время обработки данных.

Проиллюстрированный здесь подход к тестированию может быть использован с любой реализацией `ChannelHandler`, которая выбрасывает исключение.

## 9.4 Резюме (Summary)

Модульное тестирование с помощью тестовой среды, такой как JUnit, является чрезвычайно эффективным способом гарантировать корректность вашего кода и улучшить его поддерживаемость. В этой главе вы узнали, как использовать инструменты тестирования, предоставляемые Netty, для тестирования ваших пользовательских `ChannelHandler`.

В следующих главах мы сосредоточимся на написании реальных приложений с помощью Netty. Мы не будем приводить дальнейших примеров тестового кода, поэтому мы надеемся, что вы будете помнить о важности подхода к тестированию, который мы здесь продемонстрировали.
