После подключения зависимостей необходимо сконфигурировать главный сервлет `DispathcerServlet`, который уже реализован в Spring MVC

==Принцип работы:==
##### **Реализация интерфейса `javax.servlet.Servlet`:**
**`DispatcherServlet`** наследует от `HttpServlet` и реализует интерфейс `Servlet`. Это позволяет ему обрабатывать HTTP-запросы и работать с ними на уровне сервлетов, как любой другой обычный сервлет в Java.
##### **Запуск приложения:**
Когда веб-приложение запускается, в конфигурации сервера (например, в **web.xml**) определяется **`DispatcherServlet`** с маппингом на все или определенные URL-паттерны (например, `/` для всех запросов).
##### **Инициализация:**
При старте приложения, **`DispatcherServlet`** инициализируется, и Spring загружает контекст приложения. В частности, он читает конфигурационный файл (например, **`applicationContext.xml`**) и загружает все бины, необходимые для работы приложения.
##### **Процесс обработки запроса:**
Когда поступает HTTP-запрос, **`DispatcherServlet`** анализирует его и решает, какой контроллер должен обработать этот запрос. Процесс работы с запросом включает несколько шагов:
**1. Выбор обработчика (Handler Mapping):**
- `DispatcherServlet` использует ==**Handler Mapping** (делает Spring)== для выбора соответствующего обработчика (метода контроллера), который должен обработать данный запрос. В Spring это происходит через аннотации, такие как **`@RequestMapping`** или **`@GetMapping`**.

**2. Вызов обработчика (Handler Execution):**
- После того как нужный обработчик найден, **`DispatcherServlet`** вызывает соответствующий метод контроллера.

**3. Обработка результата (View Resolution):**
- Когда метод контроллера возвращает строку (например, имя представления), **`DispatcherServlet`** передает это имя **`ViewResolver`**, который разрешает его в фактическое представление (например, JSP или Thymeleaf-шаблон).
- Если контроллер возвращает объект, а не строку, Spring автоматически будет искать шаблон для рендеринга этих данных.

**4. Рендеринг представления (View Rendering):**
- После того как **`ViewResolver`** находит представление, **`DispatcherServlet`** передает данные модели в представление и инициирует его рендеринг, чтобы вернуть пользователю окончательную HTML-страницу.
# 1. Конфигурация файла web.xml
Чтобы приложение заработало, ==необходимо сконфигурировать главный сервлет== `DispathcerServlet` ==(уже реализован в SpringMVC)== приложения, который будет распределять запросы между нашими контроллерами

> [!NOTE] Важно
> Иногда лучше проверять ссылки на работоспособность
> 
> Атрибут `version` ==обязателен==

```xml
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
	xmlns="http://xmlns.jcp.org/xml/ns/javaee" 
	xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/javaee 
	http://xmlns.jcp.org/xml/ns/javaee/web-app_3_1.xsd" 
	id="WebApp_ID" version="3.1">
         
    <servlet>
        <servlet-name>dispatcher</servlet-name>
        <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
        <init-param>
            <param-name>contextConfigLocation</param-name>
            <param-value>/WEB-INF/applicationContextMVC.xml</param-value>
        </init-param>
        <load-on-startup>1</load-on-startup>
    </servlet>

    <servlet-mapping>
        <servlet-name>dispatcher</servlet-name>
        <url-pattern>/</url-pattern>
    </servlet-mapping>
</web-app>
```

В теге `<servlet>` определили `DispatcherServlet`, который уже реализован в SpringMVC. ==В параметрах инициализации указали ГДЕ РАССПОЛОЖЕН контекст приложения==, где будут находится наши контроллеры

> [!NOTE] Маппинг приложения
> Маппинг в конечном итоге определяется НА СЕРВЕРЕ. Однако, маппинг dispathcerServlet лучше оставить /
> 
> В Eclipse это выглядит так: 
> вкладка Servers-> текущий_сервер-> 2 раза ЛКМ -> подвкладка Modules

# 2. Создание контекста приложения
Теперь нужно создать контекст приложения, в котором Spring будет искать необходимый бин

> [!NOTE] Контекст приложения
> Файл XML с конфигурацией должен (рекомендуется) лежать в папке WEB-INF (в той же папке что и web.xml)

Конфигурация схожа с той, что используется в Sring Core.
Замечания:
1. Будем использовать XML+Annotation
```xml
<?xml version="1.0" encoding="UTF-8"?>
<beans  xmlns="http://www.springframework.org/schema/beans"
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xmlns:context="http://www.springframework.org/schema/context"
        xsi:schemaLocation="http://www.springframework.org/schema/beans
        http://www.springframework.org/schema/beans/spring-beans.xsd
        http://www.springframework.org/schema/context
        http://www.springframework.org/schema/context/spring-context.xsd">

	<context:component-scan base-package="ru.MvcInAction"/>
</beans>
```

> [!NOTE] Spring 3.1 и НИЖЕ
> Если используются зависимости версии НИЖЕ чем 3.1, то для ==подключения== аннотаций `@RequestMapping`, `@GetMapping`, `@PostMapping` , `@Controller` 
> 
> НЕОБХОДИМО использовать тег `<mvc:annotation-driven />`

В **Spring 6+** `<mvc:annotation-driven />` в XML **обычно не нужен**, потому что Spring MVC автоматически регистрирует необходимые обработчики аннотаций

# 3. Настройка бинов (Thymleaf) для представлений 
==Нужно подключить зависимость Thymeleaf==

Для работы с Thymleaf необходимо создать бины:
- **TemplateResolver** – определяет, где хранятся шаблоны.
- **TemplateEngine** – основной компонент Thymeleaf, который рендерит шаблоны.
- **ViewResolver** – сообщает Spring MVC, как находить и обрабатывать шаблоны.

```xml
	<!-- 1. Определение TemplateResolver -->
    <bean id="templateResolver" class="org.thymeleaf.spring6.templateresolver.SpringResourceTemplateResolver">
        <property name="prefix" value="/WEB-INF/views/" />
        <property name="suffix" value=".html" />
    </bean>


    <!-- 2. Определение TemplateEngine -->
    <bean id="templateEngine" class="org.thymeleaf.spring6.SpringTemplateEngine">
        <property name="templateResolver" ref="templateResolver" />
    </bean>


    <!-- 3. Определение ViewResolver для Thymeleaf -->
    <bean class="org.thymeleaf.spring6.view.ThymeleafViewResolver">
        <property name="templateEngine" ref="templateEngine" />
    </bean>
```

В бине `ThymeleafViewResolver` можно также добавить тег `<property name="viewNames" value="*" />` чтобы определить имена шаблонов, который будет обрабатывать Thymeleaf (полезно если используются несколько шаблонизаторов)

> [!NOTE] Где хранить представления?
> Представления ЛУЧШЕ хранить в подпапке папки WEB-INF, чтобы представления не были доступны через браузер (обычно /WEB-INF/templates)
