# 0. Важная аннотация @EnableWebMvc
Аннотация `@EnableWebMvc` используется для **включения полноценной инфраструктуры Spring MVC** в приложениях, которые **не используют Spring Boot**. По сути, она сообщает Spring, что данное приложение должно работать как MVC-приложение с контроллерами, маппингами запросов, обработкой аргументов методов и сериализацией ответов.

Когда ты добавляешь `@EnableWebMvc` в конфигурационный класс, Spring импортирует специальную конфигурацию `DelegatingWebMvcConfiguration`. Именно этот класс регистрирует в контексте все ключевые бины, необходимые для работы MVC.

## Jackson и @EnableWebMvc
==Критически важной частью является регистрация== `HttpMessageConverter`
Именно они отвечают за преобразование Java-объектов в HTTP-ответ и обратно. 

После включения `@EnableWebMvc` Spring ==автоматически регистрирует стандартный набор конвертеров==, в который входят:
- `MappingJackson2HttpMessageConverter` для JSON (если Jackson есть в classpath),
    
- `StringHttpMessageConverter` для строк,
    
- `ByteArrayHttpMessageConverter`,
    
- `FormHttpMessageConverter` и другие.


Когда ты просто добавляешь `@EnableWebMvc`, Spring делает **две вещи одновременно**:
1. **Включает всю инфраструктуру MVC**, включая `RequestMappingHandlerAdapter`, `RequestMappingHandlerMapping` и базовые HandlerMethodReturnValueHandler.
    
2. **Подключает набор стандартных конвертеров**, **но только те, которые предусмотрены в `DelegatingWebMvcConfiguration`**, то есть:
	- `StringHttpMessageConverter`
	    
	- `ByteArrayHttpMessageConverter`
	    
	- `ResourceHttpMessageConverter`
	    
	- `MappingJackson2HttpMessageConverter` (если в classpath есть Jackson)

➡️ Вот тут важный нюанс: **если Jackson есть в classpath**, Spring автоматически создаёт и регистрирует `MappingJackson2HttpMessageConverter`. Именно поэтому у тебя JSON заработал сразу после добавления аннотации.

Если Jackson бы не было — конвертер бы не появился, и ты получил бы ту же ошибку `406 Not Acceptable`.

## 1.2 Настройка бинов (Thymleaf) для представлений 
Предыдущая конфигурация будет работать, если метод контроллера используется аннотацию `@ResponseBody` и НЕ ВОЗВРАЩАЕТ страницу html.

В противном случае, необходимо создать бины для шаблонизаторов

Для работы с Thymleaf необходимо создать бины:
- **TemplateResolver** – определяет, где хранятся шаблоны.
- **TemplateEngine** – основной компонент Thymeleaf, который рендерит шаблоны.
- **ViewResolver** – сообщает Spring MVC, как находить и обрабатывать шаблоны

Здесь поступаем по АНАЛОГИИ как создавали бины в XML конфигурации (используем свойства `setPrefix` , `setSuffix`)

==Thymeleaf МОЖНО передать КОНТЕКСТ приложения==, чтобы он работал в контексте Spring с **внешними ресурсами или с динамическими настройками**, которые управляются контейнером Spring.
```java
@Configuration
@ComponentScan("mvcInAction.MvcApplication.MyControllers")
@EnableWebMvc
public class ContextConfig implements WebMvcConfigurer{
	private final ApplicationContext applicationContext;
	@Autowired
	public ContextConfig(ApplicationContext applicationContext) {
		this.applicationContext = applicationContext;
	}
	@Bean
	public SpringResourceTemplateResolver templateResolver() {
		SpringResourceTemplateResolver templateResolver = new 
		SpringResourceTemplateResolver();
		
		templateResolver.setApplicationContext(applicationContext); // Передача контекста
		templateResolver.setPrefix("/WEB-INF/views/");
		templateResolver.setSuffix(".html");
		return templateResolver;
	}
	@Bean
	public SpringTemplateEngine templateEngine() {
		SpringTemplateEngine templateEngine = new SpringTemplateEngine();
		
		templateEngine.setTemplateResolver(templateResolver());
		templateEngine.setEnableSpringELCompiler(true);
		return templateEngine;
	}
	
	@Override
	public void configureViewResolvers(ViewResolverRegistry registry) {
		ThymeleafViewResolver resolver = new ThymeleafViewResolver();
		resolver.setTemplateEngine(templateEngine());
		registry.viewResolver(resolver);
	}
}
```

#### 1.`templateResolver()`
Этот метод создает и настраивает `SpringResourceTemplateResolver`, который ==отвечает за нахождение и обработку шаблонов== (HTML файлов). Вот что делает этот метод:
- **`setApplicationContext(applicationContext)`**: Связывает `templateResolver` с контекстом Spring, что позволяет ему находить ресурсы в приложении.
#### 2.`templateEngine()`
Этот метод ==создает и настраивает== `SpringTemplateEngine`, который является ==ядром движка Thymeleaf==. Он отвечает за обработку шаблонов, используя `templateResolver` для их поиска.
- **`setTemplateResolver(templateResolver())`**: Связывает движок с ранее настроенным `templateResolver`.
- **`setEnableSpringELCompiler(true)`**: Включает компилятор выражений Spring EL, что может улучшить производительность при работе с выражениями внутри шаблонов.
#### 3.`configureViewResolvers(ViewResolverRegistry registry)`
Этот метод ==настраивает резолвер представлений==, который Spring MVC использует для определения представления, возвращаемого контроллером.
- **Создание `ThymeleafViewResolver`**: `ThymeleafViewResolver` — это адаптер между Spring MVC и Thymeleaf, который преобразует логические имена представлений (например, строки, возвращаемые контроллерами) в конечные представления с использованием движка Thymeleaf.
- **`resolver.setTemplateEngine(templateEngine())`**: Связывает `ThymeleafViewResolver` с настроенным движком `templateEngine`, чтобы тот мог использовать шаблоны для рендеринга.
- **`registry.viewResolver(resolver)`**: Регистрирует резолвер представлений в Spring MVC.

# 2. Замена файла web.xml
Для замены файла конфигурации СЕРВЕРА, может использоваться Java-класс начиная с версии Spring 3.0

Данный файл будет заменен на Java-класс, реализующий **интерфейс** ==WebApplicationInitializer== из пакета (org.springframework.web.WebApplicationInitializer)

> [!NOTE] Два варианта реализации
> Здесь есть два варианта:
> 
> 1.Реализовать интерфейс `WebApplicationInitializer` 
> В методе` void onStartup(ServletContext container)` описывается то же самое что и в web.xml (dispatcherServelt)
> 
> 2.Унаследовать ==абстрактный== класс который уже реализовал этот интерфейс
>`AbstractAnnotationConfigDispatcherServletInitialiser` представленный в Spring 3.2
>(пакет org.springframework.web.servlet.support.AbstractAnnotationConfigDispatcherServletInitializer)

***
## 2.2 extends AbstractAnnotationConfigDispatcherServletInitialiser

Для конфигурации ВТОРЫМ способом, рекомендуется создать отдельный пакет, в котором будет храниться конфигурационные классы

```java
public class ServerConfig extends AbstractAnnotationConfigDispatcherServletInitializer{

	@Override
	protected Class<?>[] getRootConfigClasses() {
		// TODO Auto-generated method stub
		return null;
	}

	@Override
	protected Class<?>[] getServletConfigClasses() {
		// TODO Auto-generated method stub
		return new Class[] {ContextConfig.class};
	}

	@Override
	protected String[] getServletMappings() {
		// TODO Auto-generated method stub
		return new String[]{"/"};
	}
}
```

Созданный класс должен реализовать абстрактные методы абстрактного класса
***
### **1. `getRootConfigClasses()`**

Этот метод **указывает классы конфигурации для корневого контекста (`ApplicationContext`)**, который управляет **бизнес-логикой и доступом к данным**.

✅ **Что здесь настраивается?**

- Бины, связанные с **сервисами** (`@Service`), **репозиториями** (`@Repository`), **безопасностью**, **JPA/Hibernate** и т. д.
- Конфигурация базы данных (например, `DataSource`, `EntityManagerFactory`).
- Любые другие **глобальные бины**, которые **не зависят от веб-уровня**.

#### Пример
```java
@Configuration
@ComponentScan(basePackages = "com.example.service") // Сканирует сервисы
public class AppConfig {
    
    @Bean
    public DataSource dataSource() {
        DriverManagerDataSource dataSource = new DriverManagerDataSource();
        dataSource.setDriverClassName("com.mysql.cj.jdbc.Driver");
        dataSource.setUrl("jdbc:mysql://localhost:3306/mydb");
        dataSource.setUsername("root");
        dataSource.setPassword("password");
        return dataSource;
    }

    @Bean
    public LocalSessionFactoryBean sessionFactory() {
        LocalSessionFactoryBean sessionFactory = new LocalSessionFactoryBean();
        sessionFactory.setDataSource(dataSource());
        return sessionFactory;
    }
}
```

```java
@Override
protected Class<?>[] getRootConfigClasses() {
    return new Class[]{AppConfig.class}; // Глобальные бины (сервисы, репозитории, БД)
}
```
***
### **2. `getServletConfigClasses()`**

Этот метод **указывает классы конфигурации для веб-контекста (`WebApplicationContext`)**, который управляет **Spring MVC (контроллеры, представления, обработка HTTP-запросов)**.

✅ **Что здесь настраивается?**

- Контроллеры (`@Controller`, `@RestController`).
- Настройка представлений (например, `ViewResolver` для JSP, Thymeleaf).
- Конвертеры форматов данных (JSON, XML).
- Интерцепторы и фильтры Spring MVC.

#### Пример
```java
@Configuration
@EnableWebMvc
@ComponentScan(basePackages = "com.example.controllers") // Сканирует контроллеры
public class WebConfig {

    @Bean
    public ViewResolver viewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setPrefix("/WEB-INF/views/");
        resolver.setSuffix(".jsp");
        return resolver;
    }
}
```

```java
@Override
protected Class<?>[] getServletConfigClasses() {
    return new Class[]{WebConfig.class}; // Конфигурация Spring MVC
}
```
***
### **3. `getServletMappings()`**

Этот метод **указывает, какие URL-адреса будут обрабатываться `DispatcherServlet`**.
```java
@Override
protected String[] getServletMappings() {
    return new String[]{"/"}; // Все запросы идут в Spring MVC
}
```
