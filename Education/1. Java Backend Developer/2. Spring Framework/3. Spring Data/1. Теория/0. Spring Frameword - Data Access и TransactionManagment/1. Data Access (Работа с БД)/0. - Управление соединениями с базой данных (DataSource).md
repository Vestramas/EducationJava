# 1. Использование `DataSource`

> [!IMPORTANT] `DataSource`
> **Spring получает соединение с БД через** `DataSource` — это часть спецификации JDBC, обобщённая фабрика соединений.
> 
> Она ***позволяет скрыть от кода приложения детали пула соединений и управления транзакциями***.

## 1.1 **Основные способы получения `DataSource`:**
- Из **JNDI** (если приложение работает в контейнере).
    
- **Сконфигурировать самостоятельно** с использованием стороннего пула соединений.

### **Рекомендации по выбору пула:**
- **Традиционные:** ==Apache Commons DBCP==, C3P0 (bean-style классы).
    
- **Современный выбор:** ==HikariCP== (builder-style API, лучшая производительность).
```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>
```

```java
@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();
        
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
        config.setUsername("user");
        config.setPassword("pass");
        config.setDriverClassName("org.postgresql.Driver");

        // Настройки пула
        
        // Максимальное количество соединений
        config.setMaximumPoolSize(10);
        
        // Минимальное количество простаивающих соединений
        config.setMinimumIdle(5);
        
        // Таймаут простаивания (5 минут)                   
        config.setIdleTimeout(300000); 
        
        // Таймаут получения соединения (20 секунд)
        config.setConnectionTimeout(20000);
        
        // Максимальное время жизни соединения (20 минут)
        config.setMaxLifetime(1200000);              

        return new HikariDataSource(config);
    }
}
```


> [!WARNING] **Важнейшее предупреждение:**
> Классы `DriverManagerDataSource` и `SimpleDriverDataSource` (из дистрибутива Spring) предназначены **только для тестирования!** 
> 
> Они НЕ ИСПОЛЬЗУЮ ПУЛ СОЕДИНЕНИЙ и работают плохо при множественных запросах.

##### **Настройка `DriverManagerDataSource` (только для тестов!):**
```java
@Bean
DriverManagerDataSource dataSource() {
    DriverManagerDataSource ds = new DriverManagerDataSource();
    
    ds.setDriverClassName("org.hsqldb.jdbcDriver");
    ds.setUrl("jdbc:hsqldb:hsql://localhost:");
    ds.setUsername("sa");
    ds.setPassword("");
    
    return ds;
}
```

`DriverManagerDataSource` — это простейшая реализация `DataSource`, которая **создаёт новое соединение с БД при каждом запросе**. 

Это создаёт две серьёзные проблемы:

**Проблема 1: Производительность.** Установка соединения с БД — дорогая операция (TCP handshake, аутентификация, выделение ресурсов на стороне сервера БД). Если на каждый запрос создавать новое соединение, приложение будет тратить большую часть времени не на выполнение SQL, а на подключение к БД.

**Проблема 2: Отсутствие пула соединений.** `DriverManagerDataSource` не хранит открытые соединения. После закрытия соединение уничтожается. Это значит, что при 100 одновременных запросах будет создано 100 соединений, выполнено по одному запросу, и все 100 будут закрыты. Это крайне неэффективно.

##### **Пример конфигурации DBCP:**
```java
@Bean(destroyMethod = "close")
BasicDataSource dataSource() {
    BasicDataSource ds = new BasicDataSource();
    ds.setDriverClassName("org.hsqldb.jdbcDriver");
    ds.setUrl("jdbc:hsqldb:hsql://localhost:");
    ds.setUsername("sa");
    ds.setPassword("");
    return ds;
}
```

XML-конфигурация для DBCP
```xml
<bean id="dataSource" class="org.apache.commons.dbcp2.BasicDataSource" destroy-method="close">
    <property name="driverClassName" value="${jdbc.driverClassName}"/>
    <property name="url" value="${jdbc.url}"/>
    <property name="username" value="${jdbc.username}"/>
    <property name="password" value="${jdbc.password}"/>
</bean>
<context:property-placeholder location="jdbc.properties"/>
```

---
##### **Пример конфигурации C3P0:**
```java
@Bean(destroyMethod = "close")
ComboPooledDataSource dataSource() throws PropertyVetoException {
    ComboPooledDataSource ds = new ComboPooledDataSource();
    
    ds.setDriverClass("org.hsqldb.jdbcDriver");
    ds.setJdbcUrl("jdbc:hsqldb:hsql://localhost:");
    ds.setUser("sa");
    ds.setPassword("");
    
    return ds;
}
```

# 2. Вспомогательные классы для работы с `DataSource`

## 2.1 **`DataSourceUtils`**  
Вспомогательный класс **со статическими методами для получения соединений из JNDI и их закрытия**. 

Поддерживает привязку JDBC-соединения к потоку через `DataSourceTransactionManager`. 

> **Важно:** `JdbcTemplate` внутри себя всегда использует `DataSourceUtils`, поэтому ==все операции неявно участвуют в текущей транзакции.==

## 2.2 **`SmartDataSource`**  
Интерфейс, расширяющий `DataSource`. 

Реализующие его классы могут подсказать, нужно ли закрывать соединение после операции. *Полезно, когда вы знаете, что соединение будет переиспользоваться.*

## 2.3 **`AbstractDataSource`**  
Абстрактный базовый класс для собственных реализаций `DataSource`. Реализует общий код.

## 2.4 **`SingleConnectionDataSource`**  
Реализация `SmartDataSource`, которая **оборачивает одно соединение** и ==не закрывает его после каждого использования==. 

**Не поддерживает многопоточность.** 
В основном используется для тестирования. 
	Отличие от `DriverManagerDataSource`: переиспользует одно и то же физическое соединение, избегая избыточного создания новых.

## 2.5 **`DriverManagerDataSource`**  
Реализация `DataSource`, которая возвращает **новое соединение каждый раз**. 

Полезна только для тестов и standalone-окружений. Даже в тестах обычно лучше использовать пул соединений (например, `commons-dbcp`), так как это очень просто.

## 2.6 **`TransactionAwareDataSourceProxy`**  
Прокси для целевого `DataSource`, который добавляет поддержку Spring-управляемых транзакций. **Редко используется** в новом коде. Предназначен для случаев, когда существующий код требует стандартный JDBC `DataSource`, но должен участвовать в транзакциях Spring. Для нового кода всегда лучше использовать `JdbcTemplate` или `DataSourceUtils`.

# 3. Управление транзакциями: `DataSourceTransactionManager` и `JdbcTransactionManager`

## 3.1 **`DataSourceTransactionManager`**  
Реализация `PlatformTransactionManager` **для одного JDBC** `DataSource`. Привязывает JDBC-соединение к текущему потоку. 

Поддерживает:
- **Точки сохранения (savepoints)** — уровень `PROPAGATION_NESTED`
    
- **Пользовательские уровни изоляции**
    
- **Таймауты**


Код приложения должен получать соединение через 
```java
DataSourceUtils.getConnection(DataSource)
```

а не через 
```java
DataSource.getConnection()
```
напрямую. 

Все классы фреймворка (включая `JdbcTemplate`) делают это неявно.

---
## 3.2 **`JdbcTransactionManager` (новинка Spring 5.3)**  
Расширенная версия `DataSourceTransactionManager`. 
	Главное отличие — добавляет **трансляцию исключений** при commit/rollback (как это делает `JdbcTemplate`). 


В то время как `DataSourceTransactionManager` **всегда бросает только** `TransactionSystemException`,
	новый `JdbcTransactionManager` **переводит ошибки блокировок БД в соответствующие подклассы** `DataAccessException`.

**Рекомендация:** Если код приложения готов обрабатывать `DataAccessException` (а не только `TransactionSystemException`), используйте **`JdbcTransactionManager`**.

**Сравнение по исключениям:**

|Менеджер|Аналог по поведению|
|---|---|
|`JdbcTransactionManager`|Примерно эквивалентен `JpaTransactionManager` и `R2dbcTransactionManager`|
|`DataSourceTransactionManager`|Эквивалентен `JtaTransactionManager`|

## 3.3 **`LazyConnectionDataSourceProxy`**  
Прокси для пула соединений, который откладывает фактическое получение соединения до момента выполнения SQL-оператора. 

Полезен для:
- «Пустых» транзакций без реальных запросов
    
- Маршрутизации по признаку read-only
    
- Специальной поддержки read-only пула соединений
