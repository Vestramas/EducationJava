# 1. `JdbcTemplate` — центральный класс JDBC

Это основа всего. 

`JdbcTemplate` берёт на себя всю рутину: 
- открытие/закрытие соединений, 
- выполнение SQL,
- итерацию по `ResultSet` 
- и **главное** — перехват JDBC-исключений и их перевод в иерархию `DataAccessException` (unchecked).

### Архитектура и логирование
**Callback-интерфейсы:** При использовании `JdbcTemplate` код приложения должен реализовать только callback-интерфейсы с чётко определённым контрактом:

- **`PreparedStatementCreator`** — создаёт `PreparedStatement`, предоставляя SQL и параметры. `JdbcTemplate` предоставляет соединение (`Connection`).
    
- **`CallableStatementCreator`** — аналогично создаёт `CallableStatement` для хранимых процедур.
    
- **`RowCallbackHandler`** — извлекает значения из каждой строки `ResultSet`.

> [!WARNING] ПОЯСНЕНИЕ
> То есть, при вызове методов `query...`, `update...`, этим методам НУЖНО ПЕРЕДОВАТЬ в качестве параметров
> - `PreparedStatement` - строку SQL с подстановочными знаками `?`
> - `RowMapper` - для извлечения данных из `ResultSet`
> - `CallableStatement` - для работы с хранимыми процедурами



**Способы использования `JdbcTemplate`:**
1. **Прямое создание** в DAO с передачей ссылки на `DataSource`.
    
2. **Конфигурация в IoC-контейнере** и внедрение как бина. `DataSource` **всегда** должен быть бином в контейнере. 
   В первом случае бин отдаётся напрямую в сервис, во втором — в шаблон.


> [!WARNING] ЛОГИРОВАНИЕ
> **Логирование:** Весь SQL, выполняемый этим классом, логируется на уровне **DEBUG** с категорией, соответствующей полному имени класса шаблона (обычно `JdbcTemplate`). Это полезно для отладки.


Код, использующий `JdbcTemplate`, должен предоставить только SQL и способ извлечения результата.

> [!IMPORTANT] Потокобезопасен
> **Важнейшее свойство:** Экземпляр `JdbcTemplate` **потокобезопасен** после настройки. Можно создать один экземпляр и использовать его во всех DAO.

## 1.1. Конфигурация и Best Practices
**Рекомендуемый подход:** Настроить `DataSource` как бин и внедрять его в DAO, где в конструкторе создаётся `JdbcTemplate`.

```xml
// XML-конфигурация DataSource
<bean id="dataSource" class="org.apache.commons.dbcp2.BasicDataSource" destroy-method="close">
    <property name="driverClassName" value="${jdbc.driverClassName}"/>
    <property name="url" value="${jdbc.url}"/>
    <property name="username" value="${jdbc.username}"/>
    <property name="password" value="${jdbc.password}"/>
</bean>

<context:property-placeholder location="jdbc.properties"/>
```

```java
// Java-конфигурация
@Configuration
public class AppConfig {
    @Bean(destroyMethod = "close")
    BasicDataSource dataSource() {
    
        BasicDataSource ds = new BasicDataSource();
        
        ds.setDriverClassName("org.hsqldb.jdbcDriver");
        ds.setUrl("jdbc:hsqldb:hsql://localhost:");
        ds.setUsername("sa");
        ds.setPassword("");
        
        return ds;
    }

    @Bean
    JdbcCorporateEventDao corporateEventDao(DataSource dataSource) {
        return new JdbcCorporateEventDao(dataSource);
    }
}
```

```java
// Сам DAO
public class JdbcCorporateEventDao implements CorporateEventDao {
    private final JdbcTemplate jdbcTemplate;

    public JdbcCorporateEventDao(DataSource dataSource) {
        this.jdbcTemplate = new JdbcTemplate(dataSource);
    }
    // ... методы
}
```

**Альтернатива со Spring-аннотациями:**
```java
@Repository
public class JdbcCorporateEventRepository implements CorporateEventRepository {
    private JdbcTemplate jdbcTemplate;

    // DataSource внедряется автоматически через конструктор
    public JdbcCorporateEventRepository(DataSource dataSource) {
        this.jdbcTemplate = new JdbcTemplate(dataSource);
    }
}
```

## 1.2. Примеры операций
### 1.2.1 SELECT операции
```java
// Получить количество строк
int rowCount = jdbcTemplate
	.queryForObject("select count(*) from t_actor", Integer.class);

// Запрос с параметром-плейсхолдером (?)
int count = jdbcTemplate.queryForObject(
    "select count(*) from t_actor where first_name = ?", 
    Integer.class, 
    "Joe");

// Получить одну строку как объект (❗❗через RowMapper❗❗)
Actor actor = jdbcTemplate.queryForObject(
    "select first_name, last_name from t_actor where id = ?",
    (resultSet, rowNum) -> {
        Actor a = new Actor();
        a.setFirstName(resultSet.getString("first_name"));
        a.setLastName(resultSet.getString("last_name"));
        return a;
    }, 1212L);

// Получить список объектов
List<Actor> actors = jdbcTemplate.query(
    "select first_name, last_name from t_actor",
    (resultSet, rowNum) -> {
        Actor a = new Actor();
        a.setFirstName(resultSet.getString("first_name"));
        a.setLastName(resultSet.getString("last_name"));
        return a;
    });
```

**Лучшая практика: Вынос `RowMapper` в константу**
```java
private final RowMapper<Actor> actorRowMapper = (resultSet, rowNum) -> {
    Actor actor = new Actor();
    actor.setFirstName(resultSet.getString("first_name"));
    actor.setLastName(resultSet.getString("last_name"));
    return actor;
};

public List<Actor> findAllActors() {
    return this.jdbcTemplate.query(
	    "select first_name, last_name from t_actor",
		actorRowMapper
	);
}
```

### 1.2.2 **Обновления (INSERT, UPDATE, DELETE)**
**Все операции модификации** данных выполняются **через метод** `update(..)`:

```java
// Вставка
jdbcTemplate.update(
    "insert into t_actor (first_name, last_name) values (?, ?)", 
    "Leonor", 
    "Watling");

// Обновление
jdbcTemplate.update(
    "update t_actor set last_name = ? where id = ?", 
    "Banjo",
    5276L);

// Удаление
jdbcTemplate.update(
	"delete from t_actor where id = ?",
	 Long.valueOf(actorId));
```

### 1.2.3 **Выполнение DDL и произвольного SQL**
Для этого используется метод `execute(..)`:
```java
jdbcTemplate.execute(
	"create table mytable (id integer, name varchar(100))"
);
```

**Вызов хранимых процедур (простой)**
```java
jdbcTemplate.update(
	"call SUPPORT.REFRESH_ACTORS_SUMMARY(?)", 
	Long.valueOf(unionId));
```

**Получение автосгенерированных ключей**
```java
final String INSERT_SQL = "insert into my_test (name) values(?)";
KeyHolder keyHolder = new GeneratedKeyHolder();

jdbcTemplate.update(connection -> {
    PreparedStatement ps = 
	    connection.prepareStatement(
		    INSERT_SQL, 
		    new String[] { "id" }
		);
		
    ps.setString(1, "Rob");
    return ps;
    
}, keyHolder);

// keyHolder.getKey() содержит сгенерированный ключ
```

# 2. `NamedParameterJdbcTemplate` — именованные параметры
Это **обёртка над `JdbcTemplate`**, которая позволяет использовать именованные параметры (`:firstName`) вместо `?`. Это значительно повышает читаемость при большом количестве параметров.

```java
// Создание (требует DataSource)
NamedParameterJdbcTemplate namedTemplate = 
	new NamedParameterJdbcTemplate(dataSource);

// Использование с MapSqlParameterSource
String sql = "select count(*) from t_actor where first_name = :first_name";

SqlParameterSource params = new MapSqlParameterSource(
	"first_name", 
	firstName);
	
	
int count = namedTemplate.queryForObject(sql, params, Integer.class);
```

**Альтернатива через `Map`**:
```java
...
Map<String, String> params = Collections.singletonMap(
	"first_name", 
	firstName);
	
int count = namedTemplate.queryForObject(sql, params, Integer.class);
```

**ПРОДВИНУТЫЙ СПОСОБ: `BeanPropertySqlParameterSource`**
Позволяет ***использовать свойства JavaBean'а как источник параметров.*** 

Ключ `:firstName` ==в SQL будет автоматически сопоставлен с методом== `getFirstName()` бина:
```java
public class Actor {
    private String firstName;
    private String lastName;
    // геттеры и сеттеры
}

Actor exampleActor = new Actor("Leonor", "Watling");

String sql = "select count(*) from t_actor where first_name = :firstName and last_name = :lastName";

SqlParameterSource params = new 
	BeanPropertySqlParameterSource(exampleActor); //Передаем класс, а свойства подтянутся сами
	
int count = namedTemplate.queryForObject(sql, params, Integer.class);
```

> [!NOTE] **Доступ к обёрнутому `JdbcTemplate`:** 
> Через метод `getJdbcOperations()`
>

# 3. `JdbcClient` — Новинка Spring Framework 6.1
Унифицированный **fluent API**, который ***объединяет возможности позиционных и именованных параметров.*** 

Является более современной и лаконичной альтернативой прямым вызовам `JdbcTemplate`.
```java
// Инициализация
JdbcClient jdbcClient = JdbcClient.create(dataSource);
```

**Запросы с позиционными параметрами:**
```java
int count = jdbcClient.sql("select count(*) from t_actor where first_name = ?")
    .param("Joe")
    .query(Integer.class).single();
```

**Запросы с именованными параметрами:**
```java
int count = jdbcClient.sql("select count(*) from t_actor where first_name = :firstName")
    .param("firstName", "Joe")
    .query(Integer.class).single();
```

## 3.1 Маппинги
**Маппинг на объекты (автоматический):**

> [!NOTE] Ограничение
> предполагается, что класс Actor — record или POJO

```java
// Возвращает список (предполагается, что класс Actor — record или POJO)
List<Actor> actors = jdbcClient.sql("select first_name, last_name from t_actor")
    .query(Actor.class)
    .list();

// Возвращает один объект
Actor actor = jdbcClient.sql("select first_name, last_name from t_actor where id = ?")
    .param(1212L)
    .query(Actor.class)
    .single();

// Optional-результат
Optional<Actor> actor = jdbcClient.sql("select first_name, last_name from t_actor where id = ?")
    .param(1212L)
    .query(Actor.class)
    .optional();
```

### 3.1.1 **Маппинг с ручным RowMapper:**
```java
List<Actor> actors = jdbcClient.sql("select first_name, last_name from t_actor")
    .query((rs, rowNum) -> 
	    new Actor(rs.getString("first_name"), rs.getString("last_name"))
	)
    .list();
```

# 5. Обработка ошибок: `SQLExceptionTranslator`

Этот механизм — одна из главных причин использования `JdbcTemplate`

Он перехватывает JDBC-исключения (`SQLException`) и переводит их в **непроверяемые** (unchecked) исключения из пакета `org.springframework.dao` (`DataAccessException`).

- **По умолчанию (Spring 6.0+):** 
  Используется `SQLExceptionSubclassTranslator`, который анализирует подклассы `SQLException` (JDBC 4). Этого достаточно для большинства БД.
    
- **Если на classpath есть `sql-error-codes.xml`:** 
  Используется `SQLErrorCodeSQLExceptionTranslator`, который более точен, так как опирается на коды ошибок конкретного вендора.


## 5.1 **Создание кастомного переводчика:**
Если нужно **заточить перевод исключений под конкретную БД:**
```java
public class CustomSQLErrorCodesTranslator 
	extends SQLErrorCodeSQLExceptionTranslator 
{
    @Override
    protected DataAccessException customTranslate(
	    String task, 
	    String sql, 
	    SQLException sqlEx) 
	{
        if (sqlEx.getErrorCode() == -12345) {
            return new DeadlockLoserDataAccessException(task, sqlEx);
        }
        return null; // Возвращаем null, чтобы сработал стандартный переводчик
    }
}
```

И затем передаём его в `JdbcTemplate`:
```java
JdbcTemplate jdbcTemplate = new JdbcTemplate(dataSource);

CustomSQLErrorCodesTranslator tr = new CustomSQLErrorCodesTranslator();
tr.setDataSource(dataSource);

jdbcTemplate.setExceptionTranslator(tr);
```
