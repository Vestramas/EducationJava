Большинство JDBC-драйверов обеспечивают повышенную производительность при объединении множества вызовов одного и того же подготовленного выражения (prepared statement) в пакет. Группировка обновлений в пакеты сокращает количество обращений к базе данных.

# 1. Базовые пакетные операции с `JdbcTemplate`
Для пакетной обработки используется метод `batchUpdate` и реализация интерфейса `BatchPreparedStatementSetter`. 

Нужно реализовать два метода:
- **`getBatchSize()`** — возвращает размер текущего пакета.
    
- **`setValues(PreparedStatement ps, int i)`** — устанавливает значения параметров для i-го элемента пакета.

```java
public class JdbcActorDao implements ActorDao {
    private JdbcTemplate jdbcTemplate;

    public void setDataSource(DataSource dataSource) {
        this.jdbcTemplate = new JdbcTemplate(dataSource);
    }

    public int[] batchUpdate(final List<Actor> actors) 
    {
    return this.jdbcTemplate.batchUpdate(
        "update t_actor set first_name = ?, last_name = ? where id = ?",
		
		new BatchPreparedStatementSetter() {
			public void setValues(PreparedStatement ps, int i) 
			throws SQLException 
			{
				Actor actor = actors.get(i);
				ps.setString(1, actor.getFirstName());
				ps.setString(2, actor.getLastName());
				ps.setLong(3, actor.getId().longValue());
			}
			public int getBatchSize() {
				return actors.size();
			}
		});
    }
}
```

**`InterruptibleBatchPreparedStatementSetter`:** Если вы обрабатываете поток обновлений или читаете из файла, последний пакет может быть неполным. Этот интерфейс позволяет прервать пакетную обработку, когда входные данные исчерпаны, через метод `isBatchExhausted`.

# 2. Пакетные операции со списком объектов
Более простой способ — ==передать все значения параметров в виде списка==. Фреймворк сам проходит по значениям и использует внутренний установщик параметров.

---
## 2.1 **С именованными параметрами (через `NamedParameterJdbcTemplate`):**


> [!NOTE] `SqlParameterSourceUtils` - утилитарный класс, для создание массива параметров
> Используется `SqlParameterSourceUtils.createBatch()` для создания массива `SqlParameterSource` из списка объектов:

```java
public class JdbcActorDao implements ActorDao {
    private NamedParameterJdbcTemplate namedParameterJdbcTemplate;

    public void setDataSource(DataSource dataSource) {
        this.namedParameterJdbcTemplate = 
	        new NamedParameterJdbcTemplate(dataSource);
    }

    public int[] batchUpdate(List<Actor> actors) {
        return 
	        this.namedParameterJdbcTemplate
		        .batchUpdate(
		            "update t_actor set first_name = :firstName, last_name = :lastName where id = :id",
		            SqlParameterSourceUtils.createBatch(actors)
		        );
    }
}
```

---
## 2.2 **С классическими плейсхолдерами `?`:**
==Передаётся список объектных массивов ==(`List<Object[]>`), где каждый массив содержит значения для одного набора плейсхолдеров в правильном порядке:
```java
public int[] batchUpdate(final List<Actor> actors) {
    List<Object[]> batch = new ArrayList<>();
    
    for (Actor actor : actors) {
        Object[] values = new Object[] {
            actor.getFirstName(), 
            actor.getLastName(), 
            actor.getId()
        };
        batch.add(values);
    }
    
    return this.jdbcTemplate.batchUpdate(
        "update t_actor set first_name = ?, last_name = ? where id = ?", 
        batch);
}
```

**Возвращаемое значение:** Все методы пакетного обновления возвращают массив `int[]`, содержащий количество затронутых строк для каждой записи пакета. Если JDBC-драйвер не может предоставить количество, возвращается значение **-2**.

## 2.3 **Проблема производительности при определении типов JDBC:**
При автоматической установке значений Spring по умолчанию вызывает `ParameterMetaData.getParameterType` для определения JDBC-типа по Java-типу. 

Это может быть дорогостоящей операцией.
- **Совет:** Используйте свежую версию JDBC-драйвера.
    
- **Оптимизация:** Можно установить системное свойство `spring.jdbc.getParameterType.ignore=true`, чтобы отключить этот механизм.
    
- **Spring 6.1.2+:** Начиная с версии 6.1.2, Spring **по умолчанию отключает** `getParameterType` для PostgreSQL и MS SQL Server, так как это значительно повышает производительность пакетных операций. 
  Если возникнут побочные эффекты (например, установка byte-массива в null без указания типа), можно вернуть старое поведение флагом `spring.jdbc.getParameterType.ignore=false`.
    

**Явное указание JDBC-типов (альтернативы):**
- Использовать `BatchPreparedStatementSetter` (как в разделе 1)
    
- Передать явный массив типов с `List<Object[]>`
    
- Использовать `registerSqlType` на `MapSqlParameterSource`
    
- Использовать `BeanPropertySqlParameterSource` (выводит SQL-тип из типа свойства Java)
    
- Использовать `SqlParameterValue` вместо обычных `null`

## 2.4 Пакетные операции с разбиением на несколько пакетов
**Если данных очень много**, ==можно РАЗБИТЬ ИХ НА НЕСКОЛЬКО ППАКЕТОВ меньшего размера ОДНОЙ ОПЕРАЦИЕЙ==. 

Для этого используется перегруженный метод `batchUpdate`, принимающий:
- SQL-выражение
    
- `Collection` объектов с параметрами
    
- Размер одного пакета
    
- `ParameterizedPreparedStatementSetter` (лямбда для установки значений)

```java
public int[][] batchUpdate(final Collection<Actor> actors) {
    int[][] updateCounts = jdbcTemplate.batchUpdate(
        "update t_actor set first_name = ?, last_name = ? where id = ?",
        actors,
        100, // размер одного пакета
        (PreparedStatement ps, Actor actor) -> {
            ps.setString(1, actor.getFirstName());
            ps.setString(2, actor.getLastName());
            ps.setLong(3, actor.getId().longValue());
        });
    return updateCounts;
}
```

**Возвращаемое значение:** 
Метод возвращает **двумерный массив** `int[][]`:
- Внешний массив — каждый элемент соответствует одному выполненному пакету.
    
- Внутренний массив — количество затронутых строк для каждого обновления внутри пакета.
    
- Если JDBC-драйвер не может предоставить количество, возвращается **-2**.
