# 1. Управление транзакциями
Всесторонняя поддержка транзакций — одна из главных причин использовать Spring Framework. 

Spring предоставляет **единую абстракцию** для управления транзакциями, которая работает одинаково для разных API.

## 1.1. Преимущества транзакционной модели Spring
- **Единая модель программирования** для разных API: JTA, JDBC, Hibernate, JPA. Вы используете одни и те же аннотации/методы независимо от технологии.
    
- **Более простой API**, чем у сложных API вроде JTA.
    
- **Отличная интеграция** с абстракциями доступа к данным Spring (`JdbcTemplate`, `JpaRepository` и т.д.).

# 2. Преимущества транзакционной модели Spring
Традиционно у разработчиков было два выбора: **глобальные** или **локальные** транзакции. Оба подхода имеют серьёзные ограничения.

## 2.1. Глобальные транзакции (JTA)
- Позволяют работать с **несколькими транзакционными ресурсами** (разные БД, очереди сообщений).
    
- Управляются сервером приложений через **JTA** (Java Transaction API).
    
- **Недостатки:**
    - JTA — громоздкий API (в том числе из-за модели исключений).
        
    - `UserTransaction` нужно получать через **JNDI**, что привязывает код к серверу приложений.
        
    - **Ограничивает переиспользование кода** — JTA доступен только в среде сервера приложений.
        
- **EJB CMT** (Container-Managed Transactions) — декларативный подход, но всё равно привязан к JTA, JNDI и EJB. Непривлекателен из-за общих недостатков EJB.
    

## 2.2. Локальные транзакции
- Привязаны к конкретному ресурсу (например, транзакция JDBC-соединения).
    
- Проще в использовании, но **не могут работать с несколькими ресурсами**.
    
- **Инвазивны** для программной модели.
    
- Большинство приложений используют один транзакционный ресурс, но это всё равно ограничение.
    

## 2.3. Решение Spring: Единая модель программирования
Spring устраняет недостатки обоих подходов, предоставляя **единую согласованную модель**:

- **Один и тот же код работает в любом окружении** — вы пишете код один раз, а стратегия управления транзакциями выбирается в конфигурации.
    
- **Декларативные транзакции** (рекомендуемый способ): разработчик пишет минимум кода, не зависит от API Spring или других API.
    
- **Программные транзакции**: работа с абстракцией Spring, которая может работать поверх любой транзакционной инфраструктуры.
