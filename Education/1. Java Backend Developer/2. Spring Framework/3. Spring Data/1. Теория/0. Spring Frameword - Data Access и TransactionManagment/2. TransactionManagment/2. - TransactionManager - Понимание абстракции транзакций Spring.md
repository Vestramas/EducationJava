# 1. Понимание абстракции транзакций Spring

> [!NOTE] `TransactionManager`
> Ключ к абстракции транзакций Spring — понятие **стратегии транзакции**, определяемой через `TransactionManager`.

Он обязателен, когда при работе с БД (так как нужно управлять транзакциями)

НО здесь есть важный нюанс, который зависит от того, как вы работаете с БД.

Если вы используете Spring JDBC напрямую (через `JdbcTemplate`)
`JdbcTemplate` **может работать и без `PlatformTransactionManager`**. 

Каждый вызов `jdbcTemplate.update(...)` выполнится в режиме auto-commit — каждое изменение сразу фиксируется в БД.

**Но!** Как только вам нужно, чтобы **несколько операций выполнились как единое целое** (все или ничего), — `PlatformTransactionManager` становится **обязательным**. 

> [!WARNING] СВЯЗЬ С АННОТАЦИЕЙ `@Transactional`
> Без него аннотация `@Transactional` просто не сработает.


## 1.1. `PlatformTransactionManager` (Императивные транзакции)

Основной SPI-интерфейс для императивного управления транзакциями:
```java
public interface PlatformTransactionManager 
	extends TransactionManager 
{
    TransactionStatus getTransaction(
	    TransactionDefinition definition
	) throws TransactionException;
	
    void commit(TransactionStatus status) 
	    throws TransactionException;
	    
    void rollback(TransactionStatus status) 
	    throws TransactionException;
}
```

**Ключевые особенности:**
- Это **SPI** (Service Provider Interface), но его можно использовать и напрямую из кода.
    
- Это **интерфейс** — его легко замокать/застабить для тестов.
    
- **Не привязан к JNDI** или другой стратегии поиска.
    
- `TransactionException` (бросается методами) — **unchecked** (наследуется от `RuntimeException`). Разработчик не обязан её перехватывать.
    
- `getTransaction(..)` возвращает `TransactionStatus`, который может представлять **новую** или **существующую** транзакцию (если есть активная в текущем стеке вызовов). `TransactionStatus` привязан к **потоку выполнения**.

---
## 1.2. `ReactiveTransactionManager` (Реактивные транзакции)

Для реактивных приложений (Project Reactor, Kotlin Coroutines):
```java
public interface ReactiveTransactionManager 
	extends TransactionManager 
{
    Mono<ReactiveTransaction> getReactiveTransaction(
	    TransactionDefinition definition
	) throws TransactionException;
	
    Mono<Void> commit(ReactiveTransaction status) 
	    throws TransactionException;
	    
    Mono<Void> rollback(ReactiveTransaction status) 
	    throws TransactionException;
}
```

Те же принципы, что и у `PlatformTransactionManager`, но возвращает реактивные типы (`Mono`).

---
## 1.3. `TransactionDefinition` — параметры транзакции
| Параметр                          | Описание                                                                                                                                                   |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Propagation (Распространение)** | *Как себя вести*, ***если метод вызывается внутри существующей транзакции***. <br><br>Например: продолжить в ней же, создать новую, приостановить текущую. |
| **Isolation (Изоляция)**          | Степень изолированности от других транзакций (видит ли незакоммиченные изменения?).                                                                        |
| **Timeout (Таймаут)**             | Через сколько транзакция будет автоматически откачена.                                                                                                     |
| **Read-only**                     | Транзакция только для чтения (оптимизация для Hibernate и др.).                                                                                            |

## 1.4. `TransactionStatus` — управление и статус
Интерфейс для программного контроля выполнения транзакции:
```java
public interface TransactionStatus extends TransactionExecution, SavepointManager, Flushable 
{
    boolean isNewTransaction();   // Это новая транзакция?
    boolean hasSavepoint();       // Есть ли точка сохранения?
    void setRollbackOnly();       // Пометить транзакцию на откат
    boolean isRollbackOnly();     // Помечена ли на откат?
    void flush();                 // Принудительный сброс
    boolean isCompleted();        // Завершена ли транзакция?
}
```

## 1.5. КОНФИГУРАЦИЯ `TransactionManager`

Определение правильной реализации `TransactionManager` критически важно. 

Обычно настраивается через dependency injection.

> [!NOTE] Имя `txManager` - НЕОБЯЗАТЕЛЕНО
> 

**Для локальных JDBC-транзакций:**
```xml
<!-- DataSource -->
<bean id="dataSource" class="org.apache.commons.dbcp.BasicDataSource" destroy-method="close">
    <property name="driverClassName" value="${jdbc.driverClassName}"/>
    <property name="url" value="${jdbc.url}"/>
    <property name="username" value="${jdbc.username}"/>
    <property name="password" value="${jdbc.password}"/>
</bean>

<!-- TransactionManager -->
<bean id="txManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource"/>
</bean>
```

```java
@Bean
public DataSource dataSource() {
    BasicDataSource ds = new BasicDataSource();
    ds.setDriverClassName("org.hsqldb.jdbcDriver");
    ds.setUrl("jdbc:hsqldb:hsql://localhost:");
    ds.setUsername("sa");
    ds.setPassword("");
    return ds;
}

@Bean
public PlatformTransactionManager txManager(
	DataSource dataSource
) {
    return new DataSourceTransactionManager(dataSource);
}
```


**Для JTA (глобальные транзакции в Java EE-контейнере):**
```xml
<jee:jndi-lookup id="dataSource" jndi-name="jdbc/jpetstore"/>

<bean id="txManager" class="org.springframework.transaction.jta.JtaTransactionManager"/>
```

```java
@Bean
public DataSource dataSource() {
    JndiDataSourceLookup lookup = new JndiDataSourceLookup();
    return lookup.getDataSource("jdbc/jpetstore");
}

@Bean
public PlatformTransactionManager txManager() {
    return new JtaTransactionManager();
}
```

`JtaTransactionManager` не нужно знать о `DataSource` — он использует глобальную инфраструктуру контейнера. 

> **Важно:** При использовании JTA определение `txManager` одинаково для JDBC, Hibernate, JPA — потому что JTA-транзакции глобальны и могут вовлекать любые ресурсы.

# 2. КОНФИГУРАЦИЯ для Hibernate

**Локальные транзакции Hibernate:**
```xml
<bean id="sessionFactory" class="org.springframework.orm.hibernate5.LocalSessionFactoryBean">
    <property name="dataSource" ref="dataSource"/>
    <!-- маппинги и настройки -->
</bean>

<bean id="txManager" class="org.springframework.orm.hibernate5.HibernateTransactionManager">
    <property name="sessionFactory" ref="sessionFactory"/>
</bean>
```

```java
@Bean
public LocalSessionFactoryBean sessionFactory(
	DataSource dataSource
) {
    LocalSessionFactoryBean factory = 
	    new LocalSessionFactoryBean();
	    
    factory.setDataSource(dataSource);
    factory.setMappingResources("org/springframework/samples/petclinic/hibernate/petclinic.hbm.xml");
    
    Properties props = new Properties();
    props.setProperty(
	    "hibernate.dialect", 
	    "${hibernate.dialect}"
	);
	
    factory.setHibernateProperties(props);
    return factory;
}

@Bean
public PlatformTransactionManager txManager(SessionFactory sessionFactory) {
    return new HibernateTransactionManager(sessionFactory);
}
```

**Hibernate с JTA:**
```xml
<!-- Hibernate должен знать о JTA -->
<bean id="sessionFactory" class="org.springframework.orm.hibernate5.LocalSessionFactoryBean">
    <property name="dataSource" ref="dataSource"/>
    <property name="hibernateProperties">
        <value>
            hibernate.dialect=${hibernate.dialect}
            hibernate.transaction.coordinator_class=jta
            hibernate.connection.handling_mode=DELAYED_ACQUISITION_AND_RELEASE_AFTER_STATEMENT
        </value>
    </property>
</bean>

<bean id="txManager" class="org.springframework.transaction.jta.JtaTransactionManager"/>
```

```java
@Bean
public LocalSessionFactoryBean sessionFactory(
	DataSource dataSource
) {
    LocalSessionFactoryBean factory = 
	    new LocalSessionFactoryBean();
	    
    factory.setDataSource(dataSource);
    factory.setMappingResources("org/springframework/samples/petclinic/hibernate/petclinic.hbm.xml");
    
    
    Properties props = new Properties();
    props.setProperty(
	    "hibernate.dialect", 
	    "${hibernate.dialect}"
	);
    props.setProperty(
	    "hibernate.transaction.coordinator_class", 
	    "jta"
	);
    props.setProperty(
	    "hibernate.connection.handling_mode",
	    "DELAYED_ACQUISITION_AND_RELEASE_AFTER_STATEMENT"
	);
	
	
    factory.setHibernateProperties(props);
    return factory;
}

@Bean
public PlatformTransactionManager txManager() {
    return new JtaTransactionManager();
}
```

**Главный принцип:** Код приложения **не меняется** при смене стратегии транзакций. Вы просто меняете конфигурацию — даже при переходе от локальных транзакций к глобальным.
