После настройки `TransactionManager` 
(например, 
- `DataSourceTransactionManager` → `DataSource`, 
- `HibernateTransactionManager` → `SessionFactory`) 

встаёт вопрос: **как код приложения обеспечивает создание, переиспользование и очистку этих ресурсов в рамках транзакции**?

##### Для чего?
Представьте: у вас есть метод с `@Transactional`, внутри которого вызывается два разных DAO. 

Без синхронизации каждый DAO может открыть своё собственное соединение с БД. 

Но для транзакции нужно, чтобы **оба DAO использовали одно и то же соединение** — иначе `commit`/`rollback` для них будет работать раздельно, что сломает атомарность.

---
## 1.1. Высокоуровневый подход (рекомендуемый)
Использовать **шаблонные API Spring** или **нативные ORM API с транзакционно-осведомлёнными фабриками/прокси**. 

Эти решения **внутри себя** обрабатывают:
- Создание и переиспользование ресурсов.
    
- Очистку.
    
- Опциональную синхронизацию с транзакцией.
    
- Маппинг исключений.
    

**Примеры:**
- `JdbcTemplate` (для JDBC).
    
- Нативные ORM API (Hibernate `SessionFactory`, JPA `EntityManager`) при правильной настройке.

Код приложения фокусируется только на бизнес-логике и не занимается рутиной.

---
## 1.2. Низкоуровневый подход (для прямого доступа)
Если нужно работать с ресурсами напрямую (минуя `JdbcTemplate`), используются утилитные классы:

|Класс|Для технологии|
|---|---|
|`DataSourceUtils`|JDBC (`Connection`)|
|`EntityManagerFactoryUtils`|JPA (`EntityManager`)|
|`SessionFactoryUtils`|Hibernate (`Session`)|

**Пример с JDBC:**
```java
Connection conn = DataSourceUtils.getConnection(dataSource);
```

**Что делает `DataSourceUtils.getConnection()`:**
1. Если существует активная транзакция с уже синхронизированным соединением — **возвращает его** (переиспользование).
    
2. Если нет — создаёт новое соединение, **опционально синхронизирует** с текущей транзакцией и делает доступным для последующего переиспользования в той же транзакции.
    
3. Все `SQLException` оборачивает в `CannotGetJdbcConnectionException` (unchecked, часть иерархии `DataAccessException`).
    

**Важно:** Этот подход работает **даже без управления транзакциями Spring** (синхронизация опциональна).

## 1.3. `TransactionAwareDataSourceProxy` (самый низкий уровень)

Прокси для `DataSource`, который добавляет осведомлённость о Spring-транзакциях. Аналог транзакционного JNDI `DataSource` из Java EE.

**Почти никогда не нужен.** Используется только если существующий код требует стандартный JDBC `DataSource`, но должен участвовать в транзакциях Spring. Для нового кода всегда лучше использовать `JdbcTemplate` или `DataSourceUtils`.
