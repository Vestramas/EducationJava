Большинство пользователей Spring выбирают **декларативное** управление транзакциями. 

Этот подход минимально влияет на код приложения и лучше всего соответствует идеалам _неинвазивного_ лёгкого контейнера.

## 1. Как это работает
Декларативное управление транзакциями в Spring реализовано через **Spring AOP** (аспектно-ориентированное программирование). 

Однако вам **не обязательно** ***глубоко понимать AOP***, чтобы эффективно использовать этот механизм — весь необходимый код уже есть в дистрибутиве Spring.

## 2. Сходства и различия с EJB CMT
Spring-подход похож на EJB Container Managed Transactions, но с важными отличиями:

| Особенность                   | Spring                                                                             | EJB CMT                                                        |
| ----------------------------- | ---------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| **Окружение**                 | Работает в любом окружении (JTA или локальные транзакции через JDBC/JPA/Hibernate) | Привязан к JTA и серверу приложений                            |
| **К каким классам применять** | К любым классам (POJO)                                                             | Только к специальным классам (EJB)                             |
| **Правила отката**            | Декларативные правила отката по исключениям                                        | Автоматический откат только на системные исключения (runtime)  |
| **Кастомизация**              | Можно добавить ==произвольное поведение при откате== через AOP                     | Только `setRollbackOnly()`<br>(на объекте `TransactionStatus`) |
| **Удалённые вызовы**          | Не поддерживает propagation транзакций через удалённые вызовы                      | Поддерживают high-end серверы приложений                       |

---
## 3. Правила отката (Rollback Rules)
Одна из ключевых возможностей Spring — **декларативные правила отката**. 

*Вы можете указать*, ***какие исключения должны вызывать автоматический откат транзакции***, **без написания Java-кода**.

**Поведение по умолчанию:**
- Как и в EJB, автоматический откат происходит только на **unchecked-исключения** (наследники `RuntimeException` и `Error`).
    
- На **checked-исключения** (`Exception` и его подклассы, кроме `RuntimeException`) откат **не** происходит.


**Главное преимущество:** Бизнес-объекты не зависят от инфраструктуры транзакций. Им не нужно импортировать Spring API или другие транзакционные API.

**Пример:** Вы можете настроить правило, что `MyApplicationException` всегда вызывает откат — это делается в конфигурации или через аннотацию, а не в коде.

# 1. Устройство декларативной реализации транзакций Spring

Недостаточно просто знать, ==что нужно добавить== `@Transactional` и `@EnableTransactionManagement`. 


Этот раздел объясняет **внутреннее устройство** декларативной транзакционной инфраструктуры.

## 1.1. AOP-прокси и TransactionInterceptor
Ключевые концепции декларативных транзакций Spring:
1. Поддержка включается через **AOP-прокси**.
    
2. Транзакционное поведение (advice) **управляется** **метаданными** (==аннотации== или XML).
    
3. ==Результат их объединения== — `AOP-прокси`, использующий **`TransactionInterceptor`** вместе с **`TransactionManager`** для управления транзакциями вокруг вызовов методов.

> [!IMPORTANT] Вызов прокси
> **Простыми словами:** Когда вы вызываете метод с `@Transactional`, **вы ОБРАЩАЕТЕСЬ НЕ напрямую В ВАШЕМУ БИНУ, а К ЕГО ПРОКСИ.** 
> 
> Прокси перехватывает вызов и запускает логику транзакции (начать → вызвать реальный метод → закоммитить или откатить).

## 1.2. Императивные vs Реактивные транзакции
`TransactionInterceptor` автоматически определяет «вкус» управления транзакциями **по возвращаемому типу метода**:

| Возвращаемый тип                                                    | Тип транзакции                  | Требуемый менеджер           |
| ------------------------------------------------------------------- | ------------------------------- | ---------------------------- |
| Обычные типы, `void`                                                | **Императивные** <br>транзакции | `PlatformTransactionManager` |
| Реактивные типы: <br>`Publisher` <br>(Mono/Flux), <br>`Kotlin Flow` | **Реактивные** <br>транзакции   | `ReactiveTransactionManager` |

## 1.3. Привязка транзакций, помеченных @Transactional

**Императивные транзакции:**
- Привязаны к **потоку выполнения** (thread-bound) через `PlatformTransactionManager`.
    
- Транзакция доступна всем операциям доступа к данным в текущем потоке.
    
- **Важно:** Транзакция **НЕ распространяется** на новые потоки, запущенные внутри метода.
    

**Реактивные транзакции:**
- Используют **контекст Reactor** вместо thread-local переменных.
    
- Все участвующие операции доступа к данным должны выполняться в **одном реактивном конвейере** (pipeline) в одном контексте Reactor.
    
- **Требование:** Если используется `ReactiveTransactionManager`, все методы с `@Transactional` **должны возвращать реактивный тип**.
