> [!IMPORTANT] Бросай исключение для отката
> **Рекомендуемый способ** указать, что транзакцию нужно откатить — **ВЫБРОСИТЬ ИСКЛЮЧЕНИЕ** из кода, выполняющегося в контексте транзакции. 

Инфраструктура Spring перехватывает любое необработанное исключение при всплытии по стеку вызовов и решает, пометить ли транзакцию на откат.

**Поведение по умолчанию:**
- Откат происходит только на **unchecked-исключения** (`RuntimeException` и его подклассы).
    
- Экземпляры `Error` также вызывают откат.
    
- **Checked-исключения** (наследники `Exception`, кроме `RuntimeException`) ==НЕ ВЫЗЫВАЮТ ОТКАТ==.

---
## 1.1. Специальная обработка (Spring 6.1+)
### 1.1.1 **`CompletableFuture` и `Future`:** 

Если метод с `@Transactional` (часто в сочетании с `@Async`) возвращает `CompletableFuture`, которая была **завершена исключением** (exceptionally completed) на момент возврата из метода, Spring автоматически инициирует откат:
```java
@Transactional @Async
public CompletableFuture<String> myTransactionalMethod() {
    try {
        return CompletableFuture.completedFuture(
	        delegate.myDataAccessOperation()
	    );
    } catch (DataAccessException ex) {
	    // ИСКЛЮЧЕНИЕ Приведёт к откату
        return CompletableFuture.failedFuture(ex); 
    }
}
```

### 1.1.2 **Vavr `Try`:**  
Spring также поддерживает автоматический откат, если метод возвращает `Try` и он содержит `Failure`:
```java
@Transactional
public Try<String> myTransactionalMethod() {
	// Failure -> откат
    return Try.of(delegate::myDataAccessOperation); 
}
```

## 1.2 Правила отката (Rollback Rules)
Правила отката определяют, должно ли исключение вызывать откат, на основе **типа** или **паттерна** исключения.

| Способ конфигурации            | Атрибуты                                                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| **XML**                        | `rollback-for`, <br><br>`no-rollback-for` (паттерны)                                                                      |
| **Аннотация `@Transactional`** | `rollbackFor`<br><br>`noRollbackFor` (типы),<br><br>`rollbackForClassName`<br><br>`noRollbackForClassName` <br>(паттерны) |

**Правила по типу (`rollbackFor`):**
- Типобезопасно: указанный класс `C` совпадает с исключением `T`, если `T` равен `C` или является подклассом `C`.
    
- Например: `rollbackFor = jakarta.servlet.ServletException.class` совпадёт с `ServletException` и его подклассами.


**Правила по паттерну (`rollback-for` / `rollbackForClassName`):**
- Паттерн — полное или частичное имя класса (без wildcard).
    
- Считается совпадением, если имя исключения **содержит** указанный паттерн.
    
- **Осторожно:** `"Exception"` совпадёт почти со всем. Лучше использовать `"java.lang.Exception"` для всех checked-исключений. Может привести к непреднамеренным совпадениям (например, `"CustomException"` совпадёт с `CustomExceptionV2`).

### 1.2.1 Примеры конфигурации
##### 1. **Откат на конкретное checked-исключение:**
XML-config:
```xml
<tx:advice id="txAdvice" transaction-manager="txManager">
    <tx:attributes>
        <tx:method name="get*" read-only="true" rollback-for="NoProductInStockException"/>
        <tx:method name="*"/>
    </tx:attributes>
</tx:advice>
```

Java-config (через `@Transactional`):
```java
@Transactional(
	readOnly = true, 
	rollbackFor = NoProductInStockException.class)
public Foo getFoo(String fooName) {
    // ...
}
```


Java-config (через `TransactionInterceptor` для нескольких методов):
```java
@Bean
public TransactionInterceptor txAdvice(
	PlatformTransactionManager txManager
) {
    NameMatchTransactionAttributeSource source = 
	    new NameMatchTransactionAttributeSource();

    // Методы get*: только для чтения + откат на NoProductInStockException
    RuleBasedTransactionAttribute getMethods = 
	    new RuleBasedTransactionAttribute();
    getMethods.setReadOnly(true);
    getMethods
	    .getRollbackRules()
	    .add(
		    new RollbackRuleAttribute( 
			    NoProductInStockException.class)
			);
    source.addTransactionalMethod("get*", getMethods);

    // Остальные методы
    source.addTransactionalMethod(
	    "*", 
	    new RuleBasedTransactionAttribute()
	);

    return new TransactionInterceptor(txManager, source);
}
```

---
##### 2. **Запрет отката на конкретное исключение:**
XML-config:
```xml
<tx:advice id="txAdvice">
    <tx:attributes>
        <tx:method name="updateStock" no-rollback-for="InstrumentNotFoundException"/>
        <tx:method name="*"/>
    </tx:attributes>
</tx:advice>
```

Java-config (через `@Transactional`):
```java
@Transactional(
	noRollbackFor = InstrumentNotFoundException.class)
public void updateStock(Stock stock) {
    // ...
}
```

Java-config (через `TransactionInterceptor` для нескольких методов):
```java
@Bean
public TransactionInterceptor txAdvice(
	PlatformTransactionManager txManager
) {
    NameMatchTransactionAttributeSource source = 
	    new NameMatchTransactionAttributeSource();

    // updateStock: откат на все исключения, КРОМЕ InstrumentNotFoundException
    RuleBasedTransactionAttribute updateStock = 
	    new RuleBasedTransactionAttribute();
	    
    updateStock.getRollbackRules().add(
        new RollbackRuleAttribute(
	        Throwable.class)
	    );
        
    updateStock.getRollbackRules().add(
        new NoRollbackRuleAttribute(
	        InstrumentNotFoundException.class)
	    );
	    
	    
    source.addTransactionalMethod("updateStock", updateStock);

    // Остальные методы
    source.addTransactionalMethod(
	    "*", 
	    new RuleBasedTransactionAttribute()
	);

    return new TransactionInterceptor(txManager, source);
}
```


> [!IMPORTANT] Приоритет правил
> Побеждает **наиболее специфичное** правило. 

В примере ниже все исключения (включая checked) вызывают откат, **кроме** `InstrumentNotFoundException`:

XML-config:
```xml
<tx:advice id="txAdvice">
    <tx:attributes>
        <tx:method name="*" rollback-for="Throwable" no-rollback-for="InstrumentNotFoundException"/>
    </tx:attributes>
</tx:advice>
```

Java-config (через `TransactionInterceptor`):
```java
@Bean
public TransactionInterceptor txAdvice(
	PlatformTransactionManager txManager
) {
    NameMatchTransactionAttributeSource source = 
	    new NameMatchTransactionAttributeSource();

    // Все методы: откат на всё, кроме InstrumentNotFoundException
    RuleBasedTransactionAttribute defaultMethods = 
	    new RuleBasedTransactionAttribute();
	    
    defaultMethods.getRollbackRules().add(
        new RollbackRuleAttribute(
	        Throwable.class)
	    );
    defaultMethods.getRollbackRules().add(
        new NoRollbackRuleAttribute(
	        InstrumentNotFoundException.class)
	    );
	    
	    
    source.addTransactionalMethod("*", defaultMethods);

    return new TransactionInterceptor(txManager, source);
}
```

> [!NOTE] ИСПОЛЬЗУЙ ПАРАМЕТРЫ АННОТАЦИИ `@Transactional` для указания правил
> **Примечание:** При использовании `@Transactional` на отдельных методах проще указывать правила непосредственно в аннотации, а не через `TransactionInterceptor`. 
> 
> `TransactionInterceptor` удобен, когда нужно применить одни и те же правила к множеству методов централизованно (по паттернам имён).

##### 3. Программный откат (не рекомендуется)
Можно вызвать откат вручную, но это **инвазивно** и привязывает код к Spring:
```java
public void resolvePosition() {
    try {
        // бизнес-логика...
    } catch (NoProductInStockException ex) {
        TransactionAspectSupport
	        .currentTransactionStatus().setRollbackOnly();
    }
}
```

**Настоятельно рекомендуется** использовать декларативный подход. Программный откат нарушает чистую POJO-архитектуру.
