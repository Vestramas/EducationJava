Аннотация `@Transactional` — это **альтернативный XML-конфигурации способ настройки транзакций.** 

> **Примечание:** 
> Spring также поддерживает стандартную аннотацию `jakarta.transaction.Transactional` в качестве замены собственной аннотации Spring.

Декларация семантики транзакций прямо в Java-коде размещает настройки максимально близко к коду.

## 1.1 Базовое использование и включение обработки
Аннотация `@Transactional` может применяться на уровне класса, интерфейса или отдельного метода. 
- При использовании ==на уровне класса ==она задаёт **настройки по умолчанию для всех методов** класса (а также его подклассов). 
	
- Каждый ==метод== может быть аннотирован индивидуально для переопределения этих настроек. 

**Важно:** аннотация уровня класса не применяется к методам классов-предков вверх по иерархии — унаследованные методы нужно локально переобъявить, чтобы они участвовали в аннотации подкласса.

```java
@Transactional
public class DefaultFooService implements FooService {

    @Override
    public Foo getFoo(String fooName) { ... }

    @Override
    public Foo getFoo(String fooName, String barName) { ... }

    @Override
    public void insertFoo(Foo foo) { ... }

    @Override
    public void updateFoo(Foo foo) { ... }
}
```

**Включение обработки `@Transactional`:**
Включение обработки `@Transactional` происходит двумя способами:

Java-config — аннотация `@EnableTransactionManagement` в `@Configuration`-классе:
```java
@Configuration
@EnableTransactionManagement //НЕОБХОДИМАЯ АННОТАЦИЯ
public class AppConfig {

    @Bean
    public FooService fooService() {
        return new DefaultFooService();
    }

    @Bean
    public PlatformTransactionManager txManager(
	    DataSource dataSource
	) {
        return new DataSourceTransactionManager(dataSource);
    }
}
```

ИЛИ xml:
```xml
<tx:annotation-driven/>

<bean id="fooService" class="x.y.service.DefaultFooService"/>

<bean id="txManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource"/>
</bean>
```

**Именование бина `TransactionManager`:** И в Java, и в XML можно опустить явное указание имени менеджера, если бин называется **`transactionManager`**. 
Если имя другое — нужно указать его явно через атрибут `transaction-manager` (в XML) или через квалификатор/атрибут `value` (в аннотации).

## 1.2. `@Transactional` — значения по умолчанию

|Свойство|Значение по умолчанию|
|---|---|
|`propagation`|`PROPAGATION_REQUIRED`|
|`isolation`|`ISOLATION_DEFAULT`|
|`readOnly`|`false` (чтение-запись)|
|`timeout`|Таймаут по умолчанию базовой системы (или отсутствует)|
|`rollbackFor`|`RuntimeException` и `Error` → rollback; checked-исключения → НЕ rollback|

---
## 1.3. Атрибуты аннотации `@Transactional`

|Свойство|Тип|Описание|
|---|---|---|
|`value`|`String`|Квалификатор, указывающий используемый `TransactionManager`.|
|`transactionManager`|`String`|Псевдоним для `value`.|
|`label`|`String[]`|Метки для описательного описания транзакции. Могут оцениваться менеджерами для специфичного поведения.|
|`propagation`|`Propagation`|Режим распространения.|
|`isolation`|`Isolation`|Уровень изоляции (только для `REQUIRED` или `REQUIRES_NEW`).|
|`timeout`|`int` (секунды)|Таймаут (только для `REQUIRED` или `REQUIRES_NEW`).|
|`timeoutString`|`String`|Альтернатива: таймаут как строка (например, плейсхолдер).|
|`readOnly`|`boolean`|Только для чтения (только для `REQUIRED` или `REQUIRES_NEW`).|
|`rollbackFor`|`Class<? extends Throwable>[]`|Исключения, которые **должны** вызывать откат.|
|`rollbackForClassName`|`String[]`|Паттерны имён исключений для отката.|
|`noRollbackFor`|`Class<? extends Throwable>[]`|Исключения, которые **не должны** вызывать откат.|
|`noRollbackForClassName`|`String[]`|Паттерны имён исключений, не вызывающих откат.|

## 1.4. Видимость методов и `@Transactional` в proxy-режиме
Аннотация `@Transactional` обычно используется на методах с **публичной** видимостью. 

Начиная с **Spring 6.0**, `protected` и package-visible методы также могут быть транзакционными для class-based прокси **по умолчанию**. 

Однако транзакционные методы в интерфейсных прокси **всегда должны быть `public`** и определены в проксируемом интерфейсе. 

Для обоих видов прокси **только внешние вызовы методов, проходящие через прокси, перехватываются**.

Если вам нужно единообразное поведение (как до Spring 5.3 — игнорировать не-public методы):
```java
@Bean
TransactionAttributeSource transactionAttributeSource() {
    return new AnnotationTransactionAttributeSource(true);
}
```

### **Ключевое ограничение proxy-режима:** 
==Self-invocation== — вызов метода внутри того же класса, который вызывает другой метод того же объекта — **не проходит через прокси** и, следовательно, транзакция не применяется, даже если вызываемый метод помечен `@Transactional`. 

Также прокси должен быть полностью инициализирован, поэтому не следует полагаться на эту функцию в коде инициализации (например, в `@PostConstruct`). Если self-invocation должен быть транзакционным — используйте **AspectJ mode**.


**AspectJ mode** вместо прокси изменяет байт-код целевого класса, обеспечивая поддержку `@Transactional` на любых методах. 
> Требует `spring-aspects.jar` в classpath и включения load-time weaving (или compile-time weaving).

---
## 1.5. Рекомендация: аннотировать классы, а не интерфейсы
==Spring рекомендует аннотировать методы== **конкретных классов** аннотацией `@Transactional`, а *не полагаться на аннотированные методы в интерфейсах.* 

Хотя последнее работает для интерфейсных и целевых прокси начиная с версии 5.0, аннотации Java не наследуются от интерфейсов. 

При использовании AspectJ-режима аннотации, объявленные на интерфейсе, **не распознаются** инфраструктурой weaving, и транзакционный аспект не применяется. 

Как следствие, ваши транзакционные аннотации могут быть **молча проигнорированы**: код может казаться работающим, пока вы не протестируете сценарий отката.

## 1.6. Таблица настроек `@EnableTransactionManagement` и `<tx:annotation-driven/>`
| XML-атрибут           | Атрибут аннотации  | По умолчанию                      | Описание                                                                                                                                    |
| --------------------- | ------------------ | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `transaction-manager` | N/A                | `transactionManager`              | Имя бина `TransactionManager`. <br><br>**Требуется, только если имя не** `transactionManager`                                               |
| `mode`                | `mode`             | `proxy`                           | `proxy` — AOP-прокси (перехватываются только внешние вызовы); `aspectj` — weaving байт-кода (любые вызовы, требуется `spring-aspects.jar`). |
| `proxy-target-class`  | `proxyTargetClass` | `false`                           | Только для `proxy` mode. `true` — class-based прокси; `false` — JDK interface-based прокси.                                                 |
| `order`               | `order`            | `Ordered`<br>`.LOWEST_PRECEDENCE` | Порядок выполнения транзакционного advice.                                                                                                  |

**Важно:** 
`@EnableTransactionManagement` и `<tx:annotation-driven/>` ищут `@Transactional` **только в том же application context**, где они определены. 

Если поместить их в `WebApplicationContext` для `DispatcherServlet`, они будут проверять только контроллеры, но не сервисы.

## 1.7. Приоритетность: наиболее специфичное побеждает
При вычислении транзакционных настроек для метода **наиболее производное (специфичное) местоположение имеет приоритет**. 


> [!NOTE] БОЛЕЕ СПЕЦИФИЧНАЯ НАСТРОЙКА ПОБЕЖДАЕТ ОБЩУЮ
> Если аннотация используется и на методе, и на классе, то аннотация на методе, перекроет конфигурацию метода свой конфигурацией

Если ***аннотация стоит и на классе, и на методе*** — **метод переопределяет** настройки класса:
```java
@Transactional(readOnly = true) // По умолчанию для всех методов
public class DefaultFooService implements FooService {

	// readOnly = true (от класса)
    public Foo getFoo(String fooName) { ... } 

    @Transactional(
	    readOnly = false, 
	    propagation = Propagation.REQUIRES_NEW
	)  // Переопределяет!
    public void updateFoo(Foo foo) { ... }
}
```

## 1.8. Глобальное изменение поведения отката по умолчанию (Spring 6.2+)
Можно переопределить поведение по умолчанию, чтобы **все** исключения (включая checked) вызывали откат:

Java-config:
```java
@EnableTransactionManagement(rollbackOn = RollbackOn.ALL_EXCEPTIONS)
```

Для более тонкой настройки:
```java
@Bean
TransactionAttributeSource transactionAttributeSource() {

    AnnotationTransactionAttributeSource source = 
	    new AnnotationTransactionAttributeSource();
	    
    source.addDefaultRollbackRule(
	    new RollbackRuleAttribute(Throwable.class)
	);
	
    return source;
}
```

Транзакционно-специфичные правила отката **переопределяют** глобальное поведение по умолчанию, но сохраняют его для неуказанных исключений. 

Это касается и `@Transactional` от Spring, и `jakarta.transaction.Transactional` от JTA.


**Рекомендация:** Если вы не полагаетесь на EJB-стиль checked-исключений с поведением коммита, рекомендуется переключиться на `ALL_EXCEPTIONS` для согласованной семантики отката. 
> Для Kotlin-приложений это особенно важно, так как в Kotlin checked-исключений нет вообще.

## 1.9. Имя транзакции
Для декларативных транзакций вы **НЕ МОЖЕТЕ ЯВНО КОНТРОЛИРОВАТЬ ИМЯ** транзакции (которое отображается в мониторе транзакций и логах). 

Имя всегда формируется автоматически как:  
`полное.имя.класса.имя_метода`

Например: `com.example.BusinessService.handlePayment`.

## 1.10. Несколько `TransactionManager` с `@Transactional`
Большинству приложений нужен только один менеджер транзакций, но бывают ситуации, когда требуется несколько независимых. 

Атрибут `value` (или `transactionManager`) в `@Transactional` указывает, какой именно менеджер использовать. Это может быть имя бина или значение квалификатора.

Java-config:
```java
@Bean
@Qualifier("order")
public PlatformTransactionManager transactionManager1(
	DataSource dataSource1
) {
    return new JdbcTransactionManager(dataSource1);
}

@Bean
@Qualifier("account")
public PlatformTransactionManager transactionManager2(
	DataSource dataSource2
) {
    return new JdbcTransactionManager(dataSource2);
}

@Bean
@Qualifier("reactive-account")
public ReactiveTransactionManager transactionManager3(
    ConnectionFactory connectionFactory) 
{
    return new R2dbcTransactionManager(connectionFactory);
}
```

Использование:
```java
public class TransactionalService {

    @Transactional("order")
    public void setSomething(String name) { ... }

    @Transactional("account")
    public void doSomething() { ... }

    @Transactional("reactive-account")
    public Mono<Void> doSomethingReactive() { ... }
}
```

Если все методы класса используют один квалификатор — можно объявить аннотацию `@Qualifier` на уровне типа:
```java
@org.springframework.beans.factory.annotation.Qualifier(
	"order"
)
@Transactional
public class OrderService { ... }
```

Такой квалификатор уровня класса также влияет на определения транзакций из базового класса, эффективно переопределяя выбор менеджера по умолчанию.

## 1.11. Пользовательские составные аннотации
Если вы часто используете одни и те же атрибуты с `@Transactional` на разных методах, Spring поддерживает создание **пользовательских составных аннотаций** через механизм мета-аннотаций:
```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Transactional(
	transactionManager = "order", 
	label = "causal-consistency"
)
public @interface OrderTx {
}

@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Transactional(
	transactionManager = "account", 
	label = "retryable"
)
public @interface AccountTx {
}
```

Использование становится гораздо лаконичнее:
```java
public class TransactionalService {

    @OrderTx
    public void setSomething(String name) { ... }

    @AccountTx
    public void doSomething() { ... }
}
```
