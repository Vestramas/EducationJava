Аннотация `@Transactional` управляет поведением транзакций на уровне Spring AOP.  

Главное, что она определяет:
1. **Как начинается и заканчивается транзакция**
    
2. **Какие исключения вызывают rollback**
    
3. **Можно ли выполнять новую транзакцию внутри текущей**
    
4. **Как работает изоляция**
    
5. **Можно ли выполнить запись только для чтения**
    
6. **Тайм-ауты**
    
7. **Имя и менеджер транзакций**

# 1. `propagation` — Поведение вложенных транзакций
Управляет тем, **как методы взаимодействуют с уже существующей транзакцией**.

## Основные режимы:
### ✔ `REQUIRED` (по умолчанию)
Если есть транзакция → использовать её.  
Если нет → создать новую.

**Это дефолтный и наиболее используемый режим.**

Пример:
```java
A() @Transactional REQUIRED
 └─ B() @Transactional REQUIRED
```
`B` выполняется в транзакции `A`.

### ✔ `REQUIRES_NEW`
Всегда создаёт **новую транзакцию**, приостанавливая текущую.

**Используют для логирования, аудита, отправки писем и т.д.**
```java
A() @Transactional
 └─ B() @Transactional(REQUIRES_NEW)
```
`A` и `B` независимы — если `A` откатится, изменения `B` сохранятся.

### ✔ `SUPPORTS`
Если транзакция существует → использовать.  
Если нет → выполнить без транзакции.

Редко используется.

---
### ✔ `MANDATORY`
Обязывает выполнять метод ТОЛЬКО внутри транзакции.

Если вызвать без активной транзакции → ошибка.

---
### ✔ `NEVER`
Запрещает транзакцию.  
Если есть активная → ошибка.

---
### ✔ `NOT_SUPPORTED`
Приостанавливает активную транзакцию и выполняет метод **без неё**.

---

### ✔ `NESTED`
Создаёт **savepoint** внутри существующей транзакции.  
Откат внутри вложенного метода не откатывает всю транзакцию.

Работает только с JDBC (`DataSourceTransactionManager`).

# 2. `isolation` — Уровень изоляции транзакции
|Уровень|Dirty Read|Non-repeatable|Phantom|
|---|---|---|---|
|`DEFAULT`|DB default|DB default|DB default|
|`READ_UNCOMMITTED`|✔|✔|✔|
|`READ_COMMITTED`|✖|✔|✔|
|`REPEATABLE_READ`|✖|✖|✔*|
|`SERIALIZABLE`|✖|✖|✖|

В PostgreSQL phantom read тоже запрещён благодаря MVCC + predicate locks.

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
```
соответствует
```java
connection
	.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
```

# 3. `readOnly` — режим "только чтение"
```java
@Transactional(readOnly = true)
```
### Что делает:
- Стратегии ORM (Hibernate) — избегают "грязной проверки" (dirty checking).
    
- Hibernate может опустить обновление статистики кэша.
    
- База может оптимизировать SELECT (в PostgreSQL почти не влияет).
    
- **Запрещает вставки/обновления/удаления** (Hibernate бросает исключение, но БД — нет).


**Когда использовать:**
- Query-сервисы
- Репозитории, которые только читают
- Методы, которые возвращают большие выборки

# 4. `rollbackFor` / `noRollbackFor` — управление rollback
По умолчанию ==Spring делает rollback только для unchecked exception==:
- `RuntimeException`
- `Error`
    

Но НЕ делает rollback для:
- `Exception`
- `SQLException`
- `IOException`


Чтобы настраивать:
✔ Откатить на checked exception:
```java
@Transactional(rollbackFor = SQLException.class)
```

✔ НЕ откатывать на RuntimeException:
```java
@Transactional(noRollbackFor = RuntimeException.class)
```

---
# 5. `timeout` — ограничение по времени
```java
@Transactional(timeout = 5) // 5 секунд
```
Если транзакция длится дольше — Spring откатывает её.

Используется в:
- медленных запросах
    
- долгих операциях записи
    
- batch-processing

---
# 6. `value` / `transactionManager` — выбор транзакционного менеджера
Если в приложении несколько БД/менеджеров:
```java
@Transactional(transactionManager = "kafkaTransactionManager")
```

или
```java
@Transactional("myTxManager")
```

---
# 7. `label` (в новых версиях Spring) — логическое имя транзакции
```java
@Transactional(label = "syncUserJob")
```

Может использоваться мониторами, логгерами, APM.
