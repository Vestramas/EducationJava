Spring предоставляет **два способа** программного управления транзакциями:

1. **`TransactionTemplate`** (для императивных потоков) или **`TransactionalOperator`** (для реактивного кода) — **рекомендуемый способ**.
    
2. **`TransactionManager` напрямую** — аналог использования JTA `UserTransaction` API, но с более простой обработкой исключений.

# 1. Использование `TransactionTemplate`
`TransactionTemplate` следует тому же подходу, что и другие Spring-шаблоны (`JdbcTemplate`): использует **callback-подход**, освобождая код приложения от шаблонного получения и освобождения транзакционных ресурсов. Код фокусируется только на бизнес-логике.

> **Важно:** Использование `TransactionTemplate` **абсолютно привязывает** ваш код к транзакционной инфраструктуре Spring. 
> 
> Является ли это подходящим для ваших нужд — решать вам.

**Базовое использование:**
```java
public class SimpleService implements Service {

    // Один экземпляр TransactionTemplate на все методы
    private final TransactionTemplate transactionTemplate;

    // Внедрение PlatformTransactionManager через конструктор
    public SimpleService(PlatformTransactionManager transactionManager) {
        this.transactionTemplate = 
	        new TransactionTemplate(transactionManager);
    }

    public Object someServiceMethod() {
        return transactionTemplate.execute(new TransactionCallback() {
            // Код внутри этого метода выполняется в транзакционном контексте
            public Object doInTransaction(TransactionStatus status) {
                updateOperation1();
                return resultOfUpdateOperation2();
            }
        });
    }
}
```

**Без возвращаемого значения:**
```java
transactionTemplate.execute(
	new TransactionCallbackWithoutResult() 
{
    protected void doInTransactionWithoutResult(TransactionStatus status) 
    {
        updateOperation1();
        updateOperation2();
    }
});
```


**Программный откат внутри callback:**
Код внутри callback может откатить транзакцию, вызвав `setRollbackOnly()` на объекте `TransactionStatus`:
```java
transactionTemplate.execute(
	new TransactionCallbackWithoutResult() 
{
    protected void doInTransactionWithoutResult(
	    TransactionStatus status
	) {
        try {
            updateOperation1();
            updateOperation2();
        } catch (SomeBusinessException ex) {
            status.setRollbackOnly();
        }
    }
});
```

## 1.2 Настройка параметров `TransactionTemplate`

Настройки транзакции (режим распространения, уровень изоляции, таймаут и т.д.) можно задать **программно** или в **конфигурации**. 

По умолчанию `TransactionTemplate` имеет стандартные транзакционные настройки.
```java
public class SimpleService implements Service {
    private final TransactionTemplate transactionTemplate;

    public SimpleService(
	    PlatformTransactionManager transactionManager
	) {
        this.transactionTemplate = 
	        new TransactionTemplate(transactionManager);

        // Настройки можно задать здесь явно
        this.transactionTemplate.setIsolationLevel(
	        TransactionDefinition.ISOLATION_READ_UNCOMMITTED
	    );
        this.transactionTemplate.setTimeout(30); // 30 секунд
    }
}
```

Затем `sharedTransactionTemplate` можно внедрить в любое количество сервисов.

**Потокобезопасность:** Экземпляры `TransactionTemplate` **потокобезопасны** — они не хранят состояние диалога. Однако они **хранят конфигурационное состояние**. Поэтому, если один класс использует `TransactionTemplate` с одними настройками, а другому нужны другие (например, другой уровень изоляции) — нужно создать **два разных экземпляра**.

# 2. Использование `TransactionalOperator`

`TransactionalOperator` следует дизайну, аналогичному другим ==реактивным операторам==. Он использует callback-подход и делает код намеренно-ориентированным.

**Важно:** Использование `TransactionalOperator` **абсолютно привязывает** ваш код к транзакционной инфраструктуре Spring.

**Базовое использование:**
```java
public class SimpleService implements Service {

    // Один экземпляр на все методы
    private final TransactionalOperator transactionalOperator;

    // Внедрение ReactiveTransactionManager через конструктор
    public SimpleService(
	    ReactiveTransactionManager transactionManager
	) {
        this.transactionalOperator = 
	        TransactionalOperator.create(transactionManager);
    }

    public Mono<Object> someServiceMethod() {
        Mono<Object> update = updateOperation1();

        return update.then(resultOfUpdateOperation2)
                .as(transactionalOperator::transactional);  // Применяем транзакцию
    }
}
```

**Два способа использования `TransactionalOperator`:**
- **Операторный стиль:** через Project Reactor (`mono.as(transactionalOperator::transactional)`)
    
- **Callback-стиль:** через `transactionalOperator.execute(TransactionCallback<T>)`
    

**Программный откат:**
```java
transactionalOperator.execute(new TransactionCallback<>() {
    public Mono<Object> doInTransaction(
	    ReactiveTransaction status
	) {
        return updateOperation1()
	        .then(updateOperation2)
            .doOnError(
	            SomeBusinessException.class, 
	            e -> status.setRollbackOnly()
	        );
    }
});
```

## 2.1. Cancel Signals (Сигналы отмены)
В Reactive Streams `Subscriber` может отменить свою подписку и остановить `Publisher`. Операторы в Project Reactor (`next()`, `take(long)`, `timeout(Duration)` и др.) могут инициировать отмену. 

Невозможно узнать причину отмены — ошибка или просто отсутствие интереса к дальнейшему потреблению. **Начиная с версии 5.3, сигналы отмены приводят к откату транзакции.** 

Поэтому важно учитывать, какие операторы используются downstream от транзакционного `Publisher`. 

В случае `Flux` или другого многозначного `Publisher` **весь вывод должен быть потреблён**, чтобы транзакция могла завершиться.

## 2.2 Настройка параметров `TransactionalOperator`
По умолчанию `TransactionalOperator` имеет стандартные транзакционные настройки. Их можно переопределить:
```java
public class SimpleService implements Service {
    private final TransactionalOperator transactionalOperator;

    public SimpleService(
	    ReactiveTransactionManager transactionManager
	) {
        DefaultTransactionDefinition definition = 
	        new DefaultTransactionDefinition();

        definition.setIsolationLevel(
	        TransactionDefinition.ISOLATION_READ_UNCOMMITTED);
        definition.setTimeout(30);

        this.transactionalOperator = 
        TransactionalOperator.create(
	        transactionManager, 
	        definition
	    );
    }
}
```

# 3. Использование `PlatformTransactionManager` напрямую

Для императивных транзакций можно использовать `PlatformTransactionManager` напрямую. 

Через объекты `TransactionDefinition` и `TransactionStatus` вы можете инициировать транзакции, выполнять откат и коммит:
```java
DefaultTransactionDefinition def = 
	new DefaultTransactionDefinition();
	
// Установить имя транзакции можно только программно!
def.setName("SomeTxName");
def.setPropagationBehavior(
	TransactionDefinition.PROPAGATION_REQUIRED
);

TransactionStatus status = txManager.getTransaction(def);
try {
    // бизнес-логика
} catch (MyException ex) {
    txManager.rollback(status);
    throw ex;
}
txManager.commit(status);
```

**Примечание:** Явная установка имени транзакции (`setName`) — это то, что можно сделать **только программно**.

# 4. Использование `ReactiveTransactionManager` напрямую

Для реактивных транзакций используется `ReactiveTransactionManager` напрямую через `TransactionDefinition` и `ReactiveTransaction`:
```java
DefaultTransactionDefinition def = 
	new DefaultTransactionDefinition();
	
def.setName("SomeTxName");
def.setPropagationBehavior(
	TransactionDefinition.PROPAGATION_REQUIRED
);

Mono<ReactiveTransaction> reactiveTx = 
	txManager.getReactiveTransaction(def);


reactiveTx.flatMap(status -> {
    Mono<Object> tx = ...; // бизнес-логика

    return tx.then(txManager.commit(status))
            .onErrorResume(ex -> 
	            txManager
		            .rollback(status)
		            .then(Mono.error(ex))
			);
});
```
