Spring Framework поддерживает интеграцию с **Java Persistence API (JPA)** и **нативным Hibernate** для управления ресурсами, реализации DAO и стратегий транзакций.

**Ключевой принцип:** Рекомендуемый стиль интеграции — писать DAO, используя чистые API Hibernate или JPA, без зависимости от классов Spring.

### Преимущества использования Spring для создания ORM DAO

**1. Более лёгкое тестирование**

IoC-подход Spring позволяет легко подменять реализации и конфигурации `SessionFactory`, `DataSource`, `TransactionManager` и маппингов объектов. Это делает изолированное тестирование каждого компонента персистентности значительно проще.


**2. Единые исключения доступа к данным**

Spring оборачивает исключения ORM-инструментов, преобразуя их из проприетарных (потенциально checked) в общую иерархию **непроверяемых** `DataAccessException`. Это позволяет обрабатывать большинство невосстановимых исключений персистентности только в нужных слоях, без шаблонных `catch`, `throws` и объявлений исключений. JDBC-исключения (включая диалекто-специфичные) также преобразуются в ту же иерархию — можно выполнять операции с JDBC в рамках единой модели программирования.


**3. Общее управление ресурсами**

Spring-контекст управляет расположением и конфигурацией `SessionFactory`, `EntityManagerFactory`, `DataSource` и других ресурсов. Это делает их лёгкими в управлении и изменении. Например, код, использующий Hibernate, обычно должен работать с одним и тем же `Session` для эффективности и корректной обработки транзакций. Spring делает создание и привязку `Session` к текущему потоку прозрачным, решая многие хронические проблемы типичного использования Hibernate.


**4. Интегрированное управление транзакциями**

ORM-код можно обернуть декларативным AOP-перехватчиком через `@Transactional` или XML-конфигурацию. Семантика транзакций и обработка исключений (откат и т.д.) выполняются автоматически. Можно менять менеджеры транзакций (локальные ↔ JTA) без изменения ORM-кода. JDBC-код может полностью интегрироваться с ORM-кодом в рамках одних транзакций — полезно для операций, не подходящих для ORM (пакетная обработка, BLOB-стриминг), но требующих общих транзакций.

---
> **Примечание:** 
> Для более полной поддержки ORM, включая альтернативные технологии (MongoDB и др.), стоит обратить внимание на проекты **Spring Data**.

# 2. Перевод исключений (Exception Translation)
**Проблема:** 
Hibernate бросает `HibernateException`, JPA — `PersistenceException`, JDBC — `SQLException`. Если бизнес-логика начнёт ловить эти исключения, она намертво привяжется к конкретной технологии.


**Решение Spring: Аннотация `@Repository`.**
Вы вешаете `@Repository` на свой DAO-класс. Spring «под капотом» делает следующее:
1. Смотрит на ваш класс.
    
2. Видит, что это `@Repository`.
    
3. Создаёт **прокси** вокруг этого класса.
    
4. Когда метод DAO бросает исключение (например, `HibernateException`), **прокси перехватывает его**.
    
5. Смотрит на тип исключения и преобразует в одно из **универсальных исключений Spring** из иерархии `DataAccessException`.

| Исходное исключение                                    | Во что превратит Spring                                     |
| ------------------------------------------------------ | ----------------------------------------------------------- |
| `org.hibernate.exception.ConstraintViolationException` | `org.springframework.dao.DataIntegrityViolationException`   |
| `javax.persistence.OptimisticLockException`            | `org.springframework.dao.OptimisticLockingFailureException` |
| `java.sql.SQLException` с кодом ошибки Oracle          | Соответствующий подкласс `DataAccessException`              |
|                                                        |                                                             |

**Зачем это нужно:**
- Бизнес-логика может ловить **один тип исключений** (`DataAccessException`), не зная, какая БД или ORM-библиотека используется.
    
- Вы можете в любой момент **сменить ORM** (например, перейти с Hibernate на EclipseLink) — бизнес-код этого даже не заметит.
    
- Все исключения становятся **unchecked** — не нужно писать `throws` и `try-catch` в каждом методе.
