Spring JPA, доступный в пакете `org.springframework.orm.jpa`, **предлагает всестороннюю поддержку Java Persistence API** способом, аналогичным интеграции с Hibernate, с учетом базовой реализации для предоставления дополнительных функций.

> [!NOTE] Persistence 
> **Persistence (персистентность)** — это способность данных существовать дольше, чем процесс, который их создал. В Java-разработке это означает сохранение объектов в базу данных, чтобы они пережили перезапуск приложения.


> [!NOTE] Persistence Provider
> **Persistence Provider (провайдер персистентности)** — это конкретная **реализация спецификации JPA**. 
> 
> Спецификация JPA — это только набор интерфейсов и правил (например, `EntityManager`, `EntityManagerFactory`, JPQL). 
> 
> Провайдер — это библиотека, которая реализует всю эту логику: управляет сущностями, генерирует SQL, кэширует данные.
> 
> **Самые известные провайдеры:**
> - **Hibernate** — самый популярный, используется по умолчанию в Spring Boot.
> - **EclipseLink** — эталонная реализация JPA (Reference Implementation).
> - **Apache OpenJPA** — ещё одна реализация.


> [!NOTE] Persistence Unit - конфигурация, описывающая КАК приложение работает с базой данных. 
> **Persistence Unit** — это **контейнер конфигурации**, который объединяет:
> - Настройки подключения к БД (URL, логин, пароль, драйвер).
> - Список классов-сущностей (тех самых, что с `@Entity`, `@Table`).
> - Параметры провайдера (Hibernate, EclipseLink) — диалект SQL, стратегия генерации таблиц, кэш.
> 
> Это ==ЧАСТЬ СПЕЦИФИКАЦИИ JPA==. Спецификация JPA (JSR 338) определяет, что конфигурация Persistence Unit должна находиться в файле `META-INF/persistence.xml`.
> 
> Если вы пишете приложение **строго по спецификации JPA**, вы **обязаны** предоставить этот файл. Без него JPA-провайдер не будет знать, что делать.
>

# 1. Три способа настройки JPA в Spring
Spring JPA (пакет `org.springframework.orm.jpa`) предлагает **три способа настройки** `EntityManagerFactory`:

## 1.1. `LocalEntityManagerFactoryBean` — простейший способ
**Только для простых сред:** автономные приложения и интеграционные тесты.

Использует механизм автоопределения JPA `PersistenceProvider` (согласно Java SE bootstrapping JPA). 
- В большинстве случаев ==требует только указать ИМЯ PERSISTENCE UNIT. ==
- **Нельзя** ссылаться на существующий JDBC `DataSource`, **нет поддержки глобальных транзакций**. 
- Weaving (byte-code transformation) персистентных классов зависит от провайдера и часто требует указания JVM-агента при запуске. 

Этот вариант **достаточен только для автономных приложений и тестовых сред**.

```xml
<bean id="myEmf" class="org.springframework.orm.jpa.LocalEntityManagerFactoryBean">
    <property name="persistenceUnitName" value="myPersistenceUnit"/>
</bean>
```

```java
@Bean
public LocalEntityManagerFactoryBean myEmf() {
    LocalEntityManagerFactoryBean factory = 
	    new LocalEntityManagerFactoryBean();
	    
    factory.setPersistenceUnitName("myPersistenceUnit");
    
    return factory;
}
```

## 1.2. Получение `EntityManagerFactory` из JNDI

> [!NOTE] JNDI (Java Naming and Directory Interface)
> JNDI — это **централизованный каталог ресурсов уровня сервера приложений**. Приложение не знает и не должно знать, где физически находится БД, какой у неё пароль, какой драйвер используется. 
> 
> Оно просто просит: «Дай мне `DataSource` с именем `jdbc/MyAppDB`». Это **позднее связывание** (late binding) ресурсов.
> .
> - **С properties/env:** Вам нужно обновить конфигурацию во **всех 50 приложениях** и переразвернуть их.
> - **С JNDI:** Администратор сервера **один раз** меняет настройку в JNDI-каталоге сервера. Все 50 приложений автоматически получают новое подключение **без перекомпиляции и переразвёртывания**.
> 
> JNDI нужен в **корпоративных средах**, где приложения разворачиваются на **общих серверах приложений** (WildFly, WebSphere, GlassFish) и администрируются централизованно.


**Для Jakarta EE-серверов.** Вся конфигурация persistence unit (включая weaving) выполняется на стороне сервера.

Сервер автоопределяет persistence unit'ы (из `META-INF/persistence.xml`) и записи `persistence-unit-ref` в дескрипторе развёртывания. JDBC `DataSource` определяется через JNDI в `persistence.xml`. 

Транзакции `EntityManager` интегрируются с JTA-подсистемой сервера. Spring лишь получает готовый `EntityManagerFactory` и внедряет его.

```xml
<jee:jndi-lookup id="myEmf" jndi-name="persistence/myPersistenceUnit"/>
```

```java
@Bean
public JndiObjectFactoryBean myEmf() {
    JndiObjectFactoryBean jndiFactory = new JndiObjectFactoryBean();
    
    jndiFactory.setJndiName("persistence/myPersistenceUnit");
    
    return jndiFactory;
}
```

> **Важно для нескольких persistence unit:** 
> Имена бинов JNDI-полученных persistence unit должны совпадать с именами, которые приложение использует для обращения к ним (`@PersistenceUnit`, `@PersistenceContext`).

---
## 1.3. `LocalContainerEntityManagerFactoryBean` — самый мощный способ

**Для полных JPA-возможностей в Spring-окружении:** Tomcat, автономные приложения, интеграционные тесты со сложными требованиями к персистентности.

==Даёт полный контроль над конфигурацией== `EntityManagerFactory`. 

Создаёт `PersistenceUnitInfo` на основе `persistence.xml`, стратегии `dataSourceLookup` и указанного `loadTimeWeaver`. 

Позволяет работать с пользовательскими `DataSource` вне JNDI и контролировать процесс weaving.

```xml
<bean id="myEmf" class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean">
    <property name="dataSource" ref="someDataSource"/>
    <property name="loadTimeWeaver">
        <bean class="org.springframework.instrument.classloading.InstrumentationLoadTimeWeaver"/>
    </property>
</bean>
```

```java
@Bean
public LocalContainerEntityManagerFactoryBean myEmf(DataSource someDataSource) {
    LocalContainerEntityManagerFactoryBean factory = 
	    new LocalContainerEntityManagerFactoryBean();
	    
    factory.setDataSource(someDataSource);
    factory.setLoadTimeWeaver(new InstrumentationLoadTimeWeaver());
    
    return factory;
}
```


**Типичный `persistence.xml`:**
```xml
<persistence xmlns="http://java.sun.com/xml/ns/persistence" version="1.0">
    <persistence-unit name="myUnit" transaction-type="RESOURCE_LOCAL">
        <mapping-file>META-INF/orm.xml</mapping-file>
        <exclude-unlisted-classes/>
    </persistence-unit>
</persistence>
```

`<exclude-unlisted-classes/>` указывает, что сканирование аннотированных классов сущностей не требуется. Чтобы сканирование происходило — либо удалите элемент, либо укажите `false`.

**Преимущества:** Связь с существующим JDBC `DataSource`, поддержка локальных и глобальных транзакций. **Требования:** Weaving-capable class loader, если провайдер требует byte-code transformation.

**Конфликт с Jakarta EE:** В полной Jakarta EE-среде лучше получать `EntityManagerFactory` из JNDI. Либо укажите `persistenceXmlLocation` = `META-INF/my-persistence.xml` — сервер ищет только `META-INF/persistence.xml` и проигнорирует ваш кастомный файл, избегая конфликта.

### 1.3.1 Load-Time Weaving (LTW)

`LoadTimeWeaver` — интерфейс Spring, позволяющий подключать JPA `ClassTransformer`-ы специфичным для среды образом. 

**Использование JVM-агента неэффективно** (агент работает против всей VM и проверяет каждый загружаемый класс).

Spring предоставляет реализации `LoadTimeWeaver` для разных сред. Рекомендуемый способ — глобальный weaver через `@EnableLoadTimeWeaving` или `<context:load-time-weaver/>`:
```xml
<context:load-time-weaver/>

<bean id="emf" class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean">
    ...
</bean>
```

```java
@Configuration
@EnableLoadTimeWeaving // ДОБАВИЛИ АННОТАЦИЮ
public class AppConfig {

    @Bean
    public LocalContainerEntityManagerFactoryBean emf() {
        LocalContainerEntityManagerFactoryBean factory = 
	        new LocalContainerEntityManagerFactoryBean();
	        
        // loadTimeWeaver подхватится автоматически
        
        return factory;
    }
}
```

Такой weaver автоматически подхватывается всеми экземплярами `LocalContainerEntityManagerFactoryBean`.

**Преимущество LTW:** JPA-приложения, полагающиеся на инструментирование, могут работать без агента. JPA-трансформеры применяются только на уровне class loader и изолированы друг от друга.

### 1.3.2 Работа с несколькими Persistence Units

`PersistenceUnitManager` **выступает центральным репозиторием для нескольких persistence unit**, позволяя избежать дорогостоящего процесса обнаружения. 

Позволяет указать несколько расположений `persistence.xml` и привязать разные `DataSource` по ключам:
```xml
<bean id="pum" class="org.springframework.orm.jpa.persistenceunit.DefaultPersistenceUnitManager">
    <property name="persistenceXmlLocations">
        <list>
            <value>org/springframework/orm/jpa/domain/persistence-multi.xml</value>
            <value>classpath:/my/package/**/custom-persistence.xml</value>
            <value>classpath*:META-INF/persistence.xml</value>
        </list>
    </property>
    <property name="dataSources">
        <map>
            <entry key="localDataSource" value-ref="local-db"/>
            <entry key="remoteDataSource" value-ref="remote-db"/>
        </map>
    </property>
    <property name="defaultDataSource" ref="remoteDataSource"/>
</bean>

<bean id="emf" class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean">
    <property name="persistenceUnitManager" ref="pum"/>
    <property name="persistenceUnitName" value="myCustomUnit"/>
</bean>
```

```java
@Bean
public DefaultPersistenceUnitManager pum(
	DataSource localDb, 
	DataSource remoteDb
) {
    DefaultPersistenceUnitManager manager = 
	    new DefaultPersistenceUnitManager();
	    
    manager.setPersistenceXmlLocations(List.of(
        "org/springframework/orm/jpa/domain/persistence-multi.xml",
        "classpath:/my/package/**/custom-persistence.xml",
        "classpath*:META-INF/persistence.xml"
    ));
    
    Map<String, DataSource> dataSources = new HashMap<>();
    dataSources.put("localDataSource", localDb);
    dataSources.put("remoteDataSource", remoteDb);
    
    manager.setDataSources(dataSources);
    manager.setDefaultDataSource(remoteDb);
    
    return manager;
}

@Bean
public LocalContainerEntityManagerFactoryBean emf(
	PersistenceUnitManager pum
) {
    LocalContainerEntityManagerFactoryBean factory = 
	    new LocalContainerEntityManagerFactoryBean();
	    
    factory.setPersistenceUnitManager(pum);
    factory.setPersistenceUnitName("myCustomUnit");
    
    return factory;
}
```

### 1.3.3 Фоновый Bootstrapping

`LocalContainerEntityManagerFactoryBean` **поддерживает фоновую инициализацию через** `bootstrapExecutor`.

*Инициализация JPA-провайдера передаётся указанному executor'у* и выполняется параллельно потоку загрузки приложения. 

Прокси `EntityManagerFactory` доступен для внедрения сразу, но вызовы `createEntityManager()` блокируются до завершения фоновой инициализации.

> **Начиная с Spring 6.2:** 
> ***Инициализация JPA принудительно завершается до окончания обновления контекста***, что делает доступность инфраструктуры БД предсказуемой. 
> 
> **Не рекомендуется** помещать инициализацию БД в `@PostConstruct` — лучше использовать `Lifecycle.start` или `ContextRefreshedEvent`.

```xml
<bean id="emf" class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean">
    <property name="bootstrapExecutor">
        <bean class="org.springframework.core.task.SimpleAsyncTaskExecutor"/>
    </property>
</bean>
```

```java
@Bean
public LocalContainerEntityManagerFactoryBean emf() {

    LocalContainerEntityManagerFactoryBean factory = 
	    new LocalContainerEntityManagerFactoryBean();
	    
    factory.setBootstrapExecutor(new SimpleAsyncTaskExecutor());
    
    return factory;
}
```

# 2. DAO на основе JPA: `EntityManagerFactory` и `EntityManager`

- `EntityManagerFactory` — потокобезопасен, 
	
- `EntityManager` — нет. 

Внедрённый Spring `EntityManager` **делегирует вызовы** текущему транзакционному `EntityManager`, если он есть. 
Иначе создаёт новый `EntityManager` на каждую операцию, что делает его использование потокобезопасным.

**Чистый JPA без зависимостей от Spring:**
```java
public class ProductDaoImpl implements ProductDao {
    private EntityManagerFactory emf;

    @PersistenceUnit
    public void setEntityManagerFactory(EntityManagerFactory emf) {
        this.emf = emf;
    }

    public Collection loadProductsByCategory(String category) {
        EntityManager em = this.emf.createEntityManager();
        try {
            Query query = em.createQuery(
	            "from Product as p where p.category = ?1"
	        );
            query.setParameter(1, category);
            
            return query.getResultList();
        } finally {
            if (em != null) { em.close(); }
        }
    }
}
```

**Недостаток:** Всегда создаётся новый `EntityManager`. Этого можно избежать, запросив **транзакционный (shared) EntityManager**:
```java
public class ProductDaoImpl implements ProductDao {
    @PersistenceContext
    private EntityManager em;

    public Collection loadProductsByCategory(String category) {
        Query query = em.createQuery(
	        "from Product as p where p.category = :category"
	    );
        query.setParameter("category", category);
        
        return query.getResultList();
    }
}
```

`@PersistenceContext` с типом по умолчанию `PersistenceContextType.TRANSACTION` даёт **shared EntityManager proxy** (потокобезопасный). Альтернатива `EXTENDED` — extended EntityManager (**не потокобезопасен**, только для stateful-компонентов, жизненный цикл не привязан к транзакции).

**Главное преимущество:** Зависимость только от JPA API, без импорта классов Spring.

## 2.1. Внедрение через `@Autowired` и конструктор
`EntityManagerFactory` можно внедрить через `@Autowired` как обычный бин. Для shared `EntityManager` нужен дополнительный бин:
```xml
<bean id="emf" class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean">...</bean>

<bean id="em" class="org.springframework.orm.jpa.support.SharedEntityManagerBean">
    <property name="entityManagerFactory" ref="emf"/>
</bean>
```

Или через Java-config:
```java
@Bean("em")
public static EntityManager sharedEntityManager(EntityManagerFactory emf) {
    return SharedEntityManagerCreator.createSharedEntityManager(emf);
}
```

# 3. JPA-транзакции в Spring

Рекомендуемая стратегия — **локальные транзакции через `JpaTransactionManager`**. Он предоставляет многие возможности локальных JDBC-транзакций (уровни изоляции, resource-level read-only оптимизации) без необходимости JTA-координатора.

`JpaTransactionManager` также может предоставлять JPA-транзакцию JDBC-коду, если зарегистрированный `JpaDialect` поддерживает получение нижележащего JDBC `Connection`. Диалекты есть для EclipseLink и Hibernate.

---
# 4. `JpaDialect` и `JpaVendorAdapter`

**`JpaDialect`** (расширенная возможность) позволяет реализовать vendor-специфичные фичи:

- Специфическая семантика транзакций (кастомный уровень изоляции, таймаут).
    
- Получение транзакционного JDBC `Connection`.
    
- Продвинутый перевод `PersistenceException` в `DataAccessException`.
    

Реализация по умолчанию (`DefaultJpaDialect`) не даёт специальных возможностей.

**`JpaVendorAdapter`** — более широкая адаптация провайдера, объединяет `JpaDialect` с другими провайдер-специфичными настройками. 

`HibernateJpaVendorAdapter` или `EclipseLinkJpaVendorAdapter` — **самый удобный способ авто-конфигурации** `EntityManagerFactory` для Hibernate/EclipseLink.

---
# 5. JPA с JTA-транзакциями

Альтернатива `JpaTransactionManager` для multi-resource координации. 

Требуется:
1. JDBC connection pools должны быть XA-совместимыми и интегрированы с координатором транзакций.
    
2. `EntityManagerFactory` должен быть настроен для JTA (провайдер-специфично, через `jpaProperties`).
    
3. Для Hibernate 5.1+: установить `hibernate.connection.handling_mode` = `DELAYED_ACQUISITION_AND_RELEASE_AFTER_STATEMENT`.
    
4. Альтернатива — получить `EntityManagerFactory` из сервера через JNDI.
    

---
# 6. Нативный Hibernate и взаимодействие с JPA

Нативный `LocalSessionFactoryBean` + `HibernateTransactionManager` позволяет взаимодействовать с `@PersistenceContext` и другим JPA-кодом. 

Hibernate `SessionFactory` **нативно реализует** `EntityManagerFactory`, а `Session` — `EntityManager`.

**Преимущества:** Более сильная интеграция с Hibernate, больше гибкости конфигурации (не ограничена JPA bootstrap-контрактами), кастомные Hibernate Integrator, более сильные оптимизации для read-only транзакций.

`LocalSessionFactoryBean` и `LocalSessionFactoryBuilder` также поддерживают фоновый bootstrapping.
