# 1. Конфигурация
Spring Data JPA можно настроить двумя способами: 
- **аннотации (JavaConfig)**
- **XML (Spring Namespace)**.

> **JavaConfig или XML?** 
> XML был способом конфигурации Spring много лет назад. 
> 
> В современную эпоху быстро развивающейся Java, record-типов, аннотаций и многого другого, новые проекты обычно используют как можно больше чистого Java. 
> 
> Хотя немедленных планов по удалению поддержки XML нет, некоторые новейшие функции **могут быть недоступны через XML**.

---
## 1.1. Конфигурация через аннотации (JavaConfig)

> [!IMPORTANT] `@EnableJpaRepositories` и `@EnableTransactionManagement`
> Активация поддержки репозиториев Spring Data JPA происходит через аннотации `@EnableJpaRepositories` и `@EnableTransactionManagement`:

```java
@Configuration
@EnableJpaRepositories       // ВНИМАНИЕ
@EnableTransactionManagement // ВНИМАНИЕ
class ApplicationConfig {

  @Bean
  public DataSource dataSource() {
    EmbeddedDatabaseBuilder builder = new EmbeddedDatabaseBuilder();
    return builder.setType(EmbeddedDatabaseType.HSQL).build();
  }

  @Bean
  public LocalContainerEntityManagerFactoryBean entityManagerFactory() {
    HibernateJpaVendorAdapter vendorAdapter = 
	    new HibernateJpaVendorAdapter();
	    
    vendorAdapter.setGenerateDdl(true);


    LocalContainerEntityManagerFactoryBean factory = 
	    new LocalContainerEntityManagerFactoryBean();
	    
    factory.setJpaVendorAdapter(vendorAdapter);
    factory.setPackagesToScan("com.acme.domain");
    factory.setDataSource(dataSource());
    
    return factory;
  }

  @Bean
  public PlatformTransactionManager transactionManager(
	  EntityManagerFactory entityManagerFactory
  ) {
    JpaTransactionManager txManager = new JpaTransactionManager();
    
    txManager.setEntityManagerFactory(entityManagerFactory);
    
    return txManager;
  }
}
```

**Важные моменты:**
- **Создавать нужно `LocalContainerEntityManagerFactoryBean`**, а не `EntityManagerFactory` напрямую. Первый, помимо создания `EntityManagerFactory`, также участвует в механизмах трансляции исключений.
    
- Конфигурация настраивает встроенную HSQL БД через `EmbeddedDatabaseBuilder` из `spring-jdbc`, затем `EntityManagerFactory` с Hibernate в качестве провайдера, и `JpaTransactionManager` в качестве компонента инфраструктуры транзакций.
    
- `@EnableJpaRepositories` активирует репозитории Spring Data JPA и несёт те же атрибуты, что и XML namespace. Если базовый пакет не настроен, используется пакет, в котором находится конфигурационный класс.

---
## 1.2. Конфигурация через XML (Spring Namespace)
JPA-модуль Spring Data содержит кастомный namespace, позволяющий определять бины репозиториев:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xmlns:jpa="http://www.springframework.org/schema/data/jpa"
  xsi:schemaLocation="http://www.springframework.org/schema/beans
    https://www.springframework.org/schema/beans/spring-beans.xsd
    http://www.springframework.org/schema/data/jpa
    https://www.springframework.org/schema/data/jpa/spring-jpa.xsd">

  <jpa:repositories base-package="com.acme.repositories" />
</beans>
```

Использование элемента `repositories` активирует **трансляцию исключений персистентности** для всех бинов, аннотированных `@Repository`, чтобы исключения, выбрасываемые JPA-провайдерами, преобразовывались в иерархию `DataAccessException` от Spring.

---
## 1.3. Дополнительные атрибуты namespace
Помимо стандартных атрибутов элемента `repositories`, JPA namespace предлагает дополнительные атрибуты для более детального контроля:

|Атрибут|Описание|
|---|---|
|`entity-manager-factory-ref`|Явно указывает `EntityManagerFactory` для использования с репозиториями. Обычно нужен, если в приложении используется несколько бинов `EntityManagerFactory`. Если не настроен, Spring Data автоматически ищет бин с именем `entityManagerFactory`.|
|`transaction-manager-ref`|Явно указывает `PlatformTransactionManager` для использования с репозиториями. Обычно нужен, только если настроено несколько менеджеров транзакций или бинов `EntityManagerFactory`. По умолчанию используется единственный определённый `PlatformTransactionManager` в текущем `ApplicationContext`.|

> Spring Data JPA требует наличия бина `PlatformTransactionManager` с именем `transactionManager`, если явный `transaction-manager-ref` не определён.

# 2. Bootstrap Mode (Режим инициализации)

По умолчанию репозитории Spring Data JPA — это стандартные Spring-бины: **singleton** и с **ранней (eager) инициализацией**. 

При запуске они уже взаимодействуют с JPA `EntityManager` для проверки и анализа метаданных.

Spring Framework поддерживает инициализацию JPA `EntityManagerFactory` в фоновом потоке, так как этот процесс обычно занимает значительную часть времени запуска. Чтобы эффективно использовать этот фоновый процесс, нужно убедиться, что репозитории JPA инициализируются как можно позже.

Начиная с **Spring Data JPA 2.1**, можно настроить `BootstrapMode` (через `@EnableJpaRepositories` или XML namespace) со следующими значениями:

|Режим|Описание|
|---|---|
|**`DEFAULT`** (по умолчанию)|Репозитории инициализируются **жадно (eagerly)**, если явно не аннотированы `@Lazy`. Ленивая инициализация срабатывает, только если ни один клиентский бин не нуждается в экземпляре репозитория (что потребовало бы его инициализации).|
|**`LAZY`**|Неявно объявляет все бины репозиториев ленивыми и создаёт прокси для ленивой инициализации при внедрении. Репозитории не будут созданы, если клиентский бин просто хранит экземпляр в поле и не использует его во время инициализации. Инициализация и проверка происходят при первом взаимодействии с репозиторием.|
|**`DEFERRED`**|В основном тот же режим работы, что и `LAZY`, но инициализация репозиториев запускается в ответ на `ContextRefreshedEvent`. Это гарантирует, что репозитории будут проверены до того, как приложение полностью запустится.|

### Рекомендации по выбору режима
- **Если вы НЕ используете асинхронный JPA bootstrap** — используйте режим по умолчанию (`DEFAULT`).
    
- **Если вы используете асинхронный JPA bootstrap** — **`DEFERRED`** является разумным выбором по умолчанию. Он гарантирует, что Spring Data JPA bootstrap будет ждать завершения настройки `EntityManagerFactory`, только если та занимает больше времени, чем инициализация всех остальных компонентов приложения. При этом репозитории всё равно корректно инициализируются и проверяются до того, как приложение сигнализирует о своей готовности.
    
- **`LAZY`** — хороший выбор для сценариев тестирования и локальной разработки. Когда вы уверены, что репозитории могут корректно инициализироваться, или когда вы тестируете другие части приложения, проверка всех репозиториев может излишне увеличивать время запуска. То же самое относится к локальной разработке, где вы можете обращаться только к частям приложения, требующим инициализации одного репозитория.

# 3. Сохранение сущностей
## 3.1. Сохранение сущностей

Сохранение сущности выполняется методом `CrudRepository.save(…)`.

Он 
- либо сохраняет, 
- либо объединяет (merge) сущность, используя нижележащий JPA `EntityManager`. 


**Логика работы:**
- Если сущность **ещё не была сохранена** → вызывается `entityManager.persist(…)`.
    
- Если сущность **уже существует** в БД → вызывается `entityManager.merge(…)`.


> **Как Spring Data JPA понимает, новая сущность или нет?** Для этого существуют стратегии определения состояния сущности.

---
### 3.1.1 Стратегии определения состояния сущности (Entity State-detection Strategies)

Spring Data JPA предлагает несколько стратегий для определения, является ли сущность новой:

---
#### Стратегия 1: Проверка Version-свойства и Id-свойства (по умолчанию)

Это стратегия по умолчанию. Spring Data JPA проверяет:
1. **Сначала** — есть ли у сущности **Version-свойство непримитивного типа**. 
   Если есть, сущность считается **новой**, если ***значение этого свойства равно*** `null`.
    
2. **Если Version-свойства нет** — проверяется **свойство идентификатора** (`Id`). Если `Id == null` → сущность новая. Иначе — существующая.
    

> **Важный нюанс с примитивными типами:** 
> В отличие от других модулей Spring Data, JPA считает `0` (ноль) первой вставленной версией сущности. 
> 
> Поэтому **примитивное version-свойство не может использоваться** для определения, является ли сущность новой.

---
#### Стратегия 2: Реализация интерфейса `Persistable`

Если сущность реализует интерфейс `Persistable`, Spring Data JPA **делегирует** определение новизны методу `isNew(…)` самой сущности. Это даёт полный контроль над логикой.

---
#### Стратегия 3: Кастомизация `EntityInformation`

Можно настроить абстракцию `EntityInformation`, используемую в `SimpleJpaRepository`, создав подкласс `JpaRepositoryFactory` и переопределив метод `getEntityInformation(…)`. 

Затем нужно зарегистрировать кастомную реализацию `JpaRepositoryFactory` как Spring-бин. **Это редко требуется.**

---
## 3.3. Проблема с ручным присвоением идентификаторов

**Стратегия 1 НЕ РАБОТАЕТ** ***ДЛЯ сущностей***, у которых:
- Идентификатор **присваивается вручную**.
    
- **Нет version-атрибута**.


В этом случае идентификатор всегда будет не-null, и Spring Data JPA всегда будет вызывать `merge()` вместо `persist()`.

**Решение — общий базовый класс с флагом `isNew`:**
```java
@MappedSuperclass
public abstract class AbstractEntity<ID> implements Persistable<ID> {

  @Transient  // Не сохраняется в БД
  private boolean isNew = true;

  @Override
  public boolean isNew() {
    return isNew;
  }

  @PostPersist
  @PostLoad
  void markNotNew() {
    this.isNew = false;
  }
}
```

**Как это работает:**
1. **Поле `isNew`** — флаг, объявленный как `@Transient` (не сохраняется в БД), по умолчанию `true`.
    
2. **Метод `isNew()`** — возвращает флаг. Spring Data JPA использует его, чтобы понять: вызвать `persist()` или `merge()`.
    
3. **Метод `markNotNew()`** — помечен аннотациями `@PostPersist` (после сохранения) и `@PostLoad` (после загрузки из БД). После вызова `save(…)` или загрузки провайдером флаг переключается в `false`, показывая, что сущность теперь существующая.
