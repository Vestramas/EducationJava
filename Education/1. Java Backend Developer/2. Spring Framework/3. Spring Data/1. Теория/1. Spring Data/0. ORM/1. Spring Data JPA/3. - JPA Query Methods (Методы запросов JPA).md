Этот раздел описывает различные способы создания запросов с помощью Spring Data JPA.

# 1. Стратегии поиска запросов (Query Lookup Strategies)

Модуль JPA поддерживает определение запроса **вручную как строки** или **выведение из имени метода**.

> **Важно о производных запросах:** 
> Для предикатов `IsStartingWith`, `StartingWith`, `StartsWith`, `IsEndingWith`, `EndingWith`, `EndsWith`, `IsNotContaining`, `NotContaining`, `NotContains`, `IsContaining`, `Containing`, `Contains` соответствующие аргументы **очищаются (sanitized)**. 
> 
> Это значит, что **если аргументы содержат символы**, ***распознаваемые `LIKE` как подстановочные знаки*** (wildcards) (`%` и `_`), **они будут экранированы, чтобы соответствовать только как литералы.** 
> 
> Экранирующий символ можно настроить через `escapeCharacter` в `@EnableJpaRepositories`.


## 1.1 **Объявленные запросы:** 
Хотя получение запроса из имени метода удобно, можно столкнуться с ситуацией, когда парсер имён методов не поддерживает нужное ключевое слово или имя метода становится неоправданно громоздким. 
В таких случаях можно использовать 
- либо именованные запросы JPA (через соглашение об именовании), 
- либо аннотировать метод запроса с помощью `@Query`.

---
# 2. Создание запросов (Query Creation)

Механизм создания запросов для JPA **работает так же, как описано в общих методах запросов** (Query Methods).

Spring Data JPA преобразует имена методов в JPQL:

```java
public interface UserRepository extends Repository<User, Long> {
	List<User> findByEmailAddressAndLastname(
		String emailAddress, 
		String lastname
	);
}
```

Транслируется в JPQL: `select u from User u where u.emailAddress = ?1 and u.lastname = ?2`. Spring Data JPA выполняет проверку свойств и обход вложенных свойств.

**Поддерживаемые ключевые слова:**

| Ключевое слово         | Пример                               | JPQL-фрагмент                                                  |
| ---------------------- | ------------------------------------ | -------------------------------------------------------------- |
| `Distinct`             | `findDistinctByLastnameAndFirstname` | `select distinct … where x.lastname = ?1 and x.firstname = ?2` |
| `And`                  | `findByLastnameAndFirstname`         | `… where x.lastname = ?1 and x.firstname = ?2`                 |
| `Or`                   | `findByLastnameOrFirstname`          | `… where x.lastname = ?1 or x.firstname = ?2`                  |
| `Is`, `Equals`         | `findByFirstnameIs`                  | `… where x.firstname = ?1` (или `IS NULL`, если аргумент null) |
| `Between`              | `findByStartDateBetween`             | `… where x.startDate between ?1 and ?2`                        |
| `LessThan`             | `findByAgeLessThan`                  | `… where x.age < ?1`                                           |
| `LessThanEqual`        | `findByAgeLessThanEqual`             | `… where x.age <= ?1`                                          |
| `GreaterThan`          | `findByAgeGreaterThan`               | `… where x.age > ?1`                                           |
| `GreaterThanEqual`     | `findByAgeGreaterThanEqual`          | `… where x.age >= ?1`                                          |
| `After`                | `findByStartDateAfter`               | `… where x.startDate > ?1`                                     |
| `Before`               | `findByStartDateBefore`              | `… where x.startDate < ?1`                                     |
| `IsNull`, `Null`       | `findByAgeNull`                      | `… where x.age is null`                                        |
| `IsNotNull`, `NotNull` | `findByAgeNotNull`                   | `… where x.age is not null`                                    |
| `Like`                 | `findByFirstnameLike`                | `… where x.firstname like ?1`                                  |
| `NotLike`              | `findByFirstnameNotLike`             | `… where x.firstname not like ?1`                              |
| `StartingWith`         | `findByFirstnameStartingWith`        | `… where x.firstname like ?1` (с добавленным `%`)              |
| `EndingWith`           | `findByFirstnameEndingWith`          | `… where x.firstname like ?1` (с добавленным `%`)              |
| `Containing`           | `findByFirstnameContaining`          | `… where x.firstname like ?1` (с `%` с обеих сторон)           |
| `OrderBy`              | `findByAgeOrderByLastnameDesc`       | `… where x.age = ?1 order by x.lastname desc`                  |
| `Not`                  | `findByLastnameNot`                  | `… where x.lastname <> ?1` (или `IS NOT NULL`)                 |
| `In`                   | `findByAgeIn(Collection ages)`       | `… where x.age in ?1`                                          |
| `NotIn`                | `findByAgeNotIn(Collection ages)`    | `… where x.age not in ?1`                                      |
| `True`                 | `findByActiveTrue()`                 | `… where x.active = true`                                      |
| `False`                | `findByActiveFalse()`                | `… where x.active = false`                                     |
| `IgnoreCase`           | `findByFirstnameIgnoreCase`          | `… where UPPER(x.firstname) = UPPER(?1)`                       |

**`In` и `NotIn`** принимают любой подкласс `Collection`, а также массивы или varargs.

> **Предупреждение о `DISTINCT`:** 
> `select distinct u from User u` даёт совершенно другой результат, чем `select distinct u.lastname from User u`. 
> 
> В первом случае `User.id` включён, поэтому дубликатов не будет — вернётся вся таблица объектов `User`. 
> 
> Во втором — только уникальные фамилии в виде `List<String>`. `countDistinctByLastname(String lastname)` выведет `select count(distinct u.id) …`, что даст тот же результат, что и `countByLastname`. 
> 
> Поэтому иногда лучше писать запрос вручную через `@Query`.

# 3. Именованные запросы JPA (JPA Named Queries) - на сущности

**Конфигурация на основе аннотаций** имеет преимущество в том, что не требует редактирования другого конфигурационного файла, но ценой этому является необходимость перекомпиляции доменного класса для каждого нового объявления запроса.

```java
@Entity
@NamedQuery(
	name = "User.findByEmailAddress",
	query = "select u from User u where u.emailAddress = ?1")
public class User { }
```

**XML-конфигурация** требует добавления элемента `<named-query />` в файл `orm.xml` в папке `META-INF`:
```xml
<named-query name="User.findByLastname">
  <query>select u from User u where u.lastname = ?1</query>
</named-query>
```

**Объявление интерфейса:**
```java
public interface UserRepository extends JpaRepository<User, Long> {

	List<User> findByLastname(String lastname);
	
	User findByEmailAddress(String emailAddress);
}
```

Spring Data пытается разрешить вызов этих методов к именованному запросу, начиная с простого имени доменного класса, затем имени метода, разделённых точкой. 

Например, `findByLastname` будет искать именованный запрос `User.findByLastname`.

# 4. Использование `@Query`

Именованные запросы — допустимый подход для небольшого числа запросов. 

Но поскольку запросы привязаны к методам Java, их можно напрямую связать аннотацией `@Query` к методу репозитория, освобождая доменный класс от информации о персистентности. 

---
## 4.0 Способы передачи аргументов в запрос в @Query
### 4.0.1 Позиционные параметры (по умолчанию)
Параметры привязываются по **порядку следования** в сигнатуре метода. Используется синтаксис `?1`, `?2`, `?3` и т.д.
```java
@Query("select u from User u where u.firstname = ?1 and u.lastname = ?2")
User findByFirstnameAndLastname(String firstname, String lastname);
```

**Недостаток:** При рефакторинге легко перепутать порядок параметров, что приведёт к трудноуловимым ошибкам.

---
### 4.0.2 Именованные параметры с `@Param`
Каждому параметру метода даётся **явное имя** через аннотацию `@Param`, и в запросе используется синтаксис `:имя`.

```java
@Query("select u from User u where u.firstname = :firstname or u.lastname = :lastname")
User findByLastnameOrFirstname(
	@Param("lastname") String lastname,
    @Param("firstname") String firstname
);
```
**Альтернатива без `@Param`:** 
Начиная с Spring 4, если компиляция выполняется с флагом `-parameters`, имена параметров сохраняются в байт-коде, и `@Param` можно опустить. 
Spring Data JPA автоматически сопоставит `:firstname` с параметром `String firstname`.

---
### 4.0.3 SpEL-выражения в параметрах
Внутри `@Query` можно использовать **Spring Expression Language (SpEL)** для динамической вставки значений, в том числе с доступом к аргументам метода по имени или индексу.
```java
@Query(
	"select u from User u" + 
	"where u.firstname = ?1 and u.firstname = ?#{[0]} " +
    "and u.emailAddress = ?#{principal.emailAddress}")
List<User> findByFirstnameAndCurrentUser(String firstname);
```

**Синтаксис:**
- `?#{[0]}` — доступ к первому аргументу метода (по индексу).
    
- `?{#имя}` — доступ к SpEL-переменной (например, `principal`).
    
- `?#{entityName}` — встроенная переменная с именем сущности.

---
### 4.0.4 Wildcard-сокращение для LIKE

Для `LIKE`-условий можно добавлять `%` прямо в запросе, окружая параметр или SpEL-выражение.
```java
// % с обеих сторон от именованного параметра
@Query("select u from User u where u.lastname like %:lastname%")
List<User> findByLastnameLike(@Param("lastname") String lastname);

// % вокруг SpEL-выражения
@Query("select u from User u where u.lastname like %:#{[0]}%")
List<User> findByLastnameWithSpel(String lastname);
```

---
**Запросы, аннотированные на методе, имеют приоритет над `@NamedQuery` и `orm.xml`.**
```java
public interface UserRepository extends JpaRepository<User, Long> {
	@Query(
		"select u from User u where u.emailAddress = ?1"
	)
	User findByEmailAddress(String emailAddress);
}
```

---
**Продвинутые LIKE-выражения:**
```java
@Query(
	"select u from User u where u.firstname like %?1"
)
List<User> findByFirstnameEndsWith(String firstname);
```

Разделитель `LIKE` (`%`) распознаётся, и запрос преобразуется в валидный JPQL (с удалением `%`). При выполнении переданный параметр дополняется ранее распознанным шаблоном `LIKE`.

---
> **Нативные запросы** используют `@NativeQuery` (аналог `@Query(nativeQuery=true)`) с дополнительной поддержкой `sqlResultSetMapping`:
```java
public interface UserRepository extends JpaRepository<User, Long> {
	@NativeQuery(
		value = "SELECT * FROM USERS WHERE EMAIL_ADDRESS = ?1"
	)
	User findByEmailAddress(String emailAddress);
}
```

Для пагинации нативных запросов нужно указать `countQuery`:
```java
@NativeQuery(
	value = "SELECT * FROM USERS WHERE LASTNAME = ?1",
    countQuery = "SELECT count(*) FROM USERS WHERE LASTNAME = ?1"
)
Page<User> findByLastname(String lastname, Pageable pageable);
```

---
**Получение сырых данных:** 

> [!WARNING] Tupla - сырые колонки данных, без маппинга в класс
> **Tuple** — это **типизированный набор значений**, представляющий одну строку результата запроса. 
> 
> В отличие от возврата готового объекта-сущности, `Tuple` позволяет получить **сырые колонки** из результата SQL-запроса без маппинга на конкретный класс.

Нативные запросы также позволяют читать `Tuple` напрямую, используя `Map` как возвращаемый тип:
```java
@NativeQuery("SELECT * FROM USERS WHERE EMAIL_ADDRESS = ?1")
Map<String, Object> findRawMapByEmail(String emailAddress);

@NativeQuery("SELECT * FROM USERS WHERE LASTNAME = ?1")
List<Map<String, Object>> findRawMapByLastname(String lastname);
```

**Результат:** Каждая `Map` содержит пары ключ-значение, где ключ — **имя колонки в БД**, а значение — **значение в этой колонке**.
```java
// Пример результата для findRawMapByEmail("test@example.com")
{
  "ID": 1,
  "EMAIL_ADDRESS": "test@example.com",
  "FIRST_NAME": "John",
  "LAST_NAME": "Doe"
}
```

> **Примечание:** String-based Tuple Queries поддерживаются только Hibernate. EclipseLink поддерживает только Criteria-based Tuple Queries.

## 4.1. Интроспекция и переписывание запросов (Query Introspection and Rewriting)

Spring Data JPA предоставляет широкий спектр функций для обработки запросов:

- Интроспекция запроса для проекции и выполнение tuple-запроса для интерфейсных проекций.
    
- Использование DTO-проекций с конструкторными выражениями.
    
- Применение динамической сортировки.
    
- Выведение `COUNT`-запроса.


Для этого поставляются парсеры запросов для HQL (Hibernate) и EQL (EclipseLink). Для SQL полная поддержка невозможна из-за диалектных различий — встроенный улучшатель SQL поддерживает только простые запросы. Для сложных нужен `JSqlParser` в classpath или явный `COUNT`-запрос через `@Query(countQuery=…)`. При наличии `JSqlParser` Spring Data JPA будет использовать его для нативных запросов.

**Тонкий контроль** через `@EnableJpaRepositories(queryEnhancerSelector = …)`:
```java
@Configuration
@EnableJpaRepositories(
	queryEnhancerSelector = MyQueryEnhancerSelector.class
)
class ApplicationConfig { … }
```

`QueryEnhancerSelector` - это интерфейс, предназначенный для выбора обработчика запросов на основе конкретного запроса. 
При желании ***вы также можете предоставить свою собственную реализацию `QueryEnhancer`.***

## 4.2. Применение `QueryRewriter` - перехватчик
 
Иногда, сколько бы функций вы ни применили, невозможно заставить Spring Data JPA применить всё, что вы хотите, к запросу перед его отправкой в `EntityManager`. 

Вы можете **перехватить запрос прямо перед отправкой** и «переписать» его:
```java
public interface MyRepository extends JpaRepository<User, Long> {
    @NativeQuery(
	    value = 
	    "select original_user_alias.* from SD_USER original_user_alias",
        queryRewriter = MyQueryRewriter.class
    )
    List<User> findByNativeQuery(String param);

    @Query(
	    value = 
		    "select original_user_alias from User original_user_alias",
	    queryRewriter = MyQueryRewriter.class
    )
    List<User> findByNonNativeQuery(String param);
}
```

Spring Data JPA будет искать бин соответствующего типа в контексте приложения. Реализация `QueryRewriter`:
```java
public class MyQueryRewriter implements QueryRewriter {
     @Override
     public String rewrite(String query, Sort sort) {
         return query.replaceAll(
	         "original_user_alias", 
	         "rewritten_user_alias"
	    );
     }
}
```

**Важно:** `QueryRewriter` должен быть зарегистрирован в контексте приложения (через `@Component`, `@Bean` или репозиторий может сам реализовать интерфейс).

**Репозиторий как `QueryRewriter`:**
```java
public interface MyRepository 
extends JpaRepository<User, Long>, QueryRewriter 
{
    @Query(
	    value = "select original_user_alias from User original_user_alias",
        queryRewriter = MyRepository.class
    )
    List<User> findByNonNativeQuery(String param);

    @Override
    default String rewrite(String query, Sort sort) {
        return query.replaceAll(
	        "original_user_alias", 
	        "rewritten_user_alias"
	    );
    }
}
```

# 5. Использование `Sort`

Сортировка может быть выполнена через `PageRequest` или `Sort` напрямую. 

Свойства в `Order` должны соответствовать доменной модели. 

Использование `Sort` с `@Query` позволяет вставлять функции в `ORDER BY`. 

По умолчанию Spring Data JPA отклоняет любой `Order` с функциями, но можно использовать **`JpaSort.unsafe`**:

```java
public interface UserRepository extends JpaRepository<User, Long> {
  @Query("select u from User u where u.lastname like ?1%")
  List<User> findByAndSort(String lastname, Sort sort);
}

repo.findByAndSort("lannister", Sort.by("firstname"));               // ✅ Валидный


repo.findByAndSort("stark", Sort.by("LENGTH(firstname)"));           // ❌ Исключение

repo.findByAndSort("targaryen", JpaSort.unsafe("LENGTH(firstname)")); // ✅ Разрешён
```

**Режимы работы `JpaSort.unsafe(…)`:**
- Для производных и строковых запросов — строка добавляется в запрос.
    
- Для Query by Example/Specifications — выражения парсятся и добавляются в `CriteriaQuery`.

# 6. Скроллинг больших результатов

При работе с большими наборами данных можно использовать:

- **Пейджинг** (`Pageable`/`PageRequest`).
    
- **Offset-based scrolling** — облегчённый вариант, не требующий подсчёта общего количества результатов.
    
- **Scroll API** с query methods, Query-by-Example и Querydsl.


> **Ограничения:** Скроллинг со строковыми методами запросов пока не поддерживается. Скроллинг также не поддерживается для методов с `@Procedure`.

(Детали по Offset/Keyset-скроллингу см. в общем разделе Scrolling — глава 10.5 части II.)

# 7. Использование именованных параметров

По умолчанию Spring Data JPA использует **позиционную привязку параметров**, что может приводить к ошибкам при рефакторинге. Решение — аннотация **`@Param`**:
```java
public interface UserRepository extends JpaRepository<User, Long> {
  @Query("select u from User u where u.firstname = :firstname or u.lastname = :lastname")
  User findByLastnameOrFirstname(
	@Param("lastname") String lastname,
	@Param("firstname") String firstname);
}
```

> Начиная с версии 4, Spring полностью поддерживает обнаружение имён параметров Java 8 на основе флага компилятора `-parameters`. Используя этот флаг, можно опустить `@Param`.

# 8. Шаблонные запросы и выражения

Spring Data JPA поддерживает ограниченные выражения в запросах `@Query`. При выполнении они вычисляются с предопределённым набором переменных.

**`entityName`:** Вставляет имя сущности, связанной с репозиторием. Если у `@Entity` задано свойство `name` — используется оно, иначе — простое имя класса.
```java
@Entity
public class User {
  @Id @GeneratedValue Long id;
  String lastname;
}

public interface UserRepository extends JpaRepository<User,Long> {
  @Query("select u from #{#entityName} u where u.lastname = ?1")
  List<User> findByLastname(String lastname);
}
```

Преимущество: ссылка на `#entityName` автоматически подхватывает будущие переименования сущности (например, `@Entity(name = "MyUser")`).


**Пример с наследованием:**
```java
@MappedSuperclass
public abstract class AbstractMappedType { … }

@Entity
public class ConcreteType extends AbstractMappedType { … }

@NoRepositoryBean
public interface MappedTypeRepository<T extends AbstractMappedType>
  extends Repository<T, Long> {
  @Query("select t from #{#entityName} t where t.attribute = ?1")
  List<T> findAllByAttribute(String attribute);
}

public interface ConcreteRepository
  extends MappedTypeRepository<ConcreteType> { … }
```

Вызов `findAllByAttribute(…)` на `ConcreteRepository` превратится в `select t from ConcreteType t where t.attribute = ?1`.


**Доступ к аргументам:**
```java
@Query("select u from User u where u.firstname = ?1 and u.firstname=?#{[0]} and u.emailAddress = ?#{principal.emailAddress}")
List<User> findByFirstnameAndCurrentUserWithCustomQuery(String firstname);
```


**Wildcard-сокращение для LIKE:**
```java
@Query("select u from User u where u.lastname like %:#{[0]}% and u.lastname like %:lastname%")
List<User> findByLastnameWithSpelExpression(
	@Param("lastname") String lastname
);
```


**Санирование входных значений (экранирование wildcards):**
```java
@Query("select u from User u where u.firstname like %?#{escape([0])}% escape ?#{escapeCharacter()}")
List<User> findContainingEscaped(String namePart);
```
Метод `escape(String)` в SpEL-контексте экранирует стандартные wildcards `_` и `%`. Экранирующий символ настраивается через `escapeCharacter` в `@EnableJpaRepositories`.


**Ссылки на конфигурационные свойства:**
```java
@Query("select u from User u where u.applicationName = ?${spring.application.name:unknown}")
List<User> findContainingEscaped(String namePart);
```

# 9. Другие методы запросов
Если возможностей Spring Data JPA недостаточно:
1. **Напишите запрос вручную через `@Query`.**
    
2. **Создайте кастомную реализацию** — полный контроль: работа напрямую с `EntityManager` (чистый HQL/JPQL/native SQL или Criteria API), использование `JdbcTemplate`, или сторонних библиотек.
    
3. **Поместите запрос внутрь БД** — используйте `@StoredProcedure` или `@Query` с `CALL`.

# 10. Модифицирующие запросы

Для модифицирующих операций используйте аннотацию **`@Modifying`**:
```java
@Modifying
@Query("update User u set u.firstname = ?1 where u.lastname = ?2")
int setFixedFirstnameFor(String firstname, String lastname);
```

Это указывает, что запрос — обновляющий, а не выбирающий. **`EntityManager` может содержать устаревшие сущности после выполнения**, поэтому он не очищается автоматически. Для автоматической очистки: `@Modifying(clearAutomatically = true)`.

> `@Modifying` работает только с `@Query`. Производные методы запросов и кастомные методы не требуют этой аннотации.

## 10.1 **Производные запросы на удаление:**
```java
interface UserRepository extends Repository<User, Long> {
  // Загружает сущности, затем удаляет по одному
  void deleteByRoleId(long roleId); 

  @Modifying
  @Query("delete from User u where u.role.id = ?1") // Один JPQL-запрос
  void deleteInBulkByRoleId(long roleId);
}
```

**Разница:** `deleteByRoleId(…)` загружает сущности и удаляет по одной (вызываются `@PreRemove` callback'и). `deleteInBulkByRoleId(…)` выполняет один JPQL-запрос, но **lifecycle-callback'и не вызываются**.

> При удалении большого количества объектов все они загружаются в память перед удалением и удерживаются в сессии до сброса или завершения транзакции. Учитывайте это для производительности.

# 11. Применение подсказок запросов (Query Hints)

Используйте `@QueryHints` с массивом `@QueryHint` и флагом `forCounting`:
```java
public interface UserRepository extends Repository<User, Long> {
	@QueryHints(
		value = { @QueryHint(name = "name", value = "value")},
		forCounting = false
	)
	Page<User> findByLastname(String lastname, Pageable pageable);
}
```
Подсказка применяется к основному запросу, но **не к count-запросу**

## 11.1 Комментарии к запросу
**Добавление комментариев к запросам (для отладки):** 
Используйте `@Meta`:
```java
public interface RoleRepository extends JpaRepository<Role, Integer> {
  @Meta(comment = "find roles by name")
  List<Role> findByName(String name);

  @Override
  @Meta(comment = "count roles for a given name")
  long countByName(String name);
}
```

Эта функция распространяется на `find`, `count`, `exists` и некоторые `delete`-операции. **Не все операции `EntityManager` поддерживают комментарии** (например, `entityManager.find()` — нет).

---
## 11.2 Активация комментариев
**Активация комментариев для Hibernate:** `hibernate.use_sql_comments=true`.  

**Для EclipseLink:** `eclipselink.logging.level.sql=FINE`.

# 12. Настройка Fetch- и LoadGraphs

Спецификация JPA 2.1 ввела поддержку Fetch- и LoadGraphs. Spring Data JPA поддерживает их через аннотацию **`@EntityGraph`**:
```java
@Entity
@NamedEntityGraph(
	name = "GroupInfo.detail",
	attributeNodes = @NamedAttributeNode("members")
)
public class GroupInfo {
	@ManyToMany
	List<GroupMember> members = new ArrayList<>();
}
```

Ссылка на именованный граф в методе репозитория:
```java
public interface GroupRepository 
extends CrudRepository<GroupInfo, String> 
{
  @EntityGraph(value = "GroupInfo.detail", type = EntityGraphType.LOAD)
  GroupInfo getByGroupName(String name);
}
```

**Ad hoc графы** без `@NamedEntityGraph`:
```java
public interface GroupRepository extends CrudRepository<GroupInfo, String> {
  @EntityGraph(attributePaths = { "members" })
  GroupInfo getByGroupName(String name);
}
```

# 13. Скроллинг (Scrolling)

Детали по Offset/Keyset-скроллингу идентичны общему разделу Scrolling (см. главу 10.5 части II — Scrolling). Кратко:

- **Offset-based:** Пропускает результаты по смещению (похоже на пагинацию). Требует материализации полного результата.
    
- **Keyset-Filtering:** Использует ключи для продолжения скроллинга, не требует полной материализации. Требует non-nullable ключевых свойств и индекса по полям сортировки.
    
- **`WindowIterator`** упрощает итерацию по `Window<T>`.
    
- `ScrollPosition.offset()` — начало скролла (указывает на начало). `ScrollPosition.offset(0)` — указывает на первый элемент (пропускает его из-за исключающего характера скроллинга).
