# 1. Транзакционное поведение по умолчанию

Методы, унаследованные от `CrudRepository`, **уже имеют транзакционную конфигурацию**, унаследованную от `SimpleJpaRepository`:

- **Операции чтения** (`findById`, `findAll` и т.д.) — выполняются с флагом **`readOnly = true`**.
    
- **Операции записи** (`save`, `delete` и т.д.) — выполняются с обычной `@Transactional`, применяя стандартную конфигурацию транзакций.


Методы репозитория, поддерживаемые транзакционными фрагментами репозитория, **наследуют транзакционные атрибуты от фактического метода фрагмента**.

---
## 2. Переопределение транзакционной конфигурации для CRUD-методов

***Если нужно изменить настройки транзакции*** для одного из унаследованных методов, достаточно **переобъявить метод в интерфейсе репозитория**:
```java
public interface UserRepository extends CrudRepository<User, Long> {

  @Override
  @Transactional(timeout = 10)
  public List<User> findAll();

  // Дальнейшие объявления методов запросов
}
```

Это заставит метод `findAll()` выполняться с **таймаутом 10 секунд** и **без флага `readOnly`**.

---
## 3. Использование фасада/сервиса для управления транзакциями

Другой *способ изменить транзакционное поведение* — использовать **фасад или сервисную реализацию**, которая обычно охватывает более одного репозитория. 

Её цель — определить транзакционные границы для не-CRUD операций:
```java
@Service
public class UserManagementImpl implements UserManagement {

  private final UserRepository userRepository;
  private final RoleRepository roleRepository;

  public UserManagementImpl(UserRepository userRepository,
    RoleRepository roleRepository) {
    this.userRepository = userRepository;
    this.roleRepository = roleRepository;
  }

  @Transactional // ИСПОЛЬЗУЕТСЯ ЭТА ТРАНЗАКЦИЯ, а не настроенная в репозитории
  public void addRoleToAllUsers(String roleName) {

    Role role = roleRepository.findByName(roleName);

    for (User user : userRepository.findAll()) {
      user.addRole(role);
      userRepository.save(user);
    }
  }
}
```

**Как это работает:**
- Вызов `addRoleToAllUsers(…)` выполняется **внутри транзакции** (участвует в существующей или создаёт новую, если ни одна не запущена).
    
- Конфигурация транзакций на уровне репозиториев **игнорируется** — внешняя транзакционная конфигурация определяет фактически используемую.
    
- **Необходимо явно активировать** `<tx:annotation-driven/>` или `@EnableTransactionManagement`, чтобы аннотационная конфигурация фасадов работала.

> **Примечание о `save`:** 
> Вызов `save` не является строго необходимым с точки зрения JPA (Hibernate отслеживает изменения и сохраняет их автоматически), но его следует оставить для **согласованности с абстракцией репозитория**, предлагаемой Spring Data.

## 4. Транзакционные методы запросов

> [!WARNING] Явно объявленные методы НЕ создают транзакцию сами
> Объявленные методы запросов (включая default-методы) по умолчанию НЕ получают транзакционной конфигурации.

| Сценарий                                                     | Транзакция есть?                                | Режим                                          |
| ------------------------------------------------------------ | ----------------------------------------------- | ---------------------------------------------- |
| `findByLastname` вызван **без внешней `@Transactional`**     | ❌ Нет, JPA выполнит запрос в режиме auto-commit | Может привести к `LazyInitializationException` |
| `findByLastname` вызван **внутри метода с `@Transactional`** | ✅ Да, участвует во внешней транзакции           | Безопасно                                      |

Сам метод `findByLastname` без `@Transactional` просто делает запрос к БД напрямую, без создания транзакции. 

Но если ему нужна поддержка lazy-loading или согласованность с другими операциями — нужна внешняя `@Transactional`. 

Именно поэтому документация рекомендует вешать `@Transactional(readOnly = true)` на весь интерфейс репозитория для методов чтения.

---

Чтобы они выполнялись транзакционно, используйте `@Transactional` на интерфейсе репозитория:
```java
@Transactional(readOnly = true)
interface UserRepository extends JpaRepository<User, Long> {

  List<User> findByLastname(String lastname);

  @Modifying
  @Transactional
  @Query("delete from User u where u.active = false")
  void deleteInactiveUsers();
}
```

**Разбор примера:**
- **`@Transactional(readOnly = true)`** на уровне интерфейса делает все методы запросов транзакционными с флагом «только для чтения».
    
- **`deleteInactiveUsers()`** использует `@Modifying` и переопределяет конфигурацию транзакции на `@Transactional` без `readOnly` — метод будет выполняться с `readOnly = false`.
    

---
## 5. Что на самом деле делает `readOnly = true`?

**Важно понимать:** `readOnly` — это **НЕ проверка** того, что вы не выполняете модифицирующий запрос (хотя некоторые БД отклоняют `INSERT`/`UPDATE` внутри read-only транзакции).

Флаг `readOnly` работает как **подсказка** для:
1. **JDBC-драйвера** — для оптимизаций производительности.
    
2. **JPA-провайдера** — Spring выполняет оптимизации на уровне провайдера. Например, при использовании **Hibernate** режим flush устанавливается в **`NEVER`**, что заставляет Hibernate **пропускать dirty checks** (проверки изменений). Это даёт заметное улучшение производительности на больших деревьях объектов.

---
## 6. Рекомендация

Хотя примеры обсуждают использование `@Transactional` на репозитории, **общая рекомендация** — объявлять транзакционные границы на уровне **единицы работы** (unit of work), чтобы обеспечить правильную согласованность и желаемое участие в транзакциях.
