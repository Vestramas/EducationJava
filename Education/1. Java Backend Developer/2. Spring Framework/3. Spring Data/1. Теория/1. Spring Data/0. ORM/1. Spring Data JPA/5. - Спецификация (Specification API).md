Эта глава о том, как строить **динамические запросы**, когда заранее неизвестно, какие именно условия понадобятся. 

**Проблема, которую решают спецификации**
Представьте: у вас есть страница поиска клиентов с фильтрами:
- По имени
    
- По статусу
    
- По сумме покупок
    
- По дате регистрации
    
- И ещё десяток полей...

Пользователь может заполнить любую комбинацию фильтров. **Сколько методов в репозитории вам нужно?**


**Спецификации решают эту проблему.** 
Вместо тысячи методов вы создаёте **по одному методу на каждый критерий**, а потом **комбинируете их на лету** через `and()`, `or()`, `not()`:
```java
// Определяем критерии по отдельности
PredicateSpecification<Customer> hasName = (from, builder) ->
    builder.equal(from.get("name"), "Иван");

PredicateSpecification<Customer> isActive = (from, builder) ->
    builder.equal(from.get("status"), Status.ACTIVE);

PredicateSpecification<Customer> hasHighSales = (from, builder) ->
    builder.greaterThan(from.get("totalSales"), 10000);


// Комбинируем что нужно прямо сейчас
List<Customer> result = repository.findAll(
    hasName.and(isActive).or(hasHighSales)
);
```

JPA Criteria API **позволяет строить запросы программно**. 

Spring Data JPA предоставляет **`Specification`** — **компактный API для выражения предикатов над сущностями** и их переиспользования между репозиториями. Основанный на концепции спецификаций из книги Эрика Эванса «Domain Driven Design», этот API следует той же семантике, используя JPA Criteria.

Для поддержки спецификаций нужно расширить интерфейс репозитория с помощью **`JpaSpecificationExecutor`**:
```java
public interface CustomerRepository 
extends 
	CrudRepository<Customer, Long>, 
	JpaSpecificationExecutor<Customer> {
}
```

Спецификация — это **предикат над сущностью, выраженный через Criteria API**. 

Spring Data JPA предлагает две точки входа:
- **`PredicateSpecification`** — гибкий, агностичный к типу запроса интерфейс (введён в Spring Data JPA 4.0).
    
- **`Specification`**, **`UpdateSpecification`**, **`DeleteSpecification`** — варианты, привязанные к типу запроса.

---
## 1. `PredicateSpecification`

Определён с минимальным набором зависимостей, что позволяет широко комбинировать функции:
```java
public interface PredicateSpecification<T> {
  Predicate toPredicate(From<?, T> from, CriteriaBuilder builder);
}
```

Спецификации можно легко использовать для построения расширяемого набора предикатов, избавляя от необходимости объявлять отдельный метод запроса для каждой комбинации:

```java
class CustomerSpecs {

	static PredicateSpecification<Customer> isLongTermCustomer() {
		return (from, builder) -> {
		  LocalDate date = LocalDate.now().minusYears(2);
		  return builder.lessThan(from.get(Customer_.createdAt), date);
		};
	}
	
	static PredicateSpecification<Customer> hasSalesOfMoreThan(
		MonetaryAmount value
	) {
		return (from, builder) -> {
		  // построить предикат для sales > value
		};
	}
}
```

**`Customer_`** — это тип метамодели, сгенерированный с помощью JPA Metamodel Generator. Выражение `Customer_.createdAt` предполагает, что у `Customer` есть атрибут `createdAt` типа `Date`. Спецификации выражены на уровне абстракции бизнес-требований.

**Использование напрямую:**
```java
List<Customer> customers = customerRepository.findAll(isLongTermCustomer());
```

**Наибольшая ценность — в композиции:**
```java
MonetaryAmount amount = new MonetaryAmount(200.0, Currencies.DOLLAR);
List<Customer> customers = customerRepository.findAll(
  isLongTermCustomer().or(hasSalesOfMoreThan(amount))
);
```

## 2. `Specification`, `UpdateSpecification`, `DeleteSpecification`

Интерфейс `Specification` существует дольше и **привязан к конкретному типу запроса** (select, update, delete) в соответствии с ограничениями Criteria API:

```java
public interface Specification<T> {
	Predicate toPredicate(
		Root<T> root, 
		CriteriaQuery<?> query, 
		CriteriaBuilder builder
	);
}
```

Объекты `Specification` можно создавать напрямую или переиспользуя экземпляры `PredicateSpecification`:
```java
public class CustomerSpecs {

	public static UpdateSpecification<Customer> 
	updateLastnameByFirstnameAndLastname(
		String newLastName, 
		String currentFirstname, 
		String currentLastname
	) {
	return UpdateSpecification
	.<User>update(
		(root, update, criteriaBuilder) -> {
			update.set("lastname", newLastName);
		})
		.where(
			hasFirstname(currentFirstname)
			.and(hasLastname(currentLastname))
		);
}

	public static PredicateSpecification<Customer> hasFirstname(
		String firstname
	) {
		return (root, builder) -> 
			builder.equal(from.get("firstname"), value);
	}

	public static PredicateSpecification<Customer> hasLastname(
		String lastname
	) {
		return (root, builder) -> { /* построить запрос */ };
	}
}
```

## 3. Fluent API

`JpaSpecificationExecutor` определяет **fluent-методы запросов** для гибкого выполнения на основе экземпляров `Specification`:

- Для `PredicateSpecification`: **`findBy(PredicateSpecification<T> spec, Function<…> queryFunction)`**
    
- Для `Specification`: **`findBy(Specification<T> spec, Function<…> queryFunction)`**


`queryFunction` позволяет контролировать аспекты выполнения запроса, которые иначе нельзя динамически настроить, через методы `SpecificationFluentQuery`.

---
**Промежуточные методы (intermediate):**

|Метод|Описание|
|---|---|
|`sortBy`|Применяет сортировку. Повторные вызовы добавляют каждый `Sort`. `page(Pageable)` с отсортированным `Pageable` переопределяет предыдущий порядок.|
|`limit`|Ограничивает количество результатов.|
|`as`|Указывает тип для чтения или проекции.|
|`project`|Ограничивает свойства запроса.|

---
**Терминальные методы (terminal):**

|Метод|Описание|
|---|---|
|`first`, `firstValue`|Возвращает первое значение. `first` → `Optional<T>`, `firstValue` → nullable вариант.|
|`one`, `oneValue`|Возвращает единственное значение. `one` → `Optional<T>`, `oneValue` → nullable. Бросает `IncorrectResultSizeDataAccessException`, если найдено более одного совпадения.|
|`all`|Возвращает все результаты как `List<T>`.|
|`page(Pageable)`|Возвращает результаты как `Page<T>`.|
|`slice(Pageable)`|Возвращает результаты как `Slice<T>`.|
|`scroll(ScrollPosition)`|Использует скроллинг для получения результатов как `Window<T>`.|
|`stream()`|Возвращает `Stream<T>` для ленивой обработки. **Поток должен быть закрыт после использования.**|
|`count`, `exists`|Возвращает количество совпадающих сущностей или проверяет наличие совпадений.|

> Промежуточные и терминальные методы должны вызываться **внутри query function**.

**Примеры:**
```java
// Получить Page с проекцией, отсортированный по lastname
Page<CustomerProjection> page = repository.findBy(spec,
    q -> q.as(CustomerProjection.class)
          .page(PageRequest.of(0, 20, Sort.by("lastname")))
);

// Получить последний из потенциально многих результатов, отсортированный по lastname
Optional<Customer> match = repository.findBy(spec,
    q -> q.sortBy(Sort.by("lastname").descending())
          .first()
);
```
