# 1. Блокировки (Locking)

Чтобы указать режим блокировки, используется аннотация **`@Lock`** на методах запросов:
```java
interface UserRepository extends Repository<User, Long> {

  // Обычный метод запроса
  @Lock(LockModeType.READ)
  List<User> findByLastname(String lastname);
}
```

Это объявление метода приводит к тому, что инициируемый запрос будет оснащён `LockModeType.READ`. 

Также можно определить блокировку для CRUD-методов, переобъявив их в интерфейсе репозитория и добавив аннотацию `@Lock`:
```java
interface UserRepository extends Repository<User, Long> {

  // Переобъявление CRUD-метода
  @Lock(LockModeType.READ)
  List<User> findAll();
}
```

# 2. Аудит
## 1. Основы
Spring Data предоставляет продвинутую поддержку для прозрачного отслеживания того, **кто** создал или изменил сущность и **когда** это произошло. Чтобы использовать эту функциональность, необходимо:

1. Оснастить классы сущностей метаданными аудита (через аннотации или реализацию интерфейса).
    
2. Включить аудит через конфигурацию аннотациями или XML.
    

**Важно:** Приложения, которые отслеживают только даты создания и изменения, **не обязаны** реализовывать `AuditorAware` в своих сущностях.

---
## 2. Аннотационные метаданные аудита

Spring Data предоставляет аннотации 
- **`@CreatedBy`** 
- и **`@LastModifiedBy`** 
**для фиксации пользователя** 

и 
- **`@CreatedDate`** 
- и **`@LastModifiedDate`** 
**для фиксации времени:**

```java
class Customer {

  @CreatedBy
  private User user;

  @CreatedDate
  private Instant createdDate;

  // … другие свойства опущены
}
```

Аннотации можно применять **выборочно**, в зависимости от того, какую информацию вы хотите фиксировать. Аннотации, указывающие на фиксацию времени изменений, могут использоваться для свойств типов JDK8 date/time, `long`, `Long` и устаревших `java.util.Date` и `Calendar`.

Текущее время предоставляется `org.springframework.data.auditing.DateTimeProvider`. 

По умолчанию это **`CurrentDateTimeProvider`**. Его можно изменить через атрибут `dateTimeProviderRef` при включении аудита или через наличие в `ApplicationContext` выделенного бина `AuditingHandler` или `DateTimeProvider`.

Метаданные аудита не обязательно должны находиться в корневой сущности — их можно добавить во встроенный объект:
```java
class Customer {

  private AuditMetadata auditingMetadata;

  // … другие свойства опущены
}

class AuditMetadata {

  @CreatedBy
  private User user;

  @CreatedDate
  private Instant createdDate;
}
```

## 3. Аудит на основе интерфейса

Если вы не хотите использовать аннотации, можно реализовать интерфейс **`Auditable`**. Он предоставляет setter-методы для всех свойств аудита.

---
## 4. `AuditorAware`

Если вы используете `@CreatedBy` или `@LastModifiedBy`, **инфраструктура аудита должна знать текущего пользователя**. 

Для этого существует SPI-интерфейс **`AuditorAware<T>`**. 
Дженерик-тип `T` определяет, какой тип должны иметь свойства, аннотированные `@CreatedBy` или `@LastModifiedBy`.

**Пример реализации с Spring Security:**
```java
class SpringSecurityAuditorAware implements AuditorAware<User> {

  @Override
  public Optional<User> getCurrentAuditor() {

    return Optional.ofNullable(SecurityContextHolder.getContext())
            .map(SecurityContext::getAuthentication)
            .filter(Authentication::isAuthenticated)
            .map(Authentication::getPrincipal)
            .map(User.class::cast);
  }
}
```

Реализация обращается к объекту `Authentication` из Spring Security и находит экземпляр `UserDetails`, созданный в вашей реализации `UserDetailsService`.

---
## 5. `ReactiveAuditorAware`

Для реактивной инфраструктуры существует интерфейс **`ReactiveAuditorAware<T>`**:
```java
class SpringSecurityAuditorAware implements ReactiveAuditorAware<User> {

  @Override
  public Mono<User> getCurrentAuditor() {

    return ReactiveSecurityContextHolder.getContext()
                .mapNotNull(SecurityContext::getAuthentication)
                .filter(Authentication::isAuthenticated)
                .mapNotNull(Authentication::getPrincipal)
                .map(User.class::cast);
  }
}
```

## 6. Общая конфигурация аудита в JPA

Spring Data JPA поставляется с **entity listener** (`AuditingEntityListener`), который используется для запуска фиксации аудиторской информации. 

Его нужно зарегистрировать для всех сущностей в ваших persistence contexts через файл `orm.xml`:
```xml
<persistence-unit-metadata>
  <persistence-unit-defaults>
    <entity-listeners>
      <entity-listener class="….data.jpa.domain.support.AuditingEntityListener" />
    </entity-listeners>
  </persistence-unit-defaults>
</persistence-unit-metadata>
```

**Активация аудита через XML:**
```xml
<jpa:auditing auditor-aware-ref="yourAuditorAwareBean" />
```


Также можно включить `AuditingEntityListener` для отдельных сущностей через аннотацию `@EntityListeners`:
```java
@Entity
@EntityListeners(AuditingEntityListener.class)
public class MyEntity { }
```

**Активация аудита через Java-конфигурацию** (начиная с Spring Data JPA 1.5):
```java
@Configuration
@EnableJpaAuditing
class Config {

  @Bean
  public AuditorAware<AuditableUser> auditorProvider() {
    return new AuditorAwareImpl();
  }
}
```

Если вы предоставляете бин типа `AuditorAware` в `ApplicationContext`, инфраструктура аудита **автоматически подхватывает его** и использует для определения текущего пользователя. 

Если в `ApplicationContext` зарегистрировано несколько реализаций, можно выбрать нужную, явно указав атрибут `auditorAwareRef` в `@EnableJpaAuditing`.
