Мы начнём с рассмотрения Hibernate в среде Spring, используя его для демонстрации подхода Spring к интеграции ORM-мапперов. В этом разделе многие вопросы рассматриваются подробно, а также показываются различные варианты реализации DAO и разграничения транзакций. Большинство этих шаблонов могут быть напрямую перенесены на все остальные поддерживаемые ORM-инструменты. Последующие разделы этой главы затем рассматривают другие ORM-технологии и приводят краткие примеры.

> **Начиная с Spring Framework 7.0,** 
> Spring требует Hibernate ORM 7.x для `Spring` `HibernateJpaVendorAdapter`.
> 
> ***Пакет*** `org.springframework.orm.jpa.hibernate` **заменяет прежний** `orm.hibernate5`: теперь он предназначен для использования с Hibernate ORM 7.x, тесно интегрирован с `HibernateJpaVendorAdapter`, а также поддерживает нативный стиль Hibernate `SessionFactory.getCurrentSession()`.

# 1. Настройка `SessionFactory` в Spring-контейнере
Чтобы избежать привязки объектов приложения к жёстко закодированным поискам ресурсов, вы можете определять ресурсы (такие как JDBC `DataSource` или Hibernate `SessionFactory`) как бины в контейнере Spring. 

Объекты приложения, которым нужен доступ к ресурсам, получают ссылки на такие предопределённые экземпляры через бины.

Переход от локального `BasicDataSource` (Jakarta Commons DBCP) к `DataSource`, полученному через JNDI (обычно управляемому сервером приложений), является лишь вопросом конфигурации. Вы также можете получить `SessionFactory` через JNDI, используя `JndiObjectFactoryBean` / `<jee:jndi-lookup>`, однако это обычно не распространено вне контекста EJB.


## 1.1. Базовая конфигурация `SessionFactory`
В следующем отрывке из конфигурации показано, как настроить JDBC `DataSource` и Hibernate `SessionFactory` поверх него:

```xml
<beans>
    <bean id="myDataSource" class="org.apache.commons.dbcp.BasicDataSource" destroy-method="close">
        <property name="driverClassName" value="org.hsqldb.jdbcDriver"/>
        <property name="url" value="jdbc:hsqldb:hsql://localhost:9001"/>
        <property name="username" value="sa"/>
        <property name="password" value=""/>
    </bean>

    <bean id="mySessionFactory" class="org.springframework.orm.jpa.hibernate.LocalSessionFactoryBean">
        <property name="dataSource" ref="myDataSource"/>
        <property name="mappingResources">
            <list>
                <value>product.hbm.xml</value>
            </list>
        </property>
        <property name="hibernateProperties">
            <value>
                hibernate.dialect=org.hibernate.dialect.HSQLDialect
            </value>
        </property>
    </bean>
</beans>
```

```java
@Bean
public DataSource myDataSource() {
    BasicDataSource ds = new BasicDataSource();
    
    ds.setDriverClassName("org.hsqldb.jdbcDriver");
    ds.setUrl("jdbc:hsqldb:hsql://localhost:9001");
    ds.setUsername("sa");
    ds.setPassword("");
    
    return ds;
}

@Bean
public LocalSessionFactoryBean mySessionFactory(DataSource myDataSource) {
    LocalSessionFactoryBean factory = new LocalSessionFactoryBean();
    
    factory.setDataSource(myDataSource);
    factory.setMappingResources(List.of("product.hbm.xml"));
    
    Properties props = new Properties();
    props.setProperty(
	    "hibernate.dialect", 
	    "org.hibernate.dialect.HSQLDialect");
	    
    factory.setHibernateProperties(props);
    
    return factory;
}
```

## 1.2. Использование `DataSource` из JNDI
Переход на JNDI `DataSource` — лишь вопрос конфигурации:

XML:
```xml
<jee:jndi-lookup id="myDataSource" jndi-name="java:comp/env/jdbc/myds"/>
```

Java:
```java
@Bean
public JndiObjectFactoryBean myDataSource() {
    JndiObjectFactoryBean jndi = new JndiObjectFactoryBean();
    
    jndi.setJndiName("java:comp/env/jdbc/myds");
    
    return jndi;
}
```

## 1.3. `LocalSessionFactoryBuilder` и фоновый bootstrapping

`LocalSessionFactoryBuilder` — альтернатива `LocalSessionFactoryBean` для `@Bean`-конфигурации. 

И `LocalSessionFactoryBean`, и `LocalSessionFactoryBuilder` поддерживают **фоновый bootstrapping**: инициализация Hibernate выполняется параллельно с потоком загрузки приложения на заданном `bootstrapExecutor` (например, `SimpleAsyncTaskExecutor`). 

- На `LocalSessionFactoryBean` это доступно через свойство `bootstrapExecutor`. 
	
- На программном `LocalSessionFactoryBuilder` есть перегруженный метод `buildSessionFactory()`, принимающий аргумент bootstrap executor.

# 2. DAO на чистом Hibernate API
Hibernate поддерживает **contextual sessions** — сам управляет одной текущей `Session` на транзакцию.
```java
public class ProductDaoImpl implements ProductDao {

    private SessionFactory sessionFactory;

    public void setSessionFactory(SessionFactory sessionFactory) {
        this.sessionFactory = sessionFactory;
    }

    public Collection loadProductsByCategory(String category) {
        return this.sessionFactory.getCurrentSession()
                .createQuery(
	                "from test.Product product where product.category=?")
                .setParameter(0, category)
                .list();
    }
}
```

Этот стиль похож на примеры из справочной документации Hibernate, за исключением хранения `SessionFactory` в переменной экземпляра. 
**Настоятельно рекомендуется** ***использовать такой подход на основе экземпляра вместо статического класса*** `HibernateUtil` из старых примеров Hibernate. (В общем случае, не храните ресурсы в статических переменных, если это не абсолютно необходимо.)

Этот пример DAO следует паттерну dependency injection. Он хорошо вписывается в IoC-контейнер Spring, как если бы был написан с использованием `HibernateTemplate` от Spring. Такой DAO можно настроить и в чистой Java (например, в юнит-тестах): создайте экземпляр и вызовите `setSessionFactory(..)` с нужной фабрикой.


Конфигурация бина DAO в XML:
```xml
<bean id="myProductDao" class="product.ProductDaoImpl">
    <property name="sessionFactory" ref="mySessionFactory"/>
</bean>
```

**Главное преимущество** *этого стиля DAO в том, что он зависит только от Hibernate API.* Никакого импорта классов Spring не требуется. Это привлекательно с точки зрения неинвазивности и может казаться более естественным для разработчиков Hibernate.

**Недостаток:** ==DAO выбрасывает обычное== `HibernateException` (***которое является непроверяемым, так что его не нужно объявлять или перехватывать***), что означает, что вызывающие могут рассматривать исключения только как в целом фатальные — если они не хотят зависеть от собственной иерархии исключений Hibernate. 
Перехват конкретных причин (таких как сбой оптимистичной блокировки) невозможен без привязки вызывающего к стратегии реализации. Этот компромисс может быть приемлемым для приложений, которые сильно ориентированы на Hibernate, не нуждаются в специальной обработке исключений, или и то, и другое.

> К счастью, `LocalSessionFactoryBean` от Spring поддерживает метод Hibernate `SessionFactory.getCurrentSession()` для любой стратегии транзакций Spring, возвращая текущую управляемую Spring транзакционную `Session`, даже с `HibernateTransactionManager`. 
> 
> Стандартное поведение этого метода остаётся прежним: возвращать текущую `Session`, связанную с текущей транзакцией JTA, если таковая имеется. Это поведение применяется независимо от того, используете ли вы `JtaTransactionManager` от Spring, транзакции, управляемые контейнером EJB (CMT), или JTA.

**Итог:** Вы можете реализовывать DAO на основе чистого Hibernate API, при этом имея возможность участвовать в транзакциях, управляемых Spring.

# 3. Декларативное разграничение транзакций

Рекомендуется использовать декларативную поддержку транзакций Spring через `@Transactional`:

```java
public class ProductServiceImpl implements ProductService {

    private ProductDao productDao;

    public void setProductDao(ProductDao productDao) {
        this.productDao = productDao;
    }

    @Transactional
    public void increasePriceOfAllProductsInCategory(
	    final String category
	) {
        List productsToChange = 
	        this.productDao.loadProductsByCategory(category);
        // ...
    }

    @Transactional(readOnly = true)
    public List<Product> findAllProducts() {
        return this.productDao.findAllProducts();
    }
}
```

В контейнере необходимо:
- настроить реализацию`PlatformTransactionManager` (как бин) 
- и элемент `<tx:annotation-driven/>`, включающий обработку `@Transactional` во время выполнения 
	  (ИЛИ `@EnableTransactionmanager`)

XML:
```xml
<beans>
    <!-- SessionFactory, DataSource и т.д. опущены -->

    <bean id="transactionManager"
            class="org.springframework.orm.jpa.hibernate.HibernateTransactionManager">
        <property name="sessionFactory" ref="sessionFactory"/>
    </bean>

    <tx:annotation-driven/>

    <bean id="myProductService" class="product.SimpleProductService">
        <property name="productDao" ref="myProductDao"/>
    </bean>
</beans>
```

Java:
```java
@Bean
public PlatformTransactionManager transactionManager(
	SessionFactory sessionFactory
) {
    return new HibernateTransactionManager(sessionFactory);
}
```
Плюс в классе конфигурации `@EnableTransactionManager`

# 4. Программное разграничение транзакций
Вы можете разграничивать транзакции на более высоком уровне приложения, поверх нижележащих сервисов доступа к данным, которые охватывают любое количество операций. 

Никаких ограничений на реализацию окружающего бизнес-сервиса не накладывается. 

Ему нужен только Spring `PlatformTransactionManager`. 

Опять же, последний может быть получен откуда угодно, но предпочтительно как ссылка на бин через метод `setTransactionManager(..)`. 

Также, `ProductDao` должен быть установлен через метод `setProductDao(..)`.

```java
public class ProductServiceImpl implements ProductService {

    private TransactionTemplate transactionTemplate;
    private ProductDao productDao;

    public void setTransactionManager(
	    PlatformTransactionManager transactionManager
	) {
        this.transactionTemplate = 
	        new TransactionTemplate(transactionManager);
    }

    public void setProductDao(ProductDao productDao) {
        this.productDao = productDao;
    }

    public void increasePriceOfAllProductsInCategory(
	    final String category
	) {
        this.transactionTemplate.execute(
	        new TransactionCallbackWithoutResult() 
	    {
            public void doInTransactionWithoutResult(
	            TransactionStatus status
	        ) {
                List productsToChange = 
	                productDao.loadProductsByCategory(category);
	                
                // увеличение цены...
            }
        });
    }
}
```

Конфигурация в XML:
```xml
<bean id="myTxManager" class="org.springframework.orm.jpa.hibernate.HibernateTransactionManager">
    <property name="sessionFactory" ref="mySessionFactory"/>
</bean>

<bean id="myProductService" class="product.ProductServiceImpl">
    <property name="transactionManager" ref="myTxManager"/>
    <property name="productDao" ref="myProductDao"/>
</bean>
```

**Различия с декларативным подходом:**  
`TransactionInterceptor` от Spring позволяет выбрасывать любое проверяемое (checked) исключение приложения с кодом обратного вызова (callback), в то время как `TransactionTemplate` ограничен непроверяемыми (unchecked) исключениями внутри обратного вызова. 

`TransactionTemplate` инициирует откат в случае непроверяемого исключения приложения или если транзакция помечена приложением как rollback-only (путём установки `TransactionStatus`). 

По умолчанию `TransactionInterceptor` ведёт себя так же, но разрешает конфигурируемые политики отката для каждого метода.

# 5. Стратегии управления транзакциями

В Spring есть два популярных способа работы с транзакциями:

- программный:
```java
TransactionTemplate
```

- декларативный:
```java
@Transactional
```
(внутри используется `TransactionInterceptor`)

Но ни `TransactionTemplate`, ни `TransactionInterceptor` сами транзакциями не управляют.  
Они только говорят:

> «Начни транзакцию»,  
> «Зафиксируй»,  
> «Сделай rollback».

А уже конкретную реализацию выполняет объект:
```java
PlatformTransactionManager
```
Это единый интерфейс Spring для любых транзакций.

---
## 5.1. HibernateTransactionManager
Если приложение использует:
- **один** `SessionFactory`
- **одну БД**
- обычный Hibernate

то обычно используется:
```java
HibernateTransactionManager
```
Он:
- открывает Hibernate `Session`
- привязывает её к текущему потоку (`ThreadLocal`)
- начинает JDBC-транзакцию
- делает commit/rollback

То есть Spring полностью управляет жизненным циклом Hibernate Session.

Когда начинается транзакция:
```java
@Transactional
public void saveUser() {
```
Spring:
1. создаёт `Session`
2. кладёт её в `ThreadLocal`
3. все DAO получают одну и ту же Session через:
```java
sessionFactory.getCurrentSession()
```

По завершении транзакции:
- Session закрывается
- ThreadLocal очищается

---
## 5.2 JtaTransactionManager
Если приложение использует:
- **несколько БД**
- **распределённые транзакции**
- Java EE/Jakarta EE контейнер
- XA-транзакции

то нужен:
```java
JtaTransactionManager
```
Он уже не управляет транзакцией сам.

Он делегирует управление:
- серверу приложений
- JTA-подсистеме
- transaction coordinator

Например:
- WildFly
- WebLogic
- Atomikos
- Narayana

> [!NOTE] КОД НЕ ЗАВИСИТ ОТ TransactionalManager
> Так как Spring использует собственное API для управления транзакциями (`TransactionManager`)

---
## 5.3 Распределенные транзакции
Для распределённых транзакций через несколько фабрик сессий Hibernate вы можете комбинировать `JtaTransactionManager` как стратегию транзакций с несколькими определениями `LocalSessionFactoryBean`. 

Каждый DAO получает одну конкретную ссылку на `SessionFactory`, переданную в его соответствующее свойство бина. 

Если все базовые источники данных JDBC являются транзакционными контейнерными, бизнес-сервис может разграничивать транзакции через любое количество DAO и любое количество фабрик сессий без специальных оговорок, при условии использования `JtaTransactionManager` в качестве стратегии.

**ПРИМЕР**
```
Приложение работает с:  
- DB1  
- DB2  
- JMS broker
  
НУЖНО 
- либо commit везде,  
- либо rollback везде.
```

**Для этого нужен JTA.**

Допустим
```
SessionFactory #1 → DB1
SessionFactory #2 → DB2
SessionFactory #3 → DB3
```

И каждый DAO использует свою фабрику.

Например:
```java
UserDao -> sessionFactory1
OrderDao -> sessionFactory2
AuditDao -> sessionFactory3
```

Но сервис:
```java
@Transactional
public void processOrder()
```
может вызвать все DAO внутри одной общей транзакции. Это возможно благодаря `JtaTransactionManager`

---
## 5.4 Кеш
Как `HibernateTransactionManager`, так и `JtaTransactionManager` позволяют обеспечить правильную обработку кэша на уровне JVM с Hibernate, без специфичного для контейнера поиска менеджера транзакций или JCA-коннектора (если вы не используете EJB для инициации транзакций).

---
## 5.5 Смешанный доступ
**Смешанный доступ Hibernate + JDBC:** `HibernateTransactionManager` может экспортировать JDBC-соединение Hibernate для чистого кода доступа JDBC, работающего с определённым `DataSource`. 

Эта возможность позволяет осуществлять высокоуровневое разграничение транзакций со смешанным доступом к данным Hibernate и JDBC полностью без JTA, ==при условии доступа только к одной базе данных. ==

`HibernateTransactionManager` автоматически предоставляет транзакцию Hibernate как транзакцию JDBC, если вы настроили переданный `SessionFactory` с `DataSource` через свойство `dataSource` класса `LocalSessionFactoryBean`. 

В качестве альтернативы вы можете явно указать `DataSource`, для которого предполагается предоставлять транзакции, через свойство `dataSource` класса `HibernateTransactionManager`.

# 6. Сравнение ресурсов, управляемых контейнером JNDI и определённых локально
Hibernate `SessionFactory` можно получить двумя способами:

1. **Локально:** Вы сами создаёте `SessionFactory` через Spring (`LocalSessionFactoryBean`).
    
2. **Из контейнера:** Вы получаете готовую `SessionFactory` из JNDI (её туда зарегистрировал администратор сервера).

## 6.1 **Локальная `SessionFactory` + Spring-транзакции — почти всегда достаточно.**

Spring не зависит от сервера приложений. Если вы используете локальную `SessionFactory` и `HibernateTransactionManager`:

- Всё работает в **standalone-приложении**.
    
- Всё работает в **тестах**.
    
- Всё работает на **Tomcat** (который не является полноценным Java EE-сервером).
    

Для 95% приложений с **одной базой данных** этого более чем достаточно.

---
## 6.2 JNDI
**Когда нужна JNDI `SessionFactory`? Только когда вы используете EJB.**
*Если у вас EJB* (Enterprise Java Beans), ***то транзакциями управляет контейнер через CMT*** (Container-Managed Transactions). 

В этом случае JCA-коннектор Hibernate позволяет `SessionFactory` участвовать в инфраструктуре управления сервера. **Без EJB это не даёт никаких преимуществ**, только добавляет сложности: нужно настраивать JCA-коннектор, регистрировать ресурсы в сервере, писать дескрипторы развёртывания.

**А как же JTA?** JTA нужен, только если у вас **распределённые транзакции** (несколько баз данных или БД + очередь сообщений). Spring может работать с JTA через `JtaTransactionManager` даже с локальной `SessionFactory`, без JNDI и JCA.

---
## 6.3 ИТОГ
| Подход                                                         | Когда использовать                                            |
| -------------------------------------------------------------- | ------------------------------------------------------------- |
| Локальная `SessionFactory` <br>+ `HibernateTransactionManager` | Одна БД, нет EJB. 95% случаев.                                |
| Локальная `SessionFactory` <br>+ `JtaTransactionManager`       | Несколько БД, но без EJB. Spring сам управляет JTA.           |
| JNDI `SessionFactory` через JCA                                | Только с EJB и CMT. Сложно, но даёт интеграцию с контейнером. |

# 7. Ложные предупреждения сервера приложений с Hibernate

**В некоторых JTA-окружениях** с очень строгими реализациями `XADataSource` 
(некоторые версии **WebLogic Server и WebSphere**), 
==когда Hibernate настроен без учёта JTA Transaction Manager== для этого окружения, **в журнале сервера приложений могут появляться ложные предупреждения или исключения.** 

Эти предупреждения или исключения указывают на то, что соединение, к которому осуществляется доступ, больше не действительно или доступ JDBC больше не действителен, возможно, потому что транзакция больше не активна. 

Например, вот реальное исключение от WebLogic:
```java
java.sql.SQLException: The transaction is no longer active - status: 'Committed'. No
further JDBC access is allowed within this transaction.
```

Ещё одна распространённая проблема — утечка соединений после JTA-транзакций, когда сессии Hibernate (и потенциально нижележащие JDBC-соединения) не закрываются должным образом.

> [!WARNING] СУТЬ ПРОБЛЕМЫ
> Проблема возникает из-за того, что ***Hibernate пытается закрыть соединение*** **после того**, **как JTA-транзакция уже завершена**. 
> 
> Решение — дать Hibernate знать о JTA-менеджере транзакций, чтобы он синхронизировался корректно и закрывал ресурсы до коммита, а не после.


**Решение:** ==Сделать Hibernate осведомлённым о JTA Transaction== Manager, с которым он синхронизируется (вместе со Spring). Есть два варианта:
1. **Передать ваш бин** `JtaTransactionManager` от Spring **в конфигурацию Hibernate.** 
   Самый простой способ — ==ссылка на бин в свойство== `jtaTransactionManager` для вашего бина `LocalSessionFactoryBean`. Spring затем делает соответствующие стратегии JTA доступными для Hibernate.
```java
@Bean
public LocalSessionFactoryBean mySessionFactory(
	DataSource ds, 
	JtaTransactionManager tm
) {
    
    LocalSessionFactoryBean factory = 
	    new LocalSessionFactoryBean();
    
    factory.setDataSource(ds);
    
    factory.setJtaTransactionManager(tm);
	
	return factory;
}
```

2. Вы также можете явно настроить связанные с JTA свойства Hibernate, в частности "hibernate.transaction.coordinator_class", "hibernate.connection.handling_mode" и потенциально "hibernate.transaction.jta.platform" в ваших "hibernateProperties" на `LocalSessionFactoryBean`.


**Последовательность событий БЕЗ ОСВЕДОМЛЕННОСТИ о JTA:**
1. JTA-транзакция коммитится.
    
2. `JtaTransactionManager` от Spring синхронизирован с JTA-транзакцией, поэтому он вызывается через обратный вызов `afterCompletion` менеджером JTA-транзакций.
    
3. Помимо прочих действий, эта синхронизация может инициировать обратный вызов от Spring к Hibernate через `afterTransactionCompletion` Hibernate (используется для очистки кэша Hibernate), за которым следует явный вызов `close()` для сессии Hibernate, что заставляет Hibernate попытаться выполнить `close()` для JDBC `Connection`.
    
4. В некоторых окружениях этот вызов `Connection.close()` затем вызывает предупреждение или ошибку, поскольку сервер приложений больше не считает `Connection` пригодным для использования, так как транзакция уже была зафиксирована.


**Последовательность событий С ОСВЕДОМЛЕННОСТЮ о JTA:**
1. JTA-транзакция готова к коммиту.
    
2. `JtaTransactionManager` от Spring синхронизирован с JTA-транзакцией, поэтому транзакция вызывается через обратный вызов `beforeCompletion` менеджером JTA-транзакций.
    
3. Spring осведомлён, что Hibernate сам синхронизирован с JTA-транзакцией, и ведёт себя иначе, чем в предыдущем сценарии. В частности, он согласовывает управление транзакционными ресурсами с Hibernate.
    
4. JTA-транзакция коммитится.
    
5. Hibernate синхронизирован с JTA-транзакцией, поэтому транзакция вызывается через обратный вызов `afterCompletion` менеджером JTA-транзакций и может правильно очистить свой кэш.
