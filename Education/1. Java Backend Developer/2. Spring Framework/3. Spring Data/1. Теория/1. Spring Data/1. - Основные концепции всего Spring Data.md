# 1. Абстракция репозитория (Repository)
 
> [!NOTE] **Центральная идея всего Spring Data.**
> `Repository` — это **интерфейс**, который вы объявляете, а *Spring во время выполнения автоматически создаёт его реализацию*.
>  
> Используется **ДЛЯ РАБОТЫ С ДАННЫМИ В БД**
> 
> Вы пишете только _что_ хотите получить, а не _как_.

```java
// Только объявление интерфейса — без реализации
public interface UserRepository extends CrudRepository<User, Long> {
    // Spring сам создаст имплементацию при запуске
}
```


> [!NOTE] Дженерики в репозиториии
> ```java
> CrudRepository<User, Long>
> ```
> `User` - это сущность, с которой работает репозиторий
> `Long` - это ТИП первичного ключа, указанный в сущности `User`


Вам не нужно писать класс с SQL/HQL/API запросами. Магия в том, что Spring анализирует имя интерфейса, его дженерики и методы, после чего генерирует прокси-объект, который делает всю работу.

# 2. Доменный тип / Сущность / Агрегат

> [!NOTE] Сущность/агрегат/доменный тип - сущность в БД
> Это класс, с которым работает репозиторий. 

В терминологии Spring Data используются три взаимозаменяемых понятия:
- **Domain type** — ваш класс данных (например, `User`, `Order`).
    
- **Entity** — то же самое, подчёркивает, что у объекта есть уникальный идентификатор.
    
- **Aggregate** — термин из DDD (Domain-Driven Design), обозначающий корневую сущность, через которую идёт доступ к связанным объектам.


> [!WARNING] 
> **Ключевое требование:** у сущности ==должен быть идентификатор== (поле `id`). 
> С АННОТАЦИЕЙ `@Id`
> Без него Spring Data не сможет корректно работать.

```java
// Пример доменного типа
public class User {
    @Id
    private Long id;        // Идентификатор — обязателен
    
    private String name;
    private String email;
    // геттеры/сеттеры
}
```

# 3. Иерархия интерфейсов репозиториев

## 3.1. Стандартная иерархия интерфейсов
Это ваш основной инструментарий. Все интерфейсы идут от общего к частному:

| Интерфейс                               | Назначение                                                           | Что добавляет                                                                         |
| --------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `Repository<T, ID>`                     | **Маркерный корень.** Самый базовый интерфейс. Не содержит методов.  | Определяет типы: T — сущность, ID — тип её идентификатора.                            |
| `CrudRepository<T, ID>`                 | **CRUD-операции.** Наследуется от `Repository`.                      | `save()`, `findById()`, `findAll()`, `count()`, `delete()`, `existsById()`.           |
| `ListCrudRepository<T, ID>`             | То же, что `CrudRepository`, но возвращает `List` вместо `Iterable`. | Удобнее для работы с коллекциями в Java.                                              |
| `PagingAndSortingRepository<T, ID>`     | **Пагинация и сортировка.** Расширяет возможности поиска.            | `findAll(Sort)` — сортировка, `findAll(Pageable)` — постраничный вывод.               |
| `ListPagingAndSortingRepository<T, ID>` | Аналог с возвратом `List`.                                           |                                                                                       |
|                                         |                                                                      |                                                                                       |
| `JpaRepository<T, ID>`                  | **Специфичный для JPA.** Наследует всё сверху + добавляет JPA-фичи.  | `flush()`, `saveAndFlush()`, `deleteAllInBatch()`, методы для работы с EntityManager. |
**Важное изменение с версии 3.0:** `PagingAndSortingRepository` больше **не наследует** `CrudRepository`. Если вам нужны и CRUD, и пагинация — расширяйте оба интерфейса одновременно.

### 3.1.1 **Как это использовать:**
1. Всегда наследуйте **один** из верхних интерфейсов.
    
2. Для большинства задач начинайте с `CrudRepository`.
    
3. Если нужна пагинация, переключайтесь на `PagingAndSortingRepository`.
    
4. При работе с JPA — `JpaRepository`, который включает всё сразу.

## 3.2 Создание экземпляров репозитория
*Вы объявили интерфейс*, **но кто и как создаст его реализацию?** 

> [!WARNING] Кто создает РЕАЛИЗАЦИЮ репозиториев?
> За это отвечает **специальная фабричная инфраструктура Spring Data.**

### 3.2.1 Включение сканирования репозиториев (Java Config) - `@EnableJpaRepositories`

Для активации ==используется аннотация, специфичная для модуля== (например, `@EnableJpaRepositories`). 
> Её вешают на любой `@Configuration`-класс.

**Важно:** По умолчанию Spring сканирует **пакет того класса, на котором стоит аннотация**. Это можно изменить через атрибуты.
```java
@Configuration
@EnableJpaRepositories("com.acme.repositories") // Явно указываем пакет для сканирования
class ApplicationConfiguration {
  @Bean
  EntityManagerFactory entityManagerFactory() {
    // ...
  }
}
```

**Атрибуты сканирования:**
- Можно использовать **property placeholders**: `@EnableJpaRepositories("${app.scan.packages}")`
    
- Можно использовать **Ant-паттерны**: `@EnableJpaRepositories("org.example.**")` — просканирует все подпакеты.

### 3.2.2 XML-конфигурация
Аналогичный функционал доступен в XML через элемент `<repositories>`:
```xml
<jpa:repositories base-package="com.acme.repositories" />
```

**Правило именования бинов:** Для интерфейса `UserRepository` будет создан бин с именем `userRepository`. Для вложенных интерфейсов имя включает имя внешнего типа.

### 3.2.3 Фильтрация интерфейсов
Если нужно создать бины не для всех найденных интерфейсов, используйте `includeFilters` и `excludeFilters`. Семантика такая же, как у компонентных фильтров Spring.

```java
@EnableJpaRepositories(
	basePackages = "com.acme.repositories",
    includeFilters = { 
	    @Filter(type = FilterType.REGEX, pattern = ".*SomeRepository") 
	},
    excludeFilters = { 
	    @Filter(
		    type = FilterType.REGEX, 
		    pattern = ".*SomeOtherRepository"
		) 
	})
class ApplicationConfiguration { ... }
```

Этот пример создаст бины только для интерфейсов, имя которых заканчивается на `SomeRepository`, и проигнорирует те, что заканчиваются на `SomeOtherRepository`.

### 3.2.4 Автономное использование (Standalone Usage)
Можно использовать фабрику репозиториев **вне Spring-контейнера** (например, в CDI-окружении). 

Для этого модули предоставляют свою `RepositoryFactory`, которую можно инстанцировать вручную:
```java
RepositoryFactorySupport factory = ...; // Создаём фабрику

UserRepository repository = factory.getRepository(UserRepository.class);
```

## 3.3. Специализированные варианты для разных технологий
Помимо стандартных, существуют варианты для специфичных сред:
- **Реактивные:** `ReactiveCrudRepository`, `ReactiveSortingRepository` (для Project Reactor) и `RxJava3CrudRepository`, `RxJava3SortingRepository` (для RxJava 3).
    
- **Kotlin:** `CoroutineCrudRepository`, `CoroutineSortingRepository` (для корутин).
    
- **Модульные расширения:** `JpaRepository` (JPA), `MongoRepository` (MongoDB) — наследуют CRUD и добавляют специфичные для технологии методы.

## 3.4. Выборочное раскрытие методов
Если вы не хотите наследовать все методы из стандартных интерфейсов, можно:

1. **Использовать аннотацию `@RepositoryDefinition`** на своём интерфейсе (без наследования от Spring Data).
    
2. **Скопировать сигнатуры** нужных методов из стандартных интерфейсов в свой.
    
3. **Создать собственный базовый интерфейс** с аннотацией `@NoRepositoryBean` и нужным набором методов.

```java
// Создаём свой базовый интерфейс с избранными методами
@NoRepositoryBean
interface MyBaseRepository<T, ID> extends Repository<T, ID> {
  Optional<T> findById(ID id);
  <S extends T> S save(S entity);
}

// Используем его для реальных репозиториев
interface UserRepository extends MyBaseRepository<User, Long> {
  User findByEmailAddress(EmailAddress emailAddress);
}
```
> **Правило:** 
> Аннотация `@NoRepositoryBean` **обязательна для всех промежуточных интерфейсов-шаблонов**. Она говорит Spring: «Не создавай бин для этого интерфейса, он не является конечным репозиторием».

# 4. Автоматическое создание запросов по имени метода (Defining Query Methods)

Это сердце Spring Data. Репозиторий может выполнить запрос двумя способами:
1. **Автоматически вывести** запрос из имени метода.
    
2. **Использовать явно объявленный запрос** (например, через аннотацию `@Query`).

## 4.1. Стратегии поиска запроса
Перед тем как выполнить метод, Spring должен решить, где взять запрос. Это определяется стратегией `queryLookupStrategy`:

- **`CREATE`** — Создаёт запрос ТОЛЬКО из имени метода. Если в имени ошибка — при запуске будет исключение.
    
- **`USE_DECLARED_QUERY`** — Ищет ТОЛЬКО явно объявленный запрос (через `@Query` или XML). Если не находит — ошибка при запуске.
    
- **`CREATE_IF_NOT_FOUND` (по умолчанию)** — Золотая середина. Сначала ищет `@Query`, а если его нет — строит запрос по имени метода. Именно эта стратегия используется, если вы ничего не меняли.

## 4.2. Создание запроса из имени метода
Главный механизм, позволяющий не писать SQL/JPQL. 

Работает так:
1. Имя метода разбивается на **подлежащее (subject)** и **сказуемое (predicate)**.
    
2. Подлежащее — это `find…By`, `exists…By`, `count…By`, `delete…By`. Всё между началом и `By` является описательным текстом, если не содержит специальных ключевых слов.
    
3. Сказуемое (после `By`) — это сами условия.

```java
// Примеры методов — реализацию Spring создаст сам
// ... where lastname = ?
List<User> findByLastname(String lastname); 

// ...  where email = ? and lastname = ?
List<User> findByEmailAndLastname(String email, String lastname); 

// ... select count(*) where lastname = ?
long countByLastname(String lastname);

// ... delete where lastname = ?
long deleteByLastname(String lastname); 

// ... where age between ? and ? 
List<User> findByAgeBetween(int start, int end);      
```

**Полный набор ключевых слов (помимо `By`, `And`, `Or`):**
- **Сравнения:** `Between`, `LessThan`, `GreaterThan`, `Like`, `StartingWith`, `EndingWith`, `Containing`.
    
- **Логические:** `True`, `False`, `IsNull`, `IsNotNull`, `NotNull`.
    
- **Модификаторы:** `IgnoreCase` (для одного поля: `findByLastnameIgnoreCase`), `AllIgnoreCase` (для всех полей: `findByLastnameAndFirstnameAllIgnoreCase`).
    
- **Ограничение результата:** `Distinct`, `First`/`Top` (с числом или без: `findTop5By...`).
    
- **Статическая сортировка:** `OrderBy` + поле + направление (`Asc`/`Desc`), например: `findByAgeOrderByLastnameDesc`.


**Важные нюансы:**
- Для удаления можно использовать и `deleteBy…`, и `removeBy…` — разницы нет.
    
- Возвращаемый тип может быть `List`, `Optional`, `Page`, `Slice` и другие.
    
- Параметры метода должны соответствовать порядку и типам в имени.

## 4.3. Зарезервированные методы и коллизии
Базовые методы, такие как `findById`, `existsById`, `deleteById`, являются зарезервированными. 

Они **всегда работают с полем-идентификатором сущности** (==тем, что помечено== `@Id`), даже если это поле называется не `id`.

**Проблема:** Если у сущности есть поле `id`, НЕ являющееся идентификатором (а идентификатором является, скажем, поле `pk`), то метод `findById` всё равно будет искать по `pk`.

```java
class User {
    @Id Long pk; // Настоящий ID
    Long id;      // Обычное поле
}
// findById(...) будет искать по pk!
// Чтобы искать по полю id, нужно добавить описательное слово:
// findUserById(...)
```

## 4.4. Навигация по свойствам (Property Expressions)
Можно обращаться к вложенным полям через точку:  
`findByAddressZipCode(ZipCode code)` → ищет по полю `person.address.zipCode`.

**Важное правило для разрешения неоднозначностей:** 
Если у `Person` есть поля `addressZip` и `address` (с вложенным `zipCode`), алгоритм может ошибиться. 

Чтобы явно указать путь, используйте **нижнее подчёркивание (`_`)**: 
`findByAddress_ZipCode(...)` — гарантированно пойдёт по пути `address.zipCode`.

## 4.5 Продвинутая работа с результатами запросов
Методы запроса, возвращающие несколько результатов, могут использовать стандартные Java `Iterable`, `List` и `Set`. 

Кроме того, мы **поддерживаем возвращаемый** *Spring Data* `Streamable`, п*ользовательское расширение* `Iterable`, а также типы коллекций, предоставляемые [Vavr]([Vavr — Object-functional library for Java](https://vavr.io/))

### 4.5.1 Streamable и кастомные обёртки
Можно возвращать не только `List` или `Set`, но и `Streamable<T>` — он удобен для объединения результатов и фильтрации без приведения типов.

```java
interface PersonRepository extends Repository<Person, Long> {
  Streamable<Person> findByFirstnameContaining(String firstname);
  Streamable<Person> findByLastnameContaining(String lastname);
}

Streamable<Person> result = 
	repository
		.findByFirstnameContaining("av")
		.and(
			repository.findByLastnameContaining("ea")
		);
```

**Создание своей обёртки:** Вы можете создать класс-обёртку (например, `Products`), который имплементирует `Streamable`, и использовать его как тип возврата. Главное — чтобы у него был конструктор или статический фабричный метод (`of`/`valueOf`), принимающий `Streamable`.

```java
class Product {
  MonetaryAmount getPrice() { … }
}

@RequiredArgsConstructor(staticName = "of")
class Products implements Streamable<Product> {

  private final Streamable<Product> streamable;

  public MonetaryAmount getTotal() {
    return streamable.stream()
      .map(Product::getPrice)
      .reduce(Money.of(0), MonetaryAmount::add);
  }


  @Override
  public Iterator<Product> iterator() {
    return streamable.iterator();
  }
}

interface ProductRepository implements Repository<Product, Long> {
  Products findAllByDescriptionContaining(String text);
}
```

### 4.5.2 Потоковая обработка (Stream)
Метод может возвращать `Stream<T>`. Это полезно для очень больших данных.  

```java
@Query("select u from User u")
Stream<User> findAllByCustomQueryAndStream();

Stream<User> readAllByFirstnameNotNull();

@Query("select u from User u")
Stream<User> streamAllPaged(Pageable pageable);
```

**Важно:** 
Такой `Stream` ==нужно обязательно закрывать== (через `try-with-resources`), иначе соединение с БД останется висеть.
```java
try (Stream<User> stream = repository.findAllByCustomQueryAndStream()) {
  stream.forEach(…);
}
```

### 4.5.3 Асинхронные запросы
С аннотацией `@Async` метод может вернуть `Future<T>` или `CompletableFuture<T>`. 

> [!WARNING] Запрос выполнится в отдельном потоке.  
> 

```java
@Async
Future<User> findByFirstname(String firstname);

@Async
CompletableFuture<User> findOneByFirstname(String firstname);
```

**Важно:** Это НЕ реактивное программирование, не путайте.

## 4.6 Пагинация, сортировка и лимитирование
Самый мощный инструмент для работы с большими наборами данных.

### 4.6.1 Основные параметры
Вы можете добавить в метод специальные параметры:
- **`Pageable`** — пагинация + сортировка.
    
- **`Sort`** — только сортировка.
    
- **`Limit`** — простое ограничение количества (аналог `Top`/`First`, но динамический).
```java
Page<User> findByLastname(String lastname, Pageable pageable);

Slice<User> findByLastname(String lastname, Pageable pageable);

List<User> findByLastname(String lastname, Sort sort);

List<User> findByLastname(String lastname, Sort sort, Limit limit);

List<User> findByLastname(String lastname, Pageable pageable);
```

> API, использующие Sort, Pageable и Limit, ожидают, что в методы будут переданы ненулевые значения.

### 4.6.2 Типы возврата для постраничных запросов
Выбор типа возврата определяет, будет ли выполняться дополнительный `count`-запрос:

- **`Page<T>`** — **Самый полный.** Возвращает данные + **общее количество страниц и записей**. 
  ==Для этого Spring Data выполняет дополнительный== `count`-запрос, что может быть дорого.
    
- **`Slice<T>`** — **Для бесконечной прокрутки.** Возвращает данные + флаг «есть ли следующая страница». Не делает `count`, работает быстрее.
    
- **`List<T>`** — **Просто список.** Игнорирует информацию о страницах, возвращая только данные за текущую страницу. Экономит один запрос.
```java
// Полная страница с общим количеством (2 запроса: data + count)
Page<User> findByLastname(String name, Pageable pageable);

// Только данные и флаг "есть ли ещё" (1 запрос, data + 1)
Slice<User> findByLastname(String name, Pageable pageable);

// Только список записей (1 запрос)
List<User> findByLastname(String name, Pageable pageable);
```

### 4.6.3 Сортировка и лимитирование
- **Статическая сортировка** жёстко задана в имени метода (`OrderBy`).
    
- **Динамическая сортировка** задаётся через параметр `Sort`:
```java
Sort sort = Sort.by("firstname").ascending()
	.and(Sort.by("lastname").descending());
```

- **Динамическое лимитирование** через параметр `Limit` или статическое через ключевые слова `First`/`Top` в имени метода.

### 4.6.4 Важные запреты
**Нельзя использовать в одном методе одновременно**:
- `Pageable` и `Sort` (сортировка уже есть внутри `Pageable`).
    
- `Pageable` и `Limit` (лимит уже есть внутри `Pageable`).

# 5. Идентификаторы и зарезервированные методы
Spring Data **автоматически распознаёт поле `id`** как идентификатор сущности, независимо от того, как оно называется в классе — `id`, `identifier`, `pk` и т.д.


Методы, которые ==всегда работают без дополнительного объявления==:
- `findById(ID id)` — найти по идентификатору
    
- `existsById(ID id)` — проверить существование
    
- `deleteById(ID id)` — удалить по идентификатору


> Если у вас поле идентификатора называется как-то нестандартно, и вы хотите писать запросы вроде `findByCustomKey`, используйте аннотацию `@Query` с собственным запросом — но это уже усложнение, которого лучше избегать.

# 6. Специфичные для модулей абстракции

Поверх общих интерфейсов каждый модуль Spring Data добавляет свои:
- **Spring Data JPA:** `JpaRepository` + поддержка `@Entity`, `@Query` с JPQL/SQL, `@Modifying`.
    
- **Spring Data JDBC:** `CrudRepository` + поддержка `@Query` с SQL, аннотации `@Id`, `@Table`.
    
- **Spring Data Redis:** работа с ключами и хэшами, `@RedisHash`.
    
- **Spring Data MongoDB:** `MongoRepository`, `@Document`, `@Query` с JSON.


Но **базовый принцип работы с репозиториями всегда одинаков** — танец от тех же `CrudRepository`, `findBy…`, `Pageable` и т.д.

## 6.1 Использование нескольких модулей Spring Data
Когда в приложении используется только один модуль (например, только JPA), Spring Data автоматически связывает все найденные репозитории с этим модулем. 

Проблемы начинаются, когда в classpath находятся два и более модуля (JPA + MongoDB).

### 6.1.1 Режим строгой конфигурации
При обнаружении нескольких модулей Spring Data переходит в **строгий режим** (strict configuration mode). 

Он определяет привязку репозитория к модулю по трём критериям:
1. **Наследование от модуль-специфичного интерфейса:**
```java
interface MyRepo extends JpaRepository<User, Long> { … } // → Привязка к JPA
```

2. **Аннотация на доменном классе:**
```java
@Entity
class Person { … }           // → Привязка к JPA
interface PersonRepo extends Repository<Person, Long> { … }

@Document
class UserLog { … }          // → Привязка к MongoDB
interface LogRepo extends Repository<UserLog, Long> { … }
```

3. **Сканирование разных базовых пакетов:**
```java
@EnableJpaRepositories(basePackages = "com.example.jpa")
@EnableMongoRepositories(basePackages = "com.example.mongo")
class Config { … }
```

### 6.1.2 Типичные ошибки
- **Неоднозначность:** Интерфейс наследует только общий `Repository`/`CrudRepository`, а доменный класс не имеет уточняющих аннотаций → Spring не знает, с каким модулем работать.
    
- **Смешанные аннотации:** Один класс помечен и JPA-аннотацией, и MongoDB-аннотацией → поведение непредсказуемо.

**Рекомендация:** Если доменный класс используется с разными БД, создайте отдельный класс для каждого модуля. ==Смешивать аннотации в одном классе нельзя.==
