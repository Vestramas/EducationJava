Эта глава объясняет, как Spring Data **превращает строки из базы данных в Java-объекты** и обратно.

Когда вы пишете:
```java
List<Person> people = repository.findAll();
```
Spring Data JDBC делает следующее:

1. Выполняет SQL-запрос (`SELECT * FROM person`).
    
2. Получает `ResultSet` (набор строк с колонками: `id`, `firstname`, `lastname`, `birthday`, `age`).
    
3. Для **каждой строки** создаёт Java-объект `Person`.
    
4. Заполняет поля этого объекта значениями из колонок.
    

Эта глава объясняет **шаги 3 и 4**: как именно Spring Data создаёт объект и заполняет его поля.


> [!NOTE] Где вы это используете?
> **НИГДЕ ЯВНО.** Это работает автоматически, когда вы объявляете репозиторий и вызываете `findAll()`, `findById(...)` или `save(...)`.

---

Этот раздел применим только к модулям Spring Data, которые **не используют объектный маппинг нижележащего хранилища** (в отличие от JPA). 

---
**В JPA это работает иначе.** JPA требует конструктор без аргументов, создаёт прокси, потом через сеттеры или рефлексию заполняет поля.

**Spring Data JDBC не использует прокси, ленивую загрузку и dirty checking.** 
Ему нужен другой подход:
- Он может вызывать **конструктор с аргументами** напрямую.
    
- Он может работать с **иммутабельными объектами** (поля `final`, нет сеттеров).
    
- Он поддерживает паттерн `withId(...)` для создания нового объекта вместо изменения существующего.

---

Основная ответственность объектного маппинга Spring Data — **создание экземпляров доменных объектов** и **отображение на них структур данных, специфичных для хранилища**. 

Это включает два фундаментальных шага:
1. Создание экземпляра с использованием одного из доступных конструкторов.
    
2. Наполнение экземпляра для материализации всех доступных свойств.
    

---
## 1.1. Создание объектов (Object Creation)

Spring Data автоматически определяет, какой конструктор использовать для создания объекта сущности. 

**Алгоритм разрешения:**
1. Если есть **один статический фабричный метод** с аннотацией `@PersistenceCreator` — используется он.
    
2. Если есть **один конструктор** — используется он.
    
3. Если конструкторов несколько и **ровно один** аннотирован `@PersistenceCreator` — используется он.
    
4. Если тип — Java **`Record`** — используется канонический конструктор.
    
5. Если есть **конструктор без аргументов** — используется он (остальные игнорируются).
    

**Разрешение значений:** Имена аргументов конструктора/фабричного метода должны совпадать с именами свойств сущности. 
Для этого необходима либо информация об именах параметров в class-файле, либо аннотация `@ConstructorProperties` на конструкторе. 

Разрешение значений можно настроить с помощью аннотации `@Value` из Spring Framework, используя SpEL-выражение, специфичное для хранилища.

---
### 1.1.1 Внутреннее устройство создания объектов (Object Creation Internals)

Для избежания накладных расходов на рефлексию Spring Data генерирует **фабричный класс во время выполнения**, напрямую вызывающий конструктор. 

Это даёт **≈10% прироста производительности**.

**Пример:** Для класса `Person(String firstname, String lastname)` генерируется:
```java
class PersonObjectInstantiator implements ObjectInstantiator {
  Object newInstance(Object... args) {
    return new Person((String) args[0], (String) args[1]);
  }
}
```

**Ограничения для оптимизации (иначе → рефлексия):**
- ***Класс не должен быть приватным***
    
- Не должен быть нестатическим внутренним классом
    
- Не должен быть CGLib-прокси
    
- ***Конструктор не должен быть приватным***

## 1.2. Наполнение свойств (Property Population)

После создания экземпляра Spring Data заполняет все оставшиеся persistent-свойства класса. 

**Идентификатор заполняется первым** (если ещё не был заполнен через конструктор), чтобы разрешить циклические ссылки. 

Затем заполняются все не-transient свойства, не заполненные конструктором. 

**Алгоритм:**
1. Если свойство **иммутабельно**, но имеет **`with…` метод** — используется этот метод для создания нового экземпляра с новым значением.
    
2. Если определён **доступ через свойства** (геттеры/сеттеры) — вызывается сеттер.
    
3. Если свойство **мутабельно** — значение устанавливается напрямую в поле.
    
4. Если свойство **иммутабельно** — используется конструктор персистентности для создания копии экземпляра.
    
5. **По умолчанию** — значение устанавливается напрямую в поле.

### 1.2.1 Внутреннее устройство наполнения свойств (Property Population Internals) 

Аналогично созданию объектов, Spring Data генерирует **accessor-классы** для взаимодействия с экземпляром во время выполнения. Это даёт **≈25% прироста производительности** над рефлексией.

**Пример сущности:**
```java
class Person {
  private final Long id;
  private String firstname;
  private @AccessType(Type.PROPERTY) String lastname;

  Person() { this.id = null; }
  Person(Long id, String firstname, String lastname) { /* присваивания */ }
  Person withId(Long id) { 
	  return new Person(id, this.firstname, this.lastname); 
  }
  
  void setLastname(String lastname) { 
	  this.lastname = lastname;
  }
}
```

**Сгенерированный Property Accessor:**
```java
class PersonPropertyAccessor implements PersistentPropertyAccessor {

	private static final MethodHandle firstname; // MethodHandle для поля
	
	private Person person; // Текущий мутабельный экземпляр
	
	public void setProperty(PersistentProperty property, Object value) {
		String name = property.getName();
	
		if ("firstname".equals(name)) {
			// Прямая установка поля через MethodHandle
			firstname.invoke(person, (String) value); 
			
		} else if ("id".equals(name)) {
			// Иммутабельное: создаётся новый Person
			this.person = person.withId((Long) value); 
			 
		} else if ("lastname".equals(name)) {
			// Доступ через свойство: вызов сеттера
			this.person.setLastname((String) value); 
		}
	}
}
```

**Как это работает:**
- Accessor держит **мутабельную ссылку** на объект (`person`), что позволяет мутировать иммутабельные свойства.
    
- По умолчанию используется **прямой доступ к полям**. Для `private` полей применяются `MethodHandles`.
    
- `withId(…)` создаёт **новый объект** — все последующие мутации происходят с новым экземпляром.
    
- `@AccessType(Type.PROPERTY)` позволяет вызывать сеттеры напрямую, без `MethodHandles`.
    

**Ограничения для оптимизации (иначе → рефлексия):**

- Типы не должны находиться в пакете по умолчанию или под пакетом `java`
    
- Типы и конструкторы должны быть `public`
    
- Внутренние классы должны быть `static`
    
- Java Runtime должен разрешать объявление классов в исходном `ClassLoader` (Java 9+ имеет ограничения)

---
### Пример сущности с пояснениями
```java
class Person {

  private final @Id Long id;                  // (1)
  private final String firstname, lastname;   // (2)
  private final LocalDate birthday;
  private final int age;                      // (3)

  private String comment;                     // (4)
  private @AccessType(Type.PROPERTY) String remarks;    // (5)

  static Person of(String firstname, String lastname, LocalDate birthday) { // (6)
    return new Person(null, firstname, lastname, birthday,
      Period.between(birthday, LocalDate.now()).getYears());
  }

  Person(Long id, String firstname, String lastname, LocalDate birthday, int age) { // (6)
    this.id = id;
    this.firstname = firstname;
    this.lastname = lastname;
    this.birthday = birthday;
    this.age = age;
  }

  Person withId(Long id) {                                                  // (1)
    return new Person(id, this.firstname, this.lastname, this.birthday, this.age);
  }

  void setRemarks(String remarks) {                                         // (5)
    this.remarks = remarks;
  }
}
```

1. **Идентификатор** — `final`, но установлен в `null` в конструкторе. Класс предоставляет `withId(…)` для установки ID после вставки. Оригинальный экземпляр остаётся неизменным (создаётся новый). Тот же паттерн применяется для других управляемых хранилищем свойств. `with…` метод опционален, так как конструктор персистентности фактически является конструктором копирования.
    
2. **`firstname` и `lastname`** — обычные иммутабельные свойства.
    
3. **`age`** — иммутабельное, но **производное** от `birthday`. Значение из БД будет иметь приоритет, так как Spring Data использует единственный объявленный конструктор. Конструктор принимает `age` как параметр (чтобы избежать ошибки при попытке установить иммутабельное поле без `with…` метода).
    
4. **`comment`** — мутабельное, заполняется напрямую в поле.
    
5. **`remarks`** — мутабельное, заполняется через сеттер (из-за `@AccessType(Type.PROPERTY)`).
    
6. Класс предоставляет **фабричный метод** и **конструктор**. Идея: использовать фабричные методы вместо дополнительных конструкторов, чтобы избежать неоднозначности и необходимости `@PersistenceCreator`. Значения по умолчанию обрабатываются в фабричном методе. Чтобы Spring Data использовал фабричный метод, аннотируйте его `@PersistenceCreator`.

## 1.3. Общие рекомендации

- **Используйте иммутабельные объекты** — их легко создавать (достаточно вызвать конструктор), они не замусорены сеттерами. Если сеттеры нужны, делайте их package-protected. Материализация только через конструктор до **30% быстрее**.
    
- **Предоставляйте конструктор со всеми аргументами** — даже для мутабельных сущностей. Это позволяет пропустить этап наполнения свойств для оптимальной производительности.
    
- **Используйте фабричные методы вместо перегруженных конструкторов** — чтобы избежать `@PersistenceCreator`. Фабричные методы скрывают детали (например, авто-генерируемые ID).
    
- **Соблюдайте ограничения** для использования генерируемых классов instantiator и property accessor.
    
- **Для генерируемых ID используйте `final` поле** + all-args конструктор (предпочтительно) или `with…` метод.
    
- **Используйте Lombok** — `@AllArgsConstructor` избавляет от шаблонного кода присваивания полей в конструкторе.

## 1.4. Переопределение свойств (Overriding Properties)
Java позволяет подклассу определять свойство с тем же именем, что и в суперклассе. Если типы **несовместимы** — свойство суперкласса **пропускается** (считается транзиентным). Spring Data **рекомендует использовать разные имена свойств**.

```java
public class SuperType {
   private CharSequence field;
   public SuperType(CharSequence field) { this.field = field; }
   public CharSequence getField() { return this.field; }
   public void setField(CharSequence field) { this.field = field; }
}

public class SubType extends SuperType {
   private String field;
   public SubType(String field) { 
	   super(field); this.field = field; 
   }
	
   @Override 
   public String getField() { 
		return this.field; 
   }
   
   public void setField(String field) { 
	   this.field = field; super.setField(field); 
   } // опционально
}
```

**Что нужно учитывать:**

- Какое свойство сохранять? (по умолчанию — все). Исключить можно через `@Transient`.
    
- Как представить свойства в хранилище? Одинаковые имена колонок для разных значений приведут к повреждению данных — нужно явно указать имя колонки хотя бы для одного из свойств.
    
- `@AccessType(PROPERTY)` **не может использоваться**, так как супер-свойство нельзя установить без дополнительных предположений о реализации сеттера.

## 1.5. Поддержка Kotlin

**Создание объектов в Kotlin:** Алгоритм аналогичен Java, но с приоритетом для data-классов (используется первичный конструктор). Поддерживаются параметры по умолчанию (default values) — если хранилище не предоставляет значение, применяется значение по умолчанию из Kotlin.

```kotlin
data class Person(val id: String, val name: String)                           // Первичный конструктор

data class Person(var id: String, val name: String) {
    @PersistenceCreator constructor(id: String) : this(id, "unknown")        // Предпочтение конструктора
}

data class Person(var id: String, val name: String = "unknown")              // Значение по умолчанию
```

**Наполнение свойств:** Data-классы иммутабельны. Kotlin генерирует метод `copy(…)` для создания новых экземпляров с изменёнными свойствами.

**Value Classes:** Поддерживаются. Не-nullable свойства с непримитивными value-типами сплющиваются (flattened) в скомпилированном классе до типа значения.

**Переопределение свойств в Kotlin:** Аналогично Java, геттеры/сеттеры в подклассе работают только со своим полем. Рекомендуются разные имена свойств.
