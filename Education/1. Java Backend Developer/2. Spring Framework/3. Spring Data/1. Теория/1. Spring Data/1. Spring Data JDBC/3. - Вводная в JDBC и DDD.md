# 1. Почему Spring Data JDBC?

**JPA делает много вещей, чтобы помочь разработчику**: отслеживает изменения сущностей, выполняет ленивую загрузку, позволяет маппить широкий спектр объектных конструкций на разнообразные схемы баз данных. 

Это *делает многие вещи очень лёгкими*, **но часто приводит к путанице — почему JPA делает определённую вещь**, и концептуально простые вещи становятся довольно сложными.


**Spring Data JDBC стремится быть гораздо проще концептуально**, следуя трём ключевым принципам:

1. **Загрузка сущности = выполнение SQL.** 
   После завершения загрузки у вас есть **полностью загруженная сущность**. 
   ==Никакой ленивой загрузки, никакого кэширования.==
    
2. **Сохранение сущности = немедленная запись в БД.** 
   Сохранили — попало в БД. Не сохранили — не попало. 
   **Никакого dirty tracking и никакой сессии** (persistence context).
    
3. **Простая модель маппинга сущностей на таблицы.** 
   Рассчитана на простые случаи. 
   Если не подходит — нужно писать свою стратегию. 
   Spring Data JDBC предлагает лишь **очень ограниченную поддержку** настройки стратегии аннотациями.

# 2. Domain-Driven Design и реляционные базы данных

Все модули Spring Data вдохновлены концепциями **«репозиторий»**, **«агрегат»** и **«корень агрегата»** из Domain-Driven Design. 

Для Spring Data JDBC они **особенно важны**, поскольку в некоторой степени противоречат обычной практике работы с реляционными базами данных.

---
### Проблема
В JPA вы можете свободно ссылаться на любые сущности из любых других сущностей через `@OneToMany`, `@ManyToOne` и ленивую загрузку. 

В JDBC этого **нет**. 
Возникает вопрос: ==как тогда работать со связанными данными==?

---
### Решение: Агрегаты
Spring Data JDBC вводит жёсткое правило:

> **Один репозиторий = один агрегат.** Все операции (сохранение, удаление, загрузка) происходят **только через корень агрегата**.

---
### Практический пример
У вас есть **Заказ** и **Позиции заказа**:
```java
// Агрегат
class Order {              // ← Корень агрегата
  @Id Long id;
  String customerName;
  List<OrderItem> items    // ← Часть агрегата (не-корневые сущности)
  Address billingAddress;  // ← Часть агрегата (встроенный объект)
}

class OrderItem {
  int quantity;
  String productName;
}

// НЕ часть агрегата — отдельный агрегат
class Product {                        // ← Другой агрегат
  @Id Long id;
  String name;
  BigDecimal price;
}
```

**Правила работы:**
1. Вы создаёте **один репозиторий** — `OrderRepository` (для корня агрегата). **Репозитория для `OrderItem` не существует.**
    
2. Когда вы сохраняете заказ: `orderRepository.save(order)` — Spring Data JDBC **автоматически** сохранит и сам `Order`, и все его `OrderItem`, и `Address`.
    
3. Когда вы загружаете заказ: `orderRepository.findById(id)` — вернётся **полностью загруженный** `Order` со всеми позициями.
    
4. Когда вы удаляете заказ: `orderRepository.delete(order)` — удалятся и все `OrderItem`.

**Так нельзя:** `order.getItems().get(0).setProduct(productRepository.findById(...))` — прямая ссылка на другой агрегат. 

Вместо этого храните **только ID**:
```java
class OrderItem {
  int quantity;
  Long productId;    // ← Только ID продукта, а не сам Product
}
```

## СУТЬ
### Как в JPA (с Hibernate)
1. Hibernate загружает заказ с 3 позициями. Все 4 объекта находятся в **Persistent Context** (кэше первого уровня).
    
2. Вы удаляете одну позицию из списка: `order.getItems().remove(0)`.
    
3. При коммите транзакции Hibernate **сравнивает текущее состояние** объектов с тем, что было при загрузке (это и есть **dirty checking**).
    
4. Он видит: одна позиция удалена из коллекции → выполняет один `DELETE` для этой позиции.
    
5. Остальные две позиции **не трогаются** — для них не выполняется ни DELETE, ни INSERT.
    

**Итог JPA:** 1 DELETE, больше ничего.

---
### Как в Spring Data JDBC

1. JDBC загружает заказ с 3 позициями. Никакого Persistent Context'а нет.
    
2. Вы удаляете одну позицию: `order.getItems().remove(0)`.
    
3. При вызове `save()` JDBC **не знает**, что изменилось. Он просто видит текущий список из 2 позиций.
    
4. Он действует по принципу «синхронизировать список в БД со списком в объекте»:
    
    - Удаляет **все 3 старые позиции** (`DELETE FROM order_item WHERE order_id = ?`).
        
    - Вставляет **2 текущие позиции** заново (`INSERT INTO order_item ...`).
    

**Итог JDBC:** 1 массовый DELETE + 2 INSERT.

## 2.1. Агрегат (Aggregate)
**Агрегат** — это группа сущностей, согласованность которой гарантируется при атомарных изменениях. 

Классический пример: **Заказ (Order) с Позициями заказа (OrderItems)**. Свойство `numberOfItems` на `Order` остаётся согласованным с фактическим количеством `OrderItems` при любых изменениях.

> **Важное правило:** 
> Ссылки **между агрегатами** не гарантируют согласованности в каждый момент времени, но гарантируют **согласованность в конечном счёте** (eventually consistent).

## 2.2. Корень агрегата (Aggregate Root)
У каждого агрегата есть **ровно один корень** — одна из сущностей агрегата. **Все манипуляции с агрегатом выполняются только через методы этого корня.** Это и есть те атомарные изменения, о которых говорилось выше.

## 2.3. Репозиторий и Spring Data JDBC
- **Репозиторий** — это абстракция над постоянным хранилищем, которая выглядит как коллекция всех агрегатов определённого типа.
    
- В Spring Data это означает: **один `Repository` на один корень агрегата**.
    
- Для Spring Data JDBC это также означает: **все сущности, достижимые из корня агрегата, считаются частью этого агрегата**

Spring Data JDBC **предполагает**, что только у агрегата есть внешний ключ к таблице, хранящей не-корневые сущности агрегата, и никакая другая сущность не указывает на не-корневые сущности.

---
## 2.4. Важная особенность реализации
В текущей реализации **сущности, на которые ссылается корень агрегата, удаляются и пересоздаются Spring Data JDBC**. 

Вы можете переопределить методы репозитория своими реализациями, соответствующими вашему стилю работы и проектирования базы данных.

### Что означает «сущности удаляются и пересоздаются»?
У вас есть **один заказ**, у которого 3 позиции:
```java
Order order = new Order();

order.setItems(List.of(
  new OrderItem("Товар 1", 2),
  new OrderItem("Товар 2", 1),
  new OrderItem("Товар 3", 5)
));
```

После `orderRepository.save(order)` в БД: один `Order` и три `OrderItem`

Теперь вы загрузили заказ, удалили одну позицию и сохранили:
```java
Order loaded = orderRepository.findById(1L);

loaded.getItems().remove(0);  // Удалили первый товар

orderRepository.save(loaded);
```

Spring Data JDBC делает так:

1. **Удаляет ВСЕ 3 позиции** из таблицы `order_item`, где `order_id = 1`.
    
2. **Вставляет 2 новые позиции** (те, что остались в списке).
    

**Третья позиция исчезает**, но это ожидаемо — вы её удалили из списка. Первые две удаляются и вставляются заново (с теми же данными, но новыми техническими ID, если они были автосгенерированы).
