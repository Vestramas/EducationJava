# 1. Сохранение сущностей
## 1. Сохранение агрегата
Сохранение агрегата выполняется методом `CrudRepository.save(…)`. 

Поведение зависит от того, **новый агрегат или существующий**:
- **Если агрегат новый:** выполняется `INSERT` для корня агрегата, затем `INSERT` для всех прямо или косвенно связанных сущностей.
    
- **Если агрегат НЕ новый:** 
	- ==все СВЯЗАННЫЕ СУЩНОСТИ УДАЛЯЮТСЯ==, 
	- **корень** агрегата **обновляется** (`UPDATE`), 
	- затем **все связанные сущности** **вставляются заново**. Является ли экземпляр новым — часть состояния экземпляра.


**Недостатки подхода:** Если изменились лишь некоторые из связанных сущностей, удаление и вставка **расточительны**.

Spring Data JDBC **не знает предыдущего состояния агрегата**, поэтому любой процесс обновления всегда должен взять то, что находится в базе данных, и привести это к состоянию сущности, переданной в метод `save`.

---
## 2. Загрузка агрегатов

Spring Data JDBC предлагает **два способа загрузки агрегатов**:

### **Способ 1: Традиционный (до версии 3.2 — единственный).** 
Каждый запрос загружает корни агрегатов. Если корень агрегата ссылается на другие сущности, они загружаются **отдельными запросами**.

### **Способ 2: Single Query Loading (с версии 3.2).** 
***Произвольное количество агрегатов*** может быть полностью загружено **одним SQL-запросом**. 

Значительно эффективнее для сложных агрегатов из многих сущностей.

**Ограничения Single Query Loading (экспериментальная функция):**
- Агрегат **не должен иметь вложенных коллекций** (включая `Map`) — планируется убрать.
    
- Агрегат **не должен использовать `AggregateReference` или встроенные сущности** — планируется убрать.
    
- Диалект БД **должен поддерживать** аналитические (оконные) функции. Из поддерживаемых диалектов **H2 и HSQL не поддерживают**.
    
- Работает только для find-методов `CrudRepository`, **не для производных и аннотированных запросов** — планируется убрать.
    
- Требует включения через `jdbcMappingContext.setSingleQueryLoadingEnabled(true)`.


Если любое условие не выполняется — Spring Data JDBC возвращается к стандартному подходу.

## 3. Генерация ID

Spring Data использует свойства идентификатора для идентификации сущностей. 

ID сущности должен быть аннотирован `@Id`
- **Auto-increment колонка:** Сгенерированное значение устанавливается в сущности после вставки в БД.
    
- **`@Sequence`:** **Если дополнительно аннотировать идентификатор** `@Sequence`, ***для получения значений будет использоваться последовательность БД*** (если диалект поддерживает последовательности).
    
- **По умолчанию:** Spring Data **не пытается вставлять значения** в колонки идентификаторов, когда сущность новая и значение идентификатора равно начальному (`0` для примитивных типов, `null` для обёрток).


> **Важное ограничение:** 
> После сохранения сущность **не должна больше считаться новой**. Является ли сущность новой — часть её состояния. 
> 
> С auto-increment колонками это происходит автоматически (ID устанавливается Spring Data).

## 4. Template API - альтернатива репозиториям

Альтернатива репозиториям — **`JdbcAggregateTemplate`**, более прямой способ загрузки и сохранения сущностей. 

Репозитории в значительной степени используют `JdbcAggregateTemplate` для реализации своей функциональности.

**Доступ к `JdbcAggregateTemplate`:** 
Используется как Spring-бин. Если приложение настроено для Spring Data JDBC, можно объявить зависимость на `JdbcAggregateTemplate` в любом Spring-бине.

**Сохранение:** 
Три типа методов: 
- `save`, 
- `insert`, 
- `update`. 
Каждый в двух вариантах: для одного агрегата и с суффиксом `All` для `Iterable`. 
- `save` делает то же, что и одноимённый метод репозитория. 
- `insert` и `update` **пропускают проверку новизны** и предполагают новый или существующий агрегат.

**Запросы с `Query` и `Criteria`:**
```java
template.findOne(query(where("name").is("Gandalf")), Person.class);
```

`Query` определяет список колонок, where-клаузу (через `CriteriaDefinition`), limit и offset.

**Методы класса `Criteria` (соответствуют SQL-операторам):**

|Метод|Оператор SQL|
|---|---|
|`and(String column)`|Добавляет цепочечный `Criteria` с указанным свойством|
|`or(String column)`|Добавляет цепочечный `Criteria` с указанным свойством|
|`greaterThan(Object o)`|`>`|
|`greaterThanOrEquals(Object o)`|`>=`|
|`in(Object… o)` / `in(Collection<?>)`|`IN`|
|`is(Object o)`|`=`|
|`isNull()`|`IS NULL`|
|`isNotNull()`|`IS NOT NULL`|
|`lessThan(Object o)`|`<`|
|`lessThanOrEquals(Object o)`|`<=`|
|`like(Object o)`|`LIKE` (без обработки escape-символов)|
|`not(Object o)`|`!=`|
|`notIn(Object… o)` / `notIn(Collection<?>)`|`NOT IN`|

## 5. Оптимистичная блокировка
Spring Data поддерживает оптимистичную блокировку через **числовой атрибут с аннотацией `@Version`** на корне агрегата. 

При сохранении агрегата с таким атрибутом:
1. `UPDATE` для корня агрегата содержит where-клаузу, проверяющую, что версия в БД не изменилась.
    
2. Если версия изменилась — выбрасывается `OptimisticLockingFailureException`.
    
3. Атрибут версии **увеличивается** и в сущности, и в БД, чтобы конкурентное действие заметило изменение.

Этот процесс применяется и к вставке новых агрегатов: `null` или `0` версия указывает на новый экземпляр, а увеличенное значение после вставки помечает экземпляр как не новый. 

При удалении проверка версии также применяется, но версия **не увеличивается**.

# 2. Поддержка последовательностей (Sequence Support)

Свойства первичного ключа (аннотированные `@Id`) могут быть также аннотированы **`@Sequence`**. 

*Наличие этой аннотации указывает*, что начальное значение свойства должно быть получено из **последовательности базы данных** в момент сохранения объекта.

> **Важно:** 
> Возможность генерации последовательности зависит от используемого диалекта БД. 
> 
> **Без `@Sequence`** предполагается, что значение для колонки автоматически генерируется базой данных при вставке строки.

## 1. Пример сущности
```java
@Table
class MyEntity {

    @Id
    @Sequence(
        sequence = "my_seq",
        schema = "public"
    )
    private Long id;

    // …
}
```

## 2. Как это работает
При сохранении этой сущности, **перед SQL `INSERT`**, Spring Data выполнит дополнительный `SELECT` для получения следующего значения из последовательности. 

Например, для PostgreSQL:
```sql
SELECT nextval('public.my_seq');
```

Полученное значение идентификатора включается в `VALUES` при вставке:
```sql
INSERT INTO "my_entity"("id", "name") VALUES(?, ?);
```

> **Рекомендация:** 
> - Получение значения из последовательности 
> - и сам `INSERT` 
> две отдельные операции. 
> 
> **Настоятельно рекомендуется** ==выполнять их внутри транзакции для обеспечения атомарности.==
