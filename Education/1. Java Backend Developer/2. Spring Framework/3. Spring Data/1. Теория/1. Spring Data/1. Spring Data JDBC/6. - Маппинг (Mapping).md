Богатая поддержка маппинга обеспечивается классом **`MappingJdbcConverter`**. 

Он имеет богатую модель метаданных, **позволяющую отображать доменные объекты на строки данных**. 

Метаданные маппинга заполняются с помощью аннотаций на доменных объектах, но инфраструктура не ограничивается аннотациями — `MappingJdbcConverter` также позволяет маппить объекты на строки без дополнительных метаданных, следуя набору соглашений.

## 1. Маппинг на основе соглашений (Convention-based Mapping)
Если вы вообще не ставите аннотаций, `MappingJdbcConverter` сам догадается, как маппить объекты на таблицы:

- **Класс → Таблица:** Берется короткое имя класса и переводится в верхний регистр с подчёркиваниями. 
  Например, `SavingsAccount` → `SAVINGS_ACCOUNT`. 
  Аналогично для полей: `firstName` → `FIRST_NAME`. 
  
  Это поведение настраивается через `NamingStrategy`.
    
- **Поля, а не геттеры/сеттеры:** В отличие от JPA, Spring Data JDBC смотрит **на поля объекта**, а не на JavaBean-свойства. Геттеры и сеттеры вообще не используются для маппинга.
    
- **Конструктор:** Если у класса есть ровно один конструктор с параметрами, имена которых совпадают с именами колонок — используется он. Если нет — используется конструктор без аргументов.
    
- **Кавычки:** По умолчанию имена таблиц и колонок в SQL идут без кавычек. Это можно изменить флагом `setForceQuote(true)` — тогда все имена будут в кавычках и станут чувствительны к регистру.

## 2. Поддерживаемые типы в сущностях
- Все примитивные типы и их обёртки (`int`, `Integer`, `float`, `Float` и т.д.).
    
- `Enum` — маппятся на имена.
    
- `String`.
    
- `java.util.Date`, `java.time.LocalDate`, `LocalDateTime`, `LocalTime`.
    
- Массивы и коллекции вышеуказанных типов (если БД поддерживает массивы).
    
- Всё, что принимает драйвер БД.
    
- **Ссылки на другие сущности:** считаются **one-to-one** отношением или встроенным типом. 
  Для one-to-one `id` опционален. 
  Таблица связанной сущности должна иметь дополнительную колонку (см. Back References). Встроенные сущности не требуют `id`.
    
- **`Set<некая сущность>`:** **one-to-many**. Дополнительная колонка в таблице связанной сущности (Back References).
    
- **`Map<простой тип, некая сущность>`:** квалифицированный one-to-many. Две дополнительные колонки: для внешнего ключа и для ключа Map (с суффиксом `_key`).
    
- **`List<некая сущность>`:** маппится как `Map<Integer, некая сущность>`. Те же дополнительные колонки.
    
- **Типы с зарегистрированными кастомными конвертерами.**
    

**Настройка Back References для `List`, `Set`, `Map`:** через `NamingStrategy.getReverseColumnName(...)` и `NamingStrategy.getKeyColumn(...)` или аннотацией `@MappedCollection(idColumn="...", keyColumn="...")`. Для `Set` указание `keyColumn` не имеет эффекта.

## 3. Обзор аннотаций маппинга

| Аннотация             | Описание                                                                                                                                                                                                                  |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `@Embedded`           | **Свойство маппится в таблицу** *родительской сущности* (а не в отдельную таблицу). <br><br>Можно указать префикс колонок и поведение при `null` (`onEmpty`). <br><br>Может комбинироваться с `@Id` для составного ключа. |
| `@Id`                 | **Метка первичного ключа**. Может комбинироваться с `@Embedded`.                                                                                                                                                          |
| `@InsertOnlyProperty` | Свойство записывается только при вставке, не обновляется. Только для корня агрегата.                                                                                                                                      |
| `@MappedCollection`   | Настройка маппинга коллекции или одиночной вложенной сущности. `idColumn` — колонка внешнего ключа, `keyColumn` — колонка для индекса/ключа.                                                                              |
| `@Sequence`           | Указывает последовательность БД для генерации значений.                                                                                                                                                                   |
| `@Table`              | Метка класса как кандидата для маппинга в БД. Можно указать имя таблицы.                                                                                                                                                  |
| `@Transient`          | Исключает поле из сохранения в БД. Не может использоваться в persistence-конструкторе.                                                                                                                                    |
| `@PersistenceCreator` | Отмечает конструктор или фабричный метод для создания объекта из БД.                                                                                                                                                      |
| `@Value`              | Spring Framework. В конструкторе позволяет использовать SpEL для преобразования значения колонки. `#root.myProperty` ссылается на корень строки.                                                                          |
| `@Column`             | Имя колонки в БД, отличное от имени поля. Имена с `@Column` **всегда берутся в кавычки** в SQL (чувствительны к регистру, можно использовать спецсимволы — но не рекомендуется).                                          |
| `@Version`            | Оптимистичная блокировка. `null`/`0` = новая сущность. Увеличивается автоматически при каждом обновлении.                                                                                                                 |

## 4. Ссылочные сущности (Referenced Entities)

**Вложенные сущности (часть агрегата):**
- `Set<SubEntity>` — связь one-to-many внутри одного агрегата.
    
- `List<SubEntity>` — тоже one-to-many, но с сохранением порядка (добавляется колонка с индексом).
    
- `Map<String, SubEntity>` — one-to-many с ключом.
    
- Одиночная ссылка на другую сущность — one-to-one внутри агрегата.


> **Важное правило:** 
> Все эти ссылки — **часть одного агрегата**. 
> 
> При удалении ссылки связанная сущность удаляется из БД. 
> 
> Не бывает ссылок n-1 или n-m внутри агрегата.


**Ссылки на другие агрегаты:**  
Если сущность относится к другому агрегату, используйте **`AggregateReference`** — это обёртка над ID, которая явно показывает: «это ссылка на другой агрегат, я не владею этим объектом»:
```java
class Person {
    @Id long id;
    AggregateReference<Person, Long> bestFriend;
}

// Использование:
p1.bestFriend = AggregateReference.to(p2.id);
```

---
## 5. Back References

Все ссылки в агрегате создают связь внешнего ключа в обратном направлении. 

По умолчанию имя колонки внешнего ключа — **имя таблицы ссылающейся сущности**. 
- Для `@Embedded` id: колонки именуются как `<имя_таблицы>_<имя_колонки>`. 
- Для `List`/`Map` добавляется суффикс `_KEY` для хранения индекса/ключа.

## 6. Стратегия именования (Naming Strategy)

По умолчанию Spring Data использует `NamingStrategy` для определения имён таблиц, колонок и схем, преобразуя в **snake_case** (`firstName` → `first_name`). 

Можно настроить, предоставив свой бин `NamingStrategy`.

Когда у вас есть вложенная сущность (например, `OrderItem` внутри `Order`), в таблице `ORDER_ITEM` должна быть колонка, указывающая на родительский заказ. Spring Data JDBC сам ожидает такую колонку и сам её заполняет.

**По умолчанию** имя этой колонки — имя таблицы родительской сущности. Например, для `Order` → `OrderItem` колонка будет называться `ORDER`. Это можно изменить через `NamingStrategy` или аннотацию `@MappedCollection(idColumn = "...")`.

## 7. Переопределение имён таблиц и колонок (АННОТАЦИИ)
```java
@Table("CUSTOM_TABLE_NAME")
class MyEntity {
    @Id Integer id;

    @Column("CUSTOM_COLUMN_NAME")
    String name;

    @MappedCollection(idColumn = "CUSTOM_MY_ENTITY_ID_COLUMN_NAME")
    Set<MySubEntity> subEntities;

    @MappedCollection(
	    idColumn = "CUSTOM_COLUMN_NAME", 
	    keyColumn = "CUSTOM_KEY_COLUMN_NAME")
    List<MySubEntity> items;
}
```

Если соглашения по именованию не подходят, **используйте аннотации:**
- **`@Table("CUSTOM_TABLE_NAME")`** — переопределяет имя таблицы.
    
- **`@Column("CUSTOM_COLUMN_NAME")`** — переопределяет имя колонки (чувствительно к регистру, всегда в кавычках в SQL).
    
- **`@MappedCollection(idColumn = "...", keyColumn = "...")`** — переопределяет имена колонок внешнего ключа и ключа коллекции.

## 8. Встроенные сущности (Embedded Entities)

Позволяют иметь value objects в Java-модели, даже если в БД одна таблица.
```java
class MyEntity {
    @Id Integer id;

    @Embedded(onEmpty = USE_NULL)  // или USE_EMPTY
    EmbeddedEntity embeddedEntity;
}

class EmbeddedEntity {
    String name;
}
```

- `USE_NULL`: если все колонки `null` → свойство `null`.
    
- `USE_EMPTY`: создаётся экземпляр с потенциально `null`-полями.
    
- Сокращения: `@Embedded.Nullable`, `@Embedded.Empty`.
    
- Если у встроенной сущности есть `Collection` или `Map` — она никогда не будет `null`.


**Составные ID (Embedded Ids):**
```java
@Table("PERSON_WITH_COMPOSITE_ID")
record Person(
    @Id @Embedded.Nullable 
    Name pk,
    
    String nickName,
    Integer age
) {}

record Name(String first, String last) {}
```

```sql
CREATE TABLE PERSON_WITH_COMPOSITE_ID (
    FIRST VARCHAR(100),
    LAST VARCHAR(100),
    NICK_NAME VARCHAR(100),
    AGE INT,
    PRIMARY KEY (FIRST, LAST)
);
```

## 9. Read Only и Insert Only свойства
- **`@ReadOnlyProperty`:** ***Читаются из БД***, но **не записываются**. После вставки сущность нужно перезагрузить явно.
    
- **`@InsertOnlyProperty`:** ***Записываются*** **только при вставке**, при обновлении игнорируются. **Только для корня агрегата.**

## 10. Кастомное создание объектов (`@PersistenceConstructor`)

Порядок разрешения параметров конструктора:

1. `@Value` на параметре → вычисляется SpEL-выражение.
    
2. Имя параметра совпадает с именем поля → используется значение колонки (требуется информация об именах в `.class`).
    
3. Иначе → `MappingException`.

```java
class OrderItem {
    private @Id final String id;
    private final int quantity;
    private final double unitPrice;

    OrderItem(String id, int quantity, double unitPrice) {
        this.id = id;
        this.quantity = quantity;
        this.unitPrice = unitPrice;
    }
}
```

## 11. Явные конвертеры
Конвертеры применяются на уровне свойств (одиночное значение ↔ одиночное значение). Преобразование сложных объектов в несколько колонок **не поддерживается**.

**`@WritingConverter`:**
```java
@WritingConverter
public class BooleanToStringConverter implements Converter<Boolean, String> {
    public String convert(Boolean source) {
        return source != null && source ? "T" : "F";
    }
}
```


**`@ReadingConverter`:**
```java
@ReadingConverter
public class StringToBooleanConverter implements Converter<String, Boolean> {
    public Boolean convert(String source) {
        return source != null && source.equalsIgnoreCase("T") ? Boolean.TRUE : Boolean.FALSE;
    }
}
```


**Регистрация конвертеров:**
```java
class MyJdbcConfiguration extends AbstractJdbcConfiguration {
    @Override
    protected List<?> userConverters() {
        return Arrays.asList(new BooleanToStringConverter(), new StringToBooleanConverter());
    }
}
```

## 12. `JdbcValue`

Для указания JDBC-типа используйте `JdbcValue` в кастомном write-конвертере:
```java
return JdbcValue.of("T", JDBCType.VARCHAR);
```
