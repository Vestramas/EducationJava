Большинство операций доступа к данным, вызываемых на репозитории, приводят к выполнению запроса к базе данных. 

Определение такого запроса — это объявление метода на интерфейсе репозитория.

**Примеры методов запросов:**
```java
interface PersonRepository extends PagingAndSortingRepository<Person, String> {

    List<Person> findByFirstname(String firstname); // (1)

    List<Person> findByFirstnameOrderByLastname(
	    String firstname, 
	    Pageable pageable); // (2)

    Slice<Person> findByLastname(
	    String lastname, 
	    Pageable pageable); // (3)

    Page<Person> findByLastname(
	    String lastname, 
	    Pageable pageable); // (4)

    Person findByFirstnameAndLastname(
	    String firstname, 
	    String lastname); // (5)

    Person findFirstByLastname(String lastname); // (6)

    @Query("SELECT * FROM person WHERE lastname = :lastname")
    List<Person> findByLastname(String lastname); // (7)

    @Query("SELECT * FROM person WHERE lastname = :lastname")
    Stream<Person> streamByLastname(String lastname); // (8)

    @Query("SELECT * FROM person WHERE username = :#{ principal?.username }")
    Person findActiveUser(); // (9)
}
```

1. Запрос для всех людей с указанным именем. Запрос выводится из имени метода (`AND`, `OR`). Результат: `SELECT … FROM person WHERE firstname = :firstname`.
    
2. Использование `Pageable` для передачи параметров смещения и сортировки.
    
3. Возврат `Slice<Person>`. Выбирается `LIMIT+1` строк для определения наличия дополнительных данных. `ResultSetExtractor` не поддерживается.
    
4. Постраничный запрос с возвратом `Page<Person>`. Выбираются данные в пределах страницы и потенциально выполняется count-запрос. `ResultSetExtractor` не поддерживается.
    
5. Поиск одной сущности по критериям. При не уникальных результатах — `IncorrectResultSizeDataAccessException`.
    
6. В отличие от (3), всегда возвращается первый результат, даже если запрос даёт больше строк.
    
7. Запрос с `@Query`.
    
8. Возврат `Stream` — значения доступны сразу по мере получения из БД.
    
9. Использование SpEL для динамического разрешения параметров (в примере — Spring Security для получения имени текущего пользователя).

**Выведение запросов ограничено свойствами, которые можно использовать в WHERE без соединений (joins).**

---
## 1. Поддерживаемые ключевые слова

| Ключевое слово         | Пример                                 | Логический результат                            |
| ---------------------- | -------------------------------------- | ----------------------------------------------- |
| `And`                  | `findByLastnameAndFirstname`           | `WHERE lastname = ? AND firstname = ?`          |
| `Or`                   | `findByLastnameOrFirstname`            | `WHERE lastname = ? OR firstname = ?`           |
| `Is`, `Equals`         | `findByFirstnameIs`                    | `WHERE firstname = ?` (или `IS NULL`)           |
| `Between`              | `findByStartDateBetween`               | `WHERE startDate BETWEEN ? AND ?`               |
| `LessThan`             | `findByAgeLessThan`                    | `WHERE age < ?`                                 |
| `LessThanEqual`        | `findByAgeLessThanEqual`               | `WHERE age <= ?`                                |
| `GreaterThan`          | `findByAgeGreaterThan`                 | `WHERE age > ?`                                 |
| `GreaterThanEqual`     | `findByAgeGreaterThanEqual`            | `WHERE age >= ?`                                |
| `After`                | `findByStartDateAfter`                 | `WHERE startDate > ?`                           |
| `Before`               | `findByStartDateBefore`                | `WHERE startDate < ?`                           |
| `IsNull`, `Null`       | `findByAgeNull`                        | `WHERE age IS NULL`                             |
| `IsNotNull`, `NotNull` | `findByAgeNotNull`                     | `WHERE age IS NOT NULL`                         |
| `Like`                 | `findByFirstnameLike`                  | `WHERE firstname LIKE ?`                        |
| `NotLike`              | `findByFirstnameNotLike`               | `WHERE firstname NOT LIKE ?`                    |
| `StartingWith`         | `findByFirstnameStartingWith`          | `WHERE firstname LIKE ?` (с `%` на конце)       |
| `EndingWith`           | `findByFirstnameEndingWith`            | `WHERE firstname LIKE ?` (с `%` в начале)       |
| `Containing`           | `findByFirstnameContaining`            | `WHERE firstname LIKE ?` (с `%` с обеих сторон) |
| `OrderBy`              | `findByAgeOrderByLastnameDesc`         | `WHERE age = ? ORDER BY lastname DESC`          |
| `Not`                  | `findByLastnameNot`                    | `WHERE lastname <> ?` (или `IS NOT NULL`)       |
| `In`                   | `findByAgeIn(Collection<Age> ages)`    | `WHERE age IN (?)`                              |
| `NotIn`                | `findByAgeNotIn(Collection<Age> ages)` | `WHERE age NOT IN (?)`                          |
| `True`                 | `findByActiveTrue()`                   | `WHERE active = TRUE`                           |
| `False`                | `findByActiveFalse()`                  | `WHERE active = FALSE`                          |
| `IgnoreCase`           | `findByFirstnameIgnoreCase`            | `WHERE UPPER(firstname) = UPPER(?)`             |

## 2. Стратегии поиска запросов

Модуль JDBC поддерживает определение запроса 
- вручную через `@Query` 
- или как именованный запрос в property-файле. 

Выведение запроса из имени метода **ограничено простыми свойствами** (только свойства непосредственно в корне агрегата) и поддерживает **только SELECT-запросы**.

---
## 3. Использование `@Query`
```java
interface UserRepository extends CrudRepository<User, Long> {

    @Query("select firstName, lastName from User u where u.emailAddress = :email")
    User findByEmailAddress(@Param("email") String email);
}
```

**Особенности:**
- Для преобразования результата используется тот же `RowMapper`, что и для автоматически генерируемых запросов.
    
- Запрос должен соответствовать формату, ожидаемому `RowMapper`.
    
- ==Колонки для всех свойств конструктора обязательны.== Для сеттеров/полей — опциональны. Свойства без совпадающей колонки не устанавливаются.
    
- Запрос используется для заполнения корня агрегата, встроенных сущностей и one-to-one связей (включая массивы примитивов как SQL-array).
    
- Для `Map`, `List`, `Set` и массивов сущностей генерируются **отдельные запросы**.
    
- **One-to-one свойства** должны иметь префикс в виде имени отношения + `_`. Например, если у `User` есть `Address address` с полем `city`, колонка должна называться `address_city`.
    
- **Строковые запросы не поддерживают пагинацию** и не принимают `Sort`, `PageRequest`, `Limit`. Для лимитирования используйте SQL напрямую.
    
- **SpEL-выражения:**
    - **`: #{#person.id}`** — заменяется на bind-переменную со значением выражения.
        
    - **`#{tableName}`** — вычисляется один раз до первого выполнения, используется для динамических имён таблиц.
        
- Spring Data JDBC поддерживает **только именованные параметры**.
    

---
## 4. Именованные запросы (Named Queries)

Если запрос не указан в аннотации, Spring Data JDBC ищет именованный запрос. 

**По умолчанию** имя формируется как `имя_класса.имя_метода`. Альтернативно — через атрибут `name` в `@Query`.

Именованные запросы хранятся в файле **`META-INF/jdbc-named-queries.properties`**. 

Расположение можно изменить через `@EnableJdbcRepositories.namedQueriesLocation`.

---
## 5. Потоковая передача результатов (Streaming Results)

При возврате `Stream` элементы доступны по мере получения из БД.

==Стрим содержит открытое соединение== — его нужно закрыть (`try-with-resources`), иначе утечка соединения. 

После закрытия стрим не может вернуть новые элементы.

## 6. Пользовательский `RowMapper` или `ResultSetExtractor`

Аннотация `@Query` позволяет указать кастомные 
- `rowMapperClass`, `resultSetExtractorClass` 
- или ссылки на бины `rowMapperClassRef`/`resultSetExtractorClassRef`.

**Глобальная регистрация `RowMapper` через `QueryMappingConfiguration`:**
```java
@Bean
QueryMappingConfiguration rowMappers() {
    return new DefaultQueryMappingConfiguration()
        .register(Person.class, new PersonRowMapper())
        .register(Address.class, new AddressRowMapper());
}
```

**Алгоритм выбора `RowMapper`:**
1. Простой тип → без `RowMapper` (одна строка, одна колонка, конвертация).
    
2. Итерация по `QueryMappingConfiguration`: ищется суперкласс/интерфейс возвращаемого типа. Используется зарегистрированный `RowMapper`. **Порядок регистрации важен:** более общие типы регистрируйте после конкретных.
    
3. Обёртки (`Optional`, коллекции) разворачиваются.
    

**Важно:** Использование кастомного `RowMapper` или `ResultSetExtractor` **отключает Entity Callbacks и Lifecycle Events**.

---
## 7. Модифицирующие запросы
```java
@Modifying
@Query("UPDATE DUMMYENTITY SET name = :name WHERE id = :id")
boolean updateName(@Param("id") Long id, @Param("name") String name);
```

**Поддерживаемые возвращаемые типы:** `void`, `int` (количество обновлённых записей), `boolean` (была ли запись обновлена). **Модифицирующие запросы выполняются напрямую против БД — события и колбэки не вызываются.** Поля с аннотациями аудита не обновляются, если они не указаны в запросе явно.
