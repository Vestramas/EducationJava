Spring Data JDBC публикует события жизненного цикла для `ApplicationListener` (обычно бинов в контексте приложения). 

**В отличие от entity callbacks, события предназначены для уведомлений**, а не для изменения состояния сущности. Транзакционные слушатели получают события при завершении транзакции.

> **Важно:** 
> События и колбэки срабатывают **только для корней агрегатов**. Для обработки не-корневых сущностей нужно использовать слушатель для содержащего их корня агрегата.

**Предупреждение о производительности:** События жизненного цикла могут быть затратными. При загрузке больших наборов результатов можно заметить изменение производительности. События можно отключить на Template API.

## 1. Общий слушатель
```java
@Bean
ApplicationListener<BeforeSaveEvent<Object>> loggingSaves() {
    return event -> {
        Object entity = event.getEntity();
        LOG.info("{} is getting saved.", entity);
    };
}
```

---
## 2. Слушатель для конкретного типа
Если нужно обрабатывать события только для определённого доменного типа, наследуйтесь от **`AbstractRelationalEventListener`**:
```java
class PersonLoadListener extends AbstractRelationalEventListener<Person> {

    @Override
    protected void onAfterLoad(AfterLoadEvent<Person> personLoad) {
        LOG.info(personLoad.getEntity());
    }
}
```

Callback-методы вызываются только для событий, связанных с доменным типом и его подтипами — кастомизация не требуется.

---
## 3. Доступные события

|Событие|Когда публикуется|
|---|---|
|**`BeforeDeleteEvent`**|Перед удалением корня агрегата|
|**`AfterDeleteEvent`**|После удаления корня агрегата|
|**`BeforeConvertEvent`**|Перед преобразованием агрегата в план SQL-выражений, **после** определения, новый агрегат или существующий (т.е. insert или update)|
|**`BeforeSaveEvent`**|Перед сохранением (insert/update), **после** решения о том, что будет: insert или update|
|**`AfterSaveEvent`**|После сохранения (insert/update)|
|**`AfterConvertEvent`**|После создания агрегата из `ResultSet` и установки всех его свойств|

> События жизненного цикла зависят от `ApplicationEventMulticaster`. При использовании `SimpleApplicationEventMulticaster` с `TaskExecutor` **нет гарантий**, когда событие будет обработано.

## 4. Entity Callbacks (специфичные для JDBC)

Spring Data JDBC использует API Entity Callbacks для поддержки аудита. 

> [!WARNING] ИСПОЛЬЗУЙ КОЛБЭКИ А НЕ СОБЫТИЯ
> **Рекомендуется использовать колбэки вместо событий**, так как они поддерживают иммутабельные классы.


|Процесс|EntityCallback / Шаг процесса|Комментарий|
|---|---|---|
|**Удаление**|||
||`BeforeDeleteCallback`|Перед фактическим удалением|
||(удаление из БД)|Корень агрегата и все сущности агрегата удаляются|
||`AfterDeleteCallback`|После удаления агрегата|
|**Сохранение**|||
||(определение insert/update)|На основе новизны агрегата|
||`BeforeConvertCallback`|**Правильный колбэк для программной установки ID.** На предыдущем шаге новые агрегаты обнаружены, и ID, сгенерированный здесь, будет использован далее.|
||(преобразование в агрегат изменений)|Последовательность SQL-выражений. Решение: ID предоставлен агрегатом или ожидается от БД.|
||`BeforeSaveCallback`|Изменения в корне агрегата могут быть учтены, но решение об отправке ID в БД уже принято. **Не используйте для создания ID — используйте `BeforeConvertCallback`.**|
||(выполнение SQL)|SQL-выражения выполняются против БД|
||`AfterSaveCallback`|После сохранения (insert/update)|
|**Загрузка**|||
||(загрузка и конструирование)|Загрузка через 1+ SQL-запросов, конструирование из `ResultSet`|
||`AfterConvertCallback`|После создания агрегата из `ResultSet`|

> **Колбэки, связанные с удалением и сохранением, не вызываются для производных методов запросов или аннотированных запросов.** Spring Data JDBC не имеет эффективного способа определить, какие агрегаты затронуты этими операциями, поэтому не может предоставить колбэки или события.
