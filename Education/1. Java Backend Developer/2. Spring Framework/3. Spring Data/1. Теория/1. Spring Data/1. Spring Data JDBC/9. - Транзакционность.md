Методы экземпляров `CrudRepository` **транзакционны по умолчанию**. 

Для операций чтения устанавливается флаг `readOnly = true`, для всех остальных — обычный `@Transactional` со стандартными настройками.

## 1. Переопределение транзакционной конфигурации

Если нужно изменить настройки для метода, объявленного в репозитории, **переобъявите метод** в своём интерфейсе:
```java
interface UserRepository extends CrudRepository<User, Long> {

    @Override
    @Transactional(timeout = 10)
    List<User> findAll();

    // Дальнейшие объявления методов запросов
}
```

Метод `findAll()` будет выполняться с таймаутом 10 секунд и **без флага `readOnly`**.

---
## 2. Фасад для управления транзакциями
Другой способ — использовать фасад или сервис, покрывающий несколько репозиториев:
```java
@Service
public class UserManagementImpl implements UserManagement {

    private final UserRepository userRepository;
    private final RoleRepository roleRepository;

    UserManagementImpl(
	    UserRepository userRepository, 
	    RoleRepository roleRepository
	) {
        this.userRepository = userRepository;
        this.roleRepository = roleRepository;
    }

    @Transactional
    public void addRoleToAllUsers(String roleName) {
        Role role = roleRepository.findByName(roleName);
        for (User user : userRepository.findAll()) {
            user.addRole(role);
            userRepository.save(user);
        }
    }
}
```

Транзакционная конфигурация репозиториев **игнорируется** — внешняя транзакция определяет фактическое поведение. Необходимо явно активировать `<tx:annotation-driven/>` или `@EnableTransactionManagement`.

---
## 3. Транзакционные методы запросов

> [!NOTE] ДЕЛАЙ МЕТОДЫ ТРАНЗАКЦИОННЫМИ
> **Настоятельно рекомендуется делать методы запросов транзакционными.** 

Без общей транзакции Spring Data JDBC выполняет запросы в **разных соединениях**, что создаёт нагрузку на пул соединений и может привести к дедлокам.
```java
@Transactional(readOnly = true)
interface UserRepository extends CrudRepository<User, Long> {

    List<User> findByLastname(String lastname);

    @Modifying
    @Transactional
    @Query("delete from User u where u.active = false")
    void deleteInactiveUsers();
}
```

- `readOnly = true` на уровне интерфейса делает все методы запросов транзакционными только для чтения.
    
- `deleteInactiveUsers()` с `@Modifying` переопределяет конфигурацию — выполняется с `readOnly = false`.


> **Важно о `readOnly`:** 
> Это **не проверка** на отсутствие модифицирующих запросов (хотя некоторые БД отклоняют `INSERT`/`UPDATE` в read-only транзакции). 
> 
> Флаг передаётся ==как подсказка JDBC-драйверу для оптимизации== производительности.

## 4. JDBC Locking

Spring Data JDBC поддерживает блокировку на **производных методах запросов** через аннотацию **`@Lock`**:

```java
interface UserRepository extends CrudRepository<User, Long> {

    @Lock(LockMode.PESSIMISTIC_READ)
    List<User> findByLastname(String lastname);
}
```

**Значения `LockMode`:**
- **`PESSIMISTIC_READ`** — гарантирует, что читаемые данные не будут изменены.
    
- **`PESSIMISTIC_WRITE`** — получает блокировку на изменение данных.
    

Некоторые БД не делают различий — оба режима эквивалентны `PESSIMISTIC_WRITE`.

**Пример для MySQL:**
```sql
SELECT * FROM user u WHERE u.lastname = ? LOCK IN SHARE MODE
```


**Ограничения:**
- `@Lock` **не поддерживается на строковых запросах** (`@Query`). Информация о блокировке игнорируется, выводится предупреждение в логах. В будущих версиях будет выбрасываться исключение.
