Когда стандартной магии `findBy...` и `@Query` не хватает, Spring Data позволяет добавить в репозиторий **собственный код**. Есть два подхода: для одного конкретного репозитория или глобально для всех.

## 1. Кастомизация через фрагменты (Customizing Individual Repositories)
Это основной и рекомендуемый способ добавить свою логику в один или несколько репозиториев.

**Шаг 1. Создаём интерфейс фрагмента** с нужными методами:
```java
interface CustomizedUserRepository {
    void someCustomMethod(User user);
}
```

**Шаг 2. Пишем реализацию.** Имя класса должно заканчиваться на `Impl` (по умолчанию):
```java
class CustomizedUserRepositoryImpl implements CustomizedUserRepository {
    @Override
    public void someCustomMethod(User user) {
        // Ваша кастомная логика (JdbcTemplate, вызов внешнего API и т.д.)
    }
}
```

Этот класс — обычный Spring-бин, в него можно инжектить любые зависимости.

**Шаг 3. «Примешиваем» фрагмент к основному интерфейсу:**
```java
interface UserRepository extends CrudRepository<User, Long>, CustomizedUserRepository {
    // Здесь могут быть и стандартные методы findBy...
}
```

Теперь у бина `userRepository` есть и стандартные CRUD-методы, и ваш `someCustomMethod`.

**Как это работает:**
- Репозиторий — это **композиция фрагментов** (базовые CRUD, аспекты вроде Querydsl и ваши кастомные).
    
- **Приоритет:** Кастомные реализации имеют приоритет выше, чем базовая реализация. Если ваш фрагмент определяет метод `save(...)`, он переопределит стандартный `save` из `CrudRepository`.
    
- **Переиспользование:** Один фрагмент можно подмешать в несколько репозиториев.

```java
// Фрагмент, переопределяющий save
interface CustomizedSave<T> {
    <S extends T> S save(S entity);
}

class CustomizedSaveImpl<T> implements CustomizedSave<T> {
    @Override
    public <S extends T> S save(S entity) { /* кастомная логика */ }
}

// Используем в любых репозиториях
interface UserRepository 
	extends CrudRepository<User, Long>, CustomizedSave<User> {}

interface PersonRepository 
	extends CrudRepository<Person, Long>, CustomizedSave<Person> {}
```

## 2. Регистрация фрагментов через `spring.factories`

Если ваш кастомный фрагмент находится **вне базового пакета** сканирования (например, в общей библиотеке), его можно зарегистрировать в файле `META-INF/spring.factories`:

```
com.acme.search.SearchExtension=com.acme.search.DefaultSearchExtension
```

Тогда любой репозиторий в любом проекте сможет просто расширить `SearchExtension<T>` и получить его функциональность.

**Пример использования контекста метода (`RepositoryMethodContext`):**  
В реализации фрагмента можно получить метаданные (например, доменный тип репозитория):
```java
public interface SearchExtension<T> {

    List<T> search(String text, Limit limit);
}
```

```java
class DefaultSearchExtension<T> implements SearchExtension<T>, RepositoryMetadataAccess {

    private final SearchService service;

    @Override
    public List<T> search(String text, Limit limit) {
        // Получаем контекст текущего вызванного метода
        RepositoryMethodContext metadata = 
	        RepositoryMethodContext.getContext();
	        
        Class<T> domainType = metadata.getRepository().getDomainType();
        // ... используем domainType для определения имени индекса и т.д.
    }
}
```

**Важно:** Доступ к `RepositoryMethodContext` отключён по умолчанию (это дорого). Включить можно либо реализовав интерфейс-маркер `RepositoryMetadataAccess` на классе фрагмента, либо через `BeanPostProcessor`, который выставит флаг `setExposeMetadata(true)` на фабрике репозиториев.

## 3. Глобальная кастомизация базового класса (Customize the Base Repository)
Если нужно ==изменить поведение== (например, метод `save`) **для всех репозиториев сразу**, 

создаётся свой класс, 

**наследующий от стандартной реализации** (для JPA это `SimpleJpaRepository`):
```java
class MyRepositoryImpl<T, ID> extends SimpleJpaRepository<T, ID> {

    MyRepositoryImpl(
	    JpaEntityInformation entityInformation, 
	    EntityManager entityManager
	) {
        super(entityInformation, entityManager);
    }

    @Override
    @Transactional
    public <S extends T> S save(S entity) {
        // Кастомная логика для всех save во всём приложении
    }
}
```

Затем этот класс указывается в конфигурации:
```java
@EnableJpaRepositories(repositoryBaseClass = MyRepositoryImpl.class)
class ApplicationConfiguration { ... }
```

## 4. Другие точки расширения (кратко)
- **`RepositoryFactoryCustomizer`**: для добавления слушателей, советников (advisors) в процесс создания прокси.
    
- **`RepositoryFactoryBean`**: полная замена фабрики репозиториев. Самый мощный, но и самый сложный способ, если нужно изменить ядро создания репозиториев.
