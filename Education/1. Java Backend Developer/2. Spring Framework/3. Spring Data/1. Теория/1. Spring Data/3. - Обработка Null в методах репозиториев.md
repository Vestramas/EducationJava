Тема `null` — одна из самых частых причин ошибок. Spring Data предоставляет чёткие гарантии и механизмы контроля.

## 1. Базовые гарантии
- **Одиночные результаты могут быть Null:** могут вернуть `null`, если результат не найден. 
  Чтобы избежать `NullPointerException`, **рекомендуется использовать `Optional<T>`**.
    
- **Коллекции и Stream НИКОГДА НЕ ВЕРНУТ NULL:** Методы, возвращающие коллекции, альтернативные коллекции (например, Vavr), обёртки и `Stream`, **гарантированно никогда не возвращают `null`**. 
  При отсутствии результатов они вернут пустую коллекцию/поток.
    
- **Поддерживаемые обёртки для одиночных результатов:** Помимо `Optional`, можно использовать `com.google.common.base.Optional`, `scala.Option`, `io.vavr.control.Option`.

## 2. Аннотации Nullability (Spring Framework)
Можно ужесточить контроль на уровне кода с помощью аннотаций:
- **`@NonNullApi`** — вешается на пакет (`package-info.java`). Объявляет, что по умолчанию параметры и возвращаемые значения **не принимают и не производят `null`**.
    
- **`@NonNull`** — для явного указания non-null там, где `@NonNullApi` не действует.
    
- **`@Nullable`** — явно разрешает `null` для параметра или возврата.


**Как это работает в репозитории:**
```java
package com.acme; // Пакет помечен @NonNullApi

interface UserRepository extends Repository<User, Long> {

  User getByEmailAddress(EmailAddress emailAddress);                    // (1)

  @Nullable
  User findByEmailAddress(@Nullable EmailAddress emailAddress);          // (2)

  Optional<User> findOptionalByEmailAddress(EmailAddress emailAddress); // (3)
}
```

1. **Non-null по умолчанию:** Метод **выбросит исключение** (`EmptyResultDataAccessException`), если запрос не дал результата. Также **выбросит исключение**, если `emailAddress` равен `null`.
    
2. **Явный `@Nullable`:** Метод **вернёт `null`**, если результат не найден. Также **принимает `null`** в качестве аргумента.
    
3. **`Optional`:** Метод **вернёт `Optional.empty()`**, если результат не найден. Но если передать `null` в аргумент, **выбросит исключение** (так как параметр non-null).
    

### 2.1 Nullability в Kotlin
Kotlin имеет встроенную систему null-безопасности на уровне языка. Spring Data интегрируется с ней (требуется `kotlin-reflect` в classpath):
```kotlin
interface UserRepository : Repository<User, String> {
    fun findByUsername(username: String): User      // (1) Non-null параметр и результат
    fun findByFirstname(firstname: String?): User?  // (2) Nullable параметр и результат
}
```
(1) — выбросит исключение при пустом результате или передаче `null`.  
(2) — может принять `null` и вернуть `null`.

## 3. Примечание про JSpecify
Начиная с Spring Framework 7 / Spring Data 4, собственные аннотации Spring (`@NonNullApi` и т.д.) считаются устаревшими в пользу стандарта **JSpecify** (`@NullMarked`, `@Nullable`, `@NonNull`). 

Синтаксис для массивов:
- `@Nullable Object[] array` — элементы могут быть null, массив — нет.
    
- `Object @Nullable [] array` — массив может быть null, элементы — нет.
    
- `@Nullable Object @Nullable [] array` — и элементы, и массив могут быть null.
