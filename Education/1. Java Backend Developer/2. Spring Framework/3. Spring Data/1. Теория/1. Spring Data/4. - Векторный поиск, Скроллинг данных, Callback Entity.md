# 1. Векторный поиск (Vector Search) — Новый раздел
Эта функциональность создана для работы с Generative AI. Векторный поиск ***ищет данные не по точному совпадению*** (как SQL `WHERE`), а по их **смыслу**. 

Он ==сравнивает векторные представления (эмбеддинги)== и находит семантически близкие объекты. 
> Это ключ к созданию рекомендательных систем, семантического поиска и NLP-приложений.

## 1. Основные типы
Spring Data вводит три ключевых абстракции для типобезопасной работы:

1. **`Vector`**: Неизменяемая обёртка над массивом чисел с плавающей точкой. **Представляет собой сам эмбеддинг**. 
   Используется и как поле сущности, и как параметр поиска.
```java
Vector queryVector = Vector.of(0.23f, 0.11f, 0.77f);
```

2. **`SearchResults<T>`**: Результат поиска. Содержит список `SearchResult<T>`.
    
3. **`SearchResult<T>`**: Одна запись результата. Содержит найденный объект (`T`) и его `Score` (оценку релевантности).


**Важно об оценке (`Score`):**
- Это число (обычно `float`), показывающее, насколько результат близок к запросу.
    
- Её значение зависит от **функции схожести** (`ScoringFunction`). Где-то «лучше» значит «больше», где-то — «меньше». Это зависит от математики функции.
    
- `Score` не является частью доменной модели, это внешние, «out-of-band» данные.
    

## 2. Стандартные ФУНКЦИИ схожести (`ScoringFunction`)
**ВЫБОР ФУНКЦИИ влияет на производительность** и семантику поиска. 

Spring Data поддерживает константы для трёх самых распространённых:
- **Euclidean Distance (Евклидово расстояние):** Прямая геометрическая дистанция в N-мерном пространстве. Чем меньше, тем лучше.
    
- **Cosine Similarity (Косинусное сходство):** Измеряет угол между векторами, игнорируя их длину. Чем ближе к 1, тем лучше.
    
- **Dot Product (Скалярное произведение):** Сумма перемножений элементов. Часто используется в обученных моделях.

## 3. Методы векторного поиска
Работают по тем же принципам, что и обычные методы запросов. Главное — метод должен принимать `Vector` и возвращать `SearchResults<T>`.

Есть два способа объявить такой метод:

**1. Выводимый метод (Derived Search):** Строится по имени.  
***Ключевые слова:*** `Near` (или `IsNear`) и `Within` (или `IsWithin`). 
> Для ограничения результата по score эти методы требуют второй параметр: `Score`, `Range<Score>` или `ScoringFunction`.

**2. Аннотированный метод (Annotated Search):** Использует `@Query`.  
Даёт полный контроль над семантикой. Если в аннотированном запросе не указан `Score`, он будет равен `0`.

## 4. Сортировка результатов
По умолчанию результаты сортируются по их `Score`. Это поведение можно переопределить, добавив параметр `Sort`. Но сортировать можно только по свойствам доменной модели, сам `Score` как критерий в `Sort` использовать нельзя.
```java
// Примеры методов векторного поиска
interface CommentRepository extends Repository<Comment, String> {

    // По умолчанию сортировка по Score
    SearchResults<Comment> searchByEmbeddingNear(Vector vector, Score score);

    // Переопределяем сортировку на свойство доменной модели
    SearchResults<Comment> searchByEmbeddingWithin(Vector vector, Score score, Sort sort);
}
```

# 2. Scrolling
Скроллинг — это более тонкий, «порционный» способ итерации по большим результатам, альтернативный классической пагинации. Он использует стабильную сортировку и `ScrollPosition` для получения следующего окна данных. Результатом является объект `Window<T>`.

**Основной принцип:**  
> Вместо «дай мне страницу №5», вы говорите «дай мне следующую порцию данных, начиная с этой позиции». 
> 
> Это позволяет базе данных избежать полной материализации всего результата, в отличие от offset-based пагинации.

**Пример со статической сортировкой:**
```java
Window<User> users = 
	repository
		.findFirst10ByLastnameOrderByFirstname(
			"Doe", 
			ScrollPosition.offset()
		);
do {
    for (User u : users) {
        // обрабатываем пользователя
    }
    
    //получаем позицию последнего элемента,чтобы запросить следующее окно
    users = 
	    repository
		    .findFirst10ByLastnameOrderByFirstname(
			    "Doe", 
			    users.positionAt(users.size() - 1)
			);
} while (!users.isEmpty());
```

**Упрощение через `WindowIterator`:**  
Код выше довольно многословен. `WindowIterator` скрывает всю рутину:
```java
WindowIterator<User> users = 
	WindowIterator.of(
	    position -> 
		    repository
			    .findFirst10ByLastnameOrderByFirstname("Doe", position)
).startingAt(ScrollPosition.offset());

while (users.hasNext()) {
    User u = users.next();
    // обрабатываем пользователя
}
```

## 2.1 Два типа скроллинга

### **1. Offset-based Scrolling (на основе смещения)**  
Похож на пагинацию. Использует параметр `OffsetScrollPosition`.
- **Плюс:** Простой механизм.
    
- **Минус:** Большинство баз данных всё равно материализуют полный результат перед тем, как вернуть вам окно.
```java
interface UserRepository extends Repository<User, Long> {
    Window<User> findFirst10ByLastnameOrderByFirstname(
	    String lastname, 
	    OffsetScrollPosition position
	);
}

WindowIterator<User> users = 
	WindowIterator.of(	    
	    position -> 
		    repository
			    .findFirst10ByLastnameOrderByFirstname("Doe", position)
).startingAt(OffsetScrollPosition.initial()); // (1)
```
`(1)` Начинаем с нулевого смещения. 

**Важно:** `ScrollPosition.offset()` (без аргументов) означает «начало скролла». `ScrollPosition.offset(0)` означает «пропустить первый элемент». Из-за исключающей природы скроллинга, `offset(0)` даст смещение 1.

### **2. Keyset-Filtering (на основе набора ключей)**
Более продвинутый и эффективный метод.

- **Идея:** ***Запоминает значения полей (keyset) последней записи в текущем окне.*** 
  >Следующий запрос фильтрует данные так: «дай мне записи, которые идут ПОСЛЕ этой, согласно сортировке».
    
- **Плюсы:** База данных использует индексы и не строит полный результат. Гораздо эффективнее при больших объёмах.
    
- **Минусы/Требования:**
    - Все поля, используемые в сортировке, **должны быть не-nullable**. Иначе поведение непредсказуемо.
        
    - Лучше всего работает, если есть индекс, совпадающий с порядком сортировки.
        
    - Поля сортировки должны присутствовать в возвращаемой сущности (важно для проекций DTO).
```java
interface UserRepository extends Repository<User, Long> {
    Window<User> findFirst10ByLastnameOrderByFirstname(
	    String lastname, 
	    KeysetScrollPosition position
	);
}

WindowIterator<User> users = 
	WindowIterator.of(
	    position -> 
		    repository
			    .findFirst10ByLastnameOrderByFirstname("Doe", position)
).startingAt(ScrollPosition.keyset()); // (1)
```

`(1)` `ScrollPosition.keyset()` означает начало скролла без дополнительной фильтрации.

**Динамическая сортировка и лимитирование:**  
Как и с `Pageable`, вы можете использовать параметры `Sort` и `Limit` в методах скроллинга, чтобы задавать сортировку и размер окна динамически при каждом запросе.

# 3. Обратные вызовы сущностей (Entity Callbacks)
`EntityCallback` — это специализированный обработчик, похожий на `ApplicationListener`, который позволяет проверить и потенциально модифицировать сущность **до или после** вызова определённых методов репозитория.

**Зачем нужны колбэки, если есть события (`@DomainEvents`)?**
- **Проблема событий:** `ApplicationEvents` полагаются на `ApplicationEventMulticaster`. Если он настроен на асинхронную работу, события обрабатываются в другом потоке, что может привести к непредсказуемым результатам.
    
- **Решение колбэков:** Колбэки гарантируют **синхронное выполнение в строго определённых точках** цепочки обработки. Они работают как с синхронными, так и с реактивными API.


**Важно:** Колбэки разделены по типу API. Синхронный API использует только синхронные колбэки, реактивный — только реактивные.

## 1. Встроенные типы колбэков
Каждый модуль Spring Data предоставляет набор предопределённых интерфейсов, покрывающих жизненный цикл сущности. 

Самые распространённые:
- `BeforeSaveCallback<T>` — вызывается **перед сохранением** сущности
    
- `AfterSaveCallback<T>` — вызывается **после сохранения**
    
- `BeforeDeleteCallback<T>` — вызывается **перед удалением**
    
- `AfterLoadCallback<T>` — вызывается **после загрузки** из БД
    

И их реактивные аналоги: `ReactiveBeforeSaveCallback<T>` и т.д.

**Структура колбэка (синхронный вариант):**
```java
@FunctionalInterface
public interface BeforeSaveCallback<T> extends EntityCallback<T> {
    T onBeforeSave(T entity, String collection); // (1) и (2)
}
```
`(1)` — возвращает ту же или модифицированную сущность.  
`(2)` — `collection` (или другие аргументы) специфичны для конкретного модуля (например, имя коллекции MongoDB).

**Реактивный вариант:**
```java
@FunctionalInterface
public interface ReactiveBeforeSaveCallback<T> extends EntityCallback<T> {
    Publisher<T> onBeforeSave(T entity, String collection);
}
```

## 2. Реализация колбэка
Нужно реализовать подходящий интерфейс. 

Если для одного типа сущности несколько колбэков одного типа, можно управлять порядком выполнения через `@Order` или интерфейс `Ordered`.
```java
class DefaultingEntityCallback implements BeforeSaveCallback<Person>, Ordered {
    @Override
    public Object onBeforeSave(Person entity, String collection) {
        if("user".equals(collection)) {
            return // ... модифицированная сущность
        }
        return entity;
    }

    @Override
    public int getOrder() {
        return 100; // Чем ниже значение, тем раньше выполнится
    }
}
```

## 3. Регистрация колбэков
Колбэки регистрируются как обычные Spring-бины в `ApplicationContext`. 

Инфраструктура модуля сама их найдёт.

### **Способы регистрации и управления порядком:**
1. **Аннотация `@Order` на классе:**
```java
@Order(1)
@Component
class First implements BeforeSaveCallback<Person> { ... }
```

2. **Реализация интерфейса `Ordered`:**
```java
@Component
class DefaultingEntityCallback implements BeforeSaveCallback<Person>, Ordered {
    @Override
    public int getOrder() { return 100; }
}
```

3. **Через лямбда-выражение (Java Config):**
```java
@Configuration
class EntityCallbackConfiguration {
    @Bean
    BeforeSaveCallback<Person> unorderedLambdaReceiverCallback() {
        return (entity) -> { /* модификация */ return entity; };
    }
}
```
**Важно:** Лямбды не имеют порядка и вызываются последними. Также они не предоставляют информацию о типе, что может повлиять на фильтрацию. Для типовой безопасности лучше использовать обычный класс.

4. **Объединение нескольких колбэков в одном классе:**
```java
@Component
class UserCallbacks 
	implements BeforeConvertCallback<User>, BeforeSaveCallback<User> 
{
    @Override
    public User onBeforeConvert(User user) { ... }
    @Override
    public User onBeforeSave(User user) { ... }
}
```
