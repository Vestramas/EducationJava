# 1. Интеграция с контейнером сервлетов
Spring Security интегрируется с контейнером сервлетов с помощью стандартного Servlet `Filter`. 

Это означает, что он работает с любым приложением, которое запускается в контейнере сервлетов. 

Более конкретно: вам **не обязательно** использовать Spring в вашем приложении на основе сервлетов, чтобы воспользоваться преимуществами Spring Security.

---
# 2. Архитектура 
В этом разделе обсуждается высокоуровневая архитектура Spring Security в приложениях на основе сервлетов.

## 2.1 Обзор фильтров
Поддержка сервлетов в Spring Security основана на Servlet Filters, поэтому для начала полезно рассмотреть роль фильтров в целом.

На следующем изображении показано типичное распределение обработчиков для одного HTTP-запроса.
![[Pasted image 20260416111619.png]]
Рисунок 1. FilterChain (Цепочка фильтров)

Клиент отправляет запрос приложению, и **контейнер создает** `FilterChain`, которая содержит экземпляры `Filter` и `Servlet`, которые должны обработать `HttpServletRequest`, на основе пути URI запроса.  

### 2.1 Один сервлет, много фильтров
В приложении Spring MVC сервлетом является экземпляр `DispatcherServlet`.  
>Только один сервлета может обрабатывать один `HttpServletRequest` и `HttpServletResponse`.  

Однако можно использовать более одного фильтра для следующих целей:
- Предотвращение вызова нижестоящих экземпляров `Filter` или сервлета.  
    В этом случае фильтр обычно записывает ответ в `HttpServletResponse`.
    
- Изменение `HttpServletRequest` или `HttpServletResponse`, используемых нижестоящими экземплярами `Filter` и сервлетом.


Сила `Filter` заключается в `FilterChain`, которая в него передается.

_Пример использования FilterChain_
```java
@Override
public void doFilter(
	ServletRequest request, 
	ServletResponse response,
	FilterChain chain) 
	throws IOException, ServletException 
{
	// сделать что-то до вызова остальной части приложения
	chain.doFilter(request, response); // вызвать остальную часть приложения
	// сделать что-то после вызова остальной части приложения
}
```

Поскольку `Filter` влияет только на нижестоящие экземпляры `Filter` и `Servlet`, ==порядок вызова каждого фильтра чрезвычайно важен.==

## 2.2 DelegatingFilterProxy

> [!NOTE] DelegatingFilterProxy 
> Это мост между контейнером сервлетов (Tomcat) и контекстом Spring

**Spring предоставляет реализацию** `Filter` под названием `DelegatingFilterProxy`, которая позволяет соединить жизненный цикл контейнера сервлетов с `ApplicationContext` Spring.  

**Контейнер сервлетов позволяет регистрировать экземпляры** `Filter` с использованием собственных стандартов, но он не знает о бинах, определенных в Spring.  

Вы можете зарегистрировать `DelegatingFilterProxy` через стандартные механизмы контейнера сервлетов, но делегировать всю работу Spring-бину, который реализует `Filter`.

Вот изображение того, как `DelegatingFilterProxy` вписывается в экземпляры `Filter` и `FilterChain`.
![[Pasted image 20260416112602.png]]
Рисунок 2. DelegatingFilterProxy

`DelegatingFilterProxy` **ищет бин** `Filter0` в `ApplicationContext`, а затем вызывает бин `Filter0`.  

В следующем листинге показан **псевдокод** `DelegatingFilterProxy`:
_Псевдокод DelegatingFilterProxy_
```java
public void doFilter(
	ServletRequest request, 
	ServletResponse response, 
	FilterChain chain) 
{
	Filter delegate = getFilterBean(someBeanName); // (1)
	delegate.doFilter(request, response); // (2)
}
```

| №   | Описание                                                                                                                                                    |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Ленивое получение `Filter`, зарегистрированного как Spring Bean. <br><br>Для примера в `DelegatingFilterProxy` делегатом является экземпляр `Bean Filter0`. |
| (2) | Делегирование работы Spring-бину.                                                                                                                           |

Еще одно преимущество `DelegatingFilterProxy` заключается в том, что он позволяет отложить поиск экземпляров бинов `Filter`.  

Это важно, поскольку контейнер должен зарегистрировать экземпляры `Filter` до того, как контейнер сможет запуститься.  

Однако Spring обычно использует `ContextLoaderListener` для загрузки Spring-бинов, что происходит только после того, как экземпляры `Filter` должны быть зарегистрированы.

## 2.3 FilterChainProxy - Filter0 из примера выше
Поддержка сервлетов в Spring Security заключена в `FilterChainProxy`.  


> [!NOTE] `FilterChainProxy` - бин, который находится внутри `DelegatingFilterProxy`
> `FilterChainProxy` — это специальный `Filter`, предоставляемый Spring Security, который позволяет делегировать вызовы множеству экземпляров `Filter` через `SecurityFilterChain`.  

Поскольку `FilterChainProxy` является бином, он обычно оборачивается в `DelegatingFilterProxy`.

## 2.4 SecurityFilterChain - ЦЕПОЧЧКА фильтров безопасности (контейнер)
`SecurityFilterChain` используется в `FilterChainProxy` для определения того, какие экземпляры `Filter` Spring Security должны быть вызваны для текущего запроса.

> [!NOTE] SecurityFilterChain
> Это **уже цепочка фильтров SpringSecurity**

На следующем изображении показана роль `SecurityFilterChain`.
![[Pasted image 20260416113034.png]]
Рисунок 4. SecurityFilterChain

Фильтры безопасности (Security Filters) в `SecurityFilterChain` обычно являются бинами, но они регистрируются в `FilterChainProxy`, а не в `DelegatingFilterProxy`.  
> Так как скорее всего было бы глупо все наши фильтры безопасности запихивать в кастомный фильтр контейнера сервлетов. 
> 
> Проще чтобы все эти фильтры безопасности хранились в одном бине, и этот бин уже дергал наш кастомный фильтр в контейнере

### 2.4.1 Почему не напрямую в DelegatingFilterProxy создавать фильтры безопасности
`FilterChainProxy` дает ряд преимуществ по сравнению с регистрацией напрямую в контейнере сервлетов или через `DelegatingFilterProxy`.

**Во-первых**, он является отправной точкой для всей поддержки сервлетов в Spring Security.  
По этой причине, если вы пытаетесь устранить неполадки в поддержке сервлетов Spring Security, установка точки отладки в `FilterChainProxy` — отличное место для начала.

**Во-вторых**, поскольку `FilterChainProxy` занимает центральное место в использовании Spring Security, он может выполнять задачи, которые не считаются опциональными.  
Например, он очищает `SecurityContext`, чтобы избежать утечек памяти.  
Он также применяет `HttpFirewall` Spring Security для защиты приложений от определенных типов атак.

Кроме того, он обеспечивает большую гибкость при определении момента вызова `SecurityFilterChain`.  
В контейнере сервлетов экземпляры `Filter` вызываются только на основе URL.  

### 2.4.2 Несколько цепочек фильтров безопасности
Однако `FilterChainProxy` может определять вызов на основе любой информации в `HttpServletRequest`, используя интерфейс `RequestMatcher`.

На следующем изображении показано несколько экземпляров `SecurityFilterChain`:
![[Pasted image 20260416113603.png]]
Рисунок 5. Несколько SecurityFilterChain

На рисунке с несколькими `SecurityFilterChain` `FilterChainProxy` решает, какой `SecurityFilterChain` следует использовать.  

Вызывается только первый подходящий `SecurityFilterChain`.  
- Если запрашивается URL `/api/messages/`, он сначала сопоставляется с шаблоном `SecurityFilterChain0` `/api/**`, поэтому вызывается только `SecurityFilterChain0`, даже если он также соответствует `SecurityFilterChainn`. 
	
- Если запрашивается URL `/messages/`, он не соответствует шаблону `SecurityFilterChain0` `/api/**`, поэтому `FilterChainProxy` продолжает перебирать `SecurityFilterChain`.  
  Предполагая, что ни один другой экземпляр `SecurityFilterChain` не подходит, вызывается `SecurityFilterChainN`.

Обратите внимание, что в `SecurityFilterChain0` настроено только три экземпляра фильтра безопасности.  

Однако в `SecurityFilterChainN` настроено четыре экземпляра фильтра безопасности.  

Важно отметить, что каждый `SecurityFilterChain` может быть уникальным и настраиваться изолированно. 

Фактически, `SecurityFilterChain` может иметь ноль экземпляров фильтров безопасности, если приложение хочет, чтобы Spring Security игнорировал определенные запросы.

## 2.5 Фильтры безопасности (Security Filters)
Фильтры безопасности вставляются в `FilterChainProxy` с помощью API `SecurityFilterChain`. 

Эти фильтры могут использоваться для различных целей, таких как защита от эксплойтов, аутентификация, авторизация и многое другое. 

==Фильтры выполняются в определенном порядке==, чтобы гарантировать их вызов в нужное время, 
> например, `Filter`, выполняющий аутентификацию, должен вызываться до `Filter`, выполняющего авторизацию.

---
### 2.5.1 HttpSecurity - объект для описания цепочки
Обычно нет необходимости знать порядок фильтров Spring Security.  
Однако бывают случаи, когда знание порядка полезно, если вы хотите его узнать, можете посмотреть код `FilterOrderRegistration`.

> [!IMPORTANT] Описание цепочки фильтров
> Для **описания ЦЕПОЧКИ ФИЛЬТРОВ**, используется объект `HttpSecurity` - через него и **описывается стандартный порядок фильтрации**

Эти **фильтры безопасности чаще всего объявляются с использованием экземпляра** `HttpSecurity`.  
Чтобы проиллюстрировать вышесказанное, рассмотрим следующую конфигурацию безопасности:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(
	    HttpSecurity http) throws Exception 
	{
        http
            .csrf(Customizer.withDefaults())
            .httpBasic(Customizer.withDefaults())
            .formLogin(Customizer.withDefaults())
            .authorizeHttpRequests(
            (authorize) -> authorize
                .anyRequest().authenticated()
            );

        return http.build();
    }

}
```

Приведенная **выше конфигурация приведет к следующему порядку фильтров**:

|Фильтр|Добавлен через|
|---|---|
|`CsrfFilter`|`HttpSecurity#csrf`|
|`UsernamePasswordAuthenticationFilter`|`HttpSecurity#formLogin`|
|`BasicAuthenticationFilter`|`HttpSecurity#httpBasic`|
|`AuthorizationFilter`|`HttpSecurity#authorizeHttpRequests`|
1. Сначала вызывается `CsrfFilter` для защиты от CSRF-атак.
    
2. Во-вторых, вызываются фильтры аутентификации для аутентификации запроса.
    
3. В-третьих, вызывается `AuthorizationFilter` для авторизации запроса.

Могут быть и другие экземпляры `Filter`, которые не перечислены выше.  
Если вы хотите увидеть список фильтров, вызываемых для конкретного запроса, вы можете их распечатать.

### 2.5.2 Вывод списка фильтров безопасности
Часто бывает полезно увидеть список фильтров безопасности (`Filters`), которые вызываются для конкретного запроса.  

Например, вы хотите убедиться, что добавленный вами фильтр находится в списке фильтров безопасности.

==Список фильтров выводится на уровне DEBUG== при запуске приложения, поэтому вы можете увидеть что-то подобное в консоли:
```bash
2023-06-14T08:55:22.321-03:00  DEBUG 76975 --- [           main] o.s.s.web.DefaultSecurityFilterChain     : 
Will secure any request with [ 
	DisableEncodeUrlFilter, 
	WebAsyncManagerIntegrationFilter, 
	SecurityContextHolderFilter, 
	HeaderWriterFilter, 
	CsrfFilter, 
	LogoutFilter, 
	UsernamePasswordAuthenticationFilter, 
	DefaultLoginPageGeneratingFilter, 
	DefaultLogoutPageGeneratingFilter, 
	BasicAuthenticationFilter, 
	RequestCacheAwareFilter, 
	SecurityContextHolderAwareRequestFilter, 
	AnonymousAuthenticationFilter, 
	ExceptionTranslationFilter, 
	AuthorizationFilter
]
```

#### **DisableEncodeUrlFilter**
Отключает кодирование URL (добавление `jsessionid` в URL). Это нужно для безопасности, чтобы ID сессии не «светился» в ссылках и логах сервера, что снижает риск атаки Session Fixation.

#### **WebAsyncManagerIntegrationFilter**
Интегрирует `SecurityContext` (контекст безопасности) с `WebAsyncManager` Spring MVC. Это позволяет данным о пользователе (`Principal`) корректно передаваться в асинхронные потоки (`@Async`, `Callable`).

#### **SecurityContextHolderFilter**
**Ключевой фильтр.** Он достает данные аутентификации из `HttpSession` (через репозиторий) и кладет их в `SecurityContextHolder` (ThreadLocal). Благодаря ему все последующие фильтры и контроллеры знают, кто текущий пользователь.

#### **HeaderWriterFilter**
Добавляет в HTTP-ответ защитные заголовки. Например: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security`.

#### **CsrfFilter**
Проверяет наличие специального токена в запросах, изменяющих состояние (POST, PUT, DELETE). Если токена нет — запрос блокируется (403). Защищает от межсайтовой подделки запросов.

#### **LogoutFilter**
Обрабатывает запрос на ВЫХОД из системы (обычно URL `/logout`). Уничтожает сессию, очищает `SecurityContext` и перенаправляет на страницу входа.

#### **UsernamePasswordAuthenticationFilter**
Обрабатывает данные из стандартной HTML-формы входа (`username` и `password`). Если логин/пароль верны, создает объект `Authentication` и сохраняет пользователя в сессии.

#### **DefaultLoginPageGeneratingFilter**
Генерирует ту самую стандартную (некрасивую) HTML-страницу логина Spring Security, если вы не создали свою собственную.

#### **DefaultLogoutPageGeneratingFilter**
Генерирует стандартную страницу подтверждения выхода (logout).

#### **BasicAuthenticationFilter**
Обрабатывает HTTP Basic Authentication. Ищет в запросе заголовок `Authorization: Basic <base64>`. Используется для REST API или stateless-сервисов.

#### **RequestCacheAwareFilter**
Проверяет, был ли текущий запрос прерван из-за отсутствия аутентификации. Если пользователь только что залогинился, этот фильтр восстанавливает сохраненный URL (например, `/secret-page`) и перенаправляет туда вместо главной страницы.

#### **SecurityContextHolderAwareRequestFilter**
Оборачивает стандартный `HttpServletRequest` в обертку, добавляющую удобные методы безопасности, например `request.isUserInRole("ADMIN")` или `request.getUserPrincipal()`.

#### **AnonymousAuthenticationFilter**
Если **пользователь еще не залогинился**, этот ф**ильтр создает для него фиктивную аутентификацию** — `AnonymousAuthenticationToken` (с ролью `ROLE_ANONYMOUS`). Это нужно, чтобы в коде не было проверок на `null` для контекста.

#### **ExceptionTranslationFilter**
Перехватывает исключения `AccessDeniedException` и `AuthenticationException`, возникающие в фильтрах _ниже_ по цепочке. Если пользователь не вошел — запускает процесс входа, если прав не хватает — показывает 403 ошибку.

#### **AuthorizationFilter**
**Последний рубеж.** Проверяет, имеет ли текущий пользователь право доступа к запрашиваемому URL в соответствии с правилами из `.authorizeHttpRequests()`.

---
И это **даст довольно хорошее представление о фильтрах безопасности**, настроенных для каждой цепочки фильтров.

Но это еще не все. Вы также ***можете настроить свое приложение на вывод вызова каждого отдельного фильтра для каждого запроса.*** 

Это полезно, чтобы увидеть, вызывается ли добавленный вами фильтр для конкретного запроса, или чтобы проверить, откуда исходит исключение. 

> Для *этого вы можете* ***настроить приложение на логирование событий безопасности***.

### 2.5.3 Добавление фильтров в цепочку фильтров
В большинстве случаев стандартных фильтров безопасности достаточно для обеспечения безопасности вашего приложения. 

Однако могут возникнуть ситуации, когда **вы захотите добавить КАСТОМНЫЙ** `Filter` в `SecurityFilterChain`.

`HttpSecurity` предоставляет три метода для добавления фильтров:
- `#addFilterBefore(Filter, Class<?>)` — добавляет ваш фильтр перед другим фильтром.
    
- `#addFilterAfter(Filter, Class<?>)` — добавляет ваш фильтр после другого фильтра.
    
- `#addFilterAt(Filter, Class<?>)` — заменяет другой фильтр вашим фильтром.

---
#### 1. Добавление пользовательского фильтра
Если вы создаете собственный фильтр, вам нужно будет определить его местоположение в цепочке фильтров. 

Обратите внимание на следующие ключевые события, происходящие в цепочке фильтров:
- **Событие 1: `SecurityContextHolderFilter`**
    - **Что происходит:** Данные о пользователе **загружены** из сессии и готовы к использованию.
        
    - **Зачем это знать:** Если вашему кастомному фильтру нужно знать, кто пользователь (например, `UserDetails`), вы **должны** разместить свой фильтр **после** него.
     
- **Событие 2: `LogoutFilter`**
    - **Что происходит:** Проверка на запрос `/logout` уже **выполнена**. 
      Если это запрос на выход, он уже обработан. Все фильтры после него гарантированно работают в рамках живой сессии пользователя (или анонимного пользователя).
        
    - **Зачем это знать:** Если ваш фильтр предполагает работу с аутентифицированным пользователем (например, запись в лог действий), его нужно ставить **после** `LogoutFilter`. Если поставить до — фильтр может сработать даже на запрос выхода.
      
- **Событие 3: `AnonymousAuthenticationFilter`**
    - **Что происходит:** Точно известно, что в `SecurityContext` есть объект `Authentication`. Либо настоящий пользователь, либо аноним (`anonymousUser`). **Больше не бывает `null`** в контексте.
        
    - **Зачем это знать:** Это идеальное место для фильтров авторизации (проверки доступа к бизнес-логике). Вам не нужно писать `if (auth != null)`, вы точно знаете, что пользователь уже определен.

Подумайте, какие события должны произойти до вашего фильтра, чтобы определить его место. 

Вот эмпирическое правило:

|Если ваш фильтр является...|То разместите его после...|Поскольку эти события уже произошли...|
|---|---|---|
|Фильтр защиты от эксплойтов|`SecurityContextHolderFilter`|1|
|Фильтр аутентификации|`LogoutFilter`|1, 2|
|Фильтр авторизации|`AnonymousAuthenticationFilter`|1, 2, 3|

**Чаще всего приложения добавляют пользовательскую аутентификацию.** Это означает, что фильтр следует размещать после `LogoutFilter`.

##### ПРИМЕР - Создание фильтра
> Предположим, вы хотите добавить `Filter`, который получает заголовок с ID арендатора (tenant id) и проверяет, есть ли у текущего пользователя доступ к этому арендатору.

Сначала создадим `Filter`:
```java
import java.io.IOException;

import jakarta.servlet.Filter;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.ServletRequest;
import jakarta.servlet.ServletResponse;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

import org.springframework.security.access.AccessDeniedException;

public class TenantFilter implements Filter {

    @Override
    public void doFilter(
	    ServletRequest servletRequest, 
	    ServletResponse servletResponse, 
	    FilterChain filterChain
	)throws IOException, ServletException 
	{
        HttpServletRequest request = 
	        (HttpServletRequest) servletRequest;
	        
        HttpServletResponse response = 
	        (HttpServletResponse) servletResponse;

        String tenantId = 
	        request.getHeader("X-Tenant-Id"); // (1)
        
        boolean hasAccess = 
	        isUserAllowed(tenantId); // (2)
        
        if (hasAccess) {
            filterChain
	            .doFilter(request, response); // (3)
            return;
        }
        
        throw 
        new AccessDeniedException("Access denied"); // (4)
    }
}
```

Пример кода выше делает следующее:

|№|Описание|
|---|---|
|(1)|Получает ID арендатора из заголовка запроса.|
|(2)|Проверяет, есть ли у текущего пользователя доступ к этому ID арендатора.|
|(3)|Если у пользователя есть доступ, вызывает остальные фильтры в цепочке.|
|(4)|Если у пользователя нет доступа, выбрасывает `AccessDeniedException`.|

**Вместо реализации** `Filter` вы **можете унаследоваться от** `OncePerRequestFilter` — это ==базовый класс для фильтров==, которые вызываются только один раз за запрос и предоставляют метод `doFilterInternal` с параметрами `HttpServletRequest` и `HttpServletResponse`.

##### ПРИМЕР - Добавление в цепочку
Теперь вам нужно добавить фильтр в `SecurityFilterChain`. 

Предыдущее описание уже дает нам подсказку, куда добавить фильтр: *поскольку нам нужно знать текущего пользователя, нам нужно добавить его после фильтров аутентификации.*

Основываясь на эмпирическом правиле, д**обавьте его после** `AnonymousAuthenticationFilter`, последнего фильтра аутентификации в цепочке, следующим образом:
```java
@Bean
SecurityFilterChain filterChain(
	HttpSecurity http
) throws Exception 
{
    http
        // ...
        .addFilterAfter(
	        new TenantFilter(),
	        AnonymousAuthenticationFilter.class
	    ); // (1)
    return http.build();
}
```

|№|Описание|
|---|---|
|(1)|Используйте `HttpSecurity#addFilterAfter`, чтобы добавить `TenantFilter` после `AnonymousAuthenticationFilter`.|

Добавляя фильтр после `AnonymousAuthenticationFilter`, **мы гарантируем**, что `TenantFilter` **будет вызван после фильтров аутентификации.**

Вот и все. Теперь `TenantFilter` будет вызываться в цепочке фильтров и проверять, есть ли у текущего пользователя доступ к ID арендатора.

#### 2. Объявление вашего фильтра как бина (Bean)
Когда вы объявляете `Filter` как Spring-бин (либо аннотируя его `@Component`, либо объявляя как бин в вашей конфигурации), *Spring Boot автоматически регистрирует его во встроенном контейнере.* 

Это **может привести к тому, что фильтр будет вызван дважды**: 
1. один раз контейнером 
2. и один раз Spring Security, причем в разном порядке.

> [!WARNING] Часто фильтры не регистрируют как бины
> По этой причине фильтры часто не являются Spring-бинами.

**Однако**, если вашему фильтру ***необходимо*** быть Spring-бином (например, для ***использования внедрения зависимостей***), вы **можете указать Spring Boot НЕ РЕГИСТРИРОВАТЬ ЕГО в контейнере**, объявив бин `FilterRegistrationBean` и установив его свойство `enabled` в `false`:
```JAVA
@Bean
public FilterRegistrationBean<TenantFilter> tenantFilterRegistration(TenantFilter filter) 
{
    FilterRegistrationBean<TenantFilter> registration = 
	    new FilterRegistrationBean<>(filter);
	    
    registration.setEnabled(false);
    
    return registration;
}
```

Это гарантирует, что только `HttpSecurity` будет добавлять этот фильтр.

#### 3. Настройка фильтра Spring Security
Как правило, для настройки фильтров Spring Security можно использовать DSL-метод фильтра. 

> **DSL (Domain-Specific Language)** — предметно-ориентированный язык.
> 
> В Spring Security DSL — это **текучий API (Fluent API)** для настройки `HttpSecurity`. 
> 
> - **Без DSL (гипотетически):** Вам пришлось бы вручную создавать `BasicAuthenticationFilter`, настраивать `AuthenticationManager`, вычислять правильную позицию в списке и добавлять его через `addFilterAt`. Это много кода.
> - **С DSL:** Вы просто пишете `.httpBasic()`, и Spring Boot сам создает нужный фильтр, настраивает его и вставляет в правильное место цепочки. **Метод `.httpBasic()` и есть DSL-метод.**

Например, самый простой способ добавить `BasicAuthenticationFilter` — это попросить DSL сделать это:
```java
@Bean
SecurityFilterChain filterChain(
	HttpSecurity http
) throws Exception {
	http
		.httpBasic(Customizer.withDefaults())
        // ...

	return http.build();
}
```

Однако, **если вам нужно сконструировать фильтр Spring Security самостоятельно**, вы ==указываете его в DSL==, используя `addFilterAt` ==следующим образом==:
```java
@Bean
SecurityFilterChain filterChain(
	HttpSecurity http
) throws Exception {
	BasicAuthenticationFilter basic = 
		new BasicAuthenticationFilter();
	// ... настроить

	http
		// ...
		.addFilterAt(
			basic, 
			BasicAuthenticationFilter.class
		);

	return http.build();
}
```

> [!WARNING] Замена стандартного фильтра кастомным 
> ```java
> .addFilterAt(basic, BasicAuthenticationFilter.class);
> ```
> «Вставь мой фильтр `basic` **на ту же позицию** в очереди, которую обычно занимает стандартный `BasicAuthenticationFilter`».
> 
> **Что произойдет?**
> 1. Стандартный фильтр `BasicAuthenticationFilter` **будет удален** из цепочки.
> 	
> 2. Вместо него на это же место встанет ваш кастомный экземпляр.

Обратите внимание, что если этот фильтр уже был добавлен, Spring Security выбросит исключение. Например, вызов `HttpSecurity#httpBasic` добавляет `BasicAuthenticationFilter` за вас. 

Таким образом, **следующая конфигурация завершится ОШИБКОЙ**, поскольку есть два вызова, которые оба пытаются добавить `BasicAuthenticationFilter`:
```java
@Bean
SecurityFilterChain filterChain(
	HttpSecurity http
) throws Exception {
	BasicAuthenticationFilter basic = 
		new BasicAuthenticationFilter();
	// ... настроить

	http
		.httpBasic(Customizer.withDefaults())
		// ... о нет! BasicAuthenticationFilter 
		// добавляется дважды!
		.addFilterAt(
			basic, 
			BasicAuthenticationFilter.class);

	return http.build();
}
```

В этом случае удалите вызов `httpBasic`, поскольку вы конструируете `BasicAuthenticationFilter` самостоятельно.

##### ОТКЛЮЧЕНИЕ ФИЛЬТРА
Если вы не можете перенастроить `HttpSecurity`, чтобы он не добавлял определенный фильтр, **вы обычно можете отключить фильтр** Spring Security, вызвав метод `disable` его DSL, например:
```java
.httpBasic((basic) -> basic.disable())
```

## 2.6 Обработка исключений безопасности
`ExceptionTranslationFilter` позволяет преобразовывать `AccessDeniedException` и `AuthenticationException` в HTTP-ответы.

`ExceptionTranslationFilter` вставляется в `FilterChainProxy` как один из фильтров безопасности.


На следующем изображении показана связь `ExceptionTranslationFilter` с другими компонентами:
![[Pasted image 20260416122334.png]]
1. Сначала `ExceptionTranslationFilter` вызывает `FilterChain.doFilter(request, response)`, чтобы вызвать остальную часть приложения.
    
2. Если **пользователь не аутентифицирован** или возникло `AuthenticationException`, то ***Начать аутентификацию (Start Authentication)***. 
	- `SecurityContextHolder` очищается.
        
    - `HttpServletRequest` сохраняется, чтобы его можно было использовать для повторного воспроизведения исходного запроса после успешной аутентификации.
        
    - `AuthenticationEntryPoint` используется для запроса учетных данных у клиента. 
      Например, он может перенаправить на страницу входа или отправить заголовок `WWW-Authenticate`.
        
3. В противном случае, если это `AccessDeniedException`, то **Доступ запрещен (Access Denied)**. Вызывается `AccessDeniedHandler` для обработки отказа в доступе.


Если приложение не выбрасывает `AccessDeniedException` или `AuthenticationException`, то `ExceptionTranslationFilter` ничего не делает.

Псевдокод для `ExceptionTranslationFilter` выглядит примерно так:

_Псевдокод ExceptionTranslationFilter_
```java
try {
	filterChain.doFilter(request, response); // (1)
} catch (
	AccessDeniedException | AuthenticationException ex
) {
	if (!authenticated || ex instanceof AuthenticationException) {
		startAuthentication(); // (2)
	} else {
		accessDenied(); // (3)
	}
}
```

| №   | Описание                                                                                                                                                                                                                                                                                                                                                           |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| (1) | Как описано в «Обзоре фильтров», вызов `FilterChain.doFilter(request, response)` эквивалентен вызову остальной части приложения. <br><br>Это означает, что если другая часть приложения (`AuthorizationFilter` или безопасность методов) выбрасывает `AuthenticationException` или `AccessDeniedException`, это исключение перехватывается и обрабатывается здесь. |
| (2) | Если пользователь не аутентифицирован или это `AuthenticationException`, начать аутентификацию.                                                                                                                                                                                                                                                                    |
| (3) | В противном случае, доступ запрещен.                                                                                                                                                                                                                                                                                                                               |

## 2.7 Сохранение запросов между аутентификациями

Как показано в разделе «Обработка исключений безопасности», *когда запрос не имеет аутентификации и запрашивает ресурс*, ***требующий аутентификации***, **необходимо сохранить запрос к защищенному ресурсу, чтобы повторно запросить его после успешной аутентификации.** 

> Если попытались обратиться к защищенному ресурсу без входа, то нужно:
> 1. Пользователя отправить на форму входа
> 2. Сохранить исходный запрос пользователя, и после его входа перенаправить его на ресурс

В Spring Security это делается путем сохранения `HttpServletRequest` с использованием реализации `RequestCache`.

### 2.7.1 RequestCache и RequestCacheAwareFilter
`HttpServletRequest` сохраняется в `RequestCache`. 

Когда пользователь успешно аутентифицируется, `RequestCache` **используется для воспроизведения исходного запроса.**  

`RequestCacheAwareFilter` использует `RequestCache` для получения сохраненного `HttpServletRequest` после аутентификации пользователя, в то время как `ExceptionTranslationFilter` использует `RequestCache` для сохранения `HttpServletRequest` после обнаружения `AuthenticationException`, перед перенаправлением пользователя на endpoint входа в систему.

По умолчанию используется `HttpSessionRequestCache`. 

***Код ниже демонстрирует***, как настроить ***реализацию*** `RequestCache` так, ***чтобы она проверяла*** `HttpSession` **на наличие сохраненного запроса только при наличии параметра** `continue`.

_RequestCache проверяет сохраненные запросы только при наличии параметра continue_
```java
@Bean
DefaultSecurityFilterChain springSecurity(
	HttpSecurity http
) throws Exception 
{
	HttpSessionRequestCache requestCache = 
		new HttpSessionRequestCache();
	requestCache.setMatchingRequestParameterName("continue");
	http
		// ...
		.requestCache((cache) -> cache
			.requestCache(requestCache)
		);
	return http.build();
}
```

```xml
<http auto-config="true">
	<!-- ... -->
	<request-cache ref="requestCache"/>
</http>

<b:bean id="requestCache" class="org.springframework.security.web.savedrequest.HttpSessionRequestCache"
	p:matchingRequestParameterName="continue"/>
```

#### 1. Предотвращение сохранения запроса
Существует ряд причин, по которым вы можете не захотеть сохранять неаутентифицированный запрос пользователя в сессии. 

Возможно, вы захотите перенести это хранилище в браузер пользователя или хранить его в базе данных. 

Или вы можете захотеть отключить эту функцию, поскольку всегда хотите перенаправлять пользователя на домашнюю страницу, а не на страницу, которую он пытался посетить до входа в систему.

Для этого вы можете использовать реализацию `NullRequestCache`.

_Предотвращение сохранения запроса_
```java
@Bean
SecurityFilterChain springSecurity(
	HttpSecurity http
) throws Exception 
{
    RequestCache nullRequestCache = 
	    new NullRequestCache();
	    
    http
        // ...
        .requestCache((cache) -> cache
            .requestCache(nullRequestCache)
        );
    return http.build();
}
```

```xml
<http auto-config="true">
	<!-- ... -->
	<request-cache ref="nullRequestCache"/>
</http>

<b:bean id="nullRequestCache" class="org.springframework.security.web.savedrequest.NullRequestCache"/>
```

## 2.8 Логирование
Spring Security **обеспечивает всестороннее логирование всех событий, связанных с безопасностью**, ***на уровнях DEBUG и TRACE***. 

Это может быть очень полезно при отладке вашего приложения, поскольку из соображений безопасности ==Spring Security не добавляет в тело ответа никаких деталей о том, почему запрос был отклонен==. 

> Если вы столкнулись с ошибкой 401 или 403, весьма вероятно, что ***вы найдете сообщение в логе***, которое поможет вам понять, что происходит.


**Рассмотрим пример**, когда пользователь пытается сделать POST-запрос к ресурсу, для которого включена защита CSRF, без CSRF-токена.  

Без логов пользователь увидит ошибку 403 без объяснения причины отклонения запроса.  
Однако, если вы включите логирование для Spring Security, вы увидите сообщение в логе, подобное этому:
