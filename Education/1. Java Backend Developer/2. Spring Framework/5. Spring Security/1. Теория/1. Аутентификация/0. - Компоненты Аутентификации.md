# 1. Аутентификация
Spring Security обеспечивает всестороннюю поддержку аутентификации. 

Мы начнем с обсуждения **общей архитектуры аутентификации сервлетов**. Как и следовало ожидать, этот раздел более абстрактный и описывает архитектуру без подробного обсуждения того, как она применяется к конкретным сценариям.

Если хотите, вы можете обратиться к разделу «Механизмы аутентификации» для ознакомления с конкретными способами аутентификации пользователей. Эти разделы фокусируются на конкретных способах аутентификации и ссылаются обратно на разделы архитектуры, чтобы описать, как работают конкретные сценарии.

## 1.1 **Механизмы аутентификации**
- **Имя пользователя и пароль** — как аутентифицироваться с помощью имени пользователя/пароля.
    
- **OAuth 2.0 Login** — вход с помощью OAuth 2.0 с OpenID Connect и нестандартным входом через OAuth 2.0 (например, GitHub).
    
- **SAML 2.0 Login** — вход с помощью SAML 2.0.
    
- **Central Authentication Server (CAS)** — поддержка центрального сервера аутентификации (CAS).
    
- **Remember Me** — как запомнить пользователя после истечения срока действия сессии.
    
- **JAAS Authentication** — аутентификация с помощью JAAS.
    
- **Pre-Authentication Scenarios** — аутентификация с помощью внешнего механизма, такого как SiteMinder или Java EE security, но с использованием Spring Security для авторизации и защиты от распространенных эксплойтов.
    
- **X509 Authentication** — аутентификация с помощью сертификатов X509.

# 2. Архитектура аутентификации сервлетов

## 2.1 Основные компоненты для аутентификации
Это обсуждение расширяет тему «Безопасность сервлетов: общая картина»([[1. - Архитектура Spring Security]]) и описывает основные архитектурные компоненты, которые Spring Security использует при аутентификации сервлетов. 

Если вам нужны конкретные сценарии, объясняющие, как эти части сочетаются друг с другом, обратитесь к разделам, посвященным конкретным механизмам аутентификации.

- **SecurityContextHolder** — `SecurityContextHolder` — это место, где Spring Security хранит информацию о том, кто аутентифицирован.
    
- **SecurityContext** — получается из `SecurityContextHolder` и содержит `Authentication` текущего аутентифицированного пользователя.
    
- **Authentication** — может быть входными данными для `AuthenticationManager`, чтобы **предоставить учетные данные**, *которые* **пользователь** ввел для аутентификации, или представлять текущего пользователя из `SecurityContext`.
    
- **GrantedAuthority** — полномочие, предоставленное принципалу в объекте `Authentication` (например, роли, области действия и т. д.).
    
- **AuthenticationManager** — API, определяющий, как фильтры Spring Security выполняют аутентификацию.
    
- **ProviderManager** — наиболее распространенная реализация `AuthenticationManager`.
    
- **AuthenticationProvider** — используется `ProviderManager` для выполнения определенного типа аутентификации.
    
- **Запрос учетных данных с помощью `AuthenticationEntryPoint`** — используется для запроса учетных данных у клиента (например, перенаправление на страницу входа, отправка ответа `WWW-Authenticate` и т. д.).
    
- **AbstractAuthenticationProcessingFilter** — базовый `Filter`, используемый для аутентификации. Он также дает хорошее представление о высокоуровневом процессе аутентификации и о том, как компоненты работают вместе.
  Обрабатывает запрос, после того как пользователь: 
	- Отправил НЕАУТЕНТИФИЦИРОННЫЙ запрос на ресурс
	- Его перекинуло на страницу входа (`AuthenticationEntryPoint` постарался)
	- Данные пользователя отправились на проверку в `AbstractAuthenticationProcessingFilter`

## 2.2 SecurityContextHolder
В ==основе модели аутентификации== Spring Security лежит `SecurityContextHolder`. 
- Он содержит `SecurityContext`.

![[Pasted image 20260416185900.png]]

> [!NOTE] SecurityContextHolder - основа аутентификации
> `SecurityContextHolder` — это место, где Spring Security хранит **информацию о том, кто аутентифицирован.** 
> 
> А инфа о аутентифицированных пользователях лежит в `SecurityContext`, который состоит из `Authentication`

Spring Security не важно, как заполняется `SecurityContextHolder`. Если он содержит значение, оно используется в качестве текущего аутентифицированного пользователя.

Самый простой способ указать, что пользователь аутентифицирован, — установить `SecurityContextHolder` напрямую:

_Установка SecurityContextHolder_
```java
SecurityContext context = 
	SecurityContextHolder.createEmptyContext(); // (1)

Authentication authentication =
    new TestingAuthenticationToken(
	    "username", 
	    "password", 
	    "ROLE_USER"); // (2)
	    
context.setAuthentication(authentication);

SecurityContextHolder.setContext(context); // (3)
```

| №   | Описание                                                                                                                                                                                                                                                                                                                                                                       |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| (1) | Мы начинаем с создания пустого `SecurityContext`. <br><br>Следует создавать новый экземпляр `SecurityContext` вместо использования <br>`SecurityContextHolder.getContext().setAuthentication(authentication)`, чтобы **избежать состояний гонки** в многопоточных средах.                                                                                                      |
| (2) | Затем мы создаем новый объект `Authentication`. <br><br>Spring Security не важно, какой тип реализации `Authentication` установлен в `SecurityContext`. <br><br>Здесь мы используем `TestingAuthenticationToken`, потому что он очень прост. <br><br>Более распространенный сценарий в production — `UsernamePasswordAuthenticationToken(userDetails, password, authorities)`. |
| (3) | Наконец, мы устанавливаем `SecurityContext` в `SecurityContextHolder`. Spring Security будет использовать эту информацию для авторизации.                                                                                                                                                                                                                                      |

Чтобы **получить информацию об аутентифицированном** принципале, обратитесь к `SecurityContextHolder`.

_Доступ к текущему аутентифицированному пользователю_
```java
SecurityContext context = 
	SecurityContextHolder.getContext();

Authentication authentication = 
	context.getAuthentication();

String username = authentication.getName();
Object principal = authentication.getPrincipal();

Collection<? extends GrantedAuthority> authorities = 
	authentication.getAuthorities();
```

==В Spring MVC== вы можете получить текущего принципала с помощью аннотации `@AuthenticationPrincipal`, а полный `SecurityContext` — с помощью `@CurrentSecurityContext`. 

> Для доступа через Servlet API используйте `HttpServletRequest#getRemoteUser`.

**По умолчанию** `SecurityContextHolder` **использует** `ThreadLocal` ***для хранения этих данных***, что **означает**, что `SecurityContext` **всегда доступен методам в том же потоке**, даже если `SecurityContext` не передается явно в качестве аргумента в эти методы. Использование `ThreadLocal` таким образом вполне безопасно, если вы позаботитесь об очистке потока после обработки запроса текущего принципала. 

`FilterChainProxy` в Spring Security гарантирует, что `SecurityContext` всегда очищается.


**Некоторые приложения не совсем подходят для использования** `ThreadLocal` из-за особенностей их работы с потоками. 

Например, клиент Swing ***может захотеть, чтобы все потоки в виртуальной машине*** Java **использовали один и тот же контекст безопасности**. 
Вы можете настроить `SecurityContextHolder` со стратегией при запуске, чтобы указать, как вы хотите хранить контекст. 
- Для автономного приложения вы бы использовали стратегию `SecurityContextHolder.MODE_GLOBAL`. 
	
- Другие приложения могут захотеть, чтобы потоки, порожденные защищенным потоком, также наследовали ту же идентичность безопасности. Этого можно достичь, используя `SecurityContextHolder.MODE_INHERITABLETHREADLOCAL`. 

Вы можете изменить режим со значения по умолчанию `SecurityContextHolder.MODE_THREADLOCAL` двумя способами. 
- Первый — установить системное свойство. 
- Второй — вызвать статический метод на `SecurityContextHolder`. 

Большинству приложений не требуется изменять значение по умолчанию. Однако, если вы это сделаете, ознакомьтесь с JavaDoc для `SecurityContextHolder`, чтобы узнать больше.

## 2.4 Authentication
**Интерфейс** `Authentication` служит двум основным целям в Spring Security:

1. **Входные данные для** `AuthenticationManager`, чтобы **предоставить учетные данные**, ***которые пользователь ввел для аутентификации***. При использовании в этом сценарии `isAuthenticated()` возвращает `false`.
    
2. **Представление текущего аутентифицированного пользователя.** Вы можете получить текущий `Authentication` из `SecurityContext`.

### 2.4.1 Состав Authentication
`Authentication` содержит:

- `principal`: ==Идентифицирует пользователя==. При аутентификации с именем пользователя/паролем это часто экземпляр `UserDetails`.
    
- `credentials`: ==Часто пароль==. Во многих случаях он очищается после аутентификации пользователя, чтобы гарантировать, что он не утечет.
    
- `authorities`: Экземпляры `GrantedAuthority` — это ==высокоуровневые разрешения==, предоставленные пользователю. 
  Два примера — роли и области действия (scopes).

Он также оснащен `AdditionalRequiredFactorsBuilder`, который позволяет изменять существующий экземпляр `Authentication` и потенциально объединять его с другим. 

Это полезно в сценариях, таких как взятие полномочий (authorities) из одного шага аутентификации, например, входа через форму, и применение их к другому, например, входу по одноразовому токену, как показано ниже:
```java
Authentication latestResult = 
	authenticationManager
		.authenticate(authenticationRequest);
		
Authentication previousResult = 
	SecurityContextHolder
		.getContext()
		.getAuthentication();
		
if (previousResult != null 
	&& previousResult.isAuthenticated()
) {
    latestResult = latestResult.toBuilder()
            .authorities((a) -> 
	            a.addAll(previousResult.getAuthorities()))
            .build();
}
```

## 2.5 `GrantedAuthority` - права
Экземпляры `GrantedAuthority` — это **высокоуровневые разрешения**, предоставленные пользователю. 
Два примера — роли и области действия (scopes).

Вы можете получить экземпляры `GrantedAuthority` из метода `Authentication.getAuthorities()`. Этот метод предоставляет коллекцию (`Collection`) объектов `GrantedAuthority`. 

> [!NOTE] GrantedAuthority - права
> `GrantedAuthority` — **это**, как несложно догадаться, **полномочие**, ***предоставленное принципалу***. 

**Аналогия: Паспорт (Authentication) и Виза (GrantedAuthority).**
1. **Аутентификация:** Вы предъявляете свои биометрические данные, и система устанавливает вашу личность: «Это гражданин Иванов Иван Иванович». 
   Система создает объект `Authentication` с `principal = "Иванов И.И."`. 
   На этом этапе (`isAuthenticated = false` в момент проверки) система еще не знает, что вам можно.
    
2. **Загрузка прав (внутри процесса аутентификации):** Система лезет в базу МВД и видит: у вас есть «Виза A» и «Виза B» (это `GrantedAuthority`). Система вкладывает список этих виз внутрь вашего объекта «Удостоверение личности» (`Authentication`).
    
3. **Финал аутентификации:** Теперь объект `Authentication` полностью заполнен: _кто_ вы и _что_ у вас есть с собой. Он помечается `isAuthenticated = true`.
    
4. **Авторизация:** Когда вы пытаетесь пройти в здание «Для своих», охранник смотрит уже готовое удостоверение и ищет в списке виз наличие `ROLE_EMPLOYEE`.


Такими полномочиями **обычно являются «роли»**, такие как `ROLE_ADMINISTRATOR` или `ROLE_HR_SUPERVISOR`. 

Эти роли позже настраиваются для веб-авторизации, авторизации методов и авторизации объектов предметной области. 
Другие части Spring Security интерпретируют эти полномочия и ожидают их наличия. 
> При использовании аутентификации на основе имени пользователя и пароля экземпляры `GrantedAuthority` обычно загружаются `UserDetailsService`.

**Обычно** объекты `GrantedAuthority` **являются разрешениями на уровне всего приложения.** 
> Они не привязаны к конкретному объекту предметной области. 

Таким образом, *у вас вряд ли будет* `GrantedAuthority` *для представления разрешения на доступ к объекту «Сотрудник» номер 54*, потому что если таких полномочий тысячи, вы быстро исчерпаете память (или, по крайней мере, аутентификация пользователя будет занимать много времени). 
Конечно, Spring Security специально разработан для обработки этого распространенного требования, но для этой цели вам следует использовать возможности безопасности объектов предметной области в проекте.

## 2.6 `AuthenticationManager` - интерфейс

> [!NOTE] AuthenticationManager - как происходит аутентификация
> `AuthenticationManager` — это API, который **определяет**, **как фильтры** Spring Security **выполняют аутентификацию.** 
> 
> Этот менеджер ==содержит список провайдеров аутентификации==, ***если его не переопределить, Spring предоставит default реализцию***

Возвращенный объект `Authentication` затем устанавливается в `SecurityContextHolder` контроллером (то есть экземплярами фильтров Spring Security), который вызвал `AuthenticationManager`. 

Если вы не интегрируетесь с экземплярами фильтров Spring Security, вы можете установить `SecurityContextHolder` напрямую, и вам не обязательно использовать `AuthenticationManager`.

Хотя реализация `AuthenticationManager` может быть любой, наиболее распространенной реализацией является `ProviderManager`.

---
## 2.7 `ProviderManager` - реализация по умолчанию

> [!NOTE] ProviderManager - реализация AuthenticationManager
> `ProviderManager` — наиболее часто используемая реализация `AuthenticationManager`. 

`ProviderManager` делегирует работу списку (`List`) экземпляров `AuthenticationProvider`. 

![[Pasted image 20260416192240.png]]

Каждый `AuthenticationProvider` **имеет возможность указать, что аутентификация должна быть успешной, неудачной** ***или что он не может принять решение и позволяет нижестоящему*** `AuthenticationProvider` принять решение. 

***Если ни один из настроенных экземпляров*** `AuthenticationProvider` ***не может выполнить аутентификацию***, **аутентификация завершается неудачей с исключением** `ProviderNotFoundException`, которое является специальным `AuthenticationException`, указывающим, что `ProviderManager` не был настроен для поддержки типа `Authentication`, который был в него передан.


На практике каждый `AuthenticationProvider` знает, как выполнить определенный тип аутентификации. 
> Например, 
> - **один** `AuthenticationProvider` **может проверять имя пользователя/пароль,** 
> - в то время как **другой может аутентифицировать утверждение SAML.** 
> 
> Это позволяет каждому `AuthenticationProvider` выполнять очень специфичный тип аутентификации, поддерживая при этом несколько типов аутентификации и предоставляя только один бин `AuthenticationManager`.

### 2.7.1 "By default" менеджер аутентификации при неудачах
`ProviderManager` также п***озволяет настроить необязательный родительский*** `AuthenticationManager`, **к которому происходит обращение в случае**, если ни один `AuthenticationProvider` **не может выполнить аутентификацию**. 

Родитель может быть любым типом `AuthenticationManager`, но часто это экземпляр `ProviderManager`.
![[Pasted image 20260416192307.png]]

Фактически, несколько экземпляров `ProviderManager` могут совместно использовать один и тот же родительский `AuthenticationManager`. 

Это довольно распространено в сценариях, где есть несколько экземпляров `SecurityFilterChain`, которые имеют некоторую общую аутентификацию (общий родительский `AuthenticationManager`), но также и различные механизмы аутентификации (разные экземпляры `ProviderManager`).

![[Pasted image 20260416192319.png]]

**По умолчанию** `ProviderManager` **пытается очистить любую конфиденциальную информацию об учетных данных из объекта** `Authentication`, возвращаемого успешным запросом аутентификации. 

Это предотвращает хранение информации, такой как пароли, в `HttpSession` дольше, чем необходимо.

### 2.7.2 `CredentialsContainer` - контейнер данных для аутентификации
Интерфейс `CredentialsContainer` играет решающую роль в процессе аутентификации. Он позволяет удалять информацию об учетных данных, когда она больше не нужна, тем самым повышая безопасность, гарантируя, что конфиденциальные данные не хранятся дольше необходимого.

Это может вызвать проблемы, если вы используете кэш пользовательских объектов, например, для повышения производительности в приложении без сохранения состояния (stateless). 

Если `Authentication` содержит ссылку на объект в кэше (например, экземпляр `UserDetails`), и его учетные данные удалены, аутентификация по кэшированному значению становится невозможной. 
Вам нужно учитывать это, если вы используете кэш. 

Очевидное решение — сначала создать копию объекта либо в реализации кэша, либо в `AuthenticationProvider`, который создает возвращаемый объект `Authentication`. 

В качестве альтернативы вы можете отключить свойство `eraseCredentialsAfterAuthentication` в `ProviderManager`. 
> См. Javadoc для класса `ProviderManager`.

## 2.8 `AuthenticationProvider` - выполняет определенную аутентификацию
Вы можете внедрить несколько экземпляров `AuthenticationProvider` в `ProviderManager`. 

Каждый `AuthenticationProvider` ==выполняет определенный тип аутентификации.== 

Например, 
- `DaoAuthenticationProvider` поддерживает аутентификацию на основе имени пользователя и пароля, 
- а `JwtAuthenticationProvider` поддерживает аутентификацию JWT-токена.

## 2.9 Запрос учетных данных с помощью `AuthenticationEntryPoint`

`AuthenticationEntryPoint` ==используется для отправки HTTP-ответа==, который запрашивает учетные данные у клиента.

### 2.9.1 КЛИЕНТ СРАЗУ ДОБАВИЛ СВОИ КРЕДЫ В ЗАПРОС
*Иногда клиент* проактивно *включает учетные данные* (такие как имя пользователя и пароль) *в запрос ресурса*. 
В этих случаях Spring Security не нужно предоставлять HTTP-ответ с запросом учетных данных у клиента, поскольку они уже включены.

### 2.9.2 КЛИЕНТ НЕ ДОБАВИЛ СВОИ КРЕДЫ В ЗАПРОС
В других случаях клиент делает неаутентифицированный запрос к ресурсу, на доступ к которому у него нет прав. 

В этом случае реализация `AuthenticationEntryPoint` **используется для запроса учетных данных у клиента.** 

Реализация `AuthenticationEntryPoint` **может выполнить перенаправление на страницу входа**, ответить заголовком `WWW-Authenticate` или предпринять другое действие.

## 3. AbstractAuthenticationProcessingFilter - базовый фильтр аутентификации
`AbstractAuthenticationProcessingFilter` **используется в качестве базового фильтра** (`Filter`) **для аутентификации учетных** данных пользователя. 

> `UsernamePasswordAuthenticationFilter` - реализация `AbstractAuthenticationProcessingFilter` 

***Прежде чем учетные*** данные могут быть ***аутентифицированы***, **Spring Security обычно запрашивает их с помощью** `AuthenticationEntryPoint`.

Затем `AbstractAuthenticationProcessingFilter` может аутентифицировать любые запросы аутентификации, которые ему передаются.

> Короче, когда пользователь пытается получить доступ к защищенному ресурсу, то Spring Security сначала попытается запросить данные входа у клиента, когда клиент введет их, то запрос попадет в AbstractAuthenticationProcessingFilter
![[Pasted image 20260416192428.png]]
1. Когда пользователь отправляет свои учетные данные, `AbstractAuthenticationProcessingFilter` создает объект `Authentication` из `HttpServletRequest`, который должен быть аутентифицирован. 
   Тип создаваемого `Authentication` зависит от подкласса `AbstractAuthenticationProcessingFilter`. Например, `UsernamePasswordAuthenticationFilter` создает `UsernamePasswordAuthenticationToken` из имени пользователя и пароля, отправленных в `HttpServletRequest`.
    
2. Затем `Authentication` передается в `AuthenticationManager` для аутентификации.
    

**Если аутентификация не удалась, то «Неудача» (Failure):**
1. `SecurityContextHolder` очищается.
    
2. Вызывается `RememberMeServices.loginFail`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. пакет `rememberme`
    
3. Вызывается `AuthenticationFailureHandler`. См. интерфейс `AuthenticationFailureHandler`.
    

**Если аутентификация успешна, то «Успех» (Success):**
1. `SessionAuthenticationStrategy` уведомляется о новом входе в систему. См. интерфейс `SessionAuthenticationStrategy`.
    
2. Любая уже аутентифицированная `Authentication` в `SecurityContextHolder` загружается, и ее полномочия (`authorities`) добавляются к возвращенной `Authentication`.
    
3. `Authentication` устанавливается в `SecurityContextHolder`. Позже, если вам нужно сохранить `SecurityContext`, чтобы он мог автоматически устанавливаться для будущих запросов, необходимо явно вызвать `SecurityContextRepository#saveContext`. См. класс `SecurityContextHolderFilter`.
    
4. Вызывается `RememberMeServices.loginSuccess`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. пакет `rememberme`
    
5. `ApplicationEventPublisher` публикует событие `InteractiveAuthenticationSuccessEvent`.
    
6. Вызывается `AuthenticationSuccessHandler`. См. интерфейс `AuthenticationSuccessHandler`.
