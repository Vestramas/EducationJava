Spring Security предоставляет следующие встроенные **механизмы для чтения имени пользователя и пароля из** `HttpServletRequest`:
- Form
- Bacis
- Digest

---
# 1. Вход через форму (Form Login)
Spring Security обеспечивает поддержку предоставления имени пользователя и пароля через HTML-форму. 

В этом разделе подробно описывается, как работает аутентификация на основе формы в Spring Security.

Сначала мы видим, как пользователь перенаправляется на форму входа:

![[Pasted image 20260416201444.png]]
**Рисунок 1. Перенаправление на страницу входа**

Предыдущий рисунок основан на нашей диаграмме `SecurityFilterChain`.

1. Сначала пользователь делает ***неаутентифицированный*** запрос к ресурсу (`/private`), на который у него нет прав.
    
2. `AuthorizationFilter` Spring Security указывает, что неаутентифицированный запрос **Запрещен (Denied)**, выбрасывая `AccessDeniedException`.
    
3. Поскольку пользователь не аутентифицирован, `ExceptionTranslationFilter` инициирует **Начало аутентификации (Start Authentication)** и отправляет перенаправление на страницу входа с помощью настроенного `AuthenticationEntryPoint`. 
 >  В большинстве случаев `AuthenticationEntryPoint` является экземпляром `LoginUrlAuthenticationEntryPoint`.
    
4. Браузер запрашивает страницу входа, на которую он был перенаправлен.
    
5. Что-то внутри приложения должно отобразить страницу входа.

Когда имя пользователя и пароль отправлены, `UsernamePasswordAuthenticationFilter` создает `UsernamePasswordAuthenticationToken` (который является типом `Authentication`), извлекая имя пользователя и пароль из экземпляра `HttpServletRequest`. 

`UsernamePasswordAuthenticationFilter` **расширяет** `AbstractAuthenticationProcessingFilter`, поэтому следующая диаграмма должна выглядеть довольно похоже:

![[Pasted image 20260416201501.png]]
**Рисунок 2. Аутентификация имени пользователя и пароля**

Рисунок основан на нашей диаграмме `SecurityFilterChain`.

1. Когда пользователь отправляет свое имя пользователя и пароль, `UsernamePasswordAuthenticationFilter` создает `UsernamePasswordAuthenticationToken` (**тип** `Authentication`), **извлекая имя пользователя** **и пароль** из экземпляра `HttpServletRequest`.
    
2. Затем `UsernamePasswordAuthenticationToken` передается в экземпляр `AuthenticationManager` для аутентификации. 
   Детали того, как выглядит `AuthenticationManager`, зависят от того, как хранится информация о пользователе.

**Если аутентификация не удалась, то «Неудача» (Failure):**
1. `SecurityContextHolder` очищается.
    
2. Вызывается `RememberMeServices.loginFail`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. интерфейс `RememberMeServices` в Javadoc.
    
3. Вызывается `AuthenticationFailureHandler`. См. класс `AuthenticationFailureHandler` в Javadoc.
    

**Если аутентификация успешна, то «Успех» (Success):**
1. `SessionAuthenticationStrategy` уведомляется о новом входе в систему. См. интерфейс `SessionAuthenticationStrategy` в Javadoc.
    
2. `Authentication` устанавливается в `SecurityContextHolder`. См. класс `SecurityContextPersistenceFilter` в Javadoc.
    
3. Вызывается `RememberMeServices.loginSuccess`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. интерфейс `RememberMeServices` в Javadoc.
    
4. `ApplicationEventPublisher` публикует событие `InteractiveAuthenticationSuccessEvent`.
    
5. Вызывается `AuthenticationSuccessHandler`. Обычно это `SimpleUrlAuthenticationSuccessHandler`, который перенаправляет на запрос, сохраненный `ExceptionTranslationFilter` при перенаправлении на страницу входа.


**По умолчанию вход через форму в Spring Security включен.** 
> Однако, как только предоставляется любая конфигурация на основе сервлетов, *вход через форму должен быть предоставлен явно*.

Следующий пример показывает минимальную, явную Java-конфигурацию:

**Вход через форму (Form Login)**
```java
public SecurityFilterChain filterChain(HttpSecurity http) {
	http
		.formLogin(withDefaults());
	// ...
}
```

В предыдущей конфигурации Spring Security ==отображает страницу входа по умолчанию==. 

Большинству production-приложений **требуется пользовательская форма входа.**

Следующая конфигурация демонстрирует, как предоставить пользовательскую форму входа.

**Конфигурация пользовательской формы входа**
```java
public SecurityFilterChain filterChain(HttpSecurity http) {
	http
		.formLogin((form) -> form
			.loginPage("/login")
			.permitAll()
		);
	// ...
}
```

Когда страница входа указана в конфигурации Spring Security, **вы сами отвечаете за отображение этой страницы**. Следующий шаблон Thymeleaf создает HTML-форму входа, которая соответствует странице входа `/login`:

**Форма входа — `src/main/resources/templates/login.html`**
```html
<!DOCTYPE html>
<html xmlns="http://www.w3.org/1999/xhtml" xmlns:th="https://www.thymeleaf.org">
	<head>
		<title>Please Log In</title>
	</head>
	<body>
		<h1>Please Log In</h1>
		<div th:if="${param.error}">
			Invalid username and password.</div>
		<div th:if="${param.logout}">
			You have been logged out.</div>
		<form th:action="@{/login}" method="post">
			<div>
			<input type="text" name="username" placeholder="Username"/>
			</div>
			<div>
			<input type="password" name="password" placeholder="Password"/>
			</div>
			<input type="submit" value="Log in" />
		</form>
	</body>
</html>
```

Вот несколько ключевых моментов о стандартной HTML-форме:
- **Форма должна выполнять** `post` на `/login`.
    
- Форма *должна включать CSRF-токен*, который автоматически включается Thymeleaf.
    
- Форма должна указывать имя пользователя в параметре с именем `username`.
    
- Форма должна указывать пароль в параметре с именем `password`.
    
- Если найден HTTP-параметр с именем `error`, это указывает на то, что пользователь не смог предоставить действительное имя пользователя или пароль.
    
- Если найден HTTP-параметр с именем `logout`, это указывает на то, что пользователь успешно вышел из системы.

Многим пользователям не нужно ничего больше, чем кастомизировать страницу входа. Однако, при необходимости, вы можете кастомизировать все, что было показано ранее, с помощью дополнительной конфигурации.

Если вы используете Spring MVC, вам нужен контроллер, который сопоставляет `GET /login` с созданным нами шаблоном входа. 

Следующий пример показывает минимальный `LoginController`:
```java
@Controller
class LoginController {
	@GetMapping("/login")
	String login() {
		return "login";
	}
}
```

# 2. Basic Authentication
Этот раздел предоставляет подробную информацию о том, как Spring Security обеспечивает поддержку базовой HTTP-аутентификации (==Basic HTTP Authentication==) для приложений на основе сервлетов. 

![[Pasted image 20260416202704.png]]

Сначала мы видим, **как заголовок** `WWW-Authenticate` **отправляется обратно неаутентифицированному клиенту.**

Предыдущий рисунок основан на нашей диаграмме `SecurityFilterChain`.

1. Сначала пользователь делает неаутентифицированный запрос к ресурсу `/private`, на который у него нет прав.
    
2. `AuthorizationFilter` Spring Security указывает, что неаутентифицированный запрос **Запрещен (Denied)**, выбрасывая `AccessDeniedException`.
    
3. Поскольку пользователь не аутентифицирован, `ExceptionTranslationFilter` инициирует **Начало аутентификации (Start Authentication)**. Настроенный `AuthenticationEntryPoint` является экземпляром `BasicAuthenticationEntryPoint`, который отправляет заголовок `WWW-Authenticate`. `RequestCache`, как правило, является `NullRequestCache`, который не сохраняет запрос, поскольку клиент способен повторно воспроизвести запросы, которые он изначально запрашивал.

> **Примечание:** Провайдер HTTP Basic Auth по умолчанию подавляет как тело ответа, так и заголовок `WWW-Authenticate` в ответе 401, если запрос был сделан с заголовком `X-Requested-With: XMLHttpRequest`. 
> 
> Это позволяет фронтендам реализовывать свой собственный код аутентификации вместо того, чтобы вызывать диалог входа в браузере. Чтобы переопределить это поведение, реализуйте свой собственный `BasicAuthenticationEntryPoint`.

Когда клиент получает заголовок `WWW-Authenticate`, он знает, что должен повторить попытку с именем пользователя и паролем. 

На следующем изображении показан процесс обработки имени пользователя и пароля:

![[Pasted image 20260416202727.png]]
**Рисунок 2. Аутентификация имени пользователя и пароля**

Предыдущий рисунок основан на нашей диаграмме `SecurityFilterChain`.
1. Когда пользователь отправляет свое имя пользователя и пароль, `BasicAuthenticationFilter` создает `UsernamePasswordAuthenticationToken` (тип `Authentication`), извлекая имя пользователя и пароль из `HttpServletRequest`.
    
2. Затем `UsernamePasswordAuthenticationToken` передается в `AuthenticationManager` для аутентификации. Детали того, как выглядит `AuthenticationManager`, зависят от того, как хранится информация о пользователе.

**Если аутентификация не удалась, то «Неудача» (Failure):**
1. `SecurityContextHolder` очищается.
    
2. Вызывается `RememberMeServices.loginFail`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. интерфейс `RememberMeServices` в Javadoc.
    
3. Вызывается `AuthenticationEntryPoint`, чтобы снова инициировать отправку заголовка `WWW-Authenticate`. См. интерфейс `AuthenticationEntryPoint` в Javadoc.


**Если аутентификация успешна, то «Успех» (Success):**
1. Любая уже аутентифицированная `Authentication` в `SecurityContextHolder` загружается, и ее полномочия (`authorities`) добавляются к возвращенной `Authentication`.
    
2. Вызывается `RememberMeServices.loginSuccess`. Если функция «запомнить меня» не настроена, это холостая операция (no-op). См. интерфейс `RememberMeServices` в Javadoc.
    
3. `BasicAuthenticationFilter` вызывает `FilterChain.doFilter(request,response)`, чтобы продолжить выполнение остальной логики приложения. См. класс `BasicAuthenticationFilter` в Javadoc.


==По умолчанию поддержка базовой HTTP-аутентификации в Spring Security включена==. 
Однако, *как только предоставляется любая конфигурация* на основе сервлетов, **HTTP Basic ДОЛЖЕН быть ПРЕДОСТАВЛЕН ЯВНО.**

Следующий пример показывает минимальную, явную конфигурацию:
**Явная конфигурация HTTP Basic**
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) {
	http
		// ...
		.httpBasic(withDefaults());
	return http.build();
}
```

# 3. Digest Authentication)
Этот раздел предоставляет подробную информацию о том, как Spring Security обеспечивает поддержку дайджест-аутентификации, которая реализуется с помощью `DigestAuthenticationFilter`.

> **⚠️ Внимание:** Вам **не следует** использовать дайджест-аутентификацию в современных приложениях, поскольку она **не считается безопасной.** 
> 
> Наиболее очевидная проблема заключается в том, что вы должны хранить свои пароли в виде обычного текста, либо в зашифрованном формате или формате MD5. 
> 
> Все эти форматы хранения считаются небезопасными. 
> 
> Вместо этого **следует хранить учетные данные с использованием** одностороннего адаптивного ***хеширования паролей*** (bCrypt, PBKDF2, SCrypt и другие), **что не поддерживается дайджест-аутентификацией.**

Дайджест-аутентификация пытается решить многие слабые стороны базовой (Basic) аутентификации, в частности, гарантируя, что учетные данные никогда не передаются по сети в виде открытого текста. Многие браузеры поддерживают дайджест-аутентификацию.

Стандарт, регулирующий HTTP Digest Authentication, определен в RFC 2617, который обновляет более раннюю версию стандарта дайджест-аутентификации, описанную в RFC 2069. Большинство пользовательских агентов реализуют RFC 2617. Поддержка дайджест-аутентификации в Spring Security совместима с качеством защиты (qop) "auth", предписанным RFC 2617, что также обеспечивает обратную совместимость с RFC 2069. 

Дайджест-аутентификация рассматривалась как более привлекательный вариант, если вам нужно использовать незашифрованный HTTP (без TLS или HTTPS) и вы хотите максимально обезопасить процесс аутентификации. **Однако всем следует использовать HTTPS.**

Центральным элементом дайджест-аутентификации является «nonce». Это значение, которое генерирует сервер. Nonce в Spring Security использует следующий формат:

**Синтаксис дайджеста**
```bash
base64(
expirationTime + ":" + md5Hex(expirationTime + ":" + key))

expirationTime:   Дата и время истечения срока действия nonce, выраженные в миллисекундах

key:              Приватный ключ для предотвращения изменения токена nonce
```

Вам необходимо убедиться, что вы настроили **небезопасное хранение паролей в виде обычного текста**, используя `NoOpPasswordEncoder`. (См. класс `NoOpPasswordEncoder` в Javadoc.) Ниже приведен пример настройки дайджест-аутентификации с помощью Java-конфигурации:

**Дайджест-аутентификация**
```java
@Autowired
UserDetailsService userDetailsService;

DigestAuthenticationEntryPoint authenticationEntryPoint() {
	DigestAuthenticationEntryPoint result = 
		new DigestAuthenticationEntryPoint();
	result.setRealmName("My App Realm");
	result.setKey("3028472b-da34-4501-bfd8-a355c42bdf92");
	return result;
}

DigestAuthenticationFilter digestAuthenticationFilter() {
	DigestAuthenticationFilter result = 
		new DigestAuthenticationFilter();
	result.setUserDetailsService(userDetailsService);
	
	result
	.setAuthenticationEntryPoint(
		authenticationEntryPoint()
	);
	return result;
}

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
	http
		// ...
		.exceptionHandling((e) -> 
			e.authenticationEntryPoint(
				authenticationEntryPoint()
			)
		)
		.addFilter(digestAuthenticationFilter());
	return http.build();
}
```
