# 1. Хранение паролей
**Интерфейс** `PasswordEncoder` в Spring Security используется **для выполнения одностороннего преобразования пароля**, чтобы пароль можно было безопасно хранить. 

Поскольку `PasswordEncoder` — это одностороннее преобразование, он не подходит, когда требуется двустороннее преобразование пароля (например, для хранения учетных данных, используемых для аутентификации в базе данных). 

Обычно `PasswordEncoder` используется **для хранения пароля**, который необходимо сравнить с паролем, предоставленным пользователем во время аутентификации.

## 1.1 История хранения паролей
На протяжении многих лет механизмы хранения паролей существенно эволюционировали. Изначально пароли хранились в открытом виде (plaintext), предполагая, что доступ к базе данных уже защищён. Однако с развитием атак, например SQL-инъекций, злоумышленники начали получать дампы данных с учетными записями, что показало ненадёжность такого подхода.

Следующим этапом стало использование односторонних хеш-функций (например, SHA-256). Система хранила только хеши паролей, что повышало безопасность. Однако злоумышленники начали применять радужные таблицы (Rainbow Tables) — заранее вычисленные соответствия паролей и их хешей, что позволило обходить защиту.

Для противодействия этому *стали использовать «соль» (salt)* — случайные данные, добавляемые к паролю перед хешированием. Это делало каждый хеш уникальным и значительно снижало эффективность радужных таблиц.

При использовании солёных паролей д*ля каждого пользователя генерируется уникальная случайная последовательность байтов* — соль (salt), ***которая добавляется к паролю перед хешированием***. 
В **базе данных сохраняются сама соль и полученный хеш**, при этом исходный пароль не хранится. 
***При аутентификации сервер извлекает соль***, ***объединяет её с введённым пользователем паролем*** **и вычисляет хеш**, ***который затем сравнивается с сохранённым значением***. Такой подход обеспечивает уникальность хешей даже для одинаковых паролей и значительно снижает эффективность атак с использованием радужных таблиц.

Со временем стало очевидно, что обычные криптографические хеш-функции уже недостаточно безопасны, поскольку современное оборудование позволяет выполнять миллиарды операций хеширования в секунду и подбирать пароли перебором.

В настоящее время рекомендуется использовать адаптивные односторонние функции (bcrypt, PBKDF2, scrypt, Argon2), которые намеренно требуют значительных вычислительных ресурсов. Они поддерживают настройку «фактора работы», позволяющего увеличивать сложность вычислений по мере роста производительности оборудования. Оптимальным считается время проверки пароля около одной секунды.

Из-за высокой ресурсоёмкости таких функций проверка пароля при каждом запросе может снижать производительность системы. Поэтому на практике рекомендуется использовать краткосрочные учетные данные (например, сессии или токены), которые позволяют избежать частой проверки пароля без потери безопасности.

## 1.2 DelegatingPasswordEncoder
До Spring Security 5.0 `PasswordEncoder` по умолчанию был `NoOpPasswordEncoder`, который т**ребовал пароли в виде обычного текста.** 

Основываясь на разделе «История хранения паролей», вы могли бы ожидать, что теперь `PasswordEncoder` по умолчанию будет чем-то вроде `BCryptPasswordEncoder`. 

Однако это игнорирует три реальные проблемы:
1. Многие приложения используют старые кодировки паролей, которые нелегко мигрировать.
    
2. Лучшая практика хранения паролей снова изменится.
    
3. Как фреймворк, Spring Security не может часто вносить критические изменения (breaking changes).

Вместо этого Spring Security представляет `DelegatingPasswordEncoder`, который решает все эти проблемы путем:
- Обеспечения кодирования паролей с использованием текущих рекомендаций по хранению паролей.
    
- Возможности проверки паролей в современных и устаревших форматах.
    
- Возможности обновления кодировки в будущем.

### **АНАЛОГИЯ**
В офисе турникет, в 2010 году были карточки, а потом в 2020 NFC метки. Всем сотрудникам разом обновить пропуска не получится.
**`DelegatingPasswordEncoder`:** Это **универсальный считыватель**.  
Он смотрит на карточку (на пароль в базе данных):
1. Видит на карточке наклейку `{bcrypt}` -> Включает **сканер для новых NFC-меток**.
    
2. Видит на карточке наклейку `{noop}` -> Включает **старую магнитную головку**.
    
3. Если карточка без наклейки -> Говорит: «Ошибка! Я не знаю, как это читать, иди получи новую карточку».
---

Вы можете легко создать экземпляр `DelegatingPasswordEncoder` с помощью `PasswordEncoderFactories`:

**Создание DelegatingPasswordEncoder по умолчанию**
```java
PasswordEncoder passwordEncoder =
	PasswordEncoderFactories
		.createDelegatingPasswordEncoder();
```

В качестве альтернативы вы можете создать свой собственный экземпляр:

**Создание пользовательского DelegatingPasswordEncoder**
```java
String idForEncode = "bcrypt";
Map encoders = new HashMap<>();

encoders.put(
	idForEncode, //bcrypt
	new BCryptPasswordEncoder()
);
encoders.put(
	"noop", 
	NoOpPasswordEncoder.getInstance()
);
encoders.put(
	"pbkdf2", 
	Pbkdf2PasswordEncoder
		.defaultsForSpringSecurity_v5_5()
);
encoders.put(
	"pbkdf2@SpringSecurity_v5_8", 
	Pbkdf2PasswordEncoder
		.defaultsForSpringSecurity_v5_8()
);
encoders.put(
	"scrypt",
	 SCryptPasswordEncoder
		 .defaultsForSpringSecurity_v4_1()
);
encoders.put(
	"scrypt@SpringSecurity_v5_8", 
	SCryptPasswordEncoder
		.defaultsForSpringSecurity_v5_8()
);
encoders.put(
	"argon2", 
	Argon2PasswordEncoder
		.defaultsForSpringSecurity_v5_2()
);
encoders.put(
	"argon2@SpringSecurity_v5_8", 
	Argon2PasswordEncoder
		.defaultsForSpringSecurity_v5_8()
);
encoders.put(
	"sha256", 
	new StandardPasswordEncoder()
);

PasswordEncoder passwordEncoder =
	new DelegatingPasswordEncoder(idForEncode, encoders);
```

### 1.2.1 Формат хранения паролей
Общий формат пароля выглядит следующим образом:

**Формат хранения DelegatingPasswordEncoder**
```bash
{id}encodedPassword
```
Где 
- `id` — это **идентификатор**, который используется для поиска того, **какой** `PasswordEncoder` следует использовать, 
- а `encodedPassword` — это **исходный закодированный пароль** для выбранного `PasswordEncoder`. 

`id` должен находиться в начале пароля, начинаться с `{` и заканчиваться `}`. Если `id` не может быть найден, `id` устанавливается в `null`. 
Например, ниже может быть список паролей, закодированных с использованием разных значений `id`. Все исходные пароли — `password`.

**Пример закодированных паролей DelegatingPasswordEncoder**
```
{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG // (1)

{noop}password // (2)

{pbkdf2}5d923b44a6d129f3ddf3e3c8d29412723dcbde72445e8ef6bf3b508fbf17fa4ed4d6b99ca763d8dc // (3)

{sha256}97cde38028ad898ebc02e690819fa220e88c62e0699403e94fff291cfffaf8410849f27605abcbc0 // (5)
```

> Более того, большинство форматов легко определить злоумышленнику и без префикса. Например, пароли BCrypt часто начинаются с `$2a$`.

### 1.2.2 Кодирование паролей
Параметр `idForEncode`, переданный в конструктор, определяет, какой `PasswordEncoder` будет использоваться для кодирования паролей. 

В `DelegatingPasswordEncoder`, который мы создали ранее, это означает, что результат кодирования `password` делегируется `BCryptPasswordEncoder` и предваряется префиксом `{bcrypt}`. Конечный результат выглядит так, как показано в следующем примере:

**Пример кодирования DelegatingPasswordEncoder**
```
{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
```

### 1.2.3 Сопоставление паролей
Сопоставление основано на `{id}` и сопоставлении `id` с `PasswordEncoder`, предоставленным в конструкторе. 

Наш пример в разделе «Формат хранения паролей» дает рабочий пример того, как это делается. 

По умолчанию результат вызова `matches(CharSequence, String)` с паролем и `id`, который не сопоставлен (включая `null` `id`), приводит к `IllegalArgumentException`. 
> Это поведение можно настроить с помощью `DelegatingPasswordEncoder.setDefaultPasswordEncoderForMatches(PasswordEncoder)`.

Используя `id`, мы можем сопоставлять любые кодировки паролей, **но кодировать пароли, используя самую современную кодировку паролей.** 

Это важно, потому что, в отличие от шифрования, хеши паролей разработаны так, что нет простого способа восстановить исходный текст. Поскольку нет способа восстановить исходный текст, трудно мигрировать пароли. Хотя пользователям просто мигрировать `NoOpPasswordEncoder`, мы решили включить его по умолчанию, чтобы упростить начало работы.

### 1.2.4 Опыт начала работы (БЕЗ ХЕШИРОВАНИЯ ПРИ РАЗРАБОТКЕ)
**Если вы собираете демо** или пример, **тратить время на хеширование паролей** ваших пользователей ***немного обременительно.*** 

> ПРИ РАЗРАБОТКЕ НЕ ТРАТИТЬ ВРЕМЯ НА ХЕШИРОВАНИЕ ПАРОЛЕЙ

Существуют удобные механизмы, **чтобы упростить это**, но они все равно не предназначены для production.

**Пример withDefaultPasswordEncoder**
```java
UserDetails user = User.withDefaultPasswordEncoder()
		.username("user")
		.password("password")
		.roles("user")
		.build();
System.out.println(user.getPassword());
// {bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
```

Если вы создаете несколько пользователей, вы также можете повторно использовать строитель (builder):

**Повторное использование строителя withDefaultPasswordEncoder**
```JAVA
UserBuilder users = User.withDefaultPasswordEncoder();

UserDetails user = users
		.username("user")
		.password("password")
		.roles("USER")
		.build();
		
UserDetails admin = users
		.username("admin")
		.password("password")
		.roles("USER","ADMIN")
		.build();
```

Это действительно хеширует хранимый пароль, но пароли все равно остаются раскрытыми в памяти и в скомпилированном исходном коде. 

Поэтому это все еще не считается безопасным для production-среды. 
==Для production вы должны хешировать свои пароли извне.==
(То есть, чтобы пароль не был "зашит" в коде)

### 1.2.6 Устранение неполадок
Следующая ошибка возникает, к**огда один из хранимых паролей не имеет** `id`, как описано в разделе «Формат хранения паролей».

```java
java.lang.IllegalArgumentException: 
There is no PasswordEncoder mapped for the id "null"
	at org.springframework.security.crypto.password.DelegatingPasswordEncoder$UnmappedIdPasswordEncoder.matches(DelegatingPasswordEncoder.java:233)
	at org.springframework.security.crypto.password.DelegatingPasswordEncoder.matches(DelegatingPasswordEncoder.java:196)
```

Самый простой способ решить эту проблему — выяснить, как ваши пароли хранятся в настоящее время, и явно предоставить правильный `PasswordEncoder`.

Если вы мигрируете с Spring Security 4.2.x, вы можете вернуться к предыдущему поведению, предоставив бин `NoOpPasswordEncoder`.

В качестве альтернативы вы можете добавить ко всем своим паролям правильный `id` в качестве префикса и продолжать использовать `DelegatingPasswordEncoder`. 

Например, если вы используете BCrypt, вы бы мигрировали свой пароль с чего-то вроде:
```
$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
```
на
```
{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
```


Полный список сопоставлений смотрите в Javadoc для `PasswordEncoderFactories`.

## 1.3 BCryptPasswordEncoder
Реализация `BCryptPasswordEncoder` **использует широко поддерживаемый алгоритм bcrypt** ***для хеширования паролей.*** 

Чтобы сделать его более устойчивым к взлому паролей, ==bcrypt намеренно медленный==. 
> Как и другие адаптивные односторонние функции, его следует настраивать так, ***чтобы проверка пароля в вашей системе занимала около 1 секунды.*** 

Реализация `BCryptPasswordEncoder` по умолчанию использует стойкость (strength) 10, как указано в Javadoc `BCryptPasswordEncoder`. 

**Вам рекомендуется настраивать** и тестировать параметр стойкости в вашей собственной системе, **чтобы проверка пароля занимала примерно 1 секунду**.

**BCryptPasswordEncoder**
```java
// Создать кодировщик со стойкостью 16
BCryptPasswordEncoder encoder = 
	new BCryptPasswordEncoder(16);
	
String result = encoder.encode("myPassword");
assertTrue(encoder.matches("myPassword", result));
```

## 1.4 Argon2PasswordEncoder
Реализация `Argon2PasswordEncoder` использует **алгоритм Argon2** для хеширования паролей. 

Argon2 является победителем конкурса Password Hashing Competition. Чтобы противостоять взлому паролей на специализированном оборудовании, 

Argon2 — это намеренно медленный алгоритм, требующий большого объема памяти. Как и другие адаптивные односторонние функции, его следует настраивать так, чтобы проверка пароля в вашей системе занимала около 1 секунды. 

Текущая реализация `Argon2PasswordEncoder` требует BouncyCastle.

**Argon2PasswordEncoder**
```java
// Создать кодировщик со всеми настройками по умолчанию
Argon2PasswordEncoder encoder = 
	Argon2PasswordEncoder
		.defaultsForSpringSecurity_v5_8();
		
String result = encoder.encode("myPassword");
assertTrue(encoder.matches("myPassword", result));
```

## 1.5 Pbkdf2PasswordEncoder
Реализация `Pbkdf2PasswordEncoder` использует **алгоритм PBKDF2** для хеширования паролей. 

Чтобы противостоять взлому паролей, PBKDF2 является намеренно медленным алгоритмом. 

Как и другие адаптивные односторонние функции, его следует настраивать так, чтобы проверка пароля в вашей системе занимала около 1 секунды. 

Этот алгоритм — хороший выбор, когда требуется сертификация FIPS.

**Pbkdf2PasswordEncoder**
```java
// Создать кодировщик со всеми настройками по умолчанию
Pbkdf2PasswordEncoder encoder = 
	Pbkdf2PasswordEncoder
		.defaultsForSpringSecurity_v5_8();
		
String result = encoder.encode("myPassword");
assertTrue(encoder.matches("myPassword", result));
```

Чтобы противостоять взлому паролей на специализированном оборудовании, scrypt — это намеренно медленный алгоритм, требующий большого объема памяти. 

Как и другие адаптивные односторонние функции, его следует настраивать так, чтобы проверка пароля в вашей системе занимала около 1 секунды.

## 1.7 Другие PasswordEncoder'ы
Существует значительное количество других реализаций `PasswordEncoder`, которые существуют исключительно для обратной совместимости. 

Все они помечены как устаревшие (*deprecated*), чтобы указать, что они больше не считаются безопасными. 

Однако планов по их удалению нет, поскольку миграция существующих устаревших систем затруднительна.

## 1.8 Кодировщики паролей на основе Password4j
Spring Security 7.0 представляет ==альтернативные реализации кодировщиков паролей==, основанные **на библиотеке `Password4j`**. 

Эти кодировщики предоставляют дополнительные возможности для популярных алгоритмов хеширования и могут использоваться в качестве альтернатив существующим реализациям Spring Security.

> [!NOTE] Библиотека `Password4j`
> Библиотека Password4j — это ***криптографическая библиотека Java***, *которая фокусируется на хешировании паролей с поддержкой множества алгоритмов.*

Эти кодировщики особенно полезны, когда вам нужны специфические конфигурации алгоритмов или вы хотите использовать оптимизации Password4j.

Все кодировщики на основе Password4j потокобезопасны и могут совместно использоваться несколькими потоками.

### 1.8.1 Argon2Password4jPasswordEncoder
Реализация `Argon2Password4jPasswordEncoder` использует алгоритм Argon2 через библиотеку Password4j для хеширования паролей. 

Это предоставляет **альтернативу** встроенному в Spring Security `Argon2PasswordEncoder` **с другими параметрами конфигурации** и потенциальными характеристиками производительности.

Argon2 является победителем конкурса Password Hashing Competition и рекомендуется для новых приложений. 

Эта реализация использует поддержку Argon2 в Password4j, которая корректно **включает соль** в выходной хеш.

**Создание кодировщика с настройками по умолчанию:**
**Argon2Password4jPasswordEncoder**
```java
PasswordEncoder encoder = new 
	Argon2Password4jPasswordEncoder();
	
String result = encoder.encode("myPassword");
assertThat(
	encoder.matches("myPassword", result)).isTrue();
```

**Создание кодировщика с пользовательскими параметрами Argon2:**
**Argon2Password4jPasswordEncoder Custom**
```java
Argon2Function argon2Fn = 
Argon2Function
	.getInstance(
		65536, 
		3, 
		4, 
		32,
		Argon2.ID
	);
	
PasswordEncoder encoder = new 
	Argon2Password4jPasswordEncoder(argon2Fn);
	
String result = encoder.encode("myPassword");
assertThat(
	encoder.matches("myPassword", result)).isTrue();
```

### 1.8.4 Pbkdf2Password4jPasswordEncoder
Реализация `Pbkdf2Password4jPasswordEncoder` использует алгоритм PBKDF2 через библиотеку Password4j для хеширования паролей. Это предоставляет альтернативу встроенному в Spring Security `Pbkdf2PasswordEncoder` с явным управлением солью.

PBKDF2 — это функция формирования ключа, разработанная так, чтобы быть вычислительно затратной для противодействия атакам по словарю и методом грубой силы. Эта реализация управляет солью явно, поскольку реализация PBKDF2 в Password4j не включает соль в выходной хеш. Формат закодированного пароля: `{соль}:{хеш}`, где и соль, и хеш закодированы в Base64.

**Создание кодировщика с настройками по умолчанию:**
**Pbkdf2Password4jPasswordEncoder**
```java
PasswordEncoder encoder = new 
	Pbkdf2Password4jPasswordEncoder();

String result = encoder.encode("myPassword");
assertThat(encoder.matches("myPassword", result)).isTrue();
```

**Создание кодировщика с пользовательскими параметрами PBKDF2:**
**Pbkdf2Password4jPasswordEncoder Custom**
```java
PBKDF2Function pbkdf2Fn = 
	PBKDF2Function.getInstance(Hmac.SHA256, 100000, 256);

PasswordEncoder encoder = new 
	Pbkdf2Password4jPasswordEncoder(pbkdf2Fn);

String result = encoder.encode("myPassword");
assertThat(encoder.matches("myPassword", result)).isTrue();
```

### 1.8.5 BalloonHashingPassword4jPasswordEncoder

Balloon hashing — это алгоритм хеширования паролей **с требованием к памяти (memory-hard)**, разработанный для устойчивости как к атакам компромисса времени и памяти, так и к атакам по побочным каналам.

Эта **реализация управляет солью явно,** поскольку реализация Balloon hashing в Password4j не включает соль в выходной хеш.

**BalloonHashingPassword4jPasswordEncoder**
```java
PasswordEncoder encoder = new 
	BalloonHashingPassword4jPasswordEncoder();

	encoder.matches("myPassword", result)).isTrue();
```

## 1.9 Конфигурация хранения паролей
Spring Security **использует** `DelegatingPasswordEncoder` ==по умолчанию.== 

Однако вы можете настроить это, предоставив `PasswordEncoder` как Spring-бин.

Если вы мигрируете с Spring Security 4.2.x, вы можете вернуться к предыдущему поведению, предоставив бин `NoOpPasswordEncoder`.

> **⚠️ Внимание:** Возврат к `NoOpPasswordEncoder` не считается безопасным. 
> 
> Вместо этого вам следует перейти на использование `DelegatingPasswordEncoder` для поддержки безопасного кодирования паролей.

**NoOpPasswordEncoder**
```java
@Bean
public static NoOpPasswordEncoder passwordEncoder() {
    return NoOpPasswordEncoder.getInstance();
}
```
**Примечание:** Конфигурация XML требует, чтобы имя бина `NoOpPasswordEncoder` было `passwordEncoder`.

## 2.0 Конфигурация смены пароля - обновление пароля
**Большинство приложений**, позволяющих пользователю указывать пароль, также **требуют функции для обновления этого пароля.**

**Well-Known URL для смены паролей** указывает механизм, с помощью которого менеджеры паролей могут обнаружить конечную точку (endpoint) обновления пароля для данного приложения.

Вы можете настроить Spring Security для предоставления этой конечной точки обнаружения. 

Например, если **endpoint смены пароля в вашем приложении** — `/change-password`==(ПО УМОЛЧАНИЮ)==, то вы можете настроить Spring Security следующим образом:

**Конечная точка смены пароля по умолчанию**
**Java**
```java
http
    .passwordManagement(Customizer.withDefaults())
```

Затем, когда менеджер паролей перейдет по адресу `/.well-known/change-password`, Spring Security перенаправит на вашу конечную точку `/change-password`

Или, если ваша **endpoint отличается от** `/change-password`, вы также можете указать это следующим образом:
**Конечная точка смены пароля**
```java
http
    .passwordManagement((management) -> management
        .changePasswordPage("/update-password")
    )
```

С приведенной выше конфигурацией, когда менеджер паролей перейдет по адресу `/.well-known/change-password`, Spring Security перенаправит на `/update-password`.

## 2.1 Проверка скомпрометированных паролей
Существуют сценарии, когда вам нужно **проверить, был ли пароль скомпрометирован.** 

Например, если вы создаете приложение, работающее с конфиденциальными данными, часто требуется выполнять некоторые проверки паролей пользователей для подтверждения их надежности.

Одна из таких *проверок может заключаться в том*, не был ли пароль скомпрометирован, ***обычно потому***, **что он был обнаружен в утечке данных.**

Чтобы облегчить это, ==Spring Security предоставляет ИНТЕГРАЦИЯ== с API Have I Been Pwned через реализацию `HaveIBeenPwnedRestApiPasswordChecker` интерфейса `CompromisedPasswordChecker`.

> Вы можете 
> - либо использовать API `CompromisedPasswordChecker` самостоятельно, 
> - либо, если вы используете `DaoAuthenticationProvider` через механизмы аутентификации Spring Security, вы можете предоставить бин `CompromisedPasswordChecker`, и он будет автоматически подобран конфигурацией Spring Security.

#### ВАЖНОЕ ЗАМЕЧАНИЕ
Сделав это, *когда вы попытаетесь аутентифицироваться через форму входа* (Form Login), **используя слабый пароль**, скажем, `123456`, **вы получите 401 или будете перенаправлены на страницу** `/login?error` (в зависимости от вашего user-agent). 
Однако просто 401 или перенаправление в этом случае не очень полезны, ***это вызовет некоторую путаницу***, *потому что пользователь ввел правильный пароль, но ему все равно не разрешили войти.* 

В таких случаях **вы можете обработать** `CompromisedPasswordException` через `AuthenticationFailureHandler`, чтобы выполнить желаемую логику, например, перенаправить user-agent на `/reset-password`:

**Использование CompromisedPasswordChecker**
**Java**
```java
@Bean
public SecurityFilterChain filterChain(
	HttpSecurity http
) throws Exception {
	http
		.authorizeHttpRequests(authorize -> authorize
			.anyRequest().authenticated()
		)
		.formLogin((login) -> login
			.failureHandler(new CompromisedPasswordAuthenticationFailureHandler())
		);
	return http.build();
}

@Bean
public CompromisedPasswordChecker compromisedPasswordChecker() {
	return new HaveIBeenPwnedRestApiPasswordChecker();
}

static class CompromisedPasswordAuthenticationFailureHandler implements AuthenticationFailureHandler {

	private final SimpleUrlAuthenticationFailureHandler 
	defaultFailureHandler = 
		new SimpleUrlAuthenticationFailureHandler(
			"/login?error"
		);

	private final RedirectStrategy redirectStrategy = 
		new DefaultRedirectStrategy();

	@Override
	public void onAuthenticationFailure(
		HttpServletRequest request, 
		HttpServletResponse response,
		AuthenticationException exception
	) throws IOException, ServletException 
	{
		if (exception instanceof 
			CompromisedPasswordException) 
		{
			this.redirectStrategy
				.sendRedirect(
					request, 
					response, 
					"/reset-password"
				);
			return;
		}
		this.defaultFailureHandler
			.onAuthenticationFailure(
				request, 
				response, 
				exception
			);
	}

}
```
