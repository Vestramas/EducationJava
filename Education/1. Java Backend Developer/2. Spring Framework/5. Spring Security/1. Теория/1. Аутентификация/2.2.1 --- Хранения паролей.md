Каждый из поддерживаемых механизмов чтения имени пользователя и пароля может использовать любой из поддерживаемых механизмов хранения:

- Простое хранение с аутентификацией в памяти (In-Memory Authentication)
    
- Реляционные базы данных с аутентификацией через JDBC
    
- Пользовательские хранилища данных с `UserDetailsService`
    
- Хранилище LDAP с аутентификацией LDAP

# 1. UserDetails - данные пользователя из БД

> [!NOTE] Интерфейс
> **Кто это:** Контейнер для данных о пользователе.
> 
> **Содержит:** 
> - `username`, 
> - `password`, 
> - `authorities`, 
> - `isEnabled()`. 
> 
> **Аналогия:** Карточка сотрудника в отделе кадров. 
> **Реализации**: `User` (стандартный класс Spring Security).

`UserDetails` возвращается `UserDetailsService`. 

`DaoAuthenticationProvider` проверяет `UserDetails`, а затем возвращает `Authentication`, у которого принципалом являются `UserDetails`, возвращенные настроенным `UserDetailsService`.

## 1.1 Управление учетными данными (Credentials Management)
**Настоятельно рекомендуется реализовывать интерфейс** `CredentialsContainer` **в классах, которые хранят учетные данные пользователя**, таких как те, что расширяют или реализуют `UserDetails`, особенно в приложениях, где сведения о пользователе не кэшируются. 

Эта практика повышает безопасность, гарантируя, что конфиденциальные данные, такие как пароли, не хранятся в памяти дольше, чем необходимо.

В случаях, когда сведения о пользователе кэшируются, рассмотрите возможность создания копии `UserDetails`, которая не включает учетные данные, и возврата этой копии в ответе от пользовательского `AuthenticationProvider` вместо исходного объекта. 

Это может помочь предотвратить ссылки на кэшированный экземпляр, содержащий учетные данные, со стороны остальной части приложения после завершения процесса аутентификации.

### 1.1.1 Когда следует реализовывать `CredentialsContainer`
**Приложения**, которые **НЕ ИСПОЛЬЗУЮТ МЕХАНИЗМЫ КЕШИРОВАНИЯ** для `UserDetails`, должны особенно рассмотреть возможность реализации `CredentialsContainer`. 

Этот **подход помогает снизить риск**, связанный с **хранением конфиденциальной информации в памяти**, которая может быть уязвима для векторов атак, таких как дампы памяти.

```java
public class MyUserDetails 
	implements UserDetails, CredentialsContainer 
{
    private String username;

    private String password;

    // Реализация UserDetails...

    @Override
    public void eraseCredentials() {
        this.password = null; // Безопасно удаляем ссылку на поле пароля
    }
}
```

### 1.1.2 Рекомендации по реализации
1. **Немедленное стирание:** ***Учетные данные должны быть стерты сразу после того, как они больше не нужны***, обычно после аутентификации.
    
2. **Автоматический вызов:** Убедитесь, что `eraseCredentials()` автоматически вызывается вашим фреймворком аутентификации, таким как `AuthenticationManager`, как только процесс аутентификации завершен.
    
3. **Последовательность:** Применяйте эту практику единообразно во всех приложениях, чтобы предотвратить упущения в безопасности, которые могут привести к утечке данных.

### 1.1.3 За пределами базовой реализации интерфейса
Хотя такие интерфейсы, как `CredentialsContainer`, предоставляют основу для управления учетными данными, практическая реализация часто зависит от конкретных классов и их взаимодействия.

Например, класс `DaoAuthenticationProvider`, придерживаясь контракта `AuthenticationProvider`, не выполняет стирание учетных данных в своем собственном методе `authenticate`. 

Вместо этого он полагается на `ProviderManager` — реализацию `AuthenticationManager` по умолчанию в Spring Security — для обработки стирания учетных данных и других конфиденциальных данных после аутентификации. 
> Это разделение подчеркивает принцип, согласно которому `AuthenticationProvider` не должен брать на себя ответственность за управление учетными данными.

**Включение** `CredentialsContainer` в вашу реализацию `UserDetails` ==соответствует лучшим практикам безопасности,== уменьшая потенциальную подверженность утечкам данных за счет минимизации времени жизни конфиденциальных данных в памяти.

# 3. CredentialsContainer

> [!NOTE] Интерфейс
> 
> **Кто это:** Примесь к `UserDetails`.
> 
> **Зачем:** Добавляет метод `eraseCredentials()`, чтобы `ProviderManager` мог безопасно стереть пароль из памяти после проверки.
> 
> _Реализуется кем:_ `User` (поэтому он умеет стирать пароль).

Интерфейс `CredentialsContainer` ==указывает на то, что реализующий его объект содержит конфиденциальные данные==, **и используется внутри Spring Security для удаления (стирания) учетных данных** аутентификации после успешной аутентификации. 

Этот интерфейс реализован большинством внутренних классов предметной области Spring Security, таких как `User` и `UsernamePasswordAuthenticationToken`.

Менеджер `ProviderManager` **проверяет, реализует ли возвращенный объект** `Authentication` **этот интерфейс**. 
> Если да, он вызывает метод `eraseCredentials` для удаления учетных данных из объекта.


> [!IMPORTANT] Стирание данных пользователя после AuthN
> Если вы хотите, ***чтобы учетные данные ваших пользовательских объектов аутентификации стирались*** после аутентификации, вы должны **убедиться, что их классы реализуют интерфейс** `CredentialsContainer`.

Пользователям, которые пишут свои собственные реализации `AuthenticationProvider`, следует создавать и возвращать соответствующий объект `Authentication` уже без каких-либо конфиденциальных данных, а не полагаться на использование этого интерфейса.

# 4. Стирание пароля (Password Erasure)
После успешной аутентификации ==лучшей практикой безопасности является стирание учетных данных из памяти==, чтобы предотвратить их раскрытие при потенциальных атаках с использованием дампа памяти. 

`ProviderManager` в Spring Security поддерживает эту практику через метод `eraseCredentials`, который должен вызываться после завершения процесса аутентификации.

**Лучшие практики**
- **Немедленное стирание:** Учетные данные должны быть стерты сразу же после того, как они перестают быть нужны, что минимизирует временное окно, в течение которого учетные данные находятся в памяти.
    
- **Автоматическое стирание:** Настройте `ProviderManager` на автоматическое стирание учетных данных после аутентификации, установив свойство `eraseCredentialsAfterAuthentication` в значение `true` (по умолчанию).
    
- **Пользовательские стратегии стирания:** Реализуйте собственные стратегии стирания в пользовательских реализациях `AuthenticationManager`, если поведение стирания по умолчанию не соответствует конкретным требованиям безопасности.
    

**Оценка рисков**
Неправильное стирание учетных данных может привести к нескольким рискам:
- **Атаки с доступом к памяти:** Злоумышленники могут получить доступ к необработанным учетным данным из памяти через эксплойты, такие как атаки переполнения буфера или дампы памяти.
    
- **Инсайдерские угрозы:** Злонамеренные инсайдеры, имеющие доступ к системам, потенциально могут извлечь учетные данные из памяти приложения.
    
- **Случайное раскрытие:** В многопользовательских средах (multi-tenant) оставшиеся в памяти учетные данные могут быть случайно раскрыты другим пользователям (арендаторам).

**Реализация**
```java
public class CustomAuthenticationManager implements AuthenticationManager {

	@Override
	public Authentication authenticate(
		Authentication authenticationRequest
	) throws AuthenticationException {

		Authentication authenticationResult;
		// TODO: Выполнить проверки аутентификации...

		// Стереть учетные данные после проверки
		if (authenticationResult instanceof CredentialsContainer container) {
			container.eraseCredentials();
		}
	}
}
```

Внедряя эти практики, организации могут значительно повысить безопасность своих систем аутентификации, гарантируя, что учетные данные не остаются раскрытыми в системной памяти.

# 5. Аутентификация в памяти (In-Memory Authentication)

> [!NOTE] InMemoryUserDetailsManager
> **Иерархия:** `implements UserDetailsManager, UserDetailsService`.
> 
> **Суть:** Хранит список юзеров в `HashMap`. 
> При старте приложения вы наполняете его через `.username(...).password(...).build()`.

`InMemoryUserDetailsManager` в Spring Security **реализует** `UserDetailsService` для обеспечения поддержки аутентификации ***на основе имени пользователя и пароля***, **которые ХРАНЯТСЯ В ПАМЯТИ (ОЗУ).** 

`InMemoryUserDetailsManager` **обеспечивает управление** `UserDetails`, реализуя интерфейс `UserDetailsManager`. 

> [!NOTE] UserDetails - хранит данные пользователя
> ***Аутентификация на основе*** `UserDetails` ***используется Spring Security***, **когда он настроен на прием имени пользователя и пароля** для аутентификации.

В следующем примере мы используем Spring Boot CLI для кодирования значения пароля `password` и получаем закодированный пароль `{bcrypt}$2a$10$GRLdNijSQMUvl/au9ofL.eDwmoohzzS7.rmNSJZ.0FxO/BTk76klW`:

**Конфигурация InMemoryUserDetailsManager на Java**
```java
@Bean
public UserDetailsService users() {
	UserDetails user = User.builder()
		.username("user")
		.password("{bcrypt}$2a$10$GRLdNijSQMUvl/au9ofL.eDwmoohzzS7.rmNSJZ.0FxO/BTk76klW")
		.roles("USER")
		.build();
		
	UserDetails admin = User.builder()
		.username("admin")
		.password("{bcrypt}$2a$10$GRLdNijSQMUvl/au9ofL.eDwmoohzzS7.rmNSJZ.0FxO/BTk76klW")
		.roles("USER", "ADMIN")
		.build();
	return new InMemoryUserDetailsManager(user, admin);
}
```

Предыдущие примеры хранят пароли в безопасном формате, но оставляют желать лучшего с точки зрения простоты начала работы.

В следующем примере мы используем `User.withDefaultPasswordEncoder`, чтобы гарантировать, что пароль, хранящийся в памяти, защищен. 

Однако это ***не защищает от получения пароля путем декомпиляции исходного кода***. 
По этой причине `User.withDefaultPasswordEncoder` следует использовать только для «начала работы» и не предназначен для production.

**InMemoryUserDetailsManager с User.withDefaultPasswordEncoder**
```java
@Bean
public UserDetailsService users() {
	// Строитель обеспечит кодирование паролей перед сохранением в памяти
	UserBuilder users = User.withDefaultPasswordEncoder();
	UserDetails user = users
		.username("user")
		.password("password")
		.roles("USER")
		.build();
		
	UserDetails admin = users
		.username("admin")
		.password("password")
		.roles("USER", "ADMIN")
		.build();
	return new InMemoryUserDetailsManager(user, admin);
}
```

**Не существует простого способа использовать** `User.withDefaultPasswordEncoder` с конфигурацией на основе XML. Для демонстраций или просто для начала работы вы можете добавить к паролю префикс `{noop}`, чтобы указать, что кодирование не должно использоваться:

**Конфигурация XML с {noop}**
```xml
<user-service>
	<user name="user"
		password="{noop}password"
		authorities="ROLE_USER" />
	<user name="admin"
		password="{noop}password"
		authorities="ROLE_USER,ROLE_ADMIN" />
</user-service>
```

# 6. Аутентификация через JDBC(JdbcDaoImpl и JdbcUserDetailsManager)

> [!NOTE] JdbcDaoImpl
> **Иерархия:** `implements UserDetailsService`.
> 
> **Суть:** Выполняет запрос `select username, password, enabled from users where username = ?`. 
> 
> **Важно:** Он умеет только **читать** юзеров.

`JdbcDaoImpl` в Spring Security **реализует** `UserDetailsService` для обеспечения поддержки аутентификации на основе имени пользователя и пароля, которые извлекаются с использованием JDBC.


> [!NOTE] JdbcUserDetailsManager
> **Иерархия:** `extends JdbcDaoImpl` `implements UserDetailsManager`.
> 
> **Суть:** Наследует умение читать из БД + добавляет умение **писать** в БД (`createUser`, `deleteUser`).

`JdbcUserDetailsManager` **расширяет** `JdbcDaoImpl` для обеспечения управления `UserDetails` через интерфейс `UserDetailsManager`. 



Аутентификация на основе `UserDetails` используется Spring Security, когда он настроен на прием имени пользователя/пароля для аутентификации.

## 6.1 Схема по умолчанию
**Spring Security предоставляет запросы по умолчанию** для аутентификации на основе JDBC. 

В этом разделе представлены соответствующие схемы по умолчанию, используемые с запросами по умолчанию. 

> Вам необходимо адаптировать схему в соответствии с любыми изменениями запросов и диалектом используемой вами базы данных.

### 6.1.1 Схема пользователя
`JdbcDaoImpl` ==требует таблицы== для 
- загрузки пароля, 
- статуса учетной записи (включена или отключена) 
- и списка полномочий (ролей) для пользователя.

Схема по умолчанию также доступна как ресурс classpath с именем `org/springframework/security/core/userdetails/jdbc/users.ddl`.

**Схема пользователя по умолчанию**
```sql
create table users(
	username varchar_ignorecase(50) not null primary key,
	password varchar_ignorecase(500) not null,
	enabled boolean not null
);

create table authorities (
	username varchar_ignorecase(50) not null,
	authority varchar_ignorecase(50) not null,
	
	constraint fk_authorities_users 
		foreign key(username) references users(username)
);
create unique index ix_auth_username on authorities (username,authority);
```

Oracle — популярный выбор базы данных, но требует немного другой схемы:

**Схема пользователя по умолчанию для баз данных Oracle**
```sql
CREATE TABLE USERS (
    USERNAME NVARCHAR2(128) PRIMARY KEY,
    PASSWORD NVARCHAR2(128) NOT NULL,
    ENABLED CHAR(1) CHECK (ENABLED IN ('Y','N') ) NOT NULL
);

CREATE TABLE AUTHORITIES (
    USERNAME NVARCHAR2(128) NOT NULL,
    AUTHORITY NVARCHAR2(128) NOT NULL
);
ALTER TABLE AUTHORITIES ADD CONSTRAINT AUTHORITIES_UNIQUE UNIQUE (USERNAME, AUTHORITY);

ALTER TABLE AUTHORITIES ADD CONSTRAINT AUTHORITIES_FK1 FOREIGN KEY (USERNAME) REFERENCES USERS (USERNAME) ENABLE;
```

# 8. DaoAuthenticationProvider

> [!NOTE] DaoAuthenticationProvider
> `DaoAuthenticationProvider` — это реализация `AuthenticationProvider` 
> 
> Которая использует 
> - `UserDetailsService` 
> - и `PasswordEncoder` 
> для аутентификации имени пользователя и пароля.

В этом разделе рассматривается, как `DaoAuthenticationProvider` работает в Spring Security. 

Следующий рисунок объясняет работу `AuthenticationManager` на рисунках из раздела «Чтение имени пользователя и пароля».

![[Pasted image 20260418103846.png]]
**Рисунок 1. Использование DaoAuthenticationProvider**

1. `AuthenticationFilter` из раздела «Чтение имени пользователя и пароля» передает `UsernamePasswordAuthenticationToken` в `AuthenticationManager`, который реализован `ProviderManager`.
    
2. `ProviderManager` настроен на использование `AuthenticationProvider` типа `DaoAuthenticationProvider`.
    
3. `DaoAuthenticationProvider` ищет `UserDetails` в `UserDetailsService`.
    
4. `DaoAuthenticationProvider` использует `PasswordEncoder` для проверки пароля в `UserDetails`, возвращенных на предыдущем шаге.
    
5. Когда аутентификация успешна, возвращаемый объект `Authentication` имеет тип `UsernamePasswordAuthenticationToken` и содержит принципала, которым является `UserDetails`, возвращенный настроенным `UserDetailsService`, и набор полномочий (authorities), содержащий по крайней мере `FACTOR_PASSWORD`. 
   В конечном итоге возвращенный `UsernamePasswordAuthenticationToken` устанавливается в `SecurityContextHolder` фильтром аутентификации.

# 9. Аутентификация через LDAP

> [!NOTE] LDAP (Lightweight Directory Access Protocol)
> LDAP (Lightweight Directory Access Protocol — **облегченный протокол доступа к каталогам**) часто ***используется организациями как центральное хранилище информации о пользователях*** ***и как служба аутентификации.*** 

> Он также может использоваться для хранения информации о ролях пользователей приложений.

Аутентификация на основе LDAP в Spring Security **используется, когда Spring Security настроен на прием имени пользователя и пароля** для аутентификации. 

Однако, несмотря на использование имени пользователя и пароля для аутентификации, он **не использует `UserDetailsService`**, поскольку при аутентификации методом привязки (bind authentication) **сервер LDAP не возвращает пароль**, поэтому приложение не может выполнить проверку пароля.

Существует множество различных сценариев настройки сервера LDAP, поэтому провайдер LDAP в Spring Security полностью настраиваем. 

Он использует отдельные интерфейсы стратегий для аутентификации и получения ролей и предоставляет реализации по умолчанию, которые можно настроить для обработки широкого спектра ситуаций.

## 9.1 Необходимые зависимости
Для начала работы **добавьте зависимость** `spring-security-ldap` в ваш проект. 

При использовании Spring Boot добавьте следующие зависимости:

**Зависимости Spring Security LDAP**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-ldap</artifactId>
</dependency>

<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-ldap</artifactId>
</dependency>
```

## 9.2 Предварительные требования
Вам следует ознакомиться с LDAP, прежде чем пытаться использовать его со Spring Security. 

Следующая ссылка дает хорошее введение в связанные концепции и руководство по настройке каталога с использованием бесплатного сервера LDAP, OpenLDAP: [www.zytrax.com/books/ldap/](http://www.zytrax.com/books/ldap/). 

Некоторое знакомство с API JNDI, используемыми для доступа к LDAP из Java, также может быть полезным. 

Мы не используем никаких сторонних библиотек LDAP (Mozilla, JLDAP или других) в провайдере LDAP, но активно используется Spring LDAP, поэтому некоторое знакомство с этим проектом может быть полезно, если вы планируете добавлять собственные настройки.

При использовании аутентификации LDAP вы должны убедиться, что правильно настроили пул соединений LDAP. 

Если вы не знакомы с тем, как это сделать, см. документацию по Java LDAP.

Если вы хотите использовать UnboundID, укажите следующие зависимости:

Затем вы можете настроить встроенный сервер LDAP с помощью `EmbeddedLdapServerContextSourceFactoryBean`. Это укажет Spring Security запустить сервер LDAP в памяти:

```java

	return 
		new UnboundIdContainer(
			"dc=springframework,dc=org",
			"classpath:users.ldif"
		);
}
```

## 9.4 LDAP ContextSource
После того как у вас есть сервер LDAP, на который указывает ваша конфигурация, вам необходимо настроить Spring Security на указание сервера LDAP, который следует использовать для аутентификации пользователей. 

Для этого создайте LDAP `ContextSource` (эквивалент JDBC `DataSource`). Если вы уже настроили `EmbeddedLdapServerContextSourceFactoryBean`, Spring Security создаст LDAP `ContextSource`, который указывает на встроенный сервер LDAP.

**LDAP Context Source со встроенным сервером LDAP**
```java
@Bean
public EmbeddedLdapServerContextSourceFactoryBean contextSourceFactoryBean() {
	EmbeddedLdapServerContextSourceFactoryBean 
	contextSourceFactoryBean =
			EmbeddedLdapServerContextSourceFactoryBean
				.fromEmbeddedLdapServer();
				
	contextSourceFactoryBean.setPort(0);
	return contextSourceFactoryBean;
}
```

В качестве альтернативы вы можете явно настроить LDAP `ContextSource` для подключения к предоставленному серверу LDAP:

**LDAP Context Source**
```java
ContextSource contextSource(UnboundIdContainer container) {
	return new 
	DefaultSpringSecurityContextSource(
	"ldap://localhost:53389/dc=springframework,dc=org");
}
```

## 9.6 Использование аутентификации привязкой (Bind Authentication)
Аутентификация привязкой — наиболее распространенный механизм аутентификации пользователей с помощью LDAP. 

При аутентификации привязкой учетные данные пользователя (имя пользователя и пароль) передаются на сервер LDAP, который аутентифицирует их. Преимущество использования аутентификации привязкой заключается в том, что секреты пользователя (пароль) не нужно раскрывать клиентам, что помогает защитить их от утечки.

Следующий пример показывает конфигурацию аутентификации привязкой:

**Аутентификация привязкой (Bind Authentication)**
```java
@Bean
AuthenticationManager authenticationManager(
	BaseLdapPathContextSource contextSource
) {
	LdapBindAuthenticationManagerFactory factory = 
		new LdapBindAuthenticationManagerFactory(
			contextSource
		);
	
	factory.setUserDnPatterns("uid={0},ou=people");
	return factory.createAuthenticationManager();
}
```

Предыдущий простой пример получил бы DN пользователя, подставив имя входа пользователя в предоставленный шаблон, и попытался бы выполнить привязку от имени этого пользователя с паролем для входа. Это нормально, если все ваши пользователи хранятся в одном узле каталога. Если вместо этого вы хотите настроить поисковый фильтр LDAP для поиска пользователя, вы можете использовать следующее:

**Аутентификация привязкой с поисковым фильтром**
```java
@Bean
AuthenticationManager authenticationManager(
	BaseLdapPathContextSource contextSource
) {
	LdapBindAuthenticationManagerFactory factory = 
		new LdapBindAuthenticationManagerFactory(
			contextSource
		);
		
	factory.setUserSearchFilter("(uid={0})");
	factory.setUserSearchBase("ou=people");
	return factory.createAuthenticationManager();
}
```

При использовании с определением `ContextSource`, показанным ранее, это выполнит поиск в DN `ou=people,dc=springframework,dc=org`, используя `(uid={0})` в качестве фильтра. Опять же, имя входа пользователя подставляется вместо параметра в имени фильтра, поэтому он ищет запись с атрибутом `uid`, равным имени пользователя. Если база поиска пользователя не указана, поиск выполняется от корня.

## 9.7 Использование аутентификации по паролю (Password Authentication)
Сравнение паролей — это когда пароль, предоставленный пользователем, сравнивается с паролем, хранящимся в репозитории. Это можно сделать либо путем получения значения атрибута пароля и проверки его локально, либо путем выполнения операции «сравнения» LDAP, когда предоставленный пароль передается на сервер для сравнения, а реальное значение пароля никогда не извлекается. Сравнение LDAP невозможно, если пароль правильно хеширован со случайной солью.

**Минимальная конфигурация сравнения паролей**
```java
@Bean
AuthenticationManager authenticationManager(
	BaseLdapPathContextSource contextSource
) {
	LdapPasswordComparisonAuthenticationManagerFactory
	factory = new 
	LdapPasswordComparisonAuthenticationManagerFactory(
		contextSource, 
		NoOpPasswordEncoder.getInstance()
	);
	
	factory.setUserDnPatterns("uid={0},ou=people");
	return factory.createAuthenticationManager();
}
```

Следующий пример показывает более продвинутую конфигурацию с некоторыми настройками:

**Конфигурация сравнения паролей**
```java
@Bean
AuthenticationManager authenticationManager(
	BaseLdapPathContextSource contextSource
) {
	LdapPasswordComparisonAuthenticationManagerFactory 
	factory = new 
	LdapPasswordComparisonAuthenticationManagerFactory(
		contextSource, 
		new BCryptPasswordEncoder()
	);
	factory.setUserDnPatterns("uid={0},ou=people");
	factory.setPasswordAttribute("pwd");  // (1)
	return factory.createAuthenticationManager();
}
```

|№|Описание|
|---|---|
|(1)|Указывает атрибут пароля как `pwd`.|

## 9.8 LdapAuthoritiesPopulator
`LdapAuthoritiesPopulator` в Spring Security используется для определения того, какие полномочия (authorities) возвращаются для пользователя. Следующий пример показывает, как настроить `LdapAuthoritiesPopulator`:

**Конфигурация LdapAuthoritiesPopulator**
```java
@Bean
LdapAuthoritiesPopulator authorities(BaseLdapPathContextSource contextSource) {
	String groupSearchBase = "";
	DefaultLdapAuthoritiesPopulator authorities =
		new DefaultLdapAuthoritiesPopulator(
			contextSource, 
			groupSearchBase
		);
	authorities.setGroupSearchFilter("member={0}");
	return authorities;
}

@Bean
AuthenticationManager authenticationManager(
	BaseLdapPathContextSource contextSource, 
	LdapAuthoritiesPopulator authorities
) {
	LdapBindAuthenticationManagerFactory factory = 
	new LdapBindAuthenticationManagerFactory(
		contextSource
	);
	factory.setUserDnPatterns("uid={0},ou=people");
	factory.setLdapAuthoritiesPopulator(authorities);
	return factory.createAuthenticationManager();
}
```

## 9.9 Active Directory
Active Directory поддерживает свои собственные нестандартные параметры аутентификации, и обычный шаблон использования не совсем точно соответствует стандартному `LdapAuthenticationProvider`. Обычно аутентификация выполняется с использованием доменного имени пользователя (в форме `user@domain`), а не с использованием отличительного имени LDAP (distinguished name). Чтобы упростить это, в Spring Security есть провайдер аутентификации, который настроен для типичной установки Active Directory.

Настройка `ActiveDirectoryLdapAuthenticationProvider` довольно проста. Вам нужно только указать имя домена и URL-адрес LDAP, который предоставляет адрес сервера.

Также возможно получить IP-адрес сервера с помощью поиска DNS. В настоящее время это не поддерживается, но, надеюсь, будет в будущей версии.

Следующий пример настраивает Active Directory:

**Пример конфигурации Active Directory**
```java
@Bean
ActiveDirectoryLdapAuthenticationProvider authenticationProvider() {
	return new ActiveDirectoryLdapAuthenticationProvider(
		"example.com", 
		"ldap://company.example.com/"
	);
}
```
