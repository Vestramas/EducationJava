Многофакторная аутентификация (MFA) **требует, чтобы пользователь предоставил несколько факторов для аутентификации.** 

OWASP относит факторы к следующим категориям:
- Что-то, что пользователь **знает** (например, пароль)
    
- Что-то, что у пользователя **есть** (например, доступ к SMS или email)
    
- Что-то, чем пользователь **является** (например, биометрия)
    
- Где пользователь **находится** (например, геолокация)
    
- Что-то, что пользователь **делает** (например, профилирование поведения)

## 1.1 FactorGrantedAuthority
Во время аутентификации механизмы аутентификации Spring Security добавляют `FactorGrantedAuthority`. 

> **Например**, когда пользователь аутентифицируется с использованием пароля, `FactorGrantedAuthority` с полномочием `FactorGrantedAuthority.PASSWORD_AUTHORITY` автоматически добавляется в объект `Authentication`.

Чтобы требовать MFA в Spring Security, вы должны:
1. Указать правило авторизации, которое требует несколько факторов.
    
2. Настроить аутентификацию для каждого из этих факторов.

---
## 1.2 @EnableMultiFactorAuthentication
Аннотация `@EnableMultiFactorAuthentication` **упрощает включение многофакторной аутентификации**. 

Ниже приведена конфигурация, которая добавляет требование наличия как пароля, так и OTT (One-Time Token) для каждого правила авторизации.
```java
@EnableMultiFactorAuthentication(authorities = {
	FactorGrantedAuthority.PASSWORD_AUTHORITY,
	FactorGrantedAuthority.OTT_AUTHORITY })
```

```java
@Configuration
@EnableWebSecurity // (или @EnableWebFluxSecurity)
@EnableMultiFactorAuthentication // <-- ВОТ ЗДЕСЬ
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        // ...
    }
}
```


Теперь мы можем лаконично создать конфигурацию, которая всегда требует несколько факторов.
```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (1)
			.requestMatchers("/admin/**").hasRole("ADMIN")
			// (2)
			.anyRequest().authenticated()
		)
		// (3)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|URL-адреса, начинающиеся с `/admin/**`, требуют наличия полномочий `FACTOR_OTT`, `FACTOR_PASSWORD`, `ROLE_ADMIN`.|
|(2)|Любой другой URL-адрес требует наличия полномочий `FACTOR_OTT`, `FACTOR_PASSWORD`.|
|(3)|Настройка механизмов аутентификации, которые могут предоставить требуемые факторы.|

Spring Security «за кулисами» знает, на какую конечную точку (endpoint) перейти в зависимости от того, какое полномочие отсутствует. 

Если пользователь изначально вошел в систему с именем пользователя и паролем, Spring Security перенаправляет на страницу входа по одноразовому токену (One-Time-Token Login). 

Если пользователь изначально вошел в систему с токеном, Spring Security перенаправляет на страницу входа по имени пользователя и паролю.

## 1.3 AuthorizationManagerFactory
Свойство `authorities` в `@EnableMultiFactorAuthentication` — это просто сокращение для публикации бина `AuthorizationManagerFactory`. 

Когда бин `AuthorizationManagerFactory` доступен, он используется Spring Security для создания правил авторизации, таких как `hasAnyRole(String)`, которые определены в интерфейсе бина `AuthorizationManagerFactory`. 

Реализация, опубликованная с помощью `@EnableMultiFactorAuthentication`, гарантирует, что каждая авторизация будет объединена с требованием наличия указанных факторов.

Приведенный ниже бин `AuthorizationManagerFactory` — это то, что публикуется в ранее рассмотренном примере с `@EnableMultiFactorAuthentication`.
```java
@Bean
AuthorizationManagerFactory<Object> authz() {
	return AuthorizationManagerFactories.multiFactor()
		.requireFactors(
			FactorGrantedAuthority.PASSWORD_AUTHORITY,
			FactorGrantedAuthority.OTT_AUTHORITY
		)
		.build();
}
```

## 1.4 Выборочное требование MFA
Мы продемонстрировали, как настроить все приложение на требование MFA с помощью свойства `authorities` в `@EnableMultiFactorAuthentication`. 

Однако бывают случаи, когда приложение должно требовать MFA только для определенных частей. 

Рассмотрим следующие требования:
- URL-адреса, начинающиеся с `/admin/`, должны требовать полномочия `FACTOR_OTT`, `FACTOR_PASSWORD`, `ROLE_ADMIN`.
    
- URL-адреса, начинающиеся с `/user/settings`, должны требовать полномочия `FACTOR_OTT`, `FACTOR_PASSWORD`.
    
- Любой другой URL-адрес требует аутентифицированного пользователя.

В этом случае **одни URL-адреса требуют MFA, а другие — нет**. Это означает, что ***глобальный подход, который мы видели ранее, не работает.*** 

К счастью, мы можем использовать то, что мы узнали в разделе `AuthorizationManagerFactory`, чтобы решить эту задачу лаконичным способом.


**Начните с указания** `@EnableMultiFactorAuthentication` ==без каких-либо полномочий== (`authorities`). 

Поступая так, мы включаем поддержку MFA, **но бин** `AuthorizationManagerFactory` **не публикуется.**

```java
@EnableMultiFactorAuthentication(authorities = {})
```

Затем создайте экземпляр `AuthorizationManagerFactory`, но **не публикуйте его как бин**.
```java
@Bean
SecurityFilterChain securityFilterChain(
	HttpSecurity http
) throws Exception 
{
	// (1)
	var mfa = AuthorizationManagerFactories.multiFactor()
		.requireFactors(
			FactorGrantedAuthority.PASSWORD_AUTHORITY,
			FactorGrantedAuthority.OTT_AUTHORITY
		)
		.build();
		
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (2)
			.requestMatchers("/admin/**")
			.access(mfa.hasRole("ADMIN"))
			
			// (3)
			.requestMatchers("/user/settings/**")
			.access(mfa.authenticated())
			
			// (4)
			.anyRequest().authenticated()
		)
		// (5)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

| №   | Описание                                                                                                                                                                                                                                    |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Создайте `DefaultAuthorizationManagerFactory`, как мы делали ранее, но **не публикуйте его как бин**. <br><br>Не публикуя его как бин, мы можем использовать `AuthorizationManagerFactory` выборочно, а не для каждого правила авторизации. |
| (2) | Явно используйте `AuthorizationManagerFactory`, чтобы URL-адреса, начинающиеся с `/admin/**`, требовали `FACTOR_OTT`, `FACTOR_PASSWORD` и `ROLE_ADMIN`.                                                                                     |
| (3) | Явно используйте `AuthorizationManagerFactory`, чтобы URL-адреса, начинающиеся с `/user/settings`, требовали `FACTOR_OTT` и `FACTOR_PASSWORD`.                                                                                              |
| (4) | В противном случае запрос должен быть аутентифицирован. <br><br>Требования MFA нет, поскольку `AuthorizationManagerFactory` не используется.                                                                                                |
| (5) | Настройка механизмов аутентификации, которые могут предоставить требуемые факторы.                                                                                                                                                          |

## 1.5 Указание допустимой продолжительности действия
Иногда нам может потребоваться определить правила авторизации на основе того, **как давно** мы аутентифицировались. 

Например, приложение может потребовать, чтобы пользователь аутентифицировался в течение последнего часа, чтобы разрешить доступ к конечной точке `/user/settings`.

> Помните, что во время аутентификации в `Authentication` добавляется `FactorGrantedAuthority`. `FactorGrantedAuthority` указывает, **когда** он был выдан (`issuedAt`), но не описывает, как долго он действителен. 
> 
> Это сделано намеренно, поскольку позволяет использовать один и тот же `FactorGrantedAuthority` с разными `validDurations` (допустимыми продолжительностями).

Давайте рассмотрим пример, иллюстрирующий, как удовлетворить следующие требования:
- URL-адреса, начинающиеся с `/admin/`, должны требовать, чтобы пароль был предоставлен в течение последних 30 минут.
    
- URL-адреса, начинающиеся с `/user/settings`, должны требовать, чтобы пароль был предоставлен в течение последнего часа.
    
- В противном случае требуется аутентификация, но не имеет значения, был ли это пароль или как давно произошла аутентификация.

```java
@Bean
SecurityFilterChain securityFilterChain(
	HttpSecurity http
) throws Exception {
	// (1)
	var passwordIn30m = AuthorizationManagerFactories.multiFactor()
		.requireFactor( (factor) -> factor
			.passwordAuthority()
			.validDuration(Duration.ofMinutes(30))
		)
		.build();
		
	// (2)
	var passwordInHour = AuthorizationManagerFactories.multiFactor()
		.requireFactor( (factor) -> factor
			.passwordAuthority()
			.validDuration(Duration.ofHours(1))
		)
		.build();
		
		
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (3)
			.requestMatchers("/admin/**")
			.access(passwordIn30m.hasRole("ADMIN"))
			
			// (4)
			.requestMatchers("/user/settings/**")
			.access(passwordInHour.authenticated())
			
			// (5)
			.anyRequest().authenticated()
		)
		// (6)
		.formLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|Сначала мы определяем `passwordIn30m` как требование наличия пароля, введенного в течение последних 30 минут.|
|(2)|Затем мы определяем `passwordInHour` как требование наличия пароля, введенного в течение последнего часа.|
|(3)|Мы используем `passwordIn30m`, чтобы потребовать, чтобы URL-адреса, начинающиеся с `/admin/`, требовали пароль, предоставленный в последние 30 минут, и чтобы у пользователя было полномочие `ROLE_ADMIN`.|
|(4)|Мы используем `passwordInHour`, чтобы потребовать, чтобы URL-адреса, начинающиеся с `/user/settings`, требовали пароль, предоставленный в последний час.|
|(5)|В противном случае требуется аутентификация, но не имеет значения, был ли это пароль или как давно произошла аутентификация.|
|(6)|Настройка механизмов аутентификации, которые могут предоставить требуемые факторы.|

## 1.6 Программная MFA
В наших предыдущих примерах MFA — это статическое решение для каждого запроса. 

Бывают случаи, когда **мы можем захотеть требовать MFA для одних пользователей**, *но не для других*. 

Определение того, включена ли MFA для конкретного пользователя, может быть достигнуто путем создания пользовательского `AuthorizationManager`, который условно требует факторы в зависимости от `Authentication`.

```java
@Component
class AdminMfaAuthorizationManager implements AuthorizationManager<Object> {
	@Override
	public AuthorizationResult authorize(
		Supplier<? extends @Nullable Authentication> authentication, 
		Object context
	) {
		if ("admin"
			.equals(authentication.get().getName())
		) {
			AuthorizationManager<Object> admins =
				AllAuthoritiesAuthorizationManager
					.hasAllAuthorities(
						FactorGrantedAuthority
							.OTT_AUTHORITY,
						FactorGrantedAuthority
							.PASSWORD_AUTHORITY
				);
			// (1)
			return admins.authorize(
				authentication, 
				context
			);
		} else {
			// (2)
			return new AuthorizationDecision(true);
		}
	}
}
```

|№|Описание|
|---|---|
|(1)|MFA требуется для пользователя с именем пользователя `admin`.|
|(2)|В противном случае MFA не требуется.|

Чтобы включить правила MFA глобально, мы можем опубликовать бин `AuthorizationManagerFactory`.

```java
@Bean
AuthorizationManagerFactory<Object> authorizationManagerFactory(
		AdminMfaAuthorizationManager admins) 
{
	DefaultAuthorizationManagerFactory<Object> defaults = 
		new DefaultAuthorizationManagerFactory<>();
	// (1)
	defaults.setAdditionalAuthorization(admins);
	// (2)
	return defaults;
}
```

| №   | Описание                                                                                                                                                                                                                                                                                                                                                            |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Внедрите пользовательский `AuthorizationManager` <br>как `DefaultAuthorization.additionalAuthorization`. <br><br>Это указывает `DefaultAuthorizationManagerFactory`, что любое правило авторизации должно применять наш пользовательский `AuthorizationManager` вместе с любыми требованиями авторизации, определенными приложением (например, `hasRole("ADMIN")`). |
| (2) | Опубликуйте `DefaultAuthorizationManagerFactory` <br>как бин, чтобы он использовался глобально.                                                                                                                                                                                                                                                                     |

Это должно быть очень похоже на наш предыдущий пример в разделе `AuthorizationManagerFactory`. Разница в том, что в предыдущем примере `AuthorizationManagerFactories` устанавливает `DefaultAuthorization.additionalAuthorization` с помощью встроенного `AuthorizationManager`, который всегда требует одни и те же полномочия.

Теперь мы можем определить наши правила авторизации, которые объединяются с `AdminMfaAuthorizationManager`.
```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (1)
			.requestMatchers("/admin/**").hasRole("ADMIN")
			// (2)
			.anyRequest().authenticated()
		)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|URL-адреса, начинающиеся с `/admin/**`, требуют `ROLE_ADMIN`. Если имя пользователя `admin`, то также требуются `FACTOR_OTT` и `FACTOR_PASSWORD`.|
|(2)|В противном случае запрос должен быть аутентифицирован. Если имя пользователя `admin`, то также требуются `FACTOR_OTT` и `FACTOR_PASSWORD`.|

MFA включена по имени пользователя, а не по роли, потому что именно так мы реализовали `RequiredAuthoritiesAuthorizationManagerConfiguration`. Если бы мы предпочли, мы могли бы изменить нашу логику, чтобы включать MFA на основе ролей, а не имени пользователя.

## 1.7 RequiredAuthoritiesAuthorizationManager - встроенная "Программная MFA"
Мы продемонстрировали, как можно динамически определять полномочия для конкретного пользователя в разделе **«Программная MFA»** с использованием пользовательского `AuthorizationManager`. 

Однако это настолько **распространенный сценарий**, ***что Spring Security предоставляет встроенную поддержку с использованием*** 
- `RequiredAuthoritiesAuthorizationManager` 
- и `RequiredAuthoritiesRepository`.

Давайте реализуем то же требование, что и в разделе «Программная MFA», используя встроенную поддержку.

Начнем с создания бина `RequiredAuthoritiesAuthorizationManager` для использования.

```java
@Bean
RequiredAuthoritiesAuthorizationManager<Object> adminAuthorization() {
	// (1)
	MapRequiredAuthoritiesRepository authorities = 
		new MapRequiredAuthoritiesRepository();
		
	authorities.saveRequiredAuthorities(
		"admin", 
		List.of(
			FactorGrantedAuthority.PASSWORD_AUTHORITY,
			FactorGrantedAuthority.OTT_AUTHORITY
		)
	);
	// (2)
	return new RequiredAuthoritiesAuthorizationManager<>(authorities);
}
```

|№|Описание|
|---|---|
|(1)|Создайте `MapRequiredAuthoritiesRepository`, который сопоставляет пользователей с именем пользователя `admin` с требованием MFA.|
|(2)|Верните `RequiredAuthoritiesAuthorizationManager`, в который внедрен `MapRequiredAuthoritiesRepository`.|

Далее мы можем определить `AuthorizationManagerFactory`, который использует `RequiredAuthoritiesAuthorizationManager`.

```java
@Bean
AuthorizationManagerFactory<Object> authorizationManagerFactory(
		RequiredAuthoritiesAuthorizationManager admins
) {
	DefaultAuthorizationManagerFactory<Object> defaults = 
		new DefaultAuthorizationManagerFactory<>();
	// (1)
	defaults.setAdditionalAuthorization(admins);
	// (2)
	return defaults;
}
```

| №   | Описание                                                                                                                                                                                                                                                                                                                                                                |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Внедрите `RequiredAuthoritiesAuthorizationManager` <br>как `DefaultAuthorization.additionalAuthorization`. <br><br>Это указывает `DefaultAuthorizationManagerFactory`, что любое правило авторизации должно применять `RequiredAuthoritiesAuthorizationManager` <br>вместе с любыми требованиями авторизации, определенными приложением (например, `hasRole("ADMIN")`). |
| (2) | Опубликуйте `DefaultAuthorizationManagerFactory` как бин, чтобы он использовался глобально.                                                                                                                                                                                                                                                                             |

Теперь мы можем определить наши правила авторизации, которые объединяются с `RequiredAuthoritiesAuthorizationManager`.

```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.requestMatchers("/admin/**")
				.hasRole("ADMIN") // (1)
			.anyRequest().authenticated() // (2)
		)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|URL-адреса, начинающиеся с `/admin/**`, требуют `ROLE_ADMIN`. Если имя пользователя `admin`, то также требуются `FACTOR_OTT` и `FACTOR_PASSWORD`.|
|(2)|В противном случае запрос должен быть аутентифицирован. Если имя пользователя `admin`, то также требуются `FACTOR_OTT` и `FACTOR_PASSWORD`.|

В нашем примере используется отображение имен пользователей на дополнительные требуемые полномочия, хранящееся в памяти. Для более динамичных сценариев использования, которые могут определяться именем пользователя, можно создать пользовательскую реализацию `RequiredAuthoritiesRepository`. 

Возможные примеры: проверка, включил ли пользователь MFA в явной настройке, определение, зарегистрировал ли пользователь ключ доступа (passkey), и т. д.

Для случаев, когда необходимо определять MFA на основе `Authentication`, можно использовать пользовательский `AuthorizationManager`, как показано в разделе «Программная MFA».

## 1.8 Использование hasAllAuthorities
Мы показали много дополнительной инфраструктуры для поддержки MFA. 

Однако для **простых случаев использования MFA** эффективно использовать `hasAllAuthorities` для требования нескольких факторов.
```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (1)
			.anyRequest().hasAllAuthorities(
				FactorGrantedAuthority.PASSWORD_AUTHORITY,
				FactorGrantedAuthority.OTT_AUTHORITY
			)
		)
		// (2)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|Требовать `FACTOR_PASSWORD` и `FACTOR_OTT` для каждого запроса.|
|(2)|Настройка механизмов аутентификации, которые могут предоставить требуемые факторы.|

Приведенная выше конфигурация хорошо работает **только для самых простых случаев использования**. 

Если у вас много конечных точек (endpoints), вы, вероятно, не захотите повторять требования MFA в каждом правиле авторизации.

Например, рассмотрим следующую конфигурацию:

```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			// (1)
			.requestMatchers("/admin/**")
			.hasAllAuthorities(
				"ROLE_ADMIN",
				FactorGrantedAuthority.PASSWORD_AUTHORITY,
				FactorGrantedAuthority.OTT_AUTHORITY
			)
			
			// (2)
			.anyRequest()
			.hasAllAuthorities(
				"ROLE_USER",
				FactorGrantedAuthority.PASSWORD_AUTHORITY,
				FactorGrantedAuthority.OTT_AUTHORITY
			)
		)
		// (3)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|Для URL-адресов, начинающихся с `/admin/**`, требуются следующие полномочия: `FACTOR_OTT`, `FACTOR_PASSWORD`, `ROLE_ADMIN`.|
|(2)|Для любого другого URL-адреса требуются следующие полномочия: `FACTOR_OTT`, `FACTOR_PASSWORD`, `ROLE_USER`.|
|(3)|Настройка механизмов аутентификации, которые могут предоставить требуемые факторы.|

Конфигурация определяет только два правила авторизации, но этого достаточно, чтобы увидеть, что дублирование нежелательно. Можете ли вы представить, каково было бы объявить сотни таких правил?

Более того, становится трудно выразить более сложные правила авторизации. Например, как бы вы потребовали два фактора **и** либо `ROLE_ADMIN`, либо `ROLE_USER`?

Ответ на эти вопросы, как мы уже видели, заключается в использовании `@EnableMultiFactorAuthentication`.

## 1.9 Повторная аутентификация (Re-authentication)
Наиболее распространенным из них является повторная аутентификация. 

Представьте приложение, настроенное следующим образом:
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
		.anyRequest().authenticated())
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

***По умолчанию это приложение имеет два механизма аутентификации***, которые оно разрешает, а **это означает, что пользователь может использовать любой из них** ***и быть полностью аутентифицированным.***

Если есть набор конечных точек, требующих определенный фактор, мы можем указать это в `authorizeHttpRequests` следующим образом:
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.requestMatchers("/profile/**")
			.hasAuthority(
				FactorGrantedAuthority.OTT_AUTHORITY
			) // (1)
			
			.anyRequest().authenticated()
		)
		.formLogin(Customizer.withDefaults())
		.oneTimeTokenLogin(Customizer.withDefaults());
	return http.build();
}
```

|№|Описание|
|---|---|
|(1)|Указывает, что все конечные точки `/profile/**` требуют входа по одноразовому токену (one-time-token login) для авторизации.|

Учитывая приведенную выше конфигурацию, пользователи могут входить в систему с помощью любого поддерживаемого вами механизма. И, если они захотят посетить страницу профиля, Spring Security перенаправит их на страницу входа по одноразовому токену для его получения.

Таким образом, полномочия, предоставляемые пользователю, прямо пропорциональны количеству предоставленных доказательств. Этот адаптивный подход позволяет пользователям предоставлять ровно столько доказательств, сколько необходимо для выполнения их предполагаемых операций.

## 2.0 Авторизация дополнительных областей действия (Scopes)

Вы также можете настроить обработку исключений, чтобы указать Spring Security, как получить отсутствующую область действия (scope).

Рассмотрим приложение, которое требует определенную область действия OAuth 2.0 для данной конечной точки:

```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.requestMatchers("/profile/**")
				.hasAuthority("SCOPE_profile:read")
			.anyRequest().authenticated()
		)
		.x509(Customizer.withDefaults())
		.oauth2Login(Customizer.withDefaults());
	return http.build();
}
```

Если это также настроено с бином `AuthorizationManagerFactory`, подобным этому:
```java
@Bean
AuthorizationManagerFactory<Object> authz() {
	return AuthorizationManagerFactories.multiFactor()
			.requireFactors(
				FactorGrantedAuthority.X509_AUTHORITY, 
				FactorGrantedAuthority
					.AUTHORIZATION_CODE_AUTHORITY
			)
			.build();
}
```

Тогда приложение **потребует сертификат X.509**, а также авторизацию от сервера авторизации OAuth 2.0.

В случае, если пользователь не дал согласие на `profile:read`, это приложение в текущем виде выдаст ошибку 403. Однако, если у вас есть способ для приложения повторно запросить согласие, вы можете реализовать это в `AuthenticationEntryPoint`, как показано ниже:
```java
@Component
class ScopeRetrievingAuthenticationEntryPoint implements AuthenticationEntryPoint {
	@Override
	public void commence(
		HttpServletRequest request, 
		HttpServletResponse response, 
		AuthenticationException authException
	) throws IOException, ServletException {
		response
			.sendRedirect(
				"https://authz.example.org/authorize?scope=profile:read");
	}
}
```

Затем в объявлении вашей цепочки фильтров можно привязать эту точку входа к данному полномочию следующим образом:
```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http, ScopeRetrievingAuthenticationEntryPoint oauth2) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.requestMatchers("/profile/**")
				.hasAuthority("SCOPE_profile:read")
			.anyRequest().authenticated()
		)
		.x509(Customizer.withDefaults())
		.oauth2Login(Customizer.withDefaults())
		
		.exceptionHandling((exceptions) -> exceptions
			.defaultDeniedHandlerForMissingAuthority(
				oauth2, 
				"SCOPE_profile:read"
			)
		);
	return http.build();
}
```
