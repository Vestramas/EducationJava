Общая поддержка конфигурации Java была добавлена в Spring Framework в Spring 3.1. 

**Spring Security 3.2** представила конфигурацию Java, чтобы позволить пользователям **настраивать Spring Security без использования какого-либо XML.**

> Если вы знакомы с конфигурацией Security Namespace (XML) (пространства имен безопасности), вы должны найти довольно много сходств между ней и конфигурацией Java Spring Security.

Spring Security предоставляет множество примеров приложений для демонстрации использования конфигурации Java Spring Security.

# 1. Hello Web Security Java Configuration (Базовая конфигурация веб-безопасности на Java)
Первый шаг — создать нашу конфигурацию Java Spring Security. 

==Конфигурация СОЗДАЕТ СЕРВЛЕТНЫЙ ФИЛЬТР==, известный как `springSecurityFilterChain`, который **отвечает за всю безопасность** (защиту URL-адресов приложения, проверку отправленных имен пользователей и паролей, перенаправление на форму входа и так далее) **внутри вашего приложения.** 

В следующем примере показан самый ==базовый пример конфигурации Java Spring Security:==

> Здесь просто конфигурация Spring Security (без регистрации фильтра)
```java
import org.springframework.beans.factory.annotation.Autowired;

import org.springframework.context.annotation.*;
import org.springframework.security.config.annotation.authentication.builders.*;
import org.springframework.security.config.annotation.web.configuration.*;

@Configuration
@EnableWebSecurity
public class WebSecurityConfig {

	@Bean
	public UserDetailsService userDetailsService() {
		InMemoryUserDetailsManager manager = 
			new InMemoryUserDetailsManager();
		manager
		.createUser(
			User.withDefaultPasswordEncoder()
				.username("user")
				.password("password")
				.roles("USER")
				.build()
		);
		return manager;
	}
}
```

---

**Эта конфигурация** не является сложной или обширной, **но она делает многое:**
- Требует аутентификацию для каждого URL-адреса в вашем приложении
    
- Генерирует для вас форму входа
    
- Позволяет пользователю с именем пользователя `user` и паролем `password` аутентифицироваться с помощью аутентификации на основе формы
    
- Позволяет пользователю выйти из системы
    
- Предотвращение CSRF-атак
    
- Интеграция заголовков безопасности:
    - HTTP Strict Transport Security для безопасных запросов
        
    - Cache Control (который вы можете позже переопределить в своем приложении, чтобы разрешить кэширование ваших статических ресурсов)
        
    - Интеграция X-XSS-Protection
        
    - Интеграция X-Frame-Options для помощи в предотвращении Clickjacking
        
- Интеграция со следующими методами Servlet API:
    - `HttpServletRequest#getRemoteUser()`
        
    - `HttpServletRequest.html#getUserPrincipal()`
        
    - `HttpServletRequest.html#isUserInRole(java.lang.String)`
        
    - `HttpServletRequest.html#login(java.lang.String, java.lang.String)`
        
    - `HttpServletRequest.html#logout()`
        

## 1.1 AbstractSecurityWebApplicationInitializer
==Следующий шаг — зарегистрировать== `springSecurityFilterChain` в WAR-файле. 

Вы **можете сделать это** в конфигурации Java **с помощью поддержки** `WebApplicationInitializer` Spring в среде Servlet 3.0+.

Неудивительно, что **Spring Security предоставляет базовый класс** (`AbstractSecurityWebApplicationInitializer`), ***чтобы гарантировать, что `springSecurityFilterChain` будет зарегистрирован за вас.*** 

Способ использования `AbstractSecurityWebApplicationInitializer` отличается в зависимости от того, используем ли мы уже Spring, или Spring Security является единственным компонентом Spring в нашем приложении.

- [[#1.2 AbstractSecurityWebApplicationInitializer без существующего Spring]] — Используйте эти инструкции, **если вы еще не используете Spring**
    
- [[#1.3 AbstractSecurityWebApplicationInitializer с Spring MVC]] — Используйте эти инструкции, **если вы уже используете Spring**

## 1.2  AbstractSecurityWebApplicationInitializer без существующего Spring
**Если вы не используете Spring** или Spring MVC, в**ам нужно передать** `WebSecurityConfig` **в суперкласс**, ***чтобы гарантировать, что конфигурация будет подхвачена:***
```java
import org.springframework.security.web.context.*;

public class SecurityWebApplicationInitializer
	extends AbstractSecurityWebApplicationInitializer {

	public SecurityWebApplicationInitializer() {
		super(WebSecurityConfig.class);
	}
}
```

`SecurityWebApplicationInitializer`:
- ==Автоматически регистрирует фильтр== `springSecurityFilterChain` для каждого URL-адреса в вашем приложении.
    
- **Добавляет** `ContextLoaderListener`, который загружает `WebSecurityConfig`.
    

## 1.3 AbstractSecurityWebApplicationInitializer с Spring MVC
Если мы используем Spring в другом месте нашего приложения, ***у нас, вероятно, уже есть*** `WebApplicationInitializer`, **который загружает нашу конфигурацию Spring.** 

**Если мы используем предыдущую конфигурацию, мы получим ошибку.** 

Вместо этого мы должны зарегистрировать Spring Security в существующем `ApplicationContext`. 

Например, если мы используем Spring MVC, наш `SecurityWebApplicationInitializer` может выглядеть примерно так:
```java
import org.springframework.security.web.context.*;

public class SecurityWebApplicationInitializer
	extends AbstractSecurityWebApplicationInitializer {

}
```

**Это только регистрирует** `springSecurityFilterChain` для каждого URL-адреса в вашем приложении. 

После этого ***нам нужно убедиться, что*** `WebSecurityConfig` б**ыл загружен в нашем существующем** `ApplicationInitializer`. 

Например, если мы используем Spring MVC, он добавляется в `getServletConfigClasses()`:
```java
public class MvcWebApplicationInitializer extends
		AbstractAnnotationConfigDispatcherServletInitializer {

	@Override
	protected Class<?>[] getServletConfigClasses() {
		return new Class[] { WebSecurityConfig.class, WebMvcConfig.class };
	}

	// ... другие переопределения ...
}
```

Причина этого в том, что Spring Security должна иметь возможность проверять некоторую конфигурацию Spring MVC, чтобы правильно настроить базовые сопоставители запросов (request matchers), ==поэтому они должны находиться в одном контексте приложения.== 

> [!IMPORTANT] ВАЖНО УСЛОВИЕ ОДНОГО КОНТЕКСТА
> Класс конфигурации приложения И класс конфигурации безопасности, ДОЛЖНЫ БЫТЬ В ОДНОМ КОНТЕКСТЕ

Размещение Spring Security в `getRootConfigClasses` помещает его в родительский контекст приложения, который может не найти `PathPatternParser` Spring MVC.

### Настройка для нескольких диспетчеров Spring MVC
При желании любая конфигурация Spring Security, не связанная с Spring MVC, может быть размещена в другом классе конфигурации, например так:
```java
public class MvcWebApplicationInitializer extends
		AbstractAnnotationConfigDispatcherServletInitializer {

	@Override
    protected Class<?>[] getRootConfigClasses() {
		return new Class[] { NonWebSecurityConfig.class };
    }

	@Override
	protected Class<?>[] getServletConfigClasses() {
		return new Class[] { WebSecurityConfig.class, WebMvcConfig.class };
	}

	// ... другие переопределения ...
}
```

Это может быть полезно, если у вас есть несколько экземпляров `AbstractAnnotationConfigDispatcherServletInitializer` и вы не хотите дублировать общую конфигурацию безопасности в обоих из них.

# 2. HttpSecurity
До сих пор наш `WebSecurityConfig` содержит только информацию о том, как аутентифицировать наших пользователей. 

Откуда Spring Security знает, что мы хотим требовать аутентификации всех пользователей? Откуда Spring Security знает, что мы хотим поддерживать аутентификацию на основе формы? 

На самом деле существует **класс конфигурации** (называемый `SecurityFilterChain`), который вызывается за кулисами. 
==Он настроен со следующей реализацией по умолчанию:==
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.anyRequest().authenticated()
		)
		.formLogin(Customizer.withDefaults())
		.httpBasic(Customizer.withDefaults());
	return http.build();
}
```

Конфигурация по умолчанию (показанная в предыдущем примере):
- Гарантирует, что любой запрос к нашему приложению требует аутентификации пользователя
    
- Позволяет пользователям аутентифицироваться с помощью входа на основе формы
    
- Позволяет пользователям аутентифицироваться с помощью базовой аутентификации HTTP
    

Обратите внимание, что эта конфигурация параллельна конфигурации пространства имен XML:
```xml
<http>
	<intercept-url pattern="/**" access="authenticated"/>
	<form-login />
	<http-basic />
</http>
```

## 2.1 Несколько экземпляров HttpSecurity
Для эффективного управления безопасностью в приложении, где определенные области требуют разной защиты, мы можем использовать несколько цепочек фильтров вместе с DSL-методом `securityMatcher`. 
Этот подход позволяет нам определять отдельные конфигурации безопасности, адаптированные для конкретных частей приложения, повышая общую безопасность и контроль приложения.

> **Мы можем настроить несколько экземпляров** `HttpSecurity` так же, как мы можем иметь несколько блоков `<http>` в XML. 

Ключевым моментом является регистрация нескольких бинов `SecurityFilterChain @Bean`. 
В следующем примере показана другая конфигурация для URL-адресов, начинающихся с `/api/`:
```java
@Configuration
@EnableWebSecurity
public class MultiHttpSecurityConfig {
	@Bean // (1)
	public UserDetailsService userDetailsService() throws Exception {
		UserBuilder users = User.withDefaultPasswordEncoder();
		
		InMemoryUserDetailsManager manager = 
			new InMemoryUserDetailsManager();
		manager
			.createUser(
				users.username("user")
				.password("password")
				.roles("USER")
				.build()
			);
		manager
			.createUser(
				users.username("admin")
				.password("password")
				.roles("USER","ADMIN")
				.build()
			);
		return manager;
	}

	@Bean
	@Order(1) // (2)
	public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception {
		http
			.securityMatcher("/api/**") // (3)
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().hasRole("ADMIN")
			)
			.httpBasic(Customizer.withDefaults());
		return http.build();
	}

	@Bean // (4)
	public SecurityFilterChain formLoginFilterChain(HttpSecurity http) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.formLogin(Customizer.withDefaults());
		return http.build();
	}
}
```

| №   | Описание                                                                                                                                                                                                                                                                                                            |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Настройка аутентификации как обычно.                                                                                                                                                                                                                                                                                |
| (2) | Создание экземпляра `SecurityFilterChain`, который **содержит** `@Order`, **чтобы указать, какой** `SecurityFilterChain` ==должен рассматриваться первым.==                                                                                                                                                         |
| (3) | `http.securityMatcher()` ***утверждает, что этот `HttpSecurity` применим только к URL-адресам, начинающимся с `/api/`.***                                                                                                                                                                                           |
| (4) | **Создание другого экземпляра** `SecurityFilterChain`. <br><br>**Если URL-адрес НЕ начинается с** `/api/`, **используется эта конфигурация.** <br><br>Эта конфигурация рассматривается после `apiFilterChain`, поскольку она имеет значение `@Order` после 1 (отсутствие `@Order` по умолчанию означает последний). |

## 2.2 Выбор между `securityMatcher` и `requestMatchers`

Распространенный вопрос:

> В чем разница между методом `http.securityMatcher()` и `requestMatchers()`, используемым для авторизации запросов (т.е. внутри `http.authorizeHttpRequests()`)?

Чтобы ответить на этот вопрос, полезно понимать, что ***каждый экземпляр*** `HttpSecurity`, используемый для построения `SecurityFilterChain`, **содержит `RequestMatcher` для сопоставления входящих запросов.** 


> [!NOTE] `@Order(1)` - приоритет фильтров безопасности
> **Если запрос не соответствует** `SecurityFilterChain` с более высоким приоритетом (например, `@Order(1)`), ***запрос может быть опробован против цепочки фильтров с более низким приоритетом*** (например, без `@Order`).

Логика сопоставления для нескольких цепочек фильтров выполняется `FilterChainProxy`.

> [!NOTE] РАЗНИЦА
> `RequestMatcher` ==по умолчанию соответствует любому запросу==, чтобы гарантировать, что Spring Security защищает все запросы по умолчанию.
> 
> Указание `securityMatcher` ==переопределяет это значение по умолчанию.==
> 
> 
> ⭐`securityMatcher`⭐ — это метод, который говорит цепочке фильтров безопасности: **"Обрабатывай только те запросы, которые соответствуют этому шаблону"**.

Если ни одна цепочка фильтров не соответствует конкретному запросу, запрос не защищен Spring Security.

Происходит следующее:
1. Создаётся цепочка фильтров, которая применяется **только** к запросам, начинающимся с `/api/`.
    
2. Все остальные запросы (например, `/login`, `/logout`) **игнорируются** этой цепочкой.
    
3. Внутри этой цепочки создаются фильтры (например, `BasicAuthenticationFilter`), но они работают только для тех запросов, которые прошли `securityMatcher`.

Следующий пример демонстрирует одну цепочку фильтров, которая защищает только запросы, начинающиеся с `/secured/`:
```java
@Configuration
@EnableWebSecurity
public class PartialSecurityConfig {

	@Bean
	public UserDetailsService userDetailsService() throws Exception {
		// ...
	}

	@Bean
	public SecurityFilterChain securedFilterChain(HttpSecurity http) throws Exception {
		http
			.securityMatcher("/secured/**") // (1)
			.authorizeHttpRequests((authorize) -> authorize
				.requestMatchers("/secured/user").hasRole("USER") // (2)
				.requestMatchers("/secured/admin").hasRole("ADMIN") // (3)
				.anyRequest().authenticated() // (4)
			)
			.httpBasic(Customizer.withDefaults())
			.formLogin(Customizer.withDefaults());
		return http.build();
	}
}
```

⭐Происходит следующее⭐:
1. Создаётся цепочка фильтров, которая применяется **только** к запросам, начинающимся с `/secured/`.
    
2. Все остальные запросы (например, `/login`, `/logout`) **игнорируются** этой цепочкой.
    
3. Внутри этой цепочки создаются фильтры (например, `BasicAuthenticationFilter`), но они работают только для тех запросов, которые прошли `securityMatcher`.

|№|Описание|
|---|---|
|(1)|Запросы, начинающиеся с `/secured/`, будут защищены, но любые другие запросы не защищены.|
|(2)|Запросы к `/secured/user` требуют полномочия `ROLE_USER`.|
|(3)|Запросы к `/secured/admin` требуют полномочия `ROLE_ADMIN`.|
|(4)|Любые другие запросы (такие как `/secured/other`) просто требуют аутентифицированного пользователя.|

> Рекомендуется предоставить `SecurityFilterChain`, который не указывает никакого `securityMatcher`, чтобы гарантировать защиту всего приложения, как показано в предыдущем примере.

Обратите внимание, что метод `requestMatchers` применяется только к отдельным правилам авторизации. Каждый перечисленный там запрос также должен соответствовать общему `securityMatcher` для этого конкретного экземпляра `HttpSecurity`, используемого для создания `SecurityFilterChain`. 

Использование `anyRequest()` в этом примере соответствует всем остальным запросам в рамках этого конкретного `SecurityFilterChain` (которые должны начинаться с `/secured/`).

См. Authorize HttpServletRequests для получения дополнительной информации о `requestMatchers`.

## 2.3 Конечные точки SecurityFilterChain
Несколько фильтров в `SecurityFilterChain` напрямую предоставляют **конечные точки, такие как** `UsernamePasswordAuthenticationFilter`, который **настраивается с помощью** `http.formLogin()` **и предоставляет конечную точку** `POST /login`. 

> ❗❗❗ **ВОЗМОЖНАЯ ОШИБКА С Endpoint'ами** ❗❗❗
> В приведенном выше примере конечная точка `/login` не соответствует `http.securityMatcher("/secured/**")`, **и поэтому это приложение не будет иметь ни конечной точки** `GET /login`, ни `POST /login`. Такие запросы будут возвращать `404 Not Found`. 
> 
> Это часто удивляет пользователей.

Указание `http.securityMatcher()` ==влияет на то, какие запросы сопоставляются этим== `SecurityFilterChain`. 
*Однако это не влияет автоматически на конечные точки,* предоставляемые цепочкой фильтров. В таких случаях вам может потребоваться настроить URL-адрес любых конечных точек, которые вы хотите, чтобы цепочка фильтров предоставляла.

Следующий пример демонстрирует конфигурацию, которая защищает запросы, начинающиеся с `/secured/`, и отклоняет все остальные запросы, одновременно настраивая конечные точки, предоставляемые `SecurityFilterChain`:
```java
@Configuration
@EnableWebSecurity
public class SecuredSecurityConfig {

	@Bean
	public UserDetailsService userDetailsService() throws Exception {
		// ...
	}

	@Bean
	@Order(1)
	public SecurityFilterChain securedFilterChain(HttpSecurity http) throws Exception {
		http
			.securityMatcher("/secured/**") // (1)
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated() // (2)
			)
			.formLogin((formLogin) -> formLogin // (3)
				.loginPage("/secured/login")
				.loginProcessingUrl("/secured/login")
				.permitAll()
			)
			.logout((logout) -> logout // (4)
				.logoutUrl("/secured/logout")
				.logoutSuccessUrl("/secured/login?logout")
				.permitAll()
			);
		return http.build();
	}

	@Bean
	public SecurityFilterChain defaultFilterChain(HttpSecurity http) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().denyAll() // (5)
			);
		return http.build();
	}
}
```

|№|Описание|
|---|---|
|(1)|Запросы, начинающиеся с `/secured/`, будут защищены этой цепочкой фильтров.|
|(2)|Запросы, начинающиеся с `/secured/`, требуют аутентифицированного пользователя.|
|(3)|Настройка формы входа с префиксом URL-адресов `/secured/`.|
|(4)|Настройка выхода с префиксом URL-адресов `/secured/`.|
|(5)|Все остальные запросы будут отклонены.|

Этот пример настраивает страницы входа и выхода, что отключает сгенерированные Spring Security страницы. Вы должны предоставить свои собственные пользовательские конечные точки для `GET /secured/login` и `GET /secured/logout`. Обратите внимание, что Spring Security по-прежнему предоставляет для вас конечные точки `POST /secured/login` и `POST /secured/logout`.

## 2.4 Пример из реальной жизни
Следующий пример демонстрирует немного более реальную конфигурацию, объединяющую все эти элементы:
```java
@Configuration
@EnableWebSecurity
public class BankingSecurityConfig {

    @Bean // (1)
    public UserDetailsService userDetailsService() {
		UserBuilder users = User.withDefaultPasswordEncoder();
        InMemoryUserDetailsManager manager =
	        new InMemoryUserDetailsManager();
        manager
	        .createUser(
		        users.username("user1")
		        .password("password")
		        .roles("USER", "VIEW_BALANCE")
		        .build()
		    );
        manager
	        .createUser(
		        users.username("user2")
		        .password("password")
		        .roles("USER")
		        .build()
		    );
        manager
	        .createUser(
		        users.username("admin")
		        .password("password")
		        .roles("ADMIN")
		        .build()
		    );
        return manager;
    }

    @Bean
    @Order(1) // (2)
    public SecurityFilterChain approvalsSecurityFilterChain(
	    HttpSecurity http
	) throws Exception 
	{
        String[] approvalsPaths = { 
	        "/accounts/approvals/**", 
	        "/loans/approvals/**", 
	        "/credit-cards/approvals/**"
	    };
        
        http
            .securityMatcher(approvalsPaths)
            .authorizeHttpRequests((authorize) -> authorize
				.anyRequest().hasRole("ADMIN")
            )
            .httpBasic(Customizer.withDefaults());
        return http.build();
    }

    @Bean
    @Order(2) // (3)
    public SecurityFilterChain bankingSecurityFilterChain(
	    HttpSecurity http
	) throws Exception 
	{
        String[] bankingPaths = { 
	        "/accounts/**", 
	        "/loans/**", 
	        "/credit-cards/**", 
	        "/balances/**" 
	    };
        
		String[] viewBalancePaths = { "/balances/**" };
		
        http
			.securityMatcher(bankingPaths)
			.authorizeHttpRequests((authorize) -> authorize
				.requestMatchers(viewBalancePaths).hasRole("VIEW_BALANCE")
				.anyRequest().hasRole("USER")
            );
        return http.build();
    }

    @Bean                                                              // (4)
    public SecurityFilterChain defaultSecurityFilterChain(HttpSecurity http) throws Exception {
		String[] allowedPaths = { 
			"/", 
			"/user-login", 
			"/user-logout", 
			"/notices", 
			"/contact", 
			"/register"
		};
		
        http
            .authorizeHttpRequests((authorize) -> authorize
				.requestMatchers(allowedPaths).permitAll()
				.anyRequest().authenticated()
            )
			.formLogin((formLogin) -> formLogin
				.loginPage("/user-login")
				.loginProcessingUrl("/user-login")
			)
			.logout((logout) -> logout
				.logoutUrl("/user-logout")
				.logoutSuccessUrl("/?logout")
			);
        return http.build();
    }
}
```

| №   | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1) | Начните с настройки параметров аутентификации.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| (2) | Определите экземпляр `SecurityFilterChain` с `@Order(1)`, что означает, что эта цепочка фильтров будет иметь наивысший приоритет. <br><br>Эта цепочка фильтров применяется только к запросам, начинающимся с `/accounts/approvals/`, `/loans/approvals/` или `/credit-cards/approvals/`. <br><br>Запросы к этой цепочке фильтров требуют полномочия `ROLE_ADMIN` и разрешают базовую аутентификацию HTTP (HTTP Basic Authentication).                                                                                                                                                                                                                                                                    |
| (3) | Затем создайте другой экземпляр `SecurityFilterChain` с `@Order(2)`, который будет рассматриваться вторым. <br><br>Эта цепочка фильтров применяется только к запросам, начинающимся с `/accounts/`, `/loans/`, `/credit-cards/` или `/balances/`. <br><br>**Обратите внимание**, что **поскольку эта цепочка фильтров вторая**, ***любые запросы, включающие `/approvals/`, будут соответствовать предыдущей цепочке фильтров*** и ***НЕ будут сопоставлены этой цепочкой фильтров.*** <br><br>Запросы к этой цепочке фильтров требуют полномочия `ROLE_USER`. Эта цепочка фильтров не определяет никакой аутентификации, поскольку следующая (по умолчанию) цепочка фильтров содержит эту конфигурацию. |
| (4) | Наконец, создайте дополнительный экземпляр `SecurityFilterChain` без аннотации `@Order`. <br><br>Эта конфигурация будет обрабатывать запросы, не охваченные другими цепочками фильтров, и будет обработана последней (отсутствие `@Order` по умолчанию означает последнюю). <br><br>Запросы, соответствующие `/`, `/user-login`, `/user-logout`, `/notices`, `/contact` и `/register`, разрешают доступ без аутентификации. <br><br>Любые другие запросы требуют аутентификации пользователя для доступа к любому URL-адресу, явно не разрешенному или не защищенному другими цепочками фильтров.                                                                                                        |

# 3. Пользовательские DSL (Custom DSLs)
Вы можете предоставить свои **собственные пользовательские DSL** в Spring Security:
```java
public class MyCustomDsl extends AbstractHttpConfigurer<MyCustomDsl, HttpSecurity> {
	private boolean flag;

	@Override
	public void init(HttpSecurity http) {
		// любой метод, который добавляет другой конфигуратор
		// должен выполняться в методе init
		http.csrf(csrf -> csrf.disable());
	}

	@Override
	public void configure(HttpSecurity http) {
		ApplicationContext context = 
		http
			.getSharedObject(ApplicationContext.class);

		// здесь мы ищем в ApplicationContext. Вы также можете просто создать новый экземпляр.
		MyFilter myFilter = context.getBean(MyFilter.class);
		myFilter.setFlag(flag);
		
		http.addFilterBefore(
			myFilter, 
			UsernamePasswordAuthenticationFilter.class
		);
	}

	public MyCustomDsl flag(boolean value) {
		this.flag = value;
		return this;
	}

	public static MyCustomDsl customDsl() {
		return new MyCustomDsl();
	}
}
```

> Это фактически то, как реализованы такие методы, как `HttpSecurity.authorizeHttpRequests()`.

Затем вы можете использовать пользовательский DSL:
```java
@Configuration
@EnableWebSecurity
public class Config {
	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.with(MyCustomDsl.customDsl(), (dsl) -> dsl
				.flag(true)
			)
			// ...
		return http.build();
	}
}
```

Код вызывается в следующем порядке:
1. Вызывается код в методе `Config.filterChain`
    
2. Вызывается код в методе `MyCustomDsl.init`
    
3. Вызывается код в методе `MyCustomDsl.configure`

Если вы хотите, вы можете заставить `HttpSecurity` добавлять `MyCustomDsl` по умолчанию, используя `SpringFactories`. Например, вы можете создать ресурс в classpath с именем `META-INF/spring.factories` со следующим содержимым:

**META-INF/spring.factories**
```
org.springframework.security.config.annotation.web.configurers.AbstractHttpConfigurer = sample.MyCustomDsl
```

Вы также можете явно отключить значение по умолчанию:
```java
@Configuration
@EnableWebSecurity
public class Config {
	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.with(MyCustomDsl.customDsl(), (dsl) -> dsl
				.disable()
			)
			...;
		return http.build();
	}
}
```

# 4. Модульная конфигурация HttpSecurity (Modular HttpSecurity Configuration)
Многие пользователи предпочитают, чтобы их конфигурация Spring Security находилась в централизованном месте, и выбирают настройку в одном экземпляре `SecurityFilterChain`. 

Однако бывают случаи, когда пользователи могут **захотеть модуляризовать конфигурацию.** 

Это можно сделать с помощью:
- `Customizer<HttpSecurity>` Beans
    
- Top Level `HttpSecurity` Customizer Beans

Если вы используете Kotlin Configuration Spring Security, то вы также можете предоставить *Dsl → Unit Beans, как описано в Modular HttpSecurityDsl Configuration.

## 4.1 Бины `Customizer<HttpSecurity>`
Если вы хотите модуляризовать вашу конфигурацию безопасности, вы можете поместить логику в бин `Customizer<HttpSecurity>`. 

Например, следующая конфигурация гарантирует, что все экземпляры `HttpSecurity` будут настроены на:
```java
@Bean
ThrowingCustomizer<HttpSecurity> httpSecurityCustomizer() {
	return (http) -> http
		.headers((headers) -> headers
			.contentSecurityPolicy((csp) -> csp
				// (1)
				.policyDirectives("object-src 'none'")
			)
		)
		// (2)
		.redirectToHttps(Customizer.withDefaults());
}
```
1. Устанавливает для политики безопасности содержимого значение object-src "нет"
2. Перенаправляйте любой запрос на https

## 4.2 Top Level `HttpSecurity` Customizer Beans
Если вы предпочитаете дальнейшую модуляризацию вашей конфигурации безопасности, Spring Security автоматически применит любые Top Level `HttpSecurity` Customizer Beans.

Top Level `HttpSecurity` Customizer тип можно резюмировать как любой `Customizer<T>`, который соответствует `public HttpSecurity.*(Customizer<T>)`.
Это переводится как любой `Customizer<T>`, который является единственным аргументом публичного метода в `HttpSecurity`.

Несколько **примеров могут помочь прояснить.** 
Если `Customizer<ContentTypeOptionsConfig>` опубликован как Bean, он не будет автоматически применен, потому что он является аргументом для `HeadersConfigurer.contentTypeOptions(Customizer)`, который не является методом, определенным в `HttpSecurity`. 

Однако, если `Customizer<HeadersConfigurer<HttpSecurity>>` опубликован как Bean, он будет автоматически применен, потому что он является аргументом для `HttpSecurity.headers(Customizer)`.

Например, следующая конфигурация гарантирует, что Content Security Policy будет установлен в `object-src 'none'`:
```java
@Bean
Customizer<HeadersConfigurer<HttpSecurity>> headersSecurity() {
	return (headers) -> headers
		.contentSecurityPolicy((csp) -> csp
			// (1)
			.policyDirectives("object-src 'none'")
		);
}
```

## 4.3 Порядок применения Customizer Beans
Сначала каждый бин `Customizer<HttpSecurity>` применяется с использованием `ObjectProvider#orderedStream()`. Это означает, что если существует несколько бинов `Customizer<HttpSecurity>`, аннотация `@Order` может быть добавлена к определениям Bean для управления порядком.

Затем ищется каждый тип Top Level `HttpSecurity` Customizer Beans, и каждый применяется с использованием `ObjectProvider#orderedStream()`. 

Если есть два бина `Customizer<HeadersConfigurer<HttpSecurity>>` и два экземпляра `Customizer<HttpsRedirectConfigurer<HttpSecurity>>`, порядок вызова каждого типа `Customizer` не определен. Однако порядок каждого экземпляра `Customizer<HttpsRedirectConfigurer<HttpSecurity>>` определяется `ObjectProvider#orderedStream()` и может контролироваться с помощью `@Order` в определениях Bean.

Наконец, бин `HttpSecurity` внедряется как Bean. Все экземпляры `Customizer` применяются до создания бина `HttpSecurity`. Это позволяет переопределять настройки, предоставленные Customizer Beans.

Ниже приведен пример, иллюстрирующий порядок:
```java
@Bean // (4)
SecurityFilterChain springSecurity(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((requests) -> requests
			.anyRequest().authenticated()
		);
	return http.build();
}

@Bean
@Order(Ordered.LOWEST_PRECEDENCE) // (2)
ThrowingCustomizer<HttpSecurity> userAuthorization() {
	return (http) -> http
		.authorizeHttpRequests((requests) -> requests
			.requestMatchers("/users/**").hasRole("USER")
		);
}

@Bean
@Order(Ordered.HIGHEST_PRECEDENCE) // (1)
ThrowingCustomizer<HttpSecurity> adminAuthorization() {
	return (http) -> http
		.authorizeHttpRequests((requests) -> requests
			.requestMatchers("/admins/**").hasRole("ADMIN")
		);
}

// (3)

@Bean
Customizer<HeadersConfigurer<HttpSecurity>> contentSecurityPolicy() {
	return (headers) -> headers
		.contentSecurityPolicy((csp) -> csp
			.policyDirectives("object-src 'none'")
		);
}

@Bean
Customizer<HeadersConfigurer<HttpSecurity>> contentTypeOptions() {
	return (headers) -> headers
		.contentTypeOptions(Customizer.withDefaults());
}

@Bean
Customizer<HttpsRedirectConfigurer<HttpSecurity>> httpsRedirect() {
	return Customizer.withDefaults();
}
```

|№|Описание|
|---|---|
|(1)|Сначала применяются все экземпляры `Customizer<HttpSecurity>`. Бин `adminAuthorization` имеет наивысший `@Order`, поэтому он применяется первым. Если на бинах `Customizer<HttpSecurity>` нет аннотаций `@Order` или аннотации `@Order` имеют одинаковое значение, то порядок применения экземпляров `Customizer<HttpSecurity>` не определен.|
|(2)|`userAuthorization` применяется следующим, поскольку является экземпляром `Customizer<HttpSecurity>`.|
|(3)|Порядок типов `Customizer` не определен. В этом примере порядок `contentSecurityPolicy`, `contentTypeOptions` и `httpsRedirect` не определен. Если бы `@Order(Ordered.HIGHEST_PRECEDENCE)` был добавлен к `contentTypeOptions`, тогда мы бы знали, что `contentTypeOptions` находится перед `contentSecurityPolicy` (они одного типа), но мы не знаем, находится ли `httpsRedirect` до или после бинов `Customizer<HeadersConfigurer<HttpSecurity>>`.|
|(4)|После применения всех Customizer Beans бин `HttpSecurity` передается как Bean.|

# 5. Постобработка настроенных объектов (Post Processing Configured Objects)
Конфигурация Java Spring Security не раскрывает каждое свойство каждого объекта, который она настраивает. Это упрощает конфигурацию для большинства пользователей. В конце концов, если бы каждое свойство было раскрыто, пользователи могли бы использовать стандартную конфигурацию бинов.

Хотя есть веские причины не раскрывать напрямую каждое свойство, пользователям все же **могут потребоваться более продвинутые параметры конфигурации.** 

Чтобы решить эту проблему, Spring Security вводит концепцию `ObjectPostProcessor`, который можно использовать для изменения или замены многих экземпляров `Object`, созданных конфигурацией Java.

Например, чтобы настроить свойство `filterSecurityPublishAuthorizationSuccess` в `FilterSecurityInterceptor`, вы можете использовать следующее:
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
	http
		.authorizeHttpRequests((authorize) -> authorize
			.anyRequest().authenticated()
			.withObjectPostProcessor(
				new ObjectPostProcessor<FilterSecurityInterceptor>() 
				{
				public <O extends FilterSecurityInterceptor> O postProcess
					(
						O fsi
					) {
						fsi.setPublishAuthorizationSuccess(true);
						return fsi;
					}
				}
			)
		);
	return http.build();
}
```
