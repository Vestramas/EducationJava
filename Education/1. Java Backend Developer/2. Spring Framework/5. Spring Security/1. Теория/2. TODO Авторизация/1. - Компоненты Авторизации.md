После настройки аутентификации необходимо определить правила авторизации приложения. Одним из главных преимуществ Spring Security являются его развитые возможности управления доступом. 

Независимо от выбранного способа аутентификации, механизмы авторизации могут использоваться единообразно и удобно.

Обычно правила авторизации связываются с URI-запросов и методами приложения, что позволяет гибко ограничивать доступ к ресурсам. Также Spring Security позволяет отслеживать события авторизации и реагировать на результаты проверок доступа.

Начиная с Spring Security 7, устаревший Access API (`AccessDecisionManager`, `AccessDecisionVoter` и другие) вынесен в отдельный модуль `spring-security-access`. Для новых приложений рекомендуется использовать современный Authorization API.

# 1. Архитектура авторизации
В этом разделе описывается архитектура Spring Security, применимая к авторизации.

## 1.1 Полномочия (Authorities)
В разделе «Аутентификация» обсуждается, как все реализации `Authentication` хранят список объектов `GrantedAuthority`. 

Они представляют полномочия, которые были предоставлены принципалу. 

Объекты `GrantedAuthority` вставляются в объект `Authentication` с помощью `AuthenticationManager` и позже считываются экземплярами `AuthorizationManager` при принятии решений об авторизации.

Интерфейс `GrantedAuthority` имеет только один метод:
```java
String getAuthority();
```

Этот метод используется экземпляром `AuthorizationManager` для получения точного строкового представления `GrantedAuthority`. Возвращая представление в виде `String`, `GrantedAuthority` может быть легко «прочитан» большинством реализаций `AuthorizationManager`.

Если `GrantedAuthority` не может быть точно представлен в виде `String`, `GrantedAuthority` считается «сложным», и `getAuthority()` должен возвращать `null`.

***Примером сложного*** `GrantedAuthority` может быть реализация, ***которая хранит список операций и порогов полномочий***, применимых к разным номерам клиентских счетов. 
Представить такой сложный `GrantedAuthority` в виде `String` было бы довольно сложно. В результате метод `getAuthority()` должен возвращать `null`. 
Это указывает любому `AuthorizationManager`, что ему необходимо поддерживать конкретную реализацию `GrantedAuthority` для понимания ее содержимого.

### 1.1.1 Реализация SimpleGrantedAuthority
Spring Security включает **одну конкретную реализацию** `GrantedAuthority`: `SimpleGrantedAuthority`. 

Эта реализация позволяет преобразовать любую указанную пользователем строку (`String`) в `GrantedAuthority`. 
Все экземпляры `AuthenticationProvider`, включенные в архитектуру безопасности, используют `SimpleGrantedAuthority` для заполнения объекта `Authentication`.

#### ПРАВИЛО ПРЕФИКСА ROLE_
По умолчанию правила авторизации на основе ролей включают префикс `ROLE_`. Это означает, что если существует правило авторизации, требующее, чтобы контекст безопасности имел роль «USER», Spring Security по умолчанию будет искать `GrantedAuthority#getAuthority`, который возвращает «ROLE_USER».

Вы можете настроить это с помощью `GrantedAuthorityDefaults`. 
`GrantedAuthorityDefaults` существует для того, чтобы позволить настроить префикс, используемый для правил авторизации на основе ролей.

Вы можете настроить правила авторизации на использование другого префикса, предоставив бин `GrantedAuthorityDefaults`, например:

**Настройка GrantedAuthorityDefaults**
```java
@Bean
static GrantedAuthorityDefaults grantedAuthorityDefaults() {
	return new GrantedAuthorityDefaults("MYPREFIX_");
}
```

> **Примечание:** Вы предоставляете `GrantedAuthorityDefaults` с помощью статического (`static`) метода, чтобы гарантировать, что Spring опубликует его до инициализации классов конфигурации безопасности методов (`@Configuration`) Spring Security.

## 1.2 Обработка вызовов (Invocation Handling)
Spring Security предоставляет перехватчики (interceptors), которые контролируют доступ к защищенным объектам, таким как вызовы методов или веб-запросы. 

Решение о том, разрешено ли выполнение вызова до его начала (pre-invocation), принимается экземплярами `AuthorizationManager`. 

Также решения о том, может ли быть возвращено определенное значение после выполнения вызова (post-invocation), принимаются экземплярами `AuthorizationManager`.

### 1.2.1 AuthorizationManager
`AuthorizationManager` заменяет как `AccessDecisionManager`, так и `AccessDecisionVoter`.

Приложениям, которые настраивают `AccessDecisionManager` или `AccessDecisionVoter`, рекомендуется перейти на использование `AuthorizationManager`.

`AuthorizationManager` ***вызываются компонентами авторизации Spring Security на основе запросов***, методов и сообщений ==и отвечают за принятие окончательных решений по контролю доступа==. 

Интерфейс `AuthorizationManager` содержит два метода:
```java
AuthorizationResult authorize(
	Supplier<Authentication> authentication, 
	Object secureObject);

default void verify(
	Supplier<Authentication> authentication, 
	Object secureObject) throws AccessDeniedException {
    // ...
}
```

Методу `authorize` интерфейса `AuthorizationManager` **передается вся необходимая информация для принятия решения об авторизации.**
В частности, передача защищенного объекта (`secure Object`) позволяет проверять аргументы, содержащиеся в фактическом вызове защищенного объекта. 
> Например, предположим, что защищенный объект — это `MethodInvocation`. 
> 
> Было бы легко запросить `MethodInvocation` на предмет наличия аргумента `Customer`, а затем реализовать некую логику безопасности в `AuthorizationManager`, чтобы убедиться, что принципалу разрешено работать с этим клиентом.
> 
> Ожидается, что реализации будут возвращать положительное `AuthorizationDecision`, если доступ разрешен, отрицательное `AuthorizationDecision`, если в доступе отказано, и `null` `AuthorizationDecision`, если реализация воздерживается от принятия решения.

Метод `verify` вызывает `authorize` и впоследствии выбрасывает `AccessDeniedException` в случае отрицательного `AuthorizationDecision`.

### 1.2.2 Реализации AuthorizationManager на основе делегирования
Хотя пользователи могут реализовать свой собственный `AuthorizationManager` для управления всеми аспектами авторизации, **Spring Security поставляется с делегирующим** `AuthorizationManager`, который может сотрудничать с отдельными `AuthorizationManager`.

1. `RequestMatcherDelegatingAuthorizationManager` сопоставит запрос с наиболее подходящим делегатом `AuthorizationManager`. 
	
2. Для безопасности методов вы можете использовать `AuthorizationManagerBeforeMethodInterceptor` и `AuthorizationManagerAfterMethodInterceptor`.

#### AuthorityAuthorizationManager
Наиболее распространенным `AuthorizationManager`, предоставляемым Spring Security, является `AuthorityAuthorizationManager`. 

Он **настраивается с заданным набором полномочий** ***для поиска в текущем*** `Authentication`.
Он вернет положительное `AuthorizationDecision`, если `Authentication` содержит любое из настроенных полномочий. 
В противном случае он вернет отрицательное `AuthorizationDecision`

#### AuthenticatedAuthorizationManager
Другим менеджером является `AuthenticatedAuthorizationManager`. 

Его можно использовать для различения анонимных, полностью аутентифицированных и аутентифицированных через remember-me пользователей. 
Многие сайты разрешают определенный ограниченный доступ при аутентификации через remember-me, но требуют от пользователя подтверждения своей личности путем входа в систему для полного доступа.

### 1.2.3 Создание экземпляров AuthorizationManager
Интерфейс `AuthorizationManagerFactory` (представлен в Spring Security 7.0) используется для создания универсальных `AuthorizationManager` в компонентах авторизации на основе запросов и методов. 

Ниже представлен набросок интерфейса `AuthorizationManagerFactory`:
```java
public interface AuthorizationManagerFactory<T> {
	AuthorizationManager<T> permitAll();
	AuthorizationManager<T> denyAll();
	AuthorizationManager<T> hasRole(String role);
	AuthorizationManager<T> hasAnyRole(String... roles);
	AuthorizationManager<T> hasAllRoles(String... roles);
	AuthorizationManager<T> hasAuthority(String authority);
	AuthorizationManager<T> hasAnyAuthority(String... authorities);
	AuthorizationManager<T> hasAllAuthorities(String... authorities);
	AuthorizationManager<T> authenticated();
	AuthorizationManager<T> fullyAuthenticated();
	AuthorizationManager<T> rememberMe();
	AuthorizationManager<T> anonymous();
}
```

Реализацией по умолчанию является `DefaultAuthorizationManagerFactory`, которая позволяет настраивать `rolePrefix` (по умолчанию `"ROLE_"`), `RoleHierarchy` и `AuthenticationTrustManager`, которые предоставляются `AuthorizationManager`, создаваемым фабрикой.

Чтобы настроить экземпляр по умолчанию, используемый Spring Security, просто опубликуйте бин, как в следующем примере:
```java
@Bean
<T> AuthorizationManagerFactory<T> authorizationManagerFactory() {
	DefaultAuthorizationManagerFactory<T> 
	authorizationManagerFactory =
		new DefaultAuthorizationManagerFactory<>();
	
	
	authorizationManagerFactory
		.setTrustResolver(
			getAuthenticationTrustResolver()
		);
		
	authorizationManagerFactory
		.setRoleHierarchy(
			getRoleHierarchy()
		);
		
	authorizationManagerFactory.setRolePrefix("role_");

	return authorizationManagerFactory;
}
```

> **Примечание:** Также возможно настроить таргетинг на конкретное использование этой фабрики в Spring Security, предоставив конкретный параметризованный тип вместо универсального типа. 
> См. примеры каждого из них в разделах документации, посвященных авторизации на основе запросов и методов.

В дополнение к простой настройке экземпляра `AuthorizationManagerFactory` по умолчанию, вы можете предоставить свою собственную реализацию, чтобы полностью настроить создаваемые фабрикой экземпляры и предоставить собственные реализации.

> **Примечание:** Фактический интерфейс предоставляет реализации по умолчанию для всех фабричных методов, что позволяет пользовательским реализациям реализовывать только те методы, которые необходимо настроить.

---
#### AuthorizationManagerS - утилитный для создания фабрик
Существуют также полезные ==статические фабрики== в `AuthorizationManagers` для объединения отдельных `AuthorizationManager` в более сложные выражения.

---
#### Пользовательские Authorization Manager
Очевидно, вы также можете реализовать собственный `AuthorizationManager` и поместить в него практически любую логику контроля доступа, какую захотите. Она может быть специфичной для вашего приложения (связанной с бизнес-логикой) или реализовывать некоторую логику администрирования безопасности. 

Например, вы можете создать реализацию, которая может запрашивать Open Policy Agent или вашу собственную базу данных авторизации.

На веб-сайте Spring есть статья в блоге, в которой описывается, как использовать устаревший `AccessDecisionVoter` для отказа в доступе в реальном времени пользователям, чьи учетные записи были заблокированы. Вы можете достичь того же результата, реализовав `AuthorizationManager`.

## 1.3 Адаптация AccessDecisionManager и AccessDecisionVoter
До появления `AuthorizationManager` Spring Security публиковал `AccessDecisionManager` и `AccessDecisionVoter`

В некоторых случаях, например при миграции старого приложения, может быть желательно ввести `AuthorizationManager`, который вызывает `AccessDecisionManager` или `AccessDecisionVoter`.

Чтобы вызвать существующий `AccessDecisionManager`, вы можете сделать следующее:

**Адаптация AccessDecisionManager**
```java
@Component
public class AccessDecisionManagerAuthorizationManagerAdapter implements AuthorizationManager {

    private final AccessDecisionManager 
	    accessDecisionManager;
    
    private final SecurityMetadataSource 
	    securityMetadataSource;


    @Override
    public AuthorizationResult authorize(
	    Supplier<Authentication> authentication, 
	    Object object
	) {
        try {
            Collection<ConfigAttribute> attributes = 
            this.securityMetadataSource
		        .getAttributes(object);
            
            this.accessDecisionManager
	            .decide(
		            authentication.get(), 
		            object, 
		            attributes
		        );
		        
            return new AuthorizationDecision(true);
        } catch (AccessDeniedException ex) {
            return new AuthorizationDecision(false);
        }
    }

    @Override
    public void verify(
	    Supplier<Authentication> authentication, 
	    Object object
	) {
        Collection<ConfigAttribute> attributes = 
	    this.securityMetadataSource
	        .getAttributes(object);
	        
        this.accessDecisionManager
	        .decide(
		        authentication.get(), 
		        object, 
		        attributes
		    );
    }
}
```

А затем подключите его в ваш `SecurityFilterChain`.

Или, чтобы вызвать только `AccessDecisionVoter`, вы можете сделать так:

**Адаптация AccessDecisionVoter**
```java
@Component
public class AccessDecisionVoterAuthorizationManagerAdapter implements AuthorizationManager {
    private final AccessDecisionVoter accessDecisionVoter;
    private final SecurityMetadataSource securityMetadataSource;

    @Override
    public AuthorizationResult authorize(
	    Supplier<Authentication> authentication, 
	    Object object
	) {
        Collection<ConfigAttribute> attributes = 
        this.securityMetadataSource
	        .getAttributes(object);
        
        int decision = 
        this.accessDecisionVoter
	        .vote(
		        authentication.get(), 
		        object, 
		        attributes
		    );
		    
        switch (decision) {
        case ACCESS_GRANTED:
            return new AuthorizationDecision(true);
        case ACCESS_DENIED:
            return new AuthorizationDecision(false);
        }
        return null;
    }
}
```

А затем подключите его в ваш `SecurityFilterChain`.

## 1.4 Иерархические роли (Hierarchical Roles)
**Распространенным требованием** является то, что **определенная роль** в приложении д**олжна автоматически «включать» другие роли**. 

Например, в приложении, ***где есть понятие роли «admin» и «user»,*** вы можете захотеть, чтобы **администратор мог делать все, что может обычный пользователь.** 

Чтобы добиться этого, вы можете 
- либо убедиться, что ***всем пользователям-администраторам также назначена роль «user»***. 
- Либо вы можете ***изменить каждое ограничение доступа, требующее роль «user», чтобы оно также включало роль «admin»***. Это может стать довольно сложным, если в вашем приложении много разных ролей.

Использование ==иерархии ролей позволяет вам настроить, какие роли (или полномочия) должны включать другие.== 

Это поддерживается для авторизации на основе фильтров в `HttpSecurity#authorizeHttpRequests` и для авторизации на основе методов через `DefaultMethodSecurityExpressionHandler` для аннотаций pre-post, `SecuredAuthorizationManager` для `@Secured` и `Jsr250AuthorizationManager` для аннотаций JSR-250. 

Вы можете настроить поведение для всех них одновременно следующим образом:

**Конфигурация иерархических ролей**
```java
@Bean
static RoleHierarchy roleHierarchy() {
    return RoleHierarchyImpl.withDefaultRolePrefix()
        .role("ADMIN").implies("STAFF")
        .role("STAFF").implies("USER")
        .role("USER").implies("GUEST")
        .build();
}

// и, если используется безопасность методов pre-post, также добавьте
@Bean
static MethodSecurityExpressionHandler methodSecurityExpressionHandler(
	RoleHierarchy roleHierarchy
) {
	DefaultMethodSecurityExpressionHandler 
	expressionHandler = 
		new DefaultMethodSecurityExpressionHandler();
		
	expressionHandler.setRoleHierarchy(roleHierarchy);
	return expressionHandler;
}
```

Здесь у нас есть четыре роли в иерархии `ROLE_ADMIN ⇒ ROLE_STAFF ⇒ ROLE_USER ⇒ ROLE_GUEST`. Пользователь, аутентифицированный с `ROLE_ADMIN`, будет вести себя так, как если бы у него были все четыре роли, когда ограничения безопасности оцениваются по любым правилам на основе фильтров или методов.

> Символ `>` можно рассматривать как означающий «включает».

Иерархии ролей предлагают удобное средство упрощения данных конфигурации контроля доступа для вашего приложения и/или уменьшения количества полномочий, которые необходимо назначить пользователю. 

Для более сложных требований вы можете определить логическое сопоставление между конкретными правами доступа, которые требуются вашему приложению, и ролями, назначаемыми пользователям, преобразуя одно в другое при загрузке информации о пользователе.

## 1.5 Устаревшие компоненты авторизации

> **Примечание:** Spring Security содержит некоторые устаревшие компоненты. Поскольку они еще не удалены, документация включена в исторических целях. Их рекомендуемые замены указаны выше.

При доступе к устаревшим компонентам авторизации также включите зависимость `spring-security-access` следующим образом:
```xml
<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-access</artifactId>
</dependency>
```

### 1.5.1 AccessDecisionManager
`AccessDecisionManager` вызывается `AbstractSecurityInterceptor` и **отвечает за принятие окончательных решений по контролю доступа**. 

Интерфейс `AccessDecisionManager` содержит три метода:
```java
void decide(
	Authentication authentication, 
	Object secureObject,
	Collection<ConfigAttribute> attrs
) throws AccessDeniedException;

boolean supports(ConfigAttribute attribute);

boolean supports(Class clazz);
```

Методу `decide` интерфейса `AccessDecisionManager` передается вся необходимая информация для принятия решения об авторизации. В частности, передача защищенного объекта (`secure Object`) позволяет проверять аргументы, содержащиеся в фактическом вызове защищенного объекта. 

Например, предположим, что защищенный объект — это `MethodInvocation`. Вы можете запросить `MethodInvocation` на предмет наличия аргумента `Customer`, а затем реализовать некую логику безопасности в `AccessDecisionManager`, чтобы убедиться, что принципалу разрешено работать с этим клиентом. 
Ожидается, что реализации будут выбрасывать `AccessDeniedException`, если в доступе отказано.

Метод `supports(ConfigAttribute)` вызывается `AbstractSecurityInterceptor` во время запуска, чтобы определить, может ли `AccessDecisionManager` обработать переданный `ConfigAttribute`. Метод `supports(Class)` вызывается реализацией перехватчика безопасности, чтобы убедиться, что настроенный `AccessDecisionManager` поддерживает тип защищенного объекта, который представляет перехватчик безопасности.

### 1.5.2 Реализации AccessDecisionManager на основе голосования
Хотя пользователи могут реализовать свой собственный `AccessDecisionManager` для управления всеми аспектами авторизации, Spring Security включает несколько реализаций `AccessDecisionManager`, основанных на голосовании.

На рисунке «Voting Decision Manager» описаны соответствующие классы.
![[Pasted image 20260418125504.png]]
**Рисунок 2. Voting Decision Manager**

Используя этот подход, для принятия решения об авторизации опрашивается ряд реализаций `AccessDecisionVoter`. Затем `AccessDecisionManager` решает, выбрасывать ли `AccessDeniedException`, на основе своей оценки голосов.

Интерфейс `AccessDecisionVoter` имеет три метода:
```java
int vote(
	Authentication authentication, 
	Object object, 
	Collection<ConfigAttribute> attrs);

boolean supports(ConfigAttribute attribute);

boolean supports(Class clazz);
```

Конкретные реализации возвращают `int`, с возможными значениями, отраженными в статических полях `AccessDecisionVoter` с именами `ACCESS_ABSTAIN`, `ACCESS_DENIED` и `ACCESS_GRANTED`. Реализация голосования возвращает `ACCESS_ABSTAIN`, если у нее нет мнения по решению об авторизации. Если у нее есть мнение, она должна вернуть либо `ACCESS_DENIED`, либо `ACCESS_GRANTED`.

В Spring Security предоставляются **три конкретные реализации** `AccessDecisionManager` **для подсчета голосов.** 
Реализация на основе консенсуса (`ConsensusBased`) предоставляет или запрещает доступ на основе консенсуса голосов «не воздержавшихся». Предусмотрены свойства для управления поведением в случае равенства голосов или если все голоса «воздержались». 

Реализация на основе утверждения (`AffirmativeBased`) предоставляет доступ, если был получен один или более голосов `ACCESS_GRANTED` (другими словами, голос «отказать» будет проигнорирован, если есть хотя бы один голос «разрешить»). Как и в реализации `ConsensusBased`, есть параметр, управляющий поведением, если все голосующие воздержались. 

Провайдер на основе единогласия (`UnanimousBased`) ожидает единогласных голосов `ACCESS_GRANTED` для предоставления доступа, игнорируя воздержавшихся. Он отказывает в доступе, если есть хотя бы один голос `ACCESS_DENIED`. Как и в других реализациях, есть параметр, управляющий поведением, если все голосующие воздержались.

Вы можете реализовать собственный `AccessDecisionManager`, который подсчитывает голоса по-другому. Например, голоса от определенного `AccessDecisionVoter` могут иметь дополнительный вес, в то время как голос «отказать» от определенного голосующего может иметь право вето.

#### RoleVoter
Наиболее часто используемым `AccessDecisionVoter`, предоставляемым Spring Security, является `RoleVoter`, который **рассматривает атрибуты конфигурации как имена ролей и голосует за предоставление доступа**, если пользователю была назначена эта роль.

Он голосует, если какой-либо `ConfigAttribute` начинается с префикса `ROLE_`. 

Он голосует за предоставление доступа, если существует `GrantedAuthority`, который возвращает строковое представление (из метода `getAuthority()`), точно равное одному или нескольким `ConfigAttributes`, начинающимся с префикса `ROLE_`. 

Если нет точного совпадения ни с одним `ConfigAttribute`, начинающимся с `ROLE_`, `RoleVoter` голосует за отказ в доступе. Если ни один `ConfigAttribute` не начинается с `ROLE_`, голосующий воздерживается.

#### AuthenticatedVoter
Еще один голосующий, которого мы неявно видели, — это `AuthenticatedVoter`, который можно использовать для различения анонимных, полностью аутентифицированных и аутентифицированных через remember-me пользователей. 

Многие сайты разрешают определенный ограниченный доступ при аутентификации через remember-me, но требуют от пользователя подтверждения своей личности путем входа в систему для полного доступа.

Когда мы использовали атрибут `IS_AUTHENTICATED_ANONYMOUSLY` для предоставления анонимного доступа, этот атрибут обрабатывался `AuthenticatedVoter`. Для получения дополнительной информации см. `AuthenticatedVoter`.

#### Пользовательские голосующие
Вы также можете реализовать собственный `AccessDecisionVoter` и поместить в него практически любую логику контроля доступа, какую захотите. Она может быть специфичной для вашего приложения (связанной с бизнес-логикой) или реализовывать некоторую логику администрирования безопасности. 

Например, на веб-сайте Spring есть статья в блоге, в которой описывается, как использовать голосующего для отказа в доступе в реальном времени пользователям, чьи учетные записи были заблокированы.

Как и многие другие части Spring Security, `AfterInvocationManager` имеет одну конкретную реализацию, `AfterInvocationProviderManager`, которая опрашивает список `AfterInvocationProvider`. Каждому `AfterInvocationProvider` разрешено изменять возвращаемый объект или выбрасывать `AccessDeniedException`. Более того, несколько провайдеров могут изменять объект, поскольку результат предыдущего провайдера передается следующему в списке.

> **Пожалуйста, имейте в виду:** Если вы используете `AfterInvocationManager`, вам все равно понадобятся атрибуты конфигурации, которые позволяют `AccessDecisionManager` `MethodSecurityInterceptor` разрешить операцию. 
> 
> Если вы используете типичные реализации `AccessDecisionManager`, включенные в Spring Security, отсутствие атрибутов конфигурации, определенных для конкретного вызова защищенного метода, приведет к тому, что каждый `AccessDecisionVoter` воздержится от голосования. 
> 
> В свою очередь, если свойство `AccessDecisionManager` `allowIfAllAbstainDecisions` равно `false`, будет выброшено исключение `AccessDeniedException`. 
> Вы можете избежать этой потенциальной проблемы, либо (i) установив `allowIfAllAbstainDecisions` в `true` (хотя это обычно не рекомендуется), либо (ii) просто убедившись, что существует хотя бы один атрибут конфигурации, за который `AccessDecisionVoter` проголосует за предоставление доступа. 
> Этот последний (рекомендуемый) подход обычно достигается с помощью атрибута конфигурации `ROLE_USER` или `ROLE_AUTHENTICATED`.
