# 0. Состав SecurityFilterChain
#### **DisableEncodeUrlFilter**
Отключает кодирование URL (добавление `jsessionid` в URL). Это нужно для безопасности, чтобы ID сессии не «светился» в ссылках и логах сервера, что снижает риск атаки Session Fixation.

#### **WebAsyncManagerIntegrationFilter**
Интегрирует `SecurityContext` (контекст безопасности) с `WebAsyncManager` Spring MVC. Это позволяет данным о пользователе (`Principal`) корректно передаваться в асинхронные потоки (`@Async`, `Callable`).

#### **SecurityContextHolderFilter**
**Ключевой фильтр.** Он достает данные аутентификации из `HttpSession` (через репозиторий) и кладет их в `SecurityContextHolder` (ThreadLocal). Благодаря ему все последующие фильтры и контроллеры знают, кто текущий пользователь.

#### **HeaderWriterFilter**
Добавляет в HTTP-ответ защитные заголовки. Например: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security`.

#### **CsrfFilter**
Проверяет наличие специального токена в запросах, изменяющих состояние (POST, PUT, DELETE). Если токена нет — запрос блокируется (403). Защищает от межсайтовой подделки запросов.

#### **LogoutFilter**
Обрабатывает запрос на ВЫХОД из системы (обычно URL `/logout`). Уничтожает сессию, очищает `SecurityContext` и перенаправляет на страницу входа.

#### **UsernamePasswordAuthenticationFilter**
Обрабатывает данные из стандартной HTML-формы входа (`username` и `password`). Если логин/пароль верны, создает объект `Authentication` и сохраняет пользователя в сессии.

#### **DefaultLoginPageGeneratingFilter**
Генерирует ту самую стандартную (некрасивую) HTML-страницу логина Spring Security, если вы не создали свою собственную.

#### **DefaultLogoutPageGeneratingFilter**
Генерирует стандартную страницу подтверждения выхода (logout).

#### **BasicAuthenticationFilter**
Обрабатывает HTTP Basic Authentication. Ищет в запросе заголовок `Authorization: Basic <base64>`. Используется для REST API или stateless-сервисов.

#### **RequestCacheAwareFilter**
Проверяет, был ли текущий запрос прерван из-за отсутствия аутентификации. Если пользователь только что залогинился, этот фильтр восстанавливает сохраненный URL (например, `/secret-page`) и перенаправляет туда вместо главной страницы.

#### **SecurityContextHolderAwareRequestFilter**
Оборачивает стандартный `HttpServletRequest` в обертку, добавляющую удобные методы безопасности, например `request.isUserInRole("ADMIN")` или `request.getUserPrincipal()`.

#### **AnonymousAuthenticationFilter**
Если **пользователь еще не залогинился**, этот ф**ильтр создает для него фиктивную аутентификацию** — `AnonymousAuthenticationToken` (с ролью `ROLE_ANONYMOUS`). Это нужно, чтобы в коде не было проверок на `null` для контекста.

#### **ExceptionTranslationFilter**
Перехватывает исключения `AccessDeniedException` и `AuthenticationException`, возникающие в фильтрах _ниже_ по цепочке. Если пользователь не вошел — запускает процесс входа, если прав не хватает — показывает 403 ошибку.

#### **AuthorizationFilter**
Последний рубеж. ==Проверяет, имеет ли текущий пользователь право доступа к запрашиваемому URL== **в соответствии с правилами из** `.authorizeHttpRequests()`.

# 1. Подключение к приложению
## 1.1 Что нужно сделать?
После или перед конфигурацией SecurityFilterChain, необходимо не забыть подключить эту цепочку фильтров, к фильтрам контейнера сервлетов (сервера)

Для этого, необходимо:
1. Создать бин `FilterChainProxy` 
2. Зарегистрировать фильтре `DelefationFilterProxy`, который вызовет этот бин

`springSecurityFilterChain` — это **имя бина, который Spring Security создаёт в своём контексте**. 
Этот бин представляет собой `FilterChainProxy` — точку входа для всей логики безопасности. 

> [!IMPORTANT] `@EnableWebSecurity` - создает бин `springSecurityFilterChain`
> Благодаря этой аннотации создается бин `springSecurityFilterChain`

Чтобы он начал обрабатывать HTTP-запросы, его нужно зарегистрировать в сервлет-контейнере как обычный `Filter`.

## 1.2 Как это сделать?
Задачу регистрации выполняет `AbstractSecurityWebApplicationInitializer`. 

Его поведение зависит от того, используется ли в приложении Spring-контекст.

### 1.2.1 Если Spring-контекста нет (например, простое веб-приложение без Spring)
Тогда `springSecurityFilterChain` некому создать. 

Поэтому в конструктор `AbstractSecurityWebApplicationInitializer` нужно явно передать класс конфигурации (`WebSecurityConfig.class`).
```java
import org.springframework.security.web.context.*;

public class SecurityWebApplicationInitializer
	extends AbstractSecurityWebApplicationInitializer {

	public SecurityWebApplicationInitializer() {
		super(WebSecurityConfig.class);
	}
}
```

Он ==создаст минимальный Spring-контекст== (через `ContextLoaderListener`), загрузит в него переданную конфигурацию, получит из неё бин `springSecurityFilterChain` и тоже зарегистрирует его через `DelegatingFilterProxy`. 

Фактически этот **вариант разворачивает Spring-окружение «на лету»** исключительно ради безопасности.

---
### 1.2.2 Если Spring-контекст уже есть (например, Spring MVC)
Если Spring контекст уже создается в нашем приложении, нам нужно
- ***добавить в существующий контекст*** нашу конфигурацию безопасности, 
- ***и зарегистрировать бин*** `springSecurityFilterChain`

#### 1. Добавить в контекст класс конфигурации безопасности
```java
public class MvcWebApplicationInitializer extends
		AbstractAnnotationConfigDispatcherServletInitializer {

	@Override
	protected Class<?>[] getServletConfigClasses() {
		return new Class[] { WebSecurityConfig.class, WebMvcConfig.class };
	}

	// ... другие переопределения ...
}
```

После этого, бин `springSecurityFilterChain` создастся автоматически, поскольку класс, в котором мы настраивали безопасность, помечен аннотацией `@EnableWebSecurity`

> [!NOTE] ВАЖНОЕ УСЛОВИЕ
> `WebSecurityConfig` и `WebMvcConfig` должны быть в одном контексте

Это требование связано с тем, как Spring Security сопоставляет URL-запросы. 

Для правил вроде `.requestMatchers("/admin/**").hasRole("ADMIN")` **используется класс** `MvcRequestMatcher`. 
Он **не просто сравнивает строки, а обращается к** `HandlerMapping` **из Spring MVC.** Это даёт возможность понимать сложные паттерны (например, учитывать `contextPath` или переменные пути).

Если `SecurityConfig` загружается в родительском контексте (через `getRootConfigClasses`), то она не видит бины дочернего контекста, в котором находится `DispatcherServlet` и MVC-инфраструктура. `HandlerMapping` оказывается недоступен, и любой запрос рискует получить 404 или 403 ошибку.

Поэтому оба класса конфигурации помещаются в один и тот же дочерний контекст через `getServletConfigClasses()`. Тогда Spring Security может корректно взаимодействовать с MVC.

---
#### 2. Регистрация бина `springSecurityFilterChain`
```java
import org.springframework.security.web.context.*;

public class SecurityWebApplicationInitializer
	extends AbstractSecurityWebApplicationInitializer {

}
```

##### Создание бина springSecurityFilterChain
В этом случае конфигурация безопасности (`WebSecurityConfig`) ***уже загружается как часть общего контекста*** (через `getServletConfigClasses()`). 

**Поэтому `springSecurityFilterChain` будет создан автоматически.**

---
##### Регистрация фильтра DelegatingFilterProxy
`AbstractSecurityWebApplicationInitializer` ==здесь делает только одно: регистрирует в сервлет-контейнере фильтр== `DelegatingFilterProxy`.  

`DelegatingFilterProxy` — это сервлетный фильтр, который не содержит логики, а просто находит в Spring-контексте бин с именем `springSecurityFilterChain` и передаёт ему все запросы.  

Итог: `SecurityWebApplicationInitializer` выглядит пустым, потому что его работа сводится к минимуму.

---
## 1.3 ДОПОЛНИТЕЛЬНАЯ КОНФИГУРАЦИЯ
### 1.3.1 Настройка для нескольких диспетчеров Spring MVC
При желании любая конфигурация Spring Security, не связанная с Spring MVC, может быть размещена в другом классе конфигурации, например так:
```java
public class MvcWebApplicationInitializer extends
		AbstractAnnotationConfigDispatcherServletInitializer {

	@Override
    protected Class<?>[] getRootConfigClasses() {
		return new Class[] { NonWebSecurityConfig.class };
    }

	@Override
	protected Class<?>[] getServletConfigClasses() {
		return new Class[] { WebSecurityConfig.class, WebMvcConfig.class };
	}

	// ... другие переопределения ...
}
```

Это может быть полезно, если у вас есть несколько экземпляров `AbstractAnnotationConfigDispatcherServletInitializer` и вы не хотите дублировать общую конфигурацию безопасности в обоих из них.

# 2. Конфигурация SecurityFilterChain
Всю цепочку `SecurityFilterChain` мы настраиваем при помощи класса `HttpSecurity`.

> [!IMPORTANT] Описание цепочки фильтров - `HttpSecurity`
> Для **описания ЦЕПОЧКИ ФИЛЬТРОВ**, используется объект `HttpSecurity` - через него и **описывается стандартный порядок фильтрации**

> [!NOTE] Методы - активируют определенный фильтр
> Когда используется метод вроде:
> `.formLogin()`, 
> `.httpBasic()`, 
> `.oauth2Login()`, 
> 
> мы указываем Spring Security:
> *«Добавь в цепочку фильтр(ы), которые обрабатывают этот тип аутентификации».*
> 
> То есть метод `.httpBasic()` не просто включает поддержку Basic Auth – он **добавляет в цепочку фильтр `BasicAuthenticationFilter`**. 
> 
> 
> Нет, ==порядок вызова DSL-методов в== `HttpSecurity` 
> ==НЕ ВЛИЯЕТ на итоговый порядок фильтров в цепочке==.

## 2.1 DSL методы, активирующие фильтры
Некоторые из методов класса `HttpSecurity`, активируют фильтры в `SecurityFilterChain`
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .formLogin(Customizer.withDefaults())   // (1)
        .httpBasic(Customizer.withDefaults())   // (2)
        .oauth2Login(Customizer.withDefaults()) // (3)
        .rememberMe(Customizer.withDefaults())  // (4)
        .anonymous(Customizer.withDefaults())   // (5)
        .logout(Customizer.withDefaults())      // (6)
        .csrf(Customizer.withDefaults())        // (7)
        .authorizeHttpRequests(auth -> auth     // (8)
            .anyRequest().authenticated()
        );
    return http.build();
}
```

| Метод `HttpSecurity`       | Фильтр(ы), которые активируются/настраиваются                                                                                    |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `.formLogin()`             | `UsernamePasswordAuthenticationFilter` <br>+ <br>`DefaultLoginPageGeneratingFilter`<br>+<br>==ДОБАВЛЯЕТСЯ DEFAULT ФОРМА ЛОГИНА== |
| `.httpBasic()`             | `BasicAuthenticationFilter`                                                                                                      |
| `.oauth2Login()`           | `OAuth2LoginAuthenticationFilter`                                                                                                |
| `.rememberMe()`            | `RememberMeAuthenticationFilter`                                                                                                 |
| `.anonymous()`             | `AnonymousAuthenticationFilter`                                                                                                  |
| `.logout()`                | `LogoutFilter`                                                                                                                   |
| `.csrf()`                  | `CsrfFilter`                                                                                                                     |
| `.authorizeHttpRequests()` | `AuthorizationFilter`                                                                                                            |
|                            |                                                                                                                                  |
| `.headers()`               | `HeaderWriterFilter`                                                                                                             |
| `.sessionManagement()`     | `SessionManagementFilter`                                                                                                        |
### 2.1.1 Важные уточнения
1. **Фильтры, которые НЕ ТРЕБУЮТ ЯВНОЙ АКТИВАЦИИ.** 
   Часть фильтров из стандартной цепочки (`SecurityContextHolderFilter`, `ExceptionTranslationFilter`, `RequestCacheAwareFilter` и др.) **всегда присутствуют** ***и не имеют собственного DSL-метода.*** 
   Они являются обязательной инфраструктурой.
    
2. **Активация по умолчанию.** Даже если вы не вызываете метод явно (например, `.csrf()`), *во многих случаях Spring Boot или Spring Security включает защиту по умолчанию.* Явный вызов метода нужен либо для кастомизации, либо для явного отключения (`.csrf().disable()`).
    
3. **Методы, которые НЕ активируют фильтры.** `.userDetailsService()`, `.authenticationProvider()`, `.passwordEncoder()` — эти методы **не добавляют** новых фильтров. Они лишь конфигурируют компоненты, которые используются **внутри** уже активированных фильтров. Например, `DaoAuthenticationProvider` используется внутри `UsernamePasswordAuthenticationFilter`.

---
### 2.1.2 Способы кастомизации фильтров
Есть три основных способа настройки фильтров, каждый со своим сценарием использования.

Но перед тем как начать, скажем про два важных правила при замене фильтров
#### ПРАВИЛА ВСТАВКИ ФИЛЬРОВ
##### Правило 1: Три ключевых фильтра определяют место вставки:
|Фильтр-ориентир|Что гарантирует|Когда размещать после него|
|---|---|---|
|`SecurityContextHolderFilter`|`Authentication` загружен в `SecurityContext`|Фильтрам, которым нужны данные о пользователе|
|`LogoutFilter`|`/logout` уже обработан, сессия жива|Фильтрам аутентификации|
|`AnonymousAuthenticationFilter`|`Authentication` точно не `null` (либо пользователь, либо anonymousUser)|Фильтрам авторизации|

**Практическое правило:**
- Защита от эксплойтов → после `SecurityContextHolderFilter`
    
- Пользовательская аутентификация → после `LogoutFilter`
    
- Проверка прав доступа → после `AnonymousAuthenticationFilter`

---
##### Правило 2: Не дублируйте фильтры
Методы DSL (`.httpBasic()`, `.formLogin()`) уже добавляют соответствующий фильтр в цепочку. 

Попытка добавить тот же фильтр ещё раз через `.addFilterAt()` вызовет исключение.
```java
http
    .httpBasic(Customizer.withDefaults())  // уже добавил BasicAuthenticationFilter
    
    .addFilterAt(
	    myFilter, 
	    BasicAuthenticationFilter.class); // ОШИБКА
```

---
#### Способ 1: Через DSL-методы (для стандартных настроек)
Большинство фильтров **можно настроить через их собственные DSL-конфигураторы**, которые вызываются внутри методов активации.

**Пример: настройка формы входа**
```java
.formLogin(form -> form
    .loginPage("/custom-login")        // своя страница входа
    .loginProcessingUrl("/signin")     // URL для обработки POST-запроса
    .usernameParameter("email")        // кастомное имя параметра
    .defaultSuccessUrl("/home")        // куда перенаправлять после успеха
    .permitAll()                       // разрешить доступ к странице входа
)
```
**Что это даёт:** Меняет поведение `UsernamePasswordAuthenticationFilter` без замены самого фильтра.

---

**Пример: настройка CSRF**
```java
.csrf(csrf -> csrf
    .ignoringRequestMatchers("/api/**")  // отключить CSRF для API
    .csrfTokenRepository(...)           // способ хранения токена
)
```
**Что это даёт:** Настраивает исключения и способ хранения для `CsrfFilter`.

---

**Пример: настройка выхода**
```java
.logout(logout -> logout
    .logoutUrl("/signout")
    .logoutSuccessUrl("/login?logout")
    .deleteCookies("JSESSIONID")
)
```
**Что это даёт:** Меняет URL выхода и поведение `LogoutFilter`.

---
#### Способ 2: ЗАМЕНА РЕАЛИЗАЦИИ фильтра(через `addFilterAt`)
Если вам нужно полностью изменить логику стандартного фильтра, можно заменить его своей реализацией.

**Пример: замена `UsernamePasswordAuthenticationFilter`**
```JAVA
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    // Создаём свою версию фильтра
    UsernamePasswordAuthenticationFilter customFilter = 
        new UsernamePasswordAuthenticationFilter();
    customFilter.setAuthenticationManager(...);
    customFilter.setFilterProcessesUrl("/login");
    customFilter.setUsernameParameter("email");
    
    http
        .addFilterAt(
	        customFilter, // НОВЫЙ ФИЛЬТР
	        UsernamePasswordAuthenticationFilter.class // ЧТО МЕНЯЕМ
	    )
        .formLogin(form -> form
            .loginPage("/login")
            .permitAll()
        );
    
    return http.build();
}
```
**Что это даёт:** Полный контроль над фильтром аутентификации.

---
#### Способ 3: ДОБАВЛЕНИЕ своего фильтра ДО/ПОСЛЕ (через `addFilterBefore`/`addFilterAfter`)
Для добавления логики вокруг стандартных фильтров.

**Пример: логирование перед аутентификацией**
```JAVA
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .addFilterBefore(
            new CustomLoggingFilter(), // ЧТО ДОБАВЛЯЕТ
            UsernamePasswordAuthenticationFilter.class //ПОСЛЕ ЧЕГО
        )
        .formLogin(Customizer.withDefaults());
    
    return http.build();
}
```
**Что это даёт:** Позволяет вставить свою логику в нужное место цепочки фильтров.

---
#### Способ 4: Отключение фильтра
Если стандартный фильтр не нужен вообще.

**Пример: отключение CSRF для API**
```java
.csrf(AbstractHttpConfigurer::disable)
```
Или более явно:
```java
http
    .csrf(csrf -> csrf.disable())
    // остальная конфигурация
```
**Что это даёт:** Полностью удаляет фильтр из цепочки.

---
#### Способ 5: НАПИСАНИЕ СОБСТВЕННОГО ФИЛЬТРА
Чтобы написать собственный фильтр, необходимо реализовать интерфейс `Filter`

Предположим, вы хотите добавить `Filter`, который получает заголовок с ID арендатора (tenant id) и проверяет, есть ли у текущего пользователя доступ к этому арендатору.

Сначала создадим `Filter`:
```java
import java.io.IOException;

import jakarta.servlet.Filter;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.ServletRequest;
import jakarta.servlet.ServletResponse;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

import org.springframework.security.access.AccessDeniedException;

public class TenantFilter implements Filter {

    @Override
    public void doFilter(
	    ServletRequest servletRequest, 
	    ServletResponse servletResponse, 
	    FilterChain filterChain
	)throws IOException, ServletException 
	{
        HttpServletRequest request = 
	        (HttpServletRequest) servletRequest;
	        
        HttpServletResponse response = 
	        (HttpServletResponse) servletResponse;

        String tenantId = 
	        request.getHeader("X-Tenant-Id"); // (1)
        
        boolean hasAccess = 
	        isUserAllowed(tenantId); // (2)
        
        if (hasAccess) {
            filterChain
	            .doFilter(request, response); // (3)
            return;
        }
        
        throw 
        new AccessDeniedException("Access denied"); // (4)
    }
}
```

Пример кода выше делает следующее:

|№|Описание|
|---|---|
|(1)|Получает ID арендатора из заголовка запроса.|
|(2)|Проверяет, есть ли у текущего пользователя доступ к этому ID арендатора.|
|(3)|Если у пользователя есть доступ, вызывает остальные фильтры в цепочке.|
|(4)|Если у пользователя нет доступа, выбрасывает `AccessDeniedException`.|

**Вместо реализации** `Filter` вы **можете унаследоваться от** `OncePerRequestFilter` — это ==базовый класс для фильтров==, которые вызываются только один раз за запрос и предоставляют метод `doFilterInternal` с параметрами `HttpServletRequest` и `HttpServletResponse`.

> И дальше, просто добавляем фильтр в цепочку как показано в способе 2 и 3

## 2.3 Настройка провайдеров аутентификации
Провайдер аутентификации **нужен для того,** ***чтобы по переданному логину и паролю/JWT-токену/OAuth2-коду авторизации***, **аутентифицировать пользователя**

Когда **фильтр** (например, `UsernamePasswordAuthenticationFilter`) получает логин и пароль, он **не проверяет их сам**. 

Вместо этого, фильтр:
- ***создаёт объект-запрос*** `Authentication` (в виде `UsernamePasswordAuthenticationToken`) (непроверенный) 
- и ***передаёт его*** **цепочке проверяющих** — **провайдерам** аутентификации.

> [!NOTE] `AuthenticationProvider` - определяет учетку по переданным данным аутентификации пользователя
> Провайдер (`AuthenticationProvider`) — это компонент, который **умеет проверять** определённый тип учётных данных:
> - `DaoAuthenticationProvider` — логин и пароль через `UserDetailsService`
> - `JwtAuthenticationProvider` — JWT-токен
> - `OAuth2LoginAuthenticationProvider` — OAuth2-код авторизации

---
### 2.3.1 Способы конфигурации провайдеров в HttpSecurity
В `HttpSecurity` есть три метода разного уровня абстракции для настройки провайдеров.

#### Способ 1: `.userDetailsService(UserDetailsService service)` — самый простой
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .userDetailsService(myUserDetailsService)
        .formLogin(Customizer.withDefaults());
    return http.build();
}
```

**Что происходит под капотом:**
1. Spring Security создаёт `DaoAuthenticationProvider`.
    
2. Внедряет в него ваш `UserDetailsService`.
    
3. Регистрирует этого провайдера в глобальном `AuthenticationManager`.

> [!IMPORTANT] Указанный провайдер, применяется ко всей цепочке фильтров
> При указании `.userDetailsService()`, Spring создаст провайдер `DaoAuthenticationProvider`, и этот **провайдер будет применен ко всей цепочке фильтров,** ***кому нужен*** `AuthenticationManager` 

**Когда использовать:** Когда у вас стандартная аутентификация по логину/паролю, и вам нужно только указать, откуда брать пользователей.

---
#### Способ 2: `.authenticationProvider(AuthenticationProvider provider)` — явное указание

> [!NOTE] Комбинация методов конфигурации провайдеров
> Указанные провайдеры в этом методе, ПЕРЕОПРЕДЕЛЯЮТ `DaoAuthenticationProvider`, созданный при помощи `.userDetailsService()`

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
    provider.setUserDetailsService(myUserDetailsService);
    provider.setPasswordEncoder(passwordEncoder());
    
    http
        .authenticationProvider(provider)
        // Можно добавить несколько для разных типов аутентификации
        .authenticationProvider(new CustomAuthenticationProvider())
        .formLogin(Customizer.withDefaults());
    return http.build();
}
```

**Что происходит:** Вы полностью контролируете создание провайдера(ов). Spring Security добавит их в `AuthenticationManager`

> [!NOTE] Провайдер можно определить отдельно как бин, и потом внедрять его в класс конфигурации, и использовать
> 


**Когда использовать:**
- Когда нужно несколько провайдеров (логин/пароль + JWT + сертификат)
    
- Когда нужна тонкая настройка провайдера (например, `setHideUserNotFoundExceptions(false)`)
    
- Когда нужен полностью кастомный провайдер

---
#### Способ 3: `.authenticationManager(AuthenticationManager manager)` — полный контроль

> [!NOTE] Комбинация методов конфигурирования провайдеров
> `.authenticationManager()` имеет **наивысший приоритет**. Всё остальное игнорируется

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    AuthenticationManager manager = new ProviderManager(
        List.of(
            new DaoAuthenticationProvider(...),
            new CustomAuthenticationProvider(...)
        )
    );
    
    http
        .authenticationManager(manager)
        .formLogin(Customizer.withDefaults());
    return http.build();
}
```

**Что происходит:** ==Вы предоставляете готовый== `AuthenticationManager`==со своим набором провайдеров.==
> *Spring Security* **не будет** создавать свой — он ***будет использовать ваш.***

> [!NOTE] Менеджера можно определить отдельно как бин, и потом внедрять его в класс конфигурации, и использовать
> 

**Когда использовать:**
- Когда вам нужен разный набор провайдеров в разных `SecurityFilterChain`
    
- Когда нужна кастомная логика выбора провайдера
    
- Когда нужно настроить `ProviderManager` (например, `setEraseCredentialsAfterAuthentication(false)`)

## 2.5 Обработка исключений
(подробнее см [[1. - Архитектура Spring Security#2.6 Обработка исключений безопасности]])

Для обработки исключений, возникающих при ошибках аутентификации, или при запрете доступа, используется фильтр `ExceptionTranslationFilter`, который ВСЕГДА СУЩЕСТВУЕТ в цепочке фильтров.
### 2.5.1 Случаи срабатывания
С его помощью, настраиваются **две ветки поведения**.
#### 1. Пользователь не аутентифицирован
`AuthenticationException` (**пользователь не аутентифицирован**)  вызывает `AuthenticationEntryPoint` (например, перенаправляет на `/login` или возвращает 401). 

Настраивается через `.exceptionHandling(ex -> ex.authenticationEntryPoint(...))`.

#### 2. Прав недостаточно
`AccessDeniedException` (**прав недостаточно**) → вызывает `AccessDeniedHandler`. 

Настраивается через 
`.exceptionHandling(ex -> ex.accessDeniedPage("/403"))` или через свой хендлер.

---
`ExceptionTranslationFilter` — это действительно центральный обработчик, который перехватывает **только исключения, выброшенные в фильтрах, стоящих ниже него по цепочке**, либо в контроллерах. 

Если вы сами бросите `AccessDeniedException` или `AuthenticationException` в своём коде или в кастомном фильтре (расположенном после `ExceptionTranslationFilter`), он их тоже перехватит.

Для его кастомизации, используется метод  `.exceptionHandling(Customizer.withDefaults())`, который настраивает поведение при отказах доступа. 

Можно задать кастомные страницы ошибок:
```java
.exceptionHandling(exceptions -> exceptions
    .accessDeniedPage("/403")
    .authenticationEntryPoint(customEntryPoint)
)
```
**Когда использовать:** Когда нужно кастомное поведение при 403 (доступ запрещён) или 401 (не аутентифицирован).

---
### 2.5.2 AuthenticationEntryPoint - запрос данных пользователя
`AuthenticationEntryPoint` — это компонент, который определяет, **что именно произойдёт**, когда неаутентифицированный пользователь пытается получить доступ к защищённому ресурсу. 

Проще говоря, это реакция на отсутствие аутентификации.

Используется в фильтре `ExceptionTranslationFilter`, оторый перехватывает `AuthenticationException`. Когда такое исключение возникает, фильтр вызывает `AuthenticationEntryPoint.commence(request, response, exception)`.

Метод `.exceptionHandling()` в `HttpSecurity` позволяет настроить, **какой именно** `AuthenticationEntryPoint` будет использоваться для этой реакции.

#### ВАЖНОЕ УТОЧНЕНИЕ
`AuthenticationEntryPoint` — это **не фильтр**. Это стратегия, которая вызывается внутри `ExceptionTranslationFilter`. Сам этот фильтр всегда присутствует в цепочке, а через `.exceptionHandling()` вы просто выбираете, какую стратегию он будет использовать.

Если вы не настраиваете `.exceptionHandling()` явно, Spring Security автоматически выбирает подходящую реализацию в зависимости от того, какие механизмы аутентификации вы включили:

- `.formLogin()` → `LoginUrlAuthenticationEntryPoint`
    
- `.httpBasic()` → `BasicAuthenticationEntryPoint`
    
- `.oauth2ResourceServer()` → `BearerTokenAuthenticationEntryPoint`


Таким образом, в большинстве случаев вам вообще не нужно трогать эту настройку — Spring Security сам делает правильный выбор. Разработчики обращаются к ней в основном для создания REST API, где стандартные перенаправления не имеют смысла.

---
#### Встроенные реализации
Spring Security предоставляет несколько готовых реализаций, которые автоматически подставляются в зависимости от того, какие механизмы аутентификации вы включили:

| Реализация                                  | Когда используется                              | Что делает                                                                       |
| ------------------------------------------- | ----------------------------------------------- | -------------------------------------------------------------------------------- |
| `LoginUrl`<br>`AuthenticationEntryPoint`    | При использовании <br>`.formLogin()`            | Перенаправляет пользователя на страницу входа (по умолчанию `/login`)            |
| `Basic`<br>`AuthenticationEntryPoint`       | При использовании <br>`.httpBasic()`            | Возвращает HTTP-ответ `401 Unauthorized` с заголовком `WWW-Authenticate: Basic`  |
| `BearerToken`<br>`AuthenticationEntryPoint` | При использовании <br>`.oauth2ResourceServer()` | Возвращает HTTP-ответ `401 Unauthorized` с заголовком `WWW-Authenticate: Bearer` |
| `Http403Forbidden`<br>`EntryPoint`          | Если ничего не настроено (fallback)             | Просто возвращает `403 Forbidden` без каких-либо инструкций                      |

---
#### НАСТРОЙКА
Есть три основных способа настройки `AuthenticationEntryPoint`.
##### Способ 1: Через `.exceptionHandling()`
Самый простой способ — **указать кастомную реализацию для всей цепочки фильтров:**
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .exceptionHandling(exceptions -> exceptions
            .authenticationEntryPoint(
	            (request, response, authException) -> 
	        {
                response.setContentType("application/json");
                response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
                response
	                .getWriter()
		                .write(
			                "{\"error\": \"Authentication required\"}"
			            );
            })
        )
        .authorizeHttpRequests(auth -> auth
	        .anyRequest().authenticated()
	    );
    
    return http.build();
}
```

Здесь вместо стандартного перенаправления на `/login` или выдачи `WWW-Authenticate` заголовка, сервер просто возвращает JSON с ошибкой. Это полезно для REST API.

---
##### Способ 2: Для разных типов запросов
Можно настроить **разную реакцию** в зависимости от типа запроса. 

Например, для HTML-запросов показывать страницу входа, а для API-запросов возвращать JSON:
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .exceptionHandling(exceptions -> exceptions
            .defaultAuthenticationEntryPointFor(
                new LoginUrlAuthenticationEntryPoint("/login"),
                new MediaTypeRequestMatcher(MediaType.TEXT_HTML)
            )
            .defaultAuthenticationEntryPointFor(
                (request, response, authException) -> {
                    response.setContentType("application/json");
                    response.setStatus(
	                    HttpServletResponse.SC_UNAUTHORIZED
	                );
                    response
	                    .getWriter()
		                    .write("{\"error\": \"Unauthorized\"}");
                },
                new MediaTypeRequestMatcher(MediaType.APPLICATION_JSON)
            )
        );
    // ...
}
```

Здесь:
- Если клиент ожидает `text/html` (обычный браузер) → перенаправление на `/login`.
    
- Если клиент ожидает `application/json` (API-клиент) → JSON с ошибкой.

---
##### Способ 3: Создание отдельного бина
Если один и тот же `AuthenticationEntryPoint` используется в нескольких местах, его можно **объявить как бин:**
```java
@Bean
public AuthenticationEntryPoint customAuthenticationEntryPoint() {
    return (request, response, authException) -> {
        response.setContentType("application/json");
        response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
        response
	        .getWriter()
		        .write("{\"error\": \"Please log in first\"}");
    };
}

@Bean
public SecurityFilterChain filterChain(HttpSecurity http, AuthenticationEntryPoint entryPoint) throws Exception {
    http
        .exceptionHandling(exceptions -> exceptions
            .authenticationEntryPoint(entryPoint)
        );
    // ...
    return http.build();
}
```

## 2.6 Сохранение запросов между аутентификациями
Данный механизм срабатывает при помощи фильтра `RequestCacheAwareFilter`, который **всегда** есть в цепочке (между `SecurityContextHolderAwareRequestFilter` и `ExceptionTranslationFilter`).
> Его поведение мы конфигурируем при помощи метода `.requestCache()`

---
`RequestCache` — это механизм Spring Security, который **сохраняет исходный HTTP-запрос**, когда неаутентифицированный пользователь пытается получить доступ к защищённому ресурсу. 

После успешного входа этот запрос **воспроизводится**, и пользователь попадает туда, куда изначально хотел.

> Если попытались обратиться к защищенному ресурсу без входа, то нужно:
> 1. Пользователя отправить на форму входа
> 2. Сохранить исходный запрос пользователя, и после его входа перенаправить его на ресурс

**Пример**
1. Пользователь (не залогинен) запрашивает `/secret-page`.
    
2. `AuthorizationFilter` выбрасывает `AccessDeniedException`.
    
3. `ExceptionTranslationFilter` перехватывает исключение, видит, что пользователь не аутентифицирован.
    
4. **Перед** тем как отправить пользователя на `/login`, фильтр сохраняет текущий запрос (`GET /secret-page`) в `RequestCache`.
    
5. Пользователь вводит логин/пароль на странице входа.
    
6. После успешной аутентификации `RequestCacheAwareFilter` проверяет кэш: «Есть сохранённый запрос?»
    
7. Если есть — перенаправляет пользователя на `/secret-page` вместо страницы по умолчанию.


В Spring Security это делается путем сохранения `HttpServletRequest` с использованием реализации `RequestCache`.

### 2.7.1 Настройка `RequestCache` через `HttpSecurity`
Есть три основных сценария настройки.
#### Сценарий 1: Поведение по умолчанию
Если ***вы ничего не настраиваете***, **Spring Security использует `HttpSessionRequestCache`**. 

==Запрос сохраняется в HTTP-сессии==, и после входа пользователь перенаправляется туда, куда хотел.

```java
http
    .formLogin(Customizer.withDefaults());
// RequestCache работает автоматически
```

---
#### Сценарий 2: Сохранять запрос только при наличии параметра `continue` в URL
Полезно, когда у вас несколько способов попасть на страницу входа, и **вы хотите явно указать, когда нужно восстанавливать URL.**

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    HttpSessionRequestCache requestCache = new HttpSessionRequestCache();
    requestCache.setMatchingRequestParameterName("continue");
    
    http
        .requestCache(cache -> cache.requestCache(requestCache))
        .formLogin(Customizer.withDefaults());
    
    return http.build();
}
```

Вы (или фронтенд-разработчик) просто **добавляете параметр 
`?continue` к тем ссылкам**, для которых хотите, чтобы после входа пользователь вернулся обратно. 

Теперь запрос сохранится только если пользователь перешёл по URL с параметром `?continue`. 
Например:
- `/secret-page?continue` → запрос сохранится, после входа вернёт на `/secret-page`
    
- `/secret-page` → запрос НЕ сохранится, после входа попадёт на страницу по умолчанию


`HttpSessionRequestCache` — это реализация `RequestCache`, которая хранит сохранённый запрос в HTTP-сессии. 
А метод `setMatchingRequestParameterName("continue")` говорит ему: «Сохраняй запрос **только если** в URL есть параметр `continue`».

---
#### Сценарий 3: Полностью отключить сохранение запросов
Для stateless API или когда вы всегда хотите отправлять пользователя на одну и ту же страницу после входа.

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    RequestCache nullRequestCache = new NullRequestCache();
    
    http
        .requestCache(cache -> cache.requestCache(nullRequestCache))
        .formLogin(form -> form.defaultSuccessUrl("/home"));
    
    return http.build();
}
```
Теперь после входа пользователь **всегда** попадёт на `/home`, независимо от того, какой URL он запрашивал до аутентификации.

---
#### Сценарий 4. СОСБСТВЕННАЯ РЕАЛИЗАЦИЯ RequstCache
Если вам нужно хранить запросы не в сессии, а, например, в Redis или базе данных — реализуйте интерфейс `RequestCache`:

```java
public class RedisRequestCache implements RequestCache {
    
    @Override
    public void saveRequest(
	    HttpServletRequest request, 
		HttpServletResponse response
	) {
        // сохраняем в Redis
    }
    
    @Override
    public SavedRequest getRequest(
	    HttpServletRequest request, 
	    HttpServletResponse response
	) {
        // извлекаем из Redis
    }
    
    @Override
    public HttpServletRequest getMatchingRequest(
	    HttpServletRequest request, 
	    HttpServletResponse response
	) {
        // проверяем, есть ли сохранённый запрос для этого пользователя
    }
    
    @Override
    public void removeRequest(
	    HttpServletRequest request, 
	    HttpServletResponse response
	) {
        // удаляем после восстановления
    }
}
```


---

`HttpServletRequest` сохраняется в `RequestCache`. 

Когда пользователь успешно аутентифицируется, `RequestCache` **используется для воспроизведения исходного запроса.**  

`RequestCacheAwareFilter` использует `RequestCache` для получения сохраненного `HttpServletRequest` после аутентификации пользователя, 
в то время как `ExceptionTranslationFilter` использует `RequestCache` для сохранения `HttpServletRequest` после обнаружения `AuthenticationException`, перед перенаправлением пользователя на endpoint входа в систему.

По умолчанию используется `HttpSessionRequestCache`. 

***Код ниже демонстрирует***, как настроить ***реализацию*** `RequestCache` так, ***чтобы она проверяла*** `HttpSession` **на наличие сохраненного запроса только при наличии параметра** `continue`.

По сути
- `requestCache()` — это про «какой URL запросил пользователь до того, как его перебросило на логин».

_RequestCache проверяет сохраненные запросы только при наличии параметра continue_
```java
@Bean
DefaultSecurityFilterChain springSecurity(
	HttpSecurity http
) throws Exception 
{
	HttpSessionRequestCache requestCache = 
		new HttpSessionRequestCache();
	requestCache.setMatchingRequestParameterName("continue");
	
	http
		// ...
		.requestCache((cache) -> cache
			.requestCache(requestCache)
		);
	return http.build();
}
```

## 2.7 ДОПОЛНИТЕЛЬНО
### 2.7.1 Указание, где хранить SecurityContext
`SecurityContext` (настраивается через `.securityContext()`) управляет тем, **как и где хранится сам объект `SecurityContext`** (то есть данные об аутентифицированном пользователе) между запросами. 

```java
.securityContext(context -> context
    .requireExplicitSave(true)
)
```

По сути:
- `securityContext()` — это про «где хранить информацию о том, что пользователь уже залогинился».

Например: `.securityContext(ctx -> ctx.requireExplicitSave(true))` означает, что `SecurityContextHolderFilter` не будет автоматически сохранять контекст — это задача разработчика.
