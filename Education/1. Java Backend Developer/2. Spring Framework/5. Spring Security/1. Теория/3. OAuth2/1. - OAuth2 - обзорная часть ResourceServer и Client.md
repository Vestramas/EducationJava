# 1. OAuth2
Spring Security обеспечивает всестороннюю поддержку OAuth 2.0. В этом разделе обсуждается, как интегрировать OAuth 2.0 в ваше приложение на основе сервлетов.

## 1.1 Обзор
Поддержка OAuth 2.0 в Spring Security состоит из трех основных наборов функций:

- **OAuth2 Client** (Клиент OAuth2)
    
- **OAuth2 Resource Server** (Сервер ресурсов OAuth2)
    
- **OAuth2 Authorization Server** (Сервер авторизации OAuth2)
    

**OAuth2 Login** — это важная часть OAuth2 Client, но не является самостоятельной функцией и работает только вместе с ним.

В OAuth 2.0 выделяют три основные роли:
- **Client** — клиентское приложение
- **Resource Server** — сервер ресурсов (API)
- **Authorization Server** — сервер авторизации

Обычно сервер ресурсов и клиент представлены одним или несколькими серверными приложениями, а сервер авторизации может быть как внешним сервисом (например, централизованная система аутентификации), так и частью самого приложения.

Типичный пример — микросервисная архитектура, где есть:
- клиентское приложение для пользователя
- несколько внутренних Resource Server с REST API
- отдельный Authorization Server для управления пользователями и аутентификацией

Также часто одно приложение выполняет только одну роль и взаимодействует с внешними сервисами, реализующими остальные роли.

Spring Security поддерживает все эти сценарии и предоставляет инструменты для работы с каждой из ролей OAuth2.


- Я хочу защитить доступ к API с помощью OAuth2 (сервер авторизации предоставляет JWT или непрозрачный (opaque) токен доступа) 
  [[#1.2.1 Защита доступа с помощью OAuth2 Access Token]]
    
- Я хочу защитить доступ к API с помощью **пользовательского JWT** *(кастомного токена)*
  [[#1.2.2 Защита доступа с помощью пользовательского JWT]]

### 1.2.1 Защита доступа с помощью OAuth2 Access Token
==Защита доступа к API с помощью токенов== доступа OAuth2 о**чень распространена.** 

В большинстве случаев Spring Security требует лишь минимальной конфигурации для обеспечения безопасности приложения с помощью OAuth2.

Spring Security поддерживает ==два типа токенов== `Bearer`, каждый из которых использует свой компонент для проверки:
- Поддержка **JWT** использует бин `JwtDecoder` для проверки подписей и декодирования токенов.
    
- Поддержка **непрозрачных (opaque) токенов** использует бин `OpaqueTokenIntrospector` для интроспекции токенов.

---
#### Поддержка JWT
Следующий пример настраивает бин `JwtDecoder` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://my-auth-server.com
```
> При использовании Spring Boot это все, что требуется. 

*(Java-base config)* Конфигурация по умолчанию, предоставляемая Spring Boot, эквивалентна следующей:
**Настройка Resource Server с JWT**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.oauth2ResourceServer((oauth2) -> oauth2
				.jwt(Customizer.withDefaults())
			);
		return http.build();
	}

	@Bean
	public JwtDecoder jwtDecoder() {
		return JwtDecoders
			.fromIssuerLocation("https://my-auth-server.com");
	}

}
```
- `issuer-uri` = адрес сервера, который выпустил JWT
```java
Customizer.withDefaults()
```
“*Использовать стандартную (дефолтную) настройку Spring Security без дополнительной кастомизации.*”


```java
.oauth2ResourceServer((oauth2) -> oauth2  
	.jwt(Customizer.withDefaults())  
);
```
После добавления этой настройки приложение становится **OAuth2 Resource Server**, то есть *сервером, который защищает свои ресурсы (API endpoint'ы)* и принимает запросы с `Bearer JWT token`.

Теперь *Spring Security пропускает клиента только после проверки access token.*

Когда клиент отправляет запрос с токеном:
```http
Authorization: Bearer <JWT>
```
Spring Security с помощью `JwtDecoder` декодирует JWT, проверяет его срок действия, claims и подлинность цифровой подписи.

Для этого используется `issuer-uri` — **адрес Authorization Server**, ***который выпустил токен***.

По этому адресу `JwtDecoder` ***получает metadata и публичные ключи*** (JWK Set), **с помощью которых проверяется подпись JWT.**

Если подпись валидна и токен корректный — пользователь считается аутентифицированным, и доступ к ресурсу разрешается.  
Если проверка не пройдена — возвращается `401 Unauthorized`.

---
#### Поддержка непрозрачных (opaque) токенов
Следующий пример настраивает бин `OpaqueTokenIntrospector` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        opaquetoken:
          introspection-uri: https://my-auth-server.com/oauth2/introspect
          client-id: my-client-id
          client-secret: my-client-secret
```

При использовании Spring Boot это все, что требуется. Конфигурация по умолчанию, предоставляемая Spring Boot, эквивалентна следующей:

**Настройка Resource Server с непрозрачными (opaque) токенами**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.oauth2ResourceServer((oauth2) -> oauth2
				.opaqueToken(Customizer.withDefaults())
			);
		return http.build();
	}

	@Bean
	public OpaqueTokenIntrospector opaqueTokenIntrospector() {
		return new SpringOpaqueTokenIntrospector(
			"https://my-auth-server.com/oauth2/introspect", "my-client-id", "my-client-secret");
	}

}
```

### 1.2.2 Защита доступа с помощью пользовательского JWT
Защита доступа к API с помощью JWT — довольно распространенная задача, особенно когда фронтенд разрабатывается как одностраничное приложение (SPA). 
Поддержка OAuth2 Resource Server в Spring Security может использоваться для любого типа `Bearer` токена, включая пользовательский JWT.

==Все, что требуется для защиты API с помощью JWT, — это бин ==`JwtDecoder`, который используется для проверки подписей и декодирования токенов. 

Spring Security автоматически использует предоставленный бин для настройки защиты в `SecurityFilterChain`.

Следующий пример настраивает бин `JwtDecoder` с использованием конфигурационных свойств Spring Boot:
```java
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          public-key-location: classpath:my-public-key.pub
```
> Вы можете предоставить открытый ключ как ресурс classpath (в этом примере он называется `my-public-key.pub`).

При использовании Spring Boot это все, что требуется. Конфигурация по умолчанию, предоставляемая Spring Boot, эквивалентна следующей:

**Настройка Resource Server с пользовательскими JWT**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.oauth2ResourceServer((oauth2) -> oauth2
				.jwt(Customizer.withDefaults())
			);
		return http.build();
	}

	@Bean
	public JwtDecoder jwtDecoder() {
		return NimbusJwtDecoder.withPublicKey(publicKey()).build();
	}

	private RSAPublicKey publicKey() {
		// ...
	}

}
```

> **Примечание:** Spring Security не предоставляет конечную точку для создания (чеканки) токенов. 
> 
> Однако Spring Security **предоставляет интерфейс** `JwtEncoder` **вместе с одной реализацией** — `NimbusJwtEncoder`.

### 1.3.1 Вход пользователей с помощью OAuth2
До этого мы обсуждали сценарий, где ваше приложение — это **Backend API** (Resource Server).

- **Клиент:** Мобильное приложение или React SPA.
    
- **Схема:** Клиент _сам_ идет на Auth0, получает токен, и стучится этим токеном к вам в API.
    
- **Ваша задача:** Только **проверить** токен (`JwtDecoder`). Вы не участвуете в процессе получения токена. Вы — **Resource Server**.

```
[Клиент (React)] ---> (Получает токен сам) ---> Auth0

[Клиент (React)] ---> (Стучится с токеном) 
---> [Ваше API (Resource Server)]
```

Сейчас мы говорим о сценарии, ==где ваше приложение== — это **Серверное Web-приложение** (Spring MVC с шаблонами Thymeleaf, например).
- **Клиент:** Браузер пользователя (==он не умеет сам ходить в Auth0 по OAuth2 безопасно==, потому что не умеет хранить `client_secret`).
    
- **Схема:** ==Браузер просит== **ваше приложение** залогинить его через Google. 
  ❗❗❗ **Ваше приложение** само идет в Google, договаривается, получает токен, и только потом пускает пользователя на свои страницы.
    
- **Ваша задача:** ==Участвовать в процессе получения токена.== Вы — **OAuth2 Client**.
```
[Браузер] ---> (Хочу логин) ---> [Ваше Приложение (Client)] ---> (Идет за токеном) ---> Google
```

#### Суть проблемы: Как приложению представиться Google?
Чтобы **ваше приложение** могло пойти в Google и попросить данные пользователя, ==Google должен знать,== **кто именно просит**. 
> Это же не какой-то анонимный бот.

Еще в древние времена OAuth (RFC 6749) был придуман именно для этого: **Делегирование доступа**.  
==Вы, как владелец приложения, должны зарегистрировать его в консоли разработчика Google== (или GitHub, или Facebook). 

Там вам выдадут:
1. **Client ID:** Публичный идентификатор вашего приложения (например, `123456-abc.apps.googleusercontent.com`). Можно светить в браузере.
    
2. **Client Secret:** Пароль вашего приложения. **Категорически нельзя светить** в браузере или мобильном приложении. Его знает только ваш Бэкенд.

Вот эти два значения (`client-id` и `client-secret`) + настройки (куда слать запросы, какие права просить) и составляют **`ClientRegistration`**.

> [!NOTE] ClientRegistration - **ПАСПОРТ**
> `ClientRegistration` — это «**Паспорт вашего приложения»** для стороннего сервиса авторизации.

#### Зачем нужен `ClientRegistrationRepository`?
Ваше приложение может поддерживать вход через **несколько** провайдеров. 
Например:
- Вход через **Google** (для обычных юзеров).
    
- Вход через **GitHub** (для разработчиков).
    
- Вход через корпоративный **Keycloak** (для сотрудников).


`ClientRegistrationRepository` — это ==просто хранилище (репозиторий), где лежат все эти «паспорта».==
- `InMemoryClientRegistrationRepository` — паспорта лежат прямо в коде (как в вашем примере Java Config) или в `application.yml`.
    
- `JdbcClientRegistrationRepository` — паспорта лежат в базе данных (можно добавлять новых провайдеров на лету).

Очень *распространено требование* **входа пользователей через OAuth2.** 

**OpenID Connect 1.0** ***предоставляет специальный токен***, называемый `id_token`, который *предназначен для того, чтобы дать OAuth2 Client возможность выполнить проверку личности пользователя и войти в систему.* 

В определенных случаях OAuth2 можно использовать напрямую для входа пользователей (как в случае с популярными провайдерами социального входа, которые не реализуют OpenID Connect, такими как GitHub и Facebook).

Следующий пример настраивает приложение как OAuth2 Client, способный входить пользователями с помощью OAuth2 или OpenID Connect:

**Настройка OAuth2 Login**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			// ...
			.oauth2Login(Customizer.withDefaults());
		return http.build();
	}

}
```

В дополнение к приведенной выше конфигурации ==приложению требуется==, чтобы хотя бы одна `ClientRegistration` **была настроена с использованием бина** `ClientRegistrationRepository`. 

Следующий пример настраивает бин `InMemoryClientRegistrationRepository` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-oidc-client:
            provider: my-oidc-provider
            client-id: my-client-id
            client-secret: my-client-secret
            authorization-grant-type: authorization_code
            scope: openid,profile
        provider:
          my-oidc-provider:
            issuer-uri: https://my-oidc-provider.com
```

(Spring framework config)
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> authorize
                .anyRequest().authenticated()
            )
            .oauth2Login(Customizer.withDefaults());

        return http.build();
    }

    @Bean
    public ClientRegistrationRepository clientRegistrationRepository() {
        ClientRegistration clientRegistration =
            ClientRegistration.withRegistrationId("my-oidc-client")
                .clientId("my-client-id")
                .clientSecret("my-client-secret")
                .clientAuthenticationMethod(
	                ClientAuthenticationMethod.CLIENT_SECRET_BASIC)
                .authorizationGrantType(
	                AuthorizationGrantType.AUTHORIZATION_CODE)
                .redirectUri(
	                "{baseUrl}/login/oauth2/code/{registrationId}")
                .scope("openid", "profile")
                .issuerUri("https://my-oidc-provider.com")
                .build();

        return 
	        new InMemoryClientRegistrationRepository(clientRegistration);
    }
}
```

С приведенной выше конфигурацией ***приложение теперь поддерживает две дополнительные конечные точки***:
- Конечная точка входа (например, `/oauth2/authorization/my-oidc-client`) используется **для инициирования входа и выполнения перенаправления на сторонний сервер авторизации**.
    
- Конечная точка перенаправления (например, `/login/oauth2/code/my-oidc-client`) используется сервером авторизации **для перенаправления обратно в клиентское приложение** и будет содержать параметр `code`, используемый для получения `id_token` и/или `access_token` через запрос токена доступа.

> Наличие области действия (`scope`) `openid` в приведенной выше конфигурации указывает, что следует использовать OpenID Connect 1.0. 

Это предписывает Spring Security использовать компоненты, специфичные для OIDC (такие как `OidcUserService`), во время обработки запроса. Без этой области действия Spring Security будет использовать компоненты, специфичные для OAuth2 (такие как `DefaultOAuth2UserService`).

### 1.3.2 Доступ к защищенным ресурсам
Выполнение запросов к стороннему API, защищенному с помощью OAuth2, является основным вариантом использования OAuth2 Client. 

==Это достигается путем авторизации клиента (представленного классом== `OAuth2AuthorizedClient` в Spring Security) и доступа к защищенным ресурсам путем помещения `Bearer` токена в заголовок `Authorization` исходящего запроса.

Следующий пример настраивает приложение как OAuth2 Client, способный запрашивать защищенные ресурсы у стороннего API:

**Настройка OAuth2 Client**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			// ...
			.oauth2Client(Customizer.withDefaults());
		return http.build();
	}

}
```

> **Примечание:** Приведенный выше пример не предоставляет способа входа пользователей в систему. Вы можете использовать любой другой механизм входа (например, `formLogin()`). См. следующий раздел для примера комбинирования `oauth2Client()` с `oauth2Login()`.

В дополнение к приведенной **выше конфигурации приложению требуется**, чтобы хотя бы одна `ClientRegistration` **была настроена с использованием бина** `ClientRegistrationRepository`. 

Следующий пример настраивает бин `InMemoryClientRegistrationRepository` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-oauth2-client:
            provider: my-auth-server
            client-id: my-client-id
            client-secret: my-client-secret
            authorization-grant-type: authorization_code
            scope: message.read,message.write
        provider:
          my-auth-server:
            issuer-uri: https://my-auth-server.com
```


В дополнение к настройке Spring Security для поддержки функций OAuth2 Client вам также **нужно будет решить, как вы будете получать доступ к защищенным ресурсам**, и соответствующим образом настроить ваше приложение. 

Spring Security ==предоставляет реализации== `OAuth2AuthorizedClientManager` **для получения токенов доступа**, которые можно использовать для доступа к защищенным ресурсам.

Spring Security **автоматически регистрирует бин** `OAuth2AuthorizedClientManager` ***по умолчанию, если он не существует.***

==Самый простой способ использовать== `OAuth2AuthorizedClientManager` — **через** `ClientHttpRequestInterceptor`, **который перехватывает запросы через** `RestClient`, доступный, когда `spring-web` находится в classpath.

> [!NOTE] Классы
> `OAuth2AuthorizedClient` — объект, ***содержащий access/refresh token и информацию о клиенте и пользователе***.
> 
> `OAuth2AuthorizedClientManager` — компонент, ***который управляет жизненным циклом OAuth2AuthorizedClient:*** получает, обновляет и кеширует токены. 
> 
> `ClientHttpRequestInterceptor` — HTTP interceptor, ***который перехватывает исходящие запросы и добавляет Bearer токен в заголовок Authorization.***

***Следующий пример использует*** `OAuth2AuthorizedClientManager` ***по*** ***умолчанию для настройки*** `RestClient`, способного получать доступ к защищенным ресурсам, помещая `Bearer` токены в заголовок `Authorization` каждого запроса:

**Настройка `RestClient` с `ClientHttpRequestInterceptor`**
```java
@Configuration
public class RestClientConfig {

	@Bean
	public RestClient restClient(
		OAuth2AuthorizedClientManager authorizedClientManager
	) {
		OAuth2ClientHttpRequestInterceptor requestInterceptor =
			new OAuth2ClientHttpRequestInterceptor(
				authorizedClientManager
			);
			
		return RestClient.builder()
				.requestInterceptor(requestInterceptor)
				.build();
	}
}
```

Этот настроенный `RestClient` можно использовать, как в следующем примере:

**Использование `RestClient` для доступа к защищенным ресурсам**
```java
import static org.springframework.security.oauth2.client.web.client.RequestAttributeClientRegistrationIdResolver.clientRegistrationId;

@RestController
public class MessagesController {

	private final RestClient restClient;

	public MessagesController(RestClient restClient) {
		this.restClient = restClient;
	}

	@GetMapping("/messages")
	public ResponseEntity<List<Message>> messages() {
		Message[] messages = this.restClient.get()
				.uri("http://localhost:8090/messages")
				.attributes(clientRegistrationId("my-oauth2-client"))
				.retrieve()
				.body(Message[].class);
		return ResponseEntity.ok(Arrays.asList(messages));
	}

	public record Message(String message) {
	}
}
```

`RestClient` в связке с `OAuth2AuthorizedClientManager` **автоматически добавляет Bearer токен в исходящие запросы** *к сторонним сервисам.*

Токен берётся из `OAuth2AuthorizedClient`, который управляется Spring Security.

Если токена нет или он истёк, Spring автоматически получает или обновляет его через Authorization Server.

В зависимости от конфигурации токен может быть:
- пользовательским (OAuth2 Login)
- или клиентским (client_credentials)

> Spring Security управляет жизненным циклом токена,  
> а RestClient только использует эту инфраструктуру через interceptor

### 1.3.3 Доступ к защищенным ресурсам с помощью `WebClient`
Выполнение запросов к стороннему API, защищенному с помощью OAuth2, является основным вариантом использования OAuth2 Client. 

Это достигается путем авторизации клиента (представленного классом `OAuth2AuthorizedClient` в Spring Security) и доступа к защищенным ресурсам путем помещения `Bearer` токена в заголовок `Authorization` исходящего запроса.

Следующий пример настраивает приложение как OAuth2 Client, способный запрашивать защищенные ресурсы у стороннего API:

**Настройка OAuth2 Client**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			// ...
			.oauth2Client(Customizer.withDefaults());
		return http.build();
	}

}
```

> **Примечание:** Приведенный выше пример не предоставляет способа входа пользователей в систему. Вы можете использовать любой другой механизм входа (например, `formLogin()`). 
> См. предыдущий раздел для примера комбинирования `oauth2Client()` с `oauth2Login()`.

В дополнение к приведенной выше конфигурации **приложению требуется, чтобы хотя бы одна** `ClientRegistration` была настроена с использованием бина `ClientRegistrationRepository`. 

Следующий пример настраивает бин `InMemoryClientRegistrationRepository` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-oauth2-client:
            provider: my-auth-server
            client-id: my-client-id
            client-secret: my-client-secret
            authorization-grant-type: authorization_code
            scope: message.read,message.write
        provider:
          my-auth-server:
            issuer-uri: https://my-auth-server.com
```

В дополнение к настройке Spring Security для поддержки функций OAuth2 Client вам также **нужно будет решить, как вы будете получать доступ к защищенным ресурсам,** и соответствующим образом настроить ваше приложение. 

Spring Security предоставляет реализации `OAuth2AuthorizedClientManager` для получения токенов доступа, которые можно использовать для доступа к защищенным ресурсам.

Spring Security автоматически регистрирует бин `OAuth2AuthorizedClientManager` по умолчанию, если он не существует.

**Вместо настройки** `RestClient` ==другим способом использования== `OAuth2AuthorizedClientManager` **является использование** `ExchangeFilterFunction`, который перехватывает запросы через `WebClient`. 

Чтобы использовать `WebClient`, вам нужно будет добавить зависимость `spring-webflux` вместе с реализацией реактивного клиента:

**Добавление зависимости Spring WebFlux**
```xml
<dependency>
	<groupId>org.springframework</groupId>
	<artifactId>spring-webflux</artifactId>
</dependency>
<dependency>
	<groupId>io.projectreactor.netty</groupId>
	<artifactId>reactor-netty</artifactId>
</dependency>
```

Следующий пример использует `OAuth2AuthorizedClientManager` по умолчанию для настройки `WebClient`, способного получать доступ к защищенным ресурсам, помещая `Bearer` токены в заголовок `Authorization` каждого запроса:

**Настройка `WebClient` с `ExchangeFilterFunction`**
```java
@Configuration
public class WebClientConfig {

	@Bean
	public WebClient webClient(
		OAuth2AuthorizedClientManager authorizedClientManager
	) {
		ServletOAuth2AuthorizedClientExchangeFilterFunction filter =
			new ServletOAuth2AuthorizedClientExchangeFilterFunction(
				authorizedClientManager
			);
		return WebClient.builder()
				.apply(filter.oauth2Configuration())
				.build();
	}

}
```

Этот настроенный `WebClient` можно использовать, как в следующем примере:

**Использование `WebClient` для доступа к защищенным ресурсам**
```java
import static org.springframework.security.oauth2.client.web.reactive.function.client.ServletOAuth2AuthorizedClientExchangeFilterFunction.clientRegistrationId;

@RestController
public class MessagesController {

	private final WebClient webClient;

	public MessagesController(WebClient webClient) {
		this.webClient = webClient;
	}

	@GetMapping("/messages")
	public ResponseEntity<List<Message>> messages() {
		return this.webClient.get()
				.uri("http://localhost:8090/messages")
				.attributes(clientRegistrationId("my-oauth2-client"))
				.retrieve()
				.toEntityList(Message.class)
				.block();
	}

	public record Message(String message) {
	}

}
```

### 1.3.4 Доступ к защищенным ресурсам для текущего пользователя
*Когда пользователь входит в систему через OAuth2* или OpenID Connect, *сервер авторизации может предоставить токен доступа*, ***который можно использовать напрямую для доступа к защищенным ресурсам.*** 

Это удобно, поскольку требует настройки только одной `ClientRegistration` для обоих вариантов использования одновременно.

> [!NOTE] 
> Этот раздел объединяет **Вход пользователей с помощью OAuth2** и **Доступ к защищенным ресурсам** в единую конфигурацию. 
> Contents

Существуют и другие продвинутые сценарии, например, настройка одной `ClientRegistration` для входа и другой для доступа к защищенным ресурсам. Все такие сценарии будут использовать одну и ту же базовую конфигурацию.

Следующий пример настраивает приложение как OAuth2 Client, способный входить пользователя в систему и запрашивать защищенные ресурсы у стороннего API:

**Настройка OAuth2 Login и OAuth2 Client**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		http
			// ...
			.oauth2Login(Customizer.withDefaults())
			.oauth2Client(Customizer.withDefaults());
		return http.build();
	}

}
```

В дополнение к приведенной выше конфигурации приложению **требуется, чтобы хотя бы одна** `ClientRegistration` была настроена с использованием бина `ClientRegistrationRepository`. 

Следующий пример настраивает бин `InMemoryClientRegistrationRepository` с использованием конфигурационных свойств Spring Boot:
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-combined-client:
            provider: my-auth-server
            client-id: my-client-id
            client-secret: my-client-secret
            authorization-grant-type: authorization_code
            scope: openid,profile,message.read,message.write
        provider:
          my-auth-server:
            issuer-uri: https://my-auth-server.com
```

==Основное различие ==между предыдущими примерами (***Вход пользователей с помощью OAuth2**, **Доступ к защищенным ресурсам***) и этим заключается в том, **что настроено через свойство** `scope`, которое объединяет стандартные области `openid` и `profile` с пользовательскими областями `message.read` и `message.write`.

В дополнение к настройке Spring Security для поддержки функций OAuth2 Client **вам также нужно будет решить, как вы будете получать доступ к защищенным ресурсам**, и соответствующим образом настроить ваше приложение. Spring Security предоставляет реализации `OAuth2AuthorizedClientManager` для получения токенов доступа, которые можно использовать для доступа к защищенным ресурсам.

Spring Security автоматически регистрирует бин `OAuth2AuthorizedClientManager` по умолчанию, если он не существует.

Самый простой способ использовать `OAuth2AuthorizedClientManager` — через `ClientHttpRequestInterceptor`, который перехватывает запросы через `RestClient`, доступный, когда `spring-web` находится в classpath.

Следующий пример использует `OAuth2AuthorizedClientManager` по умолчанию для настройки `RestClient`, способного получать доступ к защищенным ресурсам, помещая `Bearer` токены в заголовок `Authorization` каждого запроса:

**Настройка `RestClient` с `ClientHttpRequestInterceptor`**
```java
@Configuration
public class RestClientConfig {

	@Bean
	public RestClient restClient(
		OAuth2AuthorizedClientManager authorizedClientManager
	) {
		OAuth2ClientHttpRequestInterceptor requestInterceptor =
			new OAuth2ClientHttpRequestInterceptor(
				authorizedClientManager
			);
			
		requestInterceptor.setClientRegistrationIdResolver(
			clientRegistrationIdResolver()
		);

		return RestClient.builder()
				.requestInterceptor(requestInterceptor)
				.build();
	}

	private static ClientRegistrationIdResolver clientRegistrationIdResolver() {
		return (request) -> {
			Authentication authentication = 
				SecurityContextHolder
					.getContext()
					.getAuthentication();
				
			return 
		(authentication instanceof OAuth2AuthenticationToken principal)
			? principal.getAuthorizedClientRegistrationId()
			: null;
		};
	}

}
```

Этот настроенный `RestClient` можно использовать, как в следующем примере:

**Использование `RestClient` для доступа к защищенным ресурсам (текущий пользователь)**
```java
@RestController
public class MessagesController {

	private final RestClient restClient;

	public MessagesController(RestClient restClient) {
		this.restClient = restClient;
	}

	@GetMapping("/messages")
	public ResponseEntity<List<Message>> messages() {
		Message[] messages = this.restClient.get()
				.uri("http://localhost:8090/messages")
				.retrieve()
				.body(Message[].class);
		return ResponseEntity.ok(Arrays.asList(messages));
	}

	public record Message(String message) {
	}

}
```

В отличие от предыдущего примера, обратите внимание, **что нам не нужно сообщать Spring Security, какой** `clientRegistrationId` **мы хотели бы использовать.** Это потому, что он может быть получен из текущего вошедшего в систему пользователя.

### 1.3.5 Использование типа разрешения Client Credentials
#### РЕЗЮМЕ
> [!NOTE] 
> client_credentials = "приложение получает токен само от себя"
> 
> Spring:
> 1. Берёт `client_id + client_secret`
> 2. Идёт в Authorization Server
> 3. Получает `access_token`
> 4. Использует его для вызовов API
> 👉 Всё это **без участия пользователя**

client_credentials — это OAuth2 flow без пользователя, где приложение получает access token от своего имени.

По умолчанию Spring Security привязывает токены к principal (пользователю), но для client_credentials нужно использовать единый principal для всего приложения.

Для этого используется `RequestAttributePrincipalResolver` и передача principal через attributes().

В Spring Security и Spring Web под **attributes запроса** ==понимается НЕ HTTP-параметры и НЕ headers==, а **внутренний контейнер данных, который привязывается к конкретному request’у внутри Spring**.
> ты ==НЕ отправляешь это на сервер==.
> 👉 Это **локальные метаданные внутри Spring**, которые используются interceptor-ом.

В результате создаётся один access token на всё приложение, который используется для всех запросов к сторонним сервисам.

Это применяется в server-to-server сценариях (микросервисы, интеграции, фоновые задачи).

---

В этом разделе рассматриваются дополнительные соображения по типу разрешения (`grant type`) `client_credentials`. Общие настройки и использование со всеми типами разрешения см. в разделе **Доступ к защищенным ресурсам**. ([[#1.3.2 Доступ к защищенным ресурсам]])

Тип разрешения `client_credentials` позволяет клиенту получить `access_token` от своего собственного имени. 
`client_credentials` — это простой поток, который не вовлекает владельца ресурса (т. е. пользователя).

> **Важно отметить:** Типичное использование типа разрешения `client_credentials` подразумевает, что любой запрос (или пользователь) потенциально может получить токен доступа и делать запросы к защищенным ресурсам на сервер ресурсов. Будьте осторожны при проектировании приложений, чтобы пользователи не могли делать несанкционированные запросы, поскольку каждый запрос сможет получить токен доступа.

При получении токенов доступа в веб-приложении, где пользователи могут входить в систему, поведение Spring Security по умолчанию заключается в получении токена доступа для каждого пользователя.

По умолчанию токены доступа привязаны к имени принципала (`principal name`) текущего пользователя, что означает, что каждый пользователь получит уникальный токен доступа.

**Клиенты, использующие тип разрешения** `client_credentials`, ==обычно требуют, чтобы токены доступа были привязаны к приложению==, **а не к отдельным пользователям**, ***чтобы существовал только один токен доступа на приложение.*** 

Чтобы привязать токены доступа к приложению, *вам нужно будет установить стратегию для определения пользовательского имени принципала.* 

Следующий пример делает это, настраивая `RestClient` с `RequestAttributePrincipalResolver`:

**Настройка `RestClient` для `client_credentials`**
```java
@Configuration
public class RestClientConfig {

	@Bean
	public RestClient restClient(
		OAuth2AuthorizedClientManager authorizedClientManager
	) {
		OAuth2ClientHttpRequestInterceptor requestInterceptor =
			new OAuth2ClientHttpRequestInterceptor(
				authorizedClientManager
			);
			
		requestInterceptor.setPrincipalResolver(
			new RequestAttributePrincipalResolver()
		);
		
		return RestClient.builder()
				.requestInterceptor(requestInterceptor)
				.build();
	}

}
```

С приведенной выше конфигурацией имя принципала можно указать для каждого запроса. Следующий пример демонстрирует, как привязать токены доступа к приложению, указав имя принципала:

**Привязка токенов доступа к приложению**
```java
import static org.springframework.security.oauth2.client.web.client.RequestAttributeClientRegistrationIdResolver.clientRegistrationId;
import static org.springframework.security.oauth2.client.web.client.RequestAttributePrincipalResolver.principal;

@RestController
public class MessagesController {

	private final RestClient restClient;

	public MessagesController(RestClient restClient) {
		this.restClient = restClient;
	}

	@GetMapping("/messages")
	public ResponseEntity<List<Message>> messages() {
		Message[] messages = this.restClient.get()
				.uri("http://localhost:8090/messages")
				.attributes(clientRegistrationId("my-oauth2-client"))
				.attributes(principal("my-application"))
				.retrieve()
				.body(Message[].class);
		return ResponseEntity.ok(Arrays.asList(messages));
	}

	public record Message(String message) {
	}

}
```

При указании имени принципала через атрибуты, как в примере выше, будет существовать только один токен доступа, и он будет использоваться для всех запросов.

### 1.3.6 Включение расширенного типа разрешения (Extension Grant Type)
#### РЕЗЮМЕ:
Стандартные типы разрешения OAuth 2.0 (`authorization_code`, `client_credentials`, `password`, `refresh_token`) покрывают 80% задач. Но в сложных корпоративных или микросервисных системах возникают ситуации, когда стандартных механизмов недостаточно.

**Суть проблемы:** У вас есть **один** токен, но ==вы хотите обменять его ==на **другой** токен, не спрашивая повторно логин и пароль у пользователя.

Представьте себе микросервисную архитектуру:
![[Pasted image 20260421114009.png]]
1. Пользователь залогинился в **Frontend App** и получил JWT-токен, выпущенный **Auth Server**.
    
2. **Frontend App** вызывает **API Gateway**.
    
3. **API Gateway** должен вызвать **Backend Service A**, но для доступа к нему нужен **другой** токен (например, с другими скоупами или подписанный другим ключом).
    
4. **API Gateway** не должен передавать оригинальный токен пользователя дальше (это небезопасно и нарушает принцип минимальных привилегий).
    

Здесь на помощь приходит **Token Exchange**.  
**API Gateway** идет на **Auth Server** и говорит: _«Вот тебе мой текущий токен (`subject_token`), обменяй его мне на токен для Service A (`requested_token_type`)»_.  
**Auth Server** валидирует текущий токен, проверяет политики (можно ли этому клиенту обменивать токены), и выдает **новый токен**, который Gateway использует для вызова Service A.

---

Распространенный вариант использования включает включение и/или настройку расширенного типа разрешения (`extension grant type`). Например, Spring Security предоставляет поддержку типов разрешения `jwt-bearer` и `token-exchange`, но не включает их по умолчанию, поскольку они не являются частью основной спецификации OAuth 2.0.

В Spring Security 6.2 и новее мы можем просто опубликовать бин для одного или нескольких `OAuth2AuthorizedClientProvider`, и они будут автоматически подхвачены. 
Следующий пример просто включает тип разрешения `jwt-bearer`:

**Включение типа разрешения `jwt-bearer`**
```java
@Configuration
public class SecurityConfig {

	@Bean
	public OAuth2AuthorizedClientProvider jwtBearer() {
		return new JwtBearerOAuth2AuthorizedClientProvider();
	}

}
```

`OAuth2AuthorizedClientManager` по умолчанию будет автоматически опубликован Spring Security, если он еще не предоставлен.

Любой пользовательский бин `OAuth2AuthorizedClientProvider` также будет подхвачен и применен к предоставленному `OAuth2AuthorizedClientManager` после стандартных типов разрешения.

Чтобы добиться приведенной выше конфигурации **ДО Spring Security 6.2,** нам приходилось публиковать этот бин самостоятельно и гарантировать, что мы также повторно включили стандартные типы разрешения. 

Чтобы понять, что настраивается «за кулисами», вот как могла бы выглядеть конфигурация:

**Включение типа разрешения `jwt-bearer` (до версии 6.2)**
```java
@Configuration
public class SecurityConfig {

	@Bean
	public OAuth2AuthorizedClientManager authorizedClientManager(
			ClientRegistrationRepository clientRegistrationRepository,
			OAuth2AuthorizedClientRepository authorizedClientRepository) {

		OAuth2AuthorizedClientProvider authorizedClientProvider =
			OAuth2AuthorizedClientProviderBuilder.builder()
				.authorizationCode()
				.refreshToken()
				.clientCredentials()
				.provider(new JwtBearerOAuth2AuthorizedClientProvider())
				.build();

		DefaultOAuth2AuthorizedClientManager authorizedClientManager =
			new DefaultOAuth2AuthorizedClientManager(
				clientRegistrationRepository, authorizedClientRepository);
		authorizedClientManager.setAuthorizedClientProvider(authorizedClientProvider);

		return authorizedClientManager;
	}

}
```

### 1.3.7 Настройка существующего типа разрешения
#### РЕЗЮМЕ
Представьте, что часы **вашего сервера** спешат на 2 минуты относительно точного времени, а часы **сервера авторизации** отстают на 2 минуты. Суммарная разница может составлять 4-5 минут.

**Сценарий сбоя:**
1. **Auth Server** (время 11:55) выпускает токен с истечением в **12:00**.
    
2. **Ваше приложение** (время 12:01) получает этот токен.
    
3. Ваше приложение пытается использовать токен **немедленно** (в 12:01 по своим часам).
    
4. Логика валидации: 
```JAVA
// Время на вашем сервере: 11:59:58
// Время истечения токена: 12:00:00
// Плюс стандартный запас (clockSkew) = 60 секунд.

if (currentTime > tokenExpirationTime.plus(clockSkew)) {
    throw new TokenExpiredException(); // Токен просрочен
}
```
    
5. **Результат:** `TokenExpiredException`. Токен, который должен был жить еще 5 минут, признан мертвым при рождении.

Пользователь видит ошибку 401, хотя только что получил токен. Это классическая проблема «плавающих часов» (clock drift).


Чтобы компенсировать возможное расхождение часов между серверами в разных дата-центрах (или на разных облачных провайдерах), мы **увеличиваем допустимую погрешность** при проверке срока действия токена.

Вместо стандартных 60 секунд (1 минута) мы устанавливаем, например, **5 минут**.

---
Возможность включения расширенных типов разрешения путем публикации бина также дает возможность настройки существующего типа разрешения без необходимости переопределять стандартные. 

Например, если мы хотим настроить расхождение часов (`clock skew`) для `OAuth2AuthorizedClientProvider` для типа разрешения `client_credentials`, мы можем просто опубликовать бин следующим образом:

**Настройка типа разрешения Client Credentials**
```java
@Configuration
public class SecurityConfig {

	@Bean
	public OAuth2AuthorizedClientProvider clientCredentials() {
		ClientCredentialsOAuth2AuthorizedClientProvider authorizedClientProvider =
				new ClientCredentialsOAuth2AuthorizedClientProvider();
		
		// ГОВОРИМ: "Разрешаю расхождение часов на 5 минут"
		authorizedClientProvider.setClockSkew(Duration.ofMinutes(5));

		return authorizedClientProvider;
	}

}
```

### 1.3.8 Настройка параметров запроса токена

Мы переходим от настройки **«Когда и как получать токен»** (предыдущие пункты про `Provider` и `Manager`) к настройке **«Что именно слать в HTTP-запросе за токеном»**.

---
##### **Суть проблемы:** API провайдера требует нестандартный параметр
Стандартный OAuth 2.0 запрос на получение токена по `authorization_code` выглядит так (спецификация RFC 6749):
```HTTP
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=ABC123
&redirect_uri=https://myapp.com/callback
&client_id=myclient
&client_secret=secret
```
**Но что, если ваш провайдер (Auth0, AWS Cognito, кастомный сервер) говорит:**  
	_«Эй, я не приму этот запрос! Ты забыл параметр `audience` или `resource`. Добавь его в тело запроса, иначе получишь ошибку 400»_.

Проблема в том, что стандартный `RestClientAuthorizationCodeTokenResponseClient` (==класс Spring Security, который шлет этот POST-запрос==) **не знает** про ваш кастомный параметр `audience`.

---
##### Пример проблемы: Ошибка 400 при логине через Auth0
Вы настроили OAuth2 Login через Auth0 для своего API.

1. Пользователь редиректится на Auth0, вводит логин/пароль.
    
2. Auth0 редиректит обратно на `/login/oauth2/code/auth0` с параметром `code`.
    
3. Spring Security (внутри `OAuth2LoginAuthenticationProvider`) берет этот `code` и пытается обменять его на `access_token`.
    
4. Spring Security вызывает `RestClientAuthorizationCodeTokenResponseClient`.
    
5. Этот клиент формирует **стандартное** тело запроса (как выше).
    
6. Auth0 отвечает: **400 Bad Request** — _«Missing required parameter: audience»_.

Пользователь видит белую страницу с ошибкой 500 или 400, а в логах — `invalid_token_response`. Вы в тупике: «Я же все client-id и secret указал правильно!».

---
##### Решение проблемы: Добавление конвертера параметров (`ParametersConverter`)
Нам нужно **подсунуть** *дополнительный параметр в тело запроса* перед тем, как он уйдет на сервер авторизации. 

> [!NOTE] Для этого мы создаем бин `OAuth2AccessTokenResponseClient`.
> 

**Почему именно этот класс?**  
В иерархии OAuth2 в Spring Security:
1. **`OAuth2AccessTokenResponseClient`** — это **интерфейс того самого парня, который физически ходит по HTTP за токеном.** У него есть метод `getTokenResponse(...)`.
    
2. Для каждого `grant_type` есть своя реализация:
    - `authorization_code` -> `RestClientAuthorizationCodeTokenResponseClient`
        
    - `client_credentials` -> `RestClientClientCredentialsTokenResponseClient`
        
    - `refresh_token` -> `RestClientRefreshTokenTokenResponseClient`

```java
@Bean
public OAuth2AccessTokenResponseClient<OAuth2AuthorizationCodeGrantRequest> authorizationCodeAccessTokenResponseClient() 
{
    // 1. Берем стандартный HTTP-клиент для обмена кода на токен
    RestClientAuthorizationCodeTokenResponseClient accessTokenResponseClient =
            new RestClientAuthorizationCodeTokenResponseClient();
    
    // 2. Добавляем к нему "конвертер параметров"
    accessTokenResponseClient.addParametersConverter(parametersConverter());

    // 3. Возвращаем модифицированный клиент как бин.
    return accessTokenResponseClient;
}

// Это наш кастомный конвертер
private static Converter<OAuth2AuthorizationCodeGrantRequest, MultiValueMap<String, String>> parametersConverter() 
{
    return (grantRequest) -> {
        MultiValueMap<String, String> parameters = new LinkedMultiValueMap<>();
        // ДОБАВЛЯЕМ ТО, ЧТО ТРЕБУЕТ AUTH0
        parameters.set("audience", "https://my-api-identifier");
        return parameters;
    };
}
```

**Что произойдет в рантайме:**
1. Spring Security видит, что мы опубликовали свой бин `OAuth2AccessTokenResponseClient`.
    
2. Когда нужно обменять код на токен, он берет **наш** бин вместо стандартного.
    
3. Наш бин берет стандартные параметры (`code`, `grant_type`, `redirect_uri`) и **добавляет** к ним `audience`.
    
4. Запрос уходит на Auth0 уже с параметром `audience`, и сервер отвечает успехом (`200 OK`).

Обратите внимание, что в **этом случае нам не нужно настраивать бин** `SecurityFilterChain`, и можно придерживаться значений по умолчанию. Если используется Spring Boot без дополнительных настроек, мы можем вообще опустить бин `SecurityFilterChain`.

До Spring Security 6.2 нам приходилось гарантировать, что эта настройка применяется как для компонентов OAuth2 Login (если мы используем эту функцию), так и для компонентов OAuth2 Client с использованием Spring Security DSL. 

Чтобы понять, что настраивается «за кулисами», вот как могла бы выглядеть конфигурация:

**Настройка параметров запроса токена для типа разрешения Authorization Code (до версии 6.2)**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		RestClientAuthorizationCodeTokenResponseClient 
		accessTokenResponseClient =
			new RestClientAuthorizationCodeTokenResponseClient();
		
		accessTokenResponseClient
			.addParametersConverter(parametersConverter());

		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.oauth2Login((oauth2Login) -> oauth2Login
				.tokenEndpoint((tokenEndpoint) -> tokenEndpoint
					.accessTokenResponseClient(accessTokenResponseClient)
				)
			)
			.oauth2Client((oauth2Client) -> oauth2Client
				.authorizationCodeGrant((authorizationCode) -> authorizationCode
					.accessTokenResponseClient(accessTokenResponseClient)
				)
			);

		return http.build();
	}

	private static Converter<OAuth2AuthorizationCodeGrantRequest, MultiValueMap<String, String>> parametersConverter() {
		// ...
	}

}
```

Для других типов разрешения мы можем опубликовать дополнительные бины `OAuth2AccessTokenResponseClient`, чтобы переопределить значения по умолчанию. 

Например, чтобы настроить запросы токенов для типа разрешения `client_credentials`, мы можем опубликовать следующий бин:

**Настройка параметров запроса токена для типа разрешения Client Credentials**
```java
@Configuration
public class SecurityConfig {

	@Bean
	public OAuth2AccessTokenResponseClient<OAuth2ClientCredentialsGrantRequest> clientCredentialsAccessTokenResponseClient() 
	{
		RestClientClientCredentialsTokenResponseClient 
		accessTokenResponseClient =
			new RestClientClientCredentialsTokenResponseClient();
		
		accessTokenResponseClient
			.addParametersConverter(
				parametersConverter()
			);

		return accessTokenResponseClient;
	}

	private static Converter<OAuth2ClientCredentialsGrantRequest, MultiValueMap<String, String>> parametersConverter() {
		// ...
	}

}
```

Spring Security автоматически разрешает следующие обобщенные типы бинов `OAuth2AccessTokenResponseClient`:

- `OAuth2AuthorizationCodeGrantRequest` (см. `RestClientAuthorizationCodeTokenResponseClient`)
    
- `OAuth2RefreshTokenGrantRequest` (см. `RestClientRefreshTokenTokenResponseClient`)
    
- `OAuth2ClientCredentialsGrantRequest` (см. `RestClientClientCredentialsTokenResponseClient`)
    
- `JwtBearerGrantRequest` (см. `RestClientJwtBearerTokenResponseClient`)
    
- `TokenExchangeGrantRequest` (см. `RestClientTokenExchangeTokenResponseClient`)

Публикация бина типа `OAuth2AccessTokenResponseClient<JwtBearerGrantRequest>` автоматически включит тип разрешения `jwt-bearer` без необходимости настраивать его отдельно.

Публикация бина типа `OAuth2AccessTokenResponseClient<TokenExchangeGrantRequest>` автоматически включит тип разрешения `token-exchange` без необходимости настраивать его отдельно.

### 1.3.9 Настройка `RestClient`, используемого компонентами OAuth2 Client
Еще один распространенный вариант использования — ***необходимость настройки*** `RestClient`, **используемого ПРИ ПОЛУЧЕНИИ токена доступа.** 

Это может потребоваться для настройки обработки ответа (через пользовательский `HttpMessageConverter`) или для применения настроек прокси для корпоративной сети (через настроенный `ClientHttpRequestFactory`).

В Spring Security 6.2 и новее мы можем просто опубликовать бины типа `OAuth2AccessTokenResponseClient`, и Spring Security настроит и опубликует бин `OAuth2AuthorizedClientManager` за нас.

Следующий пример настраивает `RestClient` для всех поддерживаемых типов разрешения:

`OAuth2AuthorizedClientManager` по умолчанию будет автоматически опубликован Spring Security, если он еще не предоставлен.

Обратите внимание, что в этом случае нам не нужно настраивать бин `SecurityFilterChain`, и можно придерживаться значений по умолчанию. Если используется Spring Boot без дополнительных настроек, мы можем вообще опустить бин `SecurityFilterChain`.

До Spring Security 6.2 нам приходилось гарантировать, что эта настройка применяется как для компонентов OAuth2 Login (если мы используем эту функцию), так и для компонентов OAuth2 Client. Нам нужно было использовать как Spring Security DSL (для типа разрешения `authorization_code`), так и публиковать бин типа `OAuth2AuthorizedClientManager` для других типов разрешения. 

Чтобы понять, что настраивается «за кулисами», вот как могла бы выглядеть конфигурация:

**Настройка `RestClient` для OAuth2 Client (до версии 6.2)**
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
		RestClientAuthorizationCodeTokenResponseClient 
		accessTokenResponseClient =
			new RestClientAuthorizationCodeTokenResponseClient();
			
		accessTokenResponseClient.setRestClient(restClient());

		http
			// ...
			.oauth2Login((oauth2Login) -> oauth2Login
				.tokenEndpoint((tokenEndpoint) -> tokenEndpoint
					.accessTokenResponseClient(accessTokenResponseClient)
				)
			)
			.oauth2Client((oauth2Client) -> oauth2Client
				.authorizationCodeGrant((authorizationCode) -> 
					authorizationCode
						.accessTokenResponseClient(
							accessTokenResponseClient
						)
				)
			);

		return http.build();
	}

	@Bean
	public OAuth2AuthorizedClientManager authorizedClientManager(
			ClientRegistrationRepository clientRegistrationRepository,
			OAuth2AuthorizedClientRepository authorizedClientRepository) {

		RestClientRefreshTokenTokenResponseClient 
		refreshTokenAccessTokenResponseClient =
			new RestClientRefreshTokenTokenResponseClient();
		
		refreshTokenAccessTokenResponseClient.setRestClient(restClient());


		RestClientClientCredentialsTokenResponseClient 
		clientCredentialsAccessTokenResponseClient =
			new RestClientClientCredentialsTokenResponseClient();
			
		clientCredentialsAccessTokenResponseClient
			.setRestClient(restClient());


		RestClientJwtBearerTokenResponseClient 
		jwtBearerAccessTokenResponseClient =
			new RestClientJwtBearerTokenResponseClient();
			
		jwtBearerAccessTokenResponseClient
			.setRestClient(restClient());


		JwtBearerOAuth2AuthorizedClientProvider 
		jwtBearerAuthorizedClientProvider =
			new JwtBearerOAuth2AuthorizedClientProvider();
			
		jwtBearerAuthorizedClientProvider
			.setAccessTokenResponseClient(
				jwtBearerAccessTokenResponseClient
			);


		RestClientTokenExchangeTokenResponseClient 
		tokenExchangeAccessTokenResponseClient =
			new RestClientTokenExchangeTokenResponseClient();
		
		tokenExchangeAccessTokenResponseClient
			.setRestClient(restClient());

		TokenExchangeOAuth2AuthorizedClientProvider 
		tokenExchangeAuthorizedClientProvider =
			new TokenExchangeOAuth2AuthorizedClientProvider();
			
		tokenExchangeAuthorizedClientProvider
			.setAccessTokenResponseClient(
				tokenExchangeAccessTokenResponseClient
			);



		OAuth2AuthorizedClientProvider authorizedClientProvider =
			OAuth2AuthorizedClientProviderBuilder.builder()
				.authorizationCode()
				.refreshToken((refreshToken) -> refreshToken
					.accessTokenResponseClient(
						refreshTokenAccessTokenResponseClient
					)
				)
				.clientCredentials((clientCredentials) -> 
				clientCredentials
					.accessTokenResponseClient(
						clientCredentialsAccessTokenResponseClient
					)
				)
				.provider(jwtBearerAuthorizedClientProvider)
				.provider(tokenExchangeAuthorizedClientProvider)
				.build();

		DefaultOAuth2AuthorizedClientManager authorizedClientManager =
			new DefaultOAuth2AuthorizedClientManager(
				clientRegistrationRepository, 
				authorizedClientRepository
			);
		authorizedClientManager
			.setAuthorizedClientProvider(authorizedClientProvider);

		return authorizedClientManager;
	}

	@Bean
	public RestClient restClient() {
		// ...
	}

}
```
