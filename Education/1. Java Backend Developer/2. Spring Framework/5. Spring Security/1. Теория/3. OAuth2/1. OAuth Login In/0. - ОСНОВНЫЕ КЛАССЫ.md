# 4. Два уровня абстракции ля хранения `OAuth2AuthorizedClient`
Spring Security предоставляет два интерфейса для управления сохраненными токенами доступа (`OAuth2AuthorizedClient`). 

Это сделано для того, чтобы отделить логику «где и как хранить токен» от способа «как идентифицировать владельца токена» в разных типах приложений.

## 1.1 OAuth2AuthorizedClientRepository

### Описание
Это **высокоуровневый**, универсальный API для работы с сохраненными токенами. 
Он оперирует объектами `HttpServletRequest` и `HttpServletResponse`, что делает его пригодным для использования в любых средах, где есть веб-запрос.

### Зачем нужен
- **Универсальность:** Работает как в классических Servlet-приложениях, так и в реактивном стеке (WebFlux).
    
- **Гибкость идентификации:** ***Сам извлекает информацию о текущем пользователе*** (`Authentication`) из контекста запроса, не требуя явной передачи имени пользователя (`principalName`).
    
- **Абстракция контекста:** Позволяет реализовать хранение токенов в средах, где нет глобального `SecurityContext` (например, в CLI-утилитах или при обработке сообщений из очереди).
    

### Где хранит токены
Реализация по умолчанию (`AuthenticatedPrincipalOAuth2AuthorizedClientRepository`) **не хранит токены сама**. 

Она действует как адаптер:
- Принимает вызов от фильтров Spring Security.
    
- Извлекает имя текущего пользователя из `Authentication`.
    
- Делегирует фактическое сохранение и загрузку **низкоуровневому** компоненту — `OAuth2AuthorizedClientService`.

---
## 1.2 OAuth2AuthorizedClientService
### Описание
Это **низкоуровневый** API для прямого доступа к хранилищу токенов. 

Он оперирует простыми строковыми ключами:
- `clientRegistrationId` — идентификатор OAuth2-клиента (например, `"google"`).
    
- `principalName` — имя пользователя (например, `"ivan"`).

### Зачем нужен
- **Простота реализации хранилища:** Ч*тобы сохранить или загрузить токен, достаточно знать два строковых идентификатора.* Это упрощает написание кастомных реализаций для базы данных.
    
- **Отделение от веб-слоя:** Класс не зависит от `HttpServletRequest` и `HttpSession`. Вы можете использовать его в фоновых задачах, где нет HTTP-запроса, достаточно лишь знать имя пользователя.
    
- **Основной строительный блок:** Является тем самым компонентом, который разработчик переопределяет, когда хочет хранить токены **не в памяти**, а, например, в JDBC или Redis.

### Где хранит токены
Стандартная реализация `InMemoryOAuth2AuthorizedClientService` хранит токены в оперативной памяти:
- **Структура:** `Map<String, OAuth2AuthorizedClient>`.
    
- **Ключ:** Строка, сформированная по принципу `clientRegistrationId + ":" + principalName`.
    
- **Особенность:** При перезапуске приложения все токены теряются, и пользователям потребуется заново авторизоваться у провайдера.

### Кастомные реализации (где хранить еще)
Благодаря простому интерфейсу, этот класс часто реализуют для хранения в:
- **Реляционных базах данных:** `JdbcOAuth2AuthorizedClientService`.
    
- **Распределенных кэшах:** Например, реализация на основе **Redis** для кластеризованных приложений.

# 5. OAuth2AuthorizedClientManager

## Описание
Это **главный менеджер OAuth2 токенов**.

## Что делает
- проверяет есть ли токен
- получает новый токен
- обновляет токен (refresh)
- решает какой grant type использовать

---
## Зачем нужен
👉 это главный фасад всей OAuth2 логики
