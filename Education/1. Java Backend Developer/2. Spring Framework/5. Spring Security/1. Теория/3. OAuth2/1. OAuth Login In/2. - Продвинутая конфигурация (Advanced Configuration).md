`HttpSecurity.oauth2Login()` **предоставляет ряд параметров конфигурации для настройки OAuth 2.0 Login**. Основные параметры конфигурации сгруппированы в соответствии с их аналогами — конечными точками протокола.

Например, `oauth2Login().authorizationEndpoint()` **позволяет настраивать конечную точку авторизации** (Authorization Endpoint), тогда как `oauth2Login().tokenEndpoint()` **позволяет настраивать конечную точку токена** (Token Endpoint).

Следующий код показывает пример:
**Продвинутая конфигурация OAuth2 Login**
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.oauth2Login((oauth2) -> oauth2
			    .authorizationEndpoint((authorization) -> authorization
			            ...
			    )
			    .redirectionEndpoint((redirection) -> redirection
			            ...
			    )
			    .tokenEndpoint((token) -> token
			            ...
			    )
			    .userInfoEndpoint((userInfo) -> userInfo
			            ...
			    )
			);
		return http.build();
	}
}
```

**Основная цель** DSL `oauth2Login()` заключалась в том, чтобы **тесно согласоваться с именованием, определенным в спецификациях.**

##### Prodocol Endpoints
OAuth 2.0 Authorization Framework определяет `Protocol Endpoints` следующим образом:

|Конечная точка|Назначение|
|---|---|
|**Authorization Endpoint**|Получение авторизации от владельца ресурса через перенаправление браузера.|
|**Token Endpoint**|Обмен authorization grant на access token.|
|**Redirection Endpoint**|Прием ответа с учетными данными от сервера авторизации (callback).|
##### UserInfo Enxpoints
Спецификация OpenID Connect Core 1.0 определяет конечную точку `UserInfo Endpoint` следующим образом:

Конечная точка UserInfo — это защищенный ресурс OAuth 2.0, который возвращает утверждения (claims) об аутентифицированном конечном пользователе. 

Чтобы получить запрошенные утверждения о конечном пользователе, клиент делает запрос к конечной точке UserInfo, используя токен доступа, полученный через аутентификацию OpenID Connect. 
> Эти утверждения обычно представлены объектом JSON, который содержит коллекцию пар имя-значение для утверждений.

## 1.1 Страница входа OAuth 2.0
По умолчанию страница входа OAuth 2.0 автоматически генерируется `DefaultLoginPageGeneratingFilter`. 

Стандартная страница входа показывает каждый настроенный OAuth Client с его `ClientRegistration.clientName` в виде ссылки, которая способна инициировать запрос авторизации (или OAuth 2.0 Login).

Чтобы `DefaultLoginPageGeneratingFilter` показывал ссылки для настроенных OAuth Client, зарегистрированный `ClientRegistrationRepository` должен также реализовывать `Iterable<ClientRegistration>`. 
> См. `InMemoryClientRegistrationRepository` для справки.


Назначение ссылки для каждого OAuth Client по умолчанию следующее:
```java
OAuth2AuthorizationRequestRedirectFilter
	.DEFAULT_AUTHORIZATION_REQUEST_BASE_URI + "/{registrationId}"
```

Следующая строка показывает пример:
```html
<a href="/oauth2/authorization/google">Google</a>
```

Чтобы переопределить страницу входа по умолчанию, настройте `oauth2Login().loginPage()` и (опционально) `oauth2Login().authorizationEndpoint().baseUri()`.

Следующий листинг показывает пример:

**Конфигурация страницы входа OAuth2**
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.oauth2Login((oauth2) -> oauth2
			    .loginPage("/login/oauth2")
			    ...
			    .authorizationEndpoint((authorization) -> authorization
			        .baseUri("/login/oauth2/authorization")
			        ...
			    )
			);
		return http.build();
	}
}
```

> **Примечание:** 
> ==Вам необходимо предоставить== `@Controller` с `@RequestMapping("/login/oauth2")`, **который способен отобразить пользовательскую страницу входа.**

> Как отмечалось ранее, настройка `oauth2Login().authorizationEndpoint().baseUri()` ***необязательна***. Однако, если вы решите ее настроить, убедитесь, что ссылка на каждый OAuth Client соответствует `authorizationEndpoint().baseUri()`.
> 
> Следующая строка показывает пример:
> ```html
> <a href="/login/oauth2/authorization/google">Google</a>
> ```

## 1.2 Конечная точка перенаправления (Redirection Endpoint)
**Конечная точка перенаправления используется** с***ервером авторизации*** **для возврата ответа авторизации** (Authorization Response), который содержит учетные данные авторизации, клиенту через пользовательский агент владельца ресурса.

OAuth 2.0 Login использует тип разрешения Authorization Code Grant. 
> Следовательно, учетные данные авторизации — это код авторизации (`authorization code`).

**Базовый URI ответа авторизации** (`baseUri`) по умолчанию (конечная точка перенаправления) — `/login/oauth2/code/*`, ==который определен== в `OAuth2LoginAuthenticationFilter.DEFAULT_FILTER_PROCESSES_URI`.


==Если вы хотите настроить== `baseUri` ответа авторизации, настройте его следующим образом:
**Конфигурация конечной точки перенаправления**
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

    @Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.oauth2Login((oauth2) -> oauth2
			    .redirectionEndpoint((redirection) -> redirection
			        .baseUri("/login/oauth2/callback/*")
			        ...
			    )
			);
		return http.build();
	}
}
```

**Примечание:** Вам также необходимо убедиться, что `ClientRegistration.redirectUri` соответствует пользовательскому `baseUri` ответа авторизации.

Следующий листинг показывает пример:
```java
return CommonOAuth2Provider.GOOGLE.getBuilder("google")
	.clientId("google-client-id")
	.clientSecret("google-client-secret")
	.redirectUri("{baseUrl}/login/oauth2/callback/{registrationId}")//❗❗❗
	.build();
```

## 1.3 Конечная точка UserInfo
Когда пользователь логинится через Google или GitHub, провайдер возвращает вам **свои** данные о пользователе. Например, Google может сказать: _«У этого пользователя есть scope `https://www.googleapis.com/auth/calendar.readonly`»_.

Spring Security по умолчанию честно преобразует этот scope в `GrantedAuthority` с именем:  
`SCOPE_https://www.googleapis.com/auth/calendar.readonly`

Но ваше приложение **ничего не знает** про Google Scopes. Ваша система авторизации построена на ролях вроде:
- `ROLE_USER`
    
- `ROLE_ADMIN`
    
- `ROLE_PREMIUM_CALENDAR_USER`
    

Если вы попытаетесь защитить эндпоинт через `.hasRole("PREMIUM_CALENDAR_USER")`, Spring Security **никогда не пустит** пользователя, потому что у него в кармане лежит `SCOPE_https://...`, а не `ROLE_PREMIUM_CALENDAR_USER`.

### 1.3.1 Сопоставление пользовательских полномочий (Mapping User Authorities)
**После того как пользователь успешно аутентифицируется** ***у*** ***провайдера OAuth 2.0***, `OAuth2User.getAuthorities()` (или `OidcUser.getAuthorities()`) **содержит список предоставленных полномочий, заполненный из** `OAuth2UserRequest.getAccessToken().getScopes()` и предваренный префиксом `SCOPE_`. 

Эти **предоставленные полномочия могут быть сопоставлены с новым набором экземпляров** `GrantedAuthority`, которые передаются в `OAuth2AuthenticationToken` при завершении аутентификации.

`OAuth2AuthenticationToken.getAuthorities()` ==используется для авторизации запросов,== например, в `hasRole('USER')` или `hasRole('ADMIN')`.


Существует ***несколько вариантов для выбора при сопоставлении пользовательских полномочий:***

---
#### Использование GrantedAuthoritiesMapper
`GrantedAuthoritiesMapper` получает список предоставленных полномочий, который содержит специальное полномочие типа `OAuth2UserAuthority` и строку полномочия `OAUTH2_USER` 
(или `OidcUserAuthority` и строку полномочия `OIDC_USER`).

Предоставьте реализацию `GrantedAuthoritiesMapper` и настройте ее следующим образом:
**Конфигурация Granted Authorities Mapper**
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

    @Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.oauth2Login((oauth2) -> oauth2
			    .userInfoEndpoint((userInfo) -> userInfo
			        .userAuthoritiesMapper(this.userAuthoritiesMapper())
			        ...
			    )
			);
		return http.build();
	}

	private GrantedAuthoritiesMapper userAuthoritiesMapper() {
		// 1. authorities — список от Spring Security (SCOPE_..., OAUTH2_USER)
	    return (authorities) -> { 
        Set<GrantedAuthority> mappedAuthorities = new HashSet<>();

        authorities.forEach(authority -> {
            // 2. Если это OpenID Connect провайдер (Google, Okta)
            if (authority instanceof OidcUserAuthority oidcUserAuthority) 
            {
                // 3. Достаем ID Token и UserInfo (профиль пользователя)
                OidcIdToken idToken = oidcUserAuthority.getIdToken();
                OidcUserInfo userInfo = oidcUserAuthority.getUserInfo();
                
                // 4. ПРИМЕР МАППИНГА:
                // Проверяем email. Если он принадлежит домену admin — даем ROLE_ADMIN.
                String email = userInfo.getEmail();
                if (email != null && email.endsWith("@my-company.com")) {
                    mappedAuthorities.add(
	                    new SimpleGrantedAuthority("ROLE_ADMIN"));
                } else {
                    mappedAuthorities.add(
	                    new SimpleGrantedAuthority("ROLE_USER"));
                }

                // 5. Можно также анализировать наличие определенных scopes
                if (authorities.stream().anyMatch(a -> 
                    a.getAuthority().equals("SCOPE_calendar.read"))) {
                    mappedAuthorities.add(
	                    new SimpleGrantedAuthority("ROLE_CALENDAR_USER"));
                }
            } 
            // 6. Если это обычный OAuth2 провайдер (GitHub, Facebook)
            else if (authority instanceof OAuth2UserAuthority oauth2UserAuthority) {
                
                Map<String, Object> userAttributes = 
	                oauth2UserAuthority.getAttributes();
                
                // 7. GitHub, например, возвращает поле "login" или "company"
                // Можно давать права на основе этих атрибутов.
            }
        });

        return mappedAuthorities; // 8. Возвращаем ПРЕОБРАЗОВАННЫЙ список прав
    };
}
}
```

В качестве альтернативы вы можете зарегистрировать бин `GrantedAuthoritiesMapper @Bean`, чтобы он автоматически применялся к конфигурации, следующим образом:

**Конфигурация бина Granted Authorities Mapper**
```java

public class OAuth2LoginSecurityConfig {

	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
		    .oauth2Login(withDefaults());
		return http.build();
	}

	@Bean
	public GrantedAuthoritiesMapper userAuthoritiesMapper() {
		...
	}
}
```

> **Примечание:** После завершения аутентификации она также содержит предоставленное полномочие `FACTOR_AUTHORIZATION_CODE`.

#### Стратегия на основе делегирования с OAuth2UserService
Эта стратегия является продвинутой по сравнению с использованием `GrantedAuthoritiesMapper`. 

Однако она также более гибкая, поскольку дает вам доступ к `OAuth2UserRequest` и `OAuth2User` (при использовании OAuth 2.0 UserService) или `OidcUserRequest` и `OidcUser` (при использовании OpenID Connect 1.0 UserService).

`OAuth2UserRequest` (и `OidcUserRequest`) ==предоставляет вам доступ к связанному== `OAuth2AccessToken`, что очень полезно в случаях, когда делегату необходимо извлечь информацию о полномочиях из защищенного ресурса, прежде чем он сможет сопоставить пользовательские полномочия для пользователя.

Следующий пример показывает, как реализовать и настроить стратегию на основе делегирования ***с использованием OpenID*** Connect 1.0 UserService:

**Конфигурация OAuth2UserService**
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.oauth2Login((oauth2) -> oauth2
			    .userInfoEndpoint((userInfo) -> userInfo
			        .oidcUserService(this.oidcUserService())
			        ...
			    )
			);
		return http.build();
	}

	private OAuth2UserService<OidcUserRequest, OidcUser> oidcUserService() {
		final OidcUserService delegate = new OidcUserService();

		return (userRequest) -> {
			// Делегировать реализацию по умолчанию для загрузки пользователя
			OidcUser oidcUser = delegate.loadUser(userRequest);

			OAuth2AccessToken accessToken = userRequest.getAccessToken();
			Set<GrantedAuthority> mappedAuthorities = new HashSet<>();

			// TODO
			// 1) Извлечь информацию о полномочиях из защищенного ресурса, используя accessToken
			// 2) Сопоставить информацию о полномочиях с одним или несколькими GrantedAuthority и добавить их в mappedAuthorities

			// 3) Создать копию oidcUser, но использовать вместо этого mappedAuthorities
			ProviderDetails providerDetails = 
				userRequest
					.getClientRegistration()
					.getProviderDetails();
					
			String userNameAttributeName = 
			providerDetails
				.getUserInfoEndpoint()
				.getUserNameAttributeName();
				
			if (StringUtils.hasText(userNameAttributeName)) {
				oidcUser = new DefaultOidcUser(
					mappedAuthorities, 
					oidcUser.getIdToken(), 
					oidcUser.getUserInfo(), 
					userNameAttributeName
				);
			} else {
				oidcUser = new DefaultOidcUser(
					mappedAuthorities, 
					oidcUser.getIdToken(), 
					oidcUser.getUserInfo()
				);
			}

			return oidcUser;
		};
	}
}
```

### 1.3.2 OAuth 2.0 UserService
`DefaultOAuth2UserService` — это **реализация** `OAuth2UserService`, которая поддерживает стандартных провайдеров OAuth 2.0.

`OAuth2UserService` ==получает атрибуты пользователя конечного пользователя (владельца ресурса) из конечной точки UserInfo== (используя токен доступа, предоставленный клиенту во время процесса авторизации) и возвращает `AuthenticatedPrincipal` в форме `OAuth2User`.

`DefaultOAuth2UserService` использует экземпляр `RestOperations` при запросе атрибутов пользователя в конечной точке UserInfo.

Если вам нужно настроить предварительную обработку запроса UserInfo, вы можете предоставить `DefaultOAuth2UserService.setRequestEntityConverter()` с пользовательским `Converter<OAuth2UserRequest, RequestEntity<?>>`. 

***Реализация по умолчанию*** `OAuth2UserRequestEntityConverter` строит представление `RequestEntity` для запроса UserInfo, которое по умолчанию устанавливает `OAuth2AccessToken` в заголовке `Authorization`.

С другой стороны, если вам нужно настроить постобработку ответа UserInfo, вам нужно предоставить `DefaultOAuth2UserService.setRestOperations()` с пользовательски настроенным `RestOperations`. `RestOperations` по умолчанию настраивается следующим образом:
```java
RestTemplate restTemplate = new RestTemplate();
restTemplate.setErrorHandler(
	new OAuth2ErrorResponseErrorHandler()
);
```

`OAuth2ErrorResponseErrorHandler` — это `ResponseErrorHandler`, который может обрабатывать ошибку OAuth 2.0 (400 Bad Request). Он использует `OAuth2ErrorHttpMessageConverter` для преобразования параметров ошибки OAuth 2.0 в `OAuth2Error`.

Независимо от того, настраиваете ли вы `DefaultOAuth2UserService` или предоставляете свою собственную реализацию `OAuth2UserService`, вам необходимо настроить ее следующим образом:

			        .userService(this.oauth2UserService())
			        ...
			    )
			);
		return http.build();
	}

		...
	}

## 1.4 Проверка подписи ID Token
Аутентификация OpenID Connect 1.0 вводит ID Token, который является токеном безопасности, содержащим утверждения (Claims) об аутентификации конечного пользователя сервером авторизации при использовании клиентом.

> [!NOTE] Подпись
> ID Token представлен как JSON Web Token (JWT) и ДОЛЖЕН быть подписан с использованием JSON Web Signature (JWS).

`OidcIdTokenDecoderFactory` предоставляет `JwtDecoder`, используемый для проверки подписи `OidcIdToken`. Алгоритм по умолчанию — RS256, но может отличаться, если он назначен во время регистрации клиента. Для таких случаев вы можете настроить резолвер (resolver) для возврата ожидаемого алгоритма JWS, назначенного для конкретного клиента.

Резолвер алгоритма JWS — это `Function`, которая принимает `ClientRegistration` и возвращает ожидаемый `JwsAlgorithm` для клиента, такой как `SignatureAlgorithm.RS256` или `MacAlgorithm.HS256`.

Следующий код показывает, как настроить бин `OidcIdTokenDecoderFactory @Bean` на использование по умолчанию `MacAlgorithm.HS256` для всех экземпляров `ClientRegistration`:
```java
@Bean
public JwtDecoderFactory<ClientRegistration> idTokenDecoderFactory() {
	OidcIdTokenDecoderFactory idTokenDecoderFactory = 
		new OidcIdTokenDecoderFactory();
		
	idTokenDecoderFactory
		.setJwsAlgorithmResolver((clientRegistration) -> 
			clientRegistration.HS256
		);
	return idTokenDecoderFactory;
}
```

> **Примечание:** Для алгоритмов на основе MAC (таких как `HS256`, `HS384` или `HS512`) `client-secret`, соответствующий `client-id`, используется в качестве симметричного ключа для проверки подписи.

> Если для аутентификации OpenID Connect 1.0 настроено более одного `ClientRegistration`, резолвер алгоритма JWS может оценить предоставленный `ClientRegistration`, чтобы определить, какой алгоритм вернуть.
