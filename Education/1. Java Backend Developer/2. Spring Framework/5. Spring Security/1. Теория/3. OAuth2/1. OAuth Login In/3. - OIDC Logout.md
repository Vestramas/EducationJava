После того как конечный пользователь может войти в ваше приложение, важно подумать о том, как он будет выходить из системы.

В общем случае можно рассмотреть три сценария использования:

- Я хочу выполнить только локальный выход из системы
    
- Я хочу выйти из системы как в моем приложении, так и у провайдера OIDC, инициировав выход из своего приложения
    
- Я хочу выйти из системы как в моем приложении, так и у провайдера OIDC, когда выход инициирован провайдером OIDC

---
Для начала зафиксируем термины, чтобы не путаться:
- **Провайдер (Provider):** Тот, кто хранит вашу учетную запись (Google, Okta, Keycloak). Он же **Authorization Server**.
    
- **Клиент (Client/RP — Relying Party):** Ваше приложение (например, интернет-магазин). В терминах спецификации называется **Проверяющая сторона**, потому что оно полагается на данные от Провайдера.
    
- **Пользователь:** Человек, который сидит в браузере.
---
# 1. Локальный выход (Local Logout)
Для выполнения локального выхода специальная настройка OIDC не требуется. 

Spring Security **автоматически разворачивает конечную точку (endpoint) ЛОКАЛЬНОГО ВЫХОДА**, которую вы можете настроить через DSL-метод `logout()`

---
# 2. OpenID Connect 1.0 Client-Initiated Logout (Выход, инициированный клиентом)
### **Суть проблемы:**  
Пользователь нажал кнопку «Выйти» в вашем приложении (`Клиент`). Ваше приложение удалило свою сессию (пользователь разлогинился локально). Но пользователь **остается залогиненным у Провайдера** (например, в Google).

Теперь, если кто-то откроет ваш сайт и нажмет «Войти через Google», Google скажет: «Я знаю этого пользователя, он все еще в системе!» — и **сразу пустит его обратно без ввода пароля**. Это не всегда безопасно (особенно на чужих компьютерах).

### **Как Клиент может помочь (решение):**  
Ваше приложение может сказать Google: «Эй, Гугл, пользователь нажал Выйти. Выйди из системы и у себя тоже, пожалуйста».

Для этого ваше приложение (Клиент) **перенаправляет браузер пользователя** на ==специальный URL Провайдера== — `end_session_endpoint`. 

Провайдер:
1. Удаляет свою сессию (разлогинивает пользователя у себя).
    
2. Перенаправляет браузер обратно в ваше приложение на указанный вами URL (`post_logout_redirect_uri`).
```java
// Настройка в вашем приложении
.oauth2Login(withDefaults())
.logout((logout) -> logout
    .logoutSuccessHandler(
	    oidcLogoutSuccessHandler()
	) // <-- Этот handler все делает
);
```

---
Спецификация OpenID Connect Session Management 1.0 позволяет выполнить выход конечного пользователя у провайдера с помощью клиента. 
Одна из доступных стратегий — `RP-Initiated Logout` (**выход, инициированный проверяющей стороной**).

Если провайдер OpenID поддерживает как управление сессиями (Session Management), так и обнаружение (Discovery), клиент может получить URL-адрес `end_session_endpoint` из метаданных обнаружения провайдера OpenID. 

Вы можете сделать это, настроив `ClientRegistration` с помощью `issuer-uri` следующим образом:
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          okta:
            client-id: okta-client-id
            client-secret: okta-client-secret
            ...
        provider:
          okta:
            issuer-uri: https://dev-1234.oktapreview.com
```

Также вам следует настроить `OidcClientInitiatedLogoutSuccessHandler`, который реализует RP-Initiated Logout, следующим образом:
```java
@Configuration
@EnableWebSecurity
public class OAuth2LoginSecurityConfig {

	@Autowired
	private ClientRegistrationRepository 
		clientRegistrationRepository;

	@Bean
	public SecurityFilterChain filterChain(
		HttpSecurity http
	) throws Exception {
		http
			.authorizeHttpRequests((authorize) -> authorize
				.anyRequest().authenticated()
			)
			.oauth2Login(withDefaults())
			.logout((logout) -> logout
				.logoutSuccessHandler(
					oidcLogoutSuccessHandler()
				)
			);
		return http.build();
	}

	private LogoutSuccessHandler 
		oidcLogoutSuccessHandler() 
	{
		OidcClientInitiatedLogoutSuccessHandler 
		oidcLogoutSuccessHandler =
			new OidcClientInitiatedLogoutSuccessHandler(
				this.clientRegistrationRepository
			);

		// Устанавливает местоположение, куда будет перенаправлен пользовательский агент
		// конечного пользователя после выполнения выхода у провайдера
		oidcLogoutSuccessHandler
			.setPostLogoutRedirectUri("{baseUrl}");

		return oidcLogoutSuccessHandler;
	}
}
```

---
> `OidcClientInitiatedLogoutSuccessHandler` поддерживает плейсхолдер `{baseUrl}`.  
> Если он используется, базовый URL-адрес приложения, например `app.example.org`, заменяет его во время запроса.

По умолчанию `OidcClientInitiatedLogoutSuccessHandler` **перенаправляет на URL-адрес выхода, используя стандартный HTTP-редирект с методом** `GET`.

Чтобы выполнить выход, **используя POST-запрос, установите стратегию редиректа** в `FormPostRedirectStrategy`, например, с помощью 
```java
OidcClientInitiatedLogoutSuccessHandler
	.setRedirectStrategy(new FormPostRedirectStrategy())
```

# 3. OpenID Connect 1.0 Back-Channel Logout (Фоновый выход)
### **Суть проблемы:**  
Представьте, что вы залогинились в два разных приложения через один Google-аккаунт: в **Яндекс.Почту** и в **Яндекс.Диск**. Вы нажали «Выйти» в Яндекс.Почте. Та вызвала Google, и Google разлогинил вас у себя.

Теперь Яндекс.Диск **не знает**, что вы вышли. Вы продолжаете оставаться залогиненным в Яндекс.Диске, хотя ваш центральный аккаунт уже «протух».

**Как Провайдер (Google) может помочь (решение):**  
Back-Channel Logout — это механизм, при котором **Провайдер сам стучится во все приложения** и говорит: «Слушайте, пользователь `sub=12345` вышел из системы. Убейте и вы его сессию!».

Это происходит **напрямую (server-to-server)**, без участия браузера пользователя (поэтому и называется «Back-Channel» — фоновый канал).

### **Как это работает (код + диаграмма):**
1. **При логине:** Ваше приложение сохраняет у себя связку: _«Сессия пользователя с ID `ABC` на моем сайте связана с токеном от Google, где `sid=XYZ` и `sub=ivan@mail.ru`»_. Это делает `OidcSessionRegistry`.
    
2. **При выходе:** Когда Google разлогинивает пользователя (неважно, в каком из приложений это инициировали), он отправляет **POST-запрос** прямо на ваш сервер на URL `/logout/connect/back-channel/google`. В теле запроса — **Logout Token** (JWT), в котором написано: «Вышел пользователь `sub=ivan@mail.ru`».
    
3. Ваше приложение проверяет подпись этого JWT, смотрит на `sub` или `sid` и **аннулирует все локальные сессии** этого пользователя через `OidcSessionRegistry`.

```java
// Настройка в вашем приложении
.oidcLogout((logout) -> logout
    .backChannel(Customizer.withDefaults()) // <-- Включает прием Back-Channel запросов
);
```

---
Спецификация OpenID Connect Session Management 1.0 позволяет выполнить **выход конечного пользователя** *у клиента* **путем выполнения провайдером API-вызова к клиенту.** Это называется OIDC Back-Channel Logout.

Чтобы включить эту функцию, вы можете развернуть конечную точку Back-Channel Logout в DSL следующим образом:
```java
@Bean
OidcBackChannelLogoutHandler oidcLogoutHandler() {
	return new OidcBackChannelLogoutHandler();
}

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests((authorize) -> authorize
            .anyRequest().authenticated()
        )
        .oauth2Login(withDefaults())
        .oidcLogout((logout) -> logout
            .backChannel(Customizer.withDefaults())
        );
    return http.build();
}
```

Затем вам нужен способ прослушивать события, публикуемые Spring Security, для удаления старых записей `OidcSessionInformation`, например:
```java
@Bean
public HttpSessionEventPublisher sessionEventPublisher() {
    return new HttpSessionEventPublisher();
}
```

Это гарантирует, что если `HttpSession#invalidate` будет вызван, то сессия также будет удалена из памяти.

Вот и все!

Это развернет конечную точку `/logout/connect/back-channel/{registrationId}`, которую провайдер OIDC может запросить для аннулирования данной сессии конечного пользователя в вашем приложении.

> `oidcLogout` требует, чтобы `oauth2Login` также был настроен.

> `oidcLogout` требует, чтобы сессионная cookie называлась `JSESSIONID`, чтобы корректно выполнять выход из каждой сессии через backchannel.

---
## 3.1 Архитектура Back-Channel Logout
Рассмотрим `ClientRegistration`, чей идентификатор равен `registrationId`.

Общий поток для Back-Channel Logout выглядит следующим образом:

1. Во время входа в систему Spring Security связывает ID Token, CSRF Token и идентификатор сессии провайдера (если есть) с идентификатором сессии вашего приложения в своей реализации `OidcSessionRegistry`.
    
2. Затем, во время выхода из системы, ваш провайдер OIDC выполняет API-вызов к `/logout/connect/back-channel/registrationId`, включая Logout Token, который указывает либо `sub` (конечного пользователя), либо `sid` (идентификатор сессии провайдера) для выхода.
    
3. Spring Security проверяет подпись и утверждения (claims) токена.
    
4. Если токен содержит утверждение `sid`, то завершается только та сессия клиента, которая связана с этой сессией провайдера.
    
5. В противном случае, если токен содержит утверждение `sub`, то завершаются **все** сессии этого клиента для данного конечного пользователя.

Помните, что поддержка OIDC в Spring Security является мультитенантной. Это означает, что будут завершены только те сессии, чей клиент соответствует утверждению `aud` в Logout Token.

Одна примечательная часть реализации этой архитектуры заключается в том, что она распространяет входящий back-channel запрос внутренне для каждой соответствующей сессии. Поначалу это может показаться излишним. Однако вспомните, что Servlet API не дает прямого доступа к хранилищу `HttpSession`. Выполняя внутренний вызов выхода, соответствующая сессия теперь может быть аннулирована.

Кроме того, формирование внутреннего вызова выхода позволяет запустить каждый набор `LogoutHandlers` для этой сессии и соответствующего `SecurityContext`.

## 3.2 Настройка конечной точки выхода из сессии
При опубликованном `OidcBackChannelLogoutHandler` конечная точка выхода из сессии имеет вид `{baseUrl}/logout/connect/back-channel/{registrationId}`.

Если `OidcBackChannelLogoutHandler` не подключен, то URL-адрес имеет вид `{baseUrl}/logout/connect/back-channel/{registrationId}`, что не рекомендуется, поскольку требует передачи CSRF-токена, что может быть затруднительно в зависимости от типа репозитория, используемого вашим приложением.

В случае, если вам нужно настроить конечную точку, вы можете указать URL-адрес следующим образом:
```java
http
    // ...
    .oidcLogout((oidc) -> oidc
        .backChannel((backChannel) -> backChannel
            .logoutUri("http://localhost:9000/logout/connect/back-channel/+{registrationId}+")
        )
    );
```

## 3.3 Настройка имени cookie для выхода из сессии
По умолчанию конечная точка выхода из сессии использует cookie `JSESSIONID` для связывания сессии с соответствующей `OidcSessionInformation`.

Однако имя cookie по умолчанию в Spring Session — `SESSION`.

Вы можете настроить имя cookie Spring Session в DSL следующим образом:
```java
@Bean
OidcBackChannelLogoutHandler oidcLogoutHandler(OidcSessionRegistry oidcSessionRegistry) {
    OidcBackChannelLogoutHandler logoutHandler = new OidcBackChannelLogoutHandler(oidcSessionRegistry);
    logoutHandler.setSessionCookieName("SESSION");
    return logoutHandler;
}
```

## 3.4 Настройка реестра сессий провайдера OIDC
По умолчанию Spring Security хранит в памяти все связи между сессией провайдера OIDC и сессией клиента.

Существует ряд обстоятельств, например кластеризованное приложение, когда было бы удобно хранить эту информацию в отдельном месте, например в базе данных.

Вы можете добиться этого, настроив пользовательский `OidcSessionRegistry`, например:
```java
@Component
public final class MySpringDataOidcSessionRegistry implements OidcSessionRegistry {
    private final OidcProviderSessionRepository sessions;

    // ...

    @Override
    public void saveSessionInformation(OidcSessionInformation info) {
        this.sessions.save(info);
    }

    @Override
    public OidcSessionInformation removeSessionInformation(String clientSessionId) {
       return this.sessions.removeByClientSessionId(clientSessionId);
    }

    @Override
    public Iterable<OidcSessionInformation> removeSessionInformation(OidcLogoutToken token) {
        return token.getSessionId() != null ?
            this.sessions.removeBySessionIdAndIssuerAndAudience(...) :
            this.sessions.removeBySubjectAndIssuerAndAudience(...);
    }
}
```
