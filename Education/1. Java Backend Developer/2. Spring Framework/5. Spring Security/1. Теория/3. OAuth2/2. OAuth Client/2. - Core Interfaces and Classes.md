В этом разделе описываются основные интерфейсы и классы OAuth2, которые предлагает Spring Security.

## 1. ClientRegistration

> [!IMPORTANT] ClientRegistration - клиент в OAuth
> `ClientRegistration` — это представление клиента, зарегистрированного у OAuth 2.0 или OpenID Connect 1.0 провайдера.

Объект `ClientRegistration` **содержит такую информацию, как** 
- client id, 
- client secret, 
- тип авторизационного гранта, 
- redirect URI, 
- scope(s), 
- authorization URI, 
- token URI 
- и другие детали.

`ClientRegistration` и его свойства определены следующим образом:
```java
public final class ClientRegistration {
	private String registrationId;	(1)
	private String clientId;	(2)
	private String clientSecret;	(3)
	private ClientAuthenticationMethod clientAuthenticationMethod;	(4)
	private AuthorizationGrantType authorizationGrantType;	(5)
	private String redirectUri;	(6)
	private Set<String> scopes;	(7)
	private ProviderDetails providerDetails;
	private String clientName;	(8)

	public class ProviderDetails {
		private String authorizationUri;	(9)
		private String tokenUri;	(10)
		private UserInfoEndpoint userInfoEndpoint;
		private String jwkSetUri;	(11)
		private String issuerUri;	(12)
        private Map<String, Object> configurationMetadata;  (13)

		public class UserInfoEndpoint {
			private String uri;	(14)
            private AuthenticationMethod authenticationMethod;  (15)
			private String userNameAttributeName;	(16)

		}
	}

	public static final class ClientSettings {
		private boolean requireProofKey; (17)
	}
}
```

| №    | Свойство                                       | Описание                                                                                                                                                                                                                               |
| ---- | ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| (1)  | `registrationId`                               | Идентификатор, который уникально идентифицирует `ClientRegistration`.                                                                                                                                                                  |
| (2)  | `clientId`                                     | Идентификатор клиента.                                                                                                                                                                                                                 |
| (3)  | `clientSecret`                                 | Секрет клиента.                                                                                                                                                                                                                        |
| (4)  | `clientAuthenticationMethod`                   | Метод, используемый для аутентификации Клиента у Провайдера. Поддерживаемые значения: `client_secret_basic`, `client_secret_post`, `private_key_jwt`, `client_secret_jwt` и `none` (публичные клиенты).                                |
| (5)  | `authorizationGrantType`                       | OAuth 2.0 Authorization Framework определяет четыре типа авторизационных грантов. Поддерживаемые значения: `authorization_code`, `client_credentials`, а также расширенный тип гранта `urn:ietf:params:oauth:grant-type:jwt-bearer`.   |
| (6)  | `redirectUri`                                  | Зарегистрированный redirect URI клиента, на который Authorization Server перенаправляет пользовательский агент (user-agent) конечного пользователя после того, как конечный пользователь аутентифицировался и разрешил доступ клиенту. |
| (7)  | `scopes`                                       | Scope(s), запрошенные клиентом во время потока Authorization Request, такие как `openid`, `email` или `profile`.                                                                                                                       |
| (8)  | `clientName`                                   | Описательное имя, используемое для клиента. Имя может использоваться в определенных сценариях, например, при отображении имени клиента на автоматически сгенерированной странице входа.                                                |
| (9)  | `authorizationUri`                             | URI конечной точки авторизации (Authorization Endpoint) сервера авторизации.                                                                                                                                                           |
| (10) | `tokenUri`                                     | URI конечной точки токена (Token Endpoint) сервера авторизации.                                                                                                                                                                        |
| (11) | `jwkSetUri`                                    | URI, используемый для получения набора JSON Web Key (JWK) от сервера авторизации, который содержит криптографический ключ(и), используемый для проверки JSON Web Signature (JWS) ID Token и (опционально) UserInfo Response.           |
| (12) | `issuerUri`                                    | Возвращает URI идентификатора эмитента (issuer identifier URI) для провайдера OpenID Connect 1.0 или сервера авторизации OAuth 2.0.                                                                                                    |
| (13) | `configurationMetadata`                        | Информация о конфигурации провайдера OpenID (OpenID Provider Configuration Information). Эта информация доступна, только если настроено свойство Spring Boot `spring.security.oauth2.client.provider.[providerId].issuerUri`.          |
| (14) | `(userInfoEndpoint)uri`                        | URI конечной точки UserInfo, используемый для доступа к утверждениям (claims) и атрибутам аутентифицированного конечного пользователя.                                                                                                 |
| (15) | `(userInfoEndpoint)`<br>`authenticationMethod` | Метод аутентификации, используемый при отправке токена доступа на конечную точку UserInfo. Поддерживаемые значения: `header`, `form` и `query`.                                                                                        |
| (16) | `userNameAttributeName`                        | Имя атрибута, возвращаемого в UserInfo Response, которое ссылается на имя или идентификатор конечного пользователя.                                                                                                                    |
| (17) | `requireProofKey`                              | Если `true` или если `clientAuthenticationMethod` равен `none`, то PKCE будет включен. По умолчанию `true` для типа гранта `authorization_code` и `false` для других типов грантов.                                                    |

Вы можете **изначально настроить** `ClientRegistration`, **используя** обнаружение конечной точки конфигурации провайдера OpenID Connect (**OpenID Connect Provider's Configuration endpoint**) или конечной точки метаданных сервера авторизации (Authorization Server's Metadata endpoint).

`ClientRegistrations` предоставляет удобные методы для настройки `ClientRegistration` таким способом, как показано ниже:
```java
ClientRegistration clientRegistration =
    ClientRegistrations.fromIssuerLocation("https://idp.example.com/issuer").build();
```

В качестве альтернативы вы можете использовать `ClientRegistrations.fromOidcIssuerLocation()` для запроса только конечной точки конфигурации провайдера OpenID Connect.

## 2. ClientRegistrationRepository
`ClientRegistrationRepository` **служит репозиторием** для OAuth 2.0 / OpenID Connect 1.0 `ClientRegistration`(s).

> **Информация о регистрации клиента** в конечном счете **хранится** и принадлежит связанному **серверу авторизации**. 
> 
> Этот ==репозиторий предоставляет возможность извлекать подмножество основной информации== о регистрации клиента, которая хранится на сервере авторизации.
> 
> **(КОРОЧЕ ДЛЯ РАБОТЫ С ClientRegistration)**

==Автоконфигурация Spring Boot связывает каждое из свойств в== `spring.security.oauth2.client.registration.[registrationId]` с экземпляром `ClientRegistration`, а затем объединяет каждый из экземпляров `ClientRegistration` в `ClientRegistrationRepository`.

Реализацией `ClientRegistrationRepository` по умолчанию является `InMemoryClientRegistrationRepository`.

Автоконфигурация также регистрирует `ClientRegistrationRepository` как `@Bean` в `ApplicationContext`, чтобы он был доступен для внедрения зависимостей, если это потребуется приложению.

Следующий листинг показывает пример:
```java
@Controller
public class OAuth2ClientController {

	@Autowired
	private ClientRegistrationRepository clientRegistrationRepository;

	@GetMapping("/")
	public String index() {
		ClientRegistration oktaRegistration =
			this
				.clientRegistrationRepository
					.findByRegistrationId("okta");

		...

		return "index";
	}
}
```

## 3. OAuth2AuthorizedClient

> [!NOTE] `OAuth2AuthorizedClient` - данные авторизованного клиента
> `OAuth2AuthorizedClient` — это **представление авторизованного клиента**. 
> 
> Клиент считается авторизованным, ***когда конечный пользователь (владелец ресурса) предоставил клиенту разрешение*** на доступ к своим защищенным ресурсам.

`OAuth2AuthorizedClient` ==служит для связи ==
- `OAuth2AccessToken` (и опционального `OAuth2RefreshToken`) 
- с `ClientRegistration` (клиентом) и владельцем ресурса, который является конечным пользователем (`Principal`), предоставившим авторизацию.

## 4. OAuth2AuthorizedClientRepository и OAuth2AuthorizedClientService

> [!NOTE] `OAuth2AuthorizedClientRepository` - СОХРАНЯЕТ OAuth2AuthorizedClient
> `OAuth2AuthorizedClientRepository` **отвечает за сохранение** `OAuth2AuthorizedClient`(s) между веб-запросами, тогда как основная роль 


> [!NOTE] `OAuth2AuthorizedClientService` - УПРАВЛЯЕТ OAuth2AuthorizedClient
> `OAuth2AuthorizedClientService` заключается в управлении `OAuth2AuthorizedClient`(s) ==на уровне приложения.==
> 

С точки зрения разработчика, `OAuth2AuthorizedClientRepository` или `OAuth2AuthorizedClientService` предоставляет возможность поиска `OAuth2AccessToken`, связанного с клиентом, чтобы его можно было использовать для инициирования запроса к защищенному ресурсу.

Следующий листинг показывает пример:
```java
@Controller
public class OAuth2ClientController {

    @Autowired
    private OAuth2AuthorizedClientService authorizedClientService;

    @GetMapping("/")
    public String index(Authentication authentication) {
        OAuth2AuthorizedClient authorizedClient =
            this
	            .authorizedClientService
		            .loadAuthorizedClient(
			            "okta", 
			            authentication.getName()
			        );

        OAuth2AccessToken accessToken = 
	        authorizedClient
		        .getAccessToken();

        ...

        return "index";
    }
}
```

> **Примечание:** **Автоконфигурация Spring Boot регистрирует** `OAuth2AuthorizedClientRepository` или `OAuth2AuthorizedClientService` **как** `@Bean` в `ApplicationContext`. 
> Однако *приложение может переопределить и зарегистрировать пользовательский* `OAuth2AuthorizedClientRepository` или `OAuth2AuthorizedClientService` `@Bean`.
> 
> Реализацией `OAuth2AuthorizedClientService` по умолчанию является `InMemoryOAuth2AuthorizedClientService`, которая хранит объекты `OAuth2AuthorizedClient` в памяти.
> 
> Альтернативно, вы можете настроить реализацию JDBC `JdbcOAuth2AuthorizedClientService` для сохранения экземпляров `OAuth2AuthorizedClient` в базе данных.
> 
> `JdbcOAuth2AuthorizedClientService` зависит от определения таблицы, описанной в разделе OAuth 2.0 Client Schema.

## 5. OAuth2AuthorizedClientManager и OAuth2AuthorizedClientProvider
`OAuth2AuthorizedClientManager` отвечает **за общее управление** `OAuth2AuthorizedClient`(s).

**Основные обязанности включают:**
- Авторизацию (или повторную авторизацию) OAuth 2.0 Client с использованием `OAuth2AuthorizedClientProvider`.
    
- Делегирование сохранения `OAuth2AuthorizedClient`, обычно с использованием `OAuth2AuthorizedClientService` или `OAuth2AuthorizedClientRepository`.
    
- Делегирование `OAuth2AuthorizationSuccessHandler` при успешной авторизации (или повторной авторизации) OAuth 2.0 Client.
    
- Делегирование `OAuth2AuthorizationFailureHandler` при неудачной авторизации (или повторной авторизации) OAuth 2.0 Client.

---

`OAuth2AuthorizedClientProvider` ==реализует стратегию авторизации==(или повторной авторизации) OAuth 2.0 Client. 
> **Реализации обычно реализуют какой-либо тип авторизационного гранта**, такой как `authorization_code`, `client_credentials` и другие.

Реализацией `OAuth2AuthorizedClientManager` по умолчанию является `DefaultOAuth2AuthorizedClientManager`, который связан с `OAuth2AuthorizedClientProvider`, который может поддерживать несколько типов авторизационных грантов с использованием композита на основе делегирования. Вы можете использовать `OAuth2AuthorizedClientProviderBuilder` для настройки и построения композита на основе делегирования.

Следующий код показывает **пример того, как настроить и построить композит** `OAuth2AuthorizedClientProvider`, **который обеспечивает поддержку типов авторизационных грантов** `authorization_code`, `refresh_token` и `client_credentials`:
```java
@Bean
public OAuth2AuthorizedClientManager authorizedClientManager(
	ClientRegistrationRepository 
		clientRegistrationRepository,
	OAuth2AuthorizedClientRepository 
		authorizedClientRepository
) {

	OAuth2AuthorizedClientProvider 
	authorizedClientProvider =
		OAuth2AuthorizedClientProviderBuilder.builder()
					.authorizationCode()
					.refreshToken()
					.clientCredentials()
					.build();

	DefaultOAuth2AuthorizedClientManager authorizedClientManager =
		new DefaultOAuth2AuthorizedClientManager(
				clientRegistrationRepository, 
				authorizedClientRepository
			);
	authorizedClientManager
		.setAuthorizedClientProvider(
			authorizedClientProvider
		);

	return authorizedClientManager;
}
```

### OAuth2AuthorizationSuccessHandler
Когда **попытка авторизации завершается успешно**, `DefaultOAuth2AuthorizedClientManager` **делегирует** `OAuth2AuthorizationSuccessHandler`, который (по умолчанию) сохраняет `OAuth2AuthorizedClient` через `OAuth2AuthorizedClientRepository`. 
В случае неудачи повторной авторизации (например, refresh token больше не действителен), ранее сохраненный `OAuth2AuthorizedClient` удаляется из `OAuth2AuthorizedClientRepository` через `RemoveAuthorizedClientOAuth2AuthorizationFailureHandler`.

Вы можете настроить поведение по умолчанию через 
```java
setAuthorizationSuccessHandler(
	OAuth2AuthorizationSuccessHandler
)

```
и
```java
setAuthorizationFailureHandler(
	OAuth2AuthorizationFailureHandler
)
```


`DefaultOAuth2AuthorizedClientManager` также связан с `contextAttributesMapper` типа `Function<OAuth2AuthorizeRequest, Map<String, Object>>`, который отвечает за отображение атрибута(ов) из `OAuth2AuthorizeRequest` в `Map` атрибутов, которые должны быть связаны с `OAuth2AuthorizationContext`. Это может быть полезно, когда вам нужно предоставить `OAuth2AuthorizedClientProvider` требуемые (поддерживаемые) атрибуты.

Следующий код показывает пример `contextAttributesMapper`:
```java
@Bean
public OAuth2AuthorizedClientManager authorizedClientManager(
		ClientRegistrationRepository clientRegistrationRepository,
		OAuth2AuthorizedClientRepository authorizedClientRepository) {

	OAuth2AuthorizedClientProvider 
	authorizedClientProvider =
		OAuth2AuthorizedClientProviderBuilder.builder()
					.authorizationCode()
					.refreshToken()
					.build();

	DefaultOAuth2AuthorizedClientManager authorizedClientManager =
			new DefaultOAuth2AuthorizedClientManager(
					clientRegistrationRepository, 
					authorizedClientRepository
				);
	authorizedClientManager
		.setAuthorizedClientProvider(
			authorizedClientProvider
		);

	// Предполагая, что атрибуты передаются как параметры `HttpServletRequest`,
	// отображаем параметры `HttpServletRequest` в `OAuth2AuthorizationContext.getAttributes()`
	authorizedClientManager
		.setContextAttributesMapper(
			contextAttributesMapper()
		);

	return authorizedClientManager;
}

private Function<OAuth2AuthorizeRequest, Map<String, Object>> contextAttributesMapper() {
	return authorizeRequest -> {
		Map<String, Object> contextAttributes = 
			Collections.emptyMap();
			
		HttpServletRequest servletRequest = 
		authorizeRequest
			.getAttribute(
				HttpServletRequest.class.getName()
			);
			
		String param1 = servletRequest
			.getParameter("param1");
		String param2 = servletRequest
			.getParameter("param2");
			
		if (StringUtils.hasText(param1) && StringUtils.hasText(param2)) 
		{
			contextAttributes = new HashMap<>();
			contextAttributes.put("param1", param1);
			contextAttributes.put("param2", param2);
		}
		return contextAttributes;
	};
}
```

`DefaultOAuth2AuthorizedClientManager` предназначен для использования в контексте `HttpServletRequest`. При работе вне контекста `HttpServletRequest` используйте `AuthorizedClientServiceOAuth2AuthorizedClientManager`.

### Сервисное приложение
Сервисное приложение — это распространенный случай использования `AuthorizedClientServiceOAuth2AuthorizedClientManager`. Сервисные приложения часто работают в фоновом режиме, без какого-либо взаимодействия с пользователем, и обычно работают под системной учетной записью, а не под учетной записью пользователя. OAuth 2.0 Client, настроенный с типом гранта `client_credentials`, может рассматриваться как тип сервисного приложения.

Следующий код показывает пример того, как настроить `AuthorizedClientServiceOAuth2AuthorizedClientManager`, который обеспечивает поддержку типа гранта `client_credentials`:
```java
@Bean
public OAuth2AuthorizedClientManager authorizedClientManager(
	ClientRegistrationRepository 
		clientRegistrationRepository,
	OAuth2AuthorizedClientService 
		authorizedClientService) {

	OAuth2AuthorizedClientProvider 
	authorizedClientProvider =
		OAuth2AuthorizedClientProviderBuilder.builder()
					.clientCredentials()
					.build();

	AuthorizedClientServiceOAuth2AuthorizedClientManager authorizedClientManager =
			new AuthorizedClientServiceOAuth2AuthorizedClientManager(
				clientRegistrationRepository, 
				authorizedClientService
			);
	authorizedClientManager
		.setAuthorizedClientProvider(
			authorizedClientProvider
		);

	return authorizedClientManager;
}
```
