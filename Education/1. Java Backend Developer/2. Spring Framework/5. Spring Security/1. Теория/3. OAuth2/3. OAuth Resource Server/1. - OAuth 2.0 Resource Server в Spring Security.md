Spring Security **поддерживает защиту конечных точек (endpoints) с использованием двух форм** OAuth 2.0 Bearer-токенов:
- **JWT**
- **Непрозрачные токены (Opaque Tokens)**

Это удобно в ситуациях, когда приложение делегировало управление своими полномочиями серверу авторизации (например, Okta или Ping Identity). 
К этому серверу авторизации могут обращаться серверы ресурсов для авторизации запросов.

> В этом разделе подробно описывается, как Spring Security обеспечивает поддержку OAuth 2.0 Bearer-токенов.

Рабочие примеры как для JWT, так и для непрозрачных токенов доступны в репозитории [Spring Security Samples](https://github.com/spring-projects/spring-security-samples).

---

Теперь мы можем рассмотреть, как работает аутентификация по Bearer-токену в Spring Security. 

Сначала мы видим, что, как и в случае с базовой аутентификацией (Basic Authentication), неаутентифицированному клиенту отправляется обратно заголовок `WWW-Authenticate`:
![[Pasted image 20260429124822.png]]
**Рисунок 1. Отправка заголовка WWW-Authenticate**

Рисунок выше основан на нашей диаграмме `SecurityFilterChain`.
1. Сначала пользователь делает неаутентифицированный запрос к ресурсу `/private`, на который у пользователя нет прав.
    
2. `AuthorizationFilter` Spring Security указывает, что неаутентифицированный запрос **Отклонен (Denied)** , выбрасывая `AccessDeniedException`.
    
3. Поскольку пользователь не аутентифицирован, `ExceptionTranslationFilter` инициирует **Начало аутентификации (Start Authentication)** . Настроенный `AuthenticationEntryPoint` является экземпляром `BearerTokenAuthenticationEntryPoint`, который отправляет заголовок `WWW-Authenticate`. `RequestCache` обычно является `NullRequestCache`, который не сохраняет запрос, поскольку клиент способен повторно воспроизвести запросы, которые он изначально запрашивал.

Когда клиент получает заголовок `WWW-Authenticate: Bearer`, он знает, что должен повторить попытку, используя bearer-токен. На следующем изображении показан процесс обработки bearer-токена:
![[Pasted image 20260429124912.png]]
**Рисунок 2. Аутентификация Bearer-токена**

Рисунок основан на нашей диаграмме `SecurityFilterChain`.
1. Когда пользователь отправляет свой bearer-токен, `BearerTokenAuthenticationFilter` создает `BearerTokenAuthenticationToken`, который является типом `Authentication`, извлекая токен из `HttpServletRequest`.
    
2. Затем `HttpServletRequest` передается в `AuthenticationManagerResolver`, который выбирает `AuthenticationManager`. `BearerTokenAuthenticationToken` передается в `AuthenticationManager` для аутентификации. Детали того, как выглядит `AuthenticationManager`, зависят от того, настроены ли вы на JWT или непрозрачный токен.
    

**Если аутентификация не удалась, то «Неудача» (Failure):**
1. `SecurityContextHolder` очищается.
    
2. Вызывается `AuthenticationEntryPoint`, чтобы снова инициировать отправку заголовка `WWW-Authenticate`.
    

**Если аутентификация успешна, то «Успех» (Success):**
1. Любая уже аутентифицированная `Authentication` в `SecurityContextHolder` загружается, и ее полномочия (`authorities`) добавляются к возвращенной `Authentication`.
    
2. `BearerTokenAuthenticationFilter` вызывает `FilterChain.doFilter(request, response)`, чтобы продолжить выполнение остальной логики приложения.
