# 1. Минимальные зависимости для JWT
Бо́льшая часть поддержки Resource Server собрана в `spring-security-oauth2-resource-server`. 

Однако **поддержка декодирования и проверки JWT находится в** `spring-security-oauth2-jose`, что означает, что оба артефакта необходимы для работающего сервера ресурсов, поддерживающего Bearer-токены в кодировке JWT.

---
# 2. Минимальная конфигурация для JWT
При использовании Spring Boot ***настройка приложения в качестве сервера ресурсов*** **состоит из двух основных шагов.** 
- Во-первых, включите необходимые зависимости, 
- а во-вторых, укажите расположение сервера авторизации.

## 2.1 Указание сервера авторизации
В приложении Spring Boot, чтобы указать, какой сервер авторизации использовать, просто сделайте следующее:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://idp.example.com/issuer
```

Где `idp.example.com/issuer` — это значение, содержащееся в утверждении `iss` для JWT-токенов, которые будет выпускать сервер авторизации. Resource Server будет использовать это свойство для дальнейшей самонастройки, обнаружения открытых ключей сервера авторизации и последующей проверки входящих JWT.

**Чтобы использовать свойство** `issuer-uri`, также ***должно быть истинно, что один из следующих URL-адресов является*** **поддерживаемой endpoit для сервера авторизации:** 
- `idp.example.com/issuer/.well-known/openid-configuration`, 
- `idp.example.com/.well-known/openid-configuration/issuer` 
- `idp.example.com/.well-known/oauth-authorization-server/issuer`. 

Эта конечная точка называется конечной точкой конфигурации провайдера (Provider Configuration endpoint) или конечной точкой метаданных сервера авторизации (Authorization Server Metadata endpoint).

Вот и все!

---
## 2.2 Ожидания при запуске

> [!NOTE] Самонастройка
> Когда используются это свойство и эти зависимости, Resource Server автоматически настроит себя для проверки Bearer-токенов в кодировке JWT.

*Он достигает этого с помощью детерминированного процесса обнаружения*, **который он запускает при первом запросе, содержащем JWT:**
1. Запрашивает конечную точку конфигурации провайдера или метаданных сервера авторизации для получения свойства `jwks_url`.
    
2. Запрашивает конечную точку `jwks_url` для получения поддерживаемых алгоритмов.
    
3. Настраивает стратегию проверки так, чтобы запрашивать `jwks_url` для получения действительных открытых ключей найденных алгоритмов.
    
4. Настраивает стратегию проверки так, чтобы проверять утверждение `iss` каждого JWT на соответствие `idp.example.com`.


Одним из преимуществ отсрочки этого процесса является то, что запуск Resource Server не привязан к доступности сервера авторизации.

Эта отсрочка управляется `SupplierJwtDecoder`. Подумайте о том, чтобы обернуть любой объявляемый вами бин `JwtDecoder`, чтобы сохранить это поведение.

## 2.3 Ожидания во время выполнения
Как только приложение запущено, Resource Server будет пытаться обработать любой запрос, содержащий заголовок `Authorization: Bearer`:
```http
GET / HTTP/1.1
Authorization: Bearer some-token-value # Resource Server обработает это
```

Пока указана эта схема, Resource Server будет пытаться обработать запрос в соответствии со спецификацией Bearer Token.

При наличии корректно сформированного JWT Resource Server:
1. Проверит его подпись по открытому ключу, полученному из конечной точки `jwks_url` во время запуска или при первом запросе (в зависимости от конфигурации) и сопоставленному с JWT.
    
2. Проверит временные метки JWT `exp` и `nbf`, а также утверждение `iss` JWT.
    
3. Сопоставит каждую область действия (`scope`) с полномочием (`authority`) с префиксом `SCOPE_`.

> По мере того как сервер авторизации делает доступными новые ключи, Spring Security будет автоматически ротировать ключи, используемые для проверки JWT.

Результирующий `Authentication#getPrincipal`, по умолчанию, является объектом Spring Security `Jwt`, а `Authentication#getName` сопоставляется со свойством `sub` JWT, если оно присутствует.

# 3. Как работает аутентификация по JWT
Далее рассмотрим архитектурные компоненты, которые Spring Security использует для поддержки аутентификации по JWT в приложениях на основе сервлетов, таких как тот, что мы только что видели.


> [!NOTE] `JwtAuthenticationProvider`
> `JwtAuthenticationProvider` — это реализация `AuthenticationProvider`, которая использует 
> - `JwtDecoder` 
> - и `JwtAuthenticationConverter` 
> 
> для аутентификации JWT.


Давайте посмотрим, как `JwtAuthenticationProvider` работает в Spring Security. 

Рисунок объясняет детали работы `AuthenticationManager` на рисунках из раздела «Чтение Bearer-токена».
![[Pasted image 20260429131152.png]]
**Рисунок 1. Использование JwtAuthenticationProvider**

1. `AuthenticationFilter` из раздела «Чтение Bearer-токена» передает `BearerTokenAuthenticationToken` в `AuthenticationManager`, который реализован `ProviderManager`.
    
2. `ProviderManager` настроен на использование `AuthenticationProvider` типа `JwtAuthenticationProvider`.
    
3. `JwtAuthenticationProvider` декодирует, проверяет (verifies) и валидирует (validates) `Jwt` с помощью `JwtDecoder`.
    
4. Затем `JwtAuthenticationProvider` использует `JwtAuthenticationConverter` для преобразования `Jwt` в коллекцию предоставленных полномочий.
    
5. Когда аутентификация успешна, возвращаемый объект `Authentication` имеет тип `JwtAuthenticationToken` и содержит принципала, которым является `Jwt`, возвращенный настроенным `JwtDecoder`, и набор полномочий, содержащий по крайней мере `FACTOR_BEARER`. В конечном итоге возвращенный `JwtAuthenticationToken` будет установлен в `SecurityContextHolder` фильтром аутентификации.

# 4. Прямое указание JWK Set Uri сервера авторизации
Если сервер авторизации не поддерживает никаких конечных точек конфигурации, *или если Resource Server должен иметь возможность* **инициализироваться независимо от сервера авторизации**, ***тогда можно также указать*** `jwk-set-uri`:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://idp.example.com
          jwk-set-uri: https://idp.example.com/.well-known/jwks.json
```

> JWK Set uri **не стандартизирован**, но обычно его можно найти в документации сервера авторизации.

В результате Resource Server **не будет пинговать сервер авторизации при запуске.** Мы по-прежнему указываем `issuer-uri`, чтобы Resource Server все еще проверял утверждение `iss` во входящих JWT.

> Это свойство также может быть указано непосредственно в DSL.

---
# 5. Предоставление аудиторий (Audiences)
Как уже было показано, свойство `issuer-uri` проверяет утверждение `iss`; ==это КТО отправил JWT==.

Boot также имеет свойство `audiences` для проверки утверждения `aud`; ==это КОМУ был отправлен JWT.==

Аудиторию сервера ресурсов можно указать следующим образом:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://idp.example.com
          audiences: https://my-resource-server.example.com
```

> Вы также можете добавить проверку `aud` программно, если это необходимо.

Результатом будет то, что если утверждение `iss` JWT не равно `idp.example.com`, а его утверждение `aud` не содержит `my-resource-server.example.com` в своем списке, то проверка завершится неудачей.

# 6. Переопределение или замена автоконфигурации Boot

> [!NOTE] Магия внутри
> Существует два `@Bean`, которые Spring Boot генерирует от имени Resource Server.

### Bean 1: SecurityFilterChain
Первый — это `SecurityFilterChain`, который настраивает приложение как сервер ресурсов. 

При включении `spring-security-oauth2-jose` этот `SecurityFilterChain` выглядит так:

**Конфигурация JWT по умолчанию**
```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests((authorize) -> authorize
            .anyRequest().authenticated()
        )
        .oauth2ResourceServer((oauth2) -> oauth2
	        .jwt(Customizer.withDefaults())
	    );
    return http.build();
}
```

Если приложение не предоставляет бин `SecurityFilterChain`, то Spring Boot предоставит указанный выше бин по умолчанию.

Заменить его так же просто, как предоставить бин в приложении:

**Пользовательская конфигурация JWT**
```java
import static org.springframework.security.oauth2.core.authorization.OAuth2AuthorizationManagers.hasScope;

@Configuration
@EnableWebSecurity
public class MyCustomSecurityConfiguration {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> 
            authorize
                .requestMatchers("/messages/**")
	                .access(hasScope("message:read"))
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer((oauth2) -> oauth2
                .jwt((jwt) -> 
	                jwt
	                    .jwtAuthenticationConverter(
		                    myConverter()
		                )
                )
            );
        return http.build();
    }
}
```

Приведенный выше код требует наличия области действия `message:read` для любого URL-адреса, начинающегося с `/messages/`.

**Методы DSL** `oauth2ResourceServer` также будут **переопределять или заменять автоконфигурацию.**

### Bean 2: JwtDecoder
Например, второй `@Bean`, который создает Spring Boot, — это `JwtDecoder`, который декодирует строковые токены в проверенные экземпляры `Jwt`:

**JWT Decoder**
```java
@Bean
public JwtDecoder jwtDecoder() {
    return JwtDecoders.fromIssuerLocation(issuerUri);
}
```

> Вызов `JwtDecoders#fromIssuerLocation` — это то, что ==вызывает endpoint конфигурации провайдера или метаданных сервера авторизации== для получения `JWK Set Uri`.

Если приложение не предоставляет бин `JwtDecoder`, то Spring Boot предоставит указанный выше бин по умолчанию.

И его конфигурацию можно переопределить с помощью `jwkSetUri()` или заменить с помощью `decoder()`.

Или, если вы вообще не используете Spring Boot, то оба этих компонента — цепочка фильтров и `JwtDecoder` — могут быть указаны в XML.

Цепочка фильтров указывается следующим образом:

**Конфигурация JWT по умолчанию**
```xml
<http>
    <intercept-uri pattern="/**" access="authenticated"/>
    <oauth2-resource-server>
        <jwt decoder-ref="jwtDecoder"/>
    </oauth2-resource-server>
</http>
```

А `JwtDecoder` следующим образом:
**JWT Decoder**
```xml
<bean id="jwtDecoder"
        class="org.springframework.security.oauth2.jwt.JwtDecoders"
        factory-method="fromIssuerLocation">
    <constructor-arg value="${spring.security.oauth2.resourceserver.jwt.jwk-set-uri}"/>
</bean>
```

## 6.1 Использование `jwkSetUri()`
JWK Set Uri сервера авторизации может быть настроен как конфигурационное свойство или указан в DSL:

**Конфигурация JWK Set Uri**
```java
@Configuration
@EnableWebSecurity
public class DirectlyConfiguredJwkSetUri {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> 
            authorize
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer((oauth2) -> oauth2
                .jwt((jwt) -> jwt
                    .jwkSetUri(
	    "https://idp.example.com/.well-known/jwks.json"
	                )
                )
            );
        return http.build();
    }
}
```

> Использование `jwkSetUri()` ==имеет приоритет над любым конфигурационным свойством.==

---
## 6.2 Использование `decoder()`
Более мощным, чем `jwkSetUri()`, является `decoder()`, который полностью заменит любую автоконфигурацию Boot для `JwtDecoder`:

**Конфигурация JWT Decoder**
```java
@Configuration
@EnableWebSecurity
public class DirectlyConfiguredJwtDecoder {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> authorize
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer((oauth2) -> oauth2
                .jwt((jwt) -> jwt
                    .decoder(myCustomDecoder())
                )
            );
        return http.build();
    }
}
```

Это удобно, когда необходима более глубокая настройка, такая как валидация, маппинг или таймауты запросов.

---
## 6.3 Предоставление бина `JwtDecoder`
Или же **предоставление бина** `JwtDecoder` **имеет тот же эффект, что и** `decoder()`. 

Вы можете создать его с помощью `jwkSetUri` следующим образом:
```java
@Bean
public JwtDecoder jwtDecoder() {
    return NimbusJwtDecoder.withJwkSetUri(jwkSetUri).build();
}
```

или вы можете использовать эмитент (issuer) и позволить `NimbusJwtDecoder` искать `jwkSetUri` при вызове `build()`, как показано ниже:
```java
@Bean
public JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder.withIssuerLocation(issuer).build();
    jwtDecoder.setJwtValidator(JwtValidators.createDefaultWithIssuer(issuer));
    return jwtDecoder;
}
```

Или, если значения по умолчанию вам подходят, вы также можете использовать `JwtDecoders`, который делает то же самое, что и выше, в дополнение к настройке валидатора декодера:
```java
@Bean
public JwtDecoders jwtDecoder() {
    return JwtDecoders.fromIssuerLocation(issuer);
}
```

# 7. Настройка доверенных алгоритмов

> [!WARNING] "ПРОВЕРЯТЬ ТОКЕНЫ ТОЛЬКО ПО RS256"
> По умолчанию `NimbusJwtDecoder`, и, следовательно, Resource Server, **будет доверять и проверять только токены, использующие** `RS256`.

Вы можете настроить это через Spring Boot, через строитель (`builder`) `NimbusJwtDecoder` или через ответ JWK Set.

## 7.1 Через Spring Boot
Самый простой **способ установить алгоритм** — через свойство:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          jws-algorithms: RS512
          jwk-set-uri: https://idp.example.org/.well-known/jwks.json
```

## 7.2 Использование строителя (Builder)
Для большей гибкости мы можем использовать строитель, который поставляется с `NimbusJwtDecoder`:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder
	    .withIssuerLocation(this.issuer)
            .jwsAlgorithm(RS512).build();
    
    jwtDecoder.
	    setJwtValidator(
		    JwtValidators
			    .createDefaultWithIssuer(this.issuer)
		);
    return jwtDecoder;
}
```

Многократный вызов `jwsAlgorithm()` настроит `NimbusJwtDecoder` на доверие более чем одному алгоритму, например:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder
	    .withIssuerLocation(this.issuer)
            .jwsAlgorithm(RS512)
            .jwsAlgorithm(ES512)
            .build();
    jwtDecoder.setJwtValidator(
	    JwtValidators
		    .createDefaultWithIssuer(this.issuer)
	);
    return jwtDecoder;
}
```

Или вы можете вызвать `jwsAlgorithms`:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder.withIssuerLocation(this.issuer)
            .jwsAlgorithms(algorithms -> {
                    algorithms.add(RS512);
                    algorithms.add(ES512);
            }).build();
    jwtDecoder.setJwtValidator(JwtValidators.createDefaultWithIssuer(this.issuer));
    return jwtDecoder;
}
```

## 7.3 Из ответа JWK Set

> [!NOTE] Nimbus - основа JWT в Spring Security
> Поскольку ==поддержка JWT в Spring Security основана на Nimbus,== вы также можете использовать все его замечательные возможности.


Например, Nimbus имеет реализацию `JWSKeySelector`, которая будет выбирать набор алгоритмов на основе ответа JWK Set URI. Вы можете использовать ее для создания `NimbusJwtDecoder` следующим образом:
```java
@Bean
public JwtDecoder jwtDecoder() {
    // делает запрос к конечной точке JWK Set
    JWSKeySelector<SecurityContext> jwsKeySelector =
            JWSAlgorithmFamilyJWSKeySelector.fromJWKSetURL(this.jwkSetUrl);

    DefaultJWTProcessor<SecurityContext> jwtProcessor =
            new DefaultJWTProcessor<>();
    jwtProcessor.setJWSKeySelector(jwsKeySelector);

    return new NimbusJwtDecoder(jwtProcessor);
}
```

# 8. Доверие к одному асимметричному ключу
**Проще**, *чем поддерживать Resource Server с помощью конечной точки JWK Set*, — **жестко запрограммировать открытый ключ RSA.** 

Открытый ключ может быть предоставлен через Spring Boot или с помощью строителя (Builder).

## 8.1 Через Spring Boot
Указать ключ через Spring Boot довольно просто. Расположение ключа можно указать следующим образом:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          public-key-location: classpath:my-key.pub
```

Или, чтобы обеспечить более сложный поиск, вы можете выполнить постобработку `RsaKeyConversionServicePostProcessor`:
```java
@Bean
BeanFactoryPostProcessor conversionServiceCustomizer() {
    return beanFactory ->
        beanFactory.getBean(RsaKeyConversionServicePostProcessor.class)
                .setResourceLoader(new CustomResourceLoader());
}
```

Укажите расположение вашего ключа:
```yaml
key.location: hfds://my-key.pub
```

И затем внедрите значение:
```java
@Value("${key.location}")
RSAPublicKey key;
```


## 8.2 Использование строителя (Builder)
Чтобы напрямую подключить `RSAPublicKey`, вы можете просто использовать соответствующий строитель `NimbusJwtDecoder`, например:
```java
@Bean
public JwtDecoder jwtDecoder() {
    return NimbusJwtDecoder.withPublicKey(this.key).build();
}
```

---
# 9. Доверие к одному симметричному ключу
Использование одного симметричного ключа также просто. Вы можете просто загрузить свой `SecretKey` и использовать соответствующий строитель `NimbusJwtDecoder`, например:
```java
@Bean
public JwtDecoder jwtDecoder() {
    return NimbusJwtDecoder.withSecretKey(this.key).build();
}
```

# 10. Настройка авторизации
**JWT, выпущенный сервером авторизации OAuth 2.0, обычно имеет атрибут** `scope` или `scp`, ==указывающий== области действия (**или полномочия)**, которые были предоставлены, например:

`{ …, "scope" : "messages contacts"}`

В этом случае **Resource Server** попытается *преобразовать эти области действия в список предоставленных полномочий*, д**обавляя к каждой области префикс строки** `"SCOPE_"`.

Это означает, что ***для защиты конечной точки или метода с помощью области действия***, полученной из JWT, **соответствующие выражения должны включать этот префикс**:

**Конфигурация авторизации**
```java
import static org.springframework.security.oauth2.core.authorization.OAuth2AuthorizationManagers.hasScope;

@Configuration
@EnableWebSecurity
public class DirectlyConfiguredJwkSetUri {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> 
	            authorize
	                .requestMatchers("/contacts/**")
		                .access(hasScope("contacts"))
	                .requestMatchers("/messages/**")
		                .access(hasScope("messages"))
	                .anyRequest().authenticated()
            )
            .oauth2ResourceServer((oauth2) -> oauth2
                .jwt(Customizer.withDefaults())
            );
        return http.build();
    }
}
```

Или аналогично с помощью безопасности методов:
```java
@PreAuthorize("hasAuthority('SCOPE_messages')")
public List<Message> getMessages(...) {}
```

## 10.1 Извлечение полномочий вручную
Однако существует ряд обстоятельств, при которых этого стандартного поведения недостаточно.

Например, некоторые серверы авторизации не используют атрибут `scope`, а вместо этого имеют свой собственный пользовательский атрибут. Или, в других случаях, серверу ресурсов может потребоваться адаптировать атрибут или композицию атрибутов в внутренние полномочия.

Для этой цели Spring Security поставляется с `JwtAuthenticationConverter`, который отвечает за преобразование `Jwt` в `Authentication`.

По умолчанию Spring Security подключит `JwtAuthenticationProvider` с экземпляром `JwtAuthenticationConverter` по умолчанию.

В рамках настройки `JwtAuthenticationConverter` вы можете предоставить вспомогательный конвертер для перехода от `Jwt` к коллекции предоставленных полномочий.

Допустим, ваш сервер авторизации передает полномочия в пользовательском утверждении (claim) под названием `authorities`. В этом случае вы можете настроить утверждение, которое `JwtAuthenticationConverter` должен проверять, следующим образом:

**Конфигурация утверждения полномочий**
```java
@Bean
public JwtAuthenticationConverter jwtAuthenticationConverter() {
    JwtGrantedAuthoritiesConverter 
	grantedAuthoritiesConverter = 
	    new JwtGrantedAuthoritiesConverter();
    
    grantedAuthoritiesConverter
	    .setAuthoritiesClaimName("authorities");


    JwtAuthenticationConverter jwtAuthenticationConverter 
	    = new JwtAuthenticationConverter();
    
    jwtAuthenticationConverter
	    .setJwtGrantedAuthoritiesConverter(
		    grantedAuthoritiesConverter
		);
    return jwtAuthenticationConverter;
}
```

Вы также можете настроить префикс полномочий по-другому. Вместо того чтобы добавлять к каждому полномочию **префикс** `SCOPE_`, **вы можете изменить его на** `ROLE_` следующим образом:

**Конфигурация префикса полномочий**
```java
@Bean
public JwtAuthenticationConverter jwtAuthenticationConverter() {
    JwtGrantedAuthoritiesConverter grantedAuthoritiesConverter = 
    new JwtGrantedAuthoritiesConverter();
    
    grantedAuthoritiesConverter
	    .setAuthorityPrefix("ROLE_");

    JwtAuthenticationConverter jwtAuthenticationConverter = new JwtAuthenticationConverter();
    
    jwtAuthenticationConverter
	    .setJwtGrantedAuthoritiesConverter(
		    grantedAuthoritiesConverter
		);
    return jwtAuthenticationConverter;
}
```

Или вы можете полностью удалить префикс, вызвав `JwtGrantedAuthoritiesConverter#setAuthorityPrefix("")`.

Для большей гибкости ==DSL поддерживает полную замену конвертера любым классом==, реализующим `Converter<Jwt, AbstractAuthenticationToken>`:
```java
static class CustomAuthenticationConverter implements Converter<Jwt, AbstractAuthenticationToken> {
    public AbstractAuthenticationToken convert(Jwt jwt) {
        return new CustomAuthenticationToken(jwt);
    }
}

// ...

@Configuration
@EnableWebSecurity
public class CustomAuthenticationConverterConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests((authorize) -> 
	            authorize
	                .anyRequest().authenticated()
	        )
            .oauth2ResourceServer((oauth2) -> oauth2
                .jwt((jwt) -> jwt
                    .jwtAuthenticationConverter(
                    new CustomAuthenticationConverter())
                )
            );
        return http.build();
    }
}
```

# 11. Настройка валидации
**При использовании минимальной конфигурации Spring Boot,** *указывающей URI эмитента сервера авторизации*, **Resource Server будет по умолчанию проверять утверждение** `iss`, **а также утверждения временных меток** `exp` и `nbf`.

В случаях, когда валидацию необходимо настроить, ==Resource Server поставляется с ДВУМЯ СТАНДАРТНЫМИ валидаторами==, а также принимает пользовательские экземпляры `OAuth2TokenValidator`.

## 11.1 Настройка валидации временных меток
JWT обычно имеют окно действия, причем начало окна указано в утверждении `nbf`, а конец — в утверждении `exp`.

Однако каждый **сервер может испытывать расхождение часов** (`clock drift`), **что может привести к тому, что токены будут казаться истекшими на одном сервере, но не на другом.** 
Это может вызвать некоторую головную боль по мере увеличения количества взаимодействующих серверов в распределенной системе.

Resource Server использует `JwtTimestampValidator` ==для проверки ОКНА ДЕЙСТВИЯ токена==, и его можно настроить с помощью `clockSkew`, чтобы смягчить вышеуказанную проблему:
```java
@Bean
JwtDecoder jwtDecoder() {
     NimbusJwtDecoder jwtDecoder = (NimbusJwtDecoder)
             JwtDecoders.fromIssuerLocation(issuerUri);

    OAuth2TokenValidator<Jwt> withClockSkew = 
    new DelegatingOAuth2TokenValidator<>(
        new JwtTimestampValidator(Duration.ofSeconds(60)),
        new JwtIssuerValidator(issuerUri)
    );

     jwtDecoder.setJwtValidator(withClockSkew);

     return jwtDecoder;
}
```

> По умолчанию Resource Server настраивает расхождение часов в 60 секунд.

## 11.2 Настройка валидации RFC 9068
Если вам нужно требовать токены, соответствующие RFC 9068, вы можете настроить валидацию следующим образом:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder.withIssuerLocation(issuerUri)
            .validateTypes(false).build();
    jwtDecoder
    .setJwtValidator(
	    JwtValidators.createAtJwtValidator()
			.audience("https://audience.example.org")
			.clientId("client-identifier")
			.issuer("https://issuer.example.org")
			.build()
	);
     return jwtDecoder;
}
```

## 11.3 Настройка пользовательского валидатора
Добавить проверку утверждения `aud` просто с помощью API `OAuth2TokenValidator`:
```java
OAuth2TokenValidator<Jwt> audienceValidator() {
    return 
	    new JwtClaimValidator<List<String>>(AUD, aud -> 
		    aud.contains("messaging")
		);
}
```

Или, для большего контроля, в==ы можете реализовать свой собственный== `OAuth2TokenValidator`:
```java
static class AudienceValidator implements OAuth2TokenValidator<Jwt> 
{
    OAuth2Error error = new OAuth2Error(
	    "custom_code", 
	    "Custom error message", 
	    null
	);

    @Override
    public OAuth2TokenValidatorResult validate(Jwt jwt) {
        if (jwt.getAudience().contains("messaging")) {
            return OAuth2TokenValidatorResult
	            .success();
        } else {
            return OAuth2TokenValidatorResult
	            .failure(error);
        }
    }
}

// ...
OAuth2TokenValidator<Jwt> audienceValidator() {
    return new AudienceValidator();
}
```

Затем, чтобы добавить в сервер ресурсов, нужно указать экземпляр `JwtDecoder`:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = (NimbusJwtDecoder)
        JwtDecoders.fromIssuerLocation(issuerUri);

    OAuth2TokenValidator<Jwt> audienceValidator = audienceValidator(); //Из кода выше
    
    OAuth2TokenValidator<Jwt> withIssuer = JwtValidators.createDefaultWithIssuer(issuerUri);
   
 OAuth2TokenValidator<Jwt> withAudience = 
	new DelegatingOAuth2TokenValidator<>(
		 withIssuer, 
		 audienceValidator
	);

    jwtDecoder.setJwtValidator(withAudience);

    return jwtDecoder;
}
```

> Как указывалось ранее, вы можете вместо этого настроить проверку `aud` в Boot.

# 12. Настройка маппинга набора утверждений (Claim Set Mapping)
==Spring Security использует библиотеку Nimbus для разбора JWT== и проверки их подписей. 

Следовательно, Spring Security подчиняется интерпретации Nimbus каждого значения поля и тому, как привести каждое из них к типу Java.


Например, поскольку Nimbus остается совместимым с Java 7, он *не использует* `Instant` для представления полей временных меток.

И вполне возможно использовать другую библиотеку для обработки JWT, которая может принимать свои собственные решения по приведению типов, требующие корректировки.

Или, проще говоря, серверу ресурсов может потребоваться добавить или удалить утверждения из JWT по причинам, специфичным для предметной области.

Для этих целей Resource Server поддерживает маппинг набора утверждений JWT с помощью `MappedJwtClaimSetConverter`.

## 12.1 Настройка преобразования одного утверждения
По умолчанию `MappedJwtClaimSetConverter` будет пытаться привести утверждения к следующим типам:

|Утверждение (Claim)|Тип Java|
|---|---|
|`aud`|`Collection<String>`|
|`exp`|`Instant`|
|`iat`|`Instant`|
|`iss`|`String`|
|`jti`|`String`|
|`nbf`|`Instant`|
|`sub`|`String`|

Стратегию преобразования отдельного утверждения можно настроить с помощью `MappedJwtClaimSetConverter.withDefaults`:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder.withIssuerLocation(issuer).build();

    MappedJwtClaimSetConverter converter = MappedJwtClaimSetConverter
        .withDefaults(
	        Collections.singletonMap(
			    "sub", 
		        this::lookupUserIdBySub)
		    );
    jwtDecoder.setClaimSetConverter(converter);

    return jwtDecoder;
}
```

Это сохранит все значения по умолчанию, за исключением того, что переопределит стандартный конвертер утверждений для `sub`.

## 12.2 Добавление утверждения
`MappedJwtClaimSetConverter` также можно использовать для добавления пользовательского утверждения, например, для адаптации к существующей системе:
```java
MappedJwtClaimSetConverter.withDefaults(Collections.singletonMap("custom", custom -> "value"));
```

## 12.3 Удаление утверждения
И удаление утверждения также просто, используя тот же API:
```java
MappedJwtClaimSetConverter.withDefaults(Collections.singletonMap("legacyclaim", legacy -> null));
```

## 12.4 Переименование утверждения
В более сложных сценариях, таких как одновременное обращение к нескольким утверждениям или переименование утверждения, Resource Server принимает любой класс, реализующий `Converter<Map<String, Object>, Map<String,Object>>`:
```java
public class UsernameSubClaimAdapter implements Converter<Map<String, Object>, Map<String, Object>> {
    private final MappedJwtClaimSetConverter delegate =
            MappedJwtClaimSetConverter.withDefaults(Collections.emptyMap());

    public Map<String, Object> convert(Map<String, Object> claims) {
        Map<String, Object> convertedClaims = this.delegate.convert(claims);

        String username = (String) convertedClaims.get("user_name");
        convertedClaims.put("sub", username);

        return convertedClaims;
    }
}
```

И затем экземпляр может быть передан как обычно:
```java
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder.withIssuerLocation(issuer).build();
    jwtDecoder.setClaimSetConverter(new UsernameSubClaimAdapter());
    return jwtDecoder;
}
```

# 13. Настройка таймаутов
По умолчанию Resource Server ==использует таймауты соединения и сокета по 30 секунд== *каждый для координации с сервером авторизации.*

В некоторых сценариях **это может быть слишком мало.** Кроме того, **это не учитывает более сложные шаблоны, такие как откат (back-off) и обнаружение (discovery).**

Чтобы ==настроить способ подключения Resource Server к серверу авторизации==, `NimbusJwtDecoder` принимает экземпляр `RestOperations`:
```java
@Bean
public JwtDecoder jwtDecoder(RestTemplateBuilder builder) {
    RestOperations rest = builder
            .setConnectTimeout(Duration.ofSeconds(60))
            .setReadTimeout(Duration.ofSeconds(60))
            .build();

    NimbusJwtDecoder jwtDecoder = NimbusJwtDecoder
	    .withIssuerLocation(issuer)
		    .restOperations(rest)
			    .build();
    return jwtDecoder;
}
```

Также **по умолчанию Resource Server кэширует в памяти набор JWK сервера авторизации в течение 5 минут,** что вы, возможно, захотите изменить. 
Кроме того, это не учитывает более сложные шаблоны кэширования, такие как вытеснение (eviction) или использование общего кэша.

Чтобы настроить способ кэширования Resource Server набора JWK, `NimbusJwtDecoder` принимает экземпляр `Cache`:
```java
@Bean
public JwtDecoder jwtDecoder(CacheManager cacheManager) {
    return NimbusJwtDecoder.withIssuerLocation(issuer)
            .cache(cacheManager.getCache("jwks"))
            .build();
}
```

Когда передан `Cache`, Resource Server будет использовать JWK Set Uri в качестве ключа и JSON JWK Set в качестве значения.

==Spring НЕ является провайдером кэша==, поэтому вам **нужно убедиться, что включены соответствующие зависимости, такие как** `spring-boot-starter-cache` и ваш любимый провайдер кэширования.

> Будь то таймауты сокета или кэша, *вы можете захотеть работать с Nimbus **напрямую**.* 
> 
> Для этого помните, что `NimbusJwtDecoder` поставляется с конструктором, который принимает `JWTProcessor` от Nimbus.
