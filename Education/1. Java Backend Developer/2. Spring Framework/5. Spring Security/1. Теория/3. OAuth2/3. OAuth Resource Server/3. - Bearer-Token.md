# 1. Разрешение Bearer-токена (Bearer Token Resolution)
**По умолчанию** ==Resource Server ищет bearer-токен в заголовке ==`Authorization`. 

Однако это поведение можно настроить несколькими способами.

## 1.1 Чтение Bearer-токена из пользовательского заголовка
Например, вам может понадобиться читать bearer-токен из пользовательского заголовка. 

Чтобы достичь этого, вы можете предоставить `DefaultBearerTokenResolver` как бин или подключить его экземпляр в DSL, как показано в следующем примере:

**Пользовательский заголовок для Bearer-токена**
```java
@Bean
BearerTokenResolver bearerTokenResolver() {
    DefaultBearerTokenResolver bearerTokenResolver = 
	    new DefaultBearerTokenResolver();
	    
    bearerTokenResolver.setBearerTokenHeaderName(
	    HttpHeaders.PROXY_AUTHORIZATION
	);
    return bearerTokenResolver;
}
```

Или, в обстоятельствах, когда провайдер использует и пользовательский заголовок, и значение, вы можете использовать `HeaderBearerTokenResolver` вместо этого.

## 1.2 Чтение Bearer-токена из параметра формы
Или вы можете захотеть читать токен из параметра формы, что можно сделать, настроив `DefaultBearerTokenResolver`, как показано ниже:

**Bearer-токен из параметра формы**
```java
DefaultBearerTokenResolver resolver = 
	new DefaultBearerTokenResolver();
	
resolver.setAllowFormEncodedBodyParameter(true);

http
    .oauth2ResourceServer((oauth2) -> oauth2
        .bearerTokenResolver(resolver)
    );
```

# 2. Распространение Bearer-токена (Bearer Token Propagation)
Теперь, когда ваш сервер ресурсов проверил токен, **может быть удобно передать его в нижестоящие сервисы.** 

Это довольно просто сделать с помощью `ServletBearerExchangeFilterFunction`, как показано в следующем примере:
```java
@Bean
public WebClient rest() {
    return WebClient.builder()
            .filter(
	            new ServletBearerExchangeFilterFunction()
	        )
            .build();
}
```

Когда вышеуказанный `WebClient` **используется для выполнения запросов, Spring Security будет искать текущий** `Authentication` **и извлекать любые учетные данные** `AbstractOAuth2Token`. 

Затем ==он распространит этот токен в заголовке== `Authorization`.

Например:
```java
this.rest.get()
        .uri("https://other-service.example.com/endpoint")
        .retrieve()
        .bodyToMono(String.class)
        .block()
```

Это вызовет `other-service.example.com/endpoint`, добавив за вас заголовок `Authorization` с bearer-токеном.

В местах, где вам нужно переопределить это поведение, достаточно просто указать заголовок самостоятельно, например так:
```java
this.rest.get()
        .uri("https://other-service.example.com/endpoint")
        .headers((headers) -> 
	        headers.setBearerAuth(overridingToken))
        .retrieve()
        .bodyToMono(String.class)
        .block()
```

В этом случае фильтр отступит и просто передаст запрос дальше по цепочке веб-фильтров.

> **Примечание:** В отличие от функции фильтра OAuth 2.0 Client, эта функция фильтра не предпринимает попыток обновить токен, если он истек. Для получения такого уровня поддержки, пожалуйста, используйте фильтр OAuth 2.0 Client.

## 2.1 Поддержка RestTemplate
**На данный момент НЕ существует эквивалента** `ServletBearerExchangeFilterFunction` для `RestTemplate`, но вы можете распространить bearer-токен запроса довольно просто с помощью своего собственного перехватчика:
```java
@Bean
RestTemplate rest() {
	RestTemplate rest = new RestTemplate();
	rest
		.getInterceptors()
		.add((request, body, execution) -> 
	{
		Authentication authentication = 
			SecurityContextHolder
				.getContext()
					.getAuthentication();
					
		if (authentication == null) {
			return execution.execute(request, body);
		}

		if (!(authentication.getCredentials() instanceof AbstractOAuth2Token)) {
			return execution.execute(request, body);
		}

		AbstractOAuth2Token token = 
			(AbstractOAuth2Token) authentication
				.getCredentials();
	    
	    request
		    .getHeaders()
		    .setBearerAuth(token.getTokenValue());
	    return execution.execute(request, body);
	});
	return rest;
}
```

> **Примечание:** В отличие от OAuth 2.0 Authorized Client Manager, этот перехватчик фильтра не предпринимает попыток обновить токен, если он истек. Для получения такого уровня поддержки, пожалуйста, создайте перехватчик с использованием OAuth 2.0 Authorized Client Manager.

# 3. Ошибка Bearer-токена (Bearer Token Failure)
**Bearer-токен может быть недействительным по ряду причин.** Например, токен может больше не быть активным.

**В этих обстоятельствах Resource Server выбрасывает** `InvalidBearerTokenException`. 
Как и другие исключения, это приводит к ответу с ошибкой OAuth 2.0 Bearer Token:
```
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer error_code="invalid_token", error_description="Unsupported algorithm of none", error_uri="https://tools.ietf.org/html/rfc6750#section-3.1"
```

Кроме того,==оно публикуется как ==`AuthenticationFailureBadCredentialsEvent`, **которое ВЫ МОЖЕТЕ ПРОСЛУШИВАТЬ в своем приложении следующим образом:**
```java
@Component
public class FailureEvents {
	@EventListener
    public void onFailure(
	    AuthenticationFailureBadCredentialsEvent 
	badCredentials
	) {
		if (badCredentials.getAuthentication() instanceof BearerTokenAuthenticationToken) {
		    // ... handle
        }
    }
}
```
