При подключении зависимостей Spring, нужно быть осторожным с версиями подключаемых зависимостей.

# 1. Экосистема Spring
## Уровень 1: Ядро — Spring Framework
Это фундамент. Все модули ядра выпускаются **одновременно и имеют одинаковую версию**.

|Модуль|Назначение|
|---|---|
|`spring-core`|Базовые утилиты, IoC-контейнер|
|`spring-context`|ApplicationContext, аннотации `@Configuration`, `@Bean`|
|`spring-web`|Базовые веб-абстракции|
|`spring-webmvc`|Spring MVC (DispatcherServlet, контроллеры)|

Если вы используете Spring Framework `6.2.5`, то **все** эти модули должны быть версии `6.2.5`.

Все эти модули:
- разрабатываются вместе
- всегда совместимы
- ==должны быть одной версии==

---
## Уровень 2: Проекты экосистемы Spring
Эти проекты построены **поверх** Spring Framework, но имеют **собственный цикл релизов и независимые версии**.

|Проект|Актуальная версия (весна 2026)|Какую версию Spring Framework ожидает|
|---|---|---|
|**Spring Security**|`6.4.x`, `7.0.x`|Spring 6.2.x (для Security 6.4)|
|**Spring Data JPA**|`3.4.x`, `4.0.x`|Spring 6.2.x (для Data 3.x)|
|**Spring Batch**|`5.2.x`|Spring 6.2.x|
|**Spring Integration**|`6.4.x`|Spring 6.2.x|

Каждый проект Spring Data, Spring Security и другие заявляют в своём `pom.xml`: 
> «Я совместим с Spring Framework версии X.Y.Z». 

Maven это видит и **подтягивает** Spring Framework той версии, которую запросил проект уровня 2.

👉 Важно:
- у них **своя нумерация версий**
- они **НЕ ОБЯЗАНЫ СОВПАДАТЬ ПО ВЕРСИЯМ с Spring Framework**
- но они ==жёстко завязаны на конкретные версии Framework==

## Уровень 3 — агрегатор (Spring Boot)
Spring Boot — это:
- не просто библиотека
- а **менеджер совместимости**

Он говорит:
> “Вот набор версий, которые точно работают вместе”

# 2. Возможные проблемы

Допустим вы явно указали:
- `spring-context:6.2.5` (Уровень 1)
    
- `spring-data-jpa:4.0.4` (Уровень 2)


Spring Data JPA `4.0.4` внутри себя ожидает Spring Framework **7.0.6** (это его транзитивная зависимость). 

Maven видит два требования:
- Вы хотите `spring-core:6.2.5`
- Spring Data JPA хочет `spring-core:7.0.6`


В итоге в classpath оказываются **два `spring-core` разных версий**.

## Какое правило применяет Maven
Maven использует правило **«ближайший в дереве зависимостей»** (nearest-wins). 

Если Spring Data JPA тянет `spring-core:7.0.6` на один уровень ближе, чем ваша явная зависимость `spring-core:6.2.5`, 
Maven выберет версию `7.0.6`, а ваша версия `6.2.5` будет проигнорирована (или они обе окажутся в classpath — зависит от структуры дерева).

### Почему проект может работать
Maven действительно выберет **одну** версию `spring-core` (скорее всего 7.0.6, так как она ближе в дереве зависимостей), и ваш проект скомпилируется и даже запустится. **С точки зрения Maven конфликта нет, он его разрешил.**

### Но проблема в другом
**Проблема в** **бинарной несовместимости** между мажорными версиями Spring Framework. 

Когда в classpath оказывается `spring-core:7.0.6` (решение Maven), а остальные ваши явные зависимости (`spring-context`, `spring-webmvc`) остаются версии `6.2.5` возникает следующее:

1. **Разные API в разных версиях.** Spring 6.2.5 и Spring 7.0.6 — это **разные мажорные версии**. В Spring 7.0.6 могли быть удалены классы, изменены сигнатуры методов, переименованы пакеты.
    
2. **Ожидание разных версий друг друга.** `spring-webmvc:6.2.5` ожидает найти `spring-core:6.2.5` (свою пару). Когда он в рантайме находит `spring-core:7.0.6`, он может вызывать методы, которые в 7.0.6 уже не существуют, или возвращают другой тип.
    
3. **Непредсказуемое поведение.** Maven разрешил конфликт на уровне зависимостей, но не проверил совместимость API. Результат — `NoSuchMethodError`, `ClassNotFoundException`, `NoClassDefFoundError` в рантайме.

## Maven Enforcer Plugin (для автоматической проверки)
Добавьте в `pom.xml`, чтобы сборка падала при конфликтах:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-enforcer-plugin</artifactId>
            <version>3.5.0</version>
            <executions>
                <execution>
                    <id>enforce-versions</id>
                    <goals><goal>enforce</goal></goals>
                    <configuration>
                        <rules>
                            <dependencyConvergence/>
                        </rules>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

# 4. Золотое правило
**Всегда согласовывайте версии Уровня 1 и Уровня 2.** Spring Framework и проекты экосистемы должны быть совместимы. 

Таблицу совместимости можно найти в документации каждого проекта.

## Как выбрать правильные версии
1. Сначала определитесь с версией **Spring Boot**, если вы его используете. Spring Boot сам подберёт совместимые версии всех проектов.
    
2. **Без Spring Boot** — ***посмотрите на*** **самый требовательный проект** (***тот, что просит самую новую версию Spring Framework***) и используйте её для всех модулей ядра.
