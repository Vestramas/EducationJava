**Логирование** — это процесс фиксации и структурирования информации о работе системы, приложения, сервера или сетевого устройства в виде хронологических записей (логов). 

Эти записи содержат данные о событиях, ошибках, действиях пользователей, изменениях состояния системы и других значимых моментах

## 0.1 Классический System.out System.err
Все системы логирования, построены на обычных операция ввода/вывода 
- `System.out` - поток вывода, 
- `System.err` - поток вывода ошибок

```java
package education.samarcev.logging;

public class JUL {
	public static void main(String[] args) {
		System.out.println("Hello");
		System.err.println("Hello from err");
	}
}
```

![[Pasted image 20260531213539.png]]

Также можно изменить стандартный поток вывода:
```java
public class JUL {
	public static void main(String[] args) throws FileNotFoundException {
		System.setOut(new PrintStream(new File("D://outstream.txt")));
		System.out.println("Hello");
		System.err.println("Hello from err");
	}
}
```

И тогда, весь `System.out` будет писать в файл.
Такой подход ОЧЕНЬ ПРОИЗВОДИТЕЛЬНЫЙ (на 1000 сообщений - 8-9мс)

## 0.1 JUL - java.util.logging (с JDK 1.4)
JUL — это встроенный в JDK фреймворк, не требующий никаких внешних зависимостей. Несмотря на репутацию «медленного», в небольших утилитах или при правильной настройке менеджера он вполне работоспособен.

### Конфигурация JUL
По умолчанию JUL читает настройки из файла `logging.properties`, расположенного в директории `JRE_HOME/lib/`. 

Чтобы использовать свой файл, **нужно указать его при запуске JVM**:
```bash
java -Djava.util.logging.config.file=./my-logging.properties -jar myapp.jar
```

**Пример `logging.properties`:**
```properties
# Уровень корневого логгера
.level = INFO

# Консольный хендлер (аналог аппендера)
handlers = java.util.logging.ConsoleHandler, java.util.logging.FileHandler

# Формат сообщений (аналог Layout)
java.util.logging.ConsoleHandler.formatter = 
	java.util.logging.SimpleFormatter

# Уровень консольного хендлера
java.util.logging.ConsoleHandler.level = ALL

# Настройка файлового хендлера с ротацией
java.util.logging.FileHandler.pattern   = logs/myapp_%g.log
java.util.logging.FileHandler.limit     = 1024000
java.util.logging.FileHandler.count     = 5
java.util.logging.FileHandler.formatter = java.util.logging.XMLFormatter

# Кастомные уровни для пакетов
com.example.myapp.service.level = FINE
com.example.myapp.dao.level = FINEST
```

> [!NOTE] Можно использовать настройки по умолчанию, и сразу использовать логгер
> 

### Код использования
Вместо привычных аппендеров и лейаутов в JUL используются «хендлеры» (Handlers) и «форматтеры» (Formatters).
```java
import java.util.logging.Logger;
import java.util.logging.Level;

public class JulExample {
    // Создаём логгер по имени класса
    private static final Logger log = 
	    Logger.getLogger(JulExample.class.getName());

    public void execute() {
        // Уровни в JUL: SEVERE, WARNING, INFO, CONFIG, FINE, FINER, FINEST
        log.info("Приложение запущено");

        if (log.isLoggable(Level.FINE)) {
            // FINE — это аналог DEBUG
            log.fine(
	            "Детальная отладочная информация: " + 
	            expensiveOperation()
	        );
        }

        try {
            throw new RuntimeException("Ошибка соединения");
        } catch (Exception e) {
            // Передаём исключение
            log.log(
	            Level.SEVERE, 
	            "Критическая ошибка при выполнении операции",
	            e
	        );
        }
    }

    private String expensiveOperation() {
        return "Тяжелые вычисления...";
    }

    public static void main(String[] args) {
        new JulExample().execute();
    }
}
```

### Проблема производительности JUL
Медлительность JUL связана с тем, что внутри класса `Logger` по умолчанию используется `java.util.logging.LogManager`, **который синхронизирует многие операции через** `synchronized`-блоки. 

Это создает узкое горлышко при интенсивном многопоточном логировании.

Решение — **заменить стандартный менеджер** на высокопроизводительный, например, `JBoss Log Manager`. 

Делается это через системное свойство:
```bash
java -Djava.util.logging.manager=org.jboss.logmanager.LogManager -jar myapp.jar
```

После такой замены JUL, по утверждениям разработчиков, «летает».

> [!WARNING] Замена LogManager из JUL другим (`JBosLogManager`)
> **заменить `LogManager` можно только при запуске JVM**, и вот почему.

### ФАСАД `java.lang.System.Logger` от JDK
`System.Logger` — это минималистичный API для логирования, который является **абстракцией**, а не реализацией. 

Он позволяет коду внутри самого JDK (и вашему) писать логи, не привязываясь к конкретному фреймворку.


- **По умолчанию** все вызовы `System.getLogger()` перенаправляются в JUL.
    
- **Но** если в classpath есть реализация SLF4J или Log4j2, системный логгер JDK автоматически начинает писать через них.

| Характеристика     | JUL                                                | `System.Logger`                              |
| ------------------ | -------------------------------------------------- | -------------------------------------------- |
| **Пакет**          | `java.util.logging`                                | `java.lang.System` (вложенный интерфейс)     |
| **Роль**           | **Полноценная реализация** (движок)                | **Фасад** (абстракция)                       |
| **Появился в**     | Java 1.4                                           | Java 9                                       |
| **Уровни**         | SEVERE, WARNING, INFO, CONFIG, FINE, FINER, FINEST | ALL, TRACE, DEBUG, INFO, WARNING, ERROR, OFF |
| **Кто использует** | Прикладной код                                     | Внутренний код JDK + опционально прикладной  |


```java
import java.lang.System.Logger;
import java.lang.System.Logger.Level;

public class SystemLoggerExample {
    // Используем фасад JDK
    private static final Logger log = 
	    System.getLogger(SystemLoggerExample.class.getName());

    public void execute() {
        log.log(Level.INFO, "Сообщение через System.Logger");

        // Ленивое вычисление сообщения (через Supplier)
        log.log(
	        Level.DEBUG, 
	        () -> "Тяжелое сообщение: " + expensiveOperation());

        try {
            throw new RuntimeException("Ошибка");
        } catch (Exception e) {
            log.log(Level.ERROR, "Произошла ошибка", e);
        }
    }

    private String expensiveOperation() {
        return "Данные...";
    }
}
```

# 1. Введение в системы логирования (ИСТОРИЯ)
Эволюция логирования в Java — это классический пример того, как несовершенство встроенных решений порождает мощные open-source проекты, личные амбиции и «зоопарк» технологий.

## 1.1 **Начало: Эра System.out и System.err**  
На заре Java единственным способом вывода диагностической информации были стандартные потоки вывода (`System.out`) и ошибок (`System.err`). 

Никаких уровней, фильтрации или гибкой конфигурации — только примитивная запись в консоль.

## 1.2 **Первая волна сообщества: Apache Log4j 1.x**
Сообщество разработчиков быстро осознало потребность в мощной системе логирования. 

Cообщество под руководством швейцарского разработчика Чеки Гюльджю (Ceki Gülcü) создало библиотеку **Log4j 1**. 

Она была быстрой, гибкой и легко конфигурировалась. 

Проект быстро стал индустриальным стандартом де-факто, развиваясь под крылом Apache Software Foundation.

## 1.3 **Ответ JDK: `java.util.logging` (JUL)**
Только в **Java 1.4 (2002 год)**, когда Log4j 1 уже прочно занял нишу логирования, в состав стандартной библиотеки была включена система **`java.util.logging` (JUL)**.

Она была доступна "**из коробки**", **но страдала от серьезных проблем с производительностью** из-за неудачных внутренних механизмов блокировок. Это делало её категорически непригодной для высоконагруженных систем, где логирование не должно тормозить бизнес-логику.

### **Позднее дополнение JDK: `java.lang.System.Logger`**
В Java 9 был добавлен легковесный фасад `System.Logger`, который используется внутри самого JDK и по умолчанию работает поверх JUL, но может быть перенаправлен на любой современный движок.

## 1.4 **Раскол и рождение фасада: SLF4J и Logback**
В процессе эволюции Log4j 1 у его создателя (Чеки Гюльджю) **возникли фундаментальные разногласия** с Apache Software Foundation относительно дальнейшего вектора развития. 

В результате ==он покинул проект и основал собственный==, сделав «работу над ошибками». 
Так появился **SLF4J (Simple Logging Facade for Java)** — не библиотека логирования как таковая, а **фасад**, абстракция, которая позволяла разработчикам писать код, не привязываясь к конкретной реализации движка. 
В дополнение к фасаду он создал и «родной» ==движок для него== — **Logback**, который стал прямым наследником идей Log4j 1, но более быстрым, надежным и современным. 

На сегодняшний день связка SLF4J + Logback — это признанный стандарт "must have" для большинства Java-приложений.

## 1.5 **Ответ Apache: Рождение Log4j 2**  
Осознав, что оригинальный Log4j 1 безнадежно устарел (архитектурно и морально) и не может конкурировать с Logback, Apache Software Foundation приняла радикальное решение.

Вместо доработки первой версии они создали **Log4j 2** — полностью переписанный с нуля продукт, не имеющий обратной совместимости с Log4j 1. Он учёл все современные требования: асинхронное логирование, отказоустойчивость и ультра-низкие задержки.

## 1.6 **Параллельный трек: Экосистема JBoss**  
Пока развивались основные ветки, компания Red Hat (JBoss) создала свой стек технологий для нужд своего мощного сервера приложений WildFly (ранее JBoss AS): **JBoss Logging** (фасад, аналог SLF4J) и **JBoss Log Manager** (==менеджер, заменяющий «медленный» внутренний менеджер JUL==). 

Эти инструменты и по сей день развиваются и активно используются внутри экосистемы WildFly, решая свои специфические задачи, но вне её не получили такого широкого распространения.

# 2. Компоненты систем логирования
Система логирования состоит из трех основных абстракций: 
- Логгеры, 
- Аппендеры 
- и Лейауты. 

Дополнительно могут использоваться Фильтры.

## 2.1 Логгер (Logger)

> [!IMPORTANT] Логгер - объект, который используется для вызова записи лога
> Логгер — это основной объект, который разработчик вызывает в коде для записи сообщения. Логгеры организованы в иерархию.

### 2.1.1 Именование и иерархия
- **ИМЯ ЛОГГЕРА:** 
  Обычно равно полному (ВКЛЮЧАЯ ПАКЕТ) имени класса (`com.example.service.UserService`). 
  Разделителем выступает точка.
    
- **Иерархия:** 
  Логгеры образуют дерево. 
  Родительским для логгера `com.example.service` является логгер `com.example`, а для него — `com` и корневой `root`. 
  
  Это наследование используется ==для передачи настроек (уровня, аппендеров) ВВЕРХ (к корню) по дереву.==
	![[Pasted image 20260531210143.png]]

- **Правило обхода:** 
  Когда вы логируете событие из класса `com.example.service.UserService`, ==лог-запись проходит вверх по иерархии к== `root`. 
  Каждый логгер на пути может обработать это сообщение (если у него есть свои аппендеры и уровень позволяет), вызывая прикрепленный аппендер

### 2.1.2 Уровни логирования (Levels)
Уровни определяют важность (severity) сообщения и позволяют гибко фильтровать вывод.

_Таблица: Стандартные уровни (ОТ НИЗШЕГО к высшему)_

| Уровень   | Назначение                                                                                                                                                                                        |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **TRACE** | Самая детальная информация. Используется для пошаговой трассировки (например, вход/выход из методов, значения переменных в цикле).                                                                |
| **DEBUG** | **Отладочная информация для разработчиков**. Детали работы программы (параметры запросов, ветвления логики).                                                                                      |
| **INFO**  | Важные, но **штатные события жизненного цикла** приложения (сервер стартовал, пользователь залогинился, файл обработан).                                                                          |
| **WARN**  | **Потенциально опасная ситуация**, которая не приводит к ошибке, но требует внимания (близок лимит памяти, используется deprecated-метод).                                                        |
| **ERROR** | Событие ошибки, которое нарушило работу операции, но не остановило приложение (ошибка парсинга файла, сбой в одной из бизнес-операций).                                                           |
| **FATAL** | Критическая ошибка, приводящая к аварийному завершению приложения. Встречается редко (в некоторых библиотеках, например Log4j2, отдельного уровня нет, это аналог `ERROR` с наивысшей важностью). |

> **Принцип срабатывания:** Если логгеру установлен уровень `INFO`, он будет пропускать сообщения с уровнем `INFO` и выше (`WARN`, `ERROR`), но игнорировать `DEBUG` и `TRACE`.
> 
> ==ЛОГГЕР ИГНОРИРУЕТ СООБЩЕНИЯ БОЛЕЕ НИЗКОГО УРОВНЯ ЧЕМ ОН САМ==

## 2.2 Аппендер (Appender)

> [!IMPORTANT] Аппендер - ЗАПИСЫВАЕТ ЛОГ КУДА НУЖНО
> Аппендер — это компонент, который отвечает за **конечную доставку** лог-сообщения на некий носитель ("куда писать"). Аппендеры прикрепляются к логгерам.

### 2.2.1 Популярные типы аппендеров
- **ConsoleAppender:** Вывод в `System.out` или `System.err`.
    
- **FileAppender:** Запись в файл. Может поддерживать политику ротации (например, ежедневно или по размеру).
    
- **RollingFileAppender:** Разновидность `FileAppender`, которая создает архивные файлы по расписанию или при достижении лимита размера.
    
- **JDBCAppender:** Запись логов напрямую в таблицу базы данных.
    
- **SMTPAppender:** Отправка логов (обычно уровня `ERROR` и выше) на электронную почту.
    
- **SocketAppender:** Передача логов по сети на сервер сбора логов.

### 2.2.2 Фильтрация и отказоустойчивость (на примере Log4j2)
- **Failover Appender:** Гарантирует доставку. 
  Логи пишутся в основной аппендер (например, в БД). 
  Если он падает с ошибкой, автоматически включается резервный (например, запись в локальный файл).
    
- **Async Appender:** Не блокирует основной поток приложения. Сообщение кладется в очередь в памяти, а отдельный поток асинхронно разбирает эту очередь и записывает в целевой аппендер. Это кардинально повышает производительность.
    
- **Фильтры (Filters):** Могут быть настроены глобально или на конкретный аппендер, чтобы отсеивать сообщения по более сложной логике, чем просто уровень (например, по содержимому MDC или маркеру).

## 2.3 Лейаут (Layout)

> [!NOTE] Лейаут - ФОРМАТ лога
> Лейаут — это компонент внутри аппендера, который отвечает за **форматирование** лог-сообщения ("в каком виде писать"). Преобразует объект лог-события в строку.

**Примеры форматов:**
- **PatternLayout (Самый гибкий):** Формат задается строкой-шаблоном.
```java
[%d{yyyy-MM-dd HH:mm:ss}] [%level] [%thread] %logger{36} - %msg%n
```
_Шаблон выведет: `[2026-05-31 10:15:32] [INFO] [main] com.example.App - Приложение запущено`_

# 3. Архитектурные подходы: Конкретная реализация vs. Фасад

Привязываться к конкретному логгеру - ПЛОХО. Возможно нужно будет быстро перейти на какой-то другой.

Поэтому, используют ФАСАДЫ, которые уже вызывают конкретный логгер

## 3.1 Конкретные реализации (Библиотеки-движки)
Это библиотеки, которые непосредственно выполняют всю работу по логированию.

- **`java.util.logging` (JUL):**
    - Встроен в JDK, не требует внешних зависимостей.
        
    - Считается медленным в стандартной конфигурации из-за внутренних блокировок. Производительность можно значительно улучшить, заменив стандартный `LogManager` на, например, `JBoss Log Manager`.
        
- **Log4j 1.x:**
    - **Статус: устарел (End-of-Life).** Не рекомендуется к использованию. На смену пришел Log4j 2.
        
- **Log4j 2:**
    - Полностью переписанная версия. Высокопроизводительный, асинхронный, отказоустойчивый. Устраняет проблемы предшественника.
        
    - _Конфигурация:_ `.xml` (предпочтительно), `.json`, `.yaml`.
        
- **Logback:**
    - «Духовный наследник» Log4j 1.x от того же автора (Ceki Gülcü). Очень быстрый, является «родным» движком для SLF4J. Отличная документация и поддержка.
        
    - _Конфигурация:_ `.xml` или `logback.groovy`.
        

---
## 3.2 Фасады (Абстракции)

Фасады предоставляют универсальный API для вызова логов в коде. 

Ваш код зависит только от фасада, а конкретную библиотеку-движок вы подключаете отдельно и можете заменить в любой момент без перекомпиляции кода.

- **SLF4J (Simple Logging Facade for Java):**
    - **Де-факто стандарт.** Позволяет в рантайме использовать любой совместимый движок (Logback, Log4j2, JUL и др.).
        
    - _Пример кода с использованием SLF4J:_
```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;


public class UserService {
// Фасад SLF4J
	private static final Logger log = 
		LoggerFactory.getLogger(UserService.class);
	
	public void processPayment(String orderId) {
		log.info("Начало обработки платежа для заказа: {}", orderId);
		// ... бизнес-логика ...
	}
}
```

- **JBoss Logging:**
    - Используется внутри экосистемы WildFly/JBoss, но также может служить фасадом для сторонних приложений. 
      
      Не является прямым конкурентом SLF4J в мире Spring/Standalone-приложений.
        

> **Главная рекомендация из лекции:** 
> Всегда используйте **ФАСАД + Конкретный логгер**. 
> 
> В коде пишем вызовы SLF4J, а в зависимостях подключаем Logback или Log4j2.

# 4. Конфигурирование логирования
Настройка системы логирования почти всегда выносится в конфигурационный файл, что позволяет менять поведение без пересборки приложения.

> [!WARNIGN] ПОИСК ФАЙЛА КОНФИГУРАЦИИ
> Обычно ФАЙЛ КОНФИГУРАЦИИ ИЩЕТСЯ ИЗ classpath (в папке resource к примеру)
> 
> Если ничего не найдется, то используется базовая конфигурация (все в консоль + DEBUG)


## 4.1 Связка: SLF4J (фасад) + Logback (реализация)
Это самая популярная и рекомендуемая связка "из коробки" для Spring Boot и большинства современных проектов. SLF4J и Logback созданы одним автором и работают без дополнительных "мостов" (bridges).

### Необходимые библиотеки (Maven)
Вам достаточно подключить одну зависимость `logback-classic`, она транзитивно подтянет `logback-core` и `slf4j-api`.
```xml
<!-- pom.xml -->
<dependencies>
    <dependency>
        <groupId>ch.qos.logback</groupId>
        <artifactId>logback-classic</artifactId>
        <version>1.4.14</version>
    </dependency>
</dependencies>
```
_Состав транзитивных зависимостей: 
`logback-classic` -> `logback-core` + `slf4j-api`._

---
### Конфигурация (`logback.xml`)
Разместите файл в корне classpath (обычно `src/main/resources`).
```xml
<!-- src/main/resources/logback.xml -->
<configuration>
    <!-- Аппендер для вывода В КОНСОЛЬ -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>
	            %d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n
            </pattern>
        </encoder>
    </appender>

    <!-- Аппендер для записи в файл с ежедневной ротацией -->
    <appender name="FILE" 
	class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/myapp.log</file>
        
        <rollingPolicy 
        class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>
	            logs/myapp.%d{yyyy-MM-dd}.log.zip
	        </fileNamePattern>
	        
            <maxHistory>30</maxHistory>
        
        </rollingPolicy>
        
        <encoder>
            <pattern>
	            %d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level 
	            %logger{36} - %msg%n
	        </pattern>
        </encoder>
    </appender>

	<appender name="FILE_ERRORS" class="ch.qos.logback.core.FileAppender">
        <file>logs/errors.log</file>
        <encoder>
            <pattern>
	            %d{yyyy-MM-dd HH:mm:ss} %-5level %logger{36} - %msg%n
	        </pattern>
        </encoder>
    </appender>
    
    
    <!-- Корневой логгер -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/> <!-- ИМЯ АППЕНДЕРА -->
        <appender-ref ref="FILE"/> <!-- ИМЯ АППЕНДЕРА -->
    </root>


    <!-- Специфичные настройки для пакетов -->
    
    <!-- Логгер для сервисов: дополнительно пишет DEBUG в общий файл -->
    <logger name="com.example.service" level="DEBUG">
        <appender-ref ref="FILE_ALL" />
    </logger>
    
    <!-- Логгер для ошибок безопасности: пишет ТОЛЬКО в файл ошибок -->
    <logger name="com.example.security" level="WARN" additivity="false">
        <appender-ref ref="FILE_ERRORS" />
        <appender-ref ref="CONSOLE" />
    </logger>
    
</configuration>
```

---
### Код использования
Код всегда ссылается только на классы фасада **SLF4J**.
```java
import org.slf4j.Logger; // ФАСАД
import org.slf4j.LoggerFactory; // РЕАЛИЗАЦИЯ ИЗ SLF4J

public class PaymentService {
    // Используем фасад SLF4J
    private static final Logger log = 
	    LoggerFactory.getLogger(PaymentService.class);

    public void processPayment(String orderId, double amount) {
        log.info(
	        "Инициирован процессинг платежа: orderId={}, amount={}", 
	        orderId, 
	        amount
	    );
        try {
            // Бизнес-логика...
            log.debug("Платеж успешно обработан на шлюзе");
        } catch (Exception e) {
            log.error(
	            "Критический сбой при обработке платежа orderId={}", 
	            orderId, 
	            e
	        );
        }
    }
}
```

## 4.2 Связка: SLF4J (фасад) + Log4j 2 (реализация)
Log4j 2 — мощная и асинхронная реализация, которая подключается через специальный мост-адаптер.

### Необходимые библиотеки (Maven)
Здесь нужно **три зависимости**: 
- API фасада, 
- ядро Log4j2 
- и мост между ними.
```xml
<!-- pom.xml -->
<dependencies>
    <!-- 1. API фасада SLF4J -->
    <dependency>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-api</artifactId>
        <version>2.0.9</version>
    </dependency>

    <!-- 2. Ядро и реализация Log4j 2 -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
        <version>2.22.1</version>
    </dependency>
    
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-slf4j2-impl</artifactId>
        <version>2.22.1</version> <!-- Мост: SLF4J -> Log4j2 -->
    </dependency>
    
</dependencies>
```

---
### Конфигурация (`log4j2.xml`)
Файл размещается в `src/main/resources`.
```xml
<!-- src/main/resources/log4j2.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">
    <Appenders>
        <!-- Консольный аппендер -->
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n"/>
        </Console>

        <!-- Файловый аппендер с политикой ротации по размеру -->
        <RollingFile name="RollingFile" fileName="logs/log4j2-app.log"
                     filePattern="logs/log4j2-app-%d{yyyy-MM-dd}-%i.log.gz">
            <PatternLayout>
                <pattern>
	                %d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger{36} 
	                - %msg%n
	            </pattern>
            </PatternLayout>
            
            <Policies>
                <TimeBasedTriggeringPolicy interval="1" modulate="true"/>
                <SizeBasedTriggeringPolicy size="10 MB"/>
            </Policies>
            
        </RollingFile>

        <!-- Пример асинхронного аппендера для производительности -->
        <Async name="AsyncFile" bufferSize="1024">
            <AppenderRef ref="RollingFile"/>
        </Async>
    </Appenders>


    <Loggers>
        <Root level="info">
            <AppenderRef ref="Console"/> <!-- ИМЯ АППЕНДЕРА -->
            <AppenderRef ref="AsyncFile"/> <!-- ИМЯ АППЕНДЕРА -->
        </Root>

        <Logger name="com.example.myapp.dao" level="trace" additivity="false">
            <AppenderRef ref="Console"/> <!-- ИМЯ АППЕНДЕРА -->
        </Logger>
    </Loggers>
</Configuration>
```

---
### Код использования
Код **абсолютно идентичен** примеру с Logback, так как мы используем фасад SLF4J.
```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class DataRepository {
    private static final 
	    Logger log = LoggerFactory.getLogger(DataRepository.class);

    public void save(String entity) {
        log.debug("Сохранение сущности: {}", entity);
        // логика сохранения
    }
}
```

**Важное замечание:** Если вы планируете использовать асинхронное логирование Log4j2 на полную мощность (с библиотекой LMAX Disruptor), добавьте зависимость:
```xml
<dependency>
    <groupId>com.lmax</groupId>
    <artifactId>disruptor</artifactId>
    <version>3.4.4</version>
</dependency>
```

## 4.3 Использование «чистого» Log4j 2 (БЕЗ фасада SLF4J)
Этот подход **не рекомендуется** для крупных проектов, так как жестко привязывает ваш код к конкретной реализации. 

Однако для маленьких утилит имеет право на жизнь.

### Необходимые библиотеки (Maven)
Достаточно двух зависимостей: 
- API Log4j2 
- и его ядро.
```xml
<!-- pom.xml -->
<dependencies>
    <!-- 1. API Log4j 2 (это не фасад, а проприетарный API) -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-api</artifactId>
        <version>2.22.1</version>
    </dependency>
    <!-- 2. Ядро реализации -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
        <version>2.22.1</version>
    </dependency>
</dependencies>
```

### Конфигурация (`log4j2.xml`)
Конфигурация будет точно такой же, как в примере 4.2 (файл `log4j2.xml`). 

Мы используем тот же движок, разница только в том, какие классы мы вызываем в Java-коде.

---
### Код использования
Здесь уже используются нативные классы Log4j 2 (`LogManager` и `Logger` из пакета `org.apache.logging.log4j`).

```java
// ВНИМАНИЕ: Импорты идут из проприетарного API Log4j2
import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;


public class LegacyUtility {
    // Используем нативный API Log4j2
    private static final Logger log = 
	    LogManager.getLogger(LegacyUtility.class);

    public void execute() {
        log.info("Выполнение утилиты начато");

        if (log.isDebugEnabled()) {
            log.debug(
	            "Вызов дорогостоящего toString для объекта: {}", 
	            () -> createExpensiveDump()
	        );
	        // Так как конкатенация строк ТЯЖЕЛАЯ ОПЕРАЦИЯ, особенно когда часто пишутся логи (для серверных приложения)
        }

        try {
            // полезная работа
        } catch (Exception e) {
            // В Log4j2 нативное API поддерживает лямбда-выражения для ленивых вычислений
            log.error("Ошибка выполнения", e);
        }
    }

    private String createExpensiveDump() {
        return "Очень тяжелая строка";
    }
}
```

# 5. Лучшие практики (Best Practices)

Резюмируем ключевые рекомендации из лекции и общепринятые правила.

1. **Логируйте внешние интерфейсы:** Всегда фиксируйте входящие и исходящие данные на границах системы (REST-контроллеры, слои сервисов для внешних запросов, чтение из очередей).
    
2. **Логируйте исключения:** Передавайте объект исключения в логгер, чтобы сохранить полный стектрейс. Указывайте контекстные аргументы.
```java
try {
    paymentGateway.charge(amount);
} catch (PaymentException e) {
    // Хорошо: есть и контекст, и стектрейс
    log.error(
	    "Ошибка списания средств для заказа ID: {}. Сумма: {}", 
	    orderId, 
	    amount, 
	    e
    );
}
```

3. **Используйте параметризованные сообщения:** Никогда не используйте конкатенацию строк в логах.
```java
// ПЛОХО:
// конкатенация происходит ДО проверки уровня, тратит ресурсы впустую
log.debug("Пользователь " + user + " добавил товар " + item + " в корзину");

// ХОРОШО: 
// сообщение будет собрано, только если уровень DEBUG действительно включен
log.debug("Пользователь {} добавил товар {} в корзину", user, item);
```
