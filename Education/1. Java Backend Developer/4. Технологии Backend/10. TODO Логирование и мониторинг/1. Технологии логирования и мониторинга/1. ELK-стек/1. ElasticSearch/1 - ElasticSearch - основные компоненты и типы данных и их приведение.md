> [!IMPORTNANT] ElasticSearch - поисковой движок
> `Elasticsearch` – это **распределенный поисковый и аналитический движок**, основанный на открытом исходном коде. 
> 
> Он предназначен **для быстрого и гибкого поиска и анализа данных** в реальном времени.
> 
> ElasticSearch - ***это про НЕРЕЛЯЦИОНКИ (NoSQL), если говорить про способ хранения данных***
> 
> Хранит данные в виде JSON

 `Elasticsearch` широко используется для различных задач, таких как 
 - поиск текста, 
 - мониторинг логов, 
 - аналитика приложений, 
 - обработка метрик 
 - и многие другие сценарии. 
 
 Он основан на `Apache Lucene` и предоставляет RESTful API для взаимодействия с данными, что делает его легко интегрируемым и расширяемым для различных приложений и систем.

### Полнотекстовый поиск - плевать на падежи, найду по кусочку
**Полнотекстовый поиск** — это технология, которая ***позволяет находить документы***, страницы или записи *по содержанию их текста, а не только по метаданным, заголовкам или ключевым словам.* 

В отличие от простого поиска по точному совпадению, ***полнотекстовый механизм анализирует весь объём текста***, **учитывает грамматические формы, синонимы, контекст и смысловые оттенки.**

Разница
- *Простой поиск* - ищет ТОЧНОЕ СОВПАДЕНИЕ
- *Полнотекстовый поиск* - ищет по фрагменту строки, необязательно по точному совпадению

## 1.1 Узел (Node) - единичный инстанс ElasticSearch
Относится к **одному работающему экземпляру Elasticsearch**. 

На одном физическом или виртуальном сервере может размещаться несколько узлов в зависимости от возможностей его физических ресурсов, таких как ОЗУ, дисковое пространство и вычислительная мощность.

В этом узле, и хранятся индексы

---
## 1.2 Кластер - объединение нод
Это набор из одного или более узлов. Кластер обеспечивает совместные возможности индексации и поиска по всем узлам для всех данных.

## 1.3 Документ (Document) - аналог строки данных из РСУБД
> [!IMPORTANT] Документ - аналог ЗАПИСИ в реляционных СУБД
> 

Это набор полей сущности, определённых в формате JSON. 

Сущность сыр, **определим документ**
```json
{
	"id":"XYZ123",
	"sort": "Масдам",
	"manufacturer": "Молокозавод 1",
	"price": 900
}
```

Каждый документ принадлежит какому-либо типу (сущности) и находится внутри индекса. 

> [!IMPORTANT] ВСТАВКА Документа в Индекс
> Процесс вставки документа в индекс - **ИНДЕКСИРОВАНИЕ**

Каждый документ связан с уникальным идентификатором, называемым UID.

> [!IMPORTANT] Индекс - аналог ТАБЛИЦЫ в реляционных СУБД

Это **коллекция документов** разных типов и их свойств.

Индекс также использует концепцию шардов для повышения производительности. 

Например, набор документов, содержащих данные какого-либо приложения для социальных сетей.

## 1.5 Шард (Shard)
**Индексы горизонтально подразделяются на шарды.** 
> (Части индексов хранятся по нескольким нодам)

Благодаря шардам, **мы эффективно используем ресурсы каждой ноды, а не нагружаем только одну**

> [!WARNING] По умолчанию - количество шардов 5
> После создания индекса, создается 5 шардов. 
> 
> То есть, наша коллекция документов (JSON), делится на 5 частей
> 
> **После создания индекса**, ***ИЗМЕНИТЬ КОЛ-ВО ШАРДОВ НЕЛЬЯ***


Это означает, что каждый шард содержит все свойства документа, но включает меньшее количество JSON-объектов, чем индекс. 

Горизонтальное разделение делает шард независимым узлом, который может храниться на любом узле. 

Первичный шард — это исходная горизонтальная часть индекса, и затем эти первичные шарды реплицируются в шарды-реплики.

---
## 1.6 Реплика (Replicas)
Elasticsearch позволяет создавать реплики индексов и шардов. 

Репликация не только **помогает повысить доступность данных в случае сбоя**, ***но и улучшает производительность поиска за счёт параллельного выполнения поисковых операций на этих репликах.***

## 1.7 Анализатор

> [!IMPORTANT] Анализатор - преобразует текст в текст, пригодный для полнотекстового поиска
> **Анализатор (Analyzer)** в Elasticsearch — это специальный механизм, который ==превращает текст в формат, пригодный для полнотекстового поиска==

Эта трансформация называется **анализом (Analysis)**.

### 1.7.0 Состав Анализатора
Любой анализатор — это набор из трех строительных блоков, применяемых последовательно:
#### Компонент 1. **Символьные фильтры (Character Filters)** (0 или более)
- **Что делают:** Получают ==исходный текст и очищают его== от «мусора» до того, как он будет разбит на слова. Они могут удалять, заменять или добавлять символы
	
- **Пример:** Удалить HTML-теги (`<b>текст</b>` → `текст`) или заменить `&` на слово `and

#### Компонент 2. **Токенайзер (Tokenizer)** (ровно 1)
- **Что делает:** Это сердце анализатора. 
  
  Он принимает очищенный ==текст и разбивает его на токены(Terms, термы) ==— обычно на отдельные слова
	
- **Пример:** `whitespace` токенайзер разобьет текст `"Быстрая коричневая лиса"` на токены `[Быстрая, коричневая, лиса]` при каждой встрече с пробелом

#### Компонент 3. **Фильтры токенов (Token Filters)** (0 или более)
- **Что делают:** Получают готовый ==поток токенов и приводят их к единому (нормализованному) виду==. 
  
  Они также могут удалять или добавлять токены
	
- **Примеры:**
	- `lowercase` фильтр переведет все токены в нижний регистр (`Быстрая` → `быстрая`)
		
	- `stop` фильтр удалит «слова-паразиты» (стоп-слова), вроде `и`, `в`, `the`, которые не несут смысловой нагрузки
		
	- `stemmer` фильтр выполнит **стемминг** — приведет слово к его корневой форме (`лисы` → `лис`, `бежал` → `беж`)


**Визуально процесс можно представить так:**
```
`Исходный текст` → (Символьные фильтры) → `Очищенный текст` → (Токенайзер) → `Поток токенов` → (Фильтры токенов) → `Финальные токены для индекса`
```

### 1.7.1 Виды анализаторов
- **Встроенные анализаторы (Built-in Analyzers)**: 
  Это готовые решения «под ключ» для большинства типовых задач:
    - `standard`: Анализатор **по умолчанию**. Хорош для большинства языков, разбивает текст на слова, удаляет знаки пунктуации и приводит всё к нижнему регистру
        
    - `simple`: Разбивает текст по всему, что не является буквой, и приводит к нижнему регистру
        
    - `whitespace`: Просто разбивает текст по пробелам, **не** приводя к нижнему регистру
        
    - `english` / `russian`: Специализированные **анализаторы для конкретных языков.** Они понимают особенности языка (например, английские окончания `-ing`, `-ed` или стоп-слова вроде `the`/`a`)
    
- **Пользовательские анализаторы (Custom Analyzers)**: Если встроенных недостаточно, вы можете создать свой собственный, скомбинировав из доступных строительных блоков нужные символьные фильтры, один токенайзер и несколько фильтров токенов

![[Pasted image 20260603155757.png]]

Благодаря этому индексу, мы можем по слову находить документ, где встречается это слово

> [!WARNING] ПОЛЯ ПОПАДАЮЩИЕ В ИНДЕКС - **МАППИНГ**, и УКАЗАНИЕ ТИПА ПОЛЕЙ ДОКУМЕНТА
> По умолчанию, в этот индекс попадают ВСЕ ПОЛЯ ДОКУМЕНТА, что может быть не оптимально

### ЗАПРЕТ ИНДЕКСИРОВАНИЯ ПОЛЯ ДОКУМЕНТА
Чтобы запретить полю документа быть в обратном индексе, ПРИ УКАЗАНИИ МАППИНГА используется ключевое слово `"index": false`

Создадим индекс
```json
PUT first_index
{
    "mappings": {
        "properties":{
            "title":{
                "type": "text",
                "analyzer":"russian"
            },
            "price": {
                "type":"float"
            },
            "available":{
                "type": "boolean"
            },
            "non_index_field":{
                "type": "text",
                "index": false
            }
        }
    }
}
```

Объявили документ, с полями: заголовок, цена, доступность, и non_index_field - поле, которое не будет индексироваться

Вывод:
```json
{
  "acknowledged": true,
  "shards_acknowledged": true,
  "index": "first_index"
}
```

Посмотрим маппинг
```HTTP
GET first_index/_mapping
```

```json
{
  "first_index": {
    "mappings": {
      "properties": {
        "available": {
          "type": "boolean"
        },
        "non_index_field": {
          "type": "text",
          "index": false
        },
        "price": {
          "type": "float"
        },
        "title": {
          "type": "text",
          "analyzer": "russian"
        }
      }
    }
  }
}
```

Как видно, ПОЛЕ ПОПАЛО В МАППИНГ, но это не значит что оно попало в индекс. 

МАППИНГ просто показывает какие ограничения есть у поля. И мы видим, что у поля документа `non_index_field` есть ограничение на индексирование в обратном индексе

Сейчас проверим это. Добавим документ, и попробуем сделать поиск по этом "не индексируемому" полю
```json

{
```

## 2.1 Приведение типов
При построении ОБРАТНОГО ИНДЕКСА, **ElasticSearch может приводить один тип данных к другому, если это возможно**

> [!WARNING] Когда начинает работать приведение типов
> Приведение типов происходит, ТОЛЬКО ПОСЛЕ ТОГО КАК ОБРАТНЫЙ ИНДЕКС УЖЕ ПОСТРОЕН


Для отключения приведения типов, в маппинге указывается параметр 
```json
"coerce": false
```

## 🔑 Ключевые моменты

### 1. `text` vs `keyword` — это самая важная разница[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://docs-v3-preview.elastic.dev/elastic/elasticsearch/tree/main/reference/query-languages/sql/sql-data-types#sql-multi-field)

|                            | `text`                                                                                       | `keyword`                                                                                                        |
| -------------------------- | -------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **Что делает**             | Анализирует текст (==разбивает на токены==, приводит к нижнему регистру, удаляет стоп-слова) | ==Оставляет== строку как есть, ==целиком==                                                                       |
| **Когда использовать**     | Полнотекстовый поиск («найди документы, содержащие слово "быстрая"»)                         | ПОИСКС ПО ТОЧНОМУ СОВПАДЕНИЮ,<br>Фильтрация, сортировка, агрегации («покажи все документы со статусом "active"») |
| **Пример значения**        | `"The quick brown fox"` <br>→ <br>`["the", "quick", "brown", "fox"]`                         | `"The quick brown fox"` <br>→ <br>`"The quick brown fox"`                                                        |
| **Можно ли агрегировать?** | ❌ Нет                                                                                        | ✅ Да                                                                                                             |
| **Можно ли сортировать?**  | ❌ Нет                                                                                        | ✅ Да                                                                                                             |

**Лучшая практика**: для строковых полей часто создают **multi-field** 
Одно поле: 
- и как `text`, 
- и как `keyword`:

```json
{
  "message": {
    "type": "text",
    "fields": {
      "keyword": {
        "type": "keyword"
      }
    }
  }
}
```

Тогда можно искать по `message`, а сортировать/фильтровать по `message.keyword`[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://docs-v3-preview.elastic.dev/elastic/elasticsearch/tree/main/reference/query-languages/sql/sql-data-types#sql-multi-field).

### 2. Числовые типы — выбирайте минимально достаточный

|Тип|Диапазон|Когда использовать|
|---|---|---|
|`byte`|-128 до 127|Маленькие целые|
|`short`|-32,768 до 32,767|Немного больше|
|`integer`|-2³¹ до 2³¹-1|Стандартные целые|
|`long`|-2⁶³ до 2⁶³-1|Большие числа (timestamp, большие ID)|
|`float` / `double`|Числа с плавающей точкой|Дробные значения|
|`half_float`|Меньшая точность, экономия памяти|Для float, где высокая точность не нужна|
|`scaled_float`|Масштабируемый float (хранит как long с множителем)|Цены, проценты — там, где нужна фиксированная точность|

Elasticsearch использует **меньше памяти и быстрее работает** с более компактными типами[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423).

### 3. Работа с датами и временем

| Тип          | Точность     | Пример формата                         |
| ------------ | ------------ | -------------------------------------- |
| `date`       | Миллисекунды | `2026-06-03T10:15:30.123Z` (в конце Z) |
| `date_nanos` | Наносекунды  | `2026-06-03T10:15:30.123456789Z`       |
Миллисекунды не обязательно использовать, можно просто дату: `2026-06-03`

Elasticsearch принимает даты в трёх форматах:
- Строки ISO 8601: `"2026-06-03T10:15:30Z"`
    
- Unix timestamp в миллисекундах: `1748931330123`
    
- Unix timestamp в секундах (с суффиксом `s`): `1748931330s`

> [!WARNIGN] ElasticSearch приводит тип `date` в `long`
> При вставке Эластик приводит все типа `date` к типу `long` и хранит именно в нем


### 4. Массивы — особого типа нет

В Elasticsearch **нет отдельного типа для массивов**. 

Любое поле может содержать ноль или более значений, но все значения должны быть **одного типа**[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://docs-v3-preview.elastic.dev/elastic/elasticsearch/tree/main/reference/elasticsearch/mapping-reference/field-data-types.md):

```json
{
  "tags": ["elasticsearch", "database", "search"]   // массив строк
}
```

### 5. Вложенные объекты: `object` vs `nested`

|                        |`object` (по умолчанию)|`nested`|
|---|---|---|
| **Как хранится**       |Как плоский список пар ключ-значение|Как отдельные скрытые документы, сохраняя связи между полями|
| **Проблема**           |Потеря связи между полями внутри массива объектов («кросс-объектные совпадения»)|—|
| **Когда использовать** |Простые объекты, одиночные вложения|Массивы объектов, где важны связи между полями|
| **Цена**               |Дешевле|Дороже (больше памяти и медленнее индексация)|

**Пример проблемы `object`**:  
Документ: 
```json
employees: [ 
	{name: "John", age: 30}, 
	{name: "Jane", age: 25} 
]
```  
Запрос: `employees.name = John AND employees.age = 25` — найдёт документ, хотя John и 25 не в одном объекте.

`nested` решает эту проблему, но требует специального синтаксиса запросов и агрегаций[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://www.elastic.co/guide/en/elasticsearch/reference/8.14/properties.html).

### 6. JSON-объекты без детального маппинга: `flattened`

Если вы не знаете заранее структуру вложенного JSON или она очень разнородная, используйте `flattened` — весь объект индексируется как одно поле, к которому можно применять запросы без объявления каждого подполя[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423).

### 7. Диапазоны (`range`)

Позволяют хранить интервалы значений и искать по вхождению в интервал:

```json
{
  "price_range": {
    "type": "integer_range"
  }
}
// значение: { "gte": 10, "lte": 100 }
```

Подтипы: `integer_range`, `float_range`, `long_range`, `double_range`, `date_range`, `ip_range`[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423).

### 8. Геоданные

|Тип|Что хранит|Зачем|
|---|---|---|
|`geo_point`|Координаты `{lat, lon}`|Поиск точек в радиусе, внутри bounding box, расстояние между точками|
|`geo_shape`|Полигоны, линии, мультиполигоны|«Найди все точки внутри района Москвы»|
|`point`|Декартовы координаты `{x, y}`|Для систем без географической привязки|
|`shape`|Декартовы геометрии|Для двухмерных плоскостей (карты зданий, чертежи)|

### 9. IP-адреса (`ip`)

Поддерживает IPv4 и IPv6. Можно искать точные адреса, диапазоны (`192.168.0.0/16`) и сети[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://raw.githubusercontent.com/spinscale/elasticsearch/e7f4f486ca2be6794a70b1c2d8cae3e678425968/docs/reference/mapping/types.asciidoc).

### 10. Вектора для ML и семантического поиска

|Тип|Для чего|Когда использовать|
|---|---|---|
|`dense_vector`|Плотные вектора (почти все размерности ненулевые)|У вас есть внешние модели (OpenAI, Hugging Face) и нужен полный контроль|
|`sparse_vector`|Разреженные вектора (много нулевых значений, значимые токены с весами)|ELSER, SPLADE, контроль влияния конкретных терминов|
|`semantic_text`|**Упрощённый семантический поиск** (автоматически генерирует вектора через inference endpoint)|Хотите быстро начать, не углубляясь в детали — Elasticsearch сам создаёт и хранит вектора[](https://www.elastic.co/search-labs/kr/blog/mapping-embeddings-to-elasticsearch-field-types)[](https://www.elastic.co/search-labs/blog/mapping-embeddings-to-elasticsearch-field-types)|

`semantic_text` — это надстройка над векторами. При индексации текст отправляется в модель, результат автоматически сохраняется. При запросе — то же самое. Никакой ручной работы[](https://www.elastic.co/search-labs/blog/mapping-embeddings-to-elasticsearch-field-types).

### 11. Специальные типы

|Тип|Назначение|
|---|---|

|`search_as_you_type`|Тоже автодополнение, но другой механизм[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)|
|`percolator`|Хранит **запросы** (не документы). Можно проверить, под какой заранее сохранённый запрос подпадает новый документ[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)|

### 12. Multi-fields

==Одна из самых мощных концепций:== одно и то же значение можно индексировать **несколькими разными способами** в одном поле[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://docs-v3-preview.elastic.dev/elastic/elasticsearch/tree/main/reference/query-languages/sql/sql-data-types#sql-multi-field).

**Типичный пример** — поле `city`:

```json
"city": {
  "type": "text",
  "fields": {
    "raw": { 
	    "type": "keyword" 
	}, // для фильтрации и сортировки
	
    "english": { 
	    "type": "text", 
	    "analyzer": "english" 
	}  // для англоязычного поиска
  }
}
```

> [!NOTE] Нейнимнг полей (fields)
> Может быть любым, **НЕ обязательно** `raw`

Теперь можно:
- Искать по `city` (стандартный полнотекст)
    
- Фильтровать/сортировать по `city.raw` (точное совпадение)
    
- Искать со стеммингом по `city.english`

## 💡 Практические рекомендации
1. **Всегда продумывайте маппинг заранее**. Dynamic mapping удобен для прототипов, но в продакшене лучше контролировать типы — иначе получите `text` поле там, где нужен `keyword`[](https://www.yiiframework.com/extension/yiisoft/yii2-elasticsearch/doc/guide/2.1/ru/mapping-indexing).
    
2. **Для строк используйте multi-field** `text` + `keyword` — если не уверены, нужно ли вам и то и другое[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423)[](https://docs-v3-preview.elastic.dev/elastic/elasticsearch/tree/main/reference/query-languages/sql/sql-data-types#sql-multi-field).
    
3. **Числа: берите минимальный тип**, который покрывает ваши значения. Огромные `long` для маленьких счётчиков — пустая трата памяти и снижение производительности.
    
4. **Массивы объектов** — если нужны связи между полями внутри одного элемента массива, берите `nested`. Если нет — хватит `object`.
    
5. **Для семантического поиска** начинайте с `semantic_text` — это самый простой путь. Когда упрётесь в лимиты, переходите на `dense_vector` с контролем[](https://www.elastic.co/search-labs/kr/blog/mapping-embeddings-to-elasticsearch-field-types)[](https://www.elastic.co/search-labs/blog/mapping-embeddings-to-elasticsearch-field-types).
    
6. **Тип поля нельзя изменить после создания индекса** (если только это не добавление нового `multi-field` или некоторые динамические настройки). Для серьёзных изменений нужен `_reindex` в новый индекс[](https://www.yiiframework.com/extension/yiisoft/yii2-elasticsearch/doc/guide/2.1/ru/mapping-indexing)[](https://www.elastic.co/guide/en/elasticsearch/reference/1.7/fielddata-formats.html).
    
7. **`wildcard` тип** — альтернатива `keyword` для полей с очень длинными значениями (логи, URL, сообщения), но с ограничениями по производительности[](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/field-data-types?from_column=20423&from=20423).
