Все endpoint в Elastic можно разбить на категории, в зависимости от необходимой функциональности

> [!IMPORTANT] ДОКУМЕНТАЦИЯ ПО API (ПОДРОБНОЕ И УДОБНОЕ)
> https://www.elastic.co/docs/api/doc/elasticsearch/group/endpoint-analytics


> [!WARNING] ЛАЙФХАК - через UI Kibana в консоли есть подсказки по параметрам и методам
> В консоли в Kibana, есть автозаполнитель команд и параметров

Эти endpoint разбиваются про префиксу в начале.

| Префикс эндпоинта  | Для чего используются эндпоинты с этим префиксом                                                                                                                     |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `_search`          | Полнотекстовый поиск и агрегация данных. Основной способ извлекать документы и вычислять по ним статистику (группировки, суммы, средние, процентили и т.д.).         |
| `_count`           | Подсчёт количества документов, соответствующих запросу, без возврата самих документов. Быстрее и легче, чем `_search`.                                               |
| `_validate`        | Проверка синтаксиса поискового запроса (Query DSL) без его выполнения. Используется для отладки.                                                                     |
| `_explain`         | Объяснение, почему конкретный документ был найден по запросу и как вычислялся его `_score` (релевантность).                                                          |
| `_field_caps`      | Получение информации о возможностях полей (тип данных, агрегируемость, индексируемость) по всем индексам.                                                            |
| `_render`          | Визуализация и отладка шаблонов поисковых запросов (search templates).                                                                                               |
| `_index`           | Создание или полная замена документа. Обычно используется как путь `/{index}/_doc/{id}` с методами `PUT` или `POST`.                                                 |
| `_update`          | Частичное обновление документа (изменение конкретных полей без перезаписи всего документа).                                                                          |
| `_delete`          | Удаление одного документа по его ID. Обычно `DELETE /{index}/_doc/{id}`.                                                                                             |
| `_bulk`            | Выполнение множества операций (index, update, delete) за один HTTP-запрос. Критически важен для массовой загрузки данных.                                            |
| `_delete_by_query` | Удаление всех документов, соответствующих поисковому запросу. Аналог `DELETE FROM ... WHERE`.                                                                        |
| `_update_by_query` | Обновление всех документов, соответствующих поисковому запросу (например, добавить поле или переиндексировать).                                                      |
| `_reindex`         | Копирование документов из одного индекса в другой. Используется для смены маппинга, анализатора или переименования индекса.                                          |
| `_mapping`         | Управление схемой данных индекса: просмотр и обновление типов полей, анализаторов и настроек индексации для каждого поля.                                            |
| `_settings`        | Просмотр и динамическое изменение настроек индекса или кластера (количество реплик, интервал обновления и т.д.).                                                     |
| `_analyze`         | Тестирование анализаторов: показывает, как конкретный текст будет разбит на токены. Лабораторный инструмент для отладки.                                             |
| `_cat`             | Компактные текстовые отчёты для командной строки. Возвращают табличные данные о кластере, узлах, индексах, шардах. Не предназначены для использования в приложениях. |
| `_cluster`         | Управление кластером: состояние здоровья, статистика, переназначение шардов, информация об узлах.                                                                    |
| `_nodes`           | Получение информации об узлах кластера: их настройки, статистика использования ресурсов (CPU, память, диски), плагины.                                               |
| `_security`        | Управление безопасностью: создание и удаление пользователей (`/user`), ролей (`/role`), связей с внешними провайдерами (`/role_mapping`), API-ключей.                |
| `_watcher`         | Настройка и управление Watcher — системой оповещений на основе данных из Elasticsearch (алертинг).                                                                   |
| `_monitoring`      | Сбор и управление метриками мониторинга самого кластера Elasticsearch.                                                                                               |
| `_ml`              | Управление функциями машинного обучения: создание job-ов для обнаружения аномалий, прогнозирования, классификации.                                                   |
| `_rollover`        | Автоматическое переключение индекса на новый, когда старый достиг определённого размера, возраста или количества документов.                                         |
| `_shrink`          | Уменьшение количества первичных шардов индекса (например, с 5 до 1).                                                                                                 |
| `_split`           | Увеличение количества первичных шардов индекса (например, с 1 до 4).                                                                                                 |
| `_clone`           | Создание точной копии индекса с тем же количеством шардов.                                                                                                           |
| `_ilm`             | Управление Index Lifecycle Management (ILM) — автоматическими политиками жизненного цикла индексов (текущий, горячий, тёплый, холодный, удаление).                   |
| `_snapshot`        | Создание и восстановление резервных копий (снапшотов) индексов и всего кластера.                                                                                     |
| `_migration`       | Помощь в миграции с устаревших версий Elasticsearch на новые. Показывает, что нужно исправить.                                                                       |
| `_licensing`       | Управление лицензией Elasticsearch (базовая, platinum, enterprise).                                                                                                  |
| `_graph`           | API для Graph Explore — анализа связей между документами и выявления скрытых зависимостей.                                                                           |
| `_rank_eval`       | Оценка качества поисковой выдачи (ранжирования) на основе предоставленных релевантных суждений.                                                                      |
| `_termvectors`     | Получение информации о том, как конкретный термин (слово) представлен в инвертированном индексе для конкретного документа.                                           |
| `_scripts`         | Управление хранимыми скриптами (Painless, Groovy и др.) для использования в поиске, агрегациях или обновлениях.                                                      |
| `_enrich`          | Управление политиками обогащения данных (например, добавить к IP-адресу информацию о стране из отдельного индекса).                                                  |
| `_transform`       | Управление трансформациями (преобразованиями данных) — создание сводных индексов из исходных в реальном времени.                                                     |
| `_query`           | Выполнение запросов на ES                                                                                                                                            |

## Популярные перфиксы endpoint'ов
### Уровень 1: Основные API для работы с данными (CRUD и Поиск)
Это самые важные эндпоинты, которые вы будете использовать в 90% случаев. 

Они отвечают за 
- индексацию, 
- поиск 
- и удаление данных.

- **`_search` (Поиск данных)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/the-search-api.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Это главный эндпоинт для поиска документов. Вы передаете ему **Query DSL** (специальный JSON-язык запросов), а он возвращает совпадающие документы (`hits`), их количество и может сразу посчитать **агрегации** (статистику) по ним [](https://www.elastic.co/docs/solutions/search/the-search-api.md).
        
    - **Пример**: Найти все логи уровня `ERROR` за последний час и сгруппировать их по названию сервиса.
    
- **`_index` (Индексация)**
    - **Назначение**: Фактически это не отдельный эндпоинт, а HTTP метод `PUT` или `POST` к конкретному документу (например, `/my_index/_doc/1`). 
      Используется ==для добавления или полной замены документа==.
    
- **`_update` (Обновление)**
    - **Назначение**: ==Частичное обновление документа==. Позволяет изменить значение конкретного поля, не переписывая весь документ целиком.
    
- **`_bulk` (Массовые операции)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Высокопроизводительный эндпоинт ==для выполнения множества операций== (`index`, `update`, `delete`) в **одном** HTTP-запросе. 
      
      Критически важен для загрузки больших объемов данных (например, при миграции или начальной заливке логов).
    
- **`_delete_by_query` (Массовое удаление)**
    - **Назначение**: Удаляет все документы, которые соответствуют условиям поискового запроса. Аналог `DELETE FROM table WHERE ...` в SQL.

### Уровень 2: Поисковые и аналитические инструменты

API, которые делают Elasticsearch таким мощным для полнотекстового поиска и анализа.

- **`_query` (ES|QL запросы)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/explore-analyze/query-filter/languages)
    - **Назначение**: Эндпоинт для новой, более простой и мощной "трубопроводной" (piped) грамматики запросов **ES|QL** (Elasticsearch Query Language). 
      
      Позволяет фильтровать, преобразовывать и анализировать данные в одном запросе, похожем на синтаксис `... | where ... | stats ...` [](https://www.elastic.co/docs/explore-analyze/query-filter/languages).
    
- **`_count` (Подсчет документов)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Возвращает **только количество** документов, соответствующих запросу. 
      
      Это намного быстрее и легче, чем выполнять полный `_search`, если вам не нужны сами документы.
    
- **`_validate/query` (Валидация запросов)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Проверяет синтаксис вашего поискового запроса, не выполняя его. Незаменимо для отладки сложных Query DSL конструкций.
    
- **`_explain` (Объяснение релевантности)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Показывает детальный разбор того, **почему** конкретный документ был найден по вашему запросу и как именно считался его `_score` (релевантность). Важнейший инструмент для тонкой настройки поиска.

### Уровень 3: Администрирование и безопасность
Эти API нужны для управления кластером, настройки прав доступа и контроля над инфраструктурой.

#### `_security` (Управление безопасностью) [](https://docs.rs/elasticsearch/7.11.0-alpha.1/elasticsearch/security/#blanket-implementations)[](https://docs.rs/elasticsearch/7.14.0-alpha.1/elasticsearch/security/struct.Security.html)

Это целое семейство эндпоинтов для настройки аутентификации и авторизации.
- **`/_security/user/<username>`**:
    - **Назначение**: Создание, обновление и удаление пользователей. Вы уже знакомы с этим по предыдущему вопросу о создании пользователя.
        
- **`/_security/role/<rolename>`**:
    - **Назначение**: Создание ролей и определение их прав доступа (к каким индексам можно читать, писать, выполнять поиск и т.д.).
        
- **`/_security/role_mapping`**:
    - **Назначение**: Связывает внешние учетные записи (например, из Active Directory или LDAP) с ролями Elasticsearch.

---
#### `_cat` (Компактные текстовые отчеты) [](https://elastic.sios.jp/blog/step-look-inside-elasticsearch_cat-api-usage-guide/)[](https://raw.githubusercontent.com/jmilktea/jtea/master/elasticsearch/%E5%B8%B8%E7%94%A8api.md)

Эти API созданы для **человека**. Они возвращают данные в удобной табличной форме, идеально подходящей для `curl` и командной строки. **Важно:** Их не стоит использовать в приложениях, так как формат может меняться [](https://elastic.sios.jp/blog/step-look-inside-elasticsearch_cat-api-usage-guide/).

- **`/_cat/health`**:
    - **Назначение**: Быстро посмотреть статус кластера (`green`, `yellow`, `red`), количество узлов и состояние шардов [](https://elastic.sios.jp/blog/step-look-inside-elasticsearch_cat-api-usage-guide/).
        
    - **Лог из вашего вопроса**: Статус `GREEN` в вашем логе означает, что кластер полностью здоров.
        
- **`/_cat/nodes`**:
    - **Назначение**: Получить сводку по всем узлам кластера: использование CPU, памяти (heap), дисковое пространство, роль каждого узла (master, data и т.д.) [](https://elastic.sios.jp/blog/step-look-inside-elasticsearch_cat-api-usage-guide/)[](https://raw.githubusercontent.com/jmilktea/jtea/master/elasticsearch/%E5%B8%B8%E7%94%A8api.md).
        
- **`/_cat/indices`**:
    - **Назначение**: Показать список всех индексов, их размер (`store.size`), количество документов и состояние [](https://elastic.sios.jp/blog/step-look-inside-elasticsearch_cat-api-usage-guide/)[](https://raw.githubusercontent.com/jmilktea/jtea/master/elasticsearch/%E5%B8%B8%E7%94%A8api.md).

### Уровень 4: Специализированные API для продвинутых задач

Эти API решают конкретные инженерные задачи: управление схемой данных, оптимизация производительности и миграция.

- **`_mapping` (Управление схемой данных)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/search-labs/kr/blog/elasticsearch-index-show-fields)
    - **Назначение**: Позволяет посмотреть или обновить **маппинг** (схему) индекса. Маппинг определяет, как хранится и индексируется каждое поле (является ли оно строкой, числом, датой, анализируется ли оно, и т.д.).
        
- **`_reindex` (Переиндексация данных)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Копирует документы из одного индекса в другой. Используется, когда нужно:
        
        - Изменить маппинг поля (например, сделать поле `keyword` вместо `text`).
            
        - Обновить анализатор.
            
        - Переименовать индекс.
            
        - Объединить несколько индексов в один.
            
- **`_analyze` (Тестирование анализаторов)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Показывает, как анализатор разобьет текст на токены. Это **лабораторный инструмент** для настройки анализаторов. Вы передаете ему текст и имя анализатора, а он показывает результат.
        
- **`_settings` (Настройки индекса/кластера)** [](https://www.elastic.co/docs/solutions/search/apis-and-tools.md)[](https://www.elastic.co/docs/solutions/search/apis-and-tools)
    - **Назначение**: Позволяет просматривать или динамически менять настройки, например, количество реплик для индекса или интервал обновления (`refresh_interval`).
