
> [!NOTE] DSL - JSON-язык запросов для Elastic 
> DSL (Domain Specific Language) — это JSON-язык запросов Elasticsearch.

Через него можно:
- искать документы;
- фильтровать документы;
- сортировать результаты;
- выполнять пагинацию;
- считать статистику;
- группировать данные;
- настраивать релевантность поиска;
- работать со вложенными объектами.

По сути DSL является аналогом SQL для Elasticsearch.

Например:
SQL:
```sql
SELECT *
FROM products
WHERE status = 'ACTIVE'
  AND price <= 1000
ORDER BY price
LIMIT 20
```

DSL:
```json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "ACTIVE" } },
        { "range": { "price": { "lte": 1000 } } }
      ]
    }
  },
  "sort": [
    { "price": "asc" }
  ],
  "size": 20
}
```

---
# 2. Архитектура DSL
DSL представляет собой набор **корневых секций (root elements)**.

Общая структура выглядит так:
```json
{
  "query": {},
  "aggs": {},
  "sort": [],
  "_source": [],
  "from": 0,
  "size": 10,
  "highlight": {},
  "collapse": {},
  "search_after": [],
  "post_filter": {},
  "runtime_mappings": {}
}
```

Каждый из этих блоков отвечает за отдельную часть обработки запроса.

## 2.1 Корневые элементы DSL
### 2.1.1 query - запрос "что искать"
Самый важный блок.

Отвечает на вопрос:
> **Какие документы необходимо найти?**

Именно внутри `query` находятся:
```
match
multi_match
bool
term
terms
range
nested
exists
```

```json
{
  "query": {
    "match": {
      "title": "iphone"
    }
  }
}
```

#### Как работает query
Когда Elasticsearch получает запрос:
```json
{
  "query": {
    "match": {
      "title": "iphone"
    }
  }
}
```

Он:
1. Обходит индекс.
2. Находит подходящие документы.
3. Рассчитывает релевантность (`_score`).
4. Возвращает результаты.

---
#### Важное правило
На верхнем уровне ==внутри query может находиться только один запрос.==

**ПРАВИЛЬНО**:
```json
{
  "query": {
    "match": {
      "title": "iphone"
    }
  }
}
```

**НЕПРАВИЛЬНО**:
```JSON
{
  "query": {
    "match": {...},
    "range": {...}
  }
}
```
> Для объединения условий используется `bool`.

---
### 2.1.2 aggs - агрегация 

Отвечают на вопрос:
> **Какую статистику необходимо посчитать?**

Аналог:
```sql
GROUP BY
COUNT
AVG
SUM
MIN
MAX
```

Пример:
```json
{
  "size": 0,
  "query": {
    "match_all": {}
  },
  "aggs": {
    "brands": {
      "terms": {
        "field": "brand.keyword",
        "size": 10
      }
    }
  }
}
```

Результат:
```json
{
  "took": 5,
  "timed_out": false,
  "_shards": { "total": 1, "successful": 1, "skipped": 0, "failed": 0 },
  "hits": { 
	  "total": { 
		  "value": 1000, 
		  "relation": "eq" 
	  }, 
	  "max_score": null, 
	  "hits": [] 
  },
  "aggregations": {
    "brands": {
      "buckets": [
        { "key": "Apple", "doc_count": 100 },
        { "key": "Samsung", "doc_count": 80 },
        { "key": "Xiaomi", "doc_count": 50 },
        { "key": "Google", "doc_count": 30 }
      ]
    }
  }
}
```
`"size": 0` означает "не возвращать документы, только статистику". Если опустить, Elasticsearch вернёт и документы (по умолчанию 10), и агрегации.

---
### 2.1.3 sort

Отвечает на вопрос:
> В каком порядке показывать найденные документы?

Пример:
По цене (по возрастанию):
```json
{
  "sort": [
    { "price": "asc" }
  ]
}
```

Можно сортировать:
По цене:
```json
{
	"price": "asc"}
```

По дате (сначала новые):
По дате (сначала новые):

```json
{
  "sort": [
    { "createdAt": "desc" }
  ]
}
```

По релевантности (явно):
```json
{
  "sort": [
    "_score"
  ]
}
```

**Важно:** Если `sort` не указан, Elasticsearch сортирует по `_score` (релевантности) в порядке убывания.

---
### 2.1.4 `from` — смещение (пагинация)
Отвечает на вопрос: _"Сколько документов пропустить?"_

**Аналог SQL:** `OFFSET`

**Пример:** пропустить первые 20 документов

```json
{
  "from": 20
}
```

---
### 2.1.5 `size` — лимит (пагинация)

Отвечает на вопрос: _"Сколько документов вернуть?"_

**Аналог SQL:** `LIMIT`

**Пример:** вернуть максимум 20 документов
```json
{
  "size": 20
}
```

**Совместное использование (пагинация):**
```json
{
  "from": 40,
  "size": 20
}
```

Это вернёт третью страницу результатов (документы 41–60).

---

### 2.1.6 `_source` — выбор возвращаемых полей

Управляет набором полей в ответе.

**Исходный документ в индексе:**
```json
{
  "id": 1,
  "name": "iPhone",
  "description": "Смартфон от Apple",
  "price": 1000,
  "reviews": [
    { "user": "john", "text": "Отлично!", "rating": 5 }
  ]
}
```

**Запрос:** вернуть только `id`, `name` и `price`
```json
{
  "_source": ["id", "name", "price"]
}
```

**Зачем нужен `_source`:**
- уменьшает размер ответа
    
- сокращает сетевой трафик
    
- снижает нагрузку на приложение
    

**Расширенная фильтрация (`includes`/`excludes`):**
```json
{
  "_source": {
    "includes": ["user.*", "message"],
    "excludes": ["user.password", "user.token"]
  }
}
```

---
### 2.1.7 `highlight` — подсветка найденных слов

Популярно в поисковых системах, маркетплейсах, документации.

**Запрос:**

```json
{
  "query": {
    "match": {
      "description": "gaming laptop"
    }
  },
  "highlight": {
    "fields": {
      "description": {}
    }
  }
}
```

**Ответ (фрагмент с подсветкой):**

```json
{
  "highlight": {
    "description": [
      "<em>gaming</em> <em>laptop</em> с мощной видеокартой"
    ]
  }
}
```

**Кастомизация подсветки:**

```json
{
  "highlight": {
    "pre_tags": ["<mark>"],
    "post_tags": ["</mark>"],
    "fields": {
      "description": {
        "fragment_size": 150,
        "number_of_fragments": 3
      }
    }
  }
}
```

---
### 2.1.8 `collapse` — группировка (удаление дублей)

Оставляет только один документ из каждой группы.

**Пример данных в индексе:**
- iPhone 15 (Apple)
    
- iPhone 15 Pro (Apple)
    
- Samsung S25 (Samsung)
    
- Samsung S25 Ultra (Samsung)
    

**Запрос:** сгруппировать по бренду, оставив по одному документу

```json
{
  "collapse": {
    "field": "brand.keyword"
  }
}
```

**Используется для:**
- устранения дублей
    
- группировки похожих товаров
    
- группировки версий документов
    

**С возвратом внутренних хитов (inner_hits):**

```json
{
  "collapse": {
    "field": "brand.keyword",
    "inner_hits": {
      "name": "other_products",
      "size": 5
    }
  }
}
```

---
### 2.1.9 `search_after` — продвинутая пагинация

Используется вместо `from` при очень глубокой пагинации (десятки тысяч записей).

**Проблема обычной пагинации:**

```json
{
  "from": 100000,
  "size": 20
}
```

Elasticsearch вынужден:
1. Отсортировать все документы
    
2. Отбросить первые 100000
    
3. Вернуть следующие 20 — **очень дорого**
    

**Решение — `search_after`:**

```json
{
  "size": 20,
  "sort": [
    { "createdAt": "desc" },
    { "_id": "asc" }
  ],
  "search_after": ["2025-01-01T10:00:00", "doc_12345"]
}
```

**Требования:**
- обязательна сортировка (`sort`)
    
- значения в `search_after` берутся из `sort`-полей последнего документа предыдущей страницы
    
- добавляйте `_id` как последний критерий сортировки для уникальности
    

**Используется для:**
- бесконечной прокрутки (infinite scroll)
    
- экспорта миллионов записей
    
- больших каталогов
    

---
### 2.1.10 `runtime_mappings` — временные вычисляемые поля

Создание полей "на лету" без изменения маппинга индекса.

**Пример:** добавить поле `discount_price` = `price * 0.9`

```json
{
  "runtime_mappings": {
    "discount_price": {
      "type": "double",
      "script": {
        "source": "emit(doc['price'].value * 0.9)"
      }
    }
  }
}
```

**Использование созданного поля:**

```json
{
  "runtime_mappings": {
    "discount_price": {
      "type": "double",
      "script": {
        "source": "emit(doc['price'].value * 0.9)"
      }
    }
  },
  "query": {
    "range": {
      "discount_price": {
        "lt": 1000
      }
    }
  },
  "fields": ["discount_price"]
}
```

**Используется для:**
- временных вычислений
    
- миграций данных
    
- прототипирования без изменения схемы
    

---
### 2.1.11 `post_filter` — фильтрация после агрегаций

**Один из самых непонятных блоков.**

**Проблема:** если использовать обычный `filter`, агрегации увидят только отфильтрованные документы.

**Кейс:** каталог товаров. Пользователь выбрал фильтр "Apple", но на панели фильтров нужно показать количество товаров у Samsung и Xiaomi.

**Неправильно (агрегации тоже фильтруются):**

```json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "brand.keyword": "Apple" } }
      ]
    }
  },
  "aggs": {
    "brands": {
      "terms": { "field": "brand.keyword" }
    }
  }
}
```

**Правильно (агрегации по всем данным, фильтр только для результатов):**

```json
{
  "aggs": {
    "brands": {
      "terms": { "field": "brand.keyword" }
    }
  },
  "post_filter": {
    "term": { "brand.keyword": "Apple" }
  }
}
```

**Порядок выполнения операций:**
text
1. query (поиск по всем документам)
     ↓
2. aggs (агрегации по найденным документам)
     ↓
3. post_filter (фильтрация результатов для ответа)

**Используется для:**
- фасетного поиска (фильтры с подсчётом количества)
    
- аналитики с последующей фильтрацией

# 3. Полнотекстовый поиск (Full Text Search)
Используется для поиска ==по текстовым полям== (тип `text`). 

Все полнотекстовые запросы **анализируют** входной текст (токенизация, стемминг, приведение к нижнему регистру и т.д.).

## 1.1 `match`
Самый популярный запрос. Ищет текст после анализа.

**Пример:**
```json
{
  "query": {
    "match": {
      "description": "gaming laptop"
    }
  }
}
```

Происходит это:
```
"gaming laptop" → анализ → ["gaming", "laptop"]
```
> Elasticsearch **ищет оба слова (ЛОГИКА "ИЛИ" по умолчанию).**

### 1.1.1 Параметр `operator` - логика поиска слов
Определяет логику поиска слов.

**`"or"` (по умолчанию):**
```json
{
  "match": {
    "description": {
      "query": "gaming laptop",
      "operator": "or"
    }
  }
}
```
Найдёт документы, содержащие `gaming` **или** `laptop` (или оба).


**`"and"`:**
```json
{
  "match": {
    "description": {
      "query": "gaming laptop",
      "operator": "and"
    }
  }
}
```
Найдёт только документы, где присутствуют **оба** слова.

---
### 1.1.2 Параметр `minimum_should_match`
**Минимальное количество слов** (или процент), ***которое должно совпасть.***

```json
{
  "match": {
    "description": {
      "query": "gaming laptop asus intel",
      "minimum_should_match": "75%"
    }
  }
}
```
Из 4 слов должно совпасть минимум 3.

---
### 1.1.3 Параметр `fuzziness` - Нечёткий поиск
Нечёткий поиск. Полезен **для исправления опечаток.**

```json
{
  "match": {
    "name": {
      "query": "iphon",
      "fuzziness": "AUTO"
    }
  }
}
```

Найдёт: `iphone`, `iphones`, `iphon`.

---
## 1.2 `prefix`

Поиск документов, где значение поля **начинается** с указанного префикса. Не анализирует поисковый запрос.

**Пример:**
```json
{
  "query": {
    "prefix": {
      "surname": "samar"
    }
  }
}
```

**Что происходит:**
```
"samar" → без анализа → ищет значения, начинающиеся с "samar"
```
Elasticsearch находит все документы, где поле `surname` начинается с `"samar"` (например, `"Samarcev"`, `"Samarov"`, `"Samar"`).

> **Важно:** `prefix` чувствителен к регистру, если поле типа `keyword`. Для `text` полей регистр может быть приведён при индексации.

### 1.2.2 Параметр `case_insensitive` (с версии 7.10)
Регистронезависимый поиск.
```json
{
  "query": {
    "prefix": {
      "surname": {
        "value": "samar",
        "case_insensitive": true
      }
    }
  }
}
```

### 1.2.3 Параметр `boost`
Повышение релевантности.
```json
{
  "query": {
    "prefix": {
      "surname": {
        "value": "samar",
        "boost": 2.0
      }
    }
  }
}
```

### 1.2.4 Ограничения `prefix`

|Проблема|Описание|
|---|---|
|**Производительность**|Медленнее, чем `term`, но быстрее, чем `wildcard`|
|**Короткие префиксы**|Префикс из 1-2 символов может быть очень медленным|
|**Нет поддержки суффиксов**|Ищет только с начала строки|

**Альтернативы для больших данных:**

- `search_as_you_type` (рекомендуется для автодополнения)
    
- `edge_ngram` анализатор

---
## 1.3 `wildcard` - поиск ПО ЧАСТИ СЛОВА

Поиск по шаблону с использованием wildcard-символов. Не анализирует поисковый запрос.

**Поддерживаемые символы:**

|Символ|Значение|
|---|---|
|`*`|Любая последовательность символов (включая пустую)|
|`?`|Ровно один любой символ|

**Пример:**
```json
{
  "query": {
    "wildcard": {
      "surname": "samar*"
    }
  }
}
```

**Что происходит:**
```
"samar*" → без анализа → ищет значения, начинающиеся с "samar" и имеющие любые символы после
```
Найдёт: `"Samarcev"`, `"Samarov"`, `"Samar"`, `"Samarabian"`.

### 1.3.1 Примеры использования wildcard
**Поиск по шаблону с `?` (один символ):**
```json
{
  "query": {
    "wildcard": {
      "code": "PROD-???-X"
    }
  }
}
```

Найдёт: `"PROD-123-X"`, `"PROD-ABC-X"`, но не `"PROD-12-X"` (два символа) и не `"PROD-1234-X"` (четыре символа).


**Поиск по шаблону с `*` в середине:**
```json
{
  "query": {
    "wildcard": {
      "email": "*@gmail.com"
    }
  }
}
```
Найдёт все Gmail адреса.


**Поиск по шаблону с `*` в начале (ОЧЕНЬ МЕДЛЕНО):**
```json
{
  "query": {
    "wildcard": {
      "surname": "*mar*"
    }
  }
}
```
Найдёт: `"Samarcev"`, `"Marley"`, `"Marks"`, `"Kuzmar"` (содержит подстроку `"mar"` в любом месте).

### 1.3.4 Ограничения `wildcard`

| Проблема                                 | Описание                                           |
| ---------------------------------------- | -------------------------------------------------- |
| **Производительность**                   | Очень медленный, особенно с ведущим `*`            |
| ==Не использует инвертированный индекс== | Сканирует все документы (как `LIKE '%...%'` в SQL) |
| **Нет анализа**                          | Не работает со стеммингом и синонимами             |

---
## 1.4 `match_phrase` - поиск точной фразы

Поиск точной фразы (с сохранением порядка слов).

```json
{
  "query": {
    "match_phrase": {
      "description": "gaming laptop"
    }
  }
}
```

- ✅ Найдёт: `"gaming laptop"`
    
- ❌ Не найдёт: `"gaming ultra laptop"`

---
### 1.4.1 Параметр `slop`
Разрешает расстояние между словами (на сколько позиций слова могут быть разнесены).

```json
{
  "query": {
    "match_phrase": {
      "description": {
        "query": "gaming laptop",
        "slop": 2
      }
    }
  }
}
```

Найдёт: `"gaming ultra powerful laptop"` (слова разошлись на 2 позиции).

---
## 1.5 `multi_match` - несколько полей одновременно

Поиск по нескольким полям одновременно.

```json
{
  "query": {
    "multi_match": {
      "query": "gaming laptop",
      "fields": ["title", "description"]
    }
  }
}
```

### 1.5.1 Веса полей (boosting)
```json
{
  "query": {
    "multi_match": {
      "query": "iphone",
      "fields": ["title^5", "description^2", "comments"]
    }
  }
}
```
Поле `title` имеет наибольший вес (в 5 раз важнее), `description` — в 2 раза важнее, `comments` — без множителя.

# 4. Точный поиск (Term-Level Queries)

Используется для полей типов: `keyword`, `integer`, `long`, `boolean`, `date`, `ip`.

**Важно:** эти запросы **НЕ анализируют** текст. Значение ищется как есть.

## 2.1 `term` - точное сопвпадение
Точное совпадение.

```json
{
  "query": {
    "term": {
      "status": "ACTIVE"
    }
  }
}
```

---
## 2.2 `terms` - аналог IN в SQL

Аналог SQL `IN`.

```json
{
  "query": {
    "terms": {
      "status": ["ACTIVE", "PENDING", "BLOCKED"]
    }
  }
}
```

---
## 2.3 `ids` - Поиск по `_id` документа.

Поиск по `_id` документа.
```json
{
  "query": {
    "ids": {
      "values": ["1", "5", "10"]
    }
  }
}
```

---
## 2.4 `exists`

Проверка существования поля (поле должно иметь не-`null` значение).

```json
{
  "query": {
    "exists": {
      "field": "phone"
    }
  }
}
```

# 3. Фильтрация (Filters)

Фильтры **не вычисляют `_score`**, работают **быстрее** и **кэшируются**.

## 3.1 `term` filter

```json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "ACTIVE" } }
      ]
    }
  }
}
```

---
## 3.2 `range` filter

Диапазоны чисел, дат, IP.

**Операторы:**
- `gte` — greater than or equal (≥)
    
- `gt` — greater than (>)
    
- `lte` — less than or equal (≤)
    
- `lt` — less than (<)

```json
{
  "query": {
    "range": {
      "price": {
        "gte": 1000,
        "lte": 5000
      }
    }
  }
}
```


**Диапазон дат:**
```json
{
  "query": {
    "range": {
      "createdAt": {
        "gte": "2025-01-01",
        "lte": "2025-12-31"
      }
    }
  }
}
```

**Относительные даты:**
```json
{
  "query": {
    "range": {
      "createdAt": {
        "gte": "now-7d"
      }
    }
  }
}
```

(последние 7 дней)

# 4. Bool Query

**Самая важная конструкция в Elasticsearch.** Комбинирует несколько условий.

|Блок|Логика|Влияет на `_score`|Кэшируется|
|---|---|---|---|
|`must`|AND|✅ Да|❌ Нет|
|`filter`|AND|❌ Нет|✅ Да|
|`should`|OR (желательное)|✅ Да|❌ Нет|
|`must_not`|NOT|❌ Нет|✅ Да|

---
## 4.1 `must`

Обязательное условие. Влияет на релевантность.

```json
{
  "bool": {
    "must": [
      { "match": { "description": "laptop" } }
    ]
  }
}
```

---
## 4.2 `filter`
Обязательное условие. Не влияет на `_score`.

```json
{
  "bool": {
    "filter": [
      { "term": { "status": "ACTIVE" } }
    ]
  }
}
```

---
## 4.3 `should`
Желательное условие. Повышает `_score` при совпадении.

```json
{
  "bool": {
    "should": [
      { "term": { "brand": "Apple" } }
    ]
  }
}
```

---
## 4.4 `must_not`
Исключающее условие (NOT).

```json
{
  "bool": {
    "must_not": [
      { "term": { "status": "DELETED" } }
    ]
  }
}
```

---
## 4.5 Комплексный пример

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "description": "laptop" } }
      ],
      "filter": [
        { "term": { "status": "ACTIVE" } },
        { "range": { "price": { "lte": 3000 } } }
      ],
      "must_not": [
        { "term": { "brand": "Unknown" } }
      ],
      "should": [
        { "term": { "brand": "Apple" } }
      ]
    }
  }
}
```

# 6. Пагинация

## 6.1 `from` + `size`

```json
{
  "from": 0,
  "size": 20
}
```
- Первая страница: `from: 0, size: 20`
    
- Вторая страница: `from: 20, size: 20`
    

**⚠️ Ограничение:** Глубокая пагинация (например, `from: 100000`) очень дорогая — плохая практика.

---
## 6.2 `search_after` (для больших объёмов)

Используется в production для прокрутки миллионов записей.

```json
{
  "size": 20,
  "sort": [
    { "createdAt": "desc" },
    { "_id": "asc" }
  ],
  "search_after": ["2025-01-01T10:00:00", "doc_12345"]
}
```

**Требования:**
- обязательна сортировка
    
- добавляйте `_id` как последний критерий для уникальности
    
- значения из `search_after` берутся из полей `sort` последнего документа предыдущей страницы

# 9. Nested Queries

Используется, когда внутри документа есть **массив объектов**, и нужно сохранить связь между полями каждого объекта.

**Документ:**
```json
{
  "name": "Laptop",
  "reviews": [
    { "user": "John", "rating": 5 },
    { "user": "Kate", "rating": 2 }
  ]
}
```

**Маппинг (обязателен):**
```json
{
  "reviews": {
    "type": "nested"
  }
}
```

**Запрос:** найти отзывы от John с рейтингом ≥ 4
```json
{
  "query": {
    "nested": {
      "path": "reviews",
      "query": {
        "bool": {
          "must": [
            { "term": { "reviews.user": "John" } },
            { "range": { "reviews.rating": { "gte": 4 } } }
          ]
        }
      }
    }
  }
}
```

# 10. Relevance (Score)

По умолчанию Elasticsearch возвращает документы, отсортированные по релевантности. Каждый документ получает поле `_score`.

```json
{
  "query": {
    "match": {
      "description": "gaming laptop"
    }
  }
}
```

**Ответ (фрагмент):**
```json
{
  "hits": {
    "hits": [
      { "_score": 12.45, ... }
    ]
  }
}
```

# 11. Boosting
Повышение важности поля или запроса.

## 11.1 `boost` в `match`
```json
{
  "query": {
    "match": {
      "title": {
        "query": "laptop",
        "boost": 3
      }
    }
  }
}
```

Совпадение в `title` будет в 3 раза важнее, чем в других полях.

---
## 11.2 `boost` в `term` (фильтры не имеют скоринга, но могут иметь boost в query контексте)

```json
{
  "query": {
    "bool": {
      "should": [
        { "term": { "brand": { "value": "Apple", "boost": 2 } } }
      ]
    }
  }
}
```
