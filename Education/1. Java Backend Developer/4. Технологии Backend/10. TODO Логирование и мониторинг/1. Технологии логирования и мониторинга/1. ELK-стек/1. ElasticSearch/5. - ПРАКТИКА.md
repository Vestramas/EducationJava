# 1. CRUD операции

## 1.1 Создание индекса с маппингом
Создавать индекс явно не обязательно. Если попытаться добавить документ в несуществующий индекс, то Elastic создаст индекс автоматически.

Но мы же, создадим его явно, и укажем ТИПЫ ДАННЫХ для полей документа
```json
PUT shop
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0
  },
  "mappings": {
    "properties": {
      "id": { "type": "keyword" },
      "name": { 
        "type": "text", 
        "analyzer": "russian" 
       },
      "brand": { "type": "keyword" },
      "price": { "type": "float" },
      "category": { "type": "keyword" },
      "in_stock": { "type": "boolean" },
      "created_at": { "type": "date" },
      "tags": { "type": "keyword" },

      "reviews": {
        "type": "nested",
        "properties": {
          "user": { "type": "keyword" },
          "rating": { "type": "integer" },
          "comment": { 
            "type": "text", 
            "analyzer": "russian" 
          }
        }
      }
    }
  }
}
```

```json
{
  "acknowledged": true,
  "shards_acknowledged": true,
  "index": "shop"
}
```

## 1.2 [CREATE] Индексация документа (Добавление документа)
Для создания документа используется 
POST запрос на URL `<index>/_doc`
### 1.2.1 АВТОМАТИЧЕСКАЯ ГЕНЕРАЦИЯ ID
Создадим документ с **АВТОМАТИЧЕСКОЙ ГЕНЕРАЦИЕЙ ID**

> ВНУТРЕННЕ Поле документа ID НЕ СВЯЗАНО С ID документа
```json
POST shop/_doc
{
  "id": "P001",
  "name": "Игровой ноутбук Asus ROG",
  "brand": "Asus",
  "price": 120000,
  "category": "laptops",
  "in_stock": true,
  "created_at": "2026-06-01T10:00:00Z",
  "tags": ["gaming", "laptop", "asus", "high-end"],
  
  "reviews": [
    { "user": "alex", "rating": 5, "comment": "Отличный ноутбук!" },
    { "user": "ivan", "rating": 4, "comment": "Хороший, но дорогой" }
  ]
}
```

```json
{
  "_index": "shop",
  "_id": "1F-wop4BujqXmdAECAFf",
  "_version": 1,
  "result": "created",
  "_shards": {
    "total": 1,
    "successful": 1,
    "failed": 0
  },
  "_seq_no": 0,
  "_primary_term": 1
}
```

### 1.2.3 Batch insert (`_bulk`)
Чтобы вставить сразу несколько документов, используется ednpoint `_bulk`
```json
POST _bulk
{ "index" : { "_index" : "test", "_id" : "1" } }
{ "field1" : "value1" }
{ "delete" : { "_index" : "test", "_id" : "2" } }
{ "create" : { "_index" : "test", "_id" : "3" } }
{ "field1" : "value3" }
{ "update" : {"_id" : "1", "_index" : "test"} }
{ "doc" : {"field2" : "value2"} }
```

НО ЧЕРЕЗ Kibana не работает

## 1.3 [READ] Получение документа по ID
Чтобы найти документ ПО СГЕНЕРИРОВАННОМУ ID, нужно сначала узнать сгенерированный ID

Для этого, чтобы получить **список ВСЕХ ДОКУМЕНТОВ**, используется endpoint `<index>/_search` 
```http
GET shop/_search
```

Как видно, ID у документа `1F-wop4BujqXmdAECAFf`, и оно НИКАК НЕ СВЯЗАНО С ПОЛЕМ id документа

Теперь, по ID `1F-wop4BujqXmdAECAFf`, можно посмотреть его детальную информацию
```HTTP
GET shop/_doc/1F-wop4BujqXmdAECAFf
```

### 1.3.1 Получение нескольких документов по ID
Для этого используется `<index>/_mget`
```json
GET shop/_mget
{
  "ids":["1F-wop4BujqXmdAECAFf","P002"]
}
```

> Первый id сгенерированный ElasticSearch, а второй мы указали явно

## 1.4 [UPDATE] Частичное обновление документа
Для частичного обновления документа, используется метод PUT

```json
POST shop/_update/1F-wop4BujqXmdAECAFf
{
  "doc": {
    "price": 115000,
    "tags": ["gaming", "laptop", "asus", "high-end", "updated"]
  }
}
```

## 1.5 [DELETE] Удаление документа
### 1.5.1 Удаление по ID
Используется метод DELETE
```http
DELETE shop/_doc/1F-wop4BujqXmdAECAFf
```

Вывод
```json
{
  "_index": "shop",
  "_id": "1F-wop4BujqXmdAECAFf",
  "_version": 3,
  "result": "deleted",
  "_shards": {
    "total": 1,
    "successful": 1,
    "failed": 0
  },
  "_seq_no": 3,
  "_primary_term": 1
}
```

### 1.5.2 Удаление по запросу
Удаляет все документы, соответствующие условию запроса.

> Для этого используется метод POST (так как нужно передать запрос в теле) на endpoint `_delete_by_query`

```json
POST shop/_delete_by_query
{
  "query": {
    "term": { "in_stock": false }
  }
}
```

## 1.6 ПОИСК (`_search`)
### 1.6.1 match — полнотекстовый поиск
Анализирует запрос (разбивает на токены, приводит к нижнему регистру). 

Логика "ИЛИ" по умолчанию — достаточно одного слова.

> **Ищет точное совпадение в ПОДСТРОКЕ**
> Условно "Google Pixel Super puper phone", мы хотим найти документ, где в описании товара есть слово Pixel, то нам для этого подойдет match, а не term

```json
GET shop/_search
{
  "query": {
    "match": {
      "name": "pixel"
    }
  }
}
```

Вывод:

### 1.6.2 match с оператором AND
**Что делает:** Требует, чтобы ВСЕ слова из запроса присутствовали в документе.

```json
GET shop/_search
{
  "query": {
    "match": {
      "name": {
        "query": "pixel 10",
        "operator": "and"
      }
    }
  }
}
```
