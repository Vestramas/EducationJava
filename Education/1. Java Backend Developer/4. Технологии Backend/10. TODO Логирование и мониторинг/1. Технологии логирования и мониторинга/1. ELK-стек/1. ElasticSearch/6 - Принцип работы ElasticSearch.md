**Elasticsearch** — распределённый поисковый движок, написанный на Java, построенный на библиотеке **Apache Lucene**.

В качестве хранения данных использует **НЕРЕЛЯЦИОННУЮ (документно-ориентированную)** модель.

---
## 2. Ключевые понятия

|Понятие|Описание|Аналог в РСУБД|
|---|---|---|
|**Документ (Document)**|Базовая единица информации. Хранится в формате JSON.|Строка (ROW)|
|**Индекс (Index)**|Коллекция документов одного типа (логическая группировка).|Таблица (TABLE)|
|**Узел (Node)**|Один работающий экземпляр Elasticsearch (Java-процесс).|Сервер БД|
|**Кластер (Cluster)**|Группа узлов, работающих вместе.|Кластер БД|
|**Шард (Shard)**|Физическая часть индекса. Индекс разбивается на шарды, которые распределяются по узлам. Позволяет масштабировать данные горизонтально.|Партиция (Partition) / Шард|
|**Реплика (Replica)**|Копия шарда. Обеспечивает отказоустойчивость и увеличивает пропускную способность чтения.|Реплика / Слейв|

**Важные уточнения:**
- Индекс — это **логическое** понятие (аналог таблицы). 
  Шарды — **физическое** (как файлы на диске).
    
- Реплика — это не просто резервная копия. Реплики могут обслуживать поисковые запросы **параллельно** с первичными шардами.
    
- Количество первичных шардов задаётся **при создании индекса и не может быть изменено** позже.
    
- Количество реплик можно менять **динамически**.

## 3. Анализатор (Analyzer) — сердце полнотекстового поиска

**Анализатор** — механизм, который преобразует входной текст в структуру, удобную для полнотекстового поиска.

### 3.1 Из чего состоит анализатор
Любой анализатор — это последовательность из трёх компонентов:

|Компонент|Что делает|Обязательность|
|---|---|---|
|**Символьные фильтры (Character Filters)**|Очищают текст ДО разбиения на слова (удаляют HTML-теги, заменяют `&` на `and`).|0 или более|
|**Токенайзер (Tokenizer)**|Разбивает очищенный текст на **токены** (обычно на отдельные слова). Это сердце анализатора.|Ровно 1|
|**Фильтры токенов (Token Filters)**|Нормализуют токены: приводят к нижнему регистру, удаляют стоп-слова, выполняют стемминг (приведение к корню).|0 или более|

### 3.2 Процесс анализа текста
```text
"Быстрые лисы бежали по полю"
        ↓
[Символьные фильтры] — очистка (например, удаление пунктуации)
        ↓
[Токенайзер] — разбиение на слова
        ↓
["Быстрые", "лисы", "бежали", "по", "полю"]
        ↓
[Фильтры токенов] — приведение к нижнему регистру, стемминг
        ↓
["быстр", "лис", "бежал", "пол"]
```

### 3.3 Где и когда применяется анализ

**Важно:** анализ применяется **ДВАЖДЫ**:

|Этап|Что происходит|
|---|---|
|**Индексация (Index time)**|Текст документа анализируется, токены сохраняются в обратном индексе.|
|**Поиск (Search time)**|Поисковый запрос анализируется **тем же анализатором**, токены запроса ищутся в обратном индексе.|

Это гарантирует, что запрос `"быстрая лиса"` найдёт документ `"Быстрые лисы"`.

## 4. Обратный индекс (Inverted Index)

**Обратный индекс** — основная структура данных, ==обеспечивающая молниеносный полнотекстовый поиск.==

### 4.1 Принцип работы (упрощённо)

**Обычный индекс:**
```text
Документ → список слов
Документ 1: кошка, собака, дом
Документ 2: собака, кот, дерево
Документ 3: дом, дерево, кошка
```

**Обратный индекс:**
```text
Слово → список документов
кошка → [1, 3]
собака → [1, 2]
дом → [1, 3]
кот → [2]
дерево → [2, 3]
```

### 4.2 Как это ускоряет поиск
Чтобы найти все документы со словом `"кошка"`, не нужно сканировать все документы — достаточно посмотреть в обратный индекс и сразу получить список `[1, 3]`.

**Сложность поиска:** O(1) вместо O(N) (полного сканирования).

#### Важное уточнение
> **Обратный индекс строится для каждого поля типа `text` по умолчанию.**

Для полей типа `keyword` тоже строится обратный индекс, но он хранит **точное значение целиком**, без анализа. Поэтому `keyword` подходит для фильтрации и сортировки.

## 5. Управление индексацией: Маппинг (Mapping)

### 5.1 Проблема
По умолчанию Elasticsearch индексирует **все поля** всех документов. 

Это удобно, но:
- Занимает лишнее дисковое пространство
    
- Снижает скорость индексации
    
- Может создавать ненужные обратные индексы
    

### 5.2 Решение: маппинг
**Маппинг** — это схема индекса, которая описывает:
- Какие поля есть в документе
    
- Какого типа каждое поле (`text`, `keyword`, `integer` и т.д.)
    
- Какие поля индексировать (`index: true/false`)
    
- Какой анализатор использовать

### 5.3 Параметр `index`
```json
{
  "mappings": {
    "properties": {
      "searchable_field": {
        "type": "text",
        "index": true
      },
      "non_indexed_field": {
        "type": "keyword",
        "index": false
      }
    }
  }
}
```

|Значение `index`|Что происходит|
|---|---|
|`true` (по умолчанию)|Поле индексируется — можно искать, фильтровать, сортировать, агрегировать|
|`false`|Поле НЕ индексируется — НЕЛЬЗЯ искать, но оно сохраняется в `_source` и может быть возвращено в ответе|

### 5.4 Динамический маппинг

Если не указать маппинг явно, Elasticsearch сам определит типы полей при первой индексации документа. Это удобно для прототипирования, но опасно для production (может создать `text` поле там, где нужен `keyword`).
