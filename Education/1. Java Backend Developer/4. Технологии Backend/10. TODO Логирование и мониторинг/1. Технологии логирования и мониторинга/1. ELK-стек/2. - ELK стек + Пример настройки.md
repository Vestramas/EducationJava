# 1. Проблема
В микросервисной архитектуре у вас есть десятки или сотни экземпляров приложений, каждый из которых пишет логи в свои локальные файлы. 

Разбирать инциденты, заходя по SSH на каждый сервер и выполняя `grep`, становится невозможно. 

Нужна централизация.

---
## 1.1 Elasticsearch - поисковой движок, документно-ориентированная БД

> [!NOTE] ElasticSearch - поисковой движок (документно-ориентированная БД)
> **Elasticsearch** — это распределённый поисково-аналитический движок, который выступает в роли **центрального хранилища** всех данных в стеке ELK.

---
## 1.2 Инструменты сбора
### 1. Logstash
> [!NOTE] Logstash - инструмент сбора
> **Logstash** — это инструмент для сбора, обработки и отправки данных. 

Он **работает по принципу конвейера (pipeline), через который проходят события**. 
- Если Elasticsearch — это «куда складываем», 
- то Logstash — это «как доставляем и что делаем по пути».

---
### 2. Beats
Хотя классический стек называется ELK, сейчас компания Elastic активно развивает семейство **Beats** — **лёгких агентов сбора данных**, написанных на Go и потребляющих минимум ресурсов. 

Они **располагаются непосредственно на серверах приложений** и *передают данные в Logstash или напрямую в Elasticsearch.*

|Beat|Назначение|
|---|---|
|**Filebeat**|Чтение лог-файлов и пересылка их содержимого. Самый популярный Beat.|
|**Metricbeat**|Сбор системных и сервисных метрик (CPU, память, сеть, Nginx, PostgreSQL, Redis).|
|**Packetbeat**|Анализ сетевого трафика в реальном времени.|
|**Heartbeat**|Проверка доступности сервисов (ping, HTTP health-check).|
|**Auditbeat**|Сбор данных аудита (входы пользователей, изменения файлов).|

**Filebeat vs прямая отправка из приложения:**
- Filebeat **разгружает** Java-приложение: запись в локальный файл происходит мгновенно, а пересылкой по сети занимается отдельный процесс;
    
- При проблемах с Logstash логи не теряются — они остаются в файлах до восстановления связи;
    
- Filebeat поддерживает **backpressure**: если Logstash не справляется, он замедляет чтение файлов, а не роняет приложение-источник.

---
## 1.3 Kibana / Grafana - UI для ElasticSearch
> [!NOTE] Kibana - интерфейс для работы с ElasticSearch
> **Kibana** — это веб-приложение, которое служит пользовательским интерфейсом для Elasticsearch. 

Если Elasticsearch — это мощный движок, то Kibana — это «приборная панель», которая позволяет с этим движком взаимодействовать без написания запросов вручную.

## 2.1 Как Java-логи попадают в ELK
Как показано на схеме выше, приложение генерирует логи, которые через сеть попадают в Logstash, а затем в Elasticsearch для хранения. И уже оттуда их забирает Kibana.

> [!WARNING] В приложении все также используются библиотеки логирования, меняются аппендеры
> Ваше Java-приложение всё так же использует SLF4J + Logbac= (или Log4j2) и пишет логи через привычные аппендеры.

Разница лишь в том, **куда** направлены эти логи и **в каком формате**.

### Часть 1: Java (Spring Boot) — Генерация и отправка логов
Чтобы ваше приложение говорило на языке, понятном Logstash, мы **переведем логи в структурированный формат JSON.** 

Это ключевой момент, так как работать с хорошо структурированными данными намного удобнее, чем с простым текстом

#### Шаг 1: Добавьте зависимость
Добавьте в ваш `pom.xml` библиотеку `logstash-logback-encoder`. 

**Она предоставляет специальные кодировщики**, которые превращают ваши **логи в JSON**.
```xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

#### Шаг 2: Настройте Logback (`logback-spring.xml`)
Теперь создадим или изменим файл 
`src/main/resources/logback-spring.xml`. 

Это основной конфигурационный файл для вашей системы логирования. 
Здесь мы укажем, куда и в каком формате отправлять логи.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- Важная настройка для Spring Boot: чтобы работало property из application.yml -->
    <springProperty scope="context" name="springAppName" source="spring.application.name"/>


    <!-- 1. Консольный вывод для разработки (оставляем как есть) -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>
	            %d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level 
	            %logger{36} - %msg%n
            </pattern>
        </encoder>
    </appender>

    <!-- 2. ОСНОВНОЙ APPENDER: отправка в Logstash -->
    <appender name="LOGSTASH" 
    class="net.logstash.logback.appender.LogstashTcpSocketAppender">
    
        <!-- Адрес и порт вашего Logstash -->
        <destination>localhost:5044</destination>

        <!-- Кодировщик, который создает JSON -->
        <encoder 
	class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
            <providers>
                <!-- Автоматически добавляет timestamp -->
                <timestamp>
                    <timeZone>UTC</timeZone>
                </timestamp>
                
                <!-- Уровень лога (INFO, DEBUG, ERROR...) -->
                <logLevel/>
                
                <!-- Имя логгера -->
                <loggerName/>
                
                <!-- Текст сообщения -->
                <message/>
                
                <!-- Поток -->
                <threadName/>
                
                <!-- MDC (Mapped Diagnostic Context)-важно для traceId -->
                <mdc/>
                
                <!-- Кастомные поля, которые мы добавляем -->
                <pattern>
                    <pattern>
                        {
                        "appName": "${springAppName:-unknown}"
                        }
                    </pattern>
                </pattern>
                
            </providers>
        </encoder>
    </appender>

    <!-- Корневой логгер: пишем лог и в консоль, и в Logstash -->
    <root level="INFO">
        <appender-ref ref="CONSOLE" />
        <appender-ref ref="LOGSTASH" />
    </root>
</configuration>
```

**Что происходит в этом коде?**

- Мы создаем `appender` с именем `LOGSTASH` типа `LogstashTcpSocketAppender`. Его задача — отправлять логи по TCP напрямую в Logstash[](https://blog.csdn.net/moumouguo/article/details/145754811).
    
- Внутри него мы используем `LoggingEventCompositeJsonEncoder`. Это гибкий инструмент, который собирает ваш лог в JSON-объект из разных "провайдеров" (`providers`)[](https://cloud.tencent.cn/developer/information/Logstash%20logback%E7%BC%96%E7%A0%81%E5%99%A8%E6%9C%AA%E5%B1%8F%E8%94%BD%E5%B5%8C%E5%A5%97%E5%AD%97%E6%AE%B5).
    
- Провайдеры (`<providers>`) определяют, какие поля попадут в JSON: `timestamp`, `logLevel`, `message`, `mdc` и т.д. Это делает ваши логи богатыми на контекст и очень удобными для поиска.
    
- Мы добавляем кастомное поле `appName`, которое динамически подтягивает имя вашего приложения из `application.yml`. Это бесценно, когда у вас много микросервисов.

После того как вы запустите приложение с такой настройкой, оно начнет отправлять готовые JSON-логи на порт `5044` вашего локального хоста, где их ждет Logstash.

### Часть 2: Logstash — Прием, обработка и маршрутизация

Logstash действует как центральный диспетчерский пункт. 

Он получает логи от многих приложений, может их как-то обработать и затем отправить дальше[](https://www.sfeir.dev/back/elasticsearch-spring-boot-premiers-pas-partie-2/). 

Давайте настроим его для приема логов по TCP.

#### Шаг 1: Создайте конфигурацию Logstash
Создайте файл, например `logstash.conf`. Его структура проста:
```
input { /* откуда берем */ }
filter { /* как обрабатываем */ }
output { /* куда отправляем */ }
```

#### Шаг 2: Заполните конфигурацию
```
input {
  # Принимаем логи по TCP на порту 5044
  tcp {
    port => 5044
    # Сразу говорим, что входящие данные в формате JSON
    codec => json
  }
}

filter {
  # Мы можем добавить логику обработки, но для начала она НЕ ОБЯЗАТЕЛЬНА
  # Например, если вы хотите добавить поле с хостом
  mutate {
    add_field => { "[host][name]" => "${HOSTNAME}" }
  }
}

output {
  # Отправляем данные в Elasticsearch
  elasticsearch {
    hosts => ["localhost:9200"]
    # Имя индекса будет содержать имя нашего приложения и дату
    index => "logs-%{+YYYY.MM.dd}"
  }

  # Дополнительно вы можете выводить обработанные логи в консоль Logstash для отладки
  stdout {
    codec => rubydebug
  }
}
```

**Что здесь важно?**
- `input { tcp }`: Мы говорим Logstash слушать порт 5044. Это тот самый порт, на который наше Java-приложение отправляло логи.
    
- `codec => json`: Эта настройка очень важна. Мы сообщаем, что данные уже приходят в формате JSON, и Logstash может интерпретировать их как готовые поля, а не как простой текст[](https://www.baeldung.com/java-application-logs-to-elastic-stack).
    
- `filter { mutate }`: Необязательный, но показательный блок. Здесь можно выполнять сложную обработку: разбирать сложные сообщения с помощью `grok`, обогащать данные geo-информацией через `geoip`, переименовывать или удалять поля[](https://cloud.tencent.cn/developer/information/Logstash%20logback%E7%BC%96%E7%A0%81%E5%99%A8%E6%9C%AA%E5%B1%8F%E8%94%BD%E5%B5%8C%E5%A5%97%E5%AD%97%E6%AE%B5). В данном случае мы просто добавляем имя хоста.
    
- `output { elasticsearch }`: Мы указываем адрес Elasticsearch и задаем шаблон для имени индекса (`index`). Индексы в Elasticsearch — это что-то вроде баз данных для документов. Формат `logs-%{+YYYY.MM.dd}` будет создавать новый индекс для каждого дня, например `logs-2026.06.01`. Это стандартная и очень удобная практика для управления хранением данных[](https://www.baeldung.com/java-application-logs-to-elastic-stack).

#### Шаг 3: Запустите Logstash
После создания конфигурации, запустите Logstash, указав путь к файлу:
```bash
bin/logstash -f /путь/к/logstash.conf
```

### Часть 3: Elasticsearch — Хранение и индексация
Это компонент, который ничего не требует от разработчика, кроме одного: он должен быть запущен и доступен. 

Все остальное он делает сам.
- **Прием данных:** Он слушает порт `9200` и получает данные от Logstash.
    
- **Индексация:** Elasticsearch анализирует полученные JSON-документы, строит по ним обратные индексы и сохраняет их.
    
- **Поиск:** Именно его мощный движок позволит нам в будущем искать логи за доли секунды.


Убедитесь, что ваш Elasticsearch запущен (обычно через `./elasticsearch` в папке с дистрибутивом) и доступен по адресу `localhost:9200`[](https://github.com/seyedreza-zoghi/springboot-elk-Logs).

### Часть 4: Kibana — Визуализация и анализ
И вот наступает самый интересный момент: мы видим результат.

#### Шаг 1: Запустите Kibana
Обычно он запускается командой `./kibana` и становится доступен в браузере по адресу `http://localhost:5601`[](https://github.com/seyedreza-zoghi/springboot-elk-Logs).

#### Шаг 2: Настройте Index Pattern
Это самый важный шаг в Kibana.

1. В боковом меню перейдите в **Stack Management** → **Index Patterns**.
    
2. Нажмите **Create index pattern**.
    
3. В поле **Index pattern name** введите `logs-*`. Звездочка означает "все индексы, которые начинаются с 'logs-'".
    
4. Нажмите **Next step**.
    
5. Из выпадающего списка **Time filter field name** выберите `@timestamp`. Это поле Kibana будет использовать для всех временных фильтров[](https://last9.io/blog/kibana-logs/?utm_source=www.techopsexamples.com&utm_medium=referral&utm_campaign=step-by-step-guide-to-aws-s3-object-encryption).
    
6. Нажмите **Create index pattern**.

После этого Kibana будет знать, где и как искать ваши логи.

#### Шаг 3: Исследуйте и визуализируйте
- **Discover (Исследование):** Перейдите в раздел **Discover**. Это ваша главная консоль для поиска и просмотра логов. Сверху вы увидите поле поиска (KQL — Kibana Query Language). Здесь можно вводить запросы, например:
    
    - `log.level: "ERROR"`: показать только ошибки.
        
    - `message: "exception"`: найти все логи, где в сообщении есть слово "exception".
        
    - `appName: "my-auth-service"`: отфильтровать логи по имени вашего сервиса.
        
    - Вы можете комбинировать условия: `log.level: "ERROR" and appName: "my-payment-service"`.
        
- **Visualize (Визуализация):** Перейдите в **Visualize Library** → **Create visualization**. Выберите тип, например, **Vertical bar**. Затем выберите ваш `logs-*` Index Pattern. Теперь вы можете строить графики, например:
    
    - **Y-Axis:** Count (количество событий).
        
    - **X-Axis:** выбрать `log.level.keyword`. Этот график наглядно покажет, сколько логов каждого уровня (INFO, WARN, ERROR) было за указанный период[](https://www.baeldung.com/java-application-logs-to-elastic-stack).
        
- **Dashboard (Панель мониторинга):** Вы можете собрать несколько созданных вами визуализаций на одной панели, создав мощный дашборд для мониторинга вашего приложения.
