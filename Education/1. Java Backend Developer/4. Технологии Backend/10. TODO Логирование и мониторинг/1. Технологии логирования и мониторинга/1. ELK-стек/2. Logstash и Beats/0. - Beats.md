> [!NOTE] Легковесные агенты сборка данных
> **Beats** — это **набор легковесных агентов сбора данных** (data shippers), **которые устанавливаются на серверы**, рабочие станции, сетевые устройства и другие узлы инфраструктуры для сбора различных типов данных и их отправки в Elastic Stack.
> 
> Отдельная прога, которая читает логи из указанного источника 

Обычно данные отправляются напрямую в Elasticsearch или через Logstash для дополнительной обработки.

Упрощённо архитектура выглядит так:
![[Pasted image 20260610164604.png]]

Основная идея Beats:
- минимальная нагрузка на систему;
- специализированный агент для конкретного типа данных;
- простая YAML-конфигурация;
- автоматическое обогащение данных метаданными;
- готовые дашборды и шаблоны индексов.

# 2. Семейство Beats

Elastic предоставляет несколько специализированных агентов.

|Beat|Назначение|
|---|---|
|Filebeat|Сбор логов|
|Metricbeat|Сбор метрик|
|Packetbeat|Анализ сетевого трафика|
|Winlogbeat|Сбор Windows Event Logs|
|Auditbeat|Аудит безопасности ОС|
|Heartbeat|Мониторинг доступности сервисов|

---
# 3. Общая архитектура Beats

Все Beats построены на библиотеке **Libbeat**.

Libbeat предоставляет:
- чтение конфигурации;
- отправку данных;
- буферизацию;
- управление подключениями;
- логирование;
- мониторинг работы агента.

Поэтому структура конфигурации у всех Beats очень похожа.

Общий шаблон:
```
<источник_данных>  
  
processors:  
  
output:  
  
logging:  
  
setup:  
  
monitoring:
```

# 4. Общая структура конфигурации Beat

Практически **любой Beat имеет следующие секции.**

---
## 4.1 Секция источников данных

Определяет:
- **откуда собирать данные**;
- как часто собирать;
- какие объекты отслеживать.

### Примеры:

*Filebeat:*
```yaml
filebeat.inputs:
```

*Metricbeat:*
```
metricbeat.modules:
```

*Heartbeat:*
```
heartbeat.monitors:
```

*Packetbeat:*
```
packetbeat.interfaces:
```

---
## 4.2 Processors - обработка данных

Позволяют модифицировать события перед отправкой.

Пример:
```yaml
processors:
  - add_host_metadata:
  - add_cloud_metadata:
```

Часто используются:

### add_host_metadata
Добавляет:
```yaml
host.name
host.ip
host.os.name
host.architecture
```

---
### add_cloud_metadata

Добавляет:
```yaml
cloud.provider
cloud.region
cloud.instance.id
```

---
### add_fields
Добавляет **пользовательские поля.**

```yaml
processors:
  - add_fields:
      target: ''
      fields:
        environment: production
```

Результат:
```json
{
  "environment": "production"
}
```

---
### drop_event
Полностью **удаляет событие**.

```
processors:
  - drop_event:
      when:
        equals:
          log.level: debug
```

---
### drop_fields
**Удаляет поля.**

```
processors:
  - drop_fields:
      fields: ["agent.ephemeral_id"]
```

---
## 4.3 Output
Определяет **место назначения данных.**

Поддерживаются:
- Elasticsearch
- Logstash
- Kafka
- Redis
- File
- Console
- Discard

---
### Elasticsearch Output
Самый популярный вариант.

```
output.elasticsearch:
  hosts:
    - "localhost:9200"

  username: elastic
  password: password
```

---
### Logstash Output

```
output.logstash:
  hosts:
    - "localhost:5044"
```

---
### Kafka Output

```
output.kafka:
  hosts:
    - kafka1:9092

  topic: logs
```

---
## 4.4 Logging
Настройка **логирования самого Beat**.

```yaml
logging.level: info
```

Уровни:
```
debug
info
warning
error
```

---
## 4.5 Setup
**Настройка объектов Elasticsearch.**

```
setup.template.enabled: true
setup.ilm.enabled: true
```

Используется для:
- шаблонов индексов;
- lifecycle policy;
- dashboards;
- ingest pipelines.

---
## 4.6 Monitoring
**Мониторинг работы самого Beat.**
```
monitoring.enabled: true
```

Позволяет собирать:
- количество событий;
- использование памяти;
- ошибки отправки;
- состояние очередей.
