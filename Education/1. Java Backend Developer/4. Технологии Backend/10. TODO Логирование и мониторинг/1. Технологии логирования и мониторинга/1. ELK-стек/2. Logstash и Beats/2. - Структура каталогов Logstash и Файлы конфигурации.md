# 1. Структура каталогов
Расположение каталогов зависит от способа установки (tar.gz, rpm/deb, Docker).

### Типичная структура (rpm/deb)

| Каталог                     | Назначение                                         |
| --------------------------- | -------------------------------------------------- |
| /usr/share/logstash         | Домашний каталог Logstash, бинарники и библиотеки. |
| /etc/logstash               | Основные *конфигурационные файлы.*                 |
| /etc/logstash/conf.d        | *Pipeline-конфигурации* (*.conf).                  |
| /var/lib/logstash           | Данные и persistent queue.                         |
| /var/log/logstash           | *Логи работы Logstash*.                            |
| /etc/logstash/jvm.options   | Настройки JVM (heap size и др.).                   |
| /etc/logstash/logstash.yml  | Глобальные настройки Logstash.                     |
| /etc/logstash/pipelines.yml | *Описание нескольких pipeline.*                    |
### Что важно помнить
- Конфигурации pipeline обычно лежат в conf.d.
    
- Глобальные настройки — в logstash.yml.
    
- При запуске Logstash читает pipelines.yml, который указывает, какие pipeline-файлы использовать.

# 2. Файлы конфигурации
В Logstash существует несколько уровней конфигурации, каждый со своим назначением.

```
┌───────────────────────────────────────────────────────────────────────┐
│                         CONFIGURATION FILES                           │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│   logstash.yml                                                        │
│   ├── Глобальные настройки                                            │
│   ├── Пути к папкам                                                   │
│   ├── Настройки очередей                                              │
│   ├── API мониторинга                                                 │
│   └── Логирование                                                     │
│                                                                       │
│   pipelines.yml                                                       │
│   ├── Список всех pipeline                                            │
│   ├── Для каждого: id, path.config, workers, queue.type               │
│   └── Позволяет запускать несколько pipeline                          │
│                                                                       │
│   conf.d/*.conf (или кастомные пути)                                  │
│   ├── input {}  ─ источники                                           │
│   ├── filter {} ─ обработка                                           │
│   └── output {} ─ получатели                                          │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

## 4.1 Pipeline configuration (`*.conf`)

**Назначение:** Описывает ==конкретный конвейер обработки данных== — от входа до выхода.

### Структура файла
```
input {
    # Источники данных — откуда забирать события
}

filter {
    # Обработка данных — как преобразовывать события
}

output {
    # Получатели данных — куда отправлять результат
}
```

---
### Input plugins (входные плагины)
**Определяют источники данных**, из которых Logstash получает события.

|Плагин|Описание|Типовой сценарий|
|---|---|---|
|`beats`|Приём данных от Beats (Filebeat, Metricbeat)|Логи с серверов|
|`file`|Чтение из файлов (как `tail -f`)|Локальные файлы логов|
|`stdin`|Ввод из стандартного ввода (консоли)|Тестирование и отладка|
|`http`|HTTP endpoint для приёма данных|Вебхуки, API|
|`kafka`|Чтение из топиков Kafka|Высоконагруженные системы|
|`tcp`/`udp`|Приём по протоколам TCP/UDP|Сетевые устройства, syslog|
|`redis`|Чтение из Redis|Буферизация между шперами|
|`generator`|Генерация тестовых событий|Нагрузочное тестирование|

**Пример с разными input:**
```ruby
input {
    beats {
        port => 5044
        host => "0.0.0.0"
    }
    
    file {
        path => "/var/log/myapp/*.log"
        start_position => "beginning"
        sincedb_path => "/dev/null"  # для тестов
    }
    
    http {
        port => 8080
        host => "0.0.0.0"
    }
}
```

### Filter plugins (фильтры)
**Преобразуют, обогащают и очищают события.**

#### Основные фильтры

| Фильтр        | Что делает                                                      | Пример использования                         |
| ------------- | --------------------------------------------------------------- | -------------------------------------------- |
| `grok`        | **Разбор текста по именованным шаблонам регулярных выражений**  | Парсинг access-логов Nginx/Apache            |
| `mutate`      | **Изменение, переименование, удаление полей**                   | Приведение типов, удаление лишних полей      |
| `date`        | Парсинг даты из поля и установка `@timestamp`                   | Логи с нестандартным форматом даты           |
| `geoip`       | *Добавление геоданных* (страна, город, координаты) по IP-адресу | Анализ географии посетителей                 |
| `json`        | **Разбор JSON-строки в** структурированные поля                 | Логи от приложений в JSON-формате            |
| `kv`          | **Разбор пар "ключ=значение"**                                  | Логи в формате `key1=value1 key2=value2`     |
| `dissect`     | ***Более быстрая альтернатива grok без использования regexp***  | Фиксированные разделители (пробелы, запятые) |
| `useragent`   | Определение браузера, ОС, устройства по User-Agent              | Анализ клиентов                              |
| `split`       | Разбиение одного события на несколько                           | Логи с несколькими строками                  |
| `clone`       | Клонирование события для параллельной обработки                 | Разные пути обработки для одного события     |
| `drop`        | Полное удаление события                                         | Отсев debug-логов из продакшена              |
| `prune`       | Удаление полей по чёрному/белому списку                         | Очистка чувствительных данных                |
| `fingerprint` | Создание хеша полей                                             | Дедупликация, анонимизация                   |

**Пример с несколькими фильтрами:**
```ruby
filter {
    # Парсим access-лог Apache
    grok {
        match => { "message" => "%{COMBINEDAPACHELOG}" }
    }
    
    # Добавляем геоданные по IP
    geoip {
        source => "clientip"
        target => "geo"
    }
    
    # Парсим дату
    date {
        match => [ "timestamp", "dd/MMM/yyyy:HH:mm:ss Z" ]
        target => "@timestamp"
    }
    
    # Удаляем исходное сообщение
    mutate {
        remove_field => ["message", "timestamp"]
    }
    
    # Переименовываем поля для единообразия
    mutate {
        rename => { "response" => "http_status_code" }
        rename => { "bytes" => "response_size" }
    }
    
    # Добавляем служебные поля
    mutate {
        add_field => {
            "[@metadata][index_prefix]" => "nginx"
            "env" => "production"
        }
    }
}
```

#### Пример парсинга через `grok`
**Исходный лог:**
```bash
192.168.1.100 - - [10/Jun/2026:15:30:45 +0000] "GET /api/users HTTP/1.1" 200 1234 "https://example.com" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
```

**Конфигурация grok:**
```ruby
grok {
    match => {
        "message" => "%{IPORHOST:client_ip} - %{DATA:ident} - %{DATA:auth} \[%{HTTPDATE:timestamp}\] \"%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}\" %{NUMBER:response} %{NUMBER:bytes} \"%{DATA:referrer}\" \"%{DATA:user_agent}\""
    }
}
```

**Результат после парсинга:**
```json
{
    "client_ip": "192.168.1.100",
    "ident": "-",
    "auth": "-",
    "timestamp": "10/Jun/2026:15:30:45 +0000",
    "method": "GET",
    "request": "/api/users",
    "http_version": "1.1",
    "response": "200",
    "bytes": "1234",
    "referrer": "https://example.com",
    "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
}
```

### Output plugins (выходные плагины)
Определяют, **куда отправлять обработанные события.**

| Плагин          | Описание                                           | Типовой сценарий                |
| --------------- | -------------------------------------------------- | ------------------------------- |
| `elasticsearch` | **Отправка в Elasticsearch**                       | Основной output для хранения    |
| `stdout`        | **Вывод в консоль** (обычно в формате `rubydebug`) | Отладка                         |
| `file`          | **Запись в файл**                                  | Резервное копирование           |
| `kafka`         | **Отправка в Kafka**                               | Дальнейшая обработка            |
| `s3`            | Сохранение в AWS S3                                | Долгосрочное хранение           |
| `tcp`/`udp`     | **Отправка по сети**                               | Интеграция со старыми системами |
| `null`          | Игнорирование событий                              | Профилирование, тесты           |
| `redis`         | Сохранение в Redis                                 | Буферизация                     |
| `http`          | **Отправка на HTTP endpoint**                      | Вебхуки                         |

**Пример output:**
```ruby
output {
    # Основной вывод в Elasticsearch
    elasticsearch {
        hosts => ["localhost:9200"]
        index => "logs-%{+YYYY.MM.dd}"
        user => "logstash_internal"
        password => "${LOGSTASH_PASSWORD}"
        # Таймауты и повторные попытки
        retry_max_interval => 60
        retry_initial_interval => 2
    }
    
    # Отладка: выводим первые 10 событий в консоль
    stdout {
        codec => rubydebug
    }
    
    # Отправляем ошибки в отдельный индекс
    if [log_level] == "ERROR" {
        elasticsearch {
            hosts => ["localhost:9200"]
            index => "errors-%{+YYYY.MM.dd}"
        }
    }
}
```

### Условные операторы (conditionals)
**Позволяют применять фильтры и выходы выборочно**, в зависимости от содержимого события.
```ruby
# Синтаксис
if [поле] == "значение" {
    # действия
} else if [поле] =~ /регулярное выражение/ {
    # действия
} else {
    # действия
}
```

**Примеры:**
```ruby
filter {
    # Обрабатываем только определённые логи
    if [type] == "nginx-access" {
        grok { match => { "message" => "%{COMBINEDAPACHELOG}" } }
    }
    
    # Разная обработка для разных уровней
    if [level] == "DEBUG" {
        drop { }
    }
    
    if [level] == "ERROR" {
        mutate { add_tag => ["critical"] }
    }
}

output {
    # Разные индексы для разных типов
    if [type] == "nginx-access" {
        elasticsearch { index => "nginx-%{+YYYY.MM.dd}" }
    } else if [type] == "app-log" {
        elasticsearch { index => "app-%{+YYYY.MM.dd}" }
    }
    
    # Ошибки отправляем в отдельную систему
    if [level] == "ERROR" {
        http {
            url => "https://alerts.example.com/webhook"
            http_method => "post"
        }
    }
}
```

### Полный пример pipeline
```ruby
input {
    beats {
        port => 5044
    }
}

filter {
    # Парсим JSON (если приложение пишет в JSON)
    json {
        source => "message"
        target => "parsed"
        skip_on_invalid_json => true
    }
    
    # Если не JSON, пробуем grok для access-лога
    if [parsed][message] == "" {
        grok {
            match => {
                "message" => "%{COMBINEDAPACHELOG}"
            }
            tag_on_failure => ["_grokparsefailure"]
        }
    }
    
    # Обогащаем геоданными
    if [clientip] {
        geoip {
            source => "clientip"
            target => "geoip"
            fields => ["city_name", "country_name", "location"]
        }
    }
    
    # Обработка даты
    if [timestamp] {
        date {
            match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]
            target => "@timestamp"
        }
    }
    
    # Удаляем служебные поля
    mutate {
        remove_field => ["message", "timestamp", "ident", "auth"]
    }
    
    # Добавляем метаданные
    mutate {
        add_field => {
            "[@metadata][environment]" => "production"
            "processed_by" => "logstash-01"
        }
    }
}

output {
    elasticsearch {
        hosts => ["elasticsearch:9200"]
        index => "logs-%{+YYYY.MM.dd}"
    }
    
    stdout {
        codec => rubydebug
    }
}
```

## 4.2 `pipelines.yml`

Позволяет **запускать несколько независимых pipeline одновременно** в одном экземпляре Logstash.

```yaml
# pipelines.yml
- pipeline.id: nginx-logs
  path.config: "/etc/logstash/conf.d/nginx/*.conf"
  pipeline.workers: 2
  pipeline.batch.size: 125
  
- pipeline.id: app-logs
  path.config: "/etc/logstash/conf.d/app/*.conf"
  pipeline.workers: 4
  pipeline.batch.size: 250
  queue.type: persisted
  queue.max_bytes: 2gb
  
- pipeline.id: audit-logs
  path.config: "/etc/logstash/conf.d/audit/*.conf"
  pipeline.workers: 1
  pipeline.batch.size: 50
```

### Параметры pipeline

|Параметр|Назначение|
|---|---|
|`pipeline.id`|Уникальный идентификатор pipeline (обязательный)|
|`path.config`|Путь к `.conf` файлам этого pipeline|
|`pipeline.workers`|Количество worker-потоков|
|`pipeline.batch.size`|Размер батча событий|
|`queue.type`|Тип очереди (`memory` или `persisted`)|
|`queue.max_bytes`|Максимальный размер очереди|
|`config.string`|Конфигурация в виде строки (альтернатива файлу)|

### Зачем нужно несколько pipeline

|Сценарий|Объяснение|
|---|---|
|**Разделение потоков данных**|Логи, метрики, аудит — разные конвейеры|
|**Изоляция конфигураций**|Ошибка в одном не валит все|
|**Разные настройки производительности**|Для важных потоков больше ресурсов|
|**Разные очереди**|Для критичных данных — persistent queue|
|**Поэтапное обновление**|Можно перезапустить только один pipeline|

## 4.3 `logstash.yml`
**Глобальные настройки всего экземпляра Logstash.**

### Важные параметры

| Параметр                  | Значение по умолчанию         | Назначение                             |
| ------------------------- | ----------------------------- | -------------------------------------- |
| `path.config`             | `/etc/logstash/conf.d/*.conf` | Путь к pipeline-конфигам               |
| `pipeline.workers`        | Число CPU ядер                | Количество worker-потоков              |
| `pipeline.batch.size`     | `125`                         | Размер батча событий                   |
| `pipeline.batch.delay`    | `50`                          | Задержка в мс перед отправкой батча    |
| `pipeline.ordered`        | `true`                        | Сохранять порядок событий              |
| `queue.type`              | `memory`                      | Тип очереди (`memory` или `persisted`) |
| `queue.max_bytes`         | `1gb`                         | Максимальный размер persistent queue   |
| `queue.page_capacity`     | `64mb`                        | Размер страницы в persistent queue     |
| `path.queue`              | `data/queue`                  | Путь для persistent queue              |
| `config.reload.automatic` | `false`                       | Автоперезагрузка конфигов              |
| `config.reload.interval`  | `3s`                          | Интервал проверки изменений            |
| `http.host`               | `127.0.0.1`                   | Адрес для API мониторинга              |
| `http.port`               | `9600`                        | Порт для API мониторинга               |
| `log.level`               | `info`                        | Уровень логирования                    |
| `path.logs`               | `LOGS_HOME`                   | Путь к логам Logstash                  |
| `path.data`               | `DATA_HOME`                   | Путь к данным (очереди и т.д.)         |

### Пример `logstash.yml`
```yaml
# logstash.yml

# Пути
path.config: /etc/logstash/conf.d/*.conf
path.data: /var/lib/logstash
path.logs: /var/log/logstash
path.queue: /var/lib/logstash/queue

# Pipeline настройки
pipeline.workers: 4
pipeline.batch.size: 250
pipeline.batch.delay: 10
pipeline.ordered: true

# Очередь (persisted для надёжности)
queue.type: persisted
queue.max_bytes: 5gb
queue.page_capacity: 128mb
queue.drain: true  # Очищать очередь при остановке

# Перезагрузка конфигов (только для разработки!)
config.reload.automatic: false
config.reload.interval: 10s

# API мониторинга
http.host: 0.0.0.0
http.port: 9600

# Логирование
log.level: info
log.format: json
```
