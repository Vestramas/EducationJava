## Часть 1: Управление жизненным циклом проекта через Maven (`mvn`)

Maven построен вокруг концепции **жизненного цикла** (Build Lifecycle). Это последовательность фаз, которые определяют стадию сборки проекта.

Самая важная команда, которую вам нужно запомнить:
```bash
mvn <фаза_1> <фаза_2> ...
```

### Основные фазы жизненного цикла (стандартные команды)

| Команда          | Что делает                                                                                                                                                                                                                                                                                                                                                                   | Результат                                    |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `mvn clean`      | Удаляет директорию `target` (**очищает проект от старых файлов сборки**) [](https://opensource.ncsa.illinois.edu/confluence/plugins/viewsource/viewpagesrc.action?pageId=108104822).                                                                                                                                                                                         | Исчезает папка `target`                      |
| `mvn compile`    | **Компилирует исходный код** (`src/main/java`) в байт-код.                                                                                                                                                                                                                                                                                                                   | Появляются `.class` файлы в `target/classes` |
| `mvn test`       | Запускает тесты (обычно JUnit) [](https://issues.apache.org/jira/browse/FALCON-2024?page=com.atlassian.streams.streams-jira-plugin%3Aactivity-stream-issue-tab)[](https://issues.apache.org/jira/browse/SUREFIRE-1171?focusedCommentId=17994579&page=com.atlassian.jira.plugin.system.issuetabpanels%3Acomment-tabpanel#comment-17994579). Компилирует тесты и выполняет их. | Отчет о прохождении тестов в консоли         |
| `mvn package`    | **Собирает скомпилированный код в дистрибутив** <br>(JAR, WAR и т.д.).                                                                                                                                                                                                                                                                                                       | Файл в папке `target` (например, `app.jar`)  |
| `mvn install`    | **Устанавливает собранный пакет в локальный репозиторий** (`~/.m2/repository`). <br><br>Теперь его могут использовать другие ваши Maven-проекты [](https://opensource.ncsa.illinois.edu/confluence/plugins/viewsource/viewpagesrc.action?pageId=108104822).                                                                                                                  | Пакет появляется в локальном репозитории     |
| `mvn deploy`     | Загружает пакет на удаленный репозиторий **(например, Nexus/Artifactory) для** общего доступа [](https://opensource.ncsa.illinois.edu/confluence/plugins/viewsource/viewpagesrc.action?pageId=108104822).                                                                                                                                                                    | Пакет опубликован на сервере                 |
| **`mvn verify`** | **Выполняет проверки интеграционных тестов** (стоит между `package` и `install`).                                                                                                                                                                                                                                                                                            | Проверка качества                            |

**Важное правило:** Когда вы вызываете фазу, Maven автоматически выполняет **все предыдущие фазы**.
- `mvn package` → сначала сделает `validate`, `compile`, `test`, и только потом `package`.

### Параметры команды `mvn`
|Параметр|Пример|Описание|
|---|---|---|
|`-Dproperty=value`|`mvn package -DskipTests`|Определяет системное свойство или параметр плагина.|
|`-Pprofile`|`mvn package -Pproduction`|Активирует профиль сборки, указанный в `pom.xml`.|
|`-T`|`mvn -T 4 test`|Включает параллельную сборку модулей (количество потоков).|
|`-X`|`mvn clean -X`|Включает debug-логи (выводит детальную информацию о ходе сборки).|
|`-pl` (module)|`mvn package -pl module-name`|Собрать только конкретный модуль (в многомодульном проекте).|
|`-am` (also-make)|`mvn package -pl module-name -am`|Собрать модуль и те, от которых он зависит.|
|`-o`|`mvn package -o`|Работа в оффлайн-режиме (не пытаться качать зависимости из сети).|

### Распространенные плагины и их вызов (утилиты Maven)

Maven — это набор плагинов. Иногда нужно вызвать конкретную цель (goal) плагина напрямую, минуя стандартный жизненный цикл.

---
#### 1. Запуск Java-программы (Exec Maven Plugin) [](https://www.mojohaus.org/exec-maven-plugin/exec-mojo.html)

Используется для запуска вашего приложения прямо из исходников, без ручной сборки JAR.
```bash
# Базовый запуск (mainClass должен быть указан в <configuration> pom.xml)
mvn exec:java

# Запуск с указанием класса и аргументов (работает без дополнительной настройки)
mvn exec:java -Dexec.mainClass="com.example.Main" -Dexec.args="arg1 arg2"

# Запуск в фоновом режиме (полезно для серверов)
mvn exec:exec -Dexec.executable="java" -Dexec.args="-jar target/app.jar"
```

---
#### 2. Сборка "Толстого" (Fat/Uber) JAR (Assembly/Shade Plugins)

Если ваш проект зависит от множества библиотек, обычный `mvn package` сделает JAR только с вашими классами. 

Для запуска через `java -jar` ==нужно собрать один файл, содержащий все зависимости.==

**Вариант А: Assembly Plugin (простой, может конфликтовать файлами)**
```bash
# Выполняем прямо из командной строки, не трогая pom.xml
mvn assembly:single -DdescriptorId=jar-with-dependencies
```

> _Нюанс:_ Если в разных библиотеках есть файлы с одинаковым именем (например, `META-INF/spring.schemas`), они могут затереть друг друга [](https://browse.library.kiwix.org/content/stackoverflow.com_en_all_2023-11/questions/1015520/java-how-do-i-build-standalone-distributions-of-maven-based-projects)[](https://www.programmersought.com/article/27912046167/).

---
**Вариант Б: Shade Plugin (рекомендуемый, умный)** [](https://www.programmersought.com/article/27912046167/)  
Требует добавления в `pom.xml`, но решает проблемы с конфликтами файлов.
```bash
mvn package shade:shade
```

---
#### 3. Копирование зависимостей (Dependency Plugin)

Используется, чтобы собрать все JAR-библиотеки в одну папку (например, `lib`), чтобы потом прописать их в `classpath`.
```bash
mvn dependency:copy-dependencies
# Все библиотеки появятся в target/dependency
```

---
#### 4. Запуск тестов в параллели (Surefire Plugin) [](https://issues.apache.org/jira/browse/FALCON-2024?page=com.atlassian.streams.streams-jira-plugin%3Aactivity-stream-issue-tab)[](https://issues.apache.org/jira/browse/SUREFIRE-1171?focusedCommentId=17994579&page=com.atlassian.jira.plugin.system.issuetabpanels%3Acomment-tabpanel#comment-17994579)

Для ускорения работы тестов можно управлять количеством потоков.
```bash
# -T означает количество потоков (2 потока, либо 2C - 2 ядра ЦП)
mvn -T 2 test

# Параметры для параллельного запуска внутри surefire (настраиваются в pom.xml)
# <forkCount>4</forkCount> - количество JVM процессов
# <parallel>methods</parallel> - параллельно запускать методы тестов
```

## Часть 2: Запуск через утилиты JDK (без Maven)

==Когда проект собран== (`mvn package`), Maven больше не нужен. 

**Вы используете стандартную команду** `java` из состава JDK. Ключевое различие между подходами — способ указания `classpath` и точки входа [](https://browse.library.kiwix.org/content/stackoverflow.com_en_all_nopic_2022-07/questions/23029076/the-difference-java-cp-java-jar).

### Режим 1: Запуск класса из обычного JAR (через `-cp` / `-classpath`)

Этот метод используется, когда в JAR-файле не прописан `Main-Class` в манифесте или вы хотите запустить другой класс.

**Формат:**
```bash
java -cp "путь_к_JAR_файлу" com.example.YourMainClass
```

#### **Практические примеры:**
**Простой проект (один JAR):**
```bash
java -cp target/my-app-1.0.jar com.example.Main
```


**Проект с библиотеками (JAR + папка lib):**
```bash
# Linux/Mac (разделитель :)
java -cp "target/my-app-1.0.jar:target/lib/*" com.example.Server

# Windows (разделитель ;)
java -cp "target/my-app-1.0.jar;target/lib/*" com.example.Server
```

- _Символ `*` подгружает все JAR в указанной папке._

---
### Режим 2: Исполняемый (Executable) JAR (через `-jar`)

Используется для готовых приложений. Требует, чтобы в `MANIFEST.MF` внутри JAR были указаны `Main-Class` и опционально `Class-Path`.

**Формат:**
```bash
java -jar app.jar
```

**Как получить такой JAR через Maven:**  
Настройте плагин `maven-jar-plugin` в `pom.xml`:
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <version>3.3.0</version>
    <configuration>
        <archive>
            <manifest>
                <mainClass>com.example.Main</mainClass>
                <addClasspath>true</addClasspath> <!-- Добавляет зависимости в classpath манифеста -->
            </manifest>
        </archive>
    </configuration>
</plugin>
```

После этого собираем и запускаем:
```bash
mvn package
java -jar target/my-app-1.0.jar
```

> **Важно:** Нельзя одновременно использовать `-cp` и `-jar`. Опция `-jar` заставляет Java игнорировать параметр `-cp` и полагаться только на `Class-Path` из манифеста внутри JAR [](https://browse.library.kiwix.org/content/stackoverflow.com_en_all_nopic_2022-07/questions/12047758/java-using-cp-and-jar-together)[](https://browse.library.kiwix.org/content/stackoverflow.com_en_all_nopic_2022-07/questions/23029076/the-difference-java-cp-java-jar).

---
### Режим 3: Fat/Uber JAR (Лучшее решение для сложных проектов)

Этот метод объединяет достоинства двух предыдущих. Вы получаете ОДИН большой JAR, содержащий и ваш код, и все библиотеки. Его можно запускать через `java -jar`, и он не зависит от внешних файлов.

**Сборка через Assembly (простая):**
```bash
mvn assembly:single -DdescriptorId=jar-with-dependencies

# Запуск:
java -jar target/my-app-1.0-jar-with-dependencies.jar
```

**Сборка через Shade (профессиональная):**  
(Требует добавления плагина в pom.xml, но лучше решает конфликты).
```bash
mvn package
java -jar target/my-app-1.0-shaded.jar
```
