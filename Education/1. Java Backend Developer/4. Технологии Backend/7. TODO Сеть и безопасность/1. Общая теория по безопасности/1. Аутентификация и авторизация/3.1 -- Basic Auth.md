**Basic Auth** — это стандартный метод аутентификации в HTTP, описанный в RFC 7617. Он позволяет клиенту передать серверу свои учётные данные для доступа к защищённому ресурсу.

Для этого используются заголовок и иногда тело HTTP запроса
### 1. **Заголовки HTTP** - Authorization: Basic
Со стороны **клиента** (передача учётных данных):  
    `Authorization: Basic <base64_encoded_string>`

Значение по этому заголовку проверяется сервером.
### 2. **Формат данных для кодирования**
**Строгий формат:** `логин:пароль`  
    Пример: `alex:myPassword123`

### 3. **Кодирование**
**Алгоритм:** Base64 (RFC 4648) (НЕ ШИФРОВАНИЕ, а просто кодирование - легко получить исходные данные)

**Пример преобразования:**
```
Исходная строка: "alex:myPassword123"

Base64: "YWxleDpteVBhc3N3b3JkMTIz"
```

### 4. **Итоговый заголовок**
```
Authorization: Basic YWxleDpteVBhc3N3b3JkMTIz
```

---
**После первой успешной аутентификации** (не важно, каким способом — Basic, Form Login, OAuth2) **сервер создаёт HTTP-сессию** и помещает в неё `SecurityContext` с данными пользователя. Идентификатор этой сессии (`JSESSIONID`) отправляется браузеру в виде cookie.

Вот последовательность событий:
1. **Первый запрос:** Браузер отправляет `Authorization: Basic ...`. Сервер аутентифицирует пользователя, создаёт сессию и сохраняет в ней `SecurityContext`. В ответе приходит cookie `JSESSIONID`.
    
2. **Второй запрос:** Браузер отправляет **и** `Authorization: Basic ...`, **и** cookie `JSESSIONID`. Сервер видит, что cookie указывает на существующую сессию, находит в ней уже аутентифицированный `SecurityContext` и использует его. `BasicAuthenticationFilter` всё ещё срабатывает, но `SecurityContextHolderFilter` уже загрузил контекст из сессии, и повторная проверка пароля не требуется.
