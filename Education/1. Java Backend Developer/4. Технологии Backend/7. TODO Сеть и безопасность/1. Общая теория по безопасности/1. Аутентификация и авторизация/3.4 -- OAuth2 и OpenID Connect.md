# 1. История OAuth
`OAuth` появился как решение насущной проблемы: до него *сторонние*
*приложения должны были запрашивать логин и пароль от аккаунта*
*пользователя*, чтобы получить доступ к данным.

B 2007 году появилась первая версия протокола - `0Auth1.0`. Он был достаточно защищён, но сложен в реализации из-за криптографии и подписи запросов. 

В 2012 году вышел `0Auth2.0` - значительно проще при помощи Access и Refresh Token, с гибкой
архитектурой и множеством сценариев использования.

## Что решает OAuth
OAuth решает проблему делегирования доступа. 

*Вместо того чтобы* *пользователь вводил пароль в стороннее приложение*, **он авторизуется у надёжного сервиса** (например, Google), **подтверждает запрашиваемые права и выдаёт приложению ограниченный доступ.**

Таким образом, OAuth позволяет: 
- централизовать вход, 
- разделить уровни доступа,
- избавиться от хранения паролей в сторонних системах,
- обеспечить масштабируемую и безопасную авторизацию в распределённых системах.

> [!IMPORTANT] ИДЕЯ
> Весь **процесс получени/обновления токенов ЛЕЖИТ на стороннем сервисе**, где пользователь зарегистрирован

## Роли в OAuth2
Рассмотрим роли, определенные в OAuth 2.0:
### 1. **Resource Owner (Владелец учетных данных - аккаунта)**
**Проще говоря:** **ТЫ** — пользователь, у которого есть УЧЕТНЫЕ данные.
- **Что делает?** Даёт разрешение приложению использовать свои учетные данные
- **Пример:** Твой аккаунт в Google/Facebook
    
- **Аналогия:** Ты — владелец квартиры (данных). Ты решаешь, кому дать ключ
![[Pasted image 20260112192957.png]]
### 2. **Resource Server (Сервер ресурсов)**
**Проще говоря:** ==ХРАНИЛИЩЕ данных==, которое защищено
- **Что делает?** Хранит твои данные и отдаёт их только по пропуску (токену)
    
- **Пример:** Сервер Google (хранит твои контакты, фото)
    
- **Аналогия:** Сама квартира с замком. Без ключа не пускает
![[Pasted image 20260112193114.png]]
### 3. Authorization Server
Сервис, проверяющий личность и выдающий токены
![[Pasted image 20260112193040.png]]
### 4. **Client (Клиент)**
**Проще говоря:** **Приложение**, которое хочет получить доступ к твоим данным
- **Что делает?** Просит у тебя разрешение, потом получает доступ
    
- **Пример:** Spotify (хочет получить твой список друзей из Facebook)
![[Pasted image 20260112193024.png]]
#### **Клиенты бывают двух типов:**

```
Пример: Бэкенд веб-приложения (серверная часть)
Хранит СЕКРЕТЫ безопасно: никто их не видит
```

##### 📢 **Public Client (Публичный)**
```

Работает в браузере, все видят его код
Не может хранить секреты — их украдут

Пример №2: Мобильное приложение
Установлено на телефоне, можно декомпилировать
Тоже не может хранить секреты
```

### Если коротко: 
- **web-application client** — confidential клиент. Например, backend web-приложение, которое взаимодействует Resource owner с помощью user-agent-а (web-браузера и т.д.);
    
- **user-agent-based application** — public клиент, весь код которого выполняется на пользовательском user-agent-е. SPA — отличный пример такого типа клиента;
    
- **native application** — также public клиент. Нативное приложение, которое устанавливается на устройство пользователя. Например, любое нативное приложение, установленное к вам на телефон.

# 2. Потоки авторизации OAuth 2.0:
Популярные авторизационные потоки 0Auth 2.0:
- Authorization Code Flow
- Implicit Flow
- Client Credentials Flow
- Device Code Flow
## 1. **Authorization Code Flow** (Самый безопасный и распространенный)

**Authorization code** — этот flow ==основан на редиректах==. 

![[Pasted image 20260421095443.png]]
RFC 6749 OAuth 2.0 October 2012

Поток, показанный на рисунке 3, включает следующие шаги:
**(A)** Клиент инициирует поток, направляя user-agent владельца ресурса к authorization endpoint. 
Клиент передает свой client identifier, запрашиваемую область доступа (scope), локальное состояние (state), а также redirection URI, на который сервер авторизации должен  
вернуть user-agent после того, как доступ будет предоставлен  
(или отклонен).

**(B)** Сервер авторизации аутентифицирует владельца ресурса  
(через user-agent) и определяет, разрешает ли владелец ресурса  
клиенту доступ к запрашиваемому ресурсу или отказывает в нем.

**(C)** ***Если владелец ресурса предоставляет доступ***, сервер авторизации перенапправляет user-agent обратно к клиенту, используя ранее указанный redirection URI (в запросе или при регистрации клиента).  
Этот redirection URI включает authorization code, а также любое локальное состояние (state), переданное клиентом ранее.

**(D)** Клиент запрашивает access token у token endpoint сервера  
авторизации, передавая authorization code, полученный на  
предыдущем шаге. При выполнении этого запроса клиент проходит  
аутентификацию на сервере авторизации. 
Также клиент включает redirection URI, использованный для получения authorization code, чтобы сервер мог выполнить проверку.

**(E)** Сервер авторизации аутентифицирует клиента, проверяет  
authorization code и убеждается, что полученный redirection URI совпадает с URI, использованным для перенаправления клиента  
на шаге (C). Если все данные корректны, сервер авторизации  
возвращает access token и, при необходимости, refresh token.

### РКСЕ и мобильные приложения
PKCЕ (Proof Key for Code Exchange) - это механизм безопасности, защищающий Authorization Code Flow на устройствах, где нельзя хранить клиентский секрет (мобильные приложения, SPA).

Приложение генерирует случайную строку и хеширует её. Хеш отправляется в запросе, а при обмене кода на токен сервер проверяет, совпадает ли оригинал с присланным хешем.

Если злоумышленник украдёт код авторизации, он не сможет использовать его без оригинальной строки. Благодаря РКСЕ можно безопасно использовать OAuth даже там, где раньше требовался backend.

Алгоритм работы flow следующий:
1. Происходит пользовательский интент на авторизацию. Например, нажатие на кнопку Login.
    
2. Целевое приложение генерирует случайный code_verifier и из него code_challenge. Сохраняет их в приложении (подробнее в [RFC-7636](https://datatracker.ietf.org/doc/html/rfc7636)).
    
3. Приложение делает редирект в браузер на эндпоинт *сервера авторизации* (**ТАМ ГДЕ ПОЛЬЗОВАТЕЛЬ УЖЕ ЗАРЕГИСТРИРОВАН**) вместе со сгенерированным ранее code_challenge. 
    
4. *Сервер авторизации* (**ОТКУДА ПОЛЬЗОВАТЕЛЬ ХОЧЕТ ПРОИЗВЕСТИ ВХОД**) также запоминает code_challenge для данной сессии. Соль в том, что code_verifier остается только внутри приложения и не передается по сети. 
    
5. Пользователь вводит логин и пароль на странице сервера авторизации. При необходимости определяет список разрешений, которые будут выданы целевому приложению. 
    
6. Сервер авторизации делает обратный редирект на клиент и возвращает код авторизации. При этом code_challenge обратно не возвращается. 
    
7. Приложение обменивает код на токен(-ы) практически точно так же, как при  Authorization Flow, но дополнительно передает code_verifier, который был сохранен в пункте 2
    
8. Сервер авторизации вычисляет code_challenge на основе данных пайлоада запроса на получение токенов и сравнивает с code_challenge, который был отправлен в пункте 3. Если значения совпадают, возвращает набор токенов для доступа к защищенным ресурсам.

Заметки
- **`code_verifier`** — длинная случайная строка, **никогда не покидает клиентское приложение** (сохраняется в памяти, локальном хранилище) до момента обмена кода на токен (шаг 7).
    
- **`code_challenge`** — производная от `code_verifier` (чаще всего его SHA256 хэш, преобразованный в base64url). **Передается публично** при начальном запросе (шаг 3).
    
- **Валидация (шаг 8):** Сервер авторизации хранит `code_challenge` с момента шага 3. Когда приложение присылает `code_verifier` (шаг 7), сервер вычисляет хэш от него по тому же алгоритму и сравнивает с сохраненным `code_challenge`. Если они совпали — значит, тот, кто инициировал запрос (шаг 3), тот же самый и обменивает код на токен (шаг 7). Это защищает от перехвата кода авторизации (`Authorization Code`).

## 2. Implicit - устаревший
**Implicit** — первая попытка адаптировать OAuth для **публичных клиентов** (SPA, мобильные приложения), которые не могут безопасно хранить секреты.

**Ключевая идея:**
> "Если публичный клиент не может безопасно обменивать код на токен (ибо нет секрета), давайте сразу отдадим токен!"

![[Pasted image 20260111210729.png]]

Основное отличие в том, что после успешной авторизации resource owner вместо получения authorization-code и его обмена на access token, *сервер авторизации сразу возвращает access token* **в query в redirect URI.** 
```http
https://myapp.com/callback#access_token=eyJhbGci...
```
- Видно в истории браузера
- Может попасть в логи серверов
- JavaScript может его прочитать (риск XSS-атак)

Сейчас основная рекомендация стандарта — использовать Authorization Code Flow with PKCE (Proof Key for Code Exchange). Он более безопасен: не передает токен в открытом виде, а еще позволяет возвращать и использовать клиенту refresh токен.

### **Ключевое отличие от Authorization Code Flow:**
|                                  | **Implicit Flow (старый)**               | **Authorization Code Flow (современный)** |
| -------------------------------- | ---------------------------------------- | ----------------------------------------- |
| **Что получает клиент сначала?** | Сразу `access_token`                     | `authorization_code` (просто код)         |
| **Где токен?**                   | В URL-фрагменте (`#token=...`)           | Нигде не виден, только на бэкенде         |
| **Refresh token?**               | ❌ Нельзя использовать                    | ✅ Можно (с PKCE)                          |
| **Безопасность**                 | ⚠️ Токен виден в истории браузера, логах | 🔒 Токен только в защищённом обмене       |

## 3.Client credentials 
**Client credentials** — этот flow подходит только для Machine-to-Machine (M2M) приложений (кроны, CLis, backend сервисы). 

Напоминает «обычную» авторизацию, которая выполняется на основе client_id и client_secret: по сути, логин и пароль, в случае пользователя. 

Клиент выполняет запрос на сервер авторизации с client_id и client_secret и в ответ получает access token для доступа к запрашиваемому ресурсу.

# 3. Scopes и Permissions OAuth 2.0:
**Scopes и Permissions** — это система "разрешений" как в смартфоне, когда приложение просит:
- "Доступ к геолокации"
- "Доступ к камере"
- "Доступ к контактам"

Только в мире OAuth это работает для API и данных.

![[Pasted image 20260111212003.png]]

## **Scope (Область) — "ЧТО?"**
**Что это:** Общая категория данных или функций  

**Примеры:**
- `profile` — доступ к профилю
    
- `email` — доступ к email
    
- `calendar` — доступ к календарю
    
- `drive` — доступ к файлам


**Как выглядит в запросе:**
```
https://oauth.com/authorize?
  client_id=...&
  scope=profile+email+calendar.read
```

## **Permission (Разрешение) — "ЧТО МОЖНО СДЕЛАТЬ?"**
**Что это:** Конкретное действие внутри scope  

**Примеры внутри scope `calendar`:**
- `calendar:read` — только читать события
    
- `calendar:write` — создавать/изменять события
    
- `calendar:delete` — удалять события

**Обычно объединяются:**
- `calendar.read` = scope + permission
    
- `calendar.readonly` = только чтение
    
- `calendar` = полный доступ (читай, пиши, удаляй)

## **Что видит пользователь:**
Пользователь перед авторизацией видит список запрашиваемых прав и может принять или отклонить.
```
Приложение "Планировщик встреч" запрашивает доступ к:

✅ Основная информация профиля
  • Ваше имя и фото

✅ Электронная почта
  • Просмотр вашего email-адреса

✅ Календарь
  • Чтение событий календаря
  • Создание новых событий
  • Изменение существующих событий
  └── [x] Только чтение (рекомендуется)
      [ ] Полный доступ

[ Разрешить ] [ Отклонить ]
```

Пользователь **может изменить уровень доступа**! Часто можно выбрать:
- "Только чтение" вместо "Чтение и запись"
    
- "Только базовый профиль" вместо "Полный профиль"

## **Принцип наименьших привилегий (Least Privilege)**
**Философия:** "Давать минимально необходимый доступ для работы"
### ❌ **Плохо:**
```java
// Приложение для показа аватарок просит:

scope = "profile email calendar contacts drive admin"
// Зачем аватаркам доступ к календарю и диску?!
```

### ✅ **Хорошо:**
```java
// Приложение для показа аватарок просит:
scope = "profile.readonly"
// Только чтение публичной информации профиля
```

# 4. OpenID Connect и identity layer
OpenID Connect (OIDC) **расширяет возможности OAuth 2.0**, добавляя аутентификацию пользователя. 

Вместо простой передачи доступа к ресурсам как это делает OAuth, OIDC позволяет стороннему приложению узнать, кто именно вошёл в систему.

## ID Token в OIDC
==В основе OpenID Connect лежит концепция «ID Tokens»==. 
Как раз таки в ней и возвращается аутентификационная ***информация о пользователе.*** 
> **При запросе токенов клиент вместе с access и refresh** токенами **получает также** и `ID-token`.

Помимо этого, позволяет реализовать Single Sign-On (SSO) и тем самым дает возможность входить в несколько приложений/несколько веб-сайтов с использованием единовременной аутентификации пользователя**.** Например, сервис посредством OpenID может определить, что зашедший пользователь – именно Иванов Иван Иваныч, определенного _identity provider._

Другими словами (заглянул в Wikipedia):

> **OpenID** —  открытый стандарт децентрализованной системы аутентификации, предоставляющей пользователю возможность создать единую учетную запись для аутентификации на множестве не связанных друг с другом интернет-ресурсов, используя услуги третьих лиц. Информацию о пользователе, вошедшем в систему (о Resource Owner'е), часто называют  личными данными [identity], а Authorization Server в контексте OIDC — поставщиком личных данных  **IDP.**

## Что такое ID Token
ID Token - это JSON Web Token (JWT), подписанный сервером авторизации. 
> Он **содержит сведения о пользователе**: от уникального ID до имени и email.

> [!IMPORTANT] ID Token должен быть ВСЕГДА подписан и даже зашифрован
> 

**Обязательными являются поля**
- Поле `sub` - содержит уникальный ID пользователя
- `is` - адрес сервера авторизации
- `auth` - идентификатор клиента (которое делает запрос)
- стандартные поля (`iat` - время выпуска, `exp` - срок действия)

Его структура стандартизирована и понятна клиентским приложениям. После получения токена приложение может сразу узнать, кто вошёл в систему, не обращаясь к серверу заново. Это ключевое отличие OpenID Connect от OAuth.

## Разница между OAuth и OpenID Connect
Главное отличие - в цели. 
- *OAuth 2.0 делегирует доступ:* «пользователь разрешил приложению читать его почту». 
- **OIDC же отвечает на вопрос: «а кто этот пользователь вообще?»**

C помощью OpenID Connect приложение получает ID Token - специальный токен, содержащий сведения о личности. 

Это превращает протокол в полноценный способ входа и часто заменяет локальную регистрацию.

OIDC добавляет:
- Аутентификацию
- ID Token
- UserInfo Endpoint

## Identity Layer в OpenID Connect
Под "identity layer" понимается дополнительный уровень, который
превращает 0Auth 2.0 в протокол аутентификации. 

Он **включает правила обработки ID Token**, стандартные поля и механизмы подтверждения личности.

## UserInfo Endpoint
OpenID Connect также определяет специальный endpoint - userinfo. 
Через него приложение **может получить дополнительную информацию о пользователе**, используя access token.
