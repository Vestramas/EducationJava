# 1. Режим работы (dev VS prod)
> [!NOTE] Установка режима запуска
> Для установки режима запуска используется команда `start-dev` или `start-prod`
> 
> Поэтому в compose.yaml указали - `command: start-dev`

## Dev-режим
`Dev` - предназначен **для локальной разработки** и тестирования.
- **Упрощенный запуск:** автоматически применяются настройки, не требующие сложной конфигурации.
    
- **Включён HTTP**: можно запускать без HTTPS.
    
- **Отключена проверка сертификатов**: удобно для работы с самоподписанными сертификатами.
    
- **Включены developer-friendly endpoints**: например, доступ к /admin API может быть менее защищен.
    
- **Более подробные логи:** включая stack trace.
    
- **Слабые требования к паролям** и политике безопасности.
    
- **Нет ограничений на CORS** и Content-Security-Policy, если не заданы явно.
---
## Prod-режим
`prod` - предназначен **для развертывания в боевой среде (ПРОДАКШЕН)**.
- **Требуется HTTPS**: нельзя запустить без конфигурации TLS.
    
- **Политики безопасности применяются строго** (например, CSP, CORS, защита от CSRF).
    
- **Проверка конфигурации при старте**: ошибки в настройке вызовут отказ запуска.
    
- **Пароли и другие секреты не должны быть слабыми**.
    
- **Отключены "удобные" фичи,** потенциально опасные в проде (dev endpoints и т.д.).
    
- ==Подразумевается использование внешней базы данных== (а не H2).
    
- Повышенные требования к производительности и отказоустойчивости.

---
# 2. Reaml и master-realm

> [!IMPORTANT] Reaml
> `Realm` - это **область**, которая ==УПРАВЛЯЕТ НАБОРОМ ПОЛЬЗОВАТЕЛЕЙ, их учетными данными==, ролями и группами. 
> 
> Пользователь принадлежит определённой области (realm) и входит в нее при входе в систему. 
> 
> **Области изолированы друг от друга** и могут управлять и аутентифицировать только тех пользователей, которыми они владеют

## Master-realm - админский realm для других realm 
> [!IMPORTANT] master-reaml
> master - это **главный (административный) realm**, предназначенный **для:** 
> 
> **Управления другими realms**
> - Только через master realm можно:
> 	- создавать, удалять и настраивать другие realms;
> 	- управлять пользователями-администраторами, которые имеют доступ ко всем или выбранным realms.
> 
> **Создание глобальных админов** 
> - Пользователи master realm могут быть назначены ролью realm-admin, admin, create-realm и т. д.
> - Такие пользователи могут входить в Keycloak Admin Console (/admin) и управлять всем сервером.
> 

Кроме того master-reaml используется самим Keycloak для внутренних нужд. 
Некоторые *системные клиенты и сервисные аккаунты находятся в master realm.*
> Примеры: admin-cli, account, broker.

# 3. Clients
> [!IMPORTANT] Clients
> **Клиенты (Clients)** - это ==СУЩНОСТИ, КОТОРЫЕ МОГУТ ОТПРАВЛЯТЬ ЗАПРОС В KEYCLOAK на аутентификацию== пользователя.
> 
> Чаще всего клиентами **являются приложения и сервисы**, **которые хотят использовать Keycloak** для обеспечения безопасности и реализации единого входа (Single Sign-On). 
> 
> Также клиентами могут быть **системы**, **которым просто требуется идентификационная информация или access token,** чтобы безопасно вызывать другие сервисы в сети, защищённые через Keycloak.

## Клиенты по умолчанию
По умолчанию KeyCloak создает несколько клиентов
### Client: Account
**Назначение**: клиент *для пользовательского личного кабинета*.
	
**URL**: /realms/{realm}/account
	
**Используется**: ==когда пользователь хочет изменить профиль==, сменить пароль, просмотреть сеансы входа и разрешения.

---
### Client: Account-console
**Назначение**: ==фронтенд-приложение для клиента account==.
	
**Используется**: визуальный интерфейс для управления личным аккаунтом, вызывающий API account.

---
### Client: Admin-cli
**Назначение**: клиент для ***взаимодействия с Keycloak через CLI или REST API.***
	
**Типичный use-case**: получение токена через kcadm.sh или curl для скриптов/CI/CD.
> инструмент *для выполнения административных задач* ***без использования веб-админ-консоли***. 
	
**Грант**: поддерживает `client_credentials` и `password` гранты.

---
### Client: Broker
**Назначение**: внутренний *клиент для федерации идентичности* (SSO между провайдерами).
	
**Пример**: Keycloak как Identity Broker между Google, GitHub и другим Keycloak сервером.

---
### Client: Master-realm
**Назначение**: формальный *клиент, представляющий сам master realm*.
Часто используется **для определённых внутренних механизмов** и ссылок в Admin Console.

---
### Client: Security-admin-console
**Назначение**: клиент для ==административной консоли Keycloak==.
	
**URL**: /admin/{realm}/console
	
**Используется**: при входе в админ-панель, ***управлении реалмами, пользователями и клиентами.***

# 4. Scope клиентов (Client-scope)
В Keycloak `scope` (область доступа) - это *механизм, определяющий*,==КАКИЕ ДАННЫЕ и разрешения ПОЛУЧИТ КЛИЕНТ В Access Token== / ID token **при аутентификации пользователя**.

> [!NOTE] Проще говоря:
> `Scope` - это **способ контролировать, ЧТО ИМЕННО ПОЛУЧИТ КЛИЕНТ** ***при запросе авторизации*** у пользователя.

![[Pasted image 20260120092247.png]]

## Assigned Type у каждого scope
`Default`
Этот scope **всегда включается в токен по умолчанию**, даже если клиент не запрашивает его явно.

`Optional`
Scope будет применён **только если клиент его запросит в параметре scope=...** при авторизации.

`(Unassigned)`	
Scope **не применяется клиенту вообще**, пока его не назначат вручную или не укажут явно в токене.

## Типы scope
### Client scopes - acr
**Тип**: Default
**Протокол**: OpenID Connect
**Назначение**: добавляет в токен acr (`Authentication Context Class Reference`) - ==уровень аутентификации== (например, MFA (многофакторная AuthN, пароль и т. д.).

---
### Client scopes - address
**Тип**: Optional
**Протокол**: OpenID Connect
**Назначение**: ==добавляет адрес пользователя== (`address claim`) в токен (если заполнено в профиле).

---
### Client scopes - basic
**Тип**: Default
**Протокол**: OpenID Connect
**Назначение**: ==включает базовые claims==:
- sub, 
- name, 
- preferred_username, 
- given_name, 
- family_name.

---
### Client scopes - email
**Тип**: Default
**Протокол**: OpenID Connect
**Назначение**: ==добавляет email и email_verified claims== в ID и Access токены.

---
### Client scopes - microprofile-jwt
**Тип**: Optional
**Протокол**: OpenID Connect
**Назначение**: ==обеспечивает совместимость с Eclipse MicroProfile JWT== - популярным стандартом для Java microservices.


**Eclipse MicroProfile JWT** — это стандарт для аутентификации и авторизации в микросервисных архитектурах на Java, часть инициативы Eclipse MicroProfile. Он определяет:
1. **Формат JWT-токена** (какие claims обязательны/рекомендуются)
    
2. **API для доступа к claims** внутри микросервиса
    
3. **Интеграцию с CDI (Contexts and Dependency Injection)** для внедрения информации о пользователе

Когда вы назначаете скоуп `microprofile-jwt` клиенту, токены для этого клиента **дополняются claims**, требуемыми спецификацией

---
### Client scopes - offline_access (выдача refresh_token)
**Тип**: Optional
**Протокол**: OpenID Connect
**Назначение**: ==разрешает выдачу offline refresh token== (живёт долго и используется без пользовательского взаимодействия).

---
### Client scopes - phone
**Тип**: Optional
**Протокол**: OpenID Connect
**Назначение**: *добавляет phone_number и phone_number_verified* в токен (если указано в профиле).

---
### Client scopes - profile
**Тип**: Default
**Протокол**: OpenID Connect
**Назначение**: включает claims: 
- name, 
- given_name, 
- family_name, 
- middle_name, 
- nickname, 
- preferred_username, 
- profile, 
- picture, 
- website, 
- gender, 
- birthdate, 
- zoneinfo, 
- locale, 
- updated_at.

---
### Client scopes - role_list
**Тип**: Default
**Протокол**: SAML
**Назначение**: включает список ролей (roles) пользователя в SAML assertions.

---
### Client scopes - roles
**Тип**: Default
**Протокол**: OpenID Connect
**Назначение**: добавляет ==роли пользователя== (realm roles и client roles) в access token (в realm_access.roles и resource_access).

# 5. Realm roles - реализация RBAC

> [!IMPORTANT] Realm roles
> **Realms Role (роль на уровне realm) в Keycloak** — это **глобальная роль в рамках одного realm**, которая ==назначается пользователям для управления доступом на уровне всей системы== (всех клиентов в realm).

![[Pasted image 20260120094448.png]]

Может быть:
- назначена пользователю напрямую
- включена в другие роли (composite)
- проверена в любом клиенте, если токен содержит `realm_access.roles`

## Пример: Composite role
Предположим, у нас есть такие роли:
- `read-users`
- `edit-users`
- `delete-users`

Мы создаем роль `manage-users` и делаем composite, включив в нее все три роли выше.

Теперь:
При назначении `manage-users` пользователю, он автоматически получает `read-users`, `edit-users` и `delete-users`.

Это упрощает управление: **назначаешь одну роль** - **пользователь получает комплексный доступ.**

## Reamls Role созданные по умолчанию
### Roles - admin
**Composite**: Да
**Описание**: role_admin
**Назначение**: ==включает административные права== - может управлять пользователями, клиентами, ролями и конфигурацией реалма

---
### Roles - create-realm
**Composite**: Нет
**Описание**: role_create-realm
**Назначение**: ==позволяет пользователю создавать новые realms через Admin Console== или REST API (`только в master realm`).

---
### Roles - default-roles-master
**Composite**: Да
**Описание**: role_default-roles
**Назначение**: набор ролей, назначаемых ==по умолчанию всем новым пользователям master realm==.
**Включает**: 
- offline_access, 
- uma_authorization 
- и другие (можно посмотреть внутри composite).

---
### Roles - offline_access
**Composite**: Нет
**Описание**: role_offline-access
**Назначение**: дает ==возможность получать offline refresh tokens== - живут дольше, не требуют активной сессии пользователя.

---
### Roles - uma_authorization
**Composite**: Нет
**Описание**: role_uma_authorization
**Назначение**: позволяет использовать User-Managed Access (UMA) - механизм, при котором пользователь *может делегировать доступ к своим ресурсам другим пользователям* (используется при ресурсно-ориентированном доступе).
