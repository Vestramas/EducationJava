# 1. Недостаток СТАРОЙ МЕТОДИКИ, и Преимущество НОВОЙ МЕТОДИКИ
Старая методика удерживает блокировку достаточно долгое время.
Блокировка накладывается перед запросом к разделяемому ресурсу. И держится до тех пор, пока:
- Выполняется запрос
- Выполняются проверки
- Формируются движения

В новой методике все НАОБОРОТ, сначала формируются движения **БЕЗ БЛОКИРОВКИ**, и только потом захватывается блокировка и выполняются проверки.
> То есть блокировка удерживается только на момент проверок

Из-за этой особенности, НОВАЯ МЕТОДИКА подходит ТОЛЬКО ДЛЯ ТЕХ СЛУЧАЕВ, когда данные для движений уже известны в исходной таблице (***ИЗОЛЯЦИЯ КОТОРОЙ ОБЕСПЕЧИВАЕТСЯ В РАМКАХ ТРАНЗАКЦИЙ***)

А для того что чтобы проводить манипуляции с данными, необходима блокировка (а суть новой методики как раз таки в уменьшении времени блокировки данных).

Поэтому НОВАЯ МЕТОДИКА ПОДХОДИТ ДЛЯ ТЕХ СЛУЧАЕВ, когда данных для формирования движений ХВАТАЕТ на этапе первого запроса к БД

# 2. Новая методика - "Пиши, потом проверь"

> [!NOTE] "Сначала СПИСЫВАЕМ, потом ПРОВЕРЯЕМ остаток"
> В новой методике используется принцип: 
> - **Сначала Списываем необходимые товары**, 
> - ***Далее Проверяем*** — образовались ли отрицательные остатки по товарам документа. 
> - Если да, то нужно откатить проведение документа.

> Однако, здесь также могут использоваться д***ополнительные таблицы (регистры)***, **НО ТОЛЬКО ДЛЯ ПРОВЕРКИ**, ==так как до этой проверки данные в движении уже сформированы== и записаны в БД

Обычно используется для контроля отрицательных остатков (чтобы нельзя было списать товаров больше чем есть на складе)
## 2.1 Когда использовать и алгоритм

> [!NOTE] Отличие от СТАРОЙ МЕТОДИКИ
> В НОВОЙ мы сразу получаем нужные данные из одной таблицы, и записываем движения. И только потом проверяем наши записи на валидность ПО ДОПОЛНИТЕЛЬНОЙ ТАЛИЦЕ(РЕГИСТРУ), и при необходимости откатываем
> 
> В СТАРОЙ мы сразу получаем нужные данные, ПРОВЕРЯЕМ ИХ, и только после проверки записываем в БД

> [!NOTE] БЛОКИРОВКА
> В **СТАРОЙ используется** `управляемая блокировка` (вручную выбираем поля для блокировки)
> 
> В **НОВОЙ используется** `автоматическая блокировка`, которая блокирует вообще все

### 2.1.1 Когда использовать
**Когда** ==ДАННЫЕ ДЛЯ ДВИЖЕНИЙ можно ДОСТАТЬ ИЗ ОДНОЙ ТАБЛИЦЫ==, и не нужно получать данные из других

### 2.1.2 Алгоритм
1. Получение запросом данных документа
2. Формирование движений-расход регистра
3. Запись движений в БД
4. **УСТАНОВКА БЛОКИРОВКИ**
5. (ПРОВЕРКА)Запрос, получающий отрицательные остатки из регистра
6. Вывод сообщений о недостатке товаров

## 2.3 МЕТОДИКА - Формирование ЗАПРОСА для проверки
Теперь, после записи движений, нам нужно проверить остатки, которые мы записали.

Для этого нам необходимо обратится к регистру, и провести проверку

Суть проверки схожа с той, что была в старой методике. Нужно выбрать записи из таблицы продаж (*ЧТОБЫ УЗНАТЬ СКОЛЬКО МЫ "ПРОДАЛИ"*), и затем достать остатки из регистра остатков. 

И уже на полученной выборке выполнить проверку.

### 2.3.1 Небольшая пред оптимизация
То есть, мы сейчас будем писать запрос, в котором:
- Получим данные из Табличной части документа
- Получим данные из регистра остатков, и достанем только те записи что есть в ТЧ_Документа

Но ведь мы уже доставали табличную часть документа, придется снова повторять запрос? НЕТ

И здесь есть 2 варианта решения
- ***Передать ТЧ напрямую в запрос через параметр***, так как мы находимся в документе, и имеем непосредственный доступ к ней.
	
- **Использовать** `МенеджерВременныхТаблиц`,который позволяет сохранять результат выборки временных таблиц

Мы будем использовать `МенеджерВременныхТаблиц`.

- **сам запрос поместить в ВРЕМЕННУЮ_ТАБЛИЦУ**,

#### 3.2.3.1 Поправка параметра &Момент
Когда мы используем просто `МоментВремени()` в запросе к виртуальной таблице остатков, мы получаем остатки **на начало указанного момента времени**, ==НЕ ВКЛЮЧАЯ ДВИЖЕНИЯ самого документа==

|Режим|Что возвращает|Проблема|
|---|---|---|
|`МоментВремени()`|Остатки на начало момента, **не включая текущую транзакцию**|Не видит другие параллельные транзакции|
|`Граница(..., Включая)`|Остатки с учетом всех изменений в текущей транзакции|Видит параллельные изменения (и ждет блокировок)|

> [!NOTE] Сука транзакции! 
> Движения вроде бы записаны, НО НЕ ЗАФИКСИРОВАНЫ!! Из-за этого на `МоментВремени` мы не увидим их
> 
> **Ключевой момент:** Запись движений происходит в транзакции, но данные **еще не зафиксированы** в базе. Однако запрос с `Граница(..., Включая)` **видит эти еще не зафиксированные данные в рамках текущей транзакции**!


А чтобы включить их в Период виртуальной таблицы, используется тип данных `Граница`

Поэтому код при передаче параметра в запрос изменится с такого
```java
Запрос.УстановитьПараметр("Момент", Момент);

Запрос.УстановитьПараметр("Склады", Склады);
```

НА ТАКОЙ
```JAVA
Запрос.УстановитьПараметр(
	"Момент", 
	Новый Граница(МоментВремени(), ВидГраницы.Включая));
	
Запрос.УстановитьПараметр("Склады", Склады);
```

## 2.5 МЕТОДИКА - Блокировка
Теперь наша новая методика почти готова, основная логика выполняется.

Но нужно установить блокировки, чтобы несколько пользователей могли одновременно проводить документы.

Сейчас же, когда мы вызываем метод `Движение.Записать()`, то **на уровне БД** ***происходит блокировка на регистр остатков*** (*так как только 1 транзакция может изменять данные*).

Этот же регистр мы используем для проверки отрицательных остатков уже после записи в сам регистр

Если же не делать блокировку, то допустим:

	А тем временем у Пользователя 2 микроблокировка на уровне БД

Пользователь 2

	А тем временем у Пользователя 1 микроблокировка на уровне БД

После чего, пользователь 1 решает прочитать регистр, и выполнить проверку "Хватает ли товаров на списание"

НО так как товар списали оба пользователя, то записи о списании двух пользователей появятся в регистре. 
И товар мы не сможем списать ни у Пользователя 1 ни у Пользователя 2. *ХОТЯ ДЛЯ ОДНОЙ ПРОДАЖИ ТОВАРА ТОЧНО ХВАТАЕТ*

> Допустим товара на складе 10.
> Первый пользователь списывает 5 товаров (Остаток 5)
> Второй пользователь списывает 7 товаров (Остаток -2)
> 
> После чего, у обоих пользователей начинается проверка по регистру. Они проверяют остатки по списываемому товару, и видят что остаток отрицательный, и товар продать не могут!!!

Именно поэтому, чтобы избежать такой ситуации, **нам необходимо установить блокировку ПЕРЕД методом** `Записать()`, и снимать только при выходе из обработчика проведения. 

### 2.5.1 РазделяемыеИтоги у регистра
У Регистра Накопления есть особый режим `РазделениеИтогов`
![[Pasted image 20260309211944.png]]

> [!IMPORTANT] РазделениеИтогов
> Этот режим ДОЛЖЕН БЫТЬ УСТАНОВЛЕН для ПАРАЛЛЕЛЬНОЙ РАБОТЫ ПОЛЬЗОВАТЕЛЕЙ С РЕГИСТРОМ
> 
> Система ***при одновременной записи движений*** несколькими сеансами **будет НЕ ОБНОВЛЯТЬ** одни и те же записи итогов, **а записывать изменения итогов в отдельные записи**. 
> 
> При ==получении итогов эти данные складываются==.

### 2.5.2 Установка блокировки
Чтобы установить блокировку на весь регистр, используется его метод `БлокироватьДляИзменения()`.

Устанавливает режим, при котором ***в процессе записи набора*** **будет установлена управляемая блокировка** ==для всех комбинаций измерений== в соответствии с записями набора записей. 

Имеет *смысл использовать*, **если проверка итогов регистра выполняется ПОСЛЕ ЗАПИСИ** и заблокировать нужно именно те комбинации, по которым записываются записи.
> В этом случае можно не использовать объект [БлокировкаДанных](v8help://SyntaxHelperContext/objects/catalog63/catalog1402/DataLock.html).

> "После записи", судя по всему, ч*тобы не перебирать повторно табличную часть,* по которой формировались движения

![[Pasted image 20260309212954.png]]

```java
Движения.ОстаткиНоменклатуры.БлокироватьДляИзменения = Истина;
Движения.Записать();	
```
