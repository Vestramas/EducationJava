## 1. О проекте
Проект назывался **ППВС** — Платформа предоставления внешних сервисов. Это был **внутренний проект для нужд самой компании**.


**Бизнес-задача** — *автоматизация обмена налоговыми данными с ФНС*. 
Компания формировала и отправляла налоговую отчетность в ФНС, а также обрабатывала ответы, уведомления и требования, поступающие от ФНС.

Основной целью проекта была автоматизация формирования документов по данным из 1С, их валидация по схемам ФНС и последующая отправка во внешние системы.



**Зачем это нужно:** раньше данные отправляли вручную через сайт ФНС или по email. Процесс был долгим, с ошибками, данные терялись. Нужно было автоматизировать выгрузку из 1С, валидацию по схемам ФНС и отправку по защищенному каналу.

---
## 2. Команда и процесс

**Команда состояла из 5 человек:**
- Team Lead
    
- Один Middle-разработчик
    
- Два Junior-разработчика (я и второй джун)
    
- QA-инженер

Мы работали по **Scrum со спринтами на 2 недели**. В начале каждого спринта мы собирались на **планирование**, где Team Lead показывал задачи, мы обсуждали их, распределяли между собой и оценивали в часах. Мои задачи обычно были на 4–8 часов.

---
## 3. Архитектура проекта
Архитектура была **микросервисной**. В проекте было 5 сервисов:
1. **API Gateway** (настраивал Team Lead) — на Spring Cloud Gateway с KeyCloak. 
   Использовался ==как единая точка входа ДЛЯ ВНУТРЕННИХ СИСТЕМ компании.== Проверял JWT-токены пользователей (БУХГАЛТЕРОВ ИЗ ЛИЧ. КАБИНЕТА) и сервисов, выполнял маршрутизацию запросов к микросервисам.
    
2. **Extractor Service (мой сервис)** — принимал запросы на формирование налоговых документов от внутренних систем компании, вычитывал данные из 1С, формировал XML и отправлял события в Kafka.
    
3. **Validation Service** (делал второй джун) — валидировал XML по XSD-схемам ФНС.
    
4. **Sender Service** (делал Middle) — отправлял валидный XML в ФНС по HTTPS.
    
5. **Status Service** (делал Middle) — **собирал статусы от всех сервисов**, хранил историю обработки и предоставлял статус выполнения внутренним системам компании.

## 4. Как работала система от и до

```
Личный кабинет бухгалтера / ERP / 1С
          ↓
      Gateway
          ↓
      Extractor
          ↓
        Kafka
          ↓
     Validation
          ↓
       Sender
          ↓
         ФНС
```

***Формирование налоговых документов*** **инициировали внутренние системы компании** (например, личный кабинет бухгалтера или ERP-система). 
Пользователь запускал формирование отчета, после чего ППВС выполняла всю цепочку обработки и отправки данных в ФНС.

```
Шаг 1
Внутренняя система вызывает API

Шаг 2
Gateway проверяет JWT

Шаг 3
Extractor создает задачу и возвращает 202

Шаг 4
Статус можно посмотреть через Status Service

Шаг 5
Extractor формирует XML и кладет его в Kafka

Шаг 6
Validation проверяет XML и передает Sender

Шаг 7
Sender отправляет документ в ФНС
```


**Вот как это выглядело по шагам:**

### Шаг 1. Запрос из внутренней системы
Пользователь (например, **бухгалтер**) работал **через личный кабинет** компании либо другую внутреннюю систему.

Для формирования налогового документа **система отправляла HTTP-запрос в ППВС с параметрами**:
- тип документа;
- отчетный период;
- идентификатор организации.

В запросе передавался JWT-токен, выданный Keycloak.

### Шаг 2. Аутентификация и маршрутизация
Запрос поступал в API Gateway.

Gateway проверял JWT-токен через Keycloak, а также права пользователя на выполнение операции.

После успешной проверки запрос маршрутизировался в Extractor Service.

### Шаг 3. Extractor Service (мой сервис)

Я делал следующее:

- Сохранял запрос в своей БД со статусом `PENDING`
    
- Вычислял SHA-256 хеш от параметров запроса и проверял дубликаты
    
- Отправлял событие в Kafka в **Status Service** (статус `PENDING`)
    
- Отправлял событие в Kafka в **Validation Service** (XML на валидацию)
    
- Возвращал вызывающей системе ответ `202 Accepted` с `request_id`

### Шаг 4. Получение статуса обработки

Вызывающая внутренняя система не ждала завершения обработки.

Для получения результата она периодически запрашивала статус через Status Service:

```http
GET /api/status/{request_id}
```

### Шаг 5. Фоновая обработка

В фоновом режиме мой сервис:
- Менял статус на `PROCESSING` и отправлял в Status Service
    
- Вычитывал данные из PostgreSQL (БД 1С) через JPA
    
- Формировал XML через JAXB
  JAXB - мапит Java-объект в XML, и обратно. Считали данные в объект через JPA, и смапили его в XML
    
- Сохранял XML в Outbox-таблицу (в одной транзакции с запросом)
    
- Отдельный процесс отправлял данные из Outbox в Kafka

### Шаг 6. Валидация и отправка

- **Validation Service** слушал Kafka, валидировал XML по XSD-схемам. После успешной проверки сервис публиковал сообщение для Sender Service.
    
- **Sender Service** отправлял валидный XML в ФНС по HTTPS. После отправки отправлял статус в Status Service.

### Шаг 7. Завершение обработки
После успешной отправки документа Sender Service обновлял статус обработки.

Внутренняя система компании могла получить итоговый результат через Status Service.

Финальный статус мог принимать значения:
- SUCCESS
- FAILED

## Моя задача и как я её решал

Я отвечал за **Extractor Service**.

**Что я конкретно сделал:**
1. **Endpoint для внутренних систем компании** — реализовал REST API на Spring MVC, который принимал запросы на формирование налоговых документов с параметрами периода и типа документа.
    
2. **Аутентификация и авторизация** — ***валидация JWT-токенов выполнялась на уровне API Gateway*** через интеграцию с Keycloak. В самом сервисе я не реализовывал отдельную логику проверки токенов.
    
3. **Вычитка из 1С** — подключался к PostgreSQL через JDBC и писал SELECT-запросы под конкретный период.
    
4. **Формирование XML** — использовал JAXB. Создал Java-классы по XSD-схемам ФНС, разметил аннотациями, заполнял объекты данными и через `Marshaller` превращал в XML.
    
5. **Асинхронная обработка** — возвращал 202 Accepted, а не заставлял вызывающую систему ждать завершения формирования и отправки документа.
    
6. **Outbox-паттерн** — чтобы не потерять сообщения при падении сервиса, я сохранял XML в Outbox-таблицу в той же транзакции, что и запрос. Отдельный шедулер отправлял данные в Kafka.
    
7. **Защита от дубликатов** — вычислял SHA-256 хеш от параметров запроса, сохранял в БД с уникальным индексом. Повторные одинаковые запросы игнорировал.
    
8. **Отправка статусов** — через Kafka отправлял события в Status Service при каждом изменении статуса (`PENDING`, `PROCESSING`, `FAILED`).
    
9. **Метрики** — через Micrometer собирал количество успешных и неудачных запросов, а также время обработки.

# 5. Архитектурные решения и ответы на типовые вопросы

## Какая была нагрузка?

Система **НЕ БЫЛА** **высоконагруженной**.

Это был внутренний корпоративный сервис для формирования и отправки налоговых документов в ФНС.

В обычные дни обрабатывались десятки документов в сутки. В периоды налоговой отчетности нагрузка могла увеличиваться до нескольких сотен документов в день.

Основной проблемой была не высокая нагрузка, а надежность обработки и интеграции с внешними системами.

---
## Почему использовали Kafka?

Kafka использовалась не из-за большого количества запросов.

Основные причины:

- асинхронная обработка;
- разделение ответственности между сервисами;
- возможность повторной обработки при сбоях;
- отсутствие длинных HTTP-цепочек между сервисами;
- буферизация сообщений при временной недоступности сервисов.

Например, если Validation Service временно недоступен, сообщения остаются в Kafka и будут обработаны после восстановления сервиса.

---

## Почему не использовали REST между сервисами?

REST хорошо подходит для синхронного взаимодействия.

В нашем случае процесс формирования документа состоял из нескольких этапов и мог занимать заметное время.

Использование Kafka позволило избежать длинных цепочек вызовов вида:

Extractor → Validation → Sender

и сделать обработку полностью асинхронной.

---

## Почему возвращали 202 Accepted?

На момент получения запроса документ еще не был сформирован и отправлен.

Поэтому сервис принимал задачу в обработку, сохранял ее в БД и возвращал request_id.

Далее статус можно было получить через отдельный endpoint Status Service.

---

## Почему использовали Outbox Pattern?

Нужно было гарантировать, что сообщение не потеряется между записью данных в БД и публикацией в Kafka.

Без Outbox могла возникнуть ситуация:

1. Данные сохранены в БД.
2. Приложение упало.
3. Сообщение в Kafka не отправилось.

В результате система теряла событие.

Для решения проблемы XML сначала сохранялся в Outbox-таблицу в рамках транзакции БД, а затем отдельный процесс публиковал его в Kafka.

---

## Сколько было Kafka-топиков?

Точно количество не запоминал, но их было немного — порядка 3–5.

Основные топики:

- report-validation — документы для Validation Service;
- report-sender — валидированные документы для Sender Service;
- status-events — события изменения статусов обработки;
- report-dead-letter (DLQ) — сообщения, которые не удалось обработать после нескольких попыток.

---

## Использовали ли партиционирование?

Да, но в небольшом объеме.

Так как система не была высоконагруженной, большого количества партиций не требовалось.

Партиционирование использовалось в основном для сохранения порядка обработки сообщений одного документа.

В качестве ключа обычно использовался request_id.

---

## Почему выбрали микросервисную архитектуру?

Каждый сервис отвечал за отдельный этап бизнес-процесса:

- Extractor — получение данных и формирование XML;
- Validation — проверка документа по XSD;
- Sender — отправка документа;
- Status — хранение и предоставление статусов.

Такое разделение позволяло независимо развивать и тестировать компоненты системы.

---

## Почему Validation Service был отдельным сервисом?

Логика проверки XML-документов была выделена отдельно от формирования документа.

Это позволяло независимо изменять правила валидации, тестировать их и переиспользовать для различных типов налоговых документов.

Кроме того, ошибки валидации не влияли на работу Extractor Service.

---

## Почему использовали JAXB?

ФНС работала с XML-документами, структура которых задавалась XSD-схемами.

JAXB позволял удобно преобразовывать Java-объекты в XML через аннотации и не заниматься ручной генерацией XML.
