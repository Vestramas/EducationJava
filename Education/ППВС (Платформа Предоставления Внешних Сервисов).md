## 1. О проекте
Проект назывался **ППВС** — Платформа предоставления внешних сервисов. Это был **внутренний проект для нужд самой компании**.

**Бизнес-задача** — *автоматизация обмена налоговыми данными с ФНС*. 
Компания отправляет налоговую отчетность из 1С в ФНС и обрабатывает входящие требования от ФНС.

**Зачем это нужно:** раньше данные отправляли вручную через сайт ФНС или по email. Процесс был долгим, с ошибками, данные терялись. Нужно было автоматизировать выгрузку из 1С, валидацию по схемам ФНС и отправку по защищенному каналу.

---
## 2. Команда и процесс

**Команда состояла из 5 человек:**
- Team Lead
    
- Один Middle-разработчик
    
- Два Junior-разработчика (я и второй джун)
    
- QA-инженер

Мы работали по **Scrum со спринтами на 2 недели**. В начале каждого спринта мы собирались на **планирование**, где Team Lead показывал задачи, мы обсуждали их, распределяли между собой и оценивали в часах. Мои задачи обычно были на 4–8 часов.

---
## 3. Архитектура проекта
Архитектура была **микросервисной**. В проекте было 5 сервисов:
1. **API Gateway** (настраивал Team Lead) — на Spring Cloud Gateway с KeyCloak. Он проверял JWT-токены от ФНС и маршрутизировал запросы.
    
2. **Extractor Service** (мой сервис) — принимал запросы от ФНС, вычитывал данные из 1С, формировал XML и отправлял в Kafka.
    
3. **Validation Service** (делал второй джун) — валидировал XML по XSD-схемам ФНС.
    
4. **Sender Service** (делал Middle) — отправлял валидный XML в ФНС по HTTPS.
    
5. **Status Service** (делал Middle) — собирал статусы от всех сервисов, хранил историю и отдавал статус ФНС по запросу.

## 4. Как работала система от и до

ФНС работала с нами по **pull-модели** — она сама отправляла HTTP-запрос на формирование отчета.

**Вот как это выглядело по шагам:**

### Шаг 1. Запрос от ФНС

ФНС отправляла GET-запрос с параметрами: период, тип документа. В заголовке был JWT-токен.

### Шаг 2. Аутентификация и маршрутизация

Запрос попадал в **API Gateway**, который через KeyCloak проверял токен. Если токен валидный — Gateway пропускал запрос ко мне в Extractor Service.

### Шаг 3. Extractor Service (мой сервис)

Я делал следующее:

- Сохранял запрос в своей БД со статусом `PENDING`
    
- Вычислял SHA-256 хеш от параметров запроса и проверял дубликаты
    
- Отправлял событие в Kafka в **Status Service** (статус `PENDING`)
    
- Отправлял событие в Kafka в **Validation Service** (XML на валидацию)
    
- Возвращал ФНС ответ `202 Accepted` с `request_id`

### Шаг 4. ФНС проверяет статус

ФНС больше не ждала. Она позже приходила проверять статус по отдельному endpoint в **Status Service**: `GET /api/status/{request_id}`

### Шаг 5. Фоновая обработка

В фоновом режиме мой сервис:
- Менял статус на `PROCESSING` и отправлял в Status Service
    
- Вычитывал данные из PostgreSQL (БД 1С) через JDBC
    
- Формировал XML через JAXB
    
- Сохранял XML в Outbox-таблицу (в одной транзакции с запросом)
    
- Отдельный процесс отправлял данные из Outbox в Kafka

### Шаг 6. Валидация и отправка

- **Validation Service** слушал Kafka, валидировал XML по XSD-схемам. После валидации отправлял статус в Status Service.
    
- **Sender Service** отправлял валидный XML в ФНС по HTTPS. После отправки отправлял статус в Status Service.

### Шаг 7. ФНС получает результат

ФНС периодически проверяла статус через Status Service. Когда статус становился `SUCCESS` или `FAILED`, ФНС завершала обработку.

## Моя задача и как я её решал

Я отвечал за **Extractor Service**.

**Что я конкретно сделал:**
1. **Endpoint для ФНС** — реализовал REST API на Spring MVC, который принимал GET-запросы с параметрами периода и типа документа.
    
2. **Аутентификация** — не проверял токены сам, это делал Gateway до того, как запрос доходил до меня.
    
3. **Вычитка из 1С** — подключался к PostgreSQL через JDBC и писал SELECT-запросы под конкретный период.
    
4. **Формирование XML** — использовал JAXB. Создал Java-классы по XSD-схемам ФНС, разметил аннотациями, заполнял объекты данными и через `Marshaller` превращал в XML.
    
5. **Асинхронная обработка** — возвращал `202 Accepted`, а не заставлял ФНС ждать. Сохранял запрос в БД со статусом `PENDING`.
    
6. **Outbox-паттерн** — чтобы не потерять сообщения при падении сервиса, я сохранял XML в Outbox-таблицу в той же транзакции, что и запрос. Отдельный шедулер отправлял данные в Kafka.
    
7. **Защита от дубликатов** — вычислял SHA-256 хеш от параметров запроса, сохранял в БД с уникальным индексом. Повторные одинаковые запросы игнорировал.
    
8. **Отправка статусов** — через Kafka отправлял события в Status Service при каждом изменении статуса (`PENDING`, `PROCESSING`, `FAILED`).
    
9. **Метрики** — через Micrometer собирал количество успешных и неудачных запросов, а также время обработки.
