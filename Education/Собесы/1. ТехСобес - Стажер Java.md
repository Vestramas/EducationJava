# Java Core
## 1. Как создать поток?
Есть два способа:
- Реализовать Runnable
- Наследоваться от Thread

Оба способа подразумевают реализацию метода `run`. 

Runnalbe более гибкий, так **как описывает задачу, которая выполняется в потоке.** Эту з*адачу можно передавать разным потокам*

### Runnable предпочтительнее
#### Нет проблем с наследованием
Java не поддерживает множественное наследование классов.

Если ты уже наследуешься от другого класса — `Thread` использовать нельзя.

#### Разделение задачи и механизма выполнения
`Runnable` описывает задачу.

`Thread` — механизм выполнения.

Это хороший принцип проектирования.

#### Один `Runnable` можно запускать разными потоками
```java
Runnable task = new MyTask();

new Thread(task).start();
new Thread(task).start();
```

#### Объект `Thread` нельзя запустить повторно
```java
Thread t = new Thread();

t.start();
t.start(); // IllegalThreadStateException
```

## 7. `map` vs `flatMap` в Stream API.

`map` - Преобразует **один объект в один**.
```java
List<String> names = List.of("a", "bb");  
  
names.stream()  
	.map(String::length);
	
	
//ИТОГ: Stream<List<String>>
```


`flatMap` - Преобразует **один объект в множество объектов** ***и "разворачивает" их.***
```java
List<List<String>> list = ...

list.stream()
    .flatMap(Collection::stream)
    
//ИТОГ: Stream<String>
```

## 9. Контракт `equals()` и `hashCode()`.
### Контракт equals()
**Рефлексивность** - сравнение с собой = true

**Симметричность** - x.equals(y) == y.equals(x)

**Транзитивность** - `x.equals(y)` возвращает true и `y.equals(z)`  возвращает true, то `x.equals(z)` должно возвращать true.

**Консистентность**: *повторные вызовы* x.equals(y) должны ***возвращать одинаковый результат***, если состояние объектов не изменилось.

По умолчанию метод equals() в классе Object сравнивает ссылки на объекты, а не их содержимое. Для логического сравнения объектов метод необходимо переопределять.

### Контракт метода hashCode()
Метод hashCode() возвращает целочисленное значение, которое используется в хеш-таблицах. 

Его контракт включает:
**Внутренняя согласованность**: значение hashCode() может измениться только в том случае, если изменится свойство, которое участвует в equals(). 

**Равная согласованность**: если *два объекта равны по методу equals()*, то ***вызов hashCode() для каждого из этих объектов должен возвращать одно и то же значение.*** 

**Необязательность**: не требуется, чтобы ***неравные по equals() объекты возвращали разные значения hashCode()***. Однако стремление к максимальной дисперсии значений для неравных объектов может улучшить производительность хеш-таблиц. 

**Стабильность при многократных вызовах**: многократный вызов hashCode() для одного и того же объекта в течение выполнения приложения должен возвращать одно и то же значение, если

### Ключевые правила взаимодействия equals() и hashCode()
Если **переопределён equals(), необходимо переопределить hashCode()**. Это требование обеспечивает согласованность контрактов и корректную работу хеш-структур. 

**Равенство объектов по equals() гарантирует равенство их хэш-кодов.** 

***Равенство хэш-кодов*** **не гарантирует равенства объектов по equals().** Ситуация, когда разные объекты имеют одинаковый хэш-код, называется коллизией. 

При вычислении hashCode() следует использовать те же поля, что и в `equals()»`,чтобы сохранить согласованность.

## 11. Что такое обобщённые типы (generics) и зачем они нужны?
**Обобщения (Generics)** - это ==параметризованные типы,== 

Они позволяют создавать классы, интерфейсы и методы, которые работают с различными типами данных безопасным образом.

Работают только с СЫЛОЧНЫМИ ТИПАМИ

### Основные преимущества:
- **Безопасность типов** - ошибки обнаруживаются на этапе компиляции
    
- **Устранение приведений типов** (не нужно приводить к Object)- код становится чище и понятнее
    
- **Повторное использование кода** - один алгоритм для разных типов данных

### Подстановочный знак `?`
В **обобщениях (generics)** ==подстановочный знак== `?` называется ==**wildcard== (джокер, метасимвол)**

Он используется тогда, когда мы **не знаем или не хотим указывать точный параметр типа**, а лишь ==задаём ограничения на него (на тип)==


### Правило PECS
> [!NOTE] ⭐⭐**Ключевое правило (PECS):**⭐⭐
> **Producer Extends** — если коллекция только отдаёт значения (==читаем==), используем `? extends T`. 
> 
> **Consumer Super** — если коллекция только принимает значения (==пишем==), используем `? super T`.

```java
public double sumOfList(List<? extends Number> list) {
    double s = 0.0;
    for (Number n : list) {
        s += n.doubleValue(); // Читаем как Number — это безопасно
    }
    return s;
}
```

`? super Integer` - родители класса Integer
`? extends Integer` - дочерние класса Integer

## 13. Самые популярные сборщики мусора (GC) и общий алгоритм их работы.

### Общий принцип
GC удаляет объекты, на которые нет ссылок.

### Гепотиза поколений
Большинство объектов живут недолго.

Heap делится на:
- Young Generation
- Old Generation

#### Young Generation
Новые объекты.

GC быстрый — Minor GC.

#### Old Generation
Долгоживущие объекты.

Major/Full GC дороже.

---
Соответственно, ==процессы сборки мусора== разделяются на
##### 🔹 Minor GC - для младшего поколения:
- Срабатывает в Young Generation (в Eden)
- **Быстро и часто**
- Выжившие объекты перемещаются в Survivor → затем в Old
##### 🔹 Major GC (или Full GC) - для обоих поколений:
- Затрагивает Old Generation (и иногда весь heap)
- **Медленно и редко**
- Может вызвать **Stop-The-World** паузу

### Популярные GC

#### Serial GC
Один поток.

#### Parallel GC
Несколько потоков.

Throughput-oriented.

#### G1 GC
По умолчанию в современных Java.

Heap делится на регионы.

Минимизирует pause time.

**Exception** - Исключения приложения.

### Проверяемы и НЕпроверяемые исключения

| Тип исключения         | Обработка                                            | Примеры                                                                    |
| ---------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------- |
| ✅ Checked Exception    | **Обязательна** обработка (`try-catch` или `throws`) | `IOException`, `SQLException`, `ParseException`                            |
| ⚠️ Unchecked Exception | **Необязательна** обработка                          | `NullPointerException`, `ArithmeticException`, `IndexOutOfBoundsException` |

> [!NOTE] Разница
> Все что наследуется от `RuntimeException` это - `unchecked` исключения
> Все что наследуется от `Exception` НО НЕ от `RuntimeException` - `checked` исключения

##### ✅ Checked Exceptions — _проверяемые_
- Компилятор **требует** либо `try-catch`, либо `throws`.
    
- Используются в ситуациях, где **ошибка может предсказуемо произойти** и с ней можно что-то сделать.

##### ⚠️ Unchecked Exceptions — _непроверяемые_
- Компилятор **НЕ требует** обработки.
    
- Чаще всего — **ошибки логики или программирования**.
    
- Вызов таких исключений говорит, что **код написан неправильно**.

## 25. Что такое сериализация в Java? Что нужно сделать, чтобы сериализовать объект? Что такое `serialVersionUID`?

**Сериализация** - Преобразование объекта в поток байт.


Чтобы сереализовать объект, используется интерфейс **Serializable** (МАРКЕРНЫЙ)

Его задача: пометить класс как сериализуемый. 
> Тогда `ObjectOutputStream`/`ObjectInputStream` сможет автоматически преобразовать объект в байтовый поток и обратно.

Особенности и подводные камни💡
1. **Все нестатические поля сериализуются.**  
    А `static` (классовые) и `transient` (помеченные явно) — НЕТ.
    
2. **`serialVersionUID`.**
    - Это уникальный идентификатор версии класса.
        
    - Если не указать — JVM сгенерирует автоматически (на основе структуры класса). Но при малейшем изменении класса (например, сменили тип поля) десериализация старого объекта сломается с `InvalidClassException`.
        
    - Best practice: **всегда явно указывать**:
```java
private static final long serialVersionUID = 1L;
```

## 33. Что такое deadlock (взаимная блокировка) и livelock?
### Deadlock
Потоки навсегда ждут друг друга.

**Условия**
- ***Взаимное исключение*** - ***Ресурс может быть занят*** **только одним** потоком/процессом **в каждый момент времени.** 
  Если ресурс занят, другой поток не может его использовать — он должен ждать.
	
- ***Удержание и ожидание*** - Поток удерживает уже захваченные ресурсы **и при этом** ожидает дополнительные ресурсы, которые сейчас заняты другими потоками
	
- ***Отсутствие принудительного освобождения*** - Ресурсы, уже захваченные процессом, не могут быть освобождены принудительно до завершения процесса. Процесс должен освободить ресурсы добровольно
	
- ***Циклическое ожидание*** - Существует замкнутая цепочка потоков, где каждый ждёт ресурс, захваченный следующим в цепочке. Последний ждёт ресурс, захваченный первым.

### Livelock
Потоки активны, но прогресса нет.

> Простыми словами: это как ***два человека, которые встречаются в узком коридоре.*** ***Оба уступают дорогу,*** отступают в одну сторону, потом оба — в другую, потом снова одновременно пытаются пройти — и так бесконечно. Никто никуда не проходит.

## 9. ACID. Назовите принципы и уровни изоляции транзакций.

### ACID
**A — Atomicity (Атомарность)**  
Транзакция выполняется **целиком или не выполняется вообще**  
Если один шаг упал — всё откатывается  
📌 _Пример:_ деньги не могут списаться без зачисления

**C — Consistency (Согласованность)**  
После транзакции БД остаётся в **валидном состоянии**: все ограничения, связи и правила соблюдены  
📌 _Пример:_ нельзя записать заказ без существующего пользователя

**I — Isolation (Изоляция)**  
Параллельные транзакции **не мешают друг другу**  
Каждая выглядит так, будто выполняется одна  
📌 _Решает проблемы dirty read / lost update_

**D — Durability (Долговечность)**  
Если транзакция закоммичена — данные **не пропадут**, даже при падении сервера  
📌 _Обеспечивается журналами, WAL, fsync_

### Аномалии
- **Dirty read** — чтение незакоммиченных данных
    
- **Non-repeatable read** — строка изменилась между двумя SELECT
    
- **Phantom read** — появляются новые строки, подходящие под WHERE
    
- **Lost update** — потеря обновления из-за параллельной записи

### Изоляции

#### Read Uncommitted
Возможны:
- dirty reads

#### Read Committed
Нет dirty reads.

Но возможны:
- non-repeatable reads
- phantom reads

Самый популярный уровень.

#### Repeatable Read
Нет non-repeatable reads.

#### Serializable
Максимальная изоляция.

Транзакции выполняются как последовательно.

Самый медленный.

## 8. Как работает `@Transactional` в Spring? Что произойдёт, если один сервис вызывает метод с `@Transactional` из другого метода с `@Transactional`?

Spring создает прокси вокруг бина, в котором есть метод с аннотацией `@Transactional`

Этот прокси объект управляет транзакцией (начинает ее, откатывает, и коммитит). А между этим, вызываем метод бина.

```java
public enum Propagation {
    REQUIRED,      // ПО УМОЛЧАНИЮ — присоединиться к существующей или создать новую
    
    REQUIRES_NEW,  // всегда новая транзакция (приостановить текущую)
    
    
    NESTED,        // вложенная транзакция (savepoint)
    
    
    MANDATORY,     // обязательно существующая транзакция (иначе ошибка)
    SUPPORTS,      // если есть — присоединиться, если нет — без транзакции
    
    
    NOT_SUPPORTED, // вне транзакции (приостановить текущую)
    NEVER          // вне транзакции (бросает ошибку, если транзакция есть)
}
```

Если сервис вызываетм метод с `@Transactional` из метода с `@Transactional`, то поведение зависит от того, как настроенна аннотация
- По умолчанию, будет одна транзакция
- Если настроено иначе, то будет отдельная транзакция или что-то еще
```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
    
    @Transactional
    public void methodA() {
        serviceB.methodB();  // вызываем метод другого сервиса
    }
}

@Service
public class ServiceB {
    @Transactional(propagation = Propagation.REQUIRED)  // или по умолчанию
    public void methodB() {
        // бизнес-логика
    }
}
```

### Что происходит при вызове внутри того же класса?
Это самый частый сценарий, когда `@Transactional` **НЕ РАБОТАЕТ**!
```java
@Service
public class UserService {
    @Transactional
    public void methodA() {
        methodB();  // ❌ транзакция для methodB НЕ создаётся!
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        // этот метод будет вызван без транзакции 
        // (или в транзакции methodA, если он есть)
    }
}
```
**Почему?** Прокси ==оборачивает только внешние вызовы бина (вызов БД, а внутренний вызов методов нет).== Внутренний вызов (`this.methodB()`) идёт напрямую к реальному объекту, минуя прокси.

**Как исправить?**
```java
@Service
public class UserService {
    @Autowired
    private UserService selfProxy;  // внедрить прокси самого себя
    
    @Transactional
    public void methodA() {
        selfProxy.methodB();  // ✅ теперь работает через прокси
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() { }
}
```

## 4. Что такое слои (layers) в образах и какую проблему они решают?
**Слои** — это неизменяемые ==промежуточные состояния, из которых состоит Docker-образ. ==

**Каждая инструкция** в Dockerfile (`FROM`, `COPY`, `RUN`, `CMD`) **создаёт новый слой.**

**Какую проблему решают:**
- **Кэширование** — если слой не изменился, Docker его переиспользует при пересборке (ускоряет сборку)
    
- **Переиспользование** — один и тот же базовый слой используется многими образами (экономия места)

```yaml
    internal: true   # изолирована от внешнего мира
```

## 6. Кейс: я запустил PostgreSQL в Docker, но он недоступен в системе через стандартный порт 5432. Где искать и как диагностировать проблему?

Возможные проблемы:
- Порт уже используется
- Не указан маппинг портов

```bash
# 1. Проверить, что контейнер вообще запущен
docker ps
# Если нет контейнера: docker ps -a (проверить все, возможно остановлен)

# 2. Проверить маппинг портов
docker port <container_name>
# Должно показать: 5432/tcp -> 0.0.0.0:5432

# 3. Проверить логи контейнера
docker logs <container_name>
# Ищем ошибки: "port already in use", "listening on"

# 4. Проверить, не занят ли порт 5432 на хосте
sudo netstat -tulpn | grep 5432
# или
lsof -i :5432

# 5. Проверить, слушает ли PostgreSQL внутри контейнера
docker exec <container_name> netstat -tulpn | grep 5432
# Возможная проблема: PostgreSQL слушает только 127.0.0.1, а нужно 0.0.0.0

# 6. Проверить подключение из другого контейнера в той же сети
docker run --rm -it --network container:<container_name> postgres psql -h localhost -U postgres

# 7. Проверить с хоста через nc (netcat)
nc -zv localhost 5432
```
