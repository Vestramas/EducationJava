# 1. Общее понимание процессов и ролей
## 1.1 Роли на проекте

![[Pasted image 20260402091546.png]]

**PRODUCT OWNER** (manager) - чел который отчевает за итоговый продукт, формирует задачи для продукта исходя из требований

**PROJECT MANAGER** - отвечает за выполнение проекта в отведенные строки (руководитель проекта)

**TEAM-LEAD** - чел, который шарит за техническую часть.  
- Пишет код
- Проводит дейлики
- Контролирует сроки разработки в своей команде

## 1.3 ПОСТАНОВКА ЗАДАЧ - Agile/Scrum
![[Pasted image 20260402092936.png]]

Сначала **приходит бизнес** к **PRODUCT OWNER/PROJECT MANAGER** с требованием реализации определенного продукта

Projcet\Product записывает ОБЩИЕ ЗАДАЧИ в Бэклог, которые понадобится решить для реализации проекта

Задачи лежат в Беклоге, до тех пор, пока их не возьмут Project, Тим-лид и аналитики на "Грумминг (планирование задач)"

**Грумминг** - процесс распиливания и обсуждения задачи (ОЧЕНЬ ЧАСТНО НА ГРУММИНГ ХОДЯТ ТИМ-ЛИД и АНАЛИТИКИ, и распиливают задачи на спринт)

Затем происходит **ПЛАНИРОВАНИЕ**, где уже присутствуют разработчики, **РАЗБИРАЮТ ЗАДАЧИ** *и оценивают задачи* по:
- ***времени (ЧАСАХ)***
- ***сторипоинты*** - абстрактная оценка, относительно "базовой" задачи. (Обычная задача выполняется за 4 часа - 1 сторипоинт). Если ты взял задачу, и понимаешь что она чуть сложнее обычной, то  2 сторипоинта. Если очень сложная задача то 3-4 сторипоинта

И дальше начинается спринт с ежедневными дейликами (созвонами), где вы говорите что сделали и что будете делать сегодня (ГОТОВЬ РЕЧЬ ЗАРАНЕЕ)

---
После окончания спринта происходит ДЕМО - ДЕМОнатсрация того, что было разработано по окончанию двух недель спринта (ОБЫЧНО РАЗРАБЫ НЕ ХОДЯТ)

После демо, идет РЕТРО - ретроспектива, говорим что хорошо было, а что плохо. "Долгое взаимодействе с др отделом", "Долго пайплайны работали" и т.д

## 1.4 РАБОТА С ЗАДАЧАМИ - GitFlow
Дальше работа задачами происходит при помощи Git с CI/CD в GitLab/GitHub

_"Мы использовали **Git Flow** — это популярная модель ветвления в Git. Она помогает организовать работу нескольких разработчиков над одним проектом."_

### Основные ветки в Git Flow
| Ветка                         | Назначение                                                          | Кто мержит                          |
| ----------------------------- | ------------------------------------------------------------------- | ----------------------------------- |
| **`main`** <br>(или `master`) | **Продуктивная ветка**. Здесь код, который работает на проде.       | Только Team Lead                    |
| **`develop`**                 | Основная **ветка разработки**. Сюда мержим все фичи.                | Любой разработчик после Code Review |
| **`feature/*`**               | Ветки для новых функций. Создаются от `develop`.                    | Разработчик создает и мержит        |
| **`release/*`**               | Ветки для подготовки релиза. Создаются от `develop` перед выкаткой. | Team Lead                           |
| **`hotfix/*`**                | Ветки для срочных исправлений багов на проде. Создаются от `main`.  | Senior или Team Lead                |

![[Pasted image 20260402094915.png]]

### Как я работал с Git Flow (пошагово)
> _"Когда я брал задачу в Jira, я делал так:_

1. **Создавал ветку от `develop`:**
```bash
git checkout develop
git pull origin develop
git checkout -b feature/TTS-123-create-order-endpoint
```
_Название ветки включало номер задачи из Jira._

2. **Разрабатывал фичу:** делал несколько коммитов с осмысленными сообщениями:
```bash
git commit -m "TTS-123: add OrderController with create endpoint"
git commit -m "TTS-123: add validation for weight field"
```

3. **Пушил ветку в удаленный репозиторий:**
```bash
git push origin feature/TTS-123-create-order-endpoint
```

4. **Создавал Merge Request (Pull Request) в GitLab:**
    - Из `feature/TTS-123` → в `develop`
        
    - В описании писал, что сделал, и ссылку на задачу в Jira
        
    - Добавлял ревьюверов (Team Lead или Senior)
        
5. **Проходил Code Review:** исправлял замечания, пушил новые коммиты (MR обновлялся автоматически)
    
6. **После аппрува** мержил ветку (или Team Lead мержил, если были конфликты)
    
7. **Удалял ветку** (локально и на сервере)"

### 3. Code Review (ревью кода)
> _"Каждый Merge Request обязательно проходил Code Review. 
> 
> Ревьювил обычно Team Lead или Senior."_

|Что проверяли|Почему|
|---|---|
|**Читаемость кода**|Имена переменных, методы, отсутствие магических чисел|
|**Обработка ошибок**|Не глотаем исключения, возвращаем понятные HTTP статусы|
|**Транзакционность**|Правильно ли расставлены `@Transactional`|
|**Тесты**|Есть ли юнит-тесты на новую логику|
|**Безопасность**|Нет ли SQL-инъекций, правильно ли валидируем входные данные|

## 1.5. CI/CD (как код попадал на стенд)
> _"У нас был настроен **GitLab CI/CD**. 
> 
> При каждом пуше в `develop` или `main` запускалась автоматическая сборка."_

### Окружения (стенды)

| Окружение                 | Для чего                                                                                                               | Кто имеет доступ              |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ----------------------------- |
| **Local**                 | Мой компьютер, локальная разработка                                                                                    | Только я                      |
| **Dev**                   | Общий стенд разработки. Сюда автоматически деплоится `develop`. Разработчики и QA проверяют, что фичи работают вместе. | Разработчики, QA              |
| **Stage** <br>(или Test)  | Стенд, максимально похожий на прод. Здесь заказчик и QA тестируют перед релизом.                                       | QA, заказчик, Team Lead       |
| **Prod** <br>(Production) | Продуктивный сервер. Реальные пользователи работают с системой.                                                        | Только Team Lead (для деплоя) |

### Что происходило в CI/CD:

|Шаг|Что делалось|
|---|---|
|1. Build|`mvn clean compile` — проверяла, что код компилируется|
|2. Test|`mvn test` — запускались все юнит-тесты|
|3. Integration Test|Поднимались тестовые контейнеры (Testcontainers) с PostgreSQL и Kafka|
|4. Deploy to Dev|Если тесты проходили, код деплоился на **Dev-стенд**|
|5. Deploy to Stage|После ручного подтверждения Team Lead — на **Stage** (тестовый для заказчика)|
|6. Deploy to Prod|Только после успешного тестирования на Stage и отдельной команды Team Lead|
