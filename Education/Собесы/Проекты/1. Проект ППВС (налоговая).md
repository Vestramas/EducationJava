# РЕЗЮМЕ
ППВС — **внутренний проект для автоматизации обмена налоговыми данными с ФНС**. 

ЗАДАЧА: **Автоматизация обмена налоговыми данными с ФНС** — компания отправляет налоговую отчетность (данные из 1С) в ФНС и обрабатывает входящие требования от ФНС.

Команда из 5 человек: Team Lead, Middle, два Junior (я) и QA.

Я отвечал за **Extractor Service** — он принимал запросы от ФНС на получение данных за определенный период.

## 📌 Как работала система
ФНС работала с нами по **pull-модели** — она отправляла HTTP-запрос на формирование отчета:

```http
GET /api/requests?fromDate=...&toDate=...&type=...
```
Этот запрос не возвращал данные сразу, а выступал как **ТРИГГЕР на асинхронную ОБРАБОТКУ**.

## 🔁 Поток обработки
1. Запрос сначала попадал в **API Gateway**
    - Gateway через **Keycloak** валидировал JWT-токен
    - проверял подпись и claims
    - после этого проксировал запрос в мой сервис
---
2. В **Extractor Service**:
	- Я принимал запрос и **сохранял его в БД со статусом `PENDING`** и отправлял 202 код в ФНС с `request_id`
	
	- Также вычислял **SHA-256 хеш от параметров запроса** (период, тип, id)  
	    и проверял дубликаты через уникальный индекс

👉 Если запрос уже был — повторно не обрабатывал (идемпотентность)

ФНС позже дергала ручку для получения статуса запроса по `request_id`

---
3. Далее запускалась обработка:
	- Статус менялся на `PROCESSING`
	- Я вычитывал данные из PostgreSQL

👉 Важно:  
мы работали не с основной БД 1С, а через API 1С. С этим мне помогал midle

---
5. Далее использовал паттерн **Outbox**:
	- XML (или ссылка на него) сохранялся в Outbox-таблицу в БД
	- это происходило в одной транзакции с записью запроса

👉 Благодаря этому гарантировалось, что сообщение **не потеряется**

Отдельный процесс уже отправлял данные из Outbox в Kafka

---
Я собирал метрики через Micrometer:

- количество успешных запросов (`SUCCESS`)
- количество ошибок (`FAILED`)
- время обработки (average / max)

# Проект - ВНУТРЕННИЙ для самой комании
ППВС — Платформа предоставления внешних сервисов (интеграция с ФНС)

**Тип проекта:** Внутренний проект для нужд самой компании

## Бизнес-задача
**Автоматизация обмена налоговыми данными с ФНС** — компания отправляет налоговую отчетность (данные из 1С) в ФНС и обрабатывает входящие требования от ФНС.

**Зачем:** *Раньше данные отправляли вручную через сайт ФНС или по email*. Нужно было автоматизировать процесс: выгружать данные из 1С, валидировать их по схемам ФНС, отправлять через защищенный канал и принимать требования обратно.

```
		[ФНС]
		|
		| GET /api/requests?fromDate=...&toDate=...
		| (с JWT-токеном в заголовке)
		↓
┌─────────────────────────────────────┐
│         API GATEWAY                 │
│  (Spring Cloud Gateway + KeyCloak)  │
│                                     │
│  - Проверяет JWT-токен              │
│  - Маршрутизирует запрос            │
└─────────────────────────────────────┘
		|
		| (токен проверен, запрос пущен)
		↓
┌─────────────────────────────────────┐
│         EXTRACTOR SERVICE (мой)     │
│                                     │
│  1. Принимает запрос от ФНС         │
│  2. Вычитывает данные из БД 1С      │
│  3. Формирует XML (JAXB)            │
│  4. Сохраняет запрос в БД (PENDING) │
│  5. Отправляет в Kafka              │
│  6. Возвращает 202 Accepted         │
└─────────────────────────────────────┘
		|
		| Kafka (XML)
		↓
┌─────────────────────────────────────┐
│         VALIDATION SERVICE          │
│         (Junior #2)                 │
│                                     │
│  - Валидирует XML по XSD-схемам     │
└─────────────────────────────────────┘
		|
		| Kafka (валидный XML)
		↓
┌─────────────────────────────────────┐
│         SENDER SERVICE (Middle)     │
│                                     │
│  - Отправляет XML в ФНС по HTTPS    │
└─────────────────────────────────────┘
		|
		| HTTPS
		↓
	  [ФНС]
```

## Детали реализации

### 1. Как вычитывал данные из 1С
> *"1С у нас хранила данные в **PostgreSQL** (не во внутренней БД 1С, а в обычной Postgres, куда 1С дублировала нужные таблицы). 
> 
> Я подключался к этой БД через **JDBC** и писал обычные SQL-запросы.*
> 
> _По запросу от ФНС — вычитывал конкретные данные по ID или периоду"_

### 2. Как формировал XML
> _"Использовал **JAXB** — создавал Java-классы (модели) по XSD-схемам ФНС, а потом просто конвертировал объекты в XML. 
> 
> Это было удобно: JAXB сам следил за тегами и атрибутами."_

#### Суть JAXB
JAXB =  
👉 преобразование: Java объект = XML

"Для формирования XML я использовал **JAXB**. Сначала я создал Java-классы, которые повторяли структуру XML, которую ждала ФНС. Эти классы я размечал аннотациями:
- _`@XmlRootElement` — для корневого тега_
    
- _`@XmlElement` — для обычных тегов_
    
- _`@XmlAttribute` — для атрибутов тегов_
    
- _`@XmlElementWrapper` — для обертки над списком элементов_

После этого я просто заполнял объекты данными из БД и вызывал метод `marshal()`. JAXB сам превращал объекты в правильный XML-документ.

После того как я создал Java-классы с аннотациями JAXB и заполнил объекты данными из БД, я использовал **`JAXBContext`** и **`Marshaller`**

```java
JAXBContext context = 
	JAXBContext.newInstance(Screen.class);

Marshaller marshaller = context.createMarshaller();

marshaller.setProperty(
	Marshaller.JAXB_FORMATTED_OUTPUT, Boolean.TRUE);
	
marshaller.marshal(screenObject, new File("output.xml"));
```

### 3. Как отправлял в Validation Service
> _"После формирования XML я отправлял его в **Kafka** (топик `xml-to-validate`). Validation Service слушал этот топик, валидировал XML и, если всё ок, отправлял дальше в Sender Service."_

**Почему валидация может не пройти**

|Причина|Пример|
|---|---|
|Отсутствие обязательных полей|В XML нет ИНН или КПП|
|Неправильный формат|Дата в формате `2024-13-40`|
|Несоответствие схеме XSD|Лишний тег или неправильный порядок|
|Невалидные значения|ИНН состоит из 12 цифр, а прислали 10|

### 4. Endpoint для ФНС (что приходило)
ФНС отправляла запрос с JWT-токеном в заголовке `Authorization`. 

Запрос сначала попадал в **API Gateway**, который через **KeyCloak** проверял токен. 
- Если токен валидный — Gateway пропускал запрос ко мне. 
- Если нет — возвращал `401 Unauthorized`


> _"ФНС отправляла **GET-запрос** на мой endpoint, например:_
```HTTP
GET /api/requests?fromDate=2024-01-01&toDate=2024-01-31&type=INCOME
```

_В запросе были параметры:_
- **Период (дата начала и конца)**
    
- _Тип документа (счет-фактура, акт и т.д.)_
    
- _Иногда ID конкретного документа_

Вот **принцип работы Endpoint**
1. ФНС отправляет запрос на твой endpoint
    
2. Ты ***сохраняешь запрос в БД*** (статус `PENDING`)
    
3. Возвращаешь ФНС **202 Accepted** (запрос принят, обрабатывается)
    
4. ***ФНС больше не ждет***. Она позже придет проверять статус по другому endpoint
    
5. А твой сервис в фоне: вычитывает БД → формирует XML → отправляет в Kafka → ждет валидацию → обновляет статус в БД (`SUCCESS` или `FAILED`)
    
6. ФНС периодически стучится: `GET /api/requests/{id}/status` и видит результат
