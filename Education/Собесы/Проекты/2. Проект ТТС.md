# 1. Проект
**ТТС (Транспортно-Технологическая Система)** — это система для автоматизации транспортной логистики.

## **Бизнес-задача:** 
**Автоматизация транспортной логистики** — *создание заявок* на перевозку, подбор транспорта, расчет стоимости, назначение водителя, *отслеживание статусов*.

ТТС — это система для управления перевозками:
👉 пользователь создает заявку →  
👉 система подбирает транспорт →  
👉 назначает машину и водителя →  
👉 дальше заявка живет и обновляется по статусам

## Микросервисы системы:
- **API Gateway** — входная точка, авторизация через Keycloak
- **Order Service (мой)** — управление заявками
- **Transport Service** — тарифы, маршруты, транспорт
- **Status Service** — обработка и валидация статусов
- **Notification Service** — отправка уведомлений
- **Kafka** — транспорт событий между сервисами

## Основные сложности, которые я решил

### 1. Outbox паттерн — гарантированная доставка в Kafka

**Проблема:** При создании заявки я сохранял ее в БД и отправлял событие в Kafka. Если после сохранения в БД, но до отправки в Kafka, сервис падал — событие терялось. Transport Service не получал заявку.

**Решение:** Реализовал паттерн Transactional Outbox.

- В одной транзакции сохраняю заявку и событие в таблицу `outbox`
    
- Отдельный шедулер каждые 5 секунд читает неотправленные события и отправляет в Kafka
    
- После успешной отправки помечаю событие как `processed`
    

---
### 2. Retry-механизм при вызове Transport Service

**Проблема:** При создании заявки я вызывал Transport Service через REST. Если он был временно недоступен, пользователь получал ошибку 500.

**Решение:** Добавил retry с экспоненциальной задержкой (3 попытки: 1с, 2с, 4с). Если все попытки провалились — возвращаю пользователю ошибку 503.

---
### 3. Идемпотентный consumer для Kafka

**Проблема:** Kafka могла доставить одно и то же событие дважды (at-least-once семантика).

**Решение:** Перед обновлением статуса проверяю текущий статус заявки в БД. Если статус уже соответствует событию — игнорирую.

---
### 4. Автоматическая отмена зависших заявок

**Задача:** Заявки в статусе CREATED, которые не были обработаны Transport Service больше 7 дней, должны автоматически отменяться.

**Решение:** Написал @Scheduled метод, который запускается каждую ночь в 3:00, находит такие заявки и меняет статус на CANCELLED с причиной "Автоматическая отмена по таймауту".

## Обновление статусов (через Status Service)
```
1. [Диспетчер] → POST /api/status → [Status Service]
   {
     "orderNumber": "ORD-001",
     "status": "IN_PROGRESS",
     "comment": "Погрузка начата"
   }
                                         ↓
2. [Status Service]:
   - проверяет, что заказ существует
   - проверяет допустимость перехода статуса
   - обогащает событие (timestamp, кто изменил)
   - сохраняет в свою БД (история статусов)
   - отправляет событие в Kafka (топик status-updates)
                                         ↓
3. [Order Service] (слушает Kafka):
   - получает событие об изменении статуса
   - обновляет статус в своей БД
   - сохраняет запись в таблицу аудита
                                         ↓
4. [Order Service] → отправляет событие в Kafka
   (для Notification Service)
                                         ↓
5. [Notification Service] → отправляет клиенту:
   "Ваша заявка ORD-001 в статусе 'В пути'"
```

**Почему Status Service вынесли отдельно:**
- Чтобы разгрузить Order Service (не принимать HTTP статусы)
    
- Подготовка к будущим источникам статусов (мобильное приложение водителей, GPS-трекинг)
    
- Единая точка входа для всех статусов
    
- Удобный аудит истории изменений
